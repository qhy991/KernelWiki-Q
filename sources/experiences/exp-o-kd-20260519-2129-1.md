## Challenge

reshape_and_cache_kernel stores KV cache entries in the same precision as input (FP16/BF16/FP32), wasting memory and DRAM bandwidth for large models and long sequences where KV cache is a major bottleneck


## Solution

Extend the kernel with a separate cache_t type parameter and compile-time is_fp8_e5m2_kv_cache flag; when true, each scalar KV value is quantized to FP8-E5M2 via fp8_e5m2_unscaled::vec_conversion before writing to cache, halving memory footprint with zero runtime overhead on the non-FP8 path thanks to if-constexpr

## Key Lessons

- Decouple cache storage type (cache_t) from input type (scalar_t) via a separate template parameter to enable lower-precision storage without affecting compute precision — applies whenever stored data tolerates reduced precision
- Use if-constexpr for compile-time dispatch between precision paths to guarantee zero runtime overhead on the original code path — the non-FP8 template instantiation compiles identically to the original
- FP8-E5M2 quantization for KV cache halves memory footprint and DRAM write bandwidth at the cost of a small per-element conversion, which is bandwidth-dominated in typical attention workloads
