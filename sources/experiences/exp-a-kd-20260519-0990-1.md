## Challenge

MoE token-sorting uses a single CUDA block (<<<1, 1024>>>) with shared-memory atomicAdd on local_offsets, forcing each thread to process ~4 tokens sequentially and concentrating all atomic contention within one SM, underutilizing the GPU


## Solution

Extract the token-sorting phase into a separate moe_token_sort_kernel launched with <<<actual_blocks, 256>>> using a grid-stride loop and global atomicAdd on cumsum_buffer, distributing work across multiple SMs to improve occupancy and reduce per-thread serial workload

## Key Lessons

- Splitting a single-block kernel into a dedicated multi-block kernel for the parallel-scan/sort phase distributes atomic contention across SMs instead of concentrating it in shared memory within one block — applies when the sorting workload is large enough to amortize the extra kernel launch overhead
- Grid-stride loops (for i = tid; i < numel; i += stride) automatically scale to any input size and block count, making the kernel portable across different GPU configurations without recalculating tokens_per_thread
- Moving atomicAdd targets from shared memory (local_offsets) to global memory (cumsum_buffer) trades lower per-atom latency for higher aggregate throughput by enabling concurrent execution across multiple SMs
