## Solution

StreamK scheduling distributes M*N output elements evenly across grid_size=num_sms blocks instead of one-block-per-tile, ensuring all SMs receive work even when the tile count is small; each block processes a contiguous range of output elements and computes the full K dot product independently, converting an under-occupied kernel into a fully-occupied one

## Key Lessons

- For tall-and-skinny GEMMs where K >> N, the number of output tiles can be smaller than the SM count, causing severe under-occupancy; StreamK scheduling distributes output elements across all SMs at element granularity instead of tile granularity, ensuring full utilization — applies whenever total_tiles < num_sms
- Launching grid_size = num_sms rather than grid_size = total_tiles is the key dispatch change for StreamK: it guarantees every SM gets work regardless of how few output tiles exist, and the per-block work shrinks proportionally to the SM count
- The persistent tile model (one block per tile) is suboptimal when N is small because tiles along the N dimension are few; element-level work distribution decouples parallelism from tile geometry and scales with the SM count instead
