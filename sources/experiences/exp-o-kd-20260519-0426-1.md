## Solution

Replace synchronous FLOAT4 B-tile loads with cp.async.ca.shared.global instructions, issuing async B copies first then A register loads then the BK×TM×TN compute phase, so the 16-step 64-FMA compute loop hides the asynchronous B transfer latency before a CP_ASYNC_WAIT_GROUP(0) barrier

## Key Lessons

- Use cp.async.ca.shared.global to issue asynchronous global-to-shared copies for one tile (B) while the other tile's data is already in shared memory and being consumed by compute — the BK×TM×TN FMA loop provides the latency-hiding window, and a final CP_ASYNC_WAIT_GROUP(0) ensures completion before the buffer swaps
- Decouple B-tile loading from A-tile loading in the main loop: issue async B copies first, then synchronous A register loads, then compute, then A shared-memory transpose — this ordering maximizes the overlap window between async transfers and arithmetic
- Use __cvta_generic_to_shared to obtain a shared-memory pointer suitable for cp.async destination addressing, and add byte offsets (ptr + i*4) to stride through the shared tile at 16-byte granularity
