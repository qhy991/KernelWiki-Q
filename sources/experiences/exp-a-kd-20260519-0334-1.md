## Challenge

Baseline kernel uses 96 KB shared memory with 128 threads per block; the 'optimized' version halves both to 48 KB and 64 threads, doubling the number of blocks from 8192 to 16384 for 1M elements and dramatically increasing launch and synchronization overhead for this lightweight computation kernel


## Solution

Reduce shared memory per block from 96 KB to 48 KB and block size from 128 to 64 threads, intending to improve occupancy by lowering shared memory pressure — but the lighter per-block work causes total launch count to double, and kernel launch overhead dominates execution time for this trivial computation

## Key Lessons

- Halving block size doubles the number of kernel launches; for lightweight kernels where per-block compute is minimal, launch and scheduling overhead dominates and smaller blocks cause severe regression
- Reducing shared memory per block only improves occupancy when shared memory is the actual occupancy limiter — if the kernel is compute-light and the original shared memory fit within SM capacity, the reduction provides no benefit while the smaller tile size adds overhead
- Block size and shared memory changes must be evaluated holistically: reducing shared memory in isolation can be beneficial, but coupling it with a smaller block size without profiling can invert the intended optimization
