## Challenge

A reduction kernel launches 4096 blocks (ceil(N/THREADS)) where every thread calls atomicAdd on a single output address, causing ~2M serialized atomic operations that contend on the same memory location and bottleneck performance


## Solution

Cap the grid to at most 36 blocks via a BLOCK_LIMIT constant; with 36 blocks × 512 threads = 18K threads, each thread accumulates more elements in its private loop but produces far fewer (~18K) atomicAdd operations, eliminating serialization on the shared output address

## Key Lessons

- Capping grid blocks (e.g., to 36) when kernels use atomicAdd on a single shared address drastically reduces atomic contention — fewer blocks means fewer serialized atomic operations, even though each thread does more private accumulation work
- The block-limit heuristic generalizes beyond NVLink contention: any pattern where many CTAs compete for a shared resource (atomic location, interconnect bandwidth) benefits from limiting active CTAs to the SM count
- For memory-bound reduction kernels, the private per-thread accumulation loop is nearly free compared to the cost of contended atomics — letting fewer threads do more private work is a net win
