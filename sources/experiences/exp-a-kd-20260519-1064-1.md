## Challenge

The block-level MoE sum reduction kernel with 256-thread blocks and grid(16, 256) was replaced with a warp-per-token kernel using 128-thread blocks and grid(128, 64), which severely regressed performance by reducing SM occupancy and eliminating intra-thread L1 cache locality across hidden_dim iterations


## Solution

The warp-per-token kernel assigns one warp per token via warp_id/lane decomposition and uses 128-thread blocks, which doubles total block count to 8192 while halving block size — this reduces resident warps per SM and eliminates L1 cache reuse because each thread processes only one hidden_dim element per token instead of iterating across multiple positions within a token's topk rows

## Key Lessons

- Warp-per-token dispatch is not universally superior to block-level dispatch — for small token counts (<=256) with moderate hidden_dim, block-level threads iterating across multiple hidden_dim positions keep topk rows resident in L1 cache, while warp-per-token's single-element-per-thread stride eliminates this reuse
- Larger block sizes (256 threads) provide better SM occupancy than smaller blocks (128 threads) when the kernel is memory-bound — the block-level kernel's 256-thread blocks keep more warps resident to hide memory latency
- Dispatch strategies should be selected based on problem dimensions: warp-per-token benefits large token counts where block-level grid_y saturates, but regresses for small token counts where the block-level approach already provides sufficient parallelism and better cache locality
