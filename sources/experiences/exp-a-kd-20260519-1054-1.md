## Challenge

Scalar per-element fill of a sorted_token_ids padding array issues one 4-byte STG per iteration, underutilizing GPU store unit coalescing width and requiring 4x more store transactions than necessary for the 8192-element array


## Solution

Replace scalar int32 stores with vectorized int4 stores, packing 4 int32 values into a single Vec struct and writing 16 bytes per store transaction, reducing total store count by ~4x and improving memory coalescing utilization

## Key Lessons

- Use vectorized int4/float4 stores instead of scalar int32/float stores for bulk fill or copy operations — each vector store issues a single 16-byte transaction vs four separate 4-byte transactions, reducing store count by 4x and improving memory coalescing
- Precompute the fill value into a vector struct once (fill_vec.x = fill_vec.y = fill_vec.z = fill_vec.w = val) outside the loop rather than broadcasting per-element, reducing loop-body instructions
