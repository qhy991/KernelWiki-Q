## Solution

Replace the loop-based reduction with a WarpReduce helper that uses volatile shared memory accesses to perform the same 4-step fmaxf reduction without any __syncthreads() calls, exploiting the fact that all 16 threads per group are within a single warp and volatile reads/writes provide sufficient ordering guarantees

## Key Lessons

- Replacing __syncthreads()-based loop reduction with volatile shared memory WarpReduce eliminates 4 barrier calls (~80-120 cycles), but yields no measurable speedup when the kernel is memory-bandwidth-bound and the reduction phase is a tiny fraction of total execution time
- Volatile shared memory reduction is only beneficial when the reduction itself is the dominant bottleneck — in bandwidth-bound kernels with large per-group global memory reads, the barrier savings are dwarfed by memory latency
- Unrolled reduction via volatile shared memory (no __syncthreads()) works correctly only when all participating threads are in the same warp; verify the thread count per group is ≤32 before applying this pattern
