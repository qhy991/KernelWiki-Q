## Challenge

Interleaved packing splits even-indexed and odd-indexed 4-bit nibbles into separate byte lanes within each 32-bit word, forcing the upconversion kernel to deinterleave by extracting nibbles from non-adjacent byte positions (e.g., n0 from byte0, n1 from byte2) before sign-extending to int8, adding cross-lane extraction overhead without reducing total instruction count


## Solution

Switch to sequential (non-interleaved) packing where nibbles are stored in natural byte order within each 32-bit word (byte0=n0|n1, byte1=n2|n3, etc.), allowing direct sequential nibble extraction from each byte without cross-lane deinterleaving — though in this global-memory-only microbenchmark the instruction count remains identical and the measured speedup is within noise

## Key Lessons

- Sequential packing (natural nibble order within each byte) enables prmt-based and CuTe-layout-aware upconversion paths in full-weight kernels like Machete, but in a standalone global-memory-only microbenchmark the identical instruction count (4 byte extracts, 8 masks, 8 sign-extends, 8 writes) means the reordering alone is not sufficient to produce a measurable speedup
- Data layout optimizations that target shared-memory tile access patterns or tensor core (wgmma) upconversion paths may show neutral results in isolated microbenchmarks — validate layout changes within the full kernel pipeline where the layout enables downstream hardware acceleration
- Interleaved vs sequential packing is a data layout decision whose benefit depends on the consumer kernel: interleaved can be beneficial for vectorized half2/fp16 loads while sequential is better for int8/fp8 targets where nibbles map directly to output elements
