## Solution

Split the unified barrier into barrier_at_start_sim and barrier_at_end_sim with dedicated start/end signal arrays and a block-level _flag counter incremented per barrier, but the added _flag read (+1) and conditional write-back (thread 0 only after each barrier) introduces global memory overhead not amortized in lightweight 2-barrier patterns, causing ~9% regression

## Key Lessons

- Splitting a unified barrier template into specialized start/end variants adds a block-level _flag read and conditional write-back per barrier invocation — this overhead is not amortized when inter-barrier work is lightweight, making the original per-thread counter approach faster
- A single-thread conditional global store (threadIdx.x == 0 writing _flag after __syncthreads) adds an extra global memory transaction per barrier that becomes a measurable overhead when the barrier count is high and inter-barrier computation is minimal
- Per-thread counter increment (self_counter += 1) fused as a read-modify-write is cheaper than a separate _flag read (+1) followed by a single-thread write-back, because the fused approach avoids the extra global load and the single-writer serialization point
