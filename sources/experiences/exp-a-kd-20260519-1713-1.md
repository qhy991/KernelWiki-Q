## Solution

Template-parameterize warpReduceSum with a compile-time numLanes count and compute the minimum active lanes via _nextPow2(maxActiveLanes) for the final block-level reduction, reducing shuffle rounds from log2(32)=5 to log2(actual_warps) when the block size is known at compile time

## Key Lessons

- Template-parameterizing warp shuffle reduction with the actual active lane count eliminates unnecessary shuffle rounds when the number of contributing threads is known at compile time — applies to any blockReduce variant where block size is a compile-time constant smaller than 1024
- Shared memory barrier and global memory access latency dominate reduction kernels, making small instruction savings (3 SHFL instructions) negligible — significant speedup requires attacking the memory bottleneck, not the arithmetic count
- _nextPow2 via __builtin_clz computes the ceiling power-of-two at compile time, enabling static_assert-validated template parameters without runtime overhead
