## Challenge

HGEMM kernel uses a flat 2D grid without block-level swizzle, so thread blocks map to output tiles in strictly sequential order along the N dimension, causing poor L2 cache locality when multiple blocks access overlapping B-matrix rows


## Solution

Enable BLOCK_SWIZZLE=true and restructure the launch grid from 2D to 3D, interleaving thread blocks across the N dimension via blockIdx.z so that concurrently executing SMs access spatially closer rows of the B matrix, improving L2 cache hit rate

## Key Lessons

- Block-level swizzle interleaves thread blocks across the N dimension via a 3D grid so that blocks running on adjacent SMs access nearby rows of the B matrix, improving L2 cache reuse — but the benefit only materializes when N is large enough that N_SWIZZLE > 1 and multiple z-slices actually exist
- At small problem sizes (e.g. M=N=K=512 with stride=2048), N_SWIZZLE evaluates to 1, the 3D grid degenerates to 2D, and the swizzle has zero effect — always verify the tuning parameter (stride) relative to the actual problem dimensions before expecting gains
- The compile-time BLOCK_SWIZZLE flag controls whether blockIdx.z contributes to bx computation, making the same kernel source serve both swizzled and non-swizzled launch configurations without separate kernel variants
