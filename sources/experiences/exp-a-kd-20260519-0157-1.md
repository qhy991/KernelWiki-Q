## Solution

Split the K=4096 dimension into 8 slices of K_len=512, launch 8 gemm_partial kernels concurrently on separate CUDA streams so the GPU scheduler interleaves their blocks across SMs, then reduce partial sums with a lightweight reduce_k kernel — exploiting stream-level concurrency to hide the latency of shorter K-loops behind overlapping execution

## Key Lessons

- Split-K decomposition parallelizes the K-dimension across multiple CUDA streams so the GPU scheduler can interleave blocks from independent partial GEMMs on underutilized SMs — applies when the block grid is smaller than SM count and K is large enough to amortize the reduction overhead
- Stream concurrency with split-K yields sub-linear speedup when the combined block count across all splits exceeds available SMs (512 blocks vs 108 SMs here), causing SM contention that limits practical parallelism to ~2x rather than the full 8x split factor
- Use events (cudaEventRecord + cudaStreamWaitEvent) to synchronize partial kernels before a reduction step — this avoids cudaDeviceSynchronize and allows the reduction kernel to launch on the default stream only after all partials complete
