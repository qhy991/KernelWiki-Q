## Solution

Replace 512-thread scalar kernel with 256-thread vectorized kernel using int4 loads (16 bytes/lane) and uint2-based struct decomposition into fp8x8_to_bf16x8 conversion chains, aiming to increase memory throughput — but on sm_80 the conversion still decomposes to identical per-element scalar FP8→BF16 calls with higher register pressure and struct reshuffling overhead, causing a net regression

## Key Lessons

- Vectorized int4 loads do not improve throughput when the original byte-granularity loads from 512 threads already saturate memory bandwidth for small payloads (~32KB) — wider loads only help when the load pipeline itself is the bottleneck, not when compute conversion cost dominates
- On sm_80 without native FP8→BF16 vector instructions, struct-based conversion chains (fp8x8_to_bf16x8 → fp8x4_to_bf16x4 → fp8x2_to_bf162_scalar → fp8_to_bf16_scalar) decompose to identical per-element scalar conversions, adding register pressure and decomposition overhead without reducing instruction count
- Higher per-lane register footprint (int4 + uint2 + bf16_8_t structs vs single uint8_t + bfloat16) reduces occupancy and can cause net regression on register-constrained workloads — always profile register count changes when switching from scalar to vectorized kernel designs
