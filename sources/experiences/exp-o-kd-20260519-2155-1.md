## Challenge

CUTLASS GEMM with fixed TileShape 128x128x64 wastes 87.5% of M-dimension compute when M=16 because the tile covers 128 rows but only 16 contain valid data, and the large N tile (128) limits concurrency to only 16 threadblocks along the N dimension


## Solution

Switch to M-adaptive TileShape 16x64x128 with WarpShape 16x64x64 so the tile M dimension exactly matches M=16 (eliminating wasted rows), the smaller N tile (64) doubles threadblock count from 16 to 32 for higher occupancy, and the larger K tile (128) amortizes pipeline startup over more K iterations

## Key Lessons

- Match TileShape M dimension to the actual problem M size in CUTLASS when M is small — a tile larger than M wastes compute on out-of-bounds rows that get masked out, causing up to (TileM-M)/TileM fraction of wasted work
- Shrink the N tile dimension and grow K to increase threadblock count along N for small-M GEMM — smaller N tiles create more concurrent threadblocks (2048/64=32 vs 2048/128=16) and larger K tiles amortize pipeline fill overhead across more iterations
- For batched or small-M GEMM workloads on SM80+, use M-adaptive tile configurations rather than a single fixed tile — the optimal tile shape is highly sensitive to M and can yield 4x+ speedups when properly tuned
