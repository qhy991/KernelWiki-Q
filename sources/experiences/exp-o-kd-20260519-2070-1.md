## Solution

Match the TileShape M dimension exactly to the problem M=16 by switching to GemmShape<16,64,128> with WarpShape<16,64,64>, achieving 100% warp M utilization while the deeper K=128 pipeline stage compensates for reduced tile coverage

## Key Lessons

- When the GEMM M dimension is small (e.g., ≤32), set TileShape M equal to the actual M to eliminate padding waste in warp-level computation — the default large tile shapes assume batch/generic M sizes and leave most lanes idle on small-M problems
- Increase the K tile dimension (e.g., 64→128) to deepen the software pipeline and compensate for the reduced M tile, keeping shared memory occupancy and tensor core utilization high despite the narrow M dimension
