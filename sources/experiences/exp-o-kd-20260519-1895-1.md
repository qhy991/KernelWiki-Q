## Challenge

The before code dispatches two sequential GPU kernels for FP8 quantization: a segmented_max_reduction kernel (256 blocks with shared memory reduction, __syncthreads barriers, and atomicMaxFloat contention across blocks) to compute the global max-abs scale, followed by a quant kernel that redundantly re-reads the entire 1MB input for per-element division


## Solution

Compute the max-abs scale on the CPU where the input data already resides in host memory, eliminating the entire segmented_max_reduction GPU kernel launch, its shared memory reduction with __syncthreads barriers, the atomicMaxFloat contention across 256 blocks, and the redundant 1MB global memory re-read

## Key Lessons

- When input data is already available on the host, pre-computing scalar reductions (max/sum) on the CPU avoids an entire GPU kernel launch, shared memory reduction, and a full redundant global memory traversal — applies when the reduction produces a single scalar and host-side data is readily accessible
- For small-to-medium datasets (~1MB / 256K elements), the overhead of a GPU reduction kernel (block synchronization, atomic operations across hundreds of blocks, kernel launch latency) can exceed the cost of a simple CPU loop, making host-side pre-computation faster despite lacking GPU parallelism
- Eliminating a two-pass GPU pipeline (reduction + elementwise) into a single-pass design by moving the reduction to CPU nearly halves total latency when data traversal cost dominates (~2x speedup)
