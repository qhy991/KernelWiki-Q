## Solution

Switch from ld.global.cs.f32 (streaming, bypasses L2) to ld.global.cg.f32 (cache-all levels) for sub-128-bit scalar loads, allowing the first thread's load to populate the L2 cache line so subsequent threads hitting the same line benefit from L2 hits rather than re-fetching from DRAM

## Key Lessons

- For sub-128-bit (scalar) loads where neighboring threads access the same 128-byte cache line, prefer ld.global.cg over ld.global.cs to allow L2 cache line sharing — but the benefit is marginal when the working set far exceeds L2 capacity, since the bottleneck shifts to raw memory bandwidth
- Cache hint selection (.cs vs .cg) is a secondary optimization compared to widening access width to 128-bit (e.g., float4 loads) — scalar loads at 32-bit have fundamentally lower memory throughput regardless of cache policy
- When working set exceeds L2 capacity (~230MB here), cache hit rate is low regardless of hint, so cache policy changes produce near-zero improvement; prioritize vectorized loads or access pattern restructuring instead
