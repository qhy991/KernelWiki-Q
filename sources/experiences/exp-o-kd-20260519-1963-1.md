## Challenge

Two separate kernels are launched sequentially: merge_attn_states_kernel writes BF16 to an intermediate buffer via uint4 (128-bit) stores, then quantize_fp8_kernel reads the entire BF16 buffer back from global memory to convert it to FP8, incurring double memory traffic (write BF16 + read BF16 + write FP8) and an extra kernel launch overhead


## Solution

Fuse FP8 quantization directly into the merge kernel using a compile-time bool template parameter USE_FP8_OUTPUT: each thread computes the merged float32 result and immediately converts to FP8 via float_to_fp8e4 with a pre-inverted scale read from device memory, storing via narrower uint2 packs (8 bytes FP8 vs 16 bytes BF16), eliminating the intermediate BF16 buffer and the second kernel launch entirely

## Key Lessons

- Fusing a post-processing kernel (quantization/cast) into the producer kernel eliminates intermediate buffer allocation, a full write-then-read round-trip through global memory, and kernel launch overhead — especially significant when the intermediate tensor is large (4.2M elements here saved ~16 MB of traffic)
- Use compile-time bool template parameters (e.g., USE_FP8_OUTPUT) with if constexpr to switch output types without runtime branching cost, enabling a single kernel to serve both BF16 and FP8 output paths from the same source
- When fusing quantization into a compute kernel, store output via narrower packs (FP8 = uint2 at 8 bytes vs BF16 = uint4 at 16 bytes per pack) to further reduce global memory store volume
