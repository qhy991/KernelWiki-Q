## Solution

Replaces single CUB BlockReduce with groupwise shared-memory reduction: partition 256 threads into 32 groups of 8 threads each, compute per-group absmax into s_max_vals[1024] shared memory, then reduce via custom warpReduceMaxSpecialized. This adds three __syncthreads() barriers and shared-memory traffic that outweigh any parallelism benefit at this scale, causing a 9.4% regression.

## Key Lessons

- CUB BlockReduce internally uses optimized warp shuffles with no explicit __syncthreads() barriers — replacing it with manual shared-memory + warp reduction adds 3 __syncthreads() calls and 4KB shared-memory traffic per block, which overwhelms any parallelism benefit when thread count is small (256 threads, hidden_size=4096)
- Group-partitioned iteration with small threads_per_group (8 threads per group) causes poor memory coalescing since threads within a warp access non-contiguous group offsets, unlike the original blockDim.x-stride loop where consecutive threads access consecutive elements
- Custom reduction over shared memory is only worthwhile when the reduction itself is a bottleneck (large data, many threads) or when producing finer-grained outputs (per-group scales) — for simple absmax with 256 threads, CUB BlockReduce is already optimal
