## Challenge

A reduction kernel launched with 16384 blocks (ceil(8M/512)) has each thread accumulate a local sum then call atomicAdd on a single output address, causing ~8M serialized atomic operations that stall the GPU


## Solution

Cap the thread block count at 36 via a BLOCK_LIMIT constant, reducing total atomicAdd calls from ~8M to ~18K and eliminating the serialization bottleneck while each thread still processes multiple elements through its strided loop

## Key Lessons

- Capping thread block count to a small multiple of SM count avoids atomic serialization when many blocks contend on a single output address — the strided loop already distributes work so fewer blocks still cover all data
- Excessive parallelism can degrade performance when the synchronization primitive (atomicAdd on a shared location) becomes the bottleneck — more blocks means more atomic contention, not more throughput
