## Challenge

Two-kernel approach for top-K selection with offset: topk_select_kernel writes raw indices to global memory via uncoalesced scatter writes, then a separate add_offset_kernel performs a second coalesced read-modify-write pass over the intermediate 32 KB buffer, incurring one extra kernel launch overhead (~5–10 μs) and a full global-memory round-trip


## Solution

Fuse both kernels into a single topk_fused_ragged_kernel that allocates shared memory (s_indices[TopK]) for intermediate scatter writes from fast_topk_cuda_tl, then performs the offset addition inline during the final coalesced global-memory write, eliminating one kernel launch and the entire intermediate global-memory buffer allocation, write, and read

## Key Lessons

- Fuse a producer-consumer kernel pair into a single kernel when the intermediate data (here, raw top-K indices) fits in shared memory — eliminates one kernel launch overhead (~5–10 μs) and removes the global-memory write+read round-trip for the intermediate buffer
- Redirect scatter writes from global memory to shared memory when the intermediate result size (TopK elements per block) is small enough — shared memory offers ~30x lower latency and no coalescing penalty for random-index scatter patterns
- Inline post-processing (offset addition) into the final coalesced write of a fused kernel rather than using a separate pass — the offset is a per-block constant loaded once, so adding it costs one register add per element with no extra memory traffic
