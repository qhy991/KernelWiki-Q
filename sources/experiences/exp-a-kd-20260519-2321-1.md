## Challenge

MoE alignment kernel counts all tokens per expert via unconditional atomicAdd without token-mask filtering, but the attempted fix of adding per-token mask-based conditional counting regresses due to instruction overhead exceeding the savings from fewer atomicAdds in a small-count scenario (4096 elements across 1024 threads with low shared-memory contention)


## Solution

Attempted mask-based filtering by adding a null-pointer check and global-memory token_mask[i / topk_num] read per iteration to conditionally zero out atomicAdd contributions for masked tokens, but the extra null comparison, integer division, and global-memory load per iteration cost more than the saved atomicAdd operations in this low-contention small-workload regime

## Key Lessons

- Per-iteration instruction overhead (null-pointer check, global-memory load, integer division) can exceed savings from reduced atomicAdd volume when total iteration count is small and shared-memory atomic contention is low — applies to MoE alignment kernels with modest element counts
- Unconditional atomicAdd(ptr, 1) with constant 1 is preferable to mask-based conditional atomicAdd(ptr, mask) when the filtering logic costs more instructions than the filtered-out atomics — profile before adding conditional paths to tight counting loops
- Adding a global-memory read per iteration to skip shared-memory atomics is a net loss when the atomic target is low-contention shared memory and the element count per thread is small (e.g., ~4 elements per thread)
