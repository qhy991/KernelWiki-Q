## Challenge

The warp-level shared-memory iterator constructor uses a triple-nested unrolled loop over (inst_m_idx, inner_idx, access_m_idx) with a conditional branch (if access_idx == ldsm_vec_num) to locate each thread's smem origin offset, generating branch instructions and surplus arithmetic that accumulates when the iterator is freshly constructed every iteration


## Solution

Replace the triple-nested loop and conditional branch with three direct integer-division/modulo operations that decompose ldsm_vec_num into (access_m_idx, inner_idx, inst_m_idx), eliminating all loop overhead and branch instructions in the constructor hot path

## Key Lessons

- When an unrolled loop iterates over a multi-dimensional index space only to match a single target value, replace it with direct integer-division/modulo decomposition — this eliminates loop overhead, branch instructions, and redundant arithmetic that even a compiler may not fully optimize away
- Constructor-path overhead in CUDA accumulates when objects are constructed inside hot loops (e.g., warp iterators per MMA iteration); even small per-construction savings yield measurable speedups at high iteration counts
- The mapping from a flat index (ldsm_vec_num) to a multi-dimensional offset (access_m_idx, inner_idx, inst_m_idx) can be inverted via div/mod with the known dimension strides, provided the total iteration count matches the flat range
