## Solution

Replace scalar bf16 load/store with float4-width vectorized transactions using an aligned vec_n_t<__nv_bfloat16,8> struct and reinterpret_cast, reducing load/store instruction pairs per thread from 9 to 1-2 via 16-byte vector width

## Key Lessons

- Scalar strided access with CTA_SIZE=64 over bf16 already produces warp-coalesced transactions (32 consecutive threads access 32 consecutive 2-byte elements), so vectorization alone yields minimal throughput gain — the hardware was already merging requests effectively
- Vectorized load/store reduces instruction count (9 scalar pairs to 1-2 vector pairs) but instruction-count savings alone may not translate to latency reduction when the kernel is memory-bandwidth-bound and the memory subsystem is already saturated via coalescing
- The real benefit of vectorized access emerges when combined with other changes (smaller CTA, compile-time unrolling, altered launch configuration) that together reshape occupancy and scheduling — vectorization in isolation on an already-coalesced pattern is near-neutral
