## Challenge

Shared-memory matrix transpose uses a 32x33 padded array (1056 elements) to avoid bank conflicts on the transposed read path, wasting 3.1% shared memory per tile without any throughput benefit over alternative conflict-free schemes


## Solution

Replace padding-based bank conflict avoidance (32x33 array) with XOR swizzle (threadIdx.x ^ threadIdx.y) on a compact 32x32 array, which permutes column indices so each warp accesses all 32 banks without padding overhead

## Key Lessons

- XOR swizzle (threadIdx.x ^ threadIdx.y) and padding (32→33 columns) both achieve bank-conflict-free shared memory access in transpose kernels — the XOR approach saves 3.1% shared memory per tile but the savings are too small to affect occupancy for 32x32 tiles
- When comparing two equally effective bank-conflict avoidance strategies, shared memory footprint reduction alone is unlikely to produce measurable speedup unless tile dimensions are large enough for the savings to unlock higher occupancy
