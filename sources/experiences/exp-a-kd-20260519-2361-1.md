## Solution

Eliminate the transpose kernel entirely by indexing the original scale tensor directly with the transposed access pattern (bn * nbk + bk instead of bk * nbn + bn), removing one kernel launch, one device allocation, and the associated synchronization overhead

## Key Lessons

- Eliminating a small GPU transpose kernel by using the original layout with adjusted indexing removes kernel-launch overhead and a device allocation, even when the transpose itself is cheap relative to the dominant workload
- When the dominant kernel (weight-matrix traversal) is memory-bound and the scale data fits entirely in L2 cache, the layout of the small scale tensor has negligible impact on overall performance — the transpose removal saves code complexity rather than runtime
- Hardware features like SM90 CUTLASS Sm90BlockwiseScaleConfig with explicit GMMA major-mode parameters can eliminate data-layout transformations that were previously required, simplifying the software pipeline
