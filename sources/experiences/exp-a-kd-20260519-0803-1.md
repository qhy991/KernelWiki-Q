## Challenge

The kernel hardcodes 16 groups per block (256 threads) but num_groups=17 is not divisible by 16, causing block 1 to have only 1 valid group; 240 out of 256 threads must skip absmax and quantization via runtime bounds guards, producing severe branch divergence and wasted computation


## Solution

Change GROUPS_PER_BLOCK from 16 to 1, launching num_groups blocks of 16 threads each so every thread block processes exactly one valid group, eliminating all out-of-bounds guards and branch divergence at the cost of very low occupancy per SM

## Key Lessons

- Setting GROUPS_PER_BLOCK=1 eliminates out-of-bounds guards and branch divergence when num_groups is not divisible by the previous groups-per-block value, improving correctness even when timing is neutral for small workloads
- For tiny data volumes (a few KB), kernel launch overhead and memory latency dominate execution time; structural optimizations that reduce divergence or improve occupancy show no measurable speedup because the kernel is not compute- or bandwidth-bound
- Very low occupancy (16 threads per block vs 2048 per SM on A800) wastes SM resources but is harmless for sub-microsecond kernels — the trade-off favors correctness and simplicity over occupancy for small group counts
