## Solution

Fuse both kernels into a single advance_step_flashinfer_kernel so each thread reads seq_len and block_tables once, computes block_index/block_offset once, and derives all six outputs (token, seq_len, position, slot_mapping, paged_kv_last_page_len, block_table_bound) from register-held values, eliminating redundant global memory reads and the second kernel launch overhead

## Key Lessons

- Fusing two lightweight kernels that share input arrays (seq_lens, block_tables) into one kernel eliminates redundant global memory reads and kernel launch overhead (~5-10us per launch), yielding significant speedup for small batch sizes where launch cost dominates total execution time
- Kernel fusion is most beneficial when both kernels operate per-element with low arithmetic intensity — the per-thread compute remains identical but the fixed launch and memory overhead is halved
- When kernels have a producer-consumer dependency via global memory (kernel 1 writes seq_lens, kernel 2 reads it), fusing them allows the consumer to use register-resident values directly, avoiding a global memory round-trip
