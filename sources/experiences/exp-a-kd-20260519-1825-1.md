## Solution

Replace the hardcoded const pipe_stages=4 with a runtime stages=2 parameter (Turing mode), halving the stage-dependent shared memory terms (sh_a_size, sh_b_size, sh_s_size, sh_g_idx_size) and reducing total shared memory from 60544 to 34944 bytes—a 42% reduction that comfortably fits within the 64KB shared memory limit

## Key Lessons

- Reducing pipeline depth from 4 stages to 2 stages halves stage-dependent shared memory buffers (sh_a_size, sh_b_size) in quantized GEMM kernels—the tradeoff is slightly more global-memory traffic per tile, but the shared memory savings enable configurations that would otherwise exceed the 64KB Turing shared memory limit
- Parameterizing pipeline stages as a runtime argument rather than a compile-time constant enables GPU-architecture-specific tuning (e.g., stages=4 for Ampere with 100KB+ shared memory, stages=2 for Turing with 64KB) without code duplication
- Shared memory sizing must account for all stage-scaled terms (activation tiles, weight tiles, scales cache, group indices) plus fixed overhead plus framework reservations—a configuration that appears to fit in 64KB may fail when a 512-byte reservation is subtracted
