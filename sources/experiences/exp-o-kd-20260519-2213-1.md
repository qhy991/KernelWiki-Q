## Solution

Remove the split-K heuristic entirely and issue a single cublasSgemm with the full K dimension on a dedicated CUDA stream, eliminating 3 extra cuBLAS launches, the 4x partial-result buffer, and the reduction kernel while keeping all operations on one stream for minimal synchronization overhead

## Key Lessons

- A single cuBLAS Sgemm with full K dimension is more efficient than manual split-K decomposition on SM80 for moderately large K — cuBLAS internally handles tiling and occupancy, making hand-rolled split-K overhead (extra launches, partial buffers, reduction kernel) a net negative
- Removing conditional scheduling heuristics (StreamK vs Persistent) simplifies the code path and avoids branch divergence in the host loop — applies when cuBLAS already has efficient internal scheduling for the given shape
- Placing the GEMM and post-processing kernel on the same CUDA stream enables implicit serialization without cudaDeviceSynchronize, reducing synchronization overhead in the hot loop
