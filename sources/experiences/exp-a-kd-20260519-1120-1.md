## Challenge

Grid dimension is set to raw batch size m=8, launching only 8 blocks regardless of SM count. On a GPU with 108 SMs, 100 SMs sit idle while 8 blocks serially process rows via grid-stride loops, severely underutilizing hardware parallelism.


## Solution

Replace raw m with effectiveRows = round_up(m, 128) as the grid size upper bound. For m=8 this yields 128 blocks, spreading the 120 padding-row zero-fills across 120 additional blocks so all SMs participate concurrently, reducing per-block workload and improving throughput by 1.43x.

## Key Lessons

- Use the padded/effective row count (round_up to a tile multiple) rather than raw batch size when computing grid dimensions — kernels that already iterate in grid-stride loops over padding rows can safely spread that work across more blocks to saturate SMs
- When a kernel's grid-stride loop covers both real and padding rows, the grid size can be set to cover all effective rows without correctness impact, since padding rows are already handled by the loop body
