## Solution

Vectorize to half2 processing (two FP16 elements per thread) via reinterpret_cast<half2*> loads and per-lane __hadd, halving thread count from 256 to 128 per block — however this reduces concurrent memory requests and slightly lowers occupancy, and since the kernel is purely memory-bound with 1.5 MB data fitting in L2 cache, the scalar pattern already saturates bandwidth, yielding no measurable improvement

## Key Lessons

- Consecutive half-width accesses from a 256-thread block already coalesce into wide memory transactions on sm_80 — each warp's 32 half-addresses merge into full coalesced transactions, making half2 vectorization unnecessary for memory-bandwidth saturation
- Half2 vectorization halves thread count per block (256→128), reducing concurrent memory requests and potentially lowering occupancy, which can offset the doubled per-thread throughput in memory-bound kernels
- Memory-bound kernels with small working sets that fit in L2 cache may already saturate available bandwidth with scalar access patterns — vectorization only helps when the scalar path leaves bandwidth underutilized
