## Solution

Replace batch-split 2D grid (1024 threads/CTA) with a per-token 1D grid (64 threads/CTA) where each CTA handles exactly one token's cache copy, giving each thread 9 meaningful iterations (576/64) instead of 0–1, while the smaller CTA size increases concurrent CTAs per SM for the memory-bound workload and eliminates all split/partial-block boundary conditionals

## Key Lessons

- Size CTAs to match the per-element work granularity: with entry_size=576, using 64 threads gives each thread 9 iterations (576/64) versus 0–1 iterations with 1024 threads (576/1024), eliminating thread idle time in memory-bound gather kernels
- Prefer per-token (1D grid, one token per CTA) over batch-split (2D grid, split of blocks per CTA) for paged KV-cache gather: it removes split boundary conditionals, partial-block handling, and nested block/token loops, reducing divergent branching and address computation overhead
- Smaller CTA sizes increase the number of concurrent CTAs resident per SM, which is critical for memory-bound workloads where hiding latency depends on having enough outstanding memory requests from independent warps
