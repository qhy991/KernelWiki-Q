## Challenge

FP8 KV cache dequantization kernel passes k_scale and v_scale as scalar float by-value parameters through the kernel launch parameter buffer, incurring overhead in the parameter cache path and occupying more parameter slots than necessary


## Solution

Pass k_scale/v_scale as const float* pointers and dereference once into local register variables at kernel entry (float k_s = *k_scale), reducing kernel launch parameter overhead and enabling the compiler to treat the locally cached value as a pure register operand

## Key Lessons

- Passing small scalar kernel parameters as device-memory pointers (const float*) instead of by-value floats can reduce kernel launch parameter overhead and yield measurable speedups in memory-bound microbenchmarks — applies when the kernel is lightweight and parameter-cache latency is a non-trivial fraction of total runtime
- Pointer-based scale parameters enable future extension to per-head or per-token dynamic scaling (lookup from a tensor) without changing the kernel signature — prefer pointer parameters for scales that may eventually become non-uniform
- In real-world kernels where memory accesses dominate (e.g., paged_attention_kernel), scalar vs pointer parameter overhead is negligible relative to overall kernel runtime; this optimization matters most in isolated microbenchmarks and lightweight kernels
