## Challenge

Tiled GEMM with (M,K)×(K,N) layout pads M=2 to 64 rows (tile size 64×64×128), wasting 62 of every 64 rows and computing 4× more MACs than necessary across 8 experts


## Solution

Swap the multiplication to compute (N,K)×(K,M)=(1024,1536)×(1536,2) so the large N=1024 dimension fills the 128-wide tile efficiently and the small M=2 only pads to 16, reducing padded MACs from 100M to 25M per expert; transpose the result back to (M,N) afterward

## Key Lessons

- For small-M batched GEMM, swap the multiplication to (N,K)×(K,M) so the large dimension occupies the M-tile axis and the small dimension pads to a tiny N-tile — reduces padded computation proportional to (pad_M_before/pad_N_after) ratio
- Dimension swapping is transparent to the compute kernel (identical GEMM code) — all changes are host-side data layout preparation and a post-GEMM transpose, applicable to any tiled GEMM with highly asymmetric M vs N
- When tile alignment forces M=2→64 (32× over-pad) but N=1024 naturally aligns, the 32× padding waste dominates throughput; reordering dimensions to avoid that waste yields up to ~4× speedup before kernel-level tuning
