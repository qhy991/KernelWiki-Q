## Challenge

In the 8x16 SGEMM main loop (BM=128, BN=256, 256 threads), each thread synchronously loads 16 floats for the next B tile via FLOAT4 global-to-shared copies, blocking the warp and preventing overlap with the BK=16 compute loop (128 FMAs per thread per step) and the A-tile online transpose


## Solution

Issue 4 cp.async.ca instructions for the next B tile (16 bytes each, covering 16 floats) at the start of the loop body with CP_ASYNC_COMMIT_GROUP, allowing the async copies to proceed in parallel with A loading, the full BK=16 compute loop, and the A-tile online transpose, then use CP_ASYNC_WAIT_GROUP(0) before __syncthreads to guarantee completion

## Key Lessons

- Replace synchronous FLOAT4 global-to-shared loads with cp.async.ca to overlap memory transfers with compute and other shared-memory operations — applies whenever the loop body has sufficient compute work (e.g., BK iterations of FMA) to hide the async copy latency
- Use CP_ASYNC_COMMIT_GROUP after issuing async copies and CP_ASYNC_WAIT_GROUP(0) before __syncthreads to guarantee all async groups complete — the commit/wait pair forms a boundary that is safe for double-buffered shared memory swapping
- Convert the shared-memory destination pointer with __cvta_generic_to_shared before computing byte offsets for cp.async — the intrinsic requires a shared pointer, and manual offset arithmetic must use byte-scale addressing (offset * sizeof(float)) not element indexing
