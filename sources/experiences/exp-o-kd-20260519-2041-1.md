## Challenge

The Marlin GEMM kernel computes the global-memory stride of matrix A as prob_k/8 and advances the A pointer using prob_k, assuming rows are tightly packed. For non-contiguous A where stride(0) > prob_k, this causes the kernel to skip incorrect offsets and read wrong data in subsequent tile fetches and parallel batch iterations.


## Solution

Add an lda parameter (A.stride(0)) to the Marlin kernel and replace all prob_k-based A pointer advances and stride computations with lda, so the global-to-shared copy pipeline correctly navigates the actual memory layout when rows have padding between them.

## Key Lessons

- Use an explicit lda (leading dimension of A) parameter instead of assuming prob_k equals the row stride — this decouples the logical reduction dimension from the physical memory layout and prevents correctness errors on non-contiguous inputs
- When a kernel's pointer arithmetic assumes contiguous data (stride == logical_dim), audit every place the logical dimension is used as a stride and replace it with the actual stride parameter for generality
- For Marlin-style quantized GEMM, the a_gl_stride (distance between consecutive A rows in int4 units) must reflect the real memory layout; using lda/8 instead of prob_k/8 fixes misaligned tile reads when lda > prob_k
