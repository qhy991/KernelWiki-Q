## Challenge

Passing a global-memory vector pack reference directly to a read-only callable causes NVCC to potentially emit multiple narrow scalar LDG instructions (up to 16 separate LDG.2 for vec_n_t<half,16>) instead of a single wide vector load, increasing memory transaction count on a memory-bound read kernel


## Solution

Materialize the vector pack into a local vin_t variable before passing it to the callable, which forces NVCC to load the entire 32-byte pack from global memory in one wide LD.global instruction (LDG.128 or LDG.32) instead of potentially issuing multiple narrow scalar loads when the callable iterates over the internal scalar array

## Key Lessons

- Materializing a wide vector pack (e.g., vec_n_t<half,16> = 32 bytes) into a local variable before passing it to a read-only callable encourages NVCC to emit a single LDG.128 load instead of multiple narrow LDG.2 scalar loads — applies whenever a large aligned struct is consumed element-wise inside a callee
- A local copy of a read-only global-memory vector acts as a compiler hint for wide load coalescing without changing algorithmic semantics — effective for memory-bound kernels where load transaction count dominates latency
