## Challenge

MoE softmax with BF16 input uses two separate kernel launches: a conversion kernel writes an intermediate float buffer (8192*8*4=256KB) to global memory, then a softmax kernel reads it back, causing redundant global memory traffic and extra kernel launch overhead for a short loop body (only 8 elements per row)


## Solution

Fuse the BF16-to-float conversion directly into the softmax kernel by calling __bfloat162float() inline in each of the three softmax loops (max-reduction, exp-sum, output-write), eliminating the intermediate 256KB float buffer and the separate conversion kernel launch entirely

## Key Lessons

- Fusing a type-conversion kernel into the compute kernel that consumes its output eliminates an intermediate global memory buffer and one kernel launch — especially beneficial when the per-row computation is short (e.g., softmax over 8 experts) and kernel launch overhead dominates
- Inline __bfloat162float() at the point of use is essentially free (single hardware instruction on Ampere+) and avoids materializing an entire float array to global memory, making it strictly better than a separate conversion kernel for bandwidth-bound reductions
- When the softmax dimension is small (N<=warp size), kernel launch overhead and global memory round-trips become the primary bottleneck rather than compute — fusion strategies that reduce launches and global memory traffic yield the most gains
