## Challenge

In a back-to-back GEMM kernel, scale/bias is applied inline inside GEMM1's inner K1 loop, causing each of the M*N output elements to redundantly recompute scale*multiply+bias for every K1 iteration (M*N*K1 = 262,144 operations) instead of applying it once per accumulator element (M*K1 = 4,096 operations)


## Solution

Move scale/bias from GEMM1's inner loop into a centralized epilogue after GEMM0: write GEMM0 results with scale/bias pre-applied directly to a named shared memory buffer, add a __syncthreads() barrier, then let GEMM1 read the already-transformed values without any per-iteration arithmetic

## Key Lessons

- Hoist invariant element-wise transformations (scale/bias) out of inner loops into a centralized epilogue — reduces redundant arithmetic from O(M*N*K1) to O(M*K1) and simplifies the downstream GEMM mainloop to pure multiply-accumulate
- In back-to-back GEMM patterns, trading a __syncthreads() barrier for shared-memory materialization of the intermediate result with pre-applied element-wise ops is a net win when the eliminated inner-loop arithmetic exceeds the barrier cost
- Named __shared__ arrays (fixed-size) are preferable to extern __shared__ (dynamically sized) when the accumulator tile dimensions are compile-time constants — enables the compiler to optimize shared-memory bank-access patterns
