## Solution

Add explicit GMMA major-mode template parameters (cute::GMMA::Major::MN for B-side weight scales, cute::GMMA::Major::K for A-side activation scales) to Sm90BlockwiseScaleConfig, telling CUTLASS the actual memory layout so it can interpret column-major scales directly without host-side transposition

## Key Lessons

- Specify explicit GMMA major-mode layout parameters when configuring CUTLASS Sm90 scale configs — defaulting to K-major for all scale tensors forces callers to transpose data that CUTLASS could handle natively with the correct layout hint
- Host-side data transformations (transpose + contiguous copy) can dominate cost relative to lightweight downstream computation; eliminating them yields measurable speedup even when kernel arithmetic is unchanged
- Providing layout information as template parameters removes the need for device-specific branching in caller code, reducing API surface and maintenance burden
