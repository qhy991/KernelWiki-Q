Support both weight layout conventions (NN: weight[K,N] and NT: weight[N,K]) via CUTLASS stride computation, avoiding runtime transpose overhead

## Challenge

Different frameworks store weight matrices in different layouts. PyTorch convention is weight[N,K] (ColumnMajor), while some pre-processed weights are already transposed to [K,N] (RowMajor). Runtime transpose adds O(N*K) memory bandwidth overhead.

## Solution

Separate CUTLASS GEMM kernel variants for NN and NT layouts using stride-based addressing

For NN layout (weight[K,N]): use RowMajor for InputB. For NT layout (weight[N,K]): use ColumnMajor for InputB. CUTLASS handles stride computation internally so no actual transpose is needed at runtime. Python layer detects layout via stride and dispatches to correct kernel variant.

## Key Lessons

- Never transpose weight matrices at runtime; instead dispatch to CUTLASS NN or NT kernel variant based on stride detection
- PyTorch weight layout is [N,K] (ColumnMajor/NT). Pre-processed weights may be [K,N] (RowMajor/NN). Both are valid
- CUTLASS stride computation handles both RowMajor and ColumnMajor without data movement; only the stride parameters change
- For INT8 Tensor Core operations, column-major layout is required for optimal alignment; convert in Python if needed
- Detect layout in Python via tensor.stride(): stride(0)==1 means RowMajor, stride(1)==1 means ColumnMajor
- Maintaining both NN and NT code paths doubles binary size but eliminates O(N*K) transpose overhead at runtime
