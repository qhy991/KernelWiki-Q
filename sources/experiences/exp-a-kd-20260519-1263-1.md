## Solution

Doubled TILE_K from 64 to 128 to match CUTLASS sm75_config_M64 (GemmShape<64,128,128>), halving K-loop iterations from 64 to 32 and increasing shared memory tile to 4KB total; however, the simplified per-thread dot-product kernel lacks HMMA warp-level arithmetic intensity scaling so per-iteration loading cost doubles proportionally, yielding net regression (0.766x)

## Key Lessons

- Doubling TILE_K in a simplified per-thread dot-product GEMM kernel increases shared memory loading cost proportionally to the tile size increase without changing arithmetic throughput (1 MAC per K step per thread), so halving K iterations is exactly offset by doubled per-iteration work — larger tiles only help when paired with warp-level tensor instructions (HMMA/WMMA) that provide arithmetic intensity scaling
- TILE_K tuning must account for shared memory footprint: doubling TILE_K from 64 to 128 quadruples total shared memory from 2KB to 4KB (s_A and s_B both double), which can reduce occupancy on GPUs with limited shared memory per SM
