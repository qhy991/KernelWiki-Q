## Solution

Split K=2048 into 4 independent slices of K=512, launch 4 gemm_partial kernels concurrently on separate CUDA streams so they occupy distinct SMs in parallel, then merge partial sums via a lightweight reduce_k kernel — the small 16-block grid allows all 4 kernels to run simultaneously without SM contention

## Key Lessons

- Split-K decomposition with concurrent CUDA streams saturates idle SMs when the GEMM grid is small (few output tiles) — each stream's partial kernel occupies separate SMs simultaneously, yielding near-linear speedup proportional to the number of K splits
- Use cudaEventRecord per stream + cudaStreamWaitEvent on the default stream to synchronize partial kernels before a final reduction, avoiding heavyweight cudaDeviceSynchronize and enabling fine-grained dependency control
- The reduction kernel cost (O(M*N*K_SPLITS)) is negligible compared to the GEMM work (O(M*N*K)), so split-K is effective whenever K is large relative to M*N and the grid under-utilizes available SMs
