## Challenge

Warp-cooperative memory transfer kernel uses scalar 64-bit PTX load/store per lane (ld.global.nc.b64 + st.global.cg.b64), where 16 lanes each issue one 8-byte request to copy a 128-byte item


## Solution

Replace per-lane scalar 64-bit PTX load/store with paired 128-bit v2.b64 instructions (ld.global.nc.v2.b64 + st.global.cg.v2.b64), halving the number of active lanes from 16 to 8 while each lane transfers 16 bytes instead of 8, plus a tail path for non-16-byte-aligned sizes

## Key Lessons

- On Ampere GPUs (A800/A100), the hardware memory coalescing unit already merges narrow per-lane accesses (e.g., 8-byte b64) into full 128-byte cache-line transactions, so replacing scalar PTX with v2.b64 vector instructions does not reduce actual memory bus traffic
- Halving the number of active lanes in a warp (16→8) for small transfers where each lane has only one iteration reduces warp-level memory request parallelism and pipelining, hurting throughput even when per-instruction width increases
- Vectorized PTX instructions (v2.b64/v4.b32) benefit performance only when the workload per lane is large enough to amortize the reduced lane count — for single-iteration, small-item-size transfers, keeping more lanes active with narrower accesses is preferable
