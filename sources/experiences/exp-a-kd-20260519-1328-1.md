## Solution

Replace conditional all-or-nothing vectorization with vectorize_with_alignment which unconditionally vectorizes the aligned middle portion (1008 of 1021 elements) using vec_n_t<16> loads and processes only the misaligned prefix and tail (13 elements) scalar, enabling partial vectorization even when total count is not a multiple of the vector width

## Key Lessons

- Partial vectorization (aligned middle + scalar prefix/tail) is the correct pattern for non-aligned element counts — it enables vectorized memory access for the majority of data rather than falling back entirely to scalar, but gains are only visible at sufficient workload scale
- At small per-thread workloads (~4 scalar iterations per thread with 1021 elements and 256 threads), kernel launch overhead and functor indirection can offset the vectorization benefit — this pattern yields measurable speedups only when the vectorized region is large enough to amortize dispatch overhead
- The lambda-based scalar_op abstraction in vectorize_with_alignment adds flexibility but introduces indirect call overhead; for tight quantization loops on small workloads, the overhead may neutralize the vectorization gain
