## Solution

Adapt TILE_M at compile time to match the actual valid row count (TILE_M=8 for small MoE batches with 8 rows), halving per-block shared memory from 25152 to 12576 bytes and reducing cooperative load iterations from 2048 to 1024, yielding 7.9% faster execution

## Key Lessons

- Adapt tile dimensions (TILE_M) to match actual workload size at compile or launch time rather than using a fixed maximum — for small MoE batches with fewer rows than the default tile, this halves shared memory allocation and reduces wasted load/compute iterations
- Shared memory over-allocation has a compounding cost: it increases cooperative load loop iterations, expands the reduction buffer footprint, and reduces effective L1 cache available for other data — even when occupancy reports the same blocks/SM, tighter shmem sizing improves per-block throughput
- For tiled GEMM kernels serving variable-batch workloads (e.g., MoE expert routing), provide multiple kernel variants with different tile sizes (e.g., TILE_M=8 vs TILE_M=16) and select at dispatch time based on actual M to avoid wasting shared memory on under-utilized tiles
