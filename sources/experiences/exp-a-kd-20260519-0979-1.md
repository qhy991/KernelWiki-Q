## Challenge

The moeSoftmax kernel reads __half input from global memory three times across three sequential passes (find max, compute sum, compute softmax), redundantly invoking __half2float conversion on the same data each pass


## Solution

Cache converted float values by writing them to the output buffer during pass 1, then read from output (float, cache-hot) in passes 2 and 3 instead of re-reading __half input and re-converting each time

## Key Lessons

- __half2float is a single hardware instruction on modern GPUs (A800), so eliminating redundant half-to-float conversions yields negligible compute savings — the conversion cost assumption must be validated per-architecture before optimizing around it
- When the working set fits entirely within the L2 cache (~1.5MB within A800's 40MB L2), all passes already read cache-hot data regardless of whether the source is input or output, making redundant-read elimination ineffective at this scale
- Writing intermediate float values to output in pass 1 increases total memory traffic (one extra 4-byte write per element) compared to re-reading 2-byte half values from cache, potentially negating the intended bandwidth savings
