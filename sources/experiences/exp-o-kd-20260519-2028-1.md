## Challenge

The kernel uses atomicOr to pack 4 fp8 scale exponents into each uint32 output element, which requires pre-zeroing the entire output_s_packed buffer via a separate cudaMemsetAsync call before every kernel launch — adding a full extra kernel launch overhead and an additional DRAM write pass over the scale buffer


## Solution

Replace atomicOr with direct byte-level writes via reinterpret_cast<uint8_t*>, where each thread writes its scale exponent to a unique byte position, then expand the kernel grid to cover all padding groups (num_groups_padded) and write zero bytes for padding positions directly in the same kernel — eliminating both the atomic operation overhead and the separate cudaMemsetAsync call entirely

## Key Lessons

- Replace atomicOr packing with direct byte-level writes via reinterpret_cast<uint8_t*> when each thread targets a unique byte position within a packed uint32 — this eliminates both the atomic operation overhead and the requirement for pre-zeroed output buffers
- Fuse zero-initialization into the kernel by expanding the grid to cover all padding groups and writing zeros for invalid positions directly — this removes a separate cudaMemsetAsync call and its associated kernel launch overhead, DRAM write pass, and pipeline stall
- When packing sub-word data (e.g., 4 uint8 exponents into uint32), prefer direct byte stores over shift-and-atomicOr — byte-level addressing is always safe since uint32 is naturally aligned and each byte lane is independently writable
