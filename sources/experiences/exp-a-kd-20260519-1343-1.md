## Challenge

RMS norm kernel assumes contiguous input (stride == hidden_size), forcing callers with strided tensors (e.g. PyTorch .contiguous()) to launch a separate copy kernel that reads strided rows into a contiguous buffer, adding a full kernel launch and 128KB of redundant global memory read+write traffic


## Solution

Add an input_stride parameter to rms_norm_kernel and replace blockIdx.x * hidden_size with blockIdx.x * input_stride in both the variance accumulation and output write loops, allowing the kernel to read directly from strided memory and eliminating the separate copy_strided_to_contiguous_kernel launch entirely

## Key Lessons

- Add a stride parameter to elementwise/reduction kernels instead of forcing callers to copy strided tensors into contiguous buffers — the indexing arithmetic (blockIdx.x * input_stride + idx) is free on GPUs and eliminates an entire kernel launch plus its redundant global memory traffic
- A separate strided-to-contiguous copy kernel doubles memory traffic (read strided + write contiguous + read contiguous in the compute kernel) compared to a single strided read in the compute kernel itself, making stride-aware indexing strictly dominant for bandwidth-bound operations like RMS norm
