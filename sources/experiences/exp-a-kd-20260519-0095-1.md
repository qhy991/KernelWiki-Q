## Challenge

In a back-to-back GEMM pattern, scale/bias is applied inline inside the GEMM1 K-loop for every (row, col, kk) triple, causing M*N*K1 redundant global-memory accesses and repeated scale/bias FLOPs — the accumulator is read from global memory and transformed on every inner-loop iteration instead of once


## Solution

Apply scale/bias once in a centralized epilogue pass over the M*K1 accumulator elements, write the transformed values to shared memory, then let GEMM1 read directly from shared memory — eliminating redundant FLOPs and replacing global-memory reads with ~10x lower latency shared-memory reads in the inner loop

## Key Lessons

- Factor loop-invariant transformations out of inner loops into a centralized pre-pass — applying scale/bias M*K1 times instead of M*N*K1 times reduces redundant FLOPs and global-memory traffic by a factor of N
- Use shared memory as a staging buffer for transformed accumulator data between fused GEMM stages — shared memory has ~10x lower latency than global memory and the __syncthreads() barrier cost is amortized by reduced global-memory traffic
- In back-to-back GEMM (B2b) patterns, an EpilogueSmemAccumulator that materializes the transformed intermediate result once is far more efficient than per-warp inline scale/bias that re-transforms on every consumption of the same accumulator element
