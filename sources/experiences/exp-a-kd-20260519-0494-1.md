## Challenge

Flash Attention kernel uses kStage=1 single-buffered synchronous Q/K gmem-to-smem loads: each tile_K_d iteration issues CP_ASYNC_CG then immediately calls CP_ASYNC_WAIT_GROUP(0) + __syncthreads(), creating a serial load→sync→compute dependency chain that stalls MMA while waiting for data transfer


## Solution

Upgrade from kStage=1 to kStage=2 double-buffered software pipelining: add a prologue that pre-loads (kStage-1)=1 tiles of Q and K before the main loop, then inside each iteration issue async copies for the next tile into smem_sel_next while MMA runs on the current tile in smem_sel, overlapping gmem-to-smem transfer latency with computation

## Key Lessons

- Double-buffered software pipelining (kStage=2) with CP_ASYNC prefetch into an alternate shared memory buffer overlaps gmem-to-smem latency with MMA computation — applies to any tiled GEMM/attention kernel where async copy and HMMA can operate concurrently on different smem regions
- A prologue that pre-loads (kStage-1) tiles before the main loop seeds the pipeline so the first iteration already has data ready, avoiding a cold-start stall
- CP_ASYNC_WAIT_GROUP(kStage-2) with kStage=2 waits for group 0 (all committed groups complete), same as WAIT_GROUP(0), but the key change is deferring the wait to after MMA rather than issuing it before — the wait placement, not the wait argument, enables the overlap
