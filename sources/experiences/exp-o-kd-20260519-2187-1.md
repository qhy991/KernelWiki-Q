## Challenge

The original fused_add_rms_norm_kernel loads and stores FP16/BF16 elements one scalar at a time (2 bytes per transaction), severely underutilizing GPU memory bandwidth since each 2-byte access cannot fill a 128-byte memory transaction, and all arithmetic operates on single 16-bit values without exploiting packed half2/bfloat162 instructions


## Solution

Introduce _f16Vec<scalar_t,width> (alignas(16) POD struct) to reinterpret scalar pointers as 128-bit vector pointers, loading 8 FP16/BF16 values per transaction via packed half2/bfloat162 arithmetic operators (+=, *=, sum_squares), and use SFINAE to select this specialization only when _typeConvert<scalar_t>::exists, reducing global memory transactions by up to 8x

## Key Lessons

- Reinterpret scalar FP16/BF16 pointers as 128-bit aligned vector structs (_f16Vec) to load 8 elements per transaction instead of 1, reducing global memory transactions by up to 8x — applies to any memory-bandwidth-bound kernel operating on contiguous half-precision tensors
- Use packed half2/bfloat162 arithmetic operators (overloaded +=, *=, sum_squares on a vectorized POD struct) to double FLOP throughput per instruction compared to scalar element-wise operations — applicable when contiguous elements undergo the same arithmetic operation
- Use SFINAE (std::enable_if_t on a type-trait condition) to select vectorized kernel specialization only for supported types, allowing the generic scalar kernel to coexist as a fallback for unsupported types or misaligned pointers
