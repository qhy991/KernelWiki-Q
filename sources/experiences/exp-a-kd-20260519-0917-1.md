## Solution

Replace custom FP8Wrapper and inline fp8e4m3fn_from_fp32_value with native __nv_fp8_e4m3 type and static_cast<DST_DTYPE>(val) conversion; however, on sm_80 GPUs without native FP8 hardware (cvt.rna.sat.f32.f8_e4m3 is sm_89+ only), NVCC emulates static_cast through a heavier device runtime path than the compact inline integer sequence, causing 13x slowdown

## Key Lessons

- Custom inline FP8 conversion via integer bit-manipulation (sign extraction, denorm handling via float addition, rounding via mant_odd) can outperform NVCC's native __nv_fp8_e4m3 type on GPUs without FP8 hardware (sm_80) because the compiler's emulation path has more instruction overhead than a well-crafted compact inline integer sequence
- static_cast<__nv_fp8_e4m3> maps to a single hardware instruction on sm_89+ (Hopper) and would be faster there, but regresses severely on sm_80 (Ampere) where it must be emulated in software — always profile FP8 conversion paths on the target architecture before assuming native types are faster
