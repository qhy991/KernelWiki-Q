## Solution

Increasing per-thread vectorization from half2 (2 elements) to half2×4 (8 elements) reduces block thread count from 128 to 32, leaving only one warp per block — the SM has no other warps to schedule while the single warp waits on memory, causing a regression despite processing more elements per thread

## Key Lessons

- Aggressive per-thread vectorization that collapses block thread count below one full warp destroys occupancy — the SM scheduler has no other warps to issue while the single warp stalls on memory or compute latency
- Multiple per-vector-element boundary checks (one per half2 pair) introduce warp divergence when only one warp exists per block — with more warps, divergence cost is amortized across independent warp scheduling
- Vectorization width must be balanced against minimum occupancy: increasing elements-per-thread should keep blockSize >= 128 (4 warps) to maintain latency-hiding capability on most GPU architectures
