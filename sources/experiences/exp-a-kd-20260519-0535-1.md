## Challenge

HGEMM TN kernel uses a simple 2D grid launch where consecutive blocks along blockIdx.x process column-blocks sequentially, causing poor L2 cache temporal reuse for shared A and B matrix tiles across adjacent thread blocks


## Solution

Enable block-level grid swizzle by switching to a 3D grid with interleaving stride of 2048 elements, which remaps block indices so that adjacent blocks in the remapped grid share L2 cache lines for A and B matrix tiles, dramatically improving L2 temporal locality

## Key Lessons

- Block swizzle remaps thread-block indices through a 3D grid so that blocks processing nearby output tiles execute close in time, maximizing L2 cache reuse of shared A and B tiles — applies to any tiled GEMM where multiple blocks read overlapping input regions
- The swizzle benefit scales inversely with problem size: smaller matrices (M=N=K=256) see larger speedups because L2 cache covers a larger fraction of working set, while larger matrices still benefit but with diminishing returns due to greater inherent reuse
- An interleaving stride parameter (e.g., 2048 elements) controls the swizzle granularity — it should roughly match the L2 cache line capacity to ensure remapped blocks share cache lines effectively
