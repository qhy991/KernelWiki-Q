## Challenge

Scalar f32 loads in the SGEMV kernel issue one element per thread per iteration, requiring 16 loop iterations for K=512, generating excessive instruction overhead and many small uncoalesced memory transactions that underutilize available memory bandwidth


## Solution

Replace scalar f32 loads with vectorized float4 loads using the FLOAT4 macro, so each thread fetches 4 elements per iteration via 128-byte aligned memory transactions, reducing loop iterations from 16 to 4 and enabling more efficient coalesced access for the GPU memory controller

## Key Lessons

- Use float4 vectorized loads via reinterpret_cast<float4*> to process 4 elements per thread per iteration in memory-bound GEMV kernels — reduces loop iterations by 4x and issues 128-byte coalesced transactions that are more efficient for the GPU memory controller
- Vectorized load benefits are modest when warp-level reduction and kernel launch overhead dominate total execution time, as in smaller matrix sizes
