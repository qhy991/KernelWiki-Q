## Challenge

Warp-cooperative memory copy transfers 136 bytes per item as 17 independent 8-byte chunks using ld.global.nc.b64/st.global.cg.b64, with 17 lanes each issuing one load-store pair — the baseline already achieves high memory request parallelism through coalesced 64-bit accesses across many active lanes


## Solution

Replace 17 scalar 64-bit transfers with 8 vectorized 128-bit transfers (ld.global.nc.v2.b64/st.global.cg.v2.b64) plus a divergent tail path for the remaining 8 bytes, attempting to reduce instruction count — however this reduces active lanes from 17 to 8+1=9 and adds warp divergence, causing a net regression

## Key Lessons

- Vectorized 128-bit loads/stores (v2.b64) can regress performance when they reduce the number of active warp lanes (17→9), because fewer parallel memory requests decrease overall throughput more than wider per-lane transfers gain — applies whenever vectorization consolidates work into fewer lanes without saturating memory bandwidth
- On Ampere GPUs the memory subsystem already coalesces multiple 64-bit accesses from different lanes into wider 128-bit/256-bit transactions, so explicit v2.b64 vectorization of already-coalesced 64-bit accesses provides diminishing returns — prefer maximizing active lane count over widening per-lane transfers
- Adding a divergent tail path (if lane_id < tail_8B) for non-aligned sizes introduces warp divergence overhead that can negate any benefit from vectorized bulk transfers — when item sizes are not multiples of the vectorized unit, the tail cost must be weighed against the bulk gain
