## Challenge

Grid sizing function get_num_blocks() uses a waveset-rounding heuristic (ceil-div work_blocks by gpu_occupancy then multiply back) that over-allocates block count when sm_occupancy is low, computing unnecessary intermediate values and potentially overshooting the minimum viable grid


## Solution

Replace waveset-rounding heuristic with a two-branch approach: if work_blocks is small (<= 2*avail_sms), return as-is; otherwise use fast_max(work_blocks, avail_sms*4) to ensure at least 4 waves of SM utilization without over-allocation

## Key Lessons

- Waveset-rounding (ceil-div then multiply) and fast_max with a floor of 4*avail_sms can produce identical grid sizes for medium workloads — prefer the simpler fast_max form when the rounding pattern is not workload-critical, as it eliminates a divide and multiply
- Raising the passthrough threshold from <avail_sms to <=2*avail_sms avoids unnecessary grid inflation for workloads that already nearly saturate the GPU within two waves
