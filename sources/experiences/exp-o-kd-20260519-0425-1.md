## Challenge

In a double-buffered tiled SGEMM kernel, B-tile global-to-shared memory copies are issued synchronously alongside A-tile loads in the same loop, serializing memory transfers and preventing overlap with the compute-heavy FMA loop


## Solution

Replace synchronous B-tile shared memory stores with cp.async.ca.shared.global instructions (CP_ASYNC_CA macro) issued before the FMA loop and committed with cp.async.commit_group, so that B-tile async transfers execute in parallel with A-tile loads and the BK-iteration compute loop, then synchronize with cp.async.wait_group 0 before __syncthreads

## Key Lessons

- Use cp.async.ca.shared.global to issue asynchronous global-to-shared memory copies for one tile matrix (B) while the other tile (A) loads and the FMA compute loop execute, hiding global memory latency behind arithmetic — applies to any double-buffered tiled GEMM where load and compute can be overlapped
- Issue cp.async.commit_group immediately after the async copy loop to dispatch transfers early, then place cp.async.wait_group(0) just before __syncthreads so the async hardware and thread barrier synchronize in the correct order
- Convert shared memory pointers to generic via __cvta_generic_to_shared before passing to cp.async inline PTX — the async copy instruction requires a shared memory address in generic format, not a direct __shared__ variable reference
