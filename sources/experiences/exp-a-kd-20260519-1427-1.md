## Solution

Introduce adaptive max_block_size that reduces to 256 when num_tokens >= 256, allowing more blocks to run concurrently per SM and improving occupancy for large batch workloads; for small batches (num_tokens < 256) the max stays at 1024, producing identical launch configuration and neutral performance

## Key Lessons

- Adaptive block sizing based on grid size (num_tokens) can improve occupancy for large-batch scenarios by using smaller blocks that allow more concurrent blocks per SM, but produces no benefit when the grid is already small enough that occupancy is not limited by block count
- The kernel code itself was unchanged — this optimization is purely a host-side launch configuration tuning that trades per-block parallelism for block-level concurrency, and its effectiveness depends entirely on whether the original block size was limiting occupancy
