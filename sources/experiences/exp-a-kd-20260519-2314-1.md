## Solution

Replace scalar half load/store with 128-bit uint4 packed load/store and restructure thread mapping from one-block-per-head to a flat grid where threads compute a global index then derive multi-level (token, head, pack) indices via division and modulo, reducing memory transactions from 256 individual half accesses to 32 128-bit transactions per head

## Key Lessons

- Vectorized (uint4) load/store does not always improve performance over scalar coalesced access — when the working set fits entirely in L2 cache, scalar loads already achieve near-optimal bandwidth and the packing overhead can negate any transaction-count reduction
- Flat-grid thread mapping with multi-level index computation (global_idx → token_head_idx, pack_idx → token_idx, head_idx via division/modulo) adds significant arithmetic overhead compared to the simple one-block-per-head model, especially when the per-thread work is small (only 2 scalar iterations or 1 packed iteration)
- Thread mapping redesign must consider intra-block spatial locality: mixing threads from different heads in the same block reduces cache locality compared to dedicating one block per head
