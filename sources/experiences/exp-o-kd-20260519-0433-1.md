## Solution

Issue cp.async.ca instructions for the next B tile at the start of the loop body and commit the async group, then proceed with A loading, BK compute, and A transpose while B copies are in flight; CP_ASYNC_WAIT_GROUP(0) after A transpose ensures B copies complete before the next iteration

## Key Lessons

- Replace synchronous global-to-shared FLOAT4 loads with cp.async.ca to overlap next-iteration tile prefetch with current-iteration compute — applies whenever the main loop has independent load and compute phases that can be pipelined
- Decouple A and B tile loading into separate loops so that B async copies can be issued first and committed before A loading begins, maximizing overlap between B transfer and A load+compute
- With larger tiles (TM=8, TN=8, 64 FMAs per thread per BK step), there is more compute latency to hide memory transfers, making async prefetch especially effective for 8x8 tiling configurations
