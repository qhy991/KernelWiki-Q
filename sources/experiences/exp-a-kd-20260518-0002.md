## Challenge

A quantized GEMM dot product (dot22_8_h) using half2 accumulation via 4 __hfma2 instructions risks overflow when unscaled 8-bit weights occupy the full -128..127 range. half2 (FP16) has only 5 exponent bits and cannot safely represent intermediate sums that exceed ~65504 before scale application.


## Solution

Replace half2 accumulation with FP32 scalar accumulation: decompose each half2 weight pair into two FP32 values via __low2float/__high2float, convert activations with __half2float, and accumulate with 8 scalar fmaf instructions. FP32 has 8 exponent bits, safely covering the -128..127 weight range. The arithmetic instruction count increases ~3x but in the memory-bound quantized GEMM context, arithmetic latency is hidden behind global memory access.

## Results

- **speedup**: 0.990x (~1% slowdown, negligible in memory-bound context)


## Key Lessons

- When accumulating dot products of dequantized low-precision weights (int8 range -128..127), FP16 (half2) accumulation can overflow. Switch to FP32 accumulation despite the ~3x arithmetic instruction increase. The cost is negligible for memory-bound kernels where compute latency is hidden behind memory access. Applies to any quantized GEMM where unscaled weights exceed FP16 safe range.
- Each thread in this quantized GEMM dot product loads 4 half2 weights (32 bytes) and 8 half activations (16 bytes) — arithmetic-to-memory ratio is low. In memory-bound regimes, trading more instructions for numerical correctness has near-zero performance cost. This pattern applies generally: correctness-preserving precision upgrades in memory-bound kernels are almost free.
