## Challenge

The CUTLASS Epilogue class processes accumulator fragments one at a time in a loop, incurring two __syncthreads() barriers per fragment (one after warp store to SMEM, one after shared load). For tiles with many iterations this barrier overhead dominates execution time as every barrier forces all block threads to stall.


## Solution

Add a FragmentsPerPartition template parameter (defaulting to 1 for backward compatibility) to both Epilogue and EpilogueBase, enabling the epilogue to coarsen its iteration so that multiple accumulator fragments are stored to SMEM before the first barrier and loaded after the second barrier, reducing total __syncthreads() calls by a factor of FragmentsPerPartition at the cost of proportionally larger SMEM allocation.

## Key Lessons

- Coarsening a loop that contains __syncthreads() barriers by batching multiple fragments per iteration reduces barrier count proportionally — applies whenever SMEM is large enough to hold the extra fragments and the kernel is barrier-bound rather than compute-bound
- Adding a defaulted template parameter (e.g., FragmentsPerPartition = 1) preserves backward compatibility while enabling opt-in performance tuning in CUTLASS epilogue and similar tile-iterator patterns
