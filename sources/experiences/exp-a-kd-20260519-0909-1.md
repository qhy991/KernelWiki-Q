## Solution

Replace scalar int32 loads with int4 vectorized loads (4×int32 per transaction) when the pointer is 16-byte aligned, reducing global memory transactions from 8192 to 2048 per expert block, with a scalar fallback loop for misaligned pointers and remainder elements

## Key Lessons

- Vectorizing int32 loads into int4 (4×int32) reduces global memory transaction count by 4×, but the actual speedup depends on whether the counting loop dominates kernel execution time — when BlockReduce overhead, kernel launch latency, and per-block setup dominate (short loops ~16 iterations), the memory-access improvement is masked
- int4 vectorized loads require 16-byte pointer alignment; always include a runtime alignment check (reinterpret_cast<uintptr_t>(ptr) % 16 == 0) and a scalar fallback path to guarantee correctness for arbitrary input buffers
- For small workloads (e.g., topk_length=8192 with 512-thread blocks yielding ~16 loop iterations), fixed-cost overhead (CUB BlockReduce shared-memory synchronization, kernel launch) dominates — vectorization benefits become measurable only when the loop body constitutes a significant fraction of total kernel time
