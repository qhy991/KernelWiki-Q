## Solution

Expanded each thread to process 4 half2 registers (8 half elements) instead of 1 half2, reducing thread count per block from 128 to 32 to increase per-thread memory throughput, but the reduced occupancy outweighed the benefit

## Key Lessons

- Increasing per-thread vector width (f16x2 → f16x8) without compensating block size or grid dimensions reduces occupancy and can regress performance — the GPU needs enough active warps to hide memory latency, not just wider per-thread loads
- Multiple independent boundary checks (one per half2 register) within a single warp introduce warp divergence when some checks pass and others fail near array boundaries, adding branch overhead that scales with vector width
- For memory-bound elementwise kernels, occupancy is often the primary performance driver; prefer keeping higher thread counts with moderate vector width (e.g., f16x2 or f16x4) over aggressive vectorization that starves the SM of warps
