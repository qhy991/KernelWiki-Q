## Challenge

The before code stores via base[s] = val inside an if(guard), which the compiler can optimize into a predicated store or even eliminate entirely when guard is false (skipping address computation). The after code forces every thread to compute &base[s] and materialize it as a 64-bit pointer for the P


## Solution

into four uint32 registers for the 'r' constraints of st.global.v4.u32. This mandatory register materialization and the asm volatile boundary preventing surrounding optimizations likely causes the 44% regression. The NVCC compiler at -O3 already handles simple if-guarded float4 stores efficiently —

## Key Lessons

- In the real CUTLASS context, the global_store abstraction is primarily for code consistency and correctness guarantees across access sizes, not for performance improvement over compiler-generated predicated stores.
