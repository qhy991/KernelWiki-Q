## Challenge

In sm89_dispatch_shape, the fallback path for large M (>256) uses TileShape <32,64,128> with WarpShape <16,64,64>, launching many tiny threadblocks that each compute only 32x64 output elements with low arithmetic intensity, while K=128 consumes excessive shared memory per block limiting SM occupancy


## Solution

Switch the large-M fallback branch to TileShape <128,128,64> with WarpShape <64,64,64>: the larger 128x128 output tile per threadblock increases arithmetic intensity and reduces total threadblock count, while reducing K from 128 to 64 lowers shared memory pressure per block allowing higher SM occupancy, and the 64x64 warp tile improves per-warp MMA efficiency on Ada tensor cores

## Key Lessons

- Use larger TileShape (128x128) for large-M GEMM dispatch branches where the M dimension exceeds 256 — bigger tiles amortize launch overhead, increase per-block arithmetic intensity, and reduce total threadblock count for better GPU utilization
- Reduce the K dimension in TileShape (128→64) when shared memory occupancy is the bottleneck — smaller K halves the shared memory footprint per threadblock, allowing more concurrent blocks per SM and higher overall throughput
- Match WarpShape to the larger tile proportionally (<64,64,64> for <128,128,64> tile) so each warp processes enough output elements to fully utilize tensor core MMA pipelines on Ada SM89 hardware
