## Challenge

MLA K-cache concatenation kernel uses scalar bf16 reads/writes (2-byte transactions), issuing 6 individual loads and 6 individual stores per head iteration (4 NOPE + 2 ROPE per lane), generating 96 narrow memory transactions per warp-chunk and severely underutilizing available memory bandwidth


## Solution

Replace scalar bf16 read/write loops with vectorized int2 (16-byte, covering 8 bf16 values per lane for NOPE) and int (8-byte, covering 4 bf16 values per lane for ROPE) load/store instructions via reinterpret_cast, and introduce local register buffers (k_nope_buf[16], k_rope_buf) to decouple reads from writes and ensure fully coalesced warp-wide memory transactions

## Key Lessons

- Replace scalar bf16 element loops with vectorized int2/int load/store via reinterpret_cast when consecutive lanes access contiguous memory — a full warp of int2 loads produces a single 512-byte coalesced transaction vs 32 separate 16-byte scalar transactions, reducing memory transaction count by up to 8x
- Use static_assert to verify vectorized buffer types match the expected data-per-lane width (e.g., sizeof(int2) == HEAD_DIM * sizeof(bf16) / 32) — catches dimension/lane mismatches at compile time
- Decouple reads from writes using local register buffers (k_nope_buf[], k_rope_buf) when the same data is read once and written to multiple output locations — this turns interleaved read-modify-write into a batched read-then-write pattern that gives the memory controller more coalescing opportunities
