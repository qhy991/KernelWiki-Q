## Solution

Upgrade to kStage=2 double-buffered async prefetch: assign K to smem slot 0 and V to smem slot 1 (separate smem regions), issue V async copy before Q@K^T compute so the transfer overlaps with MMA operations, use CP_ASYNC_WAIT_GROUP(1) to wait only for V while next K remains in flight, and prefetch next K into slot 0 after P@V consumes V from slot 1

## Key Lessons

- Use kStage=2 double-buffered async prefetch (CP_ASYNC.cg + separate smem slots) to overlap V gmem-to-smem transfer with Q@K^T MMA compute, hiding gmem latency behind arithmetic — applies whenever the kernel has sequential load-compute phases with independent data tiles
- CP_ASYNC_WAIT_GROUP(N) enables fine-grained pipeline control: N=0 waits for all async groups, N=1 allows one group to remain in flight — use N=1 between load phases to keep the next prefetch running while consuming the current tile
- Double-buffering requires double smem (max(Q, 2*max(K,V)) vs max(Q,K,V)) — trade extra shared memory for reduced latency; verify smem fits within cudaFuncAttributeMaxDynamicSharedMemorySize
