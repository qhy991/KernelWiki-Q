## Challenge

The kernel allocates two oversized 128x128 shared-memory buffers (total 64 KB per CTA) and loads them via a single fused loop with per-element boundary checks and zero-fills, even though the actual GEMM working set is only 64x128 and 128x64, wasting shared memory and adding branch overhead


## Solution

Right-size shared memory to the exact working set (kRows*kK + kK*kCols = 32 KB, a 2x reduction), split A and B loads into two independent loops with exact bounds that eliminate boundary checks and zero-fills, and use the logical dimension kK/kCols for compute indexing

## Key Lessons

- Right-size shared-memory allocations to the exact working set rather than rounding up to power-of-two tile dimensions — a 2x reduction in smem per CTA directly increases occupancy and reduces pressure on the shared-memory cache bank
- Split fused load loops into separate per-matrix loops with exact iteration bounds to eliminate per-element boundary checks and zero-fill branches, reducing instruction count and warp divergence in the load phase
- Use named constants (kRows, kK, kCols) for shared-memory indexing instead of hardcoded power-of-two literals (128) — this makes the code correct for non-power-of-two dimensions and avoids silent out-of-bounds reads when tile dimensions change
