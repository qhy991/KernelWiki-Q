## Challenge

Warp index is computed via __shfl_sync(0xffffffff, threadIdx.x / 32, 0), which broadcasts the value from lane 0 to all lanes — this introduces an implicit cross-lane synchronization barrier on every invocation, adding unnecessary latency when each thread already holds its own threadIdx.x in a register


## Solution

Replace the __shfl_sync warp shuffle with a direct threadIdx.x / 32 integer division — since threadIdx.x is a per-thread register value, no inter-lane communication is needed and each thread independently computes its warp index with a single ALU instruction, eliminating the implicit warp-sync barrier entirely

## Key Lessons

- Use direct register-value arithmetic (threadIdx.x / 32) instead of __shfl_sync for warp index computation — each thread already holds threadIdx.x, so the shuffle is redundant overhead that adds an implicit barrier → applies to any value derivable from per-thread registers without cross-lane data dependency
- Avoid __shfl_sync in tight loops when the broadcast value is identical across all lanes or computable from each thread's own registers — the shuffle instruction's implicit warp synchronization becomes the bottleneck at high iteration counts
