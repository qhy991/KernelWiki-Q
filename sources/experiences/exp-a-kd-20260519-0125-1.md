## Solution

Replace simple threadIdx register reads with warp_uniform_func (shfl_sync broadcast from lane 0) and a rematerializeThreadIds lambda called 3 times per iteration, aiming to break register dependency chains and force NVCC to recompute thread IDs instead of keeping them live

## Key Lessons

- Using __shfl_sync warp barriers for thread-ID rematerialization adds ~20-40 cycle overhead per call compared to simple threadIdx register reads — only beneficial when register pressure is genuinely high enough to cause spilling
- Rematerialization via heavy intrinsics (shfl_sync, shared memory round-trips) can produce severe regressions (3.2x slowdown here) when the kernel has low register pressure and the rematerialization cost dominates any theoretical register savings
- Profile register occupancy before applying rematerialization patterns — if the kernel fits within the register budget without spilling, keeping thread IDs in registers is always cheaper than recomputing them through warp shuffle
