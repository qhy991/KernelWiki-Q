## Challenge

Single GEMM kernel is launched on the default stream (stream 0) with no explicit device context, which is functionally correct for isolated execution but prevents multi-GPU device selection and eliminates the ability to overlap concurrent operations on separate streams in production serving frameworks


## Solution

Add explicit cudaSetDevice(0) for multi-GPU device context and create a dedicated CUDA stream via cudaStreamCreate, then pass the stream to kernel launches, event records, and synchronization calls, enabling correct device-scoped execution and future concurrency

## Key Lessons

- Explicit stream creation and device guard do not improve latency for isolated single-kernel launches — the GPU is fully occupied regardless of which stream the kernel runs on, so there is no opportunity for overlap
- Always use explicit cudaSetDevice + dedicated streams even for single-kernel code to ensure multi-GPU correctness (device context is not inherited across threads) and to enable future concurrency without code changes
- In production serving frameworks (e.g., vLLM, SGLang), passing stream to CUDA events (cudaEventRecord) and synchronization (cudaStreamSynchronize) is required to get correct per-stream timing when multiple operations run concurrently on different streams
