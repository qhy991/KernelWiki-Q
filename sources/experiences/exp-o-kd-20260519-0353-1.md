## Challenge

A shared-memory tiled GEMM kernel allocates two 128×128 __half buffers (32 KB total) even though the actual working set is 64×128 + 128×64 (16 KB), forcing threads to loop over 16384 elements with conditional guards to skip unused rows/columns, wasting shared memory and adding branch overhead


## Solution

Right-size shared memory to the actual 64×128 and 128×64 working sets, split the single combined load loop into two separate loops that load A and B independently without conditional guards, and use dimension-correct stride arithmetic (row*kK, kk*kCols) in the compute loop

## Key Lessons

- Right-size shared memory allocations to the exact working-set dimensions — oversized buffers halve occupancy on shared-memory-constrained SMs and the reduction from 32 KB to 16 KB directly enables more concurrent CTAs per SM
- Split a combined load loop over an oversized tile into separate per-matrix load loops scoped to each matrix's actual dimensions — this eliminates conditional guards and zero-padding inside the load path, reducing instruction count and removing divergent branches
- Use dimension-correct stride arithmetic (row*kK for row-major A, kk*kCols for column-stride B) instead of a single power-of-two stride (128) — this makes indexing self-documenting and avoids subtle bugs when tile dimensions change
