## Solution

Query GPU occupancy at runtime via cudaOccupancyMaxActiveBlocksPerMultiprocessor and cudaDevAttrMultiProcessorCount; when the token-parallel grid produces fewer thread blocks than the GPU can run concurrently, dispatch a head-parallel kernel that maps each of the 72 heads (64 Q + 8 KV) to blockIdx.y, creating 72 independent thread blocks that spread work across all available SMs

## Key Lessons

- Use runtime occupancy queries (cudaOccupancyMaxActiveBlocksPerMultiprocessor + cudaDevAttrMultiProcessorCount) to detect when the launch grid cannot saturate the GPU, and adaptively switch between token-parallel and head-parallel strategies — applies to any multi-head workload with variable batch sizes
- When batch size is small but head count is large, distribute heads across blockIdx.y instead of serializing them in a loop within one thread block — each head becomes an independent block that can run on a different SM, converting serial work into cross-SM parallelism
- Combine Q heads and KV heads into a single blockIdx.y dimension (num_qo_heads + num_kv_heads) and use blockIdx.y < num_qo_heads to dispatch between Q and KV paths — maximizes grid size while keeping launch logic simple
