## Challenge

Grouped GEMM initialization launches 4 separate kernels (3 fill_stride_kernel calls for stride arrays plus 1 init_offsets_kernel), where each kernel launch incurs ~3-5us of driver dispatch overhead; with only 8 experts the actual GPU work is ~6us, so the 3 redundant launches roughly double the pipeline latency from ~6us to ~14us


## Solution

Fuse stride array writes into the existing init_offsets_kernel by passing stride pointers and values as additional parameters and adding three scalar stores (a_strides[expert_id]=a_stride_val, etc.) per thread alongside the existing offset computation, collapsing 4 kernel launches into 1 and eliminating ~15us of driver dispatch overhead

## Key Lessons

- Fuse multiple trivial fill/initialization kernels into an existing computation kernel by passing additional parameters and adding scalar stores per thread — eliminates per-kernel launch dispatch overhead (~3-5us each) which dominates when the actual GPU work is small (few elements, light computation)
- Kernel launch overhead is a fixed cost independent of work size; when launching many kernels that each do minimal work (e.g., filling 8 values), the dispatch cost can exceed the computation cost by 2-3x, making fusion the primary optimization lever
