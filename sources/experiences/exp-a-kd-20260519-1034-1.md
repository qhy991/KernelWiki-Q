## Challenge

MoE alignment counting kernel uses warp-partitioned atomicAdd on 64 shared counters with CUB BlockScan over 1024 threads; a replacement with per-thread private count rows and serial prefix sum regresses due to 33 KB shared memory overhead and serial reduction bottleneck for 64 experts x 128 threads


## Solution

Replace warp-atomicAdd counting with per-thread private token-count rows in shared memory (128 threads x 64 experts = 8192 elements) and replace CUB BlockScan with a serial per-expert prefix sum, eliminating atomics and CUB dependency to enable future integration with sorted_token_ids scattering in one kernel — but increasing shared memory from ~256 bytes to 33.3 KB and replacing O(log N) parallel scan with O(threads x experts) serial additions

## Key Lessons

- Per-thread private counting arrays in shared memory scale as O(threads × experts) — with 128 threads and 64 experts this consumes 33 KB of shared memory just for counters, and the initialization cost (8192 zero-writes) dwarfs the actual counting work (1024 token reads) → applies when num_experts is large relative to numel
- Warp-partitioned atomicAdd on shared counters has low contention when the thread-to-counter ratio is moderate (1024/64 = 16:1) — the atomics serialize minimally and CUB BlockScan completes the prefix sum in ~10 warp-shuffle steps versus O(threads × experts) serial additions in the per-thread approach
- Architectural optimizations that merge multiple kernel phases (counting + sorted_token_ids scattering) can regress in isolated microbenchmarks measuring only one phase — the per-thread pattern eliminates a separate sort-kernel launch in the full pipeline, a benefit invisible when benchmarking counting alone
