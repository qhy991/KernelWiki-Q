## Challenge

Weight and input global memory loads are issued AFTER a 500-cycle clock64() spin-wait simulating P2P communication, so the full L1 load latency (~28-60 cycles per iteration) is exposed sequentially and accumulates across 1000 inner iterations into ~0.15 ms of wasted time


## Solution

Move the weight and input loads (float2 w, inp via regular ld.global.v2.f32 cached in L1) BEFORE the spin-wait, separated by a compiler barrier (asm volatile("" ::: "memory")), so the CUDA memory controller services the loads concurrently with the volatile flag polls via instruction-level parallelism — the 500-cycle spin period far exceeds load latency so data arrives in registers before the signal

## Key Lessons

- Issue independent global memory loads BEFORE a long spin-wait or synchronization barrier so the memory subsystem services them concurrently with the wait via instruction-level parallelism — applies whenever load data is not dependent on the wait result and the wait duration exceeds L1/L2 load latency
- Use asm volatile("" ::: "memory") compiler barriers to control load placement relative to spin-waits: place the barrier AFTER prefetch loads to prevent the compiler from sinking them past the wait, and ensure correctness of the reordering
- In iterative kernels with per-iteration communication waits (e.g., P2P allreduce pipelines), even small per-iteration latency savings (~28-60 cycles) accumulate into measurable speedups (0.15 ms per 1000 iterations) when the wait period provides slack for ILP
