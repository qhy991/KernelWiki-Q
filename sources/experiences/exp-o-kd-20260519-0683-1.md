## Solution

Replace one-block-per-row launch with occupancy-aware grid sizing via cudaOccupancyMaxActiveBlocksPerMultiprocessor and a grid-strided loop (row += gridDim.x), so each of the ~6912 resident blocks processes multiple rows sequentially, eliminating block scheduling overhead and keeping row data warm in per-SM L2 cache partitions

## Key Lessons

- Use occupancy-based grid sizing (cudaOccupancyMaxActiveBlocksPerMultiprocessor) combined with grid-strided loops when the total problem size exceeds the GPU's concurrent block capacity — eliminates block scheduling overhead and L2 cache thrashing for lightweight kernels where per-block work is small relative to scheduling cost
- Grid-strided loops (for row = blockIdx.x; row < N; row += gridDim.x) keep blocks resident on their SMs across iterations, amortizing block setup cost and maintaining L2 cache warmth — applies universally to bandwidth-bound kernels with small per-row computation (e.g., RMSNorm, LayerNorm, elementwise ops at small hidden sizes)
