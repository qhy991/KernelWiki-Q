## Solution

Replace type-based KernelTraits<T> specialization with three M-range config structs (M>256, 16<M<=256, M<=16) that independently select TileShape, ClusterShape, and PerSmTileShape_MNK tuned for each M range — large M gets 256x256x256 tiles with 2-SM clusters, medium M gets 256x128x256 with 2-SM clusters, and small M gets 128x128x256 with single-SM clusters to minimize wasted computation

## Key Lessons

- Select GEMM tile shapes based on problem geometry (M-dimension ranges) rather than data type alone — M-dimension determines how much of each tile is actually used, so type-based selection wastes compute on mismatched tiles
- Use smaller tiles (128x128) with single-SM clusters for small M (<=16) to avoid wasted SIMD lanes, and larger tiles (256x256) with multi-SM clusters for large M to maximize throughput — tile size should scale with the active dimension
- Decouple kernel schedule and epilogue schedule into separate typedefs (KernelSchedule/EpilogueSchedule) alongside tile/cluster shapes to allow independent schedule overrides without touching tile configuration
