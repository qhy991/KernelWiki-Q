## Challenge

GEMM kernel uses 256x256 tiles with a 4x4 cluster layout (16 SMs per CTA) for a problem with M=8, wasting most of the 256-wide M tile capacity and incurring unnecessary cluster coordination overhead across 16 SMs for a single tile row


## Solution

Right-size tile and cluster dimensions for small M: reduce tiles from 256x256 to 128x128 and cluster from 4x4 (16 SMs) to 1x1 (single SM), eliminating wasted thread lanes in the M dimension and removing multi-SM cluster coordination overhead entirely

## Key Lessons

- Right-size tile dimensions to match the actual problem geometry — oversized tiles (256-wide M tile for M=8) waste shared memory and thread lanes that guard against out-of-bounds accesses on every iteration
- Multi-SM cluster layouts (4x4 = 16 SMs per CTA) add coordination overhead that is pure waste when a single tile covers the entire M dimension — switch to single-SM (1x1 cluster) for small-batch GEMM
- Tile and cluster selection should be M-dimension-aware: small M (≤16) benefits from smaller tiles (128x128) with no clustering, while large M benefits from large tiles (256x256) with 2D clusters for cooperative loading
