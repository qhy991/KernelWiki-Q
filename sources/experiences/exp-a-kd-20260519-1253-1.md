## Challenge

E8M0 floating-point scale dequantization uses four exp2f calls plus four __float2bfloat16 conversions per packed int32 element, routing through the FP32 arithmetic pipeline with multiple micro-ops per call and creating significant instruction count overhead


## Solution

Replace 8 FP operations per element with 3 pure bitwise integer ops: mask odd bytes and right-shift by 1 (Out1), left-shift q by 7, then mask even-byte positions (Out2). Since E8M0 values are power-of-2 exponents, the BF16 bit representation is simply the exponent bits shifted into place, making the FP pipeline entirely unnecessary

## Key Lessons

- E8M0 (8-bit exponent-only) values represent pure powers of 2, so their BF16 bit pattern is simply (exponent_bits << 7) — this makes exp2f + float2bfloat16 entirely unnecessary and replaceable with a single shift
- Bitwise mask-and-shift operations on packed int32 can convert two E8M0 bytes simultaneously into their BF16 representations, exploiting the BF16 layout (1 sign + 8 exponent + 7 mantissa) where the exponent field aligns with the E8M0 byte after a 7-bit left shift
- Eliminating FP pipeline usage (exp2f, __float2bfloat16) in favor of integer ALU ops (SHL, SHR, LOP3) reduces both instruction count and pipeline contention, yielding measurable throughput gains even on compute-bound kernels
