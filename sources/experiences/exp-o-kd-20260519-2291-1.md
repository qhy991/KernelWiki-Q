## Challenge

Per-thread sequential expert counting uses a large global memory buffer (~256KB for 256 experts x 257 threads) where each thread zeros its own row, counts tokens into that row, then a serial prefix-sum phase aggregates across rows with two __syncthreads() barriers — all operating on high-latency global memory


## Solution

Replace per-thread global memory rows with warp-grouped atomicAdd on a tiny extern __shared__ array (8 warps × 32 experts = 256 ints = 1KB): each warp owns a slice of experts, all threads atomically increment the shared counts, eliminating the serial prefix-sum phase and its two __syncthreads() barriers entirely

## Key Lessons

- Replace per-thread global memory rows with warp-grouped shared memory atomicAdd when aggregating histogram counts — reduces memory footprint from O(threads × categories) global memory to O(warps × categories_per_warp) shared memory and eliminates the serial prefix-sum phase entirely
- Shared memory atomicAdd is efficient for low-contention histogram patterns where each warp owns a disjoint slice of counters — the per-warp expert partition ensures all atomics target different cache lines within the same warp
- When shared memory requirements scale as O(blockDim × categories), partition categories across warps instead of replicating per thread — this caps shared memory at O(warps × categories_per_warp) regardless of block size
