## Solution

Software-managed cross-CTA barriers using atomicAdd on global memory counters combined with spin-wait loops via volatile loads, replacing free kernel-boundary synchronization with ~100us of busy-waiting latency per barrier on global memory

## Key Lessons

- Spin-wait loops on global memory counters for cross-CTA synchronization add significant latency (~100us per barrier) compared to free kernel-boundary synchronization — prefer separate kernel launches for small split counts where the launch overhead is negligible
- When fusion is required, hardware named barriers or cooperative group sync are far faster than software spin-waits on global memory counters
