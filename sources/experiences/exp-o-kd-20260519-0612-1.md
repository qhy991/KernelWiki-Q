## Solution

Partition the prefix sum into segments of kElementsPerThr (16) experts per thread so that CEILDIV(256,16)=16 active threads each compute a local prefix independently, store partial sums in local_offsets_buf, then after a barrier add the running total of all preceding segments to each thread's entries — reducing serial work from O(256) on one thread to O(16) on 16 threads plus O(16) cross-segment prefix

## Key Lessons

- Partition a shared-memory prefix sum into fixed-size segments computed independently by multiple threads, then add cross-segment prefixes in a second pass — reduces serial O(N) work by one thread to O(N/P + P) work by P threads, effective for moderate N (64-512) when P threads are available
- Store each segment's total in a per-thread buffer (local_offsets_buf) so the cross-segment prefix can scan only P partial sums instead of re-reading all N elements — avoids redundant recomputation and keeps the second pass proportional to thread count rather than element count
