## Challenge

For bfloat16 inputs, opmath_t<bf16> resolves to float, which is identical to the before code's hardcoded float accumulator. Both kernels perform the same float


## Solution

Replace scalar float loads with float4 vectorized loads (4 floats per thread) to improve memory throughput, but reduce thread count from 256 to 64 to keep total elements per block constant

## Results

- **speedup**: 0.986x (neutral)


## Key Lessons

- Vectorized loads (float4) alone do not improve performance when thread count is proportionally reduced (256→64), because the resulting occupancy loss eliminates the scheduler's ability to overlap memory latency — the vectorization bandwidth gain is offset by fewer resident warps
- For memory-bound reductions on large contiguous datasets, the total data volume bottleneck dominates regardless of per-thread access width; effective vectorization must increase total memory throughput without sacrificing occupancy
- Marginal speedup (<1%) is within measurement noise and indicates that the before and after kernels saturate the same memory bandwidth ceiling equally
