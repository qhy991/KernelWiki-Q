## Solution

Replace manual vec4_t (64-bit, 4-element) vectorization with flashinfer::vec_t (128-bit, 8-element) using int4-backed loads and a generic inner loop — intended to double per-iteration throughput, but the resulting block count reduction (16→8) lowers occupancy for this small problem size and the kernel regresses

## Key Lessons

- Wider vector loads (128-bit vs 64-bit) reduce the number of thread blocks proportionally — for small problem sizes where the kernel duration is already sub-3μs, the occupancy loss from fewer blocks outweighs the per-thread throughput gain, making wider vectors a net regression
- For memory-bound kernels with very short execution times, launch overhead and pipeline startup costs dominate; prefer more blocks with narrower vectors to saturate the GPU rather than fewer blocks with wider vectors
- The optimal vector width depends on problem size: large arrays benefit from 128-bit loads, but tiny arrays (tens of thousands of elements) perform better with 64-bit loads and more concurrent threads
