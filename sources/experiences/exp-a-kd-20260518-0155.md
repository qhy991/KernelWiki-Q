## Challenge

The dramatic 27x speedup arises from three compounding factors tied to the m/n swap. First, the before kernel uses M=64 blocks with N=1024 threads per block (CUDA's maximum), producing stride-64 access across 8 partition iterations — each warp fires 256 un-coalesced cache line requests for worksp...


## Solution

Swap the block/thread mapping by exchanging grid dimensions from <<<M, N>>> to <<<N, M>>> and flipping the indexing formula from row + col * ld to col + row * ld, achieving stride-1 coalesced access, higher occupancy via 64-thread blocks (2 warps), and 1024 blocks to saturate all SMs

## Results

- **speedup**: 27.168x (improved)


## Key Lessons

- Swap block/thread dimensions so that consecutive threads access consecutive memory addresses (stride-1) rather than strided access — this converts uncoalesced multi-cache-line warps into single-coalesced-transactions, and the effect is amplified when the inner loop iterates over partitions
- Smaller thread blocks (64 threads = 2 warps) yield higher SM occupancy than max-size blocks (1024 threads = 32 warps), enabling more concurrent blocks to hide memory latency
- Ensure block count equals or exceeds SM count (e.g., 1024 blocks vs 108 SMs) to fully utilize the GPU — only 64 blocks leaves most SMs idle
