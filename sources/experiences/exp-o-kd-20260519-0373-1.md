## Challenge

Flash attention Q/K gmem-to-smem transfers use single-buffer synchronous CP_ASYNC with immediate CP_ASYNC_WAIT_GROUP(0) + __syncthreads(), creating a serial load→sync→compute dependency chain that stalls MMA on every tile_K_d iteration


## Solution

Introduce kStage=2 double-buffering: a prologue pre-loads the first Q/K tile before the main loop, then each iteration issues async copies for the next tile into smem_sel_next (alternate buffer) while MMA runs on the current tile in smem_sel, with deferred CP_ASYNC_WAIT_GROUP(kStage-2) overlapping gmem latency with compute

## Key Lessons

- Double-buffer Q/K shared memory with prologue pre-load and async prefetch into alternate smem buffer (smem_sel_next) to overlap gmem-to-smem latency with MMA compute — applies when kHeadDim/kMmaAtomK >= 2 giving enough iterations to amortize the prologue overhead
- Replace CP_ASYNC_WAIT_GROUP(0) (blocking wait for all outstanding groups) with CP_ASYNC_WAIT_GROUP(kStage-2) (deferred wait allowing kStage-1 groups in flight) to decouple async copy completion from MMA start within the same iteration
- Remove __syncthreads() between s2r (smem-to-register) and the next g2s (gmem-to-smem) when using double buffers — the alternate buffer guarantees no write-after-read hazard on the same smem region
