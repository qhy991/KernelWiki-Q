## Challenge

Per-token FP8 quantization kernel uses a two-pass approach where pass-2 reloads the same fp16 input from global memory already loaded in pass-1, potentially wasting memory bandwidth for large hidden dimensions


## Solution

Stage input data to per-warp shared memory during pass-1 so pass-2 reads from smem instead of reloading from global memory, eliminating redundant global memory traffic

## Key Lessons

- For small per-token working sets (e.g. hidden_dim=512 fp16 = 1KB per token) the entire token fits in L2 cache, making global memory reload in pass-2 essentially free — shared memory staging adds overhead (extra smem stores, smem loads, __syncwarp) that negates the theoretical bandwidth savings
- Shared memory staging is only worthwhile when the working set is large enough to thrash L2 cache between pass-1 and pass-2; for small tensors, kernel execution time is dominated by launch overhead and warp reduction latency rather than memory bandwidth
- Always profile the neutral case: smem bank conflicts, extra instructions per element, and synchronization cost can erase the benefit of avoiding global reloads when the data is already cache-resident
