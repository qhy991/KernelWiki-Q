## Challenge

Per-token int8 quantization loop is inlined directly in the rms_norm_quant kernel, mixing RMS norm, scale computation, and quantization into a single monolithic kernel body, which reduces readability and prevents reuse of the quantization logic for blockwise kernel variants where scale lives in global memory


## Solution

Extract the quantization loop into a separate norm_and_quant_ptr device helper that receives scale via pointer (float* const scale_ptr), pre-invert token_scale before the call, and pass &token_scale — enabling reuse in blockwise kernels where scale resides in global memory while producing identical SASS for the per-token case since NVCC -O3 eliminates the thread-local pointer indirection

## Key Lessons

- Passing a thread-local scalar by pointer (float*) and immediately dereferencing it produces zero overhead at NVCC -O3 — the compiler recognizes the pointer targets a register and eliminates the indirection, generating identical SASS to a by-value pass
- Refactoring kernel code into device helpers with pointer-based scale parameters is safe for API extensibility (e.g., blockwise kernels where scale is in global memory) without regressing the per-token path, since the compiler fully optimizes away the pointer abstraction
- A 0.03% timing difference between two functionally identical kernels is within measurement noise on GPU benchmarks — do not interpret sub-0.1% deltas as meaningful performance changes
