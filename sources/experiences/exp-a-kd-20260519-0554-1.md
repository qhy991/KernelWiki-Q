## Challenge

FP32 RMS norm kernel uses scalar float loads and stores (one 32-bit transaction per element), requiring 256 threads per row and incurring high warp scheduling overhead and a long block reduction chain over 8 warps


## Solution

Vectorize memory access using float4 reinterpret_cast to load/store 4 contiguous floats per thread in a single 128-bit transaction, reducing thread count from 256 to 64 per row, which shortens the block reduction chain from 8 warps to 2 and cuts warp scheduling overhead

## Key Lessons

- Use float4 reinterpret_cast to vectorize contiguous FP32 loads/stores into 128-bit transactions — applies whenever threads access consecutive aligned floats, yielding 4x fewer memory instructions and proportionally fewer threads
- Reducing thread count via vectorization also shortens block reduction chains (fewer warps enter block_reduce_sum_f32), compounding the memory throughput gain with lower synchronization and scheduling overhead
- Compute per-element squares inside the float4 register (reg_x.x*reg_x.x + ...) before reduction to keep the arithmetic register-level and avoid extra memory traffic
