## Solution

Refactor the activation function to accept and return float32 directly, and move the elementwise multiply into float32 domain before the final bf16 cast — keeping the entire GELU-tanh + multiply pipeline in float32 and eliminating intermediate round-trip conversions

## Key Lessons

- When activation (GELU-tanh) involves expensive transcendental operations (tanhf, x^3), the cost of one additional bf16↔float cast is negligible — the dominant compute cost is the activation itself, not the precision conversion
- Design activation helpers to accept and return float32 natively rather than the storage type (bf16/fp16), so the caller controls when conversions happen and can fuse subsequent operations (like elementwise multiply) in the higher-precision domain
- Moving a multiply from bf16 domain to float32 domain does not produce measurable speedup when the surrounding activation (tanhf + cubic) already dominates instruction count — the optimization benefit is primarily numerical accuracy, not throughput
