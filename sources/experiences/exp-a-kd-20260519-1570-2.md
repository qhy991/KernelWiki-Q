## Challenge

When num_experts_per_group <= WARP_SIZE each thread holds at most one element, making per-thread second_largest tracking unnecessary overhead with wasted comparisons and extra register pressure


## Solution

Add a runtime branch on num_experts_per_group > WARP_SIZE: when false, each thread loads only its single element into largest and skips second_largest tracking entirely, deferring the second-largest computation to the warp-level reduce

## Key Lessons

- When data per warp is <= WARP_SIZE each thread holds at most one element — skip per-thread top-2 tracking and load only the single value, letting the warp-level reduction handle second-largest discovery → reduces register pressure and branch overhead for the common MoE routing case with small expert groups
