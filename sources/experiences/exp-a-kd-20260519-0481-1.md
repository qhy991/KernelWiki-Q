## Solution

Thread coarsening attempts to reduce thread count by 8x and have each thread issue 8 consecutive scalar half stores instead of using proper vectorized loads/stores (e.g., float4 reinterpret_cast), resulting in 4 warps per block that cannot hide memory latency

## Key Lessons

- Thread coarsening that reduces per-block thread count without using actual vectorized intrinsics (float4/half2 reinterpret_cast) degrades performance because 8 scalar stores do not coalesce into wider transactions, while the 8x thread reduction (1024→128) slashes occupancy and removes latency-hiding capability
- Proper vectorized embedding should use reinterpret_cast<float4*>(...) to issue 128-bit loads/stores while maintaining sufficient thread count for occupancy, rather than unrolling scalar accesses with fewer threads
- Memory-bound kernels like embedding lookups depend heavily on having enough active warps (≥16) to saturate memory bandwidth; reducing warps from 32 to 4 without compensating with wider transactions is a net regression
