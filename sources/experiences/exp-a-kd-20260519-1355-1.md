## Challenge

Batched GEMM with M=8 using tile configuration <64,64,128> pads M from 8 to 64, wasting 56 out of 64 rows (87.5% of M-dimension computation). The padded workspace is 64x1024x1024 per expert, launching 2048 thread blocks to compute mostly unused output elements.


## Solution

Swap A and B matrices (swap_ab) so computation becomes (N,K)x(K,M) with tile config <128,16,256>: N=1024 becomes the row dimension and aligns perfectly with the 128-wide tile (8 tiles, zero waste), while M=8 only needs padding to 16. Padded workspace drops from 64x1024x1024 to 1024x16x1024 (4x reduction), and thread blocks drop from 2048 to 512.

## Key Lessons

- For batched GEMM where one dimension is much smaller than the tile size (e.g., M=8 with tile_M=64), swap the A/B matrices so the large dimension becomes the row dimension — this eliminates padding waste by aligning the dominant dimension with the tile width, reducing padded work by up to tile_M/pad_M factor
- The swap_ab technique transforms O(tile_M/M) padding waste into O(tile_N/N) waste; when N>>M and N is already tile-aligned, this converts severe waste into negligible waste at the cost of a host-side transpose and result un-transpose
- When choosing tiling parameters for small-M batched GEMM, treat the padding ratio (padded_dim / real_dim) as a primary cost metric — a 4x reduction in padded multiply-adds can yield ~3x wall-clock speedup even without changing the kernel code
