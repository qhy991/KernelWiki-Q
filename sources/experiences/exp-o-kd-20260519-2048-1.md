## Challenge

sortAndScanExpert accepts int* (non-const) for the expert IDs parameter, forcing every call site to clone topk_ids via cudaMemcpy (128 KB device-to-device copy per invocation) before passing it, wasting GPU memory bandwidth and adding launch overhead


## Solution

Propagate const-correctness through the call chain: change CubKeyValueSorter::run's keys_in and sortAndScanExpert's expert_for_source_row parameters to const int*, which is semantically correct since CUB radix sort only reads the keys, allowing the caller to pass d_topk_ids directly and eliminating the per-invocation 128 KB device memcpy

## Key Lessons

- Propagate const-correctness through the entire call chain when a function only reads a buffer — adding const to a parameter eliminates the need for callers to defensively clone data, removing redundant device-to-device memcpy overhead that scales with input size
- CUB's DeviceRadixSort::SortPairs only reads the keys_in buffer (outputs to keys_out), so keys_in should always be const int* — a missing const qualifier is not just a style issue but can cause measurable performance loss by forcing unnecessary copies
