## Challenge

Scalar FP32 sigmoid kernel processes one float per thread with individual 4-byte load and store instructions, underutilizing memory bandwidth because each narrow memory transaction carries fixed overhead and cannot saturate the GPU memory pipelines


## Solution

Replace scalar float load/store with vectorized float4 (128-bit) load/store via FLOAT4 reinterpret cast, so each thread processes 4 elements with a single 16-byte load and a single 16-byte store, reducing memory instruction count by 4x and improving bandwidth utilization

## Key Lessons

- Vectorize scalar float load/store into float4 (128-bit) load/store via FLOAT4 reinterpret cast to reduce memory transaction count by 4x — applies to any elementwise kernel where each thread processes independent contiguous elements and the kernel is memory-bandwidth-bound
- Scaling thread index by 4 (threadIdx * 4) combined with float4 loads increases compute per thread 4x, which helps hide memory latency through instruction-level parallelism when the kernel is memory-bound
- Vectorized float4 access shows measurable speedup only at large data sizes (16M elements) where the kernel is truly memory-bandwidth-bound; at small sizes (1M) the overhead dominates and speedup is negligible (1.024x)
