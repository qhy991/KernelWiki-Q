## Challenge

FlashAttention-2 MMA kernel with kStage=1 uses synchronous Q/K loading where each inner-loop iteration issues CP_ASYNC_CG into smem_sel, then immediately blocks on CP_ASYNC_WAIT_GROUP(0) and __syncthreads() before ldmatrix and HMMA16816F32 can proceed, leaving Tensor Cores idle during global memory transfer latency


## Solution

Change kStage from 1 to 2 to activate double-buffered software pipelining: the async copy engine loads the NEXT Q/K tile into smem_sel_next alternate buffer while current MMA computes on smem_sel, with deferred CP_ASYNC_WAIT_GROUP(kStage-2) placed after MMA to overlap gmem-to-smem transfers with Tensor Core operations

## Key Lessons

- Double-buffered software pipelining (kStage=2) with CP_ASYNC_CG + deferred CP_ASYNC_WAIT_GROUP hides global memory latency behind Tensor Core MMA computation — applies when the kernel has sufficient inner-loop iterations to amortize pipeline fill/drain overhead (4+ tiles ideal)
- smem_sel / smem_sel_next ping-pong addressing via (tile_K_d % kStage) and ((tile_K_d + kStage - 1) % kStage) enables the async copy engine and Tensor Core to operate concurrently on different shared memory buffers without bank conflicts
- kStage=2 doubles shared memory footprint for Q/K tiles but trades smem capacity for compute throughput — worthwhile when gmem latency dominates over compute time, as indicated by kernel profiling showing high stall percentages on LG_THROTTLE or LSU stalls
