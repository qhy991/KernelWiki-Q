## Challenge

The before code uses a single splitk_atomic_reduce kernel where 4 thread blocks (k_rnd=4 shards) each iterate over all 16384 output elements and use atomicAdd to accumulate partial sums into a shared global buffer. The after code splits this into two kernels: splitk_write_slots writes partial result


## Solution

tor4 addition. The after code regresses because: (1) two kernel launches add launch overhead that, even at this larger 16384-element size, contributes measurably to total time; (2) the splitk_accumulate kernel's shared memory staging with __syncthreads() barriers adds synchronization cost — on NVIDI

- **speedup**: 0.834x (regression)
