## Challenge

CUTLASS int8 GEMM with TileShape 128x128x64 wastes 50% of threadblock compute when M=64 because only half of the 128-row M tile dimension is utilized, leaving half the warp threads idle on invalid rows


## Solution

Reconfigure TileShape from 128x128x64 to 64x128x128: halving the M tile dimension from 128 to 64 matches the actual M size exactly, eliminating wasted compute, while doubling the K tile from 64 to 128 increases pipeline depth and amortizes memory latency over more MAC operations per main-loop iteration

## Key Lessons

- Set the CUTLASS TileShape M dimension to match the actual problem M size exactly (or a clean divisor) to avoid threadblock-level compute waste — when M < tile_M, the excess rows produce no useful work but still consume SM resources
- When reducing the M tile dimension frees register/shared-memory budget, increase the K tile dimension to improve pipeline depth — a larger K tile amortizes memory latency across more multiply-accumulate operations per main-loop iteration
- TileShape tuning for small-M GEMM is a multi-dimensional trade-off: the M dimension must fit M snugly, N should remain wide for memory throughput, and K should be maximized within occupancy constraints to hide latency
