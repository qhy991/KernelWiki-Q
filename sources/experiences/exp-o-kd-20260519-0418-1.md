## Challenge

Shared memory s_a[BM][BK] is row-major with stride BK=8, so threads in the same column (same k) but different rows map to the same bank modulo 32, causing severe 16-way bank conflicts on every s_a read in the compute loop, and the compute loop directly reads from smem without staging into registers, serializing load and multiply operations


## Solution

Transpose A's shared memory layout from s_a[BM][BK] to s_a[BK][BM+OFFSET] so consecutive threads access consecutive BM-dimension columns (stride-1) hitting distinct banks; stage smem reads into r_comp_a/r_comp_b registers via float4 vector loads before the TM×TN compute loop; use __fmaf_rn to decouple smem latency from FFMA throughput; pad with OFFSET to break residual stride-aligned conflicts

## Key Lessons

- Transpose shared memory from [BM][BK] to [BK][BM] so that consecutive threads along the BM dimension access stride-1 columns mapped to different banks — eliminates multi-way bank conflicts when BK is a power-of-2 divisor of 32
- Stage shared memory into registers via vectorized float4 loads before the compute inner loop to decouple smem latency from arithmetic throughput, improving instruction-level parallelism
- Use __fmaf_rn instead of += for explicit fused multiply-add control and to help the compiler keep the multiply and accumulate pipelines fully occupied with register-resident operands
