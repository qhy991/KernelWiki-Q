## Challenge

MoE expert alignment uses two separate kernel launches — an align kernel with 1024 threads doing shared-memory atomicAdd for counting, and a sort kernel with global-memory atomicAdd for token placement — wasting kernel launch overhead and thread resources when numel is small (64 tokens)


## Solution

Replace the two-kernel atomicAdd pipeline with a single kernel using per-thread counting arrays and a thread-level prefix-sum over shared memory — each thread maintains its own expert counts in tokens_cnts[tid*num_experts..], then a single thread performs a cross-thread prefix-sum, enabling lock-free sorted_token_ids placement via pre-computed ranks

## Key Lessons

- For small token counts, kernel launch overhead dominates execution time — fusing two kernels into one eliminates both the extra launch and any intermediate cudaMemset, yielding significant speedup even when per-element work is trivial
- Per-thread counting arrays with a shared-memory prefix-sum eliminate atomicAdd entirely: each thread writes only its own partition, then one thread aggregates cross-thread counts — avoids shared-memory and global-memory atomic contention that is especially wasteful when most threads are idle
- Match thread count to actual work: launching 1024+256 threads for 64 elements leaves most threads idle; using 32 threads (one warp) keeps every thread productive and reduces shared-memory footprint
