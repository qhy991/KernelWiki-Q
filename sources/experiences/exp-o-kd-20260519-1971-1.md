## Challenge

cutlass::unified_encode_int4b performs int4 nibble remapping on the CPU by copying data from GPU to host via cudaMemcpy DeviceToHost, processing each byte serially with a per-nibble switch-case, then copying back via cudaMemcpy HostToDevice — incurring two synchronous PCIe transfers and serial CPU processing as a preprocessing bottleneck for large weight tensors in W4A8 quantization


## Solution

Replace the CPU round-trip with a GPU kernel (unified_encode_int4b_device) that keeps all computation on-device, using a 256-entry constant-memory lookup table (kNibbleLUT) that maps both nibbles of each byte simultaneously, processed with uint4 vectorized loads and grid-stride loops for bandwidth efficiency

## Key Lessons

- Keep data on-device to avoid GPU→Host→GPU PCIe round-trips — eliminating two synchronous transfers and serial CPU processing yields order-of-magnitude speedups when the operation can be expressed as a parallel GPU kernel, especially for large weight tensors in quantization preprocessing
- Use constant-memory LUT (256-entry nibble table) for small fixed mappings instead of serial per-element switch-case — enables parallel vectorized lookup (uint4 loads) across all threads, applicable whenever the mapping function has a small bounded domain
