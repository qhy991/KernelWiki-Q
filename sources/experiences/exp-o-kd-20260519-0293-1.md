## Challenge

Warp index computation uses __shfl_sync(0xffffffff, threadIdx.x/32, 0) inside a tight loop of 1M iterations, issuing a redundant SHFL instruction per iteration even though threadIdx.x/32 is already warp-uniform — all threads in the same warp produce the same quotient


## Solution

Replace __shfl_sync(0xffffffff, threadIdx.x/32, 0) with a plain threadIdx.x/32 computation wrapped in a __device__ __forceinline__ canonical_warp_idx() helper, eliminating the redundant SHFL instruction and its implicit warp convergence barrier so the compiler can schedule arithmetic more freely

## Key Lessons

- Avoid __shfl_sync for values that are already warp-uniform (e.g., threadIdx.x/32) — the shuffle instruction is redundant and adds both an implicit warp convergence barrier and an extra SHFL per invocation, which compounds in tight loops
- Prefer direct arithmetic (threadIdx.x / 32) over __shfl_sync broadcast for warp-index computation — since all threads in a warp share the same quotient, the shuffle communicates no new information
- Wrap host/device-portable warp utilities in __device__ __forceinline__ helpers with __CUDA_ARCH__ guards to keep code clean while ensuring correct compilation in both host and device contexts
