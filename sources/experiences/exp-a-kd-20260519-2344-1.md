## Solution

Replace vectorized BlockLoad/BlockStore with scalar BLOCK_LOAD_DIRECT/BLOCK_STORE_DIRECT variants that accept a seqlen parameter and out-of-bounds default value (0.f), enabling correct partial-block handling for varlen sequences at the cost of losing 128-bit vectorized memory transactions

## Key Lessons

- BLOCK_LOAD_VECTORIZE/BLOCK_STORE_VECTORIZE coalesce into 128-bit (float4) vector loads/stores, reducing total global memory transactions — use them when data is guaranteed full-length and aligned, but they cannot safely handle partial blocks in varlen scenarios
- Scalar BLOCK_LOAD_DIRECT/BLOCK_STORE_DIRECT with valid_items (seqlen) and out-of-bounds default value is the correct fallback for varlen correctness — the ~9% regression quantifies the performance cost of this correctness-preserving trade-off
- When designing kernels that must handle both fixed-length and variable-length sequences, gate the vectorized path behind a compile-time or runtime check (e.g., !kVarlen) so aligned full-length data uses vectorized loads while partial sequences fall back to scalar with boundary handling
