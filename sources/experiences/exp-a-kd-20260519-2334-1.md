## Challenge

Fixed-grid WMMA GEMM for M=32, N=32 output launches only a 2D grid of 2×2=4 blocks, each processing the full K=2880 dimension in one pass, leaving most of the 108 SMs on an A800 completely idle and severely underutilizing the hardware


## Solution

Split the K dimension into NUM_K_SHARDS=2 shards along a third grid dimension (blockIdx.z), doubling total blocks from 4 to 8 so twice as many SMs are active; each block computes a partial 16×16 tile stored to shared memory, then threads merge shards into global C via atomicAdd, with cudaMemset clearing C before each launch

## Key Lessons

- Split-K decomposition along a third grid dimension (blockIdx.z) doubles or multiplies block count without changing per-block shared-memory usage, making it ideal for skinny GEMM shapes where M×N produces too few tiles to fill all SMs
- When splitting K, store each shard's partial accumulator to shared memory then reduce into global memory via atomicAdd — the atomic overhead is small relative to the compute savings from shorter K-loops and better SM occupancy
- cudaMemset the output buffer to zero before launching a split-K kernel so that atomicAdd accumulates partial sums correctly across all K-shards
