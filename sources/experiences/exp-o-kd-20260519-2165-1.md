## Challenge

Fixed CUTLASS 2.x int8 GEMM configuration uses TileShape<128,128,64> with 5 stages for all problem sizes, causing excessive threadblock counts for large M dimensions and high shared memory pressure that limits occupancy


## Solution

Introduce shape-aware tile configuration selection that chooses TileShape<256,128,64> with 3 stages for large M (mp2>256) and N<=8192, doubling M-dimension coverage per threadblock to reduce total threadblock count and improve L2 cache locality, while reducing pipeline stages from 5 to 3 to lower shared memory pressure and enable higher occupancy

## Key Lessons

- Wider tile M-dimension (e.g., 256 vs 128) reduces total threadblock count along M, improving L2 cache data reuse across threadblocks — applies when M is large enough to benefit from wider tiles without exceeding shared memory capacity
- Reducing pipeline stages (e.g., 5→3) cuts shared memory consumption per threadblock, enabling higher occupancy and more concurrent threadblocks — tradeoff is less software pipelining overlap vs more SM resources available
- Shape-aware tile configuration dispatch (selecting tile dimensions and stages based on problem size rounded to power-of-2) outperforms a single fixed configuration across varying GEMM shapes — the dispatch table can be tuned per architecture (SM89) and dtype (int8→bf16)
