## Challenge

Kernel launches with a fixed block size of min(hidden_size, 1024) threads regardless of batch size, so when num_tokens is large each SM can only host a few heavy blocks, starving the scheduler of warps and leaving global memory latency unhidden


## Solution

Introduce adaptive block sizing: when num_tokens >= 256 cap max_block_size at 256 threads, and derive block_size from hidden_size divided by the vectorization factor so that more thread blocks fit per SM, raising occupancy and latency-hiding capability for large-batch workloads

## Key Lessons

- Adaptively reduce block size for large batch counts (cap at 256 threads when num_tokens >= 256) to increase concurrent blocks per SM and improve occupancy — applies whenever kernel parallelism is over tokens and each block's work is independent
- Derive block_size from hidden_size / vec_size rather than using a raw hidden_size cap, so vectorized loads naturally reduce per-thread work and the reduced block size still covers the full hidden dimension
- Tuning decisions that help large batches (smaller blocks) are often neutral for small batches where the original block size already yields sufficient occupancy — profile both regimes to confirm no regression
