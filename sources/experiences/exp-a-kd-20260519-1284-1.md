## Solution

The after code switches from GPTQ's two-step column-prefill-then-extract pattern to AWQ's one-step direct indexing, computing cur_n_packed/cur_n_pos once and reading packed values inline — but the accompanying tile layout change (sh_stride 64→8) introduces bank conflicts that cause a net regression of 14.2%

## Key Lessons

- Shared-memory stride must avoid multiples of 4 (for 32-bank, 4-byte-per-bank shared memory) — stride-8 access causes 4-way bank conflicts, while stride-64 spreads across all banks with low conflict, even though the total tile footprint is larger
- Simpler indexing code does not guarantee better performance: the GPTQ two-step pattern (column prefetch into local arrays + bit extraction) has better shared-memory access patterns than AWQ's one-step direct indexing when the underlying tile layout has poor stride properties
- When adapting a kernel from one quantization format to another (GPTQ→AWQ), the tile layout dimensions (sh_stride, tile_ints) are not interchangeable — they encode format-specific memory layouts that dominate performance over algorithmic simplification
