## Challenge

Scalar ELU kernel processes one float element per thread with separate 32-bit load and store instructions, underutilizing memory bandwidth and increasing thread scheduling overhead for large N


## Solution

Replace scalar 32-bit loads/stores with float4 (128-bit) vectorized loads/stores so each thread processes 4 elements in a single wide memory transaction, reducing total thread count by 4x and increasing per-thread memory throughput

## Key Lessons

- Use float4 vectorized loads/stores to process 4 fp32 elements per thread in elementwise kernels — one 128-bit transaction replaces four 32-bit transactions, improving memory bandwidth utilization by up to 4x
- Scale thread index by the vector width (idx * 4) and reduce total thread count proportionally — this lowers scheduling overhead especially noticeable at smaller problem sizes
