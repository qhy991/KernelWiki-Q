## Challenge

In the tiled SGEMM kernel prologue and main loop, B-tile data is loaded from global to shared memory synchronously via FLOAT4 stores, blocking the warp until each global read completes and preventing overlap with A-tile loading and transposition


## Solution

Replace synchronous FLOAT4 stores for B-tile loads with cp.async.ca (copy-async cache-all) PTX instructions, issuing two 16-byte async copies per iteration, committing the group, then proceeding with A-tile loading and online transpose while B transfers are in flight, and waiting for completion only after A-side work finishes

## Key Lessons

- Use cp.async.ca to issue asynchronous global-to-shared copies for one matrix tile (B) while the other tile (A) is loaded and transposed synchronously — the async transfer overlaps with A-side compute, hiding global memory latency without additional threads
- Issue cp.async copies and call cp.async.commit_group before independent work, then cp.async.wait_group(0) only after the overlapping work completes — this maximizes the overlap window between async DMA and synchronous register/shared operations
- __cvta_generic_to_shared converts a generic shared memory pointer to the shared address space required by cp.async PTX instructions; the byte offset (i * 4) advances the shared pointer by 4-float granularity for each 16-byte async transfer
