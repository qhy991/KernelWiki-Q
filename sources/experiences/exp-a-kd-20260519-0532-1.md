## Solution

Replace B_PAD=8 padding with XOR permutation swizzle pattern (swizzle_permuted_B_j applying (((j>>3)^(i>>2))%2)<<3) that scatters 8-byte chunks across different banks for different row groups, eliminating structural bank conflicts during LDMATRIX_X4 loads while keeping shared memory compact at BK=16 columns per row and freeing 50% of B smem space for higher K_STAGE counts

## Key Lessons

- Replace shared memory padding with XOR-based swizzle patterns to eliminate bank conflicts while reducing smem footprint — the XOR permutation scatters accesses across banks without wasting memory on padding columns, yielding both bandwidth and capacity benefits
- At moderate grid sizes (4x4 blocks), B matrix bank conflicts during LDMATRIX_X4 loads become a significant bottleneck because concurrent warps create high smem traffic that amplifies the conflict stalls, making swizzle more impactful than at smaller grid sizes
- Freeing smem space by eliminating padding (BK+8→BK, 50% reduction per tile) enables higher K_STAGE pipeline depths, providing a secondary path to improved throughput beyond bank conflict elimination
