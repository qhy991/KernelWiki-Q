## Challenge

The before code computes ptr_Vector = vector + col_start (missing the tile_m * ldr row-stride offset), while the after code computes ptr_Vector = vector + col_start + tile_m * ldr. Both perform the same number of memory accesses and arithmetic operations per thread;


## Solution

Add the missing tile_m * ldr row-stride term to the bias vector pointer offset, matching the CUTLASS convention ptr_Vector += threadblock_item_begin.column() + tiled_coord.m() * ldr, so each tile reads bias from its correct tile-row in the 2D bias array

## Results

- **speedup**: 0.997x (neutral)


## Key Lessons

- In tiled GEMM epilogues with per-tile-row bias, the bias pointer must include both column offset and row-stride offset (tiled_coord.m() * ldr) — omitting the row stride causes tiles beyond the first tile-row to read stale bias values from row zero, producing silent correctness errors
- Adding a single integer multiply-add to a pointer base computed once before the main loop has negligible performance impact since the cost is amortized over all TILE_M × TILE_N memory accesses and arithmetic inside the loop — correctness fixes of this form are effectively free
