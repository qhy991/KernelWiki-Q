## Challenge

Scalar hardswish kernel loads and stores one f32 element per thread, issuing 4 separate memory transactions per group of 4 elements and underutilizing available memory bandwidth for a memory-bound elementwise kernel


## Solution

Vectorize load/store using float4 (16-byte) transactions so each thread processes 4 elements via a single coalesced load and a single coalesced store, reducing total memory transaction count by 4x and improving memory throughput

## Key Lessons

- Float4 vectorized load/store reduces memory transaction count by 4x for elementwise kernels, but measured speedup may remain within noise margins when the scalar version already saturates memory bandwidth through L2 cache coalescing on high-bandwidth GPUs like A800
- float4 vectorization reduces thread count by 4x (256→64 threads/block), trading per-thread memory efficiency against lower occupancy — net benefit depends on whether the kernel is truly memory-bandwidth-bound or already coalescing effectively through hardware
