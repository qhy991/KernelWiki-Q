## Challenge

MoE sum-reduce kernel uses a compile-time TOPK template parameter to enable full loop unrolling via #pragma unroll, but cannot handle arbitrary runtime topk values — passing TOPK=-1 produces zeros, sacrificing correctness for performance


## Solution

Replace the compile-time TOPK template with a runtime int topk_num parameter and change #pragma unroll to #pragma unroll 1, accepting a 2.2x performance regression as a trade-off for correctness on arbitrary topk values — the general kernel eliminates code bloat from multiple template instantiations but loses the compiler's ability to pipeline 8 independent load-convert-add sequences

## Key Lessons

- Compile-time template parameters with #pragma unroll enable full loop unrolling that eliminates loop overhead and exposes instruction-level parallelism across iterations — losing this for runtime flexibility can cause 2x+ slowdown when the unrolled loop is the dominant work
- Use dispatch-by-topk pattern for MoE kernels: specialize via templates for common topk values (1,2,4,8) to retain full unrolling, and fall back to a general runtime-topk kernel only for uncommon values
- Full unrolling of small fixed-count loops (8 iterations) allows the compiler to pipeline multiple independent load-convert-add chains, overlapping memory latency with arithmetic — #pragma unroll 1 explicitly prevents this pipelining
