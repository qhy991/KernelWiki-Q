## Solution

Parameterize warpReduceSum with a compile-time numLanes template argument so the loop runs only log2(numLanes) rounds; blockReduceSum passes the exact active-lane count (e.g., 8 for 256-thread blocks) to the final reduction, cutting shuffle rounds from 5 to 3

## Key Lessons

- Parameterizing warp reductions with a compile-time active-lane count eliminates unnecessary shuffle rounds (e.g., 5→3 for 8 active lanes), but the saving is negligible when the per-warp reductions and shared-memory barrier dominate total latency
- Use static_assert and constexpr helpers (_nextPow2) to enforce power-of-2 lane counts at compile time — enables safe unrolled reductions for sub-warp active thread counts
- Template-parameterized block size (maxBlockSize) allows the compiler to optimize shared-memory allocation and reduction rounds without runtime branching, but measurable gains require the eliminated instructions to be on the critical path
