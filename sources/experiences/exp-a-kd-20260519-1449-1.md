## Challenge

Split-K GEMM reduction uses a single kernel where all shard blocks concurrently atomicAdd partial results into the same global output buffer, creating potential atomic contention across shards for each output element


## Solution

Replace single-kernel atomicAdd reduction with a two-phase approach: (1) splitk_write_slots writes each shard's partial results to per-shard slots without atomics, then (2) splitk_accumulate loads all slots into shared memory via float4 and accumulates using vector4 addition — however this regresses on NVIDIA GPUs because the extra kernel launch overhead dominates the small 2048-element workload and the shared-memory staging lacks the AMD-specific global-to-LDS direct hardware path

## Key Lessons

- Replacing atomicAdd with a multi-kernel two-phase reduction (write-slots + accumulate) regresses for small workloads on NVIDIA GPUs — the extra kernel launch overhead (~2-5μs per launch) exceeds any atomic contention savings when the working set fits entirely in L2 cache where atomicAdd resolves cheaply
- AMD's __builtin_amdgcn_global_load_lds intrinsic provides a direct hardware path from global memory to LDS without register transit, making shared-memory staging virtually free on gfx950 — this optimization is platform-specific and counter-productive on NVIDIA GPUs which lack an equivalent hardware mechanism
- atomicAdd on NVIDIA GPUs is highly efficient for small output tensors that fit in L2 cache because the L2 acts as a low-latency atomic domain — for workloads under ~8KB, atomicAdd often outperforms any multi-pass reduction scheme
