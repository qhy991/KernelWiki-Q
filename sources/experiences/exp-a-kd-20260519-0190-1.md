## Solution

Replace __shfl_sync warp shuffle with canonical_warp_idx() that computes threadIdx.x / 32 via pure arithmetic with no synchronization semantics, allowing the compiler to hoist the loop-invariant division out of the unrolled loop and keep the result in a register for all iterations

## Key Lessons

- __shfl_sync acts as a warp-level barrier that blocks compiler instruction reordering — never use it for broadcasting a lane-invariant value (like threadIdx.x / 32) that every lane computes identically, because the barrier prevents loop-invariant code motion and common subexpression elimination
- threadIdx.x is a read-only special register constant for the kernel lifetime — any pure arithmetic expression derived from it is loop-invariant and should be computed once outside loops rather than redundantly per iteration
- When computing warp-uniform values, prefer direct arithmetic (threadIdx.x / warpSize) over shuffle intrinsics — the arithmetic has no synchronization semantics, enabling the compiler to freely hoist, fold, and optimize the expression
