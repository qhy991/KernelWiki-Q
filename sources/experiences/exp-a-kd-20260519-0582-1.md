## Challenge

Scalar f32 SGEMV kernel loads one element per thread per iteration with naturally coalesced access, but the attempted float4 vectorization spreads each warp's loads across a 512-element stride (32 threads × 4 elements apart), causing L2 cache line thrashing that degrades performance for large matrices exceeding cache capacity


## Solution

Replace scalar f32 loads with float4 vectorized loads so each thread processes 4 elements per iteration, reducing loop iterations by 4× and issuing 128-bit load instructions instead of 32-bit loads — however this specific stride pattern breaks coalescing for large K

## Key Lessons

- Naive float4 vectorization in GEMV where threads index as (warp_id*WARP_SIZE+lane)*4 spreads adjacent threads 16 bytes apart, breaking coalescing — scalar sequential access within a warp can be faster because 32 threads reading 32 consecutive floats maps to 4 contiguous 128-byte cache lines
- Vectorized loads are beneficial only when the access stride preserves coalescing (e.g., float4 from a contiguous base per thread) or when the working set fits in L2 cache; for large matrices exceeding cache, coalescing quality dominates over raw bandwidth per instruction
- When applying vectorized loads to memory-bound kernels, verify that the resulting address pattern keeps warp-wide accesses within the same cache lines — use shared memory staging or restructure the thread-to-data mapping rather than simply multiplying the stride by the vector width
