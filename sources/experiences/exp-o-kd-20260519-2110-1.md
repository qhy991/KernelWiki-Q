## Challenge

Router GEMM kernel passes num_tokens as a runtime parameter, forcing the compiler to allocate a fixed-size accumulator array of 16 floats and preventing full unrolling of the token loop, which adds loop control overhead and excessive register pressure for the M=1 case


## Solution

Convert num_tokens from a runtime function parameter to a compile-time template parameter kNumTokens, enabling the compiler to fully unroll the token loop, allocate exactly the needed number of accumulator registers, and add __launch_bounds__(128,1) to minimize register pressure for the M=1 specialization

## Key Lessons

- Convert small runtime iteration counts (especially M in GEMM) into compile-time template parameters — this allows the compiler to fully unroll loops, allocate exact accumulator registers, and eliminate loop control overhead, yielding significant speedups for small M values
- Use __launch_bounds__(blockSize, minBlocksPerSM) when the compile-time register count is known exactly — with kNumTokens=1 only one accumulator is needed, so __launch_bounds__(128,1) minimizes register pressure and maximizes occupancy
- Over-allocating fixed-size arrays (float acc[16]) for runtime-variable counts wastes registers and prevents compiler optimization — prefer template-parameterized array sizes (float acc[kNumTokens]) so the compiler sees the exact allocation
