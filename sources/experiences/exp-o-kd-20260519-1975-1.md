## Challenge

Each thread counts elements less than a target expert ID by linearly scanning a 32K-entry sorted array with early exit; threads for higher expert IDs must scan increasingly large prefixes up to ~28K elements, resulting in O(N) per-thread work dominated by cache-line-sequential memory reads


## Solution

Replace the per-thread O(N) linear scan with a templated binary search (findTotalEltsLessThanTarget) that performs only ~log2(N) ≈ 15 random-access comparisons per thread regardless of expert ID, converting cache-line-sequencing reads into a handful of pointer-chasing comparisons

## Key Lessons

- Replace O(N) linear scans with O(log N) binary search when searching sorted arrays on GPU — each thread performs only ~15 comparisons instead of potentially thousands of sequential reads, eliminating the dominant memory bottleneck
- Use a template parameter for the search function (template <class T>) rather than hardcoding the element type — enables reuse across different sorted-index representations without changing the algorithm
- Binary search on GPU trades sequential cache-friendly access for random-access pointer chasing, but the dramatic reduction in total memory reads (log N vs N) makes it a net win for any array larger than a few hundred elements
