## Challenge

Kernel prolog loads sequence lengths via global memory access on the critical path, blocking barrier initialization and cluster synchronization that must complete before any warp can begin useful work


## Solution

Issue cp.async for seq_len loading immediately on thread 0 before barrier initialization, then overlap the async load latency with barrier setup work in warp 0, finally fence with cp_async_fence + cp_async_wait<0> followed by cluster_arrive_relaxed to synchronize both the load result and barrier state in one cluster_sync

## Key Lessons

- Issue latency-critical global memory loads (like seq_lens) via cp.async as early as possible in the prolog, then overlap their latency with independent initialization work (barrier setup) — the async load completes in parallel with the barrier init, reducing prolog overhead by the load latency
- Use cluster_arrive_relaxed() + cluster_wait() instead of the heavier cluster_sync() after barrier init — the relaxed variant avoids an unnecessary membar.gpu (~0.2us) because fence_barrier_init() already provides the required ordering for mbarrier init operations in cluster scope
