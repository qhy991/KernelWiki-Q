## Challenge

Zero-point constants in 4-bit dequant are constructed via expensive float subtraction/division and __float2half_rn conversion pipeline, costing multiple instructions per dequant invocation, then combined into half2 via __halves2half2 which is a two-argument combine rather than a dedicated broadcast


## Solution

Replace float-arithmetic constant construction with IEEE 754 bit-level manipulation: half_uint16(0xe400 | zero) directly ORs the runtime zero into the known bit pattern for -1024.0f bypassing all float arithmetic and conversion hardware; replace float-division path for z16 with __hsub(__int2half_rn(-64), __int2half_rn(zero)) using hardware int-to-half lookup; replace __halves2half2(x,x) broadcasts with __half2half2(x) dedicated broadcast intrinsic

## Key Lessons

- Use bit-level IEEE 754 manipulation (integer OR into known bit patterns) instead of __float2half_rn for half-precision constant construction when the base value has a known representation — eliminates float subtraction/division and the full float-to-half conversion pipeline
- Replace float-division constant paths with __int2half_rn + __hsub when the operands are small integers — __int2half_rn maps via hardware/table lookup which is cheaper than float division + rounding conversion
- Prefer __half2half2(val) over __halves2half2(x, x) for broadcasting a single half into both lanes of a half2 — it is the dedicated broadcast intrinsic with explicit intent and potentially fewer register operations
