## Solution

Introduce four M-dimension-aware SM80 GEMM config structs with progressively smaller TileShape/WarpShape values (16x64x128 for M<=16, 32x64x128 for M<=32, 64x128x128 for M<=64, 128x128x64 for large M) and a dispatch function that selects the appropriate config based on next_pow_2(M), increasing GPU occupancy by matching tile dimensions to actual problem size

## Key Lessons

- Use M-dimension-adaptive CUTLASS tile configurations instead of a single fixed tile shape — smaller tiles for small M increase occupancy by allowing more concurrent thread blocks, especially critical for LLM inference where batch sizes are often small
- Dispatch on next_pow_2(M) to select tile configurations with power-of-2 thresholds — this provides clean branch boundaries and avoids per-value branching overhead while covering the small-M regime effectively
- For intermediate M values (e.g., 64<M<=128), add an N-dimension split (N<8192 vs N>=8192) to further differentiate between tall-skinny and square-ish problems that benefit from different tile strategies
