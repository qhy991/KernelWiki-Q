## Challenge

FP8 KV-cache gather and dequantization are implemented as two separate kernel launches: a 1024-thread gather kernel copies 656 bytes per token into an intermediate buffer via a generic byte-copy loop, then a second 576-thread kernel reads that buffer to produce BF16 output, incurring redundant write-then-read traffic per token plus an extra kernel launch and an O(batch_size) linear search for batch_id in the dequant kernel


## Solution

Fuse gather and FP8-to-BF16 dequantization into a single kernel where each thread block (indexed by blockIdx.x=batch_id, blockIdx.y=split) iterates over tokens in its sequence split, reads FP8 data directly from paged cache blocks and writes converted BF16 output in-place, eliminating the intermediate buffer, reducing thread count from 1024 to 576 to match per-token work, and removing the O(batch_size) linear search since batch_id comes directly from blockIdx.x

## Key Lessons

- Fusing a paged-cache gather kernel with a per-token compute kernel (dequant/activation) into a single pass eliminates intermediate buffer allocation and the redundant write-then-read traffic — applies when both kernels operate on the same per-token granularity and the gather loop can carry the compute inline
- When the grid topology already encodes batch identity (blockIdx.x = batch_id), deriving batch_id from the grid index removes O(batch_size) linear search over workspace_starts — applies to any paged-attention or KV-cache kernel where sequences map to blocks
- Matching thread count to actual per-token work items (576 threads for 512 FP8 values + 64 RoPE values) instead of using a generic 1024-thread configuration avoids wasted threads and register pressure — applies when the fused kernel has a known work partition
