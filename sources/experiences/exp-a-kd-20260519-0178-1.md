## Challenge

Replacing a trivial threadIdx.x / 32 integer division (compiled to a single shift instruction) with __shfl_sync broadcast adds a full warp shuffle barrier per invocation, causing severe regression in tight loops calling the function millions of times


## Solution

The after code wraps threadIdx.x / 32 in __shfl_sync(0xffffffff, ..., 0) to broadcast lane 0's result warp-wide, guaranteeing warp-uniform values even under divergence — but since threadIdx.x is already uniform within each warp in standard launches, the shuffle barrier adds overhead with no benefit

## Key Lessons

- __shfl_sync introduces a warp convergence barrier and register broadcast cost per invocation — avoid it in hot loops when the input (threadIdx.x) is already guaranteed warp-uniform, as the shuffle overhead dominates the trivial arithmetic it protects
- threadIdx.x / 32 compiles to a single shift instruction (SHR) and is effectively free; __shfl_sync requires warp synchronization, a register move to the shared exchange buffer, and cross-lane communication — orders of magnitude more expensive in aggregate
- Use __shfl_sync broadcast for correctness guarantees only when threads may have diverged (e.g., warp-level voting, divergent branch results) — not as a defensive wrapper around already-uniform quantities
