## Challenge

RMS normalization kernel launches with a fixed block size of 512 threads per block; with 512 blocks (num_tokens=512), each 512-thread block consumes significant SM resources, limiting concurrent blocks per SM and reducing memory-level parallelism for this bandwidth-bound kernel


## Solution

Use adaptive max_block_size based on grid size: cap at 256 threads per block when num_tokens is large (>=256), allowing 2x more blocks to reside concurrently on each SM and improving latency hiding for the global memory transfers that dominate this bandwidth-bound kernel

## Key Lessons

- Adaptive block sizing based on grid size improves occupancy for bandwidth-bound kernels — when the grid has many blocks, reducing threads per block allows more concurrent blocks per SM, increasing memory-level parallelism and latency hiding
- For memory-bandwidth-bound kernels like RMS normalization, higher occupancy via smaller block sizes outperforms fewer large blocks because global memory latency is the bottleneck, not compute throughput
- Shared memory buffers sized for the maximum block size (s_buf[1024]) remain valid when actual block size is smaller — no shared memory waste, but the reduction loop correctly adapts via blockDim.x / 2
