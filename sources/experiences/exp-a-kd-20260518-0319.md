## Challenge

The before kernel (relu_f16_kernel) processes one half value per thread using __hmax, issuing separate 16-bit scalar load and store per element. The after kernel (relu_f16x2_kernel) packs two half values into a half2 (32-bit) load/store, with each thread processing two elements. At ~1M half elements


## Solution

(2MB), both kernels achieve coalesced memory access patterns: consecutive threads in the scalar kernel access consecutive half addresses (coalesced within a 32-bit segment), while the half2 kernel naturally coalesces two half values per 32-bit word. The half data being only 2MB fits well in L2 cach
