## Challenge

cuBLAS grouped GEMM with M=4 is padded to PADDED_M=128 to meet tile-shape requirements, causing 32x wasted FLOPs per expert (134M computed vs 4.2M useful) and correspondingly inflated latency


## Solution

Swap A and B operands with CUBLAS_OP_T on both, computing C^T = B^T * A^T where m=N=1024 and n=M=4, so the large dimension becomes the leading GEMM dimension and no M-padding is needed since N=1024 already exceeds tile boundaries

## Key Lessons

- For small-M grouped GEMM (e.g. MoE expert routing), swap A/B operands with CUBLAS_OP_T on both to transpose the problem so the large dimension becomes m — this eliminates wasteful M-padding to tile boundaries and reduces FLOPs by PADDED_M/M ratio
- The gap between theoretical FLOP reduction (32x) and measured speedup (1.75x) comes from kernel launch overhead and different internal cuBLAS tiling for the transposed dimension configuration — the FLOP reduction is real but not fully realized in wall-clock time for small problem sizes
- cuBLAS computes C^T = B^T * A^T correctly when op(A)=CUBLAS_OP_T and op(B)=CUBLAS_OP_T; the output layout changes from column-major C[M×N] to column-major C^T[N×M], so the correctness check must index C_T[n + m*N] vs C_ref[m + n*M]
