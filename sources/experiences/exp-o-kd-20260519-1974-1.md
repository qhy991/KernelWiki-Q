## Challenge

Each thread performs a linear O(N) sequential scan through a sorted expert array to count elements less than its target expert ID, causing threads handling higher expert IDs to read nearly all 262K entries before reaching their boundary — catastrophic work imbalance and redundant memory reads


## Solution

Replace the per-thread linear scan with a templated binary search (findTotalEltsLessThanTarget) that finds the upper bound of elements less than the target in O(log N) steps, exploiting the sorted-array precondition to eliminate the sequential memory traversal entirely

## Key Lessons

- When computing partition offsets into a sorted array, use binary search (O(log N)) instead of linear scan (O(N)) — the sorted precondition makes binary search directly applicable and yields orders-of-magnitude speedup as array size grows
- Thread-level parallelism does not compensate for per-thread algorithmic inefficiency: 65 threads each doing O(N) work is still O(65N), while 65 threads each doing O(log N) work is O(65 log N) — algorithm choice dominates over parallelism for this pattern
