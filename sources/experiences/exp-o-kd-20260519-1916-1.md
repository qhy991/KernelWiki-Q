## Solution

Replace the scalar strided loop with vectorize_read_with_alignment<8> which loads 8 floats (32 bytes) per transaction via a vec_n_t<float,8> struct, reducing 4096 individual 4-byte loads to 512 wider 256-bit vector loads while handling alignment edge cases with a scalar fallback

## Key Lessons

- Vectorize global memory reads with aligned vec_n_t<scalar_t,8> structs (32-byte loads) instead of scalar strided loops — each thread processes 8 elements per iteration, reducing total load instructions by 8x and improving memory throughput even for 4-byte float types that already coalesce reasonably well
- The vectorize_read_with_alignment helper handles misalignment via a scalar prefix/tail fallback, making vectorized reads safe for any pointer alignment — use it instead of manual reinterpret_cast when the alignment guarantee is uncertain
- Speedup from vectorization is moderate (~6%) for fp32 data because a warp of 32 threads reading consecutive float addresses already coalesces into 128-byte transactions; the benefit is larger for fp16/bf16 where scalar reads produce under-utilized 64-byte transactions
