## Challenge

The backward tile kernel launches 256 threads (8 warps) per block despite only kHeadDim=64 elements per row, meaning 75% of threads are idle during the computation phase but still incur synchronization overhead in the shared-memory reduction tree.


## Solution

Reduce kBlockSizeI from 128 to 64, cutting the thread count from 256 to 128 so that half the threads now perform useful work instead of one quarter, while also shrinking the shared-memory reduction tree from 8 steps to 7 and reducing barrier synchronization cost across fewer warps.

## Key Lessons

- Match block thread count to the actual data dimension rather than oversubscribing — idle threads still pay synchronization cost at every __syncthreads() barrier and waste scheduler slots without contributing computation
- Reducing block size for per-row kernels where the work dimension is smaller than the block dimension directly cuts shared-memory footprint and reduction tree depth, yielding compounding savings in both memory and barrier overhead
