## Solution

Extract the token-sorting loop into a separate sgl_moe_token_sort_kernel launched across multiple CTAs (up to 65535 blocks x 256 threads) using global-memory atomicAdd on cumsum_buffer, while simplifying the original align kernel to only handle expert counting, cumsum, and expert_id assignment

## Key Lessons

- When a phase within a single-block kernel is embarrassingly parallel and independent of the block-cooperative computation, extract it into a separate multi-block kernel to unlock full GPU occupancy — applies to MoE token sorting, histogram binning, and similar per-element scatter workloads
- Grid-stride loops (for i = tid; i < numel; i += stride) in multi-block kernels provide portable work distribution across any launch configuration and handle arbitrary input sizes without needing to compute per-thread chunk bounds
- Shared-memory atomicAdd on a small fixed-size array (e.g., 256 entries for 256 experts) creates contention that grows with thread count — moving to global-memory atomicAdd across many blocks reduces per-location contention even though global atomics are individually slower
