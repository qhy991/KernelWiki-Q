## Challenge

Stride-based iteration in a small-batch MOE alignment kernel causes increased shared memory bank conflicts because all 32 threads simultaneously access nearby offsets within the 8-column count matrix, overlapping on the same memory banks


## Solution

Replace per-thread contiguous sharding with stride-based iteration (tid, stride=32) hoping for better load balancing, but this causes more shared memory bank conflicts as all threads concurrently access adjacent columns in the narrow 8-expert count matrix

## Key Lessons

- Per-thread contiguous sharding spreads shared memory writes across distinct rows of the count matrix, reducing bank conflicts compared to stride-based iteration where all threads access nearby columns simultaneously — prefer sharding when the count matrix has few columns (small expert counts)
- Stride-based iteration is not universally superior to contiguous sharding; for narrow shared memory matrices (e.g., 8 columns for 8 experts), the stride pattern increases bank conflict probability because multiple threads hit the same 32-bank groups within adjacent rows
- When the working set (512 tokens across 8 experts) fits entirely in shared memory, load balancing via stride iteration offers no advantage over contiguous sharding but incurs higher bank conflict overhead
