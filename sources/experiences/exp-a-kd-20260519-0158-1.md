## Challenge

Split-K reduction kernel computes kCount using ElementOutput size (half_t, 16-bit) instead of ElementWorkspace size (float, 32-bit), yielding kCount=8 instead of 4, which causes oversized tile columns, underutilized grid parallelism, inflated register pressure from 8-element accumulator arrays, and mismatched vector access width for float data


## Solution

Correct kCount to match ElementWorkspace (float, 32-bit): 128/32=4, which halves tile column width from 256 to 128, doubles the number of active thread blocks in the Y grid dimension for higher occupancy, reduces per-thread accumulator registers from 8 to 4 floats, and aligns access width to the natural 128-bit vector load boundary for float data

## Key Lessons

- Size vectorization parameters (kCount, vector length) by the workspace/scratchpad element type, not the output element type — mixing types between computation and storage leads to incorrect stride calculations and suboptimal access patterns
- Halving per-thread accumulator size from 8 to 4 floats reduces register pressure, which can free registers for higher occupancy or more warps per SM
- Matching kCount to the natural 128-bit vector width of the accessed type (4×float = 128 bits) ensures coalesced memory transactions and avoids wasted bandwidth from misaligned access patterns
