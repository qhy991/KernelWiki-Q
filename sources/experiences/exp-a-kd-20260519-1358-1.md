## Solution

Swap A/B operands using CUBLAS_OP_T for both, reinterpreting the problem as C^T[N x M] = B^T[N x K] * A^T[K x M] so that cuBLAS sees m=N=1024 and n=M=4 — N=1024 is already well above any tile boundary, eliminating the need for M-padding entirely and reducing per-expert FLOPs from 134M to 4.2M

## Key Lessons

- When the M dimension is much smaller than the GEMM tile size (e.g., M=4 vs tile=128), swap A/B operands via CUBLAS_OP_T so the large N dimension takes the 'm' role in cuBLAS — this avoids M-padding and eliminates wasted FLOPs from computing zero-padded rows
- The swap_ab technique computes C^T = B^T * A^T which is mathematically equivalent to C = A * B; correctness requires transposing the output indexing (C_T[n + m*N] = C[m][n]) but saves 32x FLOPs in this regime
- Measured speedup (1.75x) can be much less than theoretical FLOP reduction (32x) due to kernel launch overhead and different cuBLAS internal tiling for the swapped dimension configuration — the optimization is still worthwhile but actual gains depend on launch overhead ratio
