## Solution

Place the reduction region at a dedicated non-overlapping offset beyond the full pipeline footprint (sh + STAGES*PIPE_STAGE_INT4 instead of sh + STAGES*PIPE_STAGE_INT4/2), and increase shared memory allocation to accommodate both regions, eliminating data races between concurrent pipeline reads and reduction writes

## Key Lessons

- When multiple shared memory sub-arrays (pipeline stages, reduction buffers) are carved from a single extern __shared__ allocation, always place them at non-overlapping offsets computed from the cumulative size of preceding regions — overlapping offsets cause silent data corruption when warps access them through conditional __syncthreads() paths
- A correctness fix that increases shared memory footprint may show neutral or slightly negative microbenchmark latency because the extra shmem allocation doesn't change the access pattern in a single-block test, but prevents real race-condition bugs in multi-warp production kernels
