## Challenge

Custom blockReduceSum dispatches through a function-pointer (ReduceFnType<T>) to detail::_sum for warp-level shuffle reduction and shared-memory cross-warp aggregation, introducing potential indirect-call overhead compared to a purpose-built library implementation


## Solution

Replace the custom function-pointer-based blockReduceSum with CUB's cub::BlockReduce using the cub::Sum compile-time functor and explicit blockDim.x parameter, which encodes the reduction operator at compile time rather than through runtime dispatch

## Key Lessons

- Replacing a custom function-pointer block reduction with CUB BlockReduce yields no performance difference when the operator body is a single trivial instruction (FADD) — nvcc -O3 inlines the function-pointer call just as effectively as the CUB functor path, because the dominant cost is the warp shuffle network and shared-memory barrier, not the dispatch mechanism
- Function-pointer dispatch overhead is negligible for simple arithmetic operators in reduction loops — the real optimization opportunities lie in the shuffle/memory access patterns, not the operator invocation mechanism
- CUB BlockReduce is still preferable for maintainability and code clarity even when performance is equivalent, as it replaces ~40 lines of custom reduction boilerplate with 3 lines of library code
