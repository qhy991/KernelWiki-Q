## Challenge

In the MoE pipeline, SiLU(gate)*up was computed by a separate silu_and_mul kernel, writing an intermediate activation tensor to global memory, which the quantization kernel then read back — incurring one extra kernel launch and a full round-trip of the intermediate through global memory


## Solution

Added a FUSE_SILU_MUL compile-time template parameter to cvt_fp16_to_fp4; when enabled, the kernel reads both gate and up vectors from the doubled-width input (gate||up layout), applies compute_silu_mul in-register, and feeds the result directly into the FP4 quantization path — eliminating one kernel launch and one full global-memory round-trip for the intermediate activation

## Key Lessons

- Fusing activation (SiLU+Mul) with a downstream consumer kernel (quantization) via a compile-time template parameter eliminates one kernel launch and one full global-memory round-trip for the intermediate tensor — applies whenever an activation's output is immediately consumed by another kernel in the pipeline
- When fusing a gate||up layout activation, double the effective column stride (colsPerRow * 2) and index the up vector at inOffset + colsPerRow to maintain contiguous gate access while reading the paired up vector
- Use if constexpr with a boolean template parameter for zero-cost compile-time branching — the unused branch is completely eliminated from the binary, preserving peak throughput for the non-fused path
