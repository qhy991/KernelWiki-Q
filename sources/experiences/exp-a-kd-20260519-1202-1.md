## Challenge

Marlin quantized GEMM host-side M-split dispatch uses a for-loop with hardcoded max_par=16 and fixed blocks=sms, limiting parallelism for small N dimensions where higher M-split parallelism would benefit throughput at larger batch sizes


## Solution

Refactor M-split dispatch from a for-loop over tot_m_blocks to a while(rest_m) loop with N-dependent max_par (128 for N<=4096 vs 16 otherwise) and blocks_per_sm for occupancy tuning, enabling more concurrent M-splits across SMs and dynamic fallback to smaller thread_m_blocks for tail work

## Key Lessons

- Increase max_par for small N dimensions (e.g., max_par=128 for N<=4096) to allow more concurrent M-splits across SMs — small N kernels have lower per-SM work, so higher parallelism fills idle SMs without shared-memory contention
- Replace for-loop M-split dispatch with while(rest_m) loop and dynamic fallback to smaller thread_m_blocks — handles arbitrary M dimensions and tail fragments without requiring external pad/iteration bookkeeping
- Use blocks_per_sm from an exec_config_t to tune occupancy based on per-block shared memory budget — enables blocks_per_sm>1 when shared memory usage is low enough to fit multiple blocks per SM, improving throughput for memory-bound kernels
