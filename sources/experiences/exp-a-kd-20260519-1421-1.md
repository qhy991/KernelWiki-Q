## Challenge

GEMM A-matrix tile reader computes row stride as prob_k / 8 int4 units, which only matches the actual leading dimension when lda == prob_k (contiguous layout). When lda > prob_k (padded rows), the stride is too small, causing reads from incorrect memory addresses that land in padding regions of adjacent rows instead of the intended row data.


## Solution

Pass the actual leading dimension (lda) as an explicit kernel parameter and compute a_gl_stride = lda / 8 instead of prob_k / 8, ensuring correct row-to-row pointer advancement for non-contiguous (padded) A matrices where lda exceeds prob_k.

## Key Lessons

- Never assume A-matrix row stride equals prob_k in GEMM kernels — always use the explicit leading dimension (lda) for address arithmetic, since the caller may supply a padded layout where lda > prob_k for alignment or batch-stride reasons
- A correctness fix that passes the actual lda through as a kernel parameter may have a small runtime cost (~6% from the extra parameter and slightly larger stride arithmetic) but eliminates silent data corruption on non-contiguous inputs
