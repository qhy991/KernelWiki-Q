## Challenge

Scalar strided int8 quantization kernel uses a per-element loop with 1024-thread blocks, processing one float-to-int8 conversion per thread iteration — the loop is memory-bound with strided scalar stores that prevent memory coalescing across warps


## Solution

Replace the scalar strided loop with vectorize_with_alignment<16> (16-element vectorized loads/stores) and reduce block size from 1024 to 256 threads, aiming to coalesce memory transactions — however the 4x thread reduction drops occupancy from 32 warps/block to 8, severely reducing the SM scheduler's ability to hide memory latency, causing a net regression

## Key Lessons

- Vectorized memory access alone does not guarantee speedup when coupled with a block size reduction that cuts occupancy below the warp count needed for latency hiding — the thread reduction must be proportional to vectorization width to maintain sufficient schedulable warps
- For memory-bound quantization kernels with lightweight per-element work (float divide + int8 convert), high occupancy from many thin threads often outperforms fewer threads doing wider vectorized loads because the SM scheduler needs enough warps to overlap memory latency with compute
- The vectorize_with_alignment<16> pattern increases register pressure per thread (16-element temporaries), further reducing achievable occupancy and compounding the latency-hiding deficit from the smaller block
