## Challenge

Marlin MoE WNA16 quantized GEMM kernel uses hardcoded pipe_stages=4 for shared memory buffer sizing, causing the combined A and B matrix buffers to exceed Turing's 64KB shared memory limit and making the kernel impossible to launch on sm75 GPUs


## Solution

Replace the hardcoded compile-time constant pipe_stages=4 with a runtime stages parameter determined by compute capability: 2 for Turing sm75 (64KB shared memory) and 4 for Ampere+ sm80 (100KB shared memory), then propagate stages through all shared-memory sizing functions to halve buffer footprint on constrained hardware

## Key Lessons

- Replace hardcoded pipeline stage counts with a runtime parameter derived from CUDA compute capability — this allows shared memory footprint to adapt to hardware constraints (Turing 64KB vs Ampere 100KB) without separate kernel compilations
- Halving pipeline stages from 4 to 2 reduces shared memory by ~40-44% (measured 1.66x-1.78x byte reduction) with a tradeoff of fewer overlapping async copies, which is acceptable when the alternative is a kernel that cannot launch at all
- When propagating a new parameter through a call chain of sizing/config functions, ensure every function that references the old constant (pipe_stages) is updated to accept and use the new parameter (stages) to avoid inconsistent shared memory calculations
