## Solution

Replace column padding (B_PAD=8) with XOR-based swizzle address permutation via swizzle_permuted_B_j, which rearranges column-to-bank mapping at load time using the formula (((j>>3)^(i>>2))%(kColStride>>3))<<3 to eliminate bank conflicts while keeping a compact BK-column layout

## Key Lessons

- XOR-based swizzle permutation eliminates shared-memory bank conflicts without column padding, reclaiming up to 50% of wasted smem per tile row — applies whenever a tiled GEMM/conv kernel uses padding for bank-conflict avoidance
- Compact smem layout (B_PAD=0 with swizzle) enables higher K_STAGE counts within the shared-memory budget, which directly increases arithmetic intensity by overlapping more async copies with computation
- Swizzle formulas like (((j>>3)^(i>>2))%(kColStride>>3))<<3 permute 8-byte chunk indices so that concurrent warp-row accesses hit different banks — the bit-shift granularity must match the access width (128-bit = 8 half-values = 3-bit shift)
