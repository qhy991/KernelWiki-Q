## Challenge

Each thread performs 16 separate byte-level shared memory loads from scattered K-major layout addresses (indexed by row_offset * packed_k + packed_idx with non-contiguous packed_idx), causing multiple smem bank-conflict stalls and 16 independent load instruction issues on the critical path


## Solution

Reorder shared memory layout on the host side so that all 16 INT4 elements owned by one thread occupy 8 contiguous bytes, then load them as a single uint64_t via shift-or assembly — reducing 16 scattered byte-level smem transactions to 1 wide 64-bit contiguous load

## Key Lessons

- Reorder shared memory layout so that all elements accessed by the same thread are stored contiguously — converts N scattered narrow loads into 1 wide vectorized load, reducing smem bank-conflict stalls and load-instruction count on the critical path
- For INT4 sub-byte types, host-side data reordering (permute thread-owned elements into contiguous byte blocks) is essential to unlock vectorized smem loads; the reorder cost is amortized across many kernel invocations
- When simulating WGMMA thread-value access patterns, the natural K-major layout scatters a thread's owned elements — a thread-contiguous layout (similar to compute_memory_reordering_atom) is the hardware-efficient alternative
