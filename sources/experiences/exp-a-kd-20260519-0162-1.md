## Challenge

A chain of dependent GEMM+tail kernel pairs launches strictly sequentially on the same stream; each successor kernel cannot begin until the predecessor kernel has fully exited (including epilogue and tail work), because there is no device-side signal at the work-exhaustion point to allow overlap


## Solution

Introduce a CUDA event recorded at the work-exhaustion point (after the GEMM kernel but before the tail kernel) so the successor iteration can wait on that event rather than full kernel exit, simulating the SM90a griddepcontrol.launch_dependents PTX instruction that lets dependent grids begin while the predecessor's tail work still runs

## Key Lessons

- Programmatic Dependent Launch (griddepcontrol.launch_dependents on SM90a) signals successor grids at the work-exhaustion point inside a kernel, allowing the successor to overlap with the predecessor's tail work (epilogue, fixup) — but on SM80 and earlier hardware, same-stream serialization already enforces ordering so an event-based proxy provides no real overlap benefit
- Use cudaEventDisableTiming when creating events used only for synchronization (not profiling) to reduce overhead — timing-enabled events force the driver to flush the GPU pipeline on record, adding latency
- Event-based work-exhaustion signaling only helps when the successor runs on a different stream or when hardware PDL support exists; on a single stream the explicit cudaStreamWaitEvent is redundant because the stream already serializes
