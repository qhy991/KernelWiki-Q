## Solution

Fuse the activation and scaling into a single templated kernel where the gated-SiLU result stays in registers and is immediately multiplied by the per-expert prequant scale before a single store, eliminating the intermediate global-memory write/read; additionally use a template-parameterized kProcessRows (1/2/4) selected at runtime based on total work to let each block process multiple rows for better occupancy

## Key Lessons

- Fuse consecutive kernels that pass data through a global-memory intermediate into a single kernel so the intermediate stays in registers — the savings are proportional to the size of the intermediate tensor and apply whenever two kernels share a producer-consumer relationship on a per-element basis
- Use a compile-time template parameter (e.g., kProcessRows) to let a single kernel adapt its per-block work granularity to the total problem size, improving occupancy on small problems without code duplication
- Kernel fusion eliminates not only the extra global-memory traffic but also kernel launch overhead, L2 cache pollution from the intermediate buffer, and the synchronization barrier between the two kernel submissions
