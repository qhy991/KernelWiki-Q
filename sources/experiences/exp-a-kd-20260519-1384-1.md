## Solution

Replace scalar element-by-element access with 4-element struct-based vectorized loads/stores via vec4_t<__half> and float4_t, processing four elements per iteration to increase memory throughput per transaction

## Key Lessons

- Vectorized struct-based loads/stores (4-element vec4_t/float4_t) widen memory transactions to improve bandwidth utilization for elementwise kernels — but only benefit when problem size is large enough for the kernel to reach steady-state throughput
- Kernel launch overhead (~5 µs) and CUDA event timing granularity dominate microbenchmarks on small datasets (e.g., 4096 elements = 24 KB total I/O), making vectorization effects invisible within measurement noise — benchmark with problem sizes that saturate memory bandwidth to validate vectorization gains
- A remainder loop handling non-divisible tail elements is necessary when vectorizing to avoid out-of-bounds access, but adds branch and instruction overhead that can negate gains on small inputs
