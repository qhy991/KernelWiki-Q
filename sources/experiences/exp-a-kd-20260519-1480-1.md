## Solution

Replace batch-split grid (1024-thread CTAs with nested block/token loops) with per-token grid (64-thread CTAs where each CTA handles exactly one token), eliminating thread idleness (576/64=9 meaningful iterations vs 576/1024<1), increasing occupancy through smaller CTAs, and removing split boundary and partial-block divergent conditionals

## Key Lessons

- Match CTA size to the work dimension (ENTRY_SIZE=576) so that every thread has multiple meaningful iterations — 64 threads give 576/64=9 iterations vs 1024 threads where 576/1024<1 leaves >40% of threads completely idle
- Prefer per-token (fine-grained) grid launches over batch-split (coarse-grained) launches for paged KV-cache gather kernels — the finer granularity eliminates nested block/token loops and split-boundary divergence while naturally load-balancing across tokens
- Smaller CTAs (64 vs 1024 threads) improve occupancy for memory-bound workloads by allowing more concurrent CTAs per SM, increasing effective memory throughput when the kernel is not compute-limited
