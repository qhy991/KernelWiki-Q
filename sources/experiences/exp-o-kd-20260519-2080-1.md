## Challenge

The original kernel launched with grid(batch_size, num_splits) and block(1024) threads, where each CTA processed a split of blocks via a two-level loop (over blocks, then tokens within each block) plus split boundary and partial-block handling. The 1024-thread CTA with only 576 scalar elements per entry left most threads idle (576/1024 < 1 iteration per thread), and the nested loop created significant overhead for long sequences.


## Solution

Replace the batch-split grid launch with a per-token launch grid(num_tokens) and reduce CTA size to 64 threads, so each CTA processes exactly one token. With 64 threads and 144 vector iterations (576/4 vec_size), each thread performs 2-3 iterations for much better utilization, while the per-token launch eliminates the block-level loop and all split/partial-block conditional logic, enabling more concurrent CTAs on GPU.

## Key Lessons

- Size CTA threads to match the actual per-element vectorized workload (work_elements / vec_size) so each thread performs ≥1 full iteration, rather than defaulting to 1024 threads which leaves most idle when the work count is small
- Per-token (or per-row) CTA launch eliminates nested loops and split bookkeeping when each CTA can independently handle one logical unit of work — applies whenever the kernel iterates over independent tokens, rows, or entries
- Smaller CTAs (64 vs 1024) increase the number of concurrent CTAs the GPU can schedule, improving occupancy for long-sequence workloads where many independent work items exist
