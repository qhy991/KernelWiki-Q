## Solution

Fuse rope_only_kernel and kv_writeback_kernel into a single fused_rope_kv_kernel where K RoPE results (y1, y2) remain in registers after computation and are written directly to both k_rope and k_buf, eliminating the intermediate global memory write-and-re-read cycle and the second kernel launch

## Key Lessons

- Fuse producer-consumer kernel pairs where the consumer immediately reads data the producer just wrote — keeping intermediate results in registers eliminates redundant global memory round-trips that dominate bandwidth-bound workloads like RoPE + KV cache write-back
- In LLM serving, RoPE + KV cache write-back is a natural fusion candidate because the RoPE-transformed K values must immediately be stored into the paged KV cache; register-resident results can be written to both the RoPE output and k_buffer without a global memory intermediary
