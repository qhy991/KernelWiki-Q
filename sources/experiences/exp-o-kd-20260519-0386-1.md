## Challenge

HardSwish half-precision kernel processes only 2 elements per thread via a single half2 register pair, resulting in high thread count, excessive warp scheduling overhead, and poor instruction-level parallelism for a memory-bound elementwise kernel


## Solution

Increase per-thread workload from 2 to 8 half elements by unrolling into 4 independent half2 register pairs (reg_x_0..3), issuing all 4 loads upfront then interleaving 4 independent compute-store sequences; this quadruples data per thread, reduces thread count 4x, and exposes instruction-level parallelism that fills the memory pipeline

## Key Lessons

- Increase per-thread element count (2→8 half elements) for memory-bound elementwise kernels by unrolling into multiple half2 register pairs — the compiler can interleave independent load-compute-store sequences to hide memory latency and improve ILP
- Issue all loads upfront before computing to let the memory controller begin fetching while registers are being prepared, enabling load-compute overlap that a single-pair load-then-compute-then-store pattern cannot achieve
- Use per-pair bounds checks (if (idx+offset < N)) instead of a single global check when unrolling beyond the natural vector width, to handle tail elements correctly without masking valid early stores
