## Solution

Adapt tile shape to TILE_M=16 TILE_N=64 so blockDim(64,16) matches M=16 exactly, eliminating idle threads in computation; however, blockDim.y=16 < TILE_K=32 forces cooperative sB loading loops (each thread iterates twice), and sB grows from 4KB to 8KB, increasing shared memory pressure and reducing occupancy, resulting in a net regression

## Key Lessons

- Adapting tile shape to match problem dimensions can eliminate idle threads but may introduce cooperative loading overhead when blockDim dimensions do not evenly cover tile dimensions — the loop iterations added when blockDim.y < TILE_K can outweigh the savings from removing idle threads
- Widening one tile dimension (TILE_N 32→64) increases shared memory footprint proportionally (sB: 4KB→8KB), which can reduce SM occupancy and overall throughput — always compute the shared memory budget when changing tile shapes
- Eliminating idle threads is not always a net win: the before version's idle threads still participate in cooperative loading with single-iteration loops (no overhead), while the after version's full thread utilization comes at the cost of doubled loading iterations and reduced occupancy
