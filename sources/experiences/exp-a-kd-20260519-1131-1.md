## Challenge

Naive serial block reduction forces thread 0 to iterate over all blockDim.x partial sums from shared memory sequentially — with 1024 threads this requires 1023 serial additions, creating a severe stall bottleneck for the entire block


## Solution

Replace the serial single-thread reduction with a two-phase warp-shuffle block reduction: each warp reduces its partial sum via __shfl_xor_sync in log2(32)=5 steps, then the first warp reduces 32 warp-level sums in another 5 steps, yielding 10 total parallel shuffle steps instead of 1023 serial additions

## Key Lessons

- Replace serial single-thread shared-memory reductions with warp-shuffle block reduction (warpReduceSum + blockReduceSum) — __shfl_xor_sync performs log2(warpSize)=5 parallel reduction steps per warp vs O(blockDim.x) serial steps, yielding near-logarithmic scaling for any block size
- Shared memory footprint for cross-warp reduction scales with the number of warps (32 floats for a 1024-thread block) rather than with blockDim.x (1024 floats), reducing shared memory pressure and improving occupancy
- For block sizes much larger than warpSize (e.g., 1024 threads = 32 warps), the serial reduction penalty grows linearly with block size while the warp-shuffle approach stays at 2×log2(32)=10 steps — the speedup gap widens as block size increases
