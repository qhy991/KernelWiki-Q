## Challenge

Inside a tight 1000-iteration loop, __shfl_sync(0xffffffff, threadIdx.x / 32, 0) is used to compute warp_idx even though all threads are already converged, incurring multi-cycle shuffle pipeline latency and warp scheduler overhead on every iteration


## Solution

Replace __shfl_sync with a __device__ helper canonical_warp_idx() that compiles to a single integer division (threadIdx.x >> 5), eliminating the warp-cooperative shuffle instruction entirely and allowing each thread to compute warp_idx independently with a low-latency ALU operation

## Key Lessons

- Using __shfl_sync to broadcast a locally-computed value (threadIdx.x / 32) that every thread already has is redundant — prefer direct integer arithmetic to avoid shuffle pipeline latency and warp coordination overhead, especially inside tight loops
- When all threads in a warp compute the same value independently, __shfl_sync adds unnecessary synchronization cost; simple per-thread ALU operations (integer division/shift) are faster and do not require warp cooperation
- Extract repeated index computations into a __device__ helper with a __CUDA_ARCH__ guard to keep the code portable for host-side compilation while enabling GPU-optimized paths
