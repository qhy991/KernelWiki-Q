## Challenge

The before code unconditionally launches persistent_gemm_kernel with grid_size=total_tiles=8, meaning only 8 thread blocks run on a 108-SM A800 — 100 SMs are idle. Each of the 8 blocks must process the entire K=4096 dot product per output element


## Solution

The after code detects K=4096 > 3*N=192 and selects streamk_gemm_kernel with grid_size=num_sms=108, distributing all M*N=32768 output elements evenly across 108 blocks. This fills all SMs and reduces per-block work from ~4096 output elements to ~304, cutting latency dramatically. The key insight ...

## Results

- **speedup**: 15.443x (improved)


## Key Lessons

- The key insight is that the k > 3*n heuristic correctly identifies shapes where the persistent tile model under-occupies the GPU and switches to a work-distribution model that achieves full SM utilization.
