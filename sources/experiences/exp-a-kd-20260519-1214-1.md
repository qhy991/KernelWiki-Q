## Challenge

The variance accumulation loop in rms_norm_kernel uses scalar strided access, loading one __half per thread per iteration; for larger hidden sizes this would underutilize per-thread memory bandwidth, but at hidden_size=256 the GPU already coalesces 256 scalar __half loads into efficient memory transactions


## Solution

Replace the scalar strided loop with vectorize_read_with_alignment<8> that issues 128-bit vector loads (8x __half) per thread to increase per-thread memory throughput, handling alignment via prefix scalar ops and vectorized body

## Key Lessons

- Vectorized loads provide no benefit when hidden_size ≤ blockDim.x (each thread already does only 1 element) — vectorization reduces active threads from 256 to 32 while adding lambda and alignment overhead, yielding neutral or worse performance
- GPU memory coalescing already converts contiguous scalar __half loads from a warp into efficient 128-byte memory transactions at small data volumes (≤512 bytes), so explicit vectorization is redundant when total data fits in a few coalesced transactions
- Apply vectorize_read_with_alignment only when per-thread element count is large enough (≥4–8) to amortize the alignment-check branching and lambda-capture overhead; at 1 element per thread the overhead dominates
