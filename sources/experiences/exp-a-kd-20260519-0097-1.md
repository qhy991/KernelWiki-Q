## Challenge

Two separate CUTLASS GEMM kernels execute sequentially with an intermediate FP16 tensor (81920x64) written to and read back from global memory, incurring an unnecessary GM round-trip between back-to-back GEMM operations that could be fused


## Solution

Fusing both GEMMs into a single cutlass::gemm::device::B2bGemm kernel that keeps the intermediate accumulator in the register file, using reduced EpilogueOutputOp0 ElementsPerAccess (InstructionShape::kM*kN/32 = 4) for RF-resident layout and expanding WarpShape1 from (32,64,32) to (32,128,32) to accommodate the in-register intermediate

## Key Lessons

- B2bGemm RF-resident fusion eliminates intermediate global memory round-trips between back-to-back GEMMs — most beneficial when the intermediate matrix is large enough that GM bandwidth is a significant fraction of total runtime
- RF-resident fusion trades full-width epilogue stores (ElementsPerAccess=8) for reduced-width stores (ElementsPerAccess=4) to keep intermediates in registers — the net benefit depends on whether saved GM traffic exceeds the store efficiency loss
- For problem sizes where the intermediate K dimension is small relative to M, the bandwidth saved by eliminating the intermediate GM write/read is modest compared to total compute, limiting fusion benefit to near-neutral speedup
