## Challenge

The naive RMSNorm kernel launches one 32-thread block per row (8192 blocks for 8192 tokens), exceeding GPU concurrent capacity (~6912 blocks), causing block scheduling overhead and poor SM residency for the lightweight per-row computation


## Solution

Replace one-block-per-row launch with occupancy-aware grid sizing and a grid-strided loop so each block processes multiple rows, keeping blocks resident on SMs and amortizing block setup overhead across rows

## Key Lessons

- Use occupancy-based grid sizing (cudaOccupancyMaxActiveBlocksPerMultiprocessor × num_sm) instead of one-block-per-row launches when the number of rows exceeds GPU concurrent capacity — keeps blocks resident on SMs and avoids block scheduling overhead
- Grid-strided loops (for row = blockIdx.x; row < N; row += gridDim.x) amortize block setup cost across multiple rows per block and improve L2 cache locality by reusing the SM-local cache partition for sequential row accesses
- Per-row compute weight matters for speedup magnitude: with kDim=256 (8 elements/thread) the speedup is 1.36x vs 3.57x for kDim=128 (4 elements/thread), because heavier per-row work makes scheduling overhead a smaller fraction of total latency
