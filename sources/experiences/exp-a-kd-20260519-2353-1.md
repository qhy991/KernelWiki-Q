## Challenge

A 256-byte nibble lookup table stored in __device__ global memory is replaced with __constant__ memory, but the warp-wide divergent LUT indexing (each thread reads kNibbleLUT[b[i]] for 16 different byte values) causes constant-cache serialization, degrading throughput by ~3.3x


## Solution

Moving the LUT from __device__ global memory to __constant__ memory causes constant-cache serialization under divergent access patterns: each warp issues 16+ unique addresses per thread, and the constant cache must service them sequentially, whereas the L2 cache behind global memory handles divergent reads in parallel via multi-bank structure

## Key Lessons

- __constant__ memory only benefits performance when all threads in a warp read the same address (uniform access); for divergent per-thread LUT lookups, the constant cache serializes requests, making __device__ global memory (backed by L2 with multi-bank parallel access) significantly faster
- Small read-only lookup tables accessed with warp-divergent indices should prefer global memory with L2 caching over __constant__ memory — the 64 KB constant cache is optimized for broadcast, not scatter-gather reads
- Benchmark before adopting __constant__ memory: the conventional wisdom that 'constant memory is faster for read-only data' is only valid when the access pattern is uniform across the warp, not when each thread indexes independently
