## Challenge

MoE block-alignment kernel uses int32_t for all shared-memory token counts, requiring ((blockDim.x+1)*num_experts + (num_experts+1))*4 bytes. With 256 experts (DeepSeek-V3) this exceeds 48KB device shared memory limits, forcing a global-memory fallback with 5-10x higher access latency.


## Solution

Reorder shared memory so cumsum (always int32_t) sits at the base and tokens_cnts is type-punned via a new template parameter token_cnts_t; instantiating with uint16_t when topk_ids.numel() <= 65535 halves the token-counts footprint, enabling shared-memory execution for large expert counts that previously exceeded device limits.

## Key Lessons

- Place the fixed-type shared memory array (cumsum → int32_t) at the base of extern __shared__ and derive the variable-type array (tokens_cnts → token_cnts_t) from an offset; this layout separation is required to safely type-pun a sub-region of shared memory without misaligning the other region.
- Template the element type of auxiliary shared-memory arrays (e.g., uint16_t vs int32_t for bounded counters) to halve shared memory footprint when the value domain fits in fewer bits — this is the difference between fitting in 48KB shared memory and forcing a global-memory fallback on large configurations (256+ experts).
- When token counts are bounded by topk_ids.numel() and that value is ≤ 65535, uint16_t is sufficient and cuts per-element storage from 4 to 2 bytes, enabling shared-memory kernels for MoE configurations that would otherwise exceed device shared memory limits.
