## Challenge

Group absmax reduction uses shared memory (1088 bytes: s_absmax[16][17]) with two __syncthreads barriers and only 8 of 16 threads actively participating in the reduction, leaving half the warp idle during the reduce phase


## Solution

Replace shared-memory GroupReduceMax with __shfl_xor_sync warp shuffle reduction operating entirely in registers, eliminating 1088 bytes of shared memory, two __syncthreads barriers, and enabling all 16 lanes to participate in the 4-round reduction

## Key Lessons

- Replace shared-memory reduction with __shfl_xor_sync warp shuffle when the reduction group size fits within a single warp (≤32 lanes) — eliminates shared memory allocation, two __syncthreads barriers, and bank-conflict padding, trading them for pure register-to-register exchange in 4 rounds
- The speedup from warp shuffle is modest (4.4%) when the reduction phase is a small fraction of total kernel time dominated by memory-bound iteration loops — profile before prioritizing this transformation
- Using a 16-bit active mask (0xffff) with __shfl_xor_sync for sub-warp groups (16 lanes here) correctly scopes the shuffle to only the participating lanes within the warp
