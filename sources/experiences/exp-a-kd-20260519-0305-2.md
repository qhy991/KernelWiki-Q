## Solution

Restructure to a 2D grid where blockIdx.x maps to row groups and blockIdx.y maps to column vector indices, so each thread owns a dedicated float4 slot; a compile-time kProcessRows template parameter allows each block to process multiple rows, and the launch function selects the optimal group size based on total work to balance occupancy and per-thread work

## Key Lessons

- A 2D grid mapping (blockIdx.x → row groups, blockIdx.y → column vectors) gives each thread a dedicated float4 slot, producing coalesced memory access when consecutive threads in a warp read adjacent float4 elements within the same row
- Adaptive row-group sizing via template parameter (kProcessRows = 1/2/4) selected at launch time based on total work allows the kernel to scale across problem sizes: fewer rows per block for small problems (more blocks = better occupancy), more rows per block for large problems (more cache reuse per block)
