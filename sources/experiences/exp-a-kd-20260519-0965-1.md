## Challenge

Monolithic RMSNorm function bundles cooperative warp load, computation, and store into a single function taking raw pointers, loading weight only after computing norm_factor rather than pre-loading both input and weight together, which prevents the compiler from overlapping weight memory latency with the reduction and norm_factor arithmetic


## Solution

Decompose monolithic function into separate phases: TileMemWarp struct for cooperative warp load/store (constructed once outside the work loop and reused), and apply_norm_warp pure computation function that takes pre-loaded half2 vectors rather than raw pointers, enabling both input and weight to be loaded before any computation begins

## Key Lessons

- Decomposing a monolithic kernel into separate load/compute/store phases yields equivalent PTX scheduling for small loop counts (kLoopCount=2, kHeadDim=128) but enables instruction-level parallelism between memory loads and computation when loop counts or data volumes increase
- Pre-loading weight before computation (rather than after norm_factor computation) allows the compiler and hardware to overlap weight memory latency with the warp reduction and norm_factor arithmetic — beneficial when the weight load latency is non-trivial
- Extracting cooperative warp load/store into a reusable TileMemWarp struct and separating pure computation into a parameterized apply_norm_warp function improves code reusability across different tile sizes and loading strategies without changing the math
