## Solution

Create two explicit CUDA streams (stream1, stream2) and launch each independent GEMM on its own stream with cudaSetDevice(0) for explicit device context, enabling the GPU scheduler to overlap both kernels on the SM pool concurrently rather than serializing them on the default stream

## Key Lessons

- Launch independent kernels on separate explicit CUDA streams to enable concurrent SM occupancy — the default stream (stream 0) serializes all work, while explicit streams allow the GPU scheduler to overlap kernels that share no data dependencies
- Explicitly set cudaSetDevice(device_id) before stream creation when targeting multi-GPU systems, mirroring the CUDAGuard pattern — this prevents silent errors where the active device context differs from the intended one
- When benchmarking concurrent streams, synchronize each stream individually (cudaStreamSynchronize) rather than using cudaDeviceSynchronize, to correctly measure overlapping wall-clock time with per-stream event timers
