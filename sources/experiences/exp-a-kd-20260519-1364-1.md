## Challenge

Small-M GEMM dispatch (M=3, K=4096, N=4096) uses CUBLAS_OP_N for both operands, passing (M,K)x(K,N) directly to cuBLAS which on SM80 selects an internally tuned low-M kernel path that handles the tiny leading dimension m=3 adequately


## Solution

Apply swapAB heuristic (swap when M<=64 or M%4!=0) by transposing the GEMM into (N,K)x(K,M) using CUBLAS_OP_T on both operands with swapped A/B buffers and leading dimensions — intended to improve tile utilization for small M, but on SM80 without TMA the transposed access pattern forces non-contiguous stride traversal (lda=k=4096 for B, ldb=m=3 for A) causing uncoalesced global memory loads and 68.26% regression

## Key Lessons

- swapAB transformations that improve GEMM tile utilization on SM120+ with TMA (which handles transposed 2D tile loads natively) can severely regress on SM80 without TMA, where transposed access patterns cause uncoalesced global memory loads — architecture-specific validation is mandatory
- cuBLAS internal kernel selection for small M dimensions (M<=256) already applies low-M tuned paths on SM80; overriding this with swapAB forces suboptimal stride patterns (ldb=m=3) that the hardware cannot efficiently service
- Dispatch heuristics like swap_ab = (M<=64) || (M%4!=0) must account for GPU generation: the condition correctly identifies poor tile utilization but the fix (transposing operands) is only beneficial when the hardware natively accelerates transposed tile loads (TMA on SM90+)
