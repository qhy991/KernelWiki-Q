## Challenge

In online softmax sum computation, atomicAdd serializes partial-row contributions from multiple warps into shared s_prime entries, causing contention, nondeterministic numerical ordering, and retry-loop overhead


## Solution

Replace atomicAdd with warp-partitioned addition_storage where each warp writes its partial sum to a dedicated slot indexed by (accum_m + kQueriesPerBlock * warp_column_index), then after __syncthreads a single lane per row sequentially accumulates all warp contributions — eliminating atomic contention and producing deterministic results

## Key Lessons

- Replace atomicAdd on shared memory with per-warp partitioned storage slots plus a single-lane sequential reduction — eliminates atomic serialization and retry overhead while producing deterministic results, applicable whenever multiple warps accumulate into the same shared-memory counters
- Warp-partitioned reduction reads from disjoint addition_storage slots, avoiding bank conflicts in the common case — the deterministic sequential sweep by one lane per row is cheaper than N-way atomic contention when warp count is modest (2–8 warps)
