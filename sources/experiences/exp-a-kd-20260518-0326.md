## Challenge

At the largest configuration (8192 tokens, 1024 hidden size), the f32x4 pack kernel processes 4 floats per thread via 128-bit transactions while the scalar kernel does 4 separate 32-bit transactions per pair. The combination of larger grid size and vectorized access gives


## Solution

Fuse the two separate compute+update loops into a single loop body and add isfinite() NaN guards on each expf result, aiming to reduce loop overhead and improve numerical robustness — but the fusion increases register pressure and the isfinite checks add extra instructions that together cause a net slowdown

## Results

- **speedup**: 1.314x (improved)


## Key Lessons

- Loop fusion is not always beneficial — when fusing two loops forces all intermediate arrays (m_delta[4], s_delta[4][32], m_prime, s_prime, m_i, si) to be live simultaneously, register pressure can spike enough to reduce occupancy and negate the saved loop overhead, especially for small tile sizes (kQ=4, kK=32)
- Adding isfinite() guards on every expf result introduces extra compare-and-select instructions inside the inner loop — the defensive NaN check costs throughput that only pays off when inf/NaN values are actually expected in the workload
- Splitting computation into phases with narrower live ranges (compute all deltas first, then update accumulators) can schedule better on GPUs by allowing the compiler to spill and reload intermediate arrays across phase boundaries rather than holding everything in registers at once
