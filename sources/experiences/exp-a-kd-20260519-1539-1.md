## Solution

Replace whole-tile persistent scheduling with StreamK decomposition: split each tile's K dimension into k_iters_per_tile independent work units, yielding total_k_iters (256 for this problem) far exceeding SM count, launch grid_size=num_sm to engage all 108 SMs, and use atomicAdd for partial accumulation since multiple SMs now share a tile

## Key Lessons

- Decompose tile work along K dimension (StreamK) when tile_count << SM_count, converting whole-tile assignments into K-iteration work units that saturate all SMs — applies to tall-skinny or small-batch GEMM where the M×N tile grid underutilizes the GPU
- When splitting a tile across multiple SMs, switch from direct write to atomicAdd for output accumulation; the atomic overhead is negligible compared to the massive parallelism gain from engaging previously idle SMs
- Use cudaLaunchCooperativeKernel with grid_size = num_sm (not min(tiles, num_sm)) so the cooperative grid spans all SMs, enabling uniform work distribution across the entire GPU
