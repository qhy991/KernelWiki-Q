## Challenge

K-cache dot-product kernel and host code hardcode block_stride and head_stride using num_heads (NH), making the stride layout incompatible with grouped-query attention (GQA) where num_kv_heads < num_heads, without any clear path to generalize the parameterization


## Solution

Replace hardcoded NH-based stride computation with NKV-based strides (block_stride = NKV * HS * BS, head_stride = HS * BS) so the same kernel and host code work for both MHA and GQA; when num_kv_heads == num_heads the computed strides are identical, making this a zero-cost abstraction

## Key Lessons

- Parameterizing stride computation with num_kv_heads instead of num_heads is a zero-cost abstraction for K-cache layouts — when num_kv_heads == num_heads (MHA), the strides are identical, and the kernel code requires no changes since it already receives strides as runtime parameters
- Designing kernel interfaces that accept stride parameters (block_stride, head_stride) rather than computing them internally from num_heads decouples the kernel from the attention head topology, enabling GQA support without kernel modification
