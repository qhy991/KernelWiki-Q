## Challenge

Scalar stride loop processes one float (4 bytes) per iteration in a memory-bound scaling kernel, issuing separate 4-byte load and store instructions that underutilize available memory bandwidth and L2 cache lines


## Solution

Replace scalar stride loop with DefaultVecOp<4>-based vectorized loop using reinterpret_cast to vec_n_t<float,4> (16-byte aligned struct), enabling 128-bit wide load/store instructions that process 4 floats per iteration, with a scalar tail loop for remaining elements when hidden_size is not a multiple of 4

## Key Lessons

- Replace scalar stride loops with vec_n_t<T,4> reinterpret_cast in memory-bound kernels to issue 128-bit wide load/store instructions — each wide transaction fully consumes a 128-byte L2 cache line instead of loading it for 4 bytes, yielding significant bandwidth improvement at large hidden sizes
- Use a DefaultVecOp-style functor wrapper around reinterpret_cast vector types to keep the vectorized loop body generic and reusable — the scalar operation lambda is applied element-wise inside the unrolled loop, separating memory width from compute logic
- Always add a scalar tail loop after the vectorized loop (starting at num_vec * VEC_SIZE) to handle remaining elements when the problem size is not a multiple of the vector width — this ensures correctness without constraining hidden_size to be aligned
