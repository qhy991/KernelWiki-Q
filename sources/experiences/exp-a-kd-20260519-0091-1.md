## Challenge

CUTLASS SharedLoadIteratorMixed's add_pointer_offset uses pointers_ += offset which decays the array name to &pointers_[0], advancing only the first pointer while pointers_[1..kLoadsPerAccess-1] stay at their original positions, causing silently incorrect shared-memory load addresses


## Solution

Replace the array-decay offset application (pointers_ += offset, which only touches pointers_[0]) with a per-element loop using pointers_[i] += offset so every pointer lane in the multi-load iterator advances correctly after add_pointer_offset

## Key Lessons

- Using an array name (pointers_) in pointer arithmetic triggers array-to-pointer decay to &pointers_[0], so only the first element is modified — always index with pointers_[i] to advance each element independently
- Array-to-pointer decay bugs produce silently incorrect results rather than compiler errors or crashes — correctness validation with known data patterns is essential to catch them in shared-memory iterator code
- In CUTLASS multi-load iterators (SharedLoadIteratorMixed), all pointer lanes must advance in lockstep after add_pointer_offset; a single stale lane causes out-of-bounds or misaligned shared-memory reads that corrupt downstream GEMM results
