## Challenge

Scalar byte-by-byte memory access pattern (one uint8 load/store per thread per iteration) underutilizes available memory bandwidth for a 144 MB dataset, requiring nbytes threads each issuing individual 1-byte transactions


## Solution

Use uint4 (16-byte) vector loads/stores to process 16 bytes per thread per iteration, aiming to coalesce memory transactions into wider 128-byte accesses and reduce total load/store instruction count by 16x

## Key Lessons

- uint4 vector loads that require per-element decomposition via reinterpret_cast<uint8_t*> force the compiler to materialize all 16 bytes as individual registers plus LUT results, causing excessive register pressure that spills to local memory
- Reducing thread count by vector width (nbytes/16 vs nbytes threads) decreases occupancy and memory-level parallelism — for large bandwidth-bound workloads, many threads each doing simple scalar work often outperforms fewer threads doing complex vectorized work
