## Challenge

Online softmax output accumulator R_D stores intermediate O values in half precision, requiring repeated __half2float→__fmaf_rn→__float2half_rn conversion cycles per rescale step that cumulatively lose precision (~1e-3 error after multiple K/V tiles) and add unnecessary conversion overhead


## Solution

Store the output accumulator R_D in native FP32 (4 uint32_t registers per element via float* reinterpretation) so rescale FMA operates directly on float values without intermediate half rounding, preserving full FP32 precision across all online softmax rescale steps and deferring the sole half conversion to the final global memory write

## Key Lessons

- Use FP32 accumulator storage for online softmax O rescaling to eliminate cumulative half-precision rounding — applies when the kernel performs multiple rescale steps over K/V tiles and the register budget permits 4 regs per element instead of 2
- Conditionally enable FP32 storage based on head dimension heuristic (kHeadDim < 256) to avoid register-pressure-induced occupancy loss for large head dimensions while gaining precision for small ones
- In FlashAttention-style kernels, defer the sole half conversion to the final global memory write stage rather than converting back and forth at each intermediate rescale step — reduces both precision loss and conversion instruction overhead
