## Challenge

MoE routing finalization kernel performs k-way weighted reduction using scalar float loads (4 bytes per instruction), issuing 8*2048=16384 global load instructions per thread and becoming memory-bandwidth bound on the inner loop


## Solution

Reinterpret float pointers as uint4 (128-bit) to load 4 floats per instruction, reducing total load instructions from 8*2048=16384 to 8*512=4096 per thread and similarly vectorizing output stores from 2048 scalar stores to 512 16-byte stores, achieving much better memory bandwidth utilization

## Key Lessons

- Reinterpret float pointers as uint4 to load 4 floats per instruction in memory-bandwidth-bound reduction kernels — reduces load instruction count by 4x and better saturates HBM bandwidth, applicable whenever column count is a multiple of 4 and the kernel is load-heavy
- Vectorize both loads and stores together when converting a scalar reduction loop — only vectorizing one direction leaves the other as a bottleneck, negating much of the benefit
- Use reinterpret_cast<const uint4*>(float_ptr) + index-by-vector (vec_idx) instead of manual pointer arithmetic — the compiler emits properly aligned 128-bit LDG/STG instructions when the pointer type matches the access width
