## Challenge

Expert ID buffer filling is limited to only the first num_experts (256) threads, each iterating over its expert's block range with a stride-1 loop — the remaining 768 threads in a 1024-thread block sit idle during this phase, wasting available parallelism


## Solution

Replace the per-expert sequential fill with cooperative binary search: all 1024 threads iterate over the global block index space and binary-search the prefix sum array to find which expert owns each block — this engages all threads uniformly and eliminates load imbalance from experts with widely varying token counts

## Key Lessons

- When filling a mapping array where ownership ranges are given by a prefix sum, use cooperative binary search across all threads (O(log N) per element) instead of per-owner sequential loops — this engages all threads uniformly and handles skewed distributions without load imbalance
- Preferring block-strided iteration over expert-strided iteration avoids the need to partition threads by expert count, so thread participation does not depend on the data dimension
