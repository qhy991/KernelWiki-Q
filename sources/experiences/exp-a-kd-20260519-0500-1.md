## Challenge

FlashAttention-2 MMA kernel uses single-stage synchronous Q/K loading (kStage=1) where each inner-loop iteration must complete the full gmem-to-smem transfer via CP_ASYNC_WAIT_GROUP(0) and __syncthreads() before HMMA16816F32 MMA computation can begin, leaving the async copy engine idle during tensor-core computation and tensor cores stalled during global-memory transfers


## Solution

Switch to two-stage software pipelining (kStage=2) which activates the if constexpr(kStage>1) code paths: an initial pre-fetch loads (kStage-1) Q/K tiles before the main inner loop, then each iteration issues async CP_ASYNC_CG into the smem_sel_next double-buffer while MMA computes on smem_sel, with deferred CP_ASYNC_WAIT_GROUP(kStage-2) synchronization placed after MMA rather than before, overlapping gmem-to-smem transfers with tensor-core computation

## Key Lessons

- Software pipelining with kStage=2 double-buffers shared memory into smem_sel (current compute) and smem_sel_next (next load target), overlapping async gmem-to-smem CP_ASYNC_CG transfers with HMMA16816F32 MMA computation — applies when the kernel has a regular tiling loop with enough iterations (>= pipeline depth) to amortize fill/drain overhead
- Deferred synchronization (CP_ASYNC_WAIT_GROUP after MMA instead of before) is the core latency-hiding mechanism — the async copy engine transfers the next tile while tensor cores compute on the current one, reducing total stall cycles when loop trip count exceeds pipeline stages
- Pipeline depth trades shared memory footprint for latency hiding: kStage=2 doubles the Q/K smem requirement but delivers ~24% speedup at seq_len=256 (4 tiles) where steady-state overlap benefit exceeds one-time pipeline fill cost
