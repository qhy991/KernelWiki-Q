## Challenge

In groupwise CTA scheduling with 4 groups of 20 tiles each (not divisible by 8 CTAs), the naive approach uses a fixed blk_offset=blockIdx.x for every group, causing CTAs 0-3 to always receive 3 tiles per group while CTAs 4-7 receive only 2, producing a skewed work distribution [12,12,12,12,8,8,8,8] with stddev=2.0 that forces low-index CTAs to wait at synchronization barriers


## Solution

Rotate the starting CTA offset across groups by tracking the cumulative tile count: compute head_cta_id = group_total_tiles % gridDim.x after each group, then set blk_offset = (blockIdx.x + (gridDim.x - head_cta_id)) % gridDim.x so that remainder tiles from one group are assigned to different CTAs in the next group, achieving perfectly uniform [10,10,10,10,10,10,10,10] distribution with stddev=0.0

## Key Lessons

- When total_tiles per group is not divisible by gridDim.x, a fixed blockIdx.x starting offset causes the same CTAs to absorb all remainder tiles across groups — rotating the starting offset via cumulative tile count modulo distributes remainder work evenly across CTAs over multiple groups
- Flatten (global offset) scheduling over multiple groups can achieve perfect load balance even when each individual group has a non-divisible tile count, by treating the assignment rotation as a cross-group concern rather than a per-group concern
- Work imbalance at CTA granularity is a hidden performance killer in grouped GEMM/convolution workloads — measure per-CTA work counts and stddev to diagnose; a stddev of 0 after rotation confirms the fix
