## Challenge

Per-token group FP8 quantization kernel launches one CUDA block per group with only 16 threads (1 warp), resulting in poor SM occupancy since each block contributes just 1 warp — far below the 64-warp capacity per SM on A800 — leaving warp schedulators starved for work


## Solution

Bundle 16 groups into each block with 256 threads (8 warps) instead of 1 group with 16 threads, giving each SM 8x more warps to schedule and hide memory latency; use a 2D shared memory array s_absmax[16][17] with column padding to avoid bank conflicts across the 16 concurrent group reductions

## Key Lessons

- Bundling multiple independent work items (groups) into one block with more threads dramatically improves occupancy when the original per-block thread count is far below SM capacity — applies whenever each work item needs few threads but many items exist
- Pad 2D shared memory arrays to odd column strides (e.g., [16][17] instead of [16][16]) to prevent bank conflicts when different rows are accessed simultaneously by different thread sub-groups
- Reducing grid size (2048 → 128 blocks) by increasing per-block work amortizes kernel launch and block-scheduling overhead, complementing the occupancy improvement from higher thread count per block
