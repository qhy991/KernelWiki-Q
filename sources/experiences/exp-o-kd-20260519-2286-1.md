## Challenge

A separate transpose_kernel is launched to remap a 128x128 scale tensor (16384 floats) before a two-stage reduction, adding a full grid dispatch and a strided memory copy that dominate execution time when the downstream computation is lightweight


## Solution

Eliminate the unnecessary transpose kernel entirely by consuming the scale tensor in its original memory layout, removing one full kernel dispatch and a 16K-element intermediate buffer write

## Key Lessons

- Eliminate data-layout transforms (transpose/reorder) when the downstream consumer can operate on the original layout — each removed kernel dispatch saves launch overhead plus an entire buffer write, which dominates cost for lightweight computations
- When hardware features (e.g. Sm90 GMMA major-mode parameters) allow the consumer to specify its expected memory layout directly, the intermediate transpose becomes redundant and should be removed rather than optimized
