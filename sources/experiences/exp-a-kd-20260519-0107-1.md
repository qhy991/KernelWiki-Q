## Solution

Replace GemmIdentityThreadblockSwizzle with ThreadblockSwizzleStreamK to split K-dimension work across multiple threadblocks mapped to different SMs, increasing occupancy when output tile count is below SM count; partial products from K-subdivisions are accumulated in shared memory via a fused epilogue before writing the final output

## Key Lessons

- When the number of output tiles (M/tileM × N/tileN) is smaller than the GPU's SM count, basic data-parallel decomposition leaves SMs idle — Stream-K splits K-dimension work across additional threadblocks to fill more SMs, improving occupancy
- Stream-K accumulates partial products from K-subdivisions in shared memory via a fused epilogue, so correctness is preserved without extra global memory round-trips for partial sums
- Pass avail_sms = -1 to GemmUniversal::Arguments when using ThreadblockSwizzleStreamK to let the runtime query and use all available SMs for the decomposition
