## Solution

Double-buffer shared memory (2x allocation) and attempt to overlap loading the next k-tile into an alternate buffer while computing on the current buffer, swapping buffers each iteration via a cur_buf toggle

## Key Lessons

- Double-buffering without async memory copies (cp.async) does not achieve true load-compute overlap — all threads execute the load loop then the compute loop serially within each iteration, so the latency hiding benefit is nullified
- Doubling shared memory per block (16KB to 32KB) can reduce SM occupancy and hurt performance when the workload is compute-bound, making double-buffering counterproductive for kernels where arithmetic intensity already hides memory latency
- Double-buffering at the thread-block level with scalar FP32 loops provides negligible benefit; the technique only pays off in CUTLASS-style warp-fragment pipelines where async shared-memory loads overlap with high-throughput tensor-core MMA operations
