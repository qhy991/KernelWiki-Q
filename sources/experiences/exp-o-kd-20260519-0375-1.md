## Challenge

Online softmax rescaling with FP16 accumulator storage requires repeated half→float→fmaf→float→half round-trips per tiling step, losing ~3-4 decimal digits of precision per step and accumulating errors across all Tc tile steps


## Solution

Enable FP32 output accumulator storage (kOStorageAccFloat32=1) for head_dim<256 so R_D holds float values directly, eliminating intermediate half conversions and performing __fmaf_rn on FP32 accumulators with only a single __half2float for the incoming R_O contribution

## Key Lessons

- Use FP32 accumulator storage for online softmax rescaling in flash attention when head_dim<256 — the doubled register pressure (4 vs 2 uint32_t per element) remains within occupancy limits and eliminates ~512 half→float→fmaf→float→half conversion instructions per thread across all tile steps
- Gate FP32 accumulator behind a compile-time constexpr (e.g., if constexpr kOStorageAccFloat32) and select based on head dimension — fall back to FP16 storage at head_dim=256 to avoid register spillage from excessive register pressure
