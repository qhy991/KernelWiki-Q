## Solution

Cache each fp32 input element in a per-thread register array (xi_cache[kElemsPerThread]) during Pass 1 and reuse those registers in Pass 2, eliminating the second global-memory load entirely — for kDim=4096 with 512 threads each thread stores 8 fp32 values (32 bytes), well within register capacity

## Key Lessons

- Cache input elements in per-thread register arrays when the same data is consumed in multiple passes over a kernel — eliminates redundant global-memory loads and their L2 latency, applicable whenever elems-per-thread × sizeof(element) fits within the per-thread register budget (e.g., 8 × 4B = 32B here is trivial)
- The speedup from register caching is modest (~5%) when the L2 cache already partially absorbs redundant reads for medium-sized hidden dimensions, but the technique still avoids L2 hit latency and cache-line contention across many concurrent CTAs, making it more impactful at larger scales or when L2 pressure is high
- Use #pragma unroll on register-cached loops to ensure the compiler keeps the cache array in registers rather than spilling to local memory — without it the compiler may not realize the array is fully indexed at compile time
