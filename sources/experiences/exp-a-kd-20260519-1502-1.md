## Challenge

RMS norm kernel uses a custom warp-shuffle blockReduceSum that is tightly tailored to the exact active thread count (512) via nextPow2 cross-warp reduction, and is being replaced by cub::BlockReduce<float,1024> with partial-block numValid=512


## Solution

Replace custom blockReduceSum with cub::BlockReduce<float,1024> passing blockDim.x as numValid for partial-block reduction, allocating TempStorage sized for the full 1024-thread template parameter but only actively reducing 512 threads

## Key Lessons

- cub::BlockReduce partial-block path (numValid < BLOCK_THREADS) introduces bookkeeping overhead for handling inactive threads that can regress performance vs a custom warp-shuffle reduction tailored to the exact active-thread count — prefer custom reductions when thread count follows a simple nextPow2 pattern
- CUB TempStorage is sized for the full BLOCK_THREADS template parameter (1024) even when only blockDim.x (512) threads participate, wasting shared memory and potentially increasing bank-conflict pressure compared to a custom reduction that allocates only maxActiveLanes=16 shared slots
- Replacing a targeted micro-optimization with a generic library call does not automatically improve performance — verify with benchmarks, especially when the custom implementation is already well-tuned for the specific workload dimensions
