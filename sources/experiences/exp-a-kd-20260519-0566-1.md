## Challenge

Shared memory s_a[BM][BK] is row-major, so threads in a warp reading s_a[ty*TM+m][k] share the same column index k with stride BK, causing multiple threads to map to the same bank — severe bank conflicts stall shared memory loads in the compute loop


## Solution

Transpose shared memory A from s_a[BM][BK] to s_a[BK][BM+OFFSET] so consecutive threads along the BM dimension get stride-1 access mapping to different banks; OFFSET padding breaks residual stride-aligned conflicts at write time

## Key Lessons

- Transpose shared memory from [M][K] to [K][M] layout when threads iterate over the K dimension in the outer loop — consecutive threads then stride along the M dimension (stride-1), accessing different banks and eliminating bank conflicts
- Add OFFSET padding (e.g., +1 column) to shared memory arrays when the column count is a power of 2 or a multiple of the bank count, to prevent residual bank conflicts from stride alignment
- Stage global memory loads into registers before writing to shared memory when the store pattern differs from the load pattern (e.g., transposed layout) — this decouples the load width from the smem layout
