## Challenge

Scalar per-element loop in INT8 AZP quantization kernel performs one float load and one int8 store per iteration with division-based scaling, limiting memory throughput for the memory-bound quantization workload at hidden_size=4096 with block_size=1024


## Solution

Replace scalar for-loop with vectorize_with_alignment<16> for coalesced 16-element vector loads/stores and replace division (val/scale) with reciprocal multiplication (val*inv_s), but block size reduced from 1024 to 256 threads, causing severe occupancy loss that outweighs vectorization gains in this memory-bound kernel

## Key Lessons

- Reducing block size (1024→256) to accommodate vectorized access can cause catastrophic occupancy loss in memory-bound kernels — the vectorization throughput gain (16x wider loads) is overwhelmed by 4x fewer concurrent threads per SM
- Reciprocal multiplication (val*inv_s vs val/scale) reduces arithmetic latency but is irrelevant when the kernel is memory-bound — profile before optimizing the compute path
- For memory-bound quantization kernels, occupancy (active warps per SM) is the dominant performance factor; vectorize_with_alignment helps only when total thread concurrency is preserved or increased
