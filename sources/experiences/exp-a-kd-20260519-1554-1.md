## Challenge

In the MoE block-size alignment kernel, all 64 threads first initialize sorted_token_ids by writing to 8192 global-memory positions sequentially (128 iterations per thread), then proceed to shared-memory counting and prefix-sum, making init and count phases serial and underutilizing the GPU during each phase


## Solution

Partition the single block's threads into 256 fill threads (stride=256, only 32 iterations each for init) and 64 working threads that immediately begin shared-memory counting, overlapping global-memory initialization writes with counting work via concurrent execution within the same block synchronized by __syncthreads barriers

## Key Lessons

- Partition a single block's threads into dedicated sub-groups for independent work phases (e.g., global-memory init vs shared-memory counting) to overlap memory-bound and compute-bound phases within one kernel launch — applies when phases have no data dependency and one phase is global-memory heavy while the other uses shared memory
- Increase the thread count for global-memory initialization (256 fill threads with stride=256 vs 64 threads with stride=64) to saturate memory bandwidth more effectively and reduce per-thread iteration count (32 vs 128 iterations for 8192 elements)
