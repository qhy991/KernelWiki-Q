## Challenge

MoE expert GEMM dispatch selects sequential per-expert cuBLAS calls based solely on K > 128, issuing 8 independent kernel launches per iteration even on GPUs with abundant SMs that could execute all expert GEMMs concurrently in a single batched launch


## Solution

Add hardware-aware guard (sm_count == 78 && K > 128) so that on GPUs with many SMs (e.g., A800 with 108 SMs), the batched cublasGemmStridedBatchedEx path is always selected, consolidating 8 expert GEMMs into a single kernel launch that distributes work across all available SMs simultaneously

## Key Lessons

- Use hardware-aware dispatch (SM count + problem dimensions) rather than dimension-only heuristics for GEMM schedule selection — GPUs with abundant SMs can absorb batched expert workloads that would starve SM-constrained GPUs
- cublasGemmStridedBatchedEx consolidates multiple independent GEMMs into a single kernel launch, eliminating per-expert launch overhead and enabling better SM occupancy across the full chip — prefer it over sequential cublasGemmEx loops when the GPU has enough SMs to service all experts concurrently
- MoE dispatch should treat SM count as a first-class scheduling parameter: on SM-rich GPUs (H100/H200/A800 with 108+ SMs) always prefer batched mode, while SM-sparse GPUs (H20 with 78 SMs) may still benefit from sequential launches for large-K problems
