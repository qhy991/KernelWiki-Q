## Challenge

Zero-point constants z1 and z16 in 4-bit dequant are constructed via expensive float subtraction and division (__float2half_rn(-1024.0f - (float)zero), __float2half_rn(-1024.0f / 16.0f - (float)zero)), each requiring a full float-to-half rounding conversion pipeline, then broadcast to half2 via __halves2half2 which combines two identical arguments


## Solution

Replace float-arithmetic constant construction with IEEE 754 bit-level manipulation (half_uint16(0xe400 | zero) directly ORs the runtime zero into the known bit pattern for -1024.0f) and half-native arithmetic (__hsub(__int2half_rn(-64), __int2half_rn(zero))), then use __half2half2 broadcast intrinsic instead of __halves2half2, eliminating expensive float conversion and reducing instruction count

## Key Lessons

- Use IEEE 754 bit-level manipulation (integer OR into known bit patterns via half_uint16) instead of __float2half_rn for half-precision constant construction when the base value has a known representation — eliminates float subtraction/division and the full float-to-half conversion pipeline
- Prefer __half2half2(val) over __halves2half2(x, x) for broadcasting a single half into both lanes of half2 — it is a dedicated broadcast intrinsic with clearer intent and potentially fewer instructions
- Replace __float2half_rn(float_constant + variable) patterns with half-native arithmetic (__int2half_rn + __hsub) to avoid the expensive float-to-half conversion pipeline for runtime-dependent constants
