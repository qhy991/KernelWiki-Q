## Challenge

RMS normalization kernel launches with 1024 threads per block computed as min(8192/8, 1024), which severely limits occupancy because only a small number of 1024-thread blocks can reside per SM due to register and shared memory pressure, starving the memory pipeline of this bandwidth-bound kernel


## Solution

Replace the hardcoded 1024-thread cap with an adaptive 256-thread cap and compute VEC_SIZE dynamically via std::gcd(16/sizeof(scalar_t), hidden_size), reducing block size from 1024 to 256 threads so that 4× more blocks can occupy each SM simultaneously, dramatically improving memory latency hiding for this bandwidth-bound kernel

## Key Lessons

- Cap thread block size to 256 (not 1024) for bandwidth-bound kernels processing large token counts — smaller blocks allow 4× more concurrent resident blocks per SM, dramatically improving memory latency hiding (1.86× speedup observed on rms_norm with 256 tokens)
- Compute vectorized loop width dynamically via std::gcd(memory_alignment / sizeof(T), hidden_size) rather than hardcoding VEC_SIZE, ensuring the template unroll factor adapts correctly across different hidden sizes while preserving the compile-time constant optimization
