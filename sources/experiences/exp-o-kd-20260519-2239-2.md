## Challenge

Pipeline depth is hardcoded to 4 stages via a global constexpr constant, causing all shared memory buffers (A-matrix, B-matrix, scales) to be allocated at 4x the single-stage size regardless of the target GPU's shared memory capacity


## Solution

Replace the hardcoded global pipe_stages=4 constant with a runtime stages parameter passed through get_scales_cache_size and get_kernel_cache_size, allowing callers to select stages=2 on memory-constrained architectures like Turing to halve the pipeline buffer footprint

## Key Lessons

- Replace hardcoded pipeline depth constants with runtime parameters so that memory-constrained GPUs (e.g., Turing 64KB shared) can use fewer pipeline stages (2 vs 4), trading overlap for occupancy — the shared memory savings unlock larger tile configurations that more than compensate for reduced pipeline depth
- Global constexpr constants for architecture-sensitive parameters like pipeline stages should be converted to function parameters when the code must run across GPU generations with different shared memory budgets
