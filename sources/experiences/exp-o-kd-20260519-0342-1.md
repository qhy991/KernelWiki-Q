## Solution

Replace the misaligned stride_last=1 with stride_heads=kHeadDim (=64) so the alignment check correctly evaluates to true, enabling the float4 vectorized copy kernel that processes 4 elements per load/store instead of 1

## Key Lessons

- When dispatching between vectorized and scalar memory kernels, verify the alignment check uses the actual data layout stride (e.g., head_dim) rather than an irrelevant dimension (e.g., stride_last=1) — a wrong stride variable silently forces the slow scalar path even when memory is properly aligned
- float4 vectorized loads/stores process 4×128-bit per instruction vs scalar 32-bit, reducing total memory instructions by 4× for contiguous aligned data — always prefer vectorized access when alignment guarantees hold
