## Challenge

Per-token group FP8 quantization kernel uses unpadded shared memory with stride 16 (s_absmax[16][16]), which is suspected to cause bank conflicts during the WarpReduce phase when multiple groups access shared memory simultaneously


## Solution

Pad shared memory stride from 16 to 17 (s_absmax[16][17]) to prevent potential bank conflicts, but analysis reveals the warp-group mapping already partitions threads 0-15 to banks 0-15 and threads 16-31 to banks 16-31, so the padding is redundant and yields no measurable improvement

## Key Lessons

- Mechanical shared memory padding (stride N+1) is not always beneficial — when thread-to-bank mapping already avoids conflicts due to natural warp partitioning (16 threads per group spanning exactly half the 32 banks), padding adds memory overhead without reducing conflicts
- Before applying shared memory padding, verify the actual bank conflict pattern by analyzing which warp lanes access which banks under the specific thread-to-data mapping — the general rule of stride-16 conflict applies only when different rows' same-column elements are accessed by threads in the same half-warp
- Shared memory padding increases per-block SMEM allocation (1088 vs 1024 bytes here), which can reduce occupancy when the kernel is near the shared memory limit — always validate with profiling, not just the heuristic
