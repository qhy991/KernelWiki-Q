## Challenge

Custom block-wide max reduction uses shared-memory sequential halving with __syncthreads() barriers at each of 10 reduction rounds for a 1024-thread block, incurring high synchronization overhead and shared-memory access latency


## Solution

Replace custom shared-memory blockReduceMax with CUB BlockReduce, which uses warp shuffle instructions (__shfl_down_sync) for intra-warp communication, reducing to only 1 inter-warp shared-memory step and eliminating most __syncthreads() barriers

## Key Lessons

- CUB BlockReduce replaces shared-memory sequential halving with warp shuffle instructions for intra-warp communication → lower latency because shuffles bypass shared memory loads/stores and do not require __syncthreads() within a warp → applies to any block-level reduction (sum, max, min) where the custom implementation uses shared memory and __syncthreads() rounds
- For large blocks (1024 threads), custom shared-memory sequential halving requires 10 rounds with __syncthreads() barriers each, while CUB performs 5 warp-level shuffle reductions plus 1 inter-warp shared-memory step → significant reduction in barrier synchronization overhead → applies when block size is large and reduction is a hot path
