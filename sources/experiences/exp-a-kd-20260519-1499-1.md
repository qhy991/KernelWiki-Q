## Challenge

Replacing a targeted custom block-wide max reduction (vllm::blockReduceMax using warp shuffle + 128-byte shared memory for inter-warp phase) with CUB's BlockReduce<float,1024> over-allocates TempStorage sized for 1024 threads when only 256 threads are launched at runtime, increasing shared memory footprint per block and reducing SM occupancy


## Solution

The custom vllm::blockReduceMax allocates only maxActiveLanes=32 floats (128 bytes) in shared memory for the inter-warp phase regardless of actual block size, while CUB BlockReduce<float,1024> allocates TempStorage padded for 1024 threads with additional intermediate buffers and executes reduction phases designed for up to 32 warps, causing unnecessary synchronization and branch overhead when only 8 warps are active

## Key Lessons

- CUB BlockReduce's compile-time BLOCK_DIMX template parameter determines TempStorage allocation size — passing a larger value (e.g., 1024) when the runtime block dimension is smaller (e.g., 256) wastes shared memory and reduces SM occupancy; match the template parameter to the actual block size or use the custom approach for sub-maximal blocks
- Custom block reductions that allocate shared memory proportional to the actual number of warps (maxActiveLanes) can outperform generic library reductions when the block size is significantly smaller than the library's compile-time block dimension
