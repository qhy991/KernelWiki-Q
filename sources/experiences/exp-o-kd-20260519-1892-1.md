## Challenge

The SM90 int8 quantized GEMM path uses a single hardcoded configuration (TileShape=_128x128x128, ClusterShape=_1x2x1, KernelTmaWarpSpecializedPingpong) for all problem sizes, causing severe compute waste for small M dimensions (1-64 rows) where large tiles produce mostly padding rows and Pingpong's two-wave pipelining overhead cannot be amortized over few tiles


## Solution

Replace the single config with five M/N-aware config structs (default, M128, M64, M32_NBig, M32_NSmall) using progressively smaller tile shapes for smaller M ranges, switching from Pingpong to WarpSpecialized schedule when M<=64 to avoid two-wave overhead, and splitting M<=32 further by an N<8192 threshold to select cluster shapes that match column parallelism

## Key Lessons

- Use M/N-aware tile shape dispatch instead of a single hardcoded GEMM config — small M dimensions (<=64) benefit dramatically from smaller tiles (TILE_M 128->32 or 64) that reduce register pressure and wasted padding computation
- Switch from KernelTmaWarpSpecializedPingpong to KernelTmaWarpSpecialized schedule when M<=64 because the Pingpong two-wave pipelining overhead cannot be amortized over the few available tiles, making single-wave TMA more efficient
- Split configs by N dimension at low M (M<=32) to match cluster parallelism to problem width: wide problems (N>=8192) use fewer larger clusters (_1x4x1) while narrow problems (N<8192) use more smaller clusters (_1x8x1) to improve SM utilization
