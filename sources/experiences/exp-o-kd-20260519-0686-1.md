## Challenge

Per-token FP8 quantization kernel uses type-dependent vec_size=8 for fp16 input, issuing 8 scalar byte stores per loop iteration (8192 total store transactions per token), severely underutilizing available memory bandwidth in the output write phase


## Solution

Double vector width to VEC_SIZE=16 (fixed for all types) and replace 8 scalar byte stores with a single uint4 128-bit vector store using a union-based OutputStorage, reducing output transactions from 8192 to 512 per token while also widening input loads from 16 to 32 bytes per vector

## Key Lessons

- Replace scalar byte stores with consolidated uint4 128-bit vector stores when outputting quantized data — a union of uint8_t[16] and uint4 lets you fill bytes individually then issue one wide store, reducing transaction count by 16x for FP8 output
- Use a fixed VEC_SIZE tied to the output element type (16 bytes = 128 bits for uint8 quantized output) rather than type-dependent vector sizes derived from input type, since the store width determines the memory bottleneck
- Wider vector loads (vec_t<__half,16> at 32 bytes vs vec_t<__half,8> at 16 bytes) improve input bandwidth utilization as a secondary benefit when doubling the vector width for output store consolidation
