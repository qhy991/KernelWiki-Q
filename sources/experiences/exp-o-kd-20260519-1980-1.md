## Solution

Cooperatively load all 65 int64_t offsets (520 bytes) into shared memory via __ldg before the accumulation loop, then read from smem (~5 cycle latency) instead of global memory (~200+ cycle L2 latency) during the prefix-sum accumulation, eliminating redundant global traffic across loop iterations

## Key Lessons

- When a kernel loop repeatedly reads the same small read-only array (520 bytes for 65 int64_t offsets), cooperatively load it into shared memory once with __ldg and read from smem in the loop — smem latency (~5 cycles) is ~40x lower than L2 (~200+ cycles), yielding significant speedup for loops with many iterations
- Use __ldg (read-only cache) during the cooperative smem load to route accesses through the texture/L1 read-only path, avoiding contention with the L2 data cache used by other global memory traffic
- For small arrays that fit comfortably in shared memory (<1 KB), cooperative loading with a stride loop (i += blockDim.x) ensures all threads participate even if the thread count exceeds the element count
