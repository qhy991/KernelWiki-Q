## Challenge

FP8 KV-cache paged gather and dequantize use two separate kernels: a 1024-thread gather kernel copies 656 bytes per token into an intermediate buffer via a generic byte-copy loop, then a 576-thread dequantize kernel reads that buffer back for FP8→BF16 conversion, causing redundant memory traffic, an extra kernel launch, and underutilized threads in the gather step


## Solution

Fuse the paged gather and FP8→BF16 dequantize into a single kernel that reads FP8 data directly from paged cache blocks and writes converted BF16 output in-place, using exactly 576 threads to match per-token work and eliminating the intermediate buffer entirely

## Key Lessons

- Fuse paged-gather and dequantize into a single kernel when the gather output is only consumed by the dequantize step — eliminates intermediate buffer write-then-read (656 bytes per token in this case), removes one kernel launch, and improves cache locality by performing FP8 read and BF16 write in the same pass
- Match thread count to actual per-token work (576 threads for 512 dequant + 64 rope copy) rather than using a generic oversized block (1024 threads for byte-copy), avoiding wasted threads and improving SM occupancy
