## Challenge

Scalar swish activation kernel processes one float element per thread, issuing separate load and store instructions per element, which underutilizes memory bandwidth and incurs high thread scheduling overhead for large tensors


## Solution

Vectorize the kernel to process 4 float elements per thread using float4 loads and stores, reducing total memory instructions from 2N to N/2, cutting thread count by 4x, and ensuring 128-bit aligned coalesced global memory accesses

## Key Lessons

- Vectorize elementwise kernels with float4 loads/stores to halve memory instruction count and ensure 128-bit aligned coalesced access — applies to any memory-bound elementwise operation (activations, scale, bias add) on contiguous fp32 arrays
- Reducing thread count by the vectorization factor (4x here) cuts scheduling overhead and improves L2 cache hit rates because each thread fetches a wider contiguous chunk, reducing redundant fetches across neighboring threads
