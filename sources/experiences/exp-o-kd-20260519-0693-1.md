## Solution

Add at::cuda::CUDAGuard to pin the device context from the input tensor's device, obtain the current CUDA stream via at::cuda::getCurrentCUDAStream, and pass the explicit stream to both initialize() and run() calls so the CUTLASS kernel launches on the intended device and stream

## Key Lessons

- Always set an explicit CUDAGuard and pass a non-default stream when launching CUTLASS (or any) kernels in multi-GPU or multi-stream serving scenarios — the default stream serializes all launches and prevents concurrent overlap across streams
- CUDAGuard pins device context for the duration of the scope, ensuring CUTLASS internal operations target the correct GPU without requiring manual cudaSetDevice/cudaResetDevice pairs
