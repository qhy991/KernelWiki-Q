## Challenge

Shared memory sizing for the A-matrix in a quantized GEMM kernel hardcodes 2 bytes per element regardless of activation precision, over-allocating 16KB per tile for INT8 activations that only need 1 byte per element and wasting shared memory budget on Ampere's 164KB limit


## Solution

Add an is_a_8bit runtime flag and conditionally select 1 byte per A-matrix element for INT8 activations versus the default 2 bytes for FP16, halving the A-matrix shared memory footprint from 16KB to 8KB per tile configuration and freeing shared memory budget for larger tiles or additional pipeline stages

## Key Lessons

- Shared memory allocation for the A-matrix tile should match actual element size (1 byte for INT8, 2 bytes for FP16) rather than hardcoding the FP16 size — halving the footprint frees shared memory for larger tile configurations or more pipeline stages
- Parameterize pipeline stage count (stages) instead of using a global constant (pipe_stages) so the shared memory cost function can reason about different pipeline depths per kernel variant
- When shared memory budget is a binding constraint on tile size selection, even moderate per-tile savings (8KB) can unlock thread configurations that were previously over-budget, yielding measurable end-to-end speedup
