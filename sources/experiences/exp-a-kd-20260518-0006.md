## Challenge

Adding a local variable to materialize a 32-byte vector pack (vec_n_t<half,16>) before passing to a read-only VecOp was intended to force wide LD.global. However, NVCC -O3 with __forceinline__ already inlines the VecOp, sees all 16 elements are consumed, and generates wide loads without the explicit copy.


## Solution

The local copy pattern (vin_t src = v_in[i]) is context-dependent: it helps when NVCC scalarizes loads in specific calling contexts, but adds 32 bytes of unnecessary register-to-register move when the compiler already fuses loads through inlining. For vec16 with inline VecOp, the before version is better — let the compiler own the load scheduling.

## Results

- **speedup**: 0.915x (9.2% regression from unnecessary register copy)


## Key Lessons

- Materializing vector packs into local variables to force wide loads is context-dependent. When NVCC can trace through __forceinline__ callables and see all elements are consumed, it already emits wide LD.global. Adding an explicit local copy then becomes dead register traffic. Verify the actual PTX/SASS before applying this pattern universally.
- For wide packs (32 bytes), the compiler is already incentivized to use wide loads. The local-copy trick is more valuable for medium-width packs (8-16 bytes) where the compiler might scalarize, or when the callable cannot be inlined. At vec16 width under -O3 with inlining, trust the compiler.
