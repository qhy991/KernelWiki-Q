## Challenge

The mindiv host-side function uses truncating integer division with hardcoded sparse offset checks (indices 0,3,4..9), missing div2 values 1,2 and skipping i=1,2. This can produce a suboptimal WvPrGrp (wavefronts-per-group) launch parameter, wasting GPU resources with more wavefronts than necessary to achieve the same round count


## Solution

Replace sparse hardcoded division/check logic with a full-range loop using ceiling division ((N + nPrRnd - 1) / nPrRnd) and array-backed scan across all 13 candidate div2 values (0..12), then scan backward to find the minimum div2 preserving the same round count

## Key Lessons

- Reducing WvPrGrp (wavefronts per group) shrinks the grid-stride loop stride (CuCount*WvPrGrp), increasing the number of loop iterations per wavefront — this adds overhead when the kernel processes one row per stride, negating any benefit from fewer wavefronts
- Ceiling division vs truncating division in launch-parameter computation can change the WvPrGrp value even when the round count stays the same (both return 1 round), but the smaller WvPrGrp increases per-wavefront work on the 1-row-per-stride proxy path
- Parameter-tuning changes must be validated on the actual target workload: a host-side algorithm that regresses on a proxy kernel (1-row-per-stride) may still be correct for the real kernel (YTILE=4 rows per round) where fewer wavefronts improve CU balance without extra iterations
