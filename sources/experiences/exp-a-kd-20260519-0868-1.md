## Solution

Reduce per-block dynamic shared memory from 128KB to 32KB (kSmem = 8*1024*sizeof(uint32_t)), which is sufficient because each radix pass needs at most ~TopK=64 candidates in the threshold bin, and 4K entries per round (2 rounds = 8K entries = 32KB) covers the working set; this permits 3-4 blocks per SM, increasing occupancy from ~512 to ~1536-2048 concurrent threads per SM

## Key Lessons

- Reduce dynamic shared memory to the minimum required by the algorithm's actual working set — on GPUs with limited per-SM shared memory (e.g., 164KB on A800), oversized allocations cap occupancy at 1 block/SM; right-sizing enables 3-4x more concurrent blocks and threads
- Analyze shared memory requirements based on the algorithm's data-dependent upper bound (e.g., ~TopK candidates per radix round), not a generous over-allocation, to determine the tightest feasible SMEM budget without sacrificing correctness
