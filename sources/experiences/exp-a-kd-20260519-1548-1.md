## Challenge

Router GEMM kernel for Mixture-of-Experts loads bf16 matrix elements one at a time via scalar __bfloat162float conversion in a loop, issuing VPT separate LDG instructions per matrix access — with 4 tokens this produces 40 scalar loads per k-iteration (1 B + 4 A loads × 8 elements each), saturating memory bandwidth and stalling the warp scheduler


## Solution

Replace 8 scalar bf16 loads per matrix access with a single uint4 (128-bit) vectorized load via reinterpret_cast, then convert the 8 packed bf16 values to float in registers — cutting load instructions by 8x and enabling coalesced 128-bit memory transactions that reduce L2 cache contention

## Key Lessons

- Replacing scalar bf16 loads (one LDG per element) with uint4 vectorized loads (one 128-bit LDG per 8 elements) reduces memory transaction count by 8x — applies to any kernel where consecutive bf16 elements are read and memory bandwidth is the bottleneck
- Multi-token GEMM kernels amplify the benefit of vectorized loads proportionally to the number of tokens, because each additional token adds another full set of A-matrix loads per iteration, increasing memory pressure and making coalesced access more critical
- Prefer reinterpret_cast<uint4 const*>(ptr) to read 8 consecutive bf16 values in one instruction over element-wise __bfloat162float in a loop — the conversion to float can still proceed element-wise in registers after the single wide load
