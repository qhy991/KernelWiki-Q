## Challenge

Per-tensor FP8 quantization kernel uses a software FP8 conversion wrapper (FP8_TYPE_WRAPPER) that compiles to ~5-6 efficient inline integer instructions per element via fp8e4m3fn_from_fp32_value, which NVCC -O3 can schedule tightly within the unrolled conversion loop


## Solution

Replaced FP8_TYPE_WRAPPER with __nv_fp8_e4m3 native CUDA type expecting hardware cvt.rn.satfinite.e4m3x2.f32 instructions, but on SM 8.0 (Ampere/A800) where no native FP8 conversion exists, the native type invokes a less optimized runtime emulation routine not tightly integrated into the kernel's instruction scheduling, causing a 20% regression in per-element conversion overhead

## Key Lessons

- On SM 8.0 (Ampere) GPUs, __nv_fp8_e4m3 lacks native hardware FP8 conversion support and its runtime emulation can be slower than a carefully crafted inline integer conversion sequence — prefer software FP8 emulation on pre-Hopper architectures where NVCC can optimize the inline path more tightly
- Native CUDA data types like __nv_fp8_e4m3 may silently fall back to runtime emulation on hardware lacking the corresponding instructions, so always verify target SM capability before replacing a proven software implementation with a 'native' type
