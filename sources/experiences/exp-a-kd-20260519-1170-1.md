## Challenge

Tiled int8 GEMM uses TILE_M=128 but M=64, so half the tile rows are padding that compute discarded results. Each thread holds RPT×CPT=8×8=64 int32 accumulators (~80 registers), limiting SM occupancy to roughly 3 concurrent blocks due to register pressure.


## Solution

Reduce TILE_M from 128 to 64 to exactly match the problem dimension M=64, eliminating all padding rows. This drops each thread's accumulator count from 64 to 32 int32 values (~52 registers), reduces shared memory from 8800 to 6496 bytes per block, and raises occupancy from ~3 to ~4-5 concurrent blocks per SM for better latency hiding.

## Key Lessons

- Match tile dimensions to actual problem sizes to avoid computing padding rows — oversized tiles waste registers, shared memory, and computation on discarded results, all of which reduce occupancy
- Reducing per-thread accumulator count (via smaller tiles) lowers register pressure, which is often the primary occupancy limiter for compute-heavy GEMM kernels — fewer registers per thread enables more concurrent blocks per SM for latency hiding
- Shared memory footprint scales with tile size; a smaller tile that fits the problem exactly reduces smem per block, which can also increase the number of resident blocks when smem is the occupancy bottleneck
