## Solution

Widen per-thread memory access from half2 (64-bit) to LDST128BITS/float4 (128-bit) loads, processing 8 half elements per thread instead of 2, and use __hmul2 for packed half2 multiply pairs to increase arithmetic intensity per load — however the resulting 32-thread block (1 warp) severely cuts occupancy, yielding only a marginal 2.83% improvement within noise

## Key Lessons

- Widening per-thread vector loads (half2→LDST128BITS) increases memory throughput per instruction but reduces thread count and occupancy proportionally — for memory-bound kernels where total bytes transferred is unchanged, the net speedup is marginal and may not justify the complexity
- A single-warp block (32 threads) eliminates cross-warp shared-memory reduction stages but leaves the warp scheduler with no other warps to interleave, removing the primary mechanism for hiding memory latency — wider loads alone cannot compensate for this loss
- LDST128BITS via reinterpret_cast<float4*> is a portable zero-overhead idiom for issuing 128-bit (16-byte) loads on GPUs that support it, coalescing 8 half elements into a single memory transaction when threads access consecutive addresses
