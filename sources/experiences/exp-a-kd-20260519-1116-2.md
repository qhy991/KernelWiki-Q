## Challenge

The quant_kernel loop body recomputes factor, numCols_padded, and numCols_SFout on every iteration despite these values depending only on the kernel-invariant numCols parameter, wasting arithmetic throughput and preventing the compiler from proving loop-invariance in complex index expressions


## Solution

Precompute int32_t const numKTiles = (numCols + 63) / 64 at kernel scope outside the loop, then pass it directly to get_sf_out_offset and use it for SFout_in_expert offset computation, eliminating three per-iteration arithmetic operations

## Key Lessons

- Hoisting loop-invariant computations (values depending only on kernel arguments, not loop indices) to kernel-scope const variables is a code clarity improvement, but nvcc -O3 already performs this LICM optimization automatically — expect neutral performance impact
- Declaring hoisted values as const communicates intent to both the compiler and human readers, making the kernel's data flow easier to audit without changing generated code
