## Solution

Increase kFragmentsPerIteration from 1 to 2, halving sync barriers from 16 to 8 by processing two fragments per iteration, but this doubles shared memory per block to 68KB, which on sm_80 (164KB/SM) reduces blocks per SM from 4 to 2 — the occupancy loss dominates the barrier reduction, causing a 3.06x slowdown

## Key Lessons

- Increasing per-block shared memory to reduce __syncthreads() barrier count can regress performance when the memory increase cuts SM occupancy — on sm_80 with 164KB/SM, going from 34KB to 68KB per block halves blocks-per-SM from 4 to 2, and the lost latency-hiding outweighs the barrier savings
- Compute the occupancy breakpoint before tuning smem-batched iteration counts: the shared memory budget per SM divided by per-block smem must allow at least as many resident blocks as the warp scheduler needs to hide arithmetic and memory latency
- In a lightweight epilogue kernel where per-element work is trivial (one smem read + one gmem write), occupancy is the dominant performance lever — sync barrier reduction is secondary because each barrier interval has too little computation to amortize the stall
