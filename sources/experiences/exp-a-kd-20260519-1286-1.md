## Challenge

4-bit weight dequantization processes 8 scalar half elements per group in an 8-iteration loop, converting each quantized value to float for zero-point subtraction and back to half, incurring redundant __float2half/__half2float round-trips per element


## Solution

Pack quantized value pairs into half2 and use __hsub2 for vectorized zero-point subtraction via a FragZP struct of 4 broadcast half2 values, halving loop iterations from 8 to 4 — but the pack/unpack overhead (FragZP construction, __halves2half2, __low2half/__high2half stores) negates the theoretical benefit in this microbenchmark

## Key Lessons

- Packed half2 operations (__hsub2) only yield speedups when data is already in half2 layout or the pack cost is amortized over long vectors — when values must be extracted individually from packed int32 bitfields and stored as scalar half, the pack/unpack overhead cancels the 2x instruction benefit
- FragZP-style zero-point fragments are profitable in real Marlin kernels because MMA output is already in half2 layout matching the fragment, avoiding the pack/unpack overhead that appears in an isolated dequant microbenchmark
- Short loops (8 elements per group with 256 threads, so most threads execute 0–1 iterations) cannot amortize vectorization setup cost — half2 vectorization needs sufficient per-thread work to justify the fragment construction and lane extraction overhead
