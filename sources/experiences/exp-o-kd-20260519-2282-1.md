## Challenge

When feat_in == feat_out, the bgmv_kernel dispatch condition (feat_in < feat_out) routes the equal-dimension case into the shrink kernel branch, which uses a multi-stage async memcpy pipeline with shared memory buffers, warp-level cross-reduction, and staged load/compute/release cycles — all unnecessary overhead when no dimension reduction occurs


## Solution

Change the dispatch condition from feat_in < feat_out to feat_in <= feat_out so that the equal-dimension case routes to the expand kernel (bgmv_expand_kernel), which uses direct global-memory vector loads and cooperative_groups::thread_block_tile reduction with shfl_down, avoiding the shrink kernel's async pipeline, shared-memory buffering, and cross-warp synchronization

## Key Lessons

- When kernel dispatch splits on a dimension comparison, include the equal-dimension case in the branch that avoids unnecessary reduction/pipeline machinery — the expand (no-shrink) path is cheaper when no actual dimensional transformation occurs
- Async memcpy pipelines and shared-memory staging in shrink kernels carry fixed overhead (buffer allocation, barrier synchronization, cross-warp reduction) that is wasted when feat_in == feat_out because no data reduction is actually needed
