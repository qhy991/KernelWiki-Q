## Solution

The after kernel assigns one thread per head-block (num_heads*(head_size/x)=64 threads) and uses vectorize_with_alignment<4> for 128-bit vectorized key stores plus an unrolled loop for value writes, but the 8x reduction in threads per block starves the SM of concurrent warps, causing a 1.26x slowdown (0.796x speedup)

## Key Lessons

- Reducing per-block thread count to enable vectorized stores can regress performance when the thread count drops below the occupancy threshold needed to hide memory latency — vectorized 128-bit transactions per thread cannot compensate for 8x fewer concurrent warps per SM
- Per-head-block thread assignment (num_heads*(head_size/x)) produces too few threads for small batch sizes where only one block resides per SM; this pattern only benefits large-batch scenarios where multiple blocks fill each SM's concurrent slots
- When restructuring a strided loop into a per-element thread assignment, verify that the new thread count (blockDim.x) still provides sufficient occupancy (typically >= 128 threads/block for memory-bound kernels) before assuming vectorization will help
