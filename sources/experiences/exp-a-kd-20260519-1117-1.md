## Challenge

The get_sf_out_offset device function uses expensive integer division/modulo for index decomposition, computes six intermediate stride variables with multiply-add chains, and redundantly recalculates numKTiles from numCols on every invocation despite it being loop-invariant across the kernel


## Solution

Replace all division/modulo with bitwise shift/AND (valid for power-of-2 divisors), collapse six stride variables into a single shift-OR offset expression, hoist numKTiles precomputation to kernel scope as a const, and add __forceinline__ to the device function to eliminate call overhead

## Key Lessons

- NVCC -O3 automatically lowers division/modulo by power-of-2 constants to shift/AND sequences and hoists loop-invariant expressions, so manual bitwise replacement produces zero speedup when the divisor is a compile-time power-of-2 — only beneficial when the divisor is a runtime variable or when compiling at -O0
- Hoisting loop-invariant precomputation (numKTiles from numCols) and collapsing stride variables into a single shift-OR expression improves readability and reduces register pressure in the source, but NVCC's optimizer already performs these transformations at -O3 — verify with profiling before assuming manual refactoring yields latency gains
