## Challenge

Alignment check uses stride_last=1 (innermost dimension stride) instead of the actual head dimension stride, so the contiguous K=64 layout always evaluates isAligned=false and falls back to scalar element-wise copies instead of the vectorized float4 path


## Solution

Switch the alignment variable from stride_last (always 1) to stride_heads=kHeadDim (64), so the contiguous memory layout correctly satisfies the 16-byte alignment check and triggers the vectorized float4 copy kernel which issues 4x fewer load/store instructions

## Key Lessons

- When dispatching between scalar and vectorized kernels via an alignment check, verify the stride variable reflects the actual memory layout dimension (e.g., head dimension stride) rather than an unrelated dimension — a wrong stride silently forces the slow path even when data is perfectly aligned
- float4 vectorized loads/stores reduce instruction count by 4x for contiguous fp32 arrays that satisfy 16-byte alignment — always confirm alignment at the correct stride level before falling back to scalar access
