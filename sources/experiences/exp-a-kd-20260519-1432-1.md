## Solution

The after code computes an adaptive max_block_size=(num_tokens<256)?1024:256 and derives block_size from hidden_size/vec_size capped by max_block_size, aiming to launch more smaller blocks for higher concurrency on large batches, but the kernel code itself is unchanged

## Key Lessons

- When grid_dim (num_tokens) << num_SMs, all blocks run concurrently regardless of block size — reducing threads per block cannot increase concurrency because no SM needs to multiplex more than one block
- Adaptive block-size heuristics that switch between 1024 and 256 threads per block only benefit scenarios where grid_dim exceeds num_SMs and smaller blocks allow more concurrent blocks per SM
