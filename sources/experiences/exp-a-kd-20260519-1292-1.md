## Challenge

Int4 nibble remapping uses synchronous cudaMemcpy round-trips (DeviceToHost + HostToDevice) for CPU-side serial per-byte switch-case processing, incurring ~2.4ms latency dominated by synchronization overhead rather than actual data transfer


## Solution

Replace CPU host-side round-trip with a GPU kernel that uses a 256-byte constant-memory lookup table for nibble remapping, combined with uint4 vectorized loads (16 bytes per thread) and a grid-stride loop, eliminating all host-device synchronization boundaries

## Key Lessons

- Eliminating GPU↔Host round-trips by moving computation to GPU with constant-memory LUT yields massive speedups (~140x) even for small data sizes, because synchronous cudaMemcpy synchronization overhead dominates over actual data transfer time
- Constant-memory LUT for byte-to-byte transformation (256 entries) eliminates per-element branching on GPU — constant cache provides low-latency lookup for small tables that fit entirely in the 64 KB constant cache
- uint4 vectorized loads (16 bytes per thread) with grid-stride loops provide efficient memory coalescing for elementwise byte transformations, processing the full 16-byte load in registers before writing back
