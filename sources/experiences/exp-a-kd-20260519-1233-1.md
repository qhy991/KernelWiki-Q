## Challenge

Grouped GEMM scheduling uses a hardcoded ASSUMED_SM_COUNT (84) to compute block count and groups-per-block, causing mismatch on devices with more SMs (e.g., A800 with 108 SMs) — fewer blocks than available SMs means less parallelism and more sequential GEMMs per block


## Solution

Replace hardcoded SM count with runtime query via cudaDeviceGetAttribute(cudaDevAttrMultiProcessorCount) so that block count and groups-per-block are computed from the actual device's SM count, enabling full SM utilization and fewer sequential GEMMs per block

## Key Lessons

- Hardcoding SM count for grouped/batched kernel launch configurations causes under-utilization on devices with more SMs — always query cudaDevAttrMultiProcessorCount at runtime to adapt block count to the actual hardware
- In grouped GEMM scheduling, reducing groups_per_block by increasing block count (more parallel blocks) directly reduces per-block sequential work and improves latency, especially when the total work count far exceeds the block count
