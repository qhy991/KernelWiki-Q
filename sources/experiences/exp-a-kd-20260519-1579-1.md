## Solution

Introduce a separate cache_t template parameter and an is_fp8_e5m2_kv_cache compile-time flag so the kernel stores FP16 key/value into FP8-E5M2 (1-byte) cache via __nv_cvt_halfraw_to_fp8, halving write bandwidth; if constexpr ensures the non-FP8 path compiles identically to the original

## Key Lessons

- For write-bandwidth-bound kernels storing KV cache, quantizing cache elements from FP16 to FP8-E5M2 halves write traffic with minimal compute overhead from __nv_cvt_halfraw_to_fp8 — applies whenever the downstream attention computation can tolerate FP8 precision loss in cached keys/values
- Use if constexpr with a boolean template flag to gate precision-conversion paths so the non-conversion path compiles identically to the original, making the optimization zero-cost when disabled
- Separate the cache storage type (cache_t) from the computation type (scalar_t) via template parameters — this decouples the arithmetic precision from the storage precision, enabling FP8 caching without changing the input/output data types
