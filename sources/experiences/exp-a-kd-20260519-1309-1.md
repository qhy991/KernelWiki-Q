## Challenge

Row expansion kernel copies 4096 half elements per row using scalar 2-byte load/store instructions in a stride loop, issuing 16 separate memory transactions per thread and poorly utilizing the 32-byte memory transaction granularity on A800 (sm_80)


## Solution

Replace scalar half-element copy loop with 128-bit vectorized uint4 load/store, transferring 8 half values per instruction; this reduces iterations from 16 to 2 per thread and allows the hardware to coalesce 16-byte accesses into full 32-byte L2/DRAM transactions, dramatically cutting loop overhead and memory instruction count

## Key Lessons

- Replace scalar element-wise copy loops with uint4 (128-bit) vectorized load/store to reduce loop iterations by 8x for half-precision data — this cuts address computation, branch overhead, and memory instruction count while enabling hardware coalescing into full 32-byte transactions
- For half-precision row-copy kernels, 128-bit vectorization is especially effective because 8 half values (16 bytes) fit in one uint4 and align well with GPU memory transaction granularity, yielding larger speedups than the equivalent float optimization
