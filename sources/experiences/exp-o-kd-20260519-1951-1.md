## Solution

Cap the number of launched thread blocks at 36 (BLOCK_LIMIT) via a min clamp after computing the naive block count, reducing total atomicAdd calls from ~8M to ~18K and eliminating atomic serialization while each thread simply iterates over more elements in its stride loop

## Key Lessons

- Capping thread block count at a small multiple of SM count (e.g., 2-4x) reduces atomic contention on shared memory locations — each thread absorbs more elements via its stride loop, trading redundant atomics for cheap register accumulation
- Excessive parallelism (thousands of blocks all hitting one atomicAdd address) can be slower than moderate parallelism — atomic serialization is a non-linear bottleneck where halving block count can yield orders-of-magnitude speedup
- For reduction-style kernels using atomicAdd to a single output, the optimal block count is bounded by the hardware's ability to serialize atomics, not by data size — prefer grid-stride loops with a bounded grid over naive ceil-div block counts
