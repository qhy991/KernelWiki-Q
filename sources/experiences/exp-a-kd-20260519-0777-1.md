## Solution

Load k_rope once into a register variable (k_rope_buf) before the loop, then reuse it across all 16 iterations, eliminating 15 redundant global load instructions and their associated address computation

## Key Lessons

- When data is small enough to fit in a single L2 cache line (e.g. 128 bytes for 64 bf16 values), redundant global memory reads within a loop are already absorbed by L2 cache — register buffering is still correct hygiene but may not yield measurable speedup in isolation
- Register buffering of read-only loop-invariant data is always worth applying for code clarity and forward-compatibility: even if the current workload fits in L2, larger problem sizes or cache-polluted scenarios may expose the benefit
