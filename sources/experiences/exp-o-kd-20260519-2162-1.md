## Solution

Replace the entire custom hip_fp8 struct and its MI300-specific inline asm conversion with the ROCm-provided __hip_fp8_e4m3_fnuz type, using static_cast<float>(f8) and __hip_cvt_fp8_to_halfraw which map to per-generation optimized hardware paths and allow the compiler full scheduling freedom

## Key Lessons

- Prefer vendor-provided SDK intrinsics (static_cast<float>, __hip_cvt_fp8_to_halfraw) over hand-written inline assembly for type conversions — SDK intrinsics are maintained per hardware generation, allow compiler scheduling freedom, and automatically target optimal paths on newer GPUs
- Inline assembly (asm volatile) creates an optimization barrier that prevents the compiler from scheduling surrounding instructions across the conversion boundary, even when the hardware supports the same operation natively through its intrinsic path
- Using platform-provided FP8 types (__hip_fp8_e4m3_fnuz) instead of custom struct wrappers eliminates MI300-specific #ifdef branching and gains forward compatibility with future GPU generations (MI325/MI350) at no code cost
