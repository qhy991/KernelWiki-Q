## Challenge

MoE expert permutation computation uses a full CUB DeviceRadixSort on all topk_ids pairs (O(N log N) with 4 radix passes over 32-bit keys), requiring large temporary storage buffers for sorted_ids, sorted_orig_idx, and sort temp storage, plus an extra init_indices kernel to prepare original indices before sorting


## Solution

Replace CUB radix sort and post-sort permutation building with an atomicAdd-based scatter kernel (compute_arg_sorts) where each thread directly claims the next slot for its expert via atomic increment, eliminating O(N log N) sort overhead, temporary sort buffers, and the init_indices kernel

## Key Lessons

- Replace O(N log N) radix sort with O(N) atomicAdd scatter when building per-group permutations from categorized indices — the scatter approach avoids sort passes, temporary buffers (sorted_ids, sorted_orig_idx, sort temp storage), and the init_indices kernel, yielding direct speedup when group count is moderate enough to keep atomic contention low
- atomicAdd-based slot claiming with pre-computed offsets eliminates the need for a sorted intermediate representation: each thread reads the source index once and writes directly to the output permutation at the atomically claimed position, reducing total memory traffic
- For MoE topk routing with ~64 experts and ~1K tokens per expert on average, atomic contention on per-expert counters is moderate enough that the atomic scatter approach outperforms the sort-based pipeline despite potential serialization
