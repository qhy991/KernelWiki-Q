## Solution

Consolidate the two-kernel pipeline into a single kernel where each thread maintains a private per-expert count array in shared memory (tokens_cnts), performs a parallel prefix-sum across threads for cumulative offsets, and sorts tokens directly using per-thread offset tracking — eliminating all atomicAdd operations, the inter-kernel cudaMemset, and reducing thread count from 1280 to 32

## Key Lessons

- Per-thread private counting arrays in shared memory eliminate atomic contention when each thread processes a disjoint subset of input elements — applies when the aggregation result fits in shared memory and thread count is modest (e.g., ≤128 threads for ≤16 experts)
- Fusing counting + prefix-sum + sorting into a single kernel avoids kernel launch overhead and intermediate global memory buffers — especially effective for small batch sizes where launch latency dominates compute time
- For small-scale MoE routing (numel ≤ few hundred, num_experts ≤ 64), a single-block per-thread-array approach outperforms multi-kernel atomic-based pipelines despite using far fewer threads (32 vs 1280)
