## Challenge

Iterative softmax uses atomicAdd to accumulate partial exp-sums (s_prime) across warps into shared memory, which serializes access when multiple warps contend for the same row accumulator in real fMHA kernels


## Solution

Replace atomicAdd with warp-partitioned addition_storage where each warp writes its partial sum to a dedicated slot (addition_storage[row + kQueriesPerBlock * warp_id]), then after __syncthreads() a designated lane per warp reduces all slots into s_prime without atomics

## Key Lessons

- Warp-partitioned storage eliminates atomicAdd contention when multiple warps contribute to the same accumulator — each warp writes to its own slot, then a cross-warp reduction combines them after barrier sync
- In microbenchmarks where only one warp targets each accumulator, atomicAdd is contention-free and equivalent to a plain store; the warp-partitioned pattern adds overhead (extra shared memory + reduction loop) without benefit — the optimization pays off only under genuine contention from multiple warps per row
- This pattern is the same one used in fMHA iterative softmax where successive key-block iterations cause multiple warps to update the same s_prime entry across tiles
