## Challenge

Tiled SGEMM kernel uses a single shared memory buffer with two __syncthreads() barriers per K-tile iteration: one after loading to smem and one after computing, serializing global memory loads behind computation and wasting ~50% of barrier slots


## Solution

Double-buffer shared memory with s_a[2][BK][BM] and s_b[2][BK][BN] to create a 2-stage software pipeline: issue global loads for the next tile into registers before computing the current tile, then write the loaded data to the alternate buffer after compute completes, reducing barriers from two to one per iteration and overlapping global memory latency with FFMA execution

## Key Lessons

- Double-buffer shared memory (s_a[2][BK][BM]) to create a 2-stage software pipeline: issue the next tile's global load into registers before computing the current tile, then write to the alternate buffer after compute — this overlaps global memory latency with FFMA execution and halves __syncthreads() overhead per iteration
- Use a prologue-epilogue pattern for double buffering: load the first tile in a prologue before the main loop, and compute the last tile in an epilogue after the loop, avoiding branch divergence inside the pipeline
- Select the active and next buffer with bitwise AND ((bk-1)&1, bk&1) for zero-overhead buffer ping-pong instead of modulo arithmetic
