## Challenge

In tile_offset_splitk_before.cu, SMEM navigation between k-slices uses a row-based tile_row_offset computed as smem_stride, equivalent to the original add_tile_offset({tile_row_offset, 0}) pattern. In tile_offset_splitk_after.cu, navigation uses smem_pointer_off


## Solution

Replace row-based add_tile_offset({tile_row_offset, 0}) with flat add_pointer_offset(kSmemPointerOffset) computed as smem_total_count / partitions_k, producing identical element offsets for current layouts while generalizing to multi-fragment SMEM layouts where partitions are not row-aligned

## Results

- **speedup**: 0.999x (neutral)


## Key Lessons

- Replacing add_tile_offset (2D row-based offset) with add_pointer_offset (flat element-count offset) in split-K SMEM navigation is a zero-cost refactor that generalizes to multi-fragment layouts where partitions are not row-aligned — both compute identical addresses for existing single-fragment layouts
- When refactoring SMEM offset computation, computing the stride as total_count / num_partitions rather than a hardcoded row stride preserves algebraic equivalence (smem_total_count / partitions_k == smem_stride) while making the formula layout-agnostic
