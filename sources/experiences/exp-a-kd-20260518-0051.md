## Challenge

The before code uses ld.global.cs.f32 (streaming) for scalar 32-bit loads, while the after code uses ld.global.cg.f32 (cache-all) for the same scalar loads. Scalar loads mean multiple threads within a warp access addresses within the same 1


## Solution

Switch cache hint from .cs (streaming, bypass L2) to .cg (cache-all levels) for sub-128-bit scalar loads so that the first thread's load brings the full 128-byte cache line into L2, allowing neighboring threads in the same warp to hit L2 for their sub-line accesses within that line

## Results

- **speedup**: 1.007x (neutral)


## Key Lessons

- For sub-128-bit access widths, prefer .cg (cache-all) over .cs (streaming) so that neighboring threads' accesses within the same 128-byte cache line share L2 hits — .cs is designed for full-cache-line streaming reads where no spatial locality exists
- Cache hint selection alone is insufficient when the dominant bottleneck is memory bandwidth for scattered scalar accesses rather than L2 hit rate — vectorized 128-bit loads achieve ~2.4x higher throughput than scalar 32-bit loads regardless of cache policy
- When the working set (~230MB) far exceeds L2 capacity (~40-60MB), cache hint differences become negligible because lines are evicted before reuse regardless of the hint
