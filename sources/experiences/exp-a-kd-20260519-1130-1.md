## Challenge

RMS norm kernel uses naiveBlockReduceSum where thread 0 serially iterates all 512 shared-memory entries (511 serial additions), forcing the remaining 511 threads to idle during the entire reduction phase, creating a severe serial bottleneck


## Solution

Replace serial block reduction with a two-phase warp-shuffle blockReduceSum: each warp reduces its 32 partial sums via __shfl_xor_sync in log2(32)=5 steps with no shared-memory traffic, then the first warp reduces the 16 warp-level sums via shuffle in log2(16)=4 steps, replacing 511 serial shared-memory additions with 9 parallel register-level shuffle steps

## Key Lessons

- Replace single-thread serial shared-memory reductions with warp-shuffle (__shfl_xor_sync) reduction to exploit full GPU parallelism — serial reductions force all threads to idle while one thread iterates, whereas shuffle completes in O(log warpSize) steps with all lanes active
- Two-phase blockReduceSum (warp-level shuffle + cross-warp shared-memory + final warp shuffle) scales to arbitrary block sizes with only 32-element shared memory instead of blockDim.x elements, reducing shared-memory pressure
- __shfl_xor_sync performs register-level exchange without shared-memory traffic, eliminating bank conflicts and reducing memory bandwidth consumption during reduction operations
