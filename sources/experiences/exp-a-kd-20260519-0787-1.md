## Challenge

Future token ID resolution uses three separate kernel launches (negate+clamp indices, gather from map, conditional write-back) with two intermediate buffers (neg_indices and gathered, totaling 256KB for 16384 int64 elements), causing redundant global memory round-trips and preventing inter-kernel register reuse


## Solution

Fuse the three sequential kernels into a single-pass kernel that reads input_ids once, performs condition-check, negate, clamp, and map-lookup entirely in registers, and writes the resolved value in-place — eliminating both intermediate buffers and two of the three kernel launches

## Key Lessons

- Fuse sequential element-wise kernels that operate on the same data into a single-pass kernel to eliminate intermediate global memory buffers — the speedup scales with array size because intermediate buffer traffic grows linearly while the fused kernel keeps all intermediates in registers
- Keeping loaded values in registers across multiple computation stages (condition-check → negate → clamp → lookup) avoids redundant global loads that occur when each stage is a separate kernel launch — applies whenever sequential per-element transforms have no cross-thread dependencies requiring global visibility
- Eliminating kernel launch overhead (3 launches → 1) removes two synchronization barriers and their associated scheduling latency, compounding the memory traffic savings at large element counts
