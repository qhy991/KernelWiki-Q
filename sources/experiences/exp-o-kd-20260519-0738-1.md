## Challenge

Per-token group FP8 quantization kernel launches 4096 blocks with only 16 threads (half a warp) per block, severely underutilizing SM capacity since each 16-thread block contributes only 0.5 warp and cannot effectively hide memory latency during the absmax reduction and quantization phases


## Solution

Consolidate 16 groups per block with 256 threads (8 warps), mapping threads via local_group_id=tid/16 and local_tid=tid%16, and using 2D shared memory s_absmax[16][17] with stride-17 padding to avoid bank conflicts across the 16 concurrent group reductions, amortizing per-block scheduling overhead across more work

## Key Lessons

- Block coarsening (multiple logical groups per block) amortizes per-block scheduling overhead and improves SM occupancy when individual group work is short — especially effective when group_size is small (e.g. 64) where fixed overhead dominates
- When coarsening multiple independent reductions into one block, use 2D shared memory with padded stride (e.g. [16][17] instead of [16][16]) to prevent bank conflicts between concurrent warp-level reductions
- Map threads to sub-groups via integer division/modulo (tid / warp_size, tid % warp_size) to keep each sub-group's reduction and computation contiguous within a single warp
