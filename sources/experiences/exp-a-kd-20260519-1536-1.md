## Challenge

The preprocessor architecture guard __CUDA_ARCH__ < 800 excludes sm_75 (Turing) GPUs from the optimized lop3.b32 dequantization path, forcing them into a slower fallback — even though Turing hardware also supports the lop3.b32 instruction


## Solution

Lower the preprocessor guard threshold from __CUDA_ARCH__ < 800 to __CUDA_ARCH__ < 750, correctly extending the optimized lop3.b32 dequantization path to sm_75 (Turing) GPUs which also support the lop3 instruction, while keeping sm_80+ behavior byte-identical

## Key Lessons

- Set architecture guard thresholds to the minimum SM version that actually supports the instruction (lop3.b32 is available from sm_75, not just sm_80) — over-conservative guards unnecessarily exclude capable hardware from optimized paths
- When benchmarking preprocessor-only changes, verify on the architecture whose behavior actually differs; testing on sm_80 where both guards produce identical SASS yields a neutral verdict even though the change materially improves sm_75 performance
