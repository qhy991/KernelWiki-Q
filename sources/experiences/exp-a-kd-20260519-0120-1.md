## Challenge

The FMHA kernel uses a 4-stage shared-memory pipeline (kStages=4) even for medium heads where head_size_v <= kKeysPerBlock=128, allocating 33792 bytes of shared memory per block. This forces the kernel to opt-in to extended shared memory (163840 bytes) via cudaFuncSetAttribute, which reduces L1 cache capacity and limits occupancy on A800 GPUs by allowing fewer concurrent blocks per SM. Additionally, the 4-stage compute loop introduces 4 __syncthreads barriers that are unnecessary when only one pipeline stage contains useful data.


## Solution

Pass kMaxK=128 to MakeCustomMma for medium heads so that kStages is computed as min(4, ceil(128/128))=1, reducing the shared-memory pipeline to a single stage. This shrinks per-block shared memory from 33792 to 8448 bytes, eliminating the need for extended shared memory opt-in and restoring full L1 cache capacity and occupancy, while also cutting compute loop iterations and __syncthreads barriers from 4 to 1.

## Key Lessons

- Right-size the pipeline stage count to the actual problem dimensions: when head_size_v <= kKeysPerBlock, only 1 stage is needed and using 4 wastes 3x shared memory and introduces 3 unnecessary __syncthreads barriers — always compute kStages as min(maxStages, ceil(headSize/kKeysPerBlock)) rather than using a fixed constant
- Exceeding the default 48KB shared-memory allocation via cudaFuncSetAttribute forces the hardware to carve out more shared memory from L1 cache, directly reducing SM occupancy; keeping per-block shared memory under 48KB preserves full L1 capacity and allows many more concurrent blocks per SM
- Reducing __syncthreads barrier count is a compounding win: each barrier forces all threads in a block to stall and synchronize, and with higher occupancy from smaller shared memory, more blocks can overlap those stalls across warps
