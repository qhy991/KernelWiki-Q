## Challenge

Per-token group 8-bit quantization kernel hardcodes the row-major scale output pointer layout as output_s + (block_group_id + local_group_id), preventing reuse of the same kernel for column-major weight layouts without a separate kernel implementation


## Solution

Add a bool IS_COLUMN_MAJOR template parameter with if constexpr dispatch: when false (row-major), the existing output_s + global_group_id path is selected at compile time with zero overhead; when true, a column-major index computation (col_idx * scale_stride + row_idx) is used, enabling a single kernel to serve both memory layouts

## Key Lessons

- Use if constexpr with a bool template parameter to add layout dispatch paths in CUDA kernels — the compiler eliminates the unused branch entirely, producing identical machine code to the original single-layout kernel with zero runtime overhead
- Prefer compile-time template dispatch over runtime if-else for memory layout variants in quantization/dequantization kernels, enabling a single kernel source to serve row-major and column-major data without code duplication or performance regression
