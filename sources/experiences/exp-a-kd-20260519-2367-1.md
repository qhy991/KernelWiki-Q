## Challenge

In a selective scan forward kernel processing batched SSM sequences with 75% padding entries, all thread blocks execute the full computation including padding blocks that write to a shared sentinel slot, wasting compute but maintaining high SM occupancy and memory pipeline utilization


## Solution

Add an early-return guard that checks if the cache_index equals the pad_slot_id sentinel value and immediately exits the thread block, skipping all downstream SSM computation for padding entries — correct in principle but causes a regression in this microbenchmark because the fixed grid launch leaves most SMs idle when 75% of blocks exit immediately

## Key Lessons

- Early-return guards for padding/invalid entries correctly save per-block computation, but when launched with a fixed grid covering all batch entries including padding, the mass block exit starves SMs of work — in production the grid should be sized to only valid entries rather than relying on in-kernel early-return
- High padding ratios (75%) with a sentinel-slot design cause padding blocks to share the same cache_index, creating favorable L2 cache hit patterns in the unoptimized version that disappear once those blocks exit early — microbenchmark regressions can reflect measurement artifacts rather than real performance loss
- When evaluating early-return or masking optimizations, consider the ratio of active-to-idle blocks: if fewer blocks remain than there are SMs (8 valid × 8 dim = 64 blocks vs 108 SMs on A800), SM occupancy drops too low to hide memory latency, making the optimization appear harmful in synthetic benchmarks
