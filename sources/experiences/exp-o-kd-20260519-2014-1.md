## Challenge

CUTLASS SM90 TMA-based grouped GEMM for MOE batches problems along the M dimension, requiring each problem's M to be padded to the tile shape (e.g., 128). For small M values (M=4), this causes 32x wasted computation due to padding a tiny M to a large tile boundary while the large N dimension incurs no padding penalty.


## Solution

Add a compile-time swap_ab template parameter that conditionally transposes A and B operand layouts in the CUTLASS collective builders, computing B^T*A^T instead of A*B. This shifts the batched dimension from M to N, so padding overhead applies to the already-large N instead of small M, reducing wasted FLOPs from 32x to negligible for small-M grouped GEMM problems.

## Key Lessons

- Swap A/B operand layouts at compile time via conditional_t to compute B^T*A^T instead of A*B in grouped GEMM — shifts padding overhead from the small batched dimension to the large unbatched dimension, eliminating most wasted FLOPs for skewed problem shapes
- Use LayoutTranspose trait and conditional_t on layout pointer types in CUTLASS collective builders to select transposed operands with zero runtime cost — the branch is resolved entirely at compile time and the generated kernel is specialized for each layout variant
- In TMA-based grouped GEMM, padding waste is proportional to (tile_dim / actual_dim) on the batched dimension — always batch along the larger dimension (N for MOE where M is small) to minimize padding overhead
