## Challenge

Warp index computed via __shfl_sync(0xffffffff, threadIdx.x / 32, 0) broadcasts warp_id from lane 0 to all 32 lanes, creating an implicit warp-synchronization barrier that stalls the pipeline when warp threads are not converged due to divergent prior control flow


## Solution

Replace __shfl_sync broadcast with canonical_warp_idx() which computes threadIdx.x / NumThreadsPerWarp directly from the per-thread register value, eliminating the shuffle instruction and implicit sync barrier since all threads in a warp already produce the same quotient

## Key Lessons

- Prefer direct register-based computation over __shfl_sync broadcast when every thread can independently derive the same value (e.g., warp index = threadIdx.x / warpSize) — the shuffle is redundant because integer division by warpSize yields identical results across all lanes in a warp, and removing it eliminates an implicit warp-sync barrier that blocks forward progress under divergent control flow
- canonical_warp_idx() is the idiomatic CUTLASS replacement for __shfl_sync-based warp index broadcast — use it in all GEMM kernel hot paths to avoid unnecessary synchronization overhead
