## Solution

Add an explicit kClearLastStage step that zeroes the next-stage SMEM slot after the mainloop to prevent stale data from persisting when the circular buffer wraps, at negligible cost (128 floats = 512 bytes vs total mainloop SMEM traffic)

## Key Lessons

- kClearLastStage is a correctness mechanism not a performance optimization — zeroing one SMEM stage (512 bytes) is negligible vs total mainloop traffic, applicable whenever circular SMEM buffers wrap across iterations
- An extra __syncthreads for a small zeroing operation does not increase critical-path stall in multistage pipelines where barriers already gate each mainloop iteration — applies when designing stage-based SMEM buffering patterns
- Explicitly clearing stale SMEM data after the mainloop prevents subtle correctness bugs when the circular buffer is reused across iterations, even when the performance overhead is within measurement noise (+0.04%)
