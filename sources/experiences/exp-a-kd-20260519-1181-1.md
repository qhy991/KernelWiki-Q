## Challenge

Replacing half2 scales with fp8x2 (e4m3) on sm_80 halves memory traffic (4→2 bytes per pair) but forces scale conversion through a software emulation path: __nv_cvt_fp8x2_to_halfraw2 plus __half22float2, generating 3-4x more instructions per element than the native __half22float2 used on half2


## Solution

The after kernel replaces half2 scales with __nv_fp8x2_e4m3, cutting scale memory traffic by 50%, and uses (float2)s cast for conversion; however on sm_80 this maps to a multi-instruction software emulation chain rather than a native hardware instruction, so the instruction overhead dominates the memory savings, producing a net regression

## Key Lessons

- Replacing half2 with fp8x2 only benefits memory-bound kernels on GPUs with native fp8 hardware (sm_89+ Ada/Hopper); on sm_80 the software emulation of (float2)s generates 3-4x more PTX instructions per element than native __half22float2, overwhelming any memory bandwidth savings
- When evaluating a precision reduction for memory traffic gains, always verify the target GPU has native hardware support for the lower-precision conversion — the per-element instruction cost of software emulation can dominate at moderate data volumes
