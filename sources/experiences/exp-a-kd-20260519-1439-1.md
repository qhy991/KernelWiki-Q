## Solution

FP32 pre-reduction scaling via scale_float_device converts each fp16 scale component to fp32 with __half2float, multiplies in fp32 with __fmul_rn before converting the result to fp16 — preserving numerical correctness for large accumulator values at the cost of extra __half2float conversions and separate fp32 multiplies versus a single __hmul2

## Key Lessons

- __hmul2 performs two fp16 multiplies in one hardware instruction — use it for post-reduction scaling when accumulator values are within fp16 range for best throughput
- Scaling in fp32 before converting to fp16 (pre-reduction scaling) preserves numerical correctness for large accumulator values but adds __half2float conversion and separate __fmul_rn instruction overhead, resulting in ~4.5% throughput loss for safe-range values
- Choose between __hmul2 post-scaling and fp32 pre-scaling based on accumulator magnitude: post-scaling for throughput when values are safe-range, pre-scaling for correctness when overflow is possible
