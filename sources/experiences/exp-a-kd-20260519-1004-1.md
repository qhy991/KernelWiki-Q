## Challenge

Per-token max-finding kernel uses scalar 2-byte loads (one bf16 element per LDG instruction), issuing 4096 individual load instructions per token and underutilizing the 128-byte GPU cache line which could service 64 elements per fetch


## Solution

Replace 4096 scalar 2-byte loads with 512 vectorized 128-bit loads via uint4 reinterpret_cast, fetching 8 bf16 elements per LDG.128 transaction and reducing total load instructions by 8x for significantly better cache line utilization

## Key Lessons

- Use uint4 (128-bit) vectorized loads via reinterpret_cast for contiguous bf16 memory traversals — each LDG.128 fetches 8 bf16 elements in one transaction, reducing load instruction count by 8x and improving cache line utilization
- Vectorized load + inner scalar processing is a practical pattern when the per-element compute (bfloat162float + fmaxf) has no vectorized intrinsic — the memory bandwidth gain dominates even though the compute remains element-wise
