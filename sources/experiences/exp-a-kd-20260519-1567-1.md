## Challenge

Naive iterative-selection top-k performs O(N*K) work by scanning all N=64 shared-memory elements K=4 times to find each maximum via warp shuffle reduction, requiring __syncthreads between passes and a used-flag array to exclude previously selected elements — motivating a more asymptotically efficient algorithm


## Solution

Replace iterative max-selection with WarpSelect incremental top-k: process elements in warp-sized rounds, use __ballot_sync to filter candidates above the current k-th threshold, compact qualifying candidates into a shared memory buffer via __popc-based indexing, and maintain the top-k set using warp-level bitonic sort and merge — achieving O(N log k) asymptotic complexity

## Key Lessons

- For small problem sizes (N=64, K=4), naive iterative top-k with simple warp max-reduction outperforms WarpSelect because the overhead of ballot_sync, popc-based compaction, bitonic sort/merge, and syncwarp exceeds savings from better asymptotic complexity — algorithm selection should be guided by actual dimensions, not big-O alone
- WarpSelect's ballot_sync early-exit provides minimal benefit when the initial k-th threshold is -INFINITY (first round), since nearly all candidates pass the filter and the popc compaction overhead is wasted for the dominant first iteration
- __syncthreads in the naive version (K=4 calls) is cheaper than multiple __syncwarp + shared-memory staging + bitonic sort in WarpSelect when K is small and the warp already holds all N elements across two strided loads
