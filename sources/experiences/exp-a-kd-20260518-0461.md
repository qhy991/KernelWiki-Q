## Challenge

The before code uses a 2D nested loop where rows are mapped to blockIdx.x (grid stride) and columns to threadIdx.x (block stride). With numRows=8, the grid is capped at min(8, multiProcessorCount*numBlocksPerSM) = 8 blocks, leaving the A800's 108 SMs mostly idle


## Solution

Replace half2 (64-bit) loads with LDST128BITS 128-bit vectorized loads to fetch 8 half elements per thread, reducing thread count from 128 to 32 but increasing per-thread work; however the resulting single-warp block severely limits occupancy and latency hiding, yielding only marginal improvement

## Results

- **speedup**: 1.474x (improved)


## Key Lessons

- Widening vectorized loads (half2→LDST128BITS) without maintaining sufficient occupancy can negate bandwidth gains; reducing threads from 128 to 32 collapses the block to a single warp, starving the scheduler of latency-hiding opportunities
- For memory-bound kernels where total bytes transferred is unchanged, vectorization primarily affects instruction count and transaction efficiency but cannot overcome an occupancy floor — the benefit plateaus or regresses if thread count drops too far
- When increasing elements-per-thread, keep block-level warp count >= 4 to preserve scheduling flexibility; a single-warp block eliminates cross-warp interleaving and makes the kernel vulnerable to any memory or pipeline stall
