## Solution

Shape-aware tile selection dispatches TileShape<16,64,128> for M<=16 and splits GEMMs along N into sub-problems via multiple cublasGemmEx calls, but the multi-launch N-split adds kernel launch overhead exceeding any tiling gain since cuBLAS chooses its own internal tiling regardless of API-level partitioning

## Key Lessons

- CUTLASS tile-shape optimization is a compile-time template parameter affecting thread mapping within a single kernel launch — it cannot be proxied by runtime N-split multi-launch because cuBLAS chooses its own internal tiling independently of API-level problem partitioning
- Splitting a single GEMM into multiple sub-GEMMs along N adds per-launch overhead (kernel launch, queueing) that can exceed any benefit from narrower problem dimensions when total compute remains identical
