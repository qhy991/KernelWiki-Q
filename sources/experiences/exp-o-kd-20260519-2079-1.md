## Challenge

A single CTA serially iterates over all paged KV-cache blocks with a nested two-level loop (blocks × tokens per block), performing repeated block_table lookups per block and divergent conditional branches for partial-block boundary handling, leaving the GPU massively underutilized


## Solution

Replace the serial single-CTA block-level loop with a parallel per-token direct dispatch (one CTA per token) that computes each token's cache address with a single block_table lookup via batch_offset/block_size arithmetic, eliminating the nested loop structure, partial-block boundary conditionals, and O(blocks) block_table reads per CTA

## Key Lessons

- Replace serial block-level CTA loops with parallel per-token CTAs when gathering from paged KV-cache — each token's address can be computed independently with a single block_table lookup (batch_offset/block_size for block index, batch_offset%block_size for slot), exposing massive parallelism that GPUs exploit via high CTA counts
- Use a token_to_seq mapping array to pre-compute batch membership for each token, avoiding per-token binary search over cu_seq_lens and enabling O(1) batch-to-sequence resolution in the kernel
- Avoid divergent conditional branches for partial-block boundary handling (last block may have fewer tokens) by computing tokens_in_block per-token rather than per-block — the per-token dispatch naturally handles ragged blocks without branches
