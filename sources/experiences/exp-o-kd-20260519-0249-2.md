## Challenge

NHWC padding kernel uses scalar 4-byte loads and stores (float per element), underutilizing available DRAM bandwidth on a memory-bound kernel where throughput is limited by the number of memory transactions rather than compute


## Solution

Reformulate the kernel around float4 (128-bit) vectorized I/O: input is cast to float4* and loaded as coalesced 16-byte transactions into shared memory, and output is assembled as float4 values written via 16-byte stores, achieving full DRAM bandwidth utilization

## Key Lessons

- Memory-bound kernels should use the widest vectorized type practical (float4/half2/float2) for both loads and stores — 128-bit transactions fully utilize each DRAM burst and reduce total transaction count by 4x versus scalar float
- Casting pointers to float4* (or using reinterpret_cast) for vectorized I/O is safe when the underlying allocation is sufficiently aligned and the element count is a multiple of the vector width — the launch_padding wrapper handles padding via boundary checks
