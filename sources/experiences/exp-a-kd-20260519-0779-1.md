## Challenge

MLA key concatenation kernel uses scalar bf16 reads/writes where each warp lane issues multiple scattered 2-byte global memory transactions per head iteration, and k_rope data is redundantly re-read from global memory in every head iteration instead of being held in registers


## Solution

Replace scalar bf16 loads/stores with vectorized int2 (8-byte) and int (4-byte) reinterpret_cast transactions for fully coalesced 256-byte/128-byte warp-wide access, and restructure the single read-write loop into separate read-buffer and write phases so k_rope is loaded once into a register and k_nope data is held in a register array across iterations

## Key Lessons

- Use int2/int reinterpret_cast for bf16 data to issue 8-byte or 4-byte coalesced transactions per lane instead of multiple scattered 2-byte scalar transactions — 32 lanes x 8 bytes = 256 bytes matches L2 cache line size for maximum memory throughput, applies whenever bf16 elements are contiguous and type-aligned
- Buffer repeatedly-read global data (e.g., k_rope shared across all head iterations) in registers to eliminate redundant global loads — applies whenever the same read-only data is accessed inside a loop
- Separate read and write phases in a copy/concat kernel into distinct loops to enable register buffering and reduce total global memory traffic, rather than interleaving reads and writes in a single loop
