## Challenge

Router GEMM kernel passes num_tokens as a runtime parameter, forcing the compiler to treat the token loop as variable-length, allocate a fixed 16-element accumulator array (wasting registers when num_tokens < 16), and preventing full unrolling of the dot-product, warp-shuffle reduction, and cross-warp reduction loops


## Solution

Promote num_tokens from a runtime parameter to a compile-time template parameter kNumTokens, enabling right-sized accumulator allocation (float acc[kNumTokens] instead of acc[16]), full #pragma unroll on all token/iteration loops, and __launch_bounds__ hints — allowing the compiler to eliminate loop control overhead, minimize register waste, and generate specialized code for each token count

## Key Lessons

- Promote small runtime loop bounds (especially M dimensions in batched GEMM) to compile-time template constants when the set of possible values is known and small — this enables full loop unrolling, right-sized register allocation, and eliminates branch overhead, yielding 2.6x speedup in this router GEMM case
- Use __launch_bounds__(blockSize, minBlocks) alongside compile-time specialization to give the compiler register-pressure hints, preventing it from over-allocating registers when unrolling creates many accumulator variables
- Size shared memory and accumulator arrays with the compile-time constant (e.g., acc[kNumTokens] not acc[16]) to avoid wasting registers and shared memory when the actual count is smaller than the worst-case allocation
