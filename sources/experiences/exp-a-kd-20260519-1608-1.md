## Challenge

MHA V-cache stores 8 independent KV heads for 8 query heads, using block_stride=NH*HS*BS (8192 elements per block), which wastes 4x memory because heads 0-3 and 4-7 access identical KV data through an identity head_mapping


## Solution

Switch to GQA with NKV=2 KV heads by changing block_stride to NKV*HS*BS and using a grouped head_mapping {0,0,0,0,1,1,1,1} so 4 query heads share each KV head, reducing V-cache memory by 4x while keeping the kernel logic identical

## Key Lessons

- GQA reduces V-cache memory footprint by the ratio num_query_heads/num_kv_heads (here 8/2=4x) with zero kernel code changes — only the host-side block_stride, V-cache allocation size, and head_mapping table need updating
- The stride value itself does not affect per-call kernel performance because each query head still reads from exactly one mapped KV head with the same iteration count — the memory savings come from smaller total allocation, not from faster per-element access
- When multiple query heads share the same KV data (heads 0-3 → kv0, heads 4-7 → kv1), the kernel naturally deduplicates through head_mapping without needing special deduplication logic
