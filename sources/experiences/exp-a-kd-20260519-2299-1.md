## Solution

Consolidate three separate kernels into one advance_step_kernel that performs all four tensor updates per thread, but launch with grid size equal to device SM count (108 on A800) instead of ceil(num_queries/256)=16, scheduling 92 empty blocks that immediately return and add dispatch overhead that outweighs the eliminated kernel launch costs

## Key Lessons

- Kernel fusion saves launch overhead but only when the consolidated kernel's grid size matches actual work; launching sm_count blocks when only ceil(N/threads) are needed wastes scheduling resources on empty blocks — use min(sm_count, ceil(N/threads)) or compute grid size from workload, not hardware capacity
- For lightweight per-thread workloads (a few scalar reads/writes), kernel launch overhead is small relative to computation, so the bar for fusion to be profitable is higher — fusion benefits increase with compute intensity per thread
