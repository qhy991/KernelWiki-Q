## Solution

Replace the scalar stride loop with vectorize_read_with_alignment<8> that issues vec_n_t<scalar_t,8> vector loads (128-bit for half/bf16, 256-bit for float32) in the aligned portion of the row, with scalar fallback for unaligned prefix/tail, reducing global memory transactions by up to 8x

## Key Lessons

- Use vectorized loads (vec_n_t<scalar_t,8>) instead of scalar strided loops for memory-bound reduction/accumulation patterns — reduces global memory transactions by up to 8x and significantly improves bandwidth utilization, especially effective for large hidden sizes (4096+)
- Precompute row base pointers (input_row = input + blockIdx.x * stride) outside the loop to avoid redundant address computation per iteration — a minor but consistent optimization when the base address is loop-invariant
- Alignment-aware vectorized access (prefix + vectorized main + tail) ensures correctness regardless of input pointer alignment, removing the need for caller-side alignment guarantees while still getting vectorized throughput on the aligned portion
