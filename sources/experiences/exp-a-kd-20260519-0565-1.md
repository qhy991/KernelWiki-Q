## Challenge

SGEMM kernel computes one output element per thread with scalar global memory loads and stores using 32x32 block tiles, resulting in low arithmetic intensity and poor memory throughput utilization


## Solution

Thread tiling (8x8 per thread) with float4 vectorized loads and larger 128x128 block tiles was applied, but the row-major s_a[128][8] layout causes severe shared memory bank conflicts — consecutive threads accessing the same column k hit only BK=8 of 32 banks, degrading smem throughput and overriding the arithmetic intensity gains

## Key Lessons

- Thread tiling with small BK in row-major shared memory layout causes severe bank conflicts — when BK << 32, threads accessing the same column k cycle through only BK banks, creating multi-way conflicts that degrade smem throughput and can override arithmetic intensity gains
- Shared memory layout must be co-designed with thread tiling parameters — transposing s_a to s_a_k_bk[BM] or adding padding to make the leading dimension a non-power-of-2 ensures consecutive threads in the compute phase access consecutive banks
- Vectorized float4 global memory loads cannot compensate for shared memory bank conflicts in the compute inner loop — when thread tiling increases arithmetic intensity, the dominant cost shifts from global memory bandwidth to shared memory throughput, making bank-conflict-free access patterns critical
