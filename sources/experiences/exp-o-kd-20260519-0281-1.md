## Challenge

Serial GEMM kernel iterates over all 128 K-tiles (K=4096) in a single kernel launch with an 8x8 block grid, underutilizing available SMs because the 64-block grid occupies only ~59% of the A800's 108 SMs while the long K-iteration serializes all computation


## Solution

Split K=4096 into 8 slices of K_len=512 and launch 8 gemm_partial kernels concurrently on separate CUDA streams, each computing a partial sum over its K-slice; a lightweight reduce_k kernel then sums the 8 partial results, exploiting SM-level concurrency to overlap K-slice computation despite full 8x parallelism being limited by SM contention

## Key Lessons

- Split-K decomposition across CUDA streams enables SM-level concurrency for small-tile GEMM grids that underutilize SMs — each partial kernel has a shorter K-loop reducing per-block latency, and the GPU scheduler interleaves blocks from different streams to make forward progress on multiple K-slices simultaneously
- Use cudaEventRecord + cudaStreamWaitEvent to enforce a barrier between concurrent partial kernels and the reduction kernel, ensuring all partials complete before summation without a heavy cudaDeviceSynchronize
- Real-world speedup from split-K is bounded by SM contention: 8 concurrent 64-block kernels need 512 blocks but only 108 SMs exist, so measured speedup (~1.7x) is less than the 8x theoretical maximum — choose K_SPLITS to match available SM headroom
