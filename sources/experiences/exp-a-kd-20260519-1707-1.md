## Challenge

The fused_add_rms_norm kernel always launches with block_size=min(hidden_size,1024)=1024 threads, limiting occupancy to fewer concurrent blocks per SM when num_tokens is large (e.g. 512 blocks of 1024 threads), reducing the scheduler's ability to hide memory latency


## Solution

Use an adaptive block size formula: when num_tokens >= 256, reduce block_size to 256 threads (4x more blocks per SM) and dispatch blockReduceSum with matching maxBlockSize template parameter to reduce shared memory usage and increase occupancy for high-batch workloads

## Key Lessons

- Adaptive block sizing based on grid size (num_tokens) can improve occupancy: smaller thread blocks allow more concurrent blocks per SM, but the benefit is modest for scalar kernels with low memory-level parallelism per thread since the increased loop iteration count partially offsets the occupancy gain
- The blockReduceSum template maxBlockSize parameter must match the actual launch block size to correctly size shared memory and active warp counts — mismatching wastes shared memory or causes incorrect reduction results
- Block size tuning alone yields marginal improvement for scalar kernels; it becomes far more impactful when paired with vectorized memory access patterns where each thread has higher memory throughput
