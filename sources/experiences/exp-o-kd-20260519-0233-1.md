## Challenge

CUTLASS GEMM epilogue compute_source_not_needed_ loop processes one fragment per iteration with two __syncthreads() barriers per fragment (before warp store and after shared load), totaling 2*kIterations barriers and creating significant synchronization overhead for large output tiles


## Solution

Coarsen the outer loop stride from 1 to kFragmentsPerIteration and add inner loops to batch multiple fragments between each pair of __syncthreads() barriers, reducing total barrier count from 2*kIterations to 2*(kIterations/kFragmentsPerIteration) while using SMEM pointer offsets to manage non-contiguous fragment storage

## Key Lessons

- Loop coarsening (increasing stride and batching multiple work items between barriers) reduces __syncthreads() count proportionally — applies whenever independent iterations share the same barrier pair and SMEM capacity permits batching
- When coarsening a loop that writes to SMEM with pointer offsets, advance the pointer after each inner iteration and reset it once after the batch using offset*(1-batch_size) to maintain correct SMEM addressing without per-fragment base restoration
- CUTLASS epilogue barriers are a non-trivial cost for large output tiles (validated at 15-16% speedup) — barrier amortization should be considered whenever kIterations exceeds 1 in shared-memory epilogue paths
