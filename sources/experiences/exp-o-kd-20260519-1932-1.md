## Challenge

Grouped GEMM scheduler uses a hardcoded ASSUMED_SM_COUNT (84) to compute launch block count, causing under-utilization on devices with more SMs and over-subscription on devices with fewer SMs


## Solution

Query the actual device SM count at runtime via cudaDeviceGetAttribute(&sm_count, cudaDevAttrMultiProcessorCount, device_id) so that num_blocks and groups_per_block match the real hardware, reducing sequential work per block and fully occupying all available SMs

## Key Lessons

- Hardcoded SM counts in grouped/kernel launch scheduling cause load imbalance on any device that differs from the assumed value — always query cudaDevAttrMultiProcessorCount at runtime for portable, hardware-adaptive block counts
- When each block sequentially processes a batch of independent work items (groups_per_block), reducing gpb by increasing num_blocks to match actual SMs directly reduces per-block serial work and improves occupancy across the GPU
