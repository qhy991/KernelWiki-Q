## Challenge

Tensor concatenation kernel uses scalar bf16 load/store per element with per-iteration integer division and modulo to decompose flat index into (token, head, dim), causing high instruction overhead and uncoalesced memory access patterns


## Solution

Assign one warp per (token, head) pair to eliminate per-element index decomposition, then use 128-bit inline-asm vector loads/stores with cache-streaming (.cs) hints for the bulk nope portion and 32-bit scalar cache-streaming transfers for the rope portion, reducing load/store instruction count by ~8x and enabling fully coalesced warp-level memory transactions

## Key Lessons

- Assign one warp per output row instead of using a flat grid-stride loop — eliminates per-element integer division/modulo for index decomposition and enables each lane to handle a fixed offset within the row, producing fully coalesced memory transactions
- Use 128-bit (int4/v4.u32) inline-asm vector loads and stores for bulk data movement — each warp transfers 32×16=512 bytes per instruction, reducing total load/store count by ~8x compared to scalar bf16 access
- Add cache-streaming (.cs) hints to load/store instructions for write-once data patterns like tensor concatenation — streams data through L2 without allocating cache lines, avoiding cache pollution and improving throughput for sequential writes
