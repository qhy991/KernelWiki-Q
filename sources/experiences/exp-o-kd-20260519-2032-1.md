## Challenge

GEMM with small M=3 dimension requires padding to M_padded=4 for cuBLAS alignment, incurring overhead from cudaMemset zero-fill of padded buffer, two cudaMemcpy2D bulk column-major copies (input padding and output slicing), and computing 1 extra output row in the padded GEMM


## Solution

Use swap_ab strategy: compute C^T = B^T * A^T via cublasHgemm with CUBLAS_OP_T on both operands, making N=4096 the effective leading dimension of the result — this eliminates padding, zero-fill, and output slicing entirely since the large N dimension is naturally aligned for tensor cores

## Key Lessons

- When GEMM has a small M dimension that requires alignment padding (e.g., M=3 padded to 4), use the transpose identity C = A*B → C^T = B^T*A^T to swap the problem so the large dimension becomes the leading GEMM dimension — this eliminates cudaMemset, cudaMemcpy2D padding copies, and output slicing overhead
- swap_ab (CUBLAS_OP_T on both operands) is a zero-copy algebraic transformation that avoids extra memory allocations and data movement, making it strictly superior to padding when the result layout is compatible with transposed storage
- cuBLAS tensor-core performance is dominated by the largest GEMM dimension; transposing to make N=4096 the leading dimension ensures full tensor-core utilization without artificial padding of the small M dimension
