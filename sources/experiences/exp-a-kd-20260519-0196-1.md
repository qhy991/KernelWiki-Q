## Challenge

Warp index is computed via __shfl_sync(0xffffffff, threadIdx.x / 32, 0) inside a 1000-iteration tight loop, incurring multi-cycle warp shuffle pipeline latency on every invocation even though all lanes are already converged and the result is independent of cross-lane communication


## Solution

Replace __shfl_sync warp shuffle with a __device__ helper canonical_warp_idx() that compiles to a single integer division (threadIdx.x / 32), eliminating the warp-cooperative shuffle instruction entirely and converting a multi-cycle warp coordination operation into an independent per-thread ALU operation

## Key Lessons

- Use __shfl_sync only when cross-lane data exchange is required — when the computation depends on another lane's value. For pure per-thread computations like warp_idx = threadIdx.x / 32, a simple integer operation avoids the shuffle pipeline entirely and eliminates warp synchronization overhead
- Extract warp index computation into a __device__ helper (canonical_warp_idx()) with a __CUDA_ARCH__ guard to ensure correct compilation in both device and host contexts, replacing scattered __shfl_sync calls with a clean inlineable function
- __shfl_sync has non-trivial instruction latency (multiple scheduler cycles) even when all lanes are converged — in tight loops the cumulative overhead compounds significantly, making direct integer arithmetic dramatically faster for repeatedly computed thread-local values
