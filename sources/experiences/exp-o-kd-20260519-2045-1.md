## Solution

Introduce an adaptive max_block_size — 1024 when num_tokens < 256, else 256 — so that large batches launch more, smaller blocks per SM, increasing resident block count by up to 4x and improving memory latency hiding

## Key Lessons

- Use a smaller max_block_size (e.g., 256) for large-batch memory-bound kernels to increase resident blocks per SM and improve latency hiding; keep 1024 for small batches to maximize per-block parallelism
- Adaptive block sizing based on workload size (num_tokens) lets a single kernel configuration perform well across both small and large batch regimes
