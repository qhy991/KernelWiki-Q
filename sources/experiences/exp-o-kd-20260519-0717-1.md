## Challenge

In Groupwise CTA scheduling, each thread block always starts at blockIdx.x offset for every group independently, so CTAs that processed many tiles in early groups are forced to also absorb many tiles in later groups, causing workload imbalance when groups have unequal tile counts (common in MoE expert models)


## Solution

Introduce Flatten scheduling with rotating block offsets: track cumulative tiles across groups via group_total_tiles and derive head_cta_id = group_total_tiles % gridDim.x as the rotation seed, then compute each CTA's starting offset as (blockIdx.x + (gridDim.x - head_cta_id)) % gridDim.x so that CTAs absorbing remainder tiles in one group get lower offsets in the next group, balancing total work across all CTAs

## Key Lessons

- Replace per-group independent CTA scheduling (blk_offset = blockIdx.x) with cross-group rotating offsets derived from cumulative tile counts — this distributes remainder tiles across different CTAs in each group, reducing workload variance
- Track head_cta_id = cumulative_tiles % gridDim.x to record how many CTAs were 'used up' by remainder tiles from prior groups, then rotate starting offsets accordingly — applies whenever a grid of CTAs iterates over multiple work groups with potentially unequal tile counts
- Even when all groups have equal dimensions, Flatten scheduling improves balance if per-group tile counts are not divisible by gridDim.x — the rotation eliminates systematic bias where the same CTAs always absorb remainder work
