## Challenge

When num_experts < WARP_SIZE (32), launching only num_experts threads creates a partially-filled warp that cannot execute efficiently; shared memory is sized for exactly num_experts threads, so any extra threads would write out-of-bounds, and accumulation/assignment loops run unconditionally producing incorrect results on padding threads


## Solution

Launch max(num_experts, WARP_SIZE) threads to guarantee full-warp execution, resize tokens_cnts shared memory from (num_experts+1)*num_experts to (blockDim.x+1)*num_experts to accommodate actual thread count, and add if (threadIdx.x < num_experts) guards so padding threads skip expert-specific accumulation and assignment

## Key Lessons

- Always launch at least WARP_SIZE threads even when the logical work count (e.g., num_experts) is smaller — a partial warp wastes an entire scheduling slot and halves occupancy; guard extra threads with if (threadIdx.x < logical_count) to maintain correctness
- Size shared memory buffers based on blockDim.x rather than the logical work count, since the actual thread count may exceed the logical count after warp alignment — this prevents out-of-bounds shared memory writes from padding threads
- Warp-aligned launches with threadIdx guards are a general pattern for any kernel where the problem dimension (experts, heads, tiles) may be sub-warp — applies to MoE routing, small-batch attention, and low-dimensional reductions
