## Challenge

The moe_permute path unconditionally allocates an align_expert_first_token_offset tensor via cudaMalloc+cudaMemset and launches getMIndicesKernel on every call, even when the CUTLASS group gemm path does not require alignment, wasting GPU memory bandwidth and driver overhead per invocation


## Solution

Guard the cudaMalloc+cudaMemset allocation and getMIndicesKernel launch behind an if(needs_alignment) condition, so when the CUTLASS group gemm path is active (align_block_size not set), the entire per-call allocation, memset, kernel launch, and free cycle is skipped, reducing the hot-path to a single host-side branch evaluation

## Key Lessons

- Guard GPU allocation (cudaMalloc+cudaMemset) and kernel launches behind a runtime condition check for optional code paths — eliminates per-call driver overhead and memory bandwidth waste when the feature is not active, achieving 2.15x speedup in this case
- Per-iteration cudaMalloc+cudaFree cycles are expensive even for small tensors (9 int64 elements) because each call involves GPU driver synchronization overhead — moving allocation outside the loop or skipping it entirely is strongly preferred
- In MoE frameworks, alignment kernels like getMIndices are only needed for certain backend paths (e.g., CUTLASS padded-GEMM) but not others (e.g., CUTLASS group-GEMM) — conditionally dispatching based on the active backend avoids unnecessary GPU work
