## Challenge

Sparse attention vertical-slash index merge kernel uses fixed global NNZ_V=64 and NNZ_S=64 as loop bounds for all heads, forcing sparse heads (actual NNZ=4 or 8) to iterate through all 64 entries in the while(1) merge loop, wasting global memory reads from vertical_indexes/slash_indexes arrays and redundant conditional branch evaluations


## Solution

Override the global NNZ_V/NNZ_S loop bounds with per-head actual topk counts read from new per_head_vertical_topkv and per_head_slash_topkv device arrays at kernel entry, so sparse heads (62.5% of heads have NNZ=4 or 8) exit the merge loop after 4-8 iterations instead of 64

## Key Lessons

- Replace global-max loop bounds with per-element actual counts when workloads have heterogeneous sparsity — sparse elements exit early while dense elements iterate the full range, yielding speedup proportional to the sparse population fraction
- Per-head metadata arrays (topk counts, actual seqlens) are cheap to read once per thread block and can dramatically reduce iteration counts in merge/traversal kernels where the global max overestimates actual work for most elements
