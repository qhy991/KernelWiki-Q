## Challenge

The getMIndices kernel reads expert_first_token_offset[i] directly from global memory inside the accumulation loop that iterates up to eidx+1 times per block, incurring repeated high-latency L1/L2 global memory accesses for the small 65-element offset array (520 bytes)


## Solution

Cooperatively load all 65 int64_t offsets (520 bytes) into shared memory via __ldg before the accumulation loop, then read from smem during iteration — replacing ~200-cycle L2 global reads with ~5-cycle shared memory reads for the hot accumulation path

## Key Lessons

- When a kernel iterates over a small lookup table (hundreds of bytes) in a hot loop, cooperatively loading it into shared memory with __ldg before the loop eliminates repeated global memory latency — applies whenever the data fits in smem and is accessed multiple times per thread
- Use __ldg (read-only cache) during the cooperative smem load to route the initial read through the texture/L1 cache path, avoiding potential L2 thrashing when many blocks load the same small array simultaneously
