## Challenge

FP8 dynamic per-token scaling kernel uses the raw std::numeric_limits max value 240.0f (0x7F) as the qmax divisor, which causes NaN boundary accuracy issues on ROCm for the Float8_e4m3fnuz type where 0x7F maps to NaN


## Solution

Replace the hardcoded 240.0f magic constant with a type-trait specialization (quant_type_max_v<SimFP8E4M3FNuz>) that returns the adjusted max 224.0f (0x7E), avoiding the NaN encoding at 0x7F while keeping identical instruction scheduling and runtime performance

## Key Lessons

- Use type-trait specializations (quant_type_max<T>) instead of raw std::numeric_limits<T>::max() for FP8 types — platform-specific NaN encodings (e.g., 0x7F on ROCm Float8_e4m3fnuz) make the raw max value incorrect for quantization divisors, and the trait centralizes the adjusted value per type
- FP8 quantization constants must be validated per-platform: the IEEE-like max bit pattern (0x7F=240) may encode NaN on some FP8 variants, requiring an adjusted max (0x7E=224) — this is a correctness fix with neutral performance impact since the instruction sequence is identical
