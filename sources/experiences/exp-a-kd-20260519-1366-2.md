## Challenge

The before kernel copies each bf16 element with a separate scalar load and store instruction, underutilizing available memory bandwidth when copying large contiguous regions (512 bf16 elements per nope section)


## Solution

Use inline PTX vectorized 128-bit int4 load/store (ld.global.cs.v4.u32 / st.global.cs.v4.u32) with cache-streaming hints, so each lane copies 8 bf16 values per instruction; two unrolled iterations cover all 512 nope elements, and the .cs hint bypasses L2 allocation for write-once data to reduce cache pollution

## Key Lessons

- Use 128-bit vectorized loads/stores (int4 / ld.global.v4.u32) instead of scalar element access for large contiguous copy operations — each instruction moves 8 bf16 values, achieving 8x fewer memory transactions
- Apply .cs (cache-streaming) PTX cache hints on loads and stores for write-once data patterns to avoid polluting L2 cache with data that will not be reused
- Combine warp-level parallelism with vectorized access: when 32 lanes each execute one int4 load/store per iteration, the warp issues a fully coalesced 256-byte transaction per instruction
