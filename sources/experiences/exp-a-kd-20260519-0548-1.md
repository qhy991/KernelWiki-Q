## Challenge

An f16x8 unpack kernel attempts to improve L2 cache locality by processing 8 half elements per thread instead of 2, but uses scalar __hadd on individual half2 .x/.y components rather than the vectorized __hadd2 intrinsic, so arithmetic throughput per element remains unchanged while structural overheads increase


## Solution

The f16x8 unpack approach widens per-thread processing from 2 to 8 elements by loading 4 half2 pairs and performing 8 scalar __hadd operations, but regresses because it launches 4x fewer thread blocks (reducing SM parallelism), requires 12 half2 temporaries versus 3 (raising register pressure and lowering occupancy), and introduces 4 separate boundary checks that cause warp divergence near array edges

## Key Lessons

- Increasing elements-per-thread without using vectorized intrinsics (__hadd2 for half2) does not reduce instruction count per element — scalar __hadd on .x/.y fields processes one element at a time regardless of how many half2 values are loaded, so the wider load strategy has no arithmetic throughput benefit
- Processing more elements per thread reduces total thread count and block count, which can leave SMs underutilized; always verify that the new grid size still saturates all SMs when widening per-thread work
- More per-thread temporaries increase register pressure and reduce occupancy — the f16x8 kernel needs 12 half2 registers (8 input + 4 output) vs 3 for f16x2, which can halve the number of concurrent warps per SM
