## Challenge

In an 8x4 SGEMM prologue (BM=64, BN=64, 128 threads), B-tile data is loaded from global memory to shared memory using synchronous FLOAT4 vector loads and stores, serializing B loading with A-tile loading and online transpose operations and stalling the warp at each global load


## Solution

Replace synchronous FLOAT4 global-to-shared loads for B with cp.async.ca asynchronous copies: issue two 16-byte CP_ASYNC_CA transfers for the B tile using __cvta_generic_to_shared for the shared-memory pointer, commit the async group, then perform A-tile loading and transpose while the B transfers are in flight, and finally CP_ASYNC_WAIT_GROUP(0) to ensure completion

## Key Lessons

- Use cp.async.ca to overlap global-to-shared memory transfers with independent computation (A-tile loading and transpose) — applies whenever there is non-dependent work available to fill the latency of global memory loads
- Issue CP_ASYNC_COMMIT_GROUP after async copies and CP_ASYNC_WAIT_GROUP(n) before consuming the data to create a structured async pipeline where n controls how many outstanding groups are tolerated
- Use __cvta_generic_to_shared to convert a generic shared-memory address to the cp.async-compatible uint32_t pointer format required by the PTX-level async copy intrinsic
