## Solution

Replace the sort-based pipeline (init_indices + radix sort + build_perms_from_sorted) with a single compute_arg_sorts kernel using atomicAdd on a per-expert counter to scatter tokens directly into pre-counted slots, achieving O(N) work and eliminating all sort-related buffer allocations and memory traffic

## Key Lessons

- For MoE expert routing where only per-expert grouping (not full ordering) is needed, atomicAdd-based scatter into pre-counted slots achieves O(N) work versus O(N log N) radix sort — applies whenever the output requires grouping by category rather than total ordering
- Eliminating temporary buffers (sorted_ids, sorted_orig_idx, sort temp_storage) reduces memory footprint and traffic overhead, which matters more than compute savings for small-to-medium N
- Atomic contention on per-expert counters is moderate when each expert has ~1024 tokens on average (64 experts × 1024 tokens = 65K total), making atomicAdd competitive — at higher per-expert counts, consider hierarchical atomics or warp-level aggregation
