## Solution

Issue weight and input global memory loads BEFORE the spin-wait using instruction-level parallelism: the CUDA memory controller services the loads concurrently with the volatile flag-poll loop, so data arrives in registers/L1 while threads are still spinning, eliminating post-wait load latency entirely

## Key Lessons

- Move independent memory loads before synchronization barriers or spin-waits to exploit instruction-level parallelism — the GPU memory controller services loads concurrently with ALU/control-flow work, hiding load latency for free
- A compiler memory barrier (asm volatile("" ::: "memory")) is needed both to prevent the compiler from sinking prefetch loads past the spin-wait and to prevent hoisting post-wait loads before it — placement determines whether the optimization is preserved
- When a spin-wait duration exceeds the load latency (e.g., 500-cycle P2P wait vs ~30-cycle L1 hit), prefetching is guaranteed to complete before the signal arrives, making it a zero-cost optimization that scales with iteration count
