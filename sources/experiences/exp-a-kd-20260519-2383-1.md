## Solution

Replace per-thread contiguous sharding with stride-based iteration (tid/stride pattern) and inline the index() helper for direct 2D array access; this causes all 32 threads to simultaneously write to different columns of the same shared memory rows during counting, creating more bank conflicts than the sharded approach where each thread owns its own row

## Key Lessons

- Per-thread contiguous sharding (each thread owns a fixed block of elements) produces fewer shared memory bank conflicts than stride-based iteration when threads write to per-thread rows of a 2D count matrix, because each thread's writes are naturally isolated to its own row, reducing simultaneous accesses to the same bank
- Stride-based iteration (tid/stride) improves global memory coalescing for reads but may worsen shared memory bank conflict patterns; when the kernel's bottleneck is shared memory throughput rather than global memory bandwidth, contiguous sharding remains superior
