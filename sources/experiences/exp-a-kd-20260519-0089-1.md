## Challenge

Tile-based shared memory pointer arithmetic omits the tile height multiplier (kRow) when computing row offset, causing byte_pointer to advance by a single row instead of kRow rows per tile step and producing incorrect data access


## Solution

Add the missing kRow multiplier to the row offset term so that byte_pointer advances by kRow * stride_bytes per tile row, correctly landing on the start of the target tile in shared memory

## Key Lessons

- When computing byte offsets for tiled shared-memory access, the row component must include the tile height multiplier (offset_row * kRow * stride) — omitting it advances by a single row per tile step instead of a full tile, producing silently incorrect data without any runtime error
- A single extra integer multiply in address computation is negligible compared to the dominant cost of shared-memory copies and __syncthreads() barriers in micro-kernels, so correctness fixes of this kind have neutral performance impact
