## Challenge

Three stride tensors (a_strides1, b_strides1, c_strides1) for grouped GEMM are initialized via torch::full(), each launching a separate GPU fill kernel to write a single constant value across all expert slots — three redundant kernel launches whose dispatch overhead dominates at small expert counts


## Solution

Replace torch::full() with torch::empty() (zero-cost allocation, no GPU kernel) and fuse the stride writes into the existing __get_group_gemm_starts kernel by passing stride tensors and constant values as additional parameters — each thread writes its expert's stride slot alongside existing pointer offset computation

## Key Lessons

- Replace torch::full() with torch::empty() plus in-kernel writes when filling constant-stride tensors — eliminates per-tensor fill kernel launches whose dispatch overhead is disproportionate to the tiny amount of data written
- Fuse trivial per-element initialization (single constant stores) into an existing single-thread-per-expert kernel rather than launching dedicated fill kernels — the scalar stores are absorbed into the kernel's existing memory traffic at zero marginal cost
- Kernel launch overhead dominates at small problem sizes (8 experts: 2.1x speedup) but is amortized at large scale (256 experts: neutral) — launch fusion is most valuable for high-frequency, small-batch MoE inference workloads
