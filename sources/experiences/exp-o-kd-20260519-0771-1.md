## Challenge

MoE expert-token alignment uses two separate kernels: an align kernel that counts tokens per expert via atomicAdd on shared memory (1024 threads), and a sort kernel that places tokens via atomicAdd on global memory, requiring a cudaMemset between launches and two kernel launch overheads


## Solution

Consolidate two kernels into one single-warp kernel (32 threads) using per-thread private counting arrays in shared memory: each thread increments its own slot (tokens_cnts[(threadIdx.x+1)*num_experts + expert_id]) eliminating all atomics, then a parallel prefix-sum across threads computes cumulative offsets, and threads place tokens directly using their private offset trackers

## Key Lessons

- For small-batch MoE routing (numel ≤ few hundred tokens, few dozen experts), per-thread private counting in shared memory eliminates all atomic contention and enables single-kernel fusion — each thread owns a private count array so increments are conflict-free
- Kernel fusion for small workloads saves disproportionate overhead: one kernel launch + one cudaMemset can dominate runtime when compute is trivial (128 elements), so consolidating two kernels into one yields material speedup even when algorithmic work is unchanged
- Per-thread prefix-sum (each of num_experts threads sums across blockDim.x rows of a column-major layout) replaces the serial cumsum loop with parallel reduction when thread count is small and data fits in shared memory
