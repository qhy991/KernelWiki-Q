## Challenge

Flash Attention output accumulator R_D is stored as FP16 (half), requiring __half2float conversion before each FMA during online softmax rescaling and __float2half_rn rounding after each FMA — 8 conversion instructions per element per K-tile iteration, accumulating overhead across all tile iterations


## Solution

Store the output accumulator R_D as FP32 (4 registers per element instead of 2), enabling direct float FMA during online rescaling without any half<->float conversion; FP16 rounding is deferred to a single final conversion before the collective store, eliminating 8 conversion instructions per element per K-tile iteration

## Key Lessons

- Store MMA output accumulators as FP32 instead of FP16 when they undergo repeated FMA operations (e.g., online softmax rescaling) — the conversion overhead (__half2float + __float2half_rn per FMA) compounds across tile iterations and dwarfs the register cost of FP32 storage
- Defer FP16 rounding to the final store step rather than performing it at every intermediate rescaling — this trades 2× register space per accumulator for elimination of N × 8 conversion instructions where N is the number of K-tile iterations
- For Flash Attention with small head dimensions (d=64), FP32 accumulator storage is affordable within 128 threads — register pressure increase from 2 to 4 registers per element is manageable when kWarpTileHeadDimV is small
