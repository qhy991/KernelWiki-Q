## Solution

Widening from 32-bit (uint32_t) to 128-bit (uint4) vectorized load/store processes 8 half elements per iteration instead of 2, but increases local array size (vals[8] vs vals[2]) and register pressure without reducing global memory transactions proportionally when the kernel is compute-bound due to expf

## Key Lessons

- Widening memory access width (32-bit to 128-bit) does not improve performance when the kernel is compute-bound (e.g., expf-dominated) — the wider load/store only helps in memory-bound regimes where transaction count is the bottleneck
- Larger vectorized access increases local array sizes (__half vals[8] vs vals[2]) which raises register pressure and can reduce occupancy, causing net slowdown even when memory throughput theoretically improves
- 128-bit epilogue alignment in CUTLASS-style kernels only delivers benefit when the epilogue writes multiple output tensors (D, N, S, Softmax) simultaneously in a memory-bound visitor pattern, not in isolated single-output kernels
