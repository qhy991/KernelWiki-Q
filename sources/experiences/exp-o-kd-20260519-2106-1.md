## Challenge

Persistent tile scheduler launches only total_tiles blocks for tall-and-skinny GEMMs (K >> N), resulting in severe SM underutilization — 8 blocks on a 108-SM A800 leaves ~93% of SMs idle while each block computes the entire K=4096 dot product serially


## Solution

Replace persistent tile scheduling with StreamK-style work distribution: launch grid_size = num_sms blocks and divide all M×N output elements evenly across them, ensuring full SM occupancy regardless of how few output tiles the GEMM shape produces

## Key Lessons

- For tall-and-skinny GEMMs (K >> N), persistent tile scheduling produces fewer blocks than SMs, leaving hardware idle — StreamK-style element-level work distribution fills all SMs by assigning output elements directly to num_sms blocks regardless of tile geometry
- Decompose work at the output-element granularity rather than tile granularity when the number of natural tiles is small — this ensures full SM occupancy for any GEMM shape
- Launch grid_size = num_sms (queried via cudaDevAttrMultiProcessorCount) instead of total_tiles to guarantee all SMs receive work in irregular problem shapes
