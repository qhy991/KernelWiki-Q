## Challenge

Scale shared-memory read index in Marlin kernel uses warp_row % 2 unconditionally, which produces incorrect scale grouping when group_blocks=2 (MXFP4 config): rows 0→0, 1→1, 2→0, 3→1 instead of grouping rows 0,1→0 and rows 2,3→1


## Solution

Replace warp_row % 2 with (warp_row / group_blocks) % 2 so that adjacent warp rows within the same quantization group share the same scale index, producing correct grouping for group_blocks=2 while remaining equivalent for group_blocks=1

## Key Lessons

- When scale indexing depends on quantization group size (group_blocks), the row-to-scale mapping must divide by group_blocks before modulo to correctly group adjacent rows — a hardcoded modulo without division assumes group_blocks=1 and silently produces wrong results for larger group sizes
- Integer division by group_blocks adds negligible latency (single SASS instruction) — correctness fixes in index arithmetic rarely affect performance because the computation is integer and already a tiny fraction of kernel time
