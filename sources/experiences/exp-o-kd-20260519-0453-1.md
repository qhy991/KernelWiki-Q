## Solution

Process 2 half elements per thread using half2 vector type: a single 32-bit load fetches two contiguous half values, swish_half is applied to each component independently, and a single 32-bit store writes both results, halving memory instruction count and thread count

## Key Lessons

- Replace scalar half element processing with half2 vectorized processing to pack two 16-bit values into one 32-bit transaction matching GPU minimum transaction width — applies to any elementwise activation kernel operating on fp16 data where operations are lane-independent
- Vectorized half2 access halves the thread count for a given problem size, reducing scheduling overhead and improving occupancy for memory-bound elementwise kernels
- Vectorization overhead may not amortize for very small problem sizes (N=1M showed regression to 0.881x) — verify speedup across target problem sizes before deploying
