## Solution

Replace the nested block-then-token iteration with direct per-token address computation using a token_to_seq mapping array: each CTA handles one token, looks up its batch_id, computes block_table_id and slot_id from the token's batch-relative offset, and performs a single block_table lookup to resolve the cache address — eliminating the outer block loop and all split/partial-block boundary logic

## Key Lessons

- Replace nested block-then-token loops over paged KV-cache with direct per-token address computation (block_table_id = offset / block_size, slot_id = offset % block_size) — reduces block_table lookups from O(blocks_per_split) per CTA to O(1) and eliminates divergent partial-block handling
- Use a token_to_seq mapping array to decouple token iteration order from batch/block structure, enabling one-CTA-per-token parallelism where each thread independently resolves its cache address via integer division and modulo on its batch-relative offset
