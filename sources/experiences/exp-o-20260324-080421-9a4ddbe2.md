## Challenge

Tried to improve beyond 237.8x speedup for RMS norm with hidden_size=7168

## Key Lessons

- For RMS norm with hidden_size=7168, 256 threads with __launch_bounds__(256,4) is near-optimal
- Reducing to 128 threads hurts memory-level parallelism even with higher occupancy
- Caching 28KB weight in shared memory limits occupancy to 1 block/SM - too costly
- Processing 2 rows per block with split warps adds reduction complexity without benefit
- The 256-thread, 4-blocks/SM sweet spot balances occupancy and per-block parallelism for this workload size
