## Challenge

MoE token-sorting kernel runs in a single block with 64 threads using a 3-phase shared-memory approach (per-thread init+count, cross-thread prefix-sum, scatter), requiring two __syncthreads() barriers and a shared-memory count matrix of shape (num_threads+1)×num_experts to process 2048 tokens


## Solution

Replace the 3-phase shared-memory single-block kernel (64 threads) with a single-pass multi-block kernel (8 blocks × 256 threads = 2048 threads) where each thread claims its output rank via atomicAdd on the global cumsum buffer and writes its token index in one loop iteration, eliminating all __syncthreads() barriers and shared-memory management

## Key Lessons

- Replace multi-phase shared-memory sort (init+count, prefix-sum, scatter) with a single-pass atomicAdd on global cumsum when contention is low — the 64 expert bins for 2048 tokens yield ~32 tokens per bin on average, making atomicAdd contention negligible while eliminating two __syncthreads() barriers and all shared-memory bookkeeping
- Scale from single-block to multi-block (64 threads → 2048 threads) for token-level parallelism where each token's work is an independent atomic claim + write — the natural granularity is one thread per token, not one thread processing many tokens sequentially
