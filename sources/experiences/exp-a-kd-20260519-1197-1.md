## Solution

Split the single fused loop into two independent loops: one iterating over num_heads*embed_dim for query rotation and another over num_kv_heads*embed_dim for key rotation, eliminating the divergent conditional branch at the cost of two loop setups and 50% thread utilization in the smaller key loop (256/512 threads active)

## Key Lessons

- Splitting a fused GQA loop into separate query/key loops to eliminate warp divergence does not always improve performance — when the divergent branch reconverges quickly (per iteration) and the separate key loop leaves threads idle (50% utilization for 8 KV heads / 512 threads), the two-loop overhead can offset or exceed the divergence savings
- Warp divergence from a conditional guard inside a tight loop is relatively cheap when the reconvergence point is the next iteration — the warp scheduler recovers quickly, making branch elimination via loop splitting a trade-off rather than a guaranteed win
