## Solution

Add a kFactor==8 branch that computes factor_in_partition from PartitionShape/TileShape ratios, then derives partition_contiguous_idx via lane_in_quad_pair division, access_contiguous_idx via XOR of lane_in_quad with lane_id/kFactor, and access_strided_idx via lane_id/kFactor to produce correct bank-conflict-free swizzle addresses for 8-element interleaved crosswise layouts

## Key Lessons

- When extending CUTLASS MmaTensorOpMultiplicandTileIterator to support new kFactor values, the lane-to-address branch chain must cover every valid kFactor — missing branches leave indices at -1 and silently produce invalid byte offsets that corrupt shared memory access rather than failing visibly
- The kFactor==8 crosswise lane mapping uses XOR between lane_in_quad and lane_id/kFactor for access_contiguous_idx to create a bank-conflict-free swizzle pattern, distinct from the simpler division-based mappings used at kFactor<=4
- Correctness fixes in address computation can be latency-neutral (1.013x) because the new branch replaces the same number of arithmetic instructions — the value is in producing valid memory offsets, not in reducing instruction count
