## Solution

Reduce TILE_M from 128 to 32 to better match the actual M=16 problem dimension, cutting per-thread accumulators from 64 to 16 int32 values (~36 registers total), which raises occupancy from ~3 to ~7 blocks/SM and shrinks shared memory from 8800 to 5344 bytes per block

## Key Lessons

- Tuning tile dimensions to match actual problem sizes eliminates wasted registers and computation — when TILE_M far exceeds M, most accumulator slots and shared memory rows serve only padding, directly throttling occupancy via register pressure
- Register pressure is often the primary occupancy bottleneck for int8 GEMM tiling: 64 int32 accumulators per thread (~80 regs) limits to ~3 blocks/SM, while reducing to 16 accumulators (~36 regs) allows ~7 blocks/SM for dramatically better latency hiding
- Shared memory footprint scales with tile dimensions and compounds the occupancy limit — shrinking TILE_M from 128 to 32 cuts smem per block from 8800 to 5344 bytes, freeing space for more concurrent blocks within the 48–164 KB shared memory ceiling
