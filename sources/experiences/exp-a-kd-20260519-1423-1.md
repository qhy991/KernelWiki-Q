## Challenge

Scalar rms_norm kernel with 1024 threads achieves good memory coalescing for half-precision; vectorizing with vec_n_t<half,8> reduces thread count to 512, halving warp-level parallelism and memory-level parallelism in both variance and normalization phases


## Solution

Replace scalar half loads/stores with vec_n_t<half,8> (128-bit) vectorized access in both variance and normalization phases, reducing block size from 1024 to 512 threads — the 128-bit vector loads do not compensate for the 2x reduction in warp-level memory request throughput

## Key Lessons

- Vectorized loads that reduce thread count can degrade performance when the original thread count already provides sufficient memory coalescing — each warp issuing scalar half accesses on a contiguous range achieves full coalescing on modern GPUs, so widening to 128-bit vectors while halving threads removes parallelism without gaining throughput
- When blockDim.x is already large enough to cover the working set (1024 threads × 1 element each = 1024 elements per iteration, iterating to cover hidden_size), vectorization that shrinks the thread count proportionally reduces memory-level parallelism and harms shared-memory reduction efficiency
- Vectorization in rms_norm is beneficial only when it preserves or increases total memory request throughput — prefer keeping thread count constant and using vectorized loads to cover more elements per iteration, rather than reducing threads to compensate for wider loads
