## Solution

Replace the custom blockReduceSum template and its runtime conditional with a single cub::BlockReduce<float, 1024> call passing blockDim.x as the valid element count, letting CUB handle partial-block reduction internally and eliminating the hand-rolled warp-shuffle plus shared-memory staging code

## Key Lessons

- Replacing a hand-rolled block reduction with cub::BlockReduce simplifies code and eliminates runtime conditionals without hurting performance — CUB's internal valid-count handling adds negligible overhead compared to the memory-bound loops that dominate kernel execution time
- A runtime branch selecting between blockReduceSum template instantiations does not cause warp divergence when all threads in a block evaluate the same condition, but it still complicates code — prefer a single CUB reduction with valid-count for clarity and maintainability
- For memory-bound kernels like fused RMSNorm with hidden_size=4096, the reduction step contributes a small fraction of total latency; optimizing it yields near-neutral speedups (~0.2%) while code simplification remains valuable
