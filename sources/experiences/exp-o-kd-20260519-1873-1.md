## Challenge

The RMS norm kernel uses a naive serial block reduction where thread 0 iterates all 512 shared-memory entries in a sequential loop (511 serial additions), forcing the remaining 511 threads to idle while incurring shared-memory bank conflicts during the serial accumulation


## Solution

Replace the serial reduction with a two-phase warp-shuffle blockReduceSum: each warp reduces its 32 partial sums via __shfl_xor_sync in 5 parallel register-level steps (no shared memory traffic), then the first warp reduces the 16 warp-level sums via another shuffle reduction, replacing 511 serial shared-memory additions with 9 parallel shuffle steps

## Key Lessons

- Warp-shuffle reductions (__shfl_xor_sync) reduce values entirely in registers with O(log warpSize) steps, eliminating shared-memory traffic and bank conflicts — prefer over serial single-thread accumulation whenever block-level reduction is needed
- Two-phase block reduction (warp-level shuffle then cross-warp via minimal shared memory) scales to any blockDim.x with only ceil(blockDim.x/32) shared-memory slots instead of blockDim.x slots, reducing SMEM pressure
- Serial single-thread reduction is a dominant bottleneck at block sizes ≥256 — the 2.69x measured speedup confirms that parallelizing the reduction phase alone can account for the majority of kernel runtime in reduction-bound kernels like RMS norm
