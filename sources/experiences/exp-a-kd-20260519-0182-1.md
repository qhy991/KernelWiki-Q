## Solution

Adopt the CUTLASS 3.0 device_kernel pattern: wrap computation in an Operator struct with compile-time launch-bounds constants, annotate the kernel template with __launch_bounds__(256, 2) and __grid_constant__ for the params, and replace the typed SharedStorage* cast with a plain char* smem parameter that is reinterpret_cast inside the operator

## Key Lessons

- Applying __launch_bounds__ and __grid_constant__ to a memory-bandwidth-bound kernel with a simple body (few registers, no spill pressure) yields negligible improvement — the compiler already stays within the register budget and NVCC already routes parameter loads through the read-only cache on CUDA 12.x
- When a kernel's dominant cost is global-memory traffic (one load + one store per element), micro-optimizations in parameter passing or register allocation are hidden by the memory bottleneck; focus optimization effort on memory coalescing and bandwidth utilization instead
- The CUTLASS 3.0 device_kernel pattern (Operator struct + __launch_bounds__ + __grid_constant__ + char smem) is primarily a code-organization and portability improvement — its performance benefit is real but only materializes for register-heavy kernels or kernels where constant-cache parameter reuse matters
