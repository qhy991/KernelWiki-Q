## Solution

Replace __shfl_sync warp-index broadcast with a __forceinline__ device function canonical_warp_idx() that returns threadIdx.x / 32, allowing each thread to compute its warp index locally without any warp barrier or cross-lane communication

## Key Lessons

- Prefer direct arithmetic (threadIdx.x / 32) over __shfl_sync for computing warp_idx when each thread can derive its own value independently — __shfl_sync introduces an unnecessary warp-level barrier and register dependency for a trivially computable quantity
- In memory-bound kernels, single-instruction overhead differences (one __shfl_sync vs one IMAD/shift) are undetectable at the kernel level because latency is dominated by memory traffic and __syncthreads barriers — the optimization is structurally cleaner but not performance-relevant in this regime
