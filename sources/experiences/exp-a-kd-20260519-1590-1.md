## Challenge

Naive GEMM always launches with a hardcoded 16x16 block configuration regardless of problem dimensions (M, N, K), missing the opportunity to tailor block shape for different workload regimes


## Solution

Add problem-size-aware dispatch logic that selects block dimensions based on M/N/K thresholds: wide matrices (N>=8192) use 32x8, tall-skinny problems (M<=16) use 16x4, with 16x16 as the default fallback

## Key Lessons

- Problem-size-aware block dimension dispatch can improve occupancy for extreme aspect ratios (very wide N or very small M), but for medium problem sizes (M=64,N=4096,K=4096) the default 16x16 configuration is already reasonable and dispatch adds no benefit
- Dispatch overhead is negligible (host-side branching), but the optimization only pays off when the selected block shape meaningfully changes grid occupancy or shared memory utilization — verify with profiling for each target problem size regime
