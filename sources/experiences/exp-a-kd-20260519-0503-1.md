## Challenge

FlashAttention MMA kernel with kStage=1 uses synchronous single-stage async copy, issuing CP_ASYNC_WAIT_GROUP(0) immediately after loading Q/K tiles and calling __syncthreads() before MMA compute, which exposes full global memory latency and starves tensor cores while waiting for each tile transfer to complete


## Solution

Switch to kStage=2 double-stage async copy pipelining where next Q/K/V tiles are prefetched into an alternate shared memory stage (smem_sel_next) via CP_ASYNC_CG while MMA computation proceeds on the current stage; the deferred CP_ASYNC_WAIT_GROUP(kStage-2) after MMA compute overlaps global memory transfers with tensor core work, hiding gmem latency across 8 K/V tile iterations (seq_len=512, Bc=64)

## Key Lessons

- Double-stage async copy pipelining (kStage=2) overlaps CP_ASYNC_CG tile prefetch into an alternate shared memory stage with MMA computation on the current stage, hiding global memory latency — applies when the kernel has enough tile iterations to amortize preload overhead
- Deferring CP_ASYNC_WAIT_GROUP(kStage-2) until after MMA compute (rather than issuing CP_ASYNC_WAIT_GROUP(0) immediately) allows the async copy engine to transfer data concurrently with tensor core work, converting a serial load-compute dependency into a pipelined parallel pattern
