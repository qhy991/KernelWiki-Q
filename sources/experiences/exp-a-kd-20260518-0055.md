## Solution

Increased kFragmentsPerIteration from 1 to 2 to halve the number of __syncthreads() barriers from 16 to 8, but this doubled shared memory per block from 34KB to 68KB, reducing occupancy from 4 to 2 concurrent blocks per SM and causing a net performance regression of 0.421x

## Key Lessons

- Reducing __syncthreads() barrier count by batching more fragments per iteration can regress performance when the extra shared memory reduces SM occupancy — always model the occupancy impact of shared memory size changes before committing to fewer sync barriers
- Lightweight per-element kernels (e.g. float-to-half conversion) have insufficient arithmetic intensity to hide occupancy loss; sync barrier reduction is only beneficial when per-fragment compute or memory work is heavy enough to dominate, as in real CUTLASS warp-tile iterators with bank-conflict-aware strided access patterns
- In isolated microbenchmarks, shared memory footprint is the primary occupancy limiter; doubling smem from 34KB to 68KB can halve the number of resident blocks per SM, overwhelming any gain from fewer barriers
