## Challenge

V gmem-to-smem loads in the P@V section use synchronous CP_ASYNC with immediate CP_ASYNC_WAIT_GROUP(0) + __syncthreads() per V tile, serializing load completion with P@V MMA compute and preventing latency hiding


## Solution

Prefetch V tiles before P@V begins using a prologue loop, then issue async copies for the next V tile into smem_sel_v_next (alternate buffer) while current P@V MMA runs, with deferred CP_ASYNC_WAIT_GROUP(kStage-2) overlapping V load latency with P@V computation

## Key Lessons

- Apply the same double-buffer async pattern to V loads in the P@V stage as used for Q/K loads in the Q@K^T stage — the prologue-prefetch-deferred-wait structure applies uniformly to any gmem-to-smem transfer in a pipelined kernel
- V shared memory can reuse Q's smem region (V_tile_smem = Q_tile_smem) since Q is no longer needed after Q@K^T completes, but with kStage=2 the double-buffered V smem still needs 2x V_tile_size to avoid overwriting the in-use buffer
- Place CP_ASYNC_WAIT_GROUP(kStage-2) after the MMA compute (not before s2r) to maximize the overlap window — the async copy runs concurrently with both s2r and MMA, only synchronizing when the compute result is needed for the next iteration
