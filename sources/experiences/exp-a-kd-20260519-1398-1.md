## Challenge

GELU activation uses the exact erf-based formulation (::erf) inside a fused activation-and-multiply kernel, but the choice of erf versus tanh approximation has unclear performance implications on GPU SFU hardware


## Solution

Replace erf-based GELU with tanh-approximation GELU (GELU(x) = 0.5x(1 + tanh(beta(x + kappa*x^3)))), which adds extra multiply-add arithmetic (x_cube, inner product) but uses tanhf instead of erf — both are SFU microcode on sm_80 with comparable latency

## Key Lessons

- Both ::erf and ::tanhf are SFU microcode on sm_80 (A800) with comparable instruction latency — swapping between them yields negligible speedup when the extra arithmetic in the tanh path offsets any SFU advantage
- For memory-bandwidth-bound fp16 kernels (load 2 half, store 1 half per element), the activation compute cost is small relative to the data volume, making the SFU function choice invisible in end-to-end latency — optimize memory access patterns instead
