## Solution

Remove the epilogue __syncthreads() since the final mainloop iteration already contains a __syncthreads() that synchronizes all threads before the epilogue write, making the extra barrier redundant; the measured speedup is only 1.004x because with 32 mainloop iterations the single removed barrier is ~1/33 of total sync overhead

## Key Lessons

- Removing a redundant __syncthreads() after the last mainloop iteration is safe because the loop's own __syncthreads() already guarantees thread convergence and shared memory consistency before the epilogue — applies whenever the mainloop unconditionally synchronizes on its final iteration
- A single removed barrier has negligible measurable impact when the kernel contains many mainloop iterations (N iterations → the removed barrier is ~1/(N+1) of total sync overhead), so epilogue barrier optimization matters most for short-K kernels or when many small GEMM tiles are launched
- The real CUTLASS optimization targets cp_async_wait<0> alongside __syncthreads — the async pipeline drain wait is more expensive than a plain barrier because it stalls until all in-flight cp.async transactions complete, so microbenchmarks using synchronous loads underestimate the true speedup
