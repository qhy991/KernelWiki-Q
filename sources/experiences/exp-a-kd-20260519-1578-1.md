## Challenge

V-cache read kernel stores values in fp16 (2 bytes/element), consuming high memory bandwidth for long sequences (2048 tokens, 128 blocks), but the kernel is actually compute-bottlenecked by shared-memory atomicAdd rather than DRAM traffic


## Solution

Replace fp16 V-cache storage with FP8-E5M2 (1 byte/element) to halve memory traffic, adding software dequantization via integer shift/mask/or to reconstruct fp16 bit fields from FP8-E5M2 representation

## Key Lessons

- Software FP8-to-fp16 dequantization (integer shift/mask/or) adds per-element compute overhead that can exceed the memory bandwidth savings from halved traffic — quantify both compute cost and bandwidth savings before adopting lower-precision storage
- On sm_80 hardware without native FP8 instructions, FP8 quantization is counterproductive for compute-bottlenecked kernels; use native hardware intrinsics (__nv_cvt_fp8_to_halfraw on sm_89+ Hopper) to eliminate software conversion overhead
- Profile whether a kernel is memory-bound or compute-bound before applying compression techniques — shared-memory atomicAdd contention can shift the bottleneck away from DRAM, making bandwidth-saving optimizations ineffective
