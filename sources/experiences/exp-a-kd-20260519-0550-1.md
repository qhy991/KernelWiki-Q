## Solution

Replace scalar float load/store with float4 vectorized load/store so each thread processes 4 elements, reducing total memory transactions and thread count by 4x via a single 128-bit load and store per thread

## Key Lessons

- float4 vectorization reduces memory transaction count and thread launch overhead by 4x, but yields neutral speedup when the working set (~4MB for 1M FP32 elements) fits entirely in L2 cache and scalar access is already coalesced — the benefit materializes at larger problem sizes that exceed cache capacity
- Scalar FP32 access with consecutive threadIdx values already achieves full coalescing, so vectorization improves instruction throughput rather than coalescing efficiency for memory-bound elementwise kernels
