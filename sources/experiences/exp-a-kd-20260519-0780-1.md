## Challenge

MLA key concatenation kernel uses scalar bf16 reads and writes (one 2-byte element per transaction), generating 96 scalar 2-byte memory transactions per lane per head-chunk and leaving significant memory bandwidth underutilized despite partial coalescing from adjacent lane addresses


## Solution

Replace scalar bf16 element-wise reads/writes with vectorized int2 (128-bit, 8 bf16 values) loads for the NOPE head dimension and int (32-bit, 2 bf16 values) loads for the ROPE dimension, combined with local register buffers to decouple the load and store phases into separate fully-coalesced bulk transfers

## Key Lessons

- Replace scalar bf16 element reads/writes with vectorized int2 (128-bit) or int (32-bit) reinterpret_cast loads/stores when each warp lane naturally owns a contiguous chunk of the head dimension — reduces memory transaction count by 4-8x and ensures fully coalesced access
- Decouple load and store phases with local register buffers when the same data is read from non-contiguous source tensors and written to a concatenated destination — enables each phase to use optimal vectorized widths independently
- Partial memory coalescing from adjacent lane addresses can mask scalar-access inefficiency at larger data volumes (where L2 prefetching helps), but vectorized access maintains consistent bandwidth utilization regardless of data size
