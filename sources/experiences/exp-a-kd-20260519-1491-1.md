## Challenge

Scalar absmax reduction loop already achieves naturally coalesced 4-byte float accesses across a 256-thread block; wrapping it in a vectorize_read_with_alignment<16> layer adds reinterpret_cast overhead, register pressure from vec_n_t<float,16> (64-byte struct), and extra function-call indirection in DefaultReadVecOp without any output-write benefit to amortize the cost


## Solution

Replace scalar strided loop with vectorize_read_with_alignment<16> that loads 16 packed floats via vec_n_t<float,16> reinterpret_cast, then decomposes back into 16 individual scalar lambda calls inside an unrolled DefaultReadVecOp — but this layered vector-load-then-decompose pattern adds overhead for a read-only reduction where scalar loads are already coalesced

## Key Lessons

- For read-only reductions (absmax, sum, etc.) where threads already access contiguous memory with uniform stride, scalar loads are naturally coalesced by the GPU hardware — vector loads add reinterpret_cast and type-conversion overhead without improving throughput
- Vectorize-then-decompose patterns (load vec_n_t<T,16> then unroll 16 scalar ops) are functionally equivalent to plain scalar iteration but with extra register pressure and function-call indirection; they only help when the vector load feeds a truly vectorized compute or output path
- Alignment-checking infrastructure (vectorize_read_with_alignment) is valuable for read-write kernels where vectorized stores amortize the overhead, but pure reductions have no output writes to benefit from
