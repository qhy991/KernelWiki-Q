## Challenge

Passing v_in[i] directly into a templated callable vec_op causes NVCC to emit scalarized narrow loads instead of a single wide vector load, because the compiler cannot see that the entire pack should be loaded atomically when the callable iterates over its internal scalar array


## Solution

Materialize the vector pack into a local variable (vin_t tmp = v_in[i]) before passing it to vec_op, forcing NVCC to emit a single wide load instruction (e.g. LDG.128 for a 16-element pack) rather than potentially VEC_SIZE narrow scalar loads

## Key Lessons

- Materialize vector packs into local variables before passing to templated callables — the explicit copy forces a single wide vector load (e.g. LDG.128) and prevents NVCC from scalarizing the global memory access when the callable iterates over the pack's internal array
- Scalarization via local copy benefits wider packs more: 16-element (32-byte) packs see 1.10x speedup while 4-element (8-byte) packs see 0.87x regression — apply this pattern when pack width exceeds the scalarization threshold (roughly 16 bytes+)
