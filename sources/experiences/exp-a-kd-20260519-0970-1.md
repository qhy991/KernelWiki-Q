## Challenge

Per-token group FP8 quantization kernel launches 4096 blocks with only 16 threads (1 warp) each, severely underutilizing SM capacity since each 16-thread block contributes only 0.5 warp per SM, leaving warp schedulers starved during memory-bound absmax reduction and quantization phases


## Solution

Coarsen the kernel by mapping 16 independent groups into a single block with 256 threads (8 warps), using 2D shared memory s_absmax[16][17] with padded stride 17 to avoid bank conflicts, and computing local_group_id = tid/16 and local_tid = tid%16 to partition threads among groups within the block

## Key Lessons

- Block coarsening — consolidating multiple independent work items into a single block — dramatically improves occupancy when individual items use too few threads per block, because each SM scheduler needs multiple warps to hide memory latency effectively
- Use padded shared memory stride (e.g., [16][17] instead of [16][16]) when multiple independent groups write to adjacent rows, to prevent bank conflicts during parallel warp reductions
- When per-group computation is short (small group_size), fixed per-block overhead (shared memory allocation, scheduling, context) dominates; coarsening amortizes this cost across multiple groups within one block
