## Challenge

MoE token-sorting is embedded inside a single-block (1024 threads) kernel using shared-memory local_offsets[256] with atomicAdd, causing severe thread contention as 1024 threads compete for only 256 shared-memory locations when distributing tokens to experts


## Solution

Decouple the token sort into a dedicated multi-block kernel (128 blocks × 256 threads) using a grid-stride loop with global-memory atomicAdd on cumsum_buffer, distributing work across all SMs and reducing contention by leveraging L2 cache for the 257-entry buffer instead of shared memory

## Key Lessons

- Decouple heavy-contention atomic operations into a separate multi-block kernel to utilize all SMs — a single-block kernel leaves most of the GPU idle and concentrates shared-memory atomics on one SM
- Use global-memory atomicAdd on small hot buffers (≤257 entries) instead of shared-memory atomics when scaling beyond one block — the L2 cache effectively absorbs the traffic for compact buffers, and the wider parallelism outweighs the higher per-atomic latency
- Grid-stride loops (tid + stride pattern) provide load-balanced input coverage across arbitrary grid sizes, making the kernel portable across different GPU architectures without recalculating block counts
