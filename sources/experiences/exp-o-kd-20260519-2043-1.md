## Solution

Use adaptive max_block_size based on batch size: 256 threads per block for large batches (num_tokens >= 256) and 1024 for small batches. Smaller blocks allow 2x more blocks to reside on each SM simultaneously, increasing occupancy and memory-level parallelism for bandwidth-bound global memory operations

## Key Lessons

- For bandwidth-bound kernels with many independent blocks (large batch), prefer smaller block sizes (e.g. 256) over larger ones (512/1024) — the resulting higher occupancy lets more blocks run concurrently per SM, improving memory-level parallelism and latency hiding
- Adaptive block sizing based on grid dimensions (num_tokens) is preferable to fixed max_block_size because the optimal thread count depends on the ratio of available SMs to total work, not just hidden_size
