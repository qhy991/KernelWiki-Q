## Solution

Adding a conditional mask parameter (token_mask) with a runtime null-pointer ternary check to support both masked and unmasked MoE expert counting in a single kernel, but the overhead of the per-iteration null comparison, extra parameters, and retained dead code causes a 26.8% regression in the nullptr-only path

## Key Lessons

- Fusing a conditional mask path into a hot inner loop via a runtime null-pointer ternary adds per-iteration comparison overhead and extra kernel parameters — for tight loops with small element counts, this overhead is measurable and should be avoided by using separate kernels or template specialization instead
- Extra kernel function parameters increase register pressure and parameter-passing cost even when unused in a given call path — prefer kernel templates or separate overloads over runtime branching on pointer nullity for performance-critical inner loops
- A ternary null-pointer check (ptr == nullptr ? A : B) cannot be fully optimized away by the compiler because it must preserve the non-null code path in the binary, affecting instruction cache utilization in tight loops
