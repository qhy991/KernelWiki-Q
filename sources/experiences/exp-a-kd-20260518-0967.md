## Challenge

The before kernel writes 2 scalar global stores per block (reverse_map and permuted_idx) via threadIdx.x==0, while the after kernel eliminates these entirely. However, for 8192 destination rows the total side-effect store volume is only ~64KB


## Solution

(2 x 4 bytes x 8192), which is negligible compared to the ~64MB of fp16 row data copied (8192 x 4096 x 2 bytes). The 128-bit vectorized load/store loop dominates kernel runtime; the two scalar writes by thread 0 are hidden by the memory subs

## Results

- **speedup**: 0.995x (neutral)


## Key Lessons

- The before kernel writes 2 scalar global stores per block (reverse_map and permuted_idx) via threadIdx.x==0, while the after kernel eliminates these entirely.
- The 128-bit vectorized load/store loop dominates kernel runtime; the two scalar writes by thread 0 are hidden by the memory subsystem's parallelism and are not on the critical path.
