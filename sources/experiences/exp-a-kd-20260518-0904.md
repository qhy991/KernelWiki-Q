## Challenge

## Solution

The optimization helps BF16 significantly because the scalar BF16 path requires a costly convert-to-float/add/convert-back sequence per element, but for FP16 where scalar add is already native, packed operations provide no measurable advantage in this arithmetic-intensive microbenchmark.

## Key Lessons

- The packing/unpacking sequence (__halves2half2 → __hadd2 → __low2half/__high2half) adds register manipulation overhead that the scalar loop avoids.
