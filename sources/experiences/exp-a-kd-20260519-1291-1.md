## Challenge

The host-side int4 nibble-remap encodes 144MB of weight data by copying it from GPU to CPU via PCIe, running a serial byte-level loop, then copying the result back to GPU — incurring two synchronous ~144MB PCIe transfers and serial CPU processing for an embarrassingly parallel elementwise operation


## Solution

Replace the CPU round-trip with an on-device GPU kernel that uses a 256-byte precomputed lookup table uploaded to __constant__ memory and processes data with uint4 (16-byte) vectorized loads, keeping all 144MB on-device and leveraging ~900 GB/s HBM bandwidth instead of ~12 GB/s PCIe

## Key Lessons

- Eliminate GPU↔Host PCIe round-trips for data-parallel transforms — keep data on-device and launch a GPU kernel instead of copying to CPU for serial processing, since PCIe bandwidth (~12 GB/s) is ~75x slower than HBM bandwidth (~900 GB/s)
- Use __constant__ memory for small frequently-accessed lookup tables (≤4KB) — constant cache provides low-latency broadcast reads to all threads in a warp when they access the same table
- Use uint4 (16-byte) vector loads/stores for byte-level elementwise kernels — vectorized memory access amortizes instruction overhead and increases effective memory throughput by processing 16 bytes per load instruction
