## Challenge

LSE (log-sum-exp) arrays use head-major layout where stride between consecutive heads equals num_tokens (1024 floats = 4096 bytes), causing threads with consecutive head_idx within the same token to access addresses far apart and preventing coalesced memory transactions


## Solution

Change LSE layout from head-major (head_idx * num_tokens + token_idx) to token-major (token_idx * num_heads + head_idx), reducing stride from 1024 to 32 (128 bytes) so consecutive threads accessing different heads read contiguous addresses — however the measured impact is neutral because LSE arrays are only 128KB (fit in L2 cache) and represent a tiny fraction of total memory traffic vs the 128-bit packed value loads

## Key Lessons

- Memory layout optimization (head-major to token-major) for small arrays that fit entirely in L2 cache (128KB << 40MB) may show negligible improvement because scattered reads hit L2 cache lines rather than issuing expensive DRAM transactions — applies when target array size is well below L2 capacity
- When evaluating memory layout changes, consider the target array's share of total memory traffic — here each thread reads only 2 scalar floats from LSE versus 16 packed half values (32 bytes) via uint4 loads for attention values, making LSE accesses an insignificant fraction of overall bandwidth regardless of coalescing quality
