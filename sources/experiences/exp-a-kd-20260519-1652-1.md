## Challenge

Fixed CUTLASS TileShape 128x128x64 wastes 87.5% of M-dimension compute when M=16 (only 16 of 128 tile rows contain valid data), and the large N-tile (128) limits concurrency to just 16 threadblocks along N, starving SMs of work


## Solution

Switch to M-adaptive TileShape 16x64x128 with WarpShape 16x64x64 so that the tile M dimension exactly matches M=16, the smaller N-tile (64) doubles threadblock count from 16 to 32 for higher occupancy, and the larger K-tile (128) amortizes tensor core pipeline startup cost across more iterations

## Key Lessons

- Match TileShape.M to the actual batch/problem M dimension when M is small — oversized M tiles waste tensor core cycles on masked-out rows, yielding near-zero useful compute per tile
- Reduce TileShape.N (e.g., 128→64) for small-M GEMMs to increase threadblock count along N, boosting SM occupancy and hiding memory latency when per-threadblock work is already small
- Increase TileShape.K (e.g., 64→128) when reducing M and N tiles, to amortize the tensor core pipeline startup overhead across more K iterations per threadblock
