## Challenge

Rotary embedding is applied across 8 LoRA adapter groups via 8 separate kernel launches, each with only 8 thread blocks, severely underutilizing 108 SMs and incurring 8x kernel launch overhead including driver-side costs and GPU idle gaps between small launches


## Solution

Consolidate 8 per-group kernel launches into a single batched_rotary_embedding_kernel with 64 thread blocks, adding a per-token cos_sin_cache_offsets array to handle group-specific cache base addresses, amortizing launch overhead to once instead of 8 times and improving SM utilization

## Key Lessons

- Batch multiple small kernel launches into a single launch with more thread blocks when work items are independent — eliminates per-launch driver overhead and fills idle SMs, yielding dramatic speedups when individual launches have too few blocks to saturate the GPU
- Use a per-token offset array (cos_sin_cache_offsets) to generalize per-group cache base addresses into a batched kernel, enabling a single launch to handle heterogeneous groups without replicating the group-loop logic on the host
- Small kernel launches (e.g., 8 blocks on a 108-SM GPU) are dominated by launch overhead rather than compute — consolidating into fewer larger launches is preferred even if it requires a minor indirection for per-item metadata
