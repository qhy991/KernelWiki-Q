## Solution

Rematerialize thread IDs between pipeline stages via a lambda that recomputes warp_id and lane_id from a single base thread_id using __shfl_sync-based warp_uniform_func, freeing registers for accumulators — but this only wins when register pressure exceeds the 255-register limit causing spills; otherwise the extra shfl_sync calls add net overhead

## Key Lessons

- Rematerializing cheap integer thread IDs from a single base register is only beneficial when register pressure exceeds the hardware limit (255 regs) and forces spills to local memory — otherwise the recomputation overhead (extra __shfl_sync calls) is a net loss
- Rematerialization benchmarks must replicate the actual register pressure of the target kernel; a synthetic kernel that fits within 255 registers will not exhibit the spill-avoidance benefit and will only measure the overhead side
- Use __shfl_sync(val,0) (warp_uniform_func) to broadcast a warp-uniform value cheaply when rematerializing warp_id from thread_id, since the division thread_id/32 is uniform across a warp
