## Solution

Replace scalar per-element load/store with vectorized float4 packed access via FLOAT4 macro (reinterpret_cast<float4*>), processing two adjacent RoPE pairs per thread in a single 128-bit load and store, halving thread count and reducing global memory transactions from 4 to 2 per thread

## Key Lessons

- Use vectorized float4 load/store (via reinterpret_cast<float4*> or FLOAT4 macro) instead of individual scalar loads/stores when processing consecutive pairs of fp32 values — coalesces 4 separate 32-bit transactions into 1 128-bit transaction, improving memory bandwidth utilization
- Pack two adjacent RoPE pairs into a single float4 per thread to halve the thread count for the same workload — each thread computes separate trigonometric coefficients (exp_f_v/exp_s_v) for each pair's dimension index while sharing the token_pos calculation
- Vectorized memory access patterns benefit bandwidth-bound kernels like RoPE more as sequence length grows — speedups increase from 1.18x at seq_len=4096 to 1.39x at seq_len=8192 because memory throughput dominates at longer sequences
