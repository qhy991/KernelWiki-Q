## Challenge

INT8 quantization kernel receives the scale factor as a host scalar via cudaMemcpy(DeviceToHost), forcing a synchronous D2H transfer that stalls the CUDA stream and blocks the CPU on every kernel invocation, creating a massive serialization bottleneck in inference loops


## Solution

Change the kernel to accept a device pointer (const scale_type* scale_ptr) and dereference it inside the kernel with *scale_ptr, eliminating the per-invocation cudaMemcpy D2H sync and keeping the entire dataflow on the GPU

## Key Lessons

- Pass device pointers (const T*) instead of host scalar copies as kernel arguments when the value already resides in GPU memory — eliminates cudaMemcpy D2H sync stalls that serialize the CPU-GPU pipeline on every invocation
- Reading a scalar via *device_ptr inside a kernel is effectively free (single L1/L2 cache hit for 4 bytes) compared to the microseconds of latency a cudaMemcpy D2H sync imposes — this pattern applies whenever a small kernel parameter originates from a device tensor
- The .item() sync pattern (extract a GPU scalar to host, then pass it back as a kernel argument) is an anti-pattern in inference loops — restructure to pass the device pointer and let the kernel read it directly
