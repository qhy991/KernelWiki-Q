## Challenge

Scalar warp-per-token MoE sum reduction kernel processes one BFloat16 element per iteration, issuing individual 16-bit global memory loads and stores that waste bandwidth on GPUs with 128-bit memory transaction granularity — each scalar access fetches a full cache line but uses only 16 bits


## Solution

Vectorize the inner loop to process 16 consecutive BFloat16 elements per iteration using uint4 (128-bit) loads/stores via a Pack16B union, reducing global memory transaction count by ~8x for this memory-bound MoE sum reduction

## Key Lessons

- Replace scalar bf16 load/store loops with vectorized uint4 (128-bit) load/store via a union reinterpret — reduces memory transaction count by 8x for memory-bound reduction kernels where computation is trivial relative to data movement
- Use ldg_cg (LDG cache-global) intrinsic for read-only input arrays in vectorized kernels to hint read-only cache usage and avoid unnecessary L1 pollution when streaming through large inputs
- Accumulate in fp32 (float acc[VEC]) when summing bf16 values across topk experts to preserve numerical precision, then convert back to bf16 only at final store
