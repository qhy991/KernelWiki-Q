## Challenge

The before code uses a 4-element 64-bit vec4_t struct with 16 blocks (8192 threads) for 32768 bf16 elements; upgrading to flashinfer::vec_t with 8-element 128-bit (int4) vectors halves the thread count to 4096 (8 blocks), reducing GPU occupancy for this tiny sub-3μs workload so that launch overhead dominates


## Solution

Replace the hand-rolled vec4_t (4-element, 64-bit aligned) with flashinfer::vec_t using 8-element 128-bit int4 loads, processing twice as many elements per thread via a parametric inner loop and cast_load/cast_store methods — intended to improve memory throughput but the wider vectors halve total thread count, causing a regression on this small workload

## Key Lessons

- Wider vector loads (128-bit int4 vs 64-bit) can regress performance on small workloads because halving the thread count reduces occupancy and launch overhead dominates when kernel runtime is under 3μs — applies to any elementwise kernel processing fewer than ~100K elements
- Vectorization width must be tuned against total element count: when numel ÷ (threads × vec_size) yields too few blocks (< ~10), the GPU is underutilized and a narrower vector with more blocks is preferable
- Parametric inner loops over vec_size (flashinfer::vec_t style) improve code maintainability over unrolled per-element field access (vec4_t .x/.y/.z/.w), but maintainability gains do not guarantee performance gains — always validate with benchmarks on representative workload sizes
