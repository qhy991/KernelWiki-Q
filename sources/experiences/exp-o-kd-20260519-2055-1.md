## Solution

Replace int32_t with uint16_t for the token_cnts working array and reorganize shared memory layout (cumsum int32_t at base, tokens_cnts uint16_t after), halving the tokens_cnts footprint from 264KB to 132KB and fitting the entire working set within the 164KB shared memory limit, converting all accesses from ~400-800 cycle global memory to ~5-20 cycle shared memory

## Key Lessons

- When shared memory requirements exceed device limits, downsize intermediate arrays from int32_t to uint16_t (or int16_t) to halve the footprint — token counts per expert per thread rarely exceed 65535 in practice, making uint16_t safe for MoE alignment workloads
- Use extern __shared__ with manual offset-based layout (cumsum at base, tokens_cnts at offset) to pack heterogeneous arrays into a single dynamic shared memory allocation, rather than passing separate global memory pointers
- Global memory to shared memory conversion for heavily-reused 2D working arrays yields multiplicative speedup proportional to access frequency — the token-count array is read/written across four phases (zero, count, prefix-sum, scatter), amplifying the per-access latency reduction
