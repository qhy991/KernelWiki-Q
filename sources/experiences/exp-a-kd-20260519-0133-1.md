## Solution

Distribute exp2f computation across warps so each warp owns kLinesPerWarp rows and only lane < kLinesPerWarp per warp computes exp, replacing the flat tid < kQueriesPerBlock guard with a per-warp lane guard. The total exp call count remains 64, but in the full fMHA kernel this pattern eliminates an extra __syncthreads() and the m_prime-overwrite that required it.

## Key Lessons

- Warp-distributed row ownership (each warp owns kLinesPerWarp rows) does not reduce total exp call count when the flat-tid guard already limits work to exactly the needed threads — the microbenchmark shows a 2.9% regression from the added integer arithmetic (wid * kLinesPerWarp + lane) for index computation
- The real benefit of warp-partitioned exp in fMHA iterative_softmax is not from fewer exp calls but from eliminating an extra __syncthreads() and the m_prime-overwrite pattern that required it — a microbenchmark that retains __syncthreads() in both variants will not capture this gain
- When evaluating an optimization in isolation, ensure the microbenchmark exercises the same control-flow and synchronization structure as the full kernel, or the measured delta will reflect noise from extra arithmetic rather than the intended structural improvement
