## Challenge

The direct thread-per-expert approach assigns each of 256 threads to one expert, sequentially filling expert_ids from a global cumsum array — only 256 of 1024 launched threads perform useful work, but each thread does minimal work (2 global reads + 1–4 writes).


## Solution

Replace thread-per-expert sequential fill with cooperative binary search across all 1024 threads: each thread strides over output blocks and binary searches the prefix sum array (O(log₂256)=8 iterations) to find the owning expert, distributing work evenly but introducing per-block global memory reads in the search loop.

## Key Lessons

- Binary search across all threads regresses when the lookup table resides in global memory — the O(log N) global memory reads per element (8 iterations for 256 experts) outweigh the benefit of using all 1024 threads versus 256 in the direct approach
- When the prefix/lookup array is in shared memory (e.g., after an in-kernel Blelloch scan), binary search becomes competitive or faster because shared memory eliminates the global memory bottleneck that caused the regression
- Low thread utilization is not always a bottleneck — when the per-thread work is trivially cheap (2 global reads + a few writes), adding computational overhead to engage more threads can be a net loss
