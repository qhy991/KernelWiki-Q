## Solution

Replace scalar half load/store with half2 vectorized operations: load two half values as __half2, convert to float2 for sigmoid and multiply computation, convert back to half2 for store — reducing load/store instruction count from 8 to 4 per vector, but expf arithmetic remains per-float lane unchanged

## Key Lessons

- Vectorized half2 loads/stores reduce memory instruction count but yield no speedup when the kernel is compute-bound by expensive transcendental operations (expf) — the 8 expf calls per vector dominate total execution time regardless of memory access pattern
- When profiling shows a tiny dataset (32KB) with heavy arithmetic (8 expf per 8-element vector), the bottleneck is compute not memory — vectorizing memory access alone cannot improve performance in this regime
- Half2 vectorization for activation kernels only helps when memory bandwidth is the limiting factor; at compute-bound operating points the per-lane float arithmetic is identical between scalar and vectorized paths
