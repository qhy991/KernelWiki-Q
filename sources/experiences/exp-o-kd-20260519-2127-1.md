## Challenge

Paged attention kernel reads K-cache vectors in the same dtype as the query (fp16/bf16), so KV cache memory footprint and DRAM bandwidth scale with full precision, becoming the primary bottleneck for long-sequence auto-regressive inference


## Solution

Conditionally read K-cache as FP8-E5M2 packed vectors (Quant_vec) and dequantize to computation dtype (K_vec) via fp8_e5m2_unscaled::vec_conversion, which maps to hardware-accelerated __nv_cvt_fp8_to_halfraw intrinsics, guarded by compile-time if constexpr (IS_FP8_E5M2_KV_CACHE) for zero overhead on non-FP8 paths

## Key Lessons

- Store KV cache in FP8-E5M2 and dequantize inline via hardware intrinsics to halve memory footprint and DRAM traffic for key vectors — applies to attention-bound workloads where K-cache bandwidth is the bottleneck
- Use cache_t (generic) instead of scalar_t for cache pointer types when supporting mixed-precision KV cache where storage type differs from computation type — decouples cache layout from compute precision
- Guard FP8 code paths with if constexpr and #ifdef to ensure zero overhead on non-FP8 code paths and compile-time safety across GPU architectures
