## Solution

Reinterpret FP16 pointers as _f16Vec<__half,8> (128-bit aligned vector struct) to issue 16-byte vectorized loads instead of scalar 2-byte loads, use native __half2 packed arithmetic for addition and multiplication (doubling throughput per instruction), and adaptively reduce block size to 256 threads for high token counts to improve SM occupancy

## Key Lessons

- Reinterpret FP16 pointers as 128-bit aligned vector structs (_f16Vec<__half,8>) to issue vectorized loads instead of scalar 2-byte loads — reduces global memory transaction count by the vector width factor (8x here), applies when hidden_size is a multiple of the vector width
- Native __half2 packed arithmetic (add/multiply) doubles compute throughput per instruction by operating on two FP16 values simultaneously — applies when adjacent elements can be processed together and memory is 4-byte aligned
- Reduce block size from 1024 to 256 threads when vectorization shrinks the per-block iteration count (hidden_size/width=256) to avoid idle threads and improve SM occupancy — applies when vectorization reduces the work iteration space below the original thread count
