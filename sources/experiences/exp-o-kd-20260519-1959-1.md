## Solution

Replace the two-step bf16-GEMM + cast-kernel pipeline with a single cublasGemmEx call that specifies CUDA_R_32F as the output type (C matrix dtype), letting cuBLAS accumulate in fp32 internally and write fp32 output directly — eliminating the intermediate bf16 buffer, the cast kernel launch, and all associated memory traffic

## Key Lessons

- Set the C-matrix output type to CUDA_R_32F in cublasGemmEx while keeping A/B inputs as CUDA_R_16BF — cuBLAS accumulates in fp32 internally and writes fp32 directly, eliminating the need for a separate bf16→fp32 cast kernel and its associated memory traffic and launch overhead
- For small GEMMs (M≤16), the relative overhead of an extra kernel launch and intermediate buffer round-trip is proportionally large (~20% of total time); fusing the type conversion into the library call is especially impactful in this regime
