## Challenge

reduce_max always performs 5 butterfly shuffle rounds (mask starting at 16, halving to 1) regardless of how many threads actually participate, wasting cycles exchanging values with inactive or irrelevant lanes when only a subset of warp lanes need the result


## Solution

Add a kNumThreads template parameter (defaulting to kWarpThreads=32 for backward compatibility) that sets the initial mask to kNumThreads/2, making butterfly reduction depth proportional to actual group size with static_assert guards enforcing power-of-2 between 1 and 32

## Key Lessons

- Parameterize warp-level reduction functions with a kNumThreads template controlling the initial shuffle mask (kNumThreads/2) so sub-warp groups execute log2(group_size) rounds instead of a fixed 5 — eliminates unnecessary cross-lane communication when the participating group is smaller than a full warp
- Use static_assert with std::has_single_bit to enforce that kNumThreads is a power of 2 within [1, 32], ensuring valid butterfly shuffle patterns at compile time
- Default the template parameter to full warp size for backward compatibility, so existing callers get unchanged behavior without code changes
