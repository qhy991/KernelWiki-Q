## Challenge

GPTQ 8-bit weight repack kernel uses a two-step pattern: first loading 8 uint32 values from shared memory into intermediate local arrays (b1_vals/b2_vals, 8 registers total), then extracting elements in a second loop — causing unnecessary register pressure and extra address computation overhead


## Solution

Replace the two-step load-then-extract pattern with one-step direct shared memory indexing: pre-compute cur_n_packed and cur_n_pos outside the loop, read packed values directly from shared memory and extract inline, eliminating 8 intermediate registers and the first load loop entirely

## Key Lessons

- Eliminate intermediate local arrays in shared-memory tile processing by reading and consuming data in one step — this reduces register pressure and removes an entire loop when the consumption pattern can be expressed with pre-computed direct indices
- Pre-compute packed row/column indices (cur_n_packed, cur_n_pos) outside the tile loop and use them for direct shared-memory addressing — replaces a two-index decode scheme (cur_int_local + cur_pos) with a simpler single-index pattern that the compiler can optimize more effectively
- Shared-memory stride affects bank-conflict patterns: stride=16 (tile_n_size/pack_factor for 8-bit) has better bank-conflict characteristics than stride=64, compounding the register-pressure reduction for larger tile_ints values
