## Challenge

Flash attention kernel uses sequential K/V loading (kStage=1) with sync barriers before computation, preventing overlap of async memory copies with MMA math and leaving memory bandwidth idle during compute phases


## Solution

Replace sequential K/V loading with kStage=2 software pipeline: double-buffer K shared memory (2 slots), prefetch next K tile during current-tile computation, and issue V load asynchronously to overlap with Q@K^T MMA — but at seqlen=256 with only 4 KV tiles, the extra sync barriers and warmup overhead exceed the overlap benefit, causing net regression

## Key Lessons

- Software pipelining (kStage=2) adds pre-loop warmup latency (K prefetch + CP_ASYNC_WAIT_GROUP + __syncthreads) and per-iteration sync barriers — these overheads must be amortized across enough iterations; at Tc=4 the pipeline never reaches steady-state
- Double-buffered K shared memory (2 KV_tile_size slots vs 1) increases total smem usage, which can reduce occupancy and further degrade performance when the pipeline overlap benefit is marginal
- Pipeline effectiveness depends on the ratio of compute time to load time per tile — when computation (64x64 Q@K^T + softmax + 64x64 P@V) is short relative to barrier overhead, pipelining provides negative returns
