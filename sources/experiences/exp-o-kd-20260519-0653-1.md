## Challenge

The transform_a XOR remapping for shared-memory bank-conflict avoidance uses the full row value (`(i % a_gl_rd_delta_o) ^ row`), which can produce indices outside the intended 8-row tile range, causing incorrect shared memory addresses and up to 64 out-of-bounds accesses in the 16-row case


## Solution

Replace the XOR operand from the full `row` to `row % 8`, constraining the remapping to operate within the 8-row tile range used by a warp so that the XOR pattern never escapes the valid tile index space

## Key Lessons

- XOR-based shared memory bank-conflict avoidance must constrain the XOR operand to the tile dimension (row % tile_rows) rather than using the full row index — without this clamp the XOR can produce out-of-bounds shared memory addresses
- Correctness fixes that eliminate out-of-bounds indexing can also yield measurable speedups by removing conditional clamping branches the hardware must evaluate on the divergent path
