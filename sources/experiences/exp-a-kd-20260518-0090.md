## Solution

Add the missing kRow multiplier into the row offset calculation so that byte_pointer advances by offset_row * kRow rows instead of only offset_row rows, making the tile navigation mathematically correct for arbitrary tile heights

## Key Lessons

- When computing byte offsets for tiled shared memory access, the row component must be scaled by both the stride and the tile height (kRow) — omitting the tile height multiplier produces a pointer that lands on a single-row offset rather than a full-tile-row offset, yielding silently incorrect data
- A single added multiply in pointer arithmetic has negligible performance impact when the kernel is memory-bound — correctness fixes to offset formulas are essentially free in ALU cost and should never be skipped for performance reasons
