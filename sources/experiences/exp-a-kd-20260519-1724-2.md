## Challenge

BlockReduce<1024> with 32 warps per block provides good latency hiding across the two-phase kernel (absmax scan + quantize), but uses ~4KB shared memory for reduction temp storage


## Solution

Reduce block size from 1024 to 256 threads (BlockReduce<256>) to lower shared memory usage from ~4KB to ~1KB for temp storage, but the 4x reduction in warps per block (32→8) reduces latency-hiding capacity in both kernel phases, causing a net regression

## Key Lessons

- Reducing block size (1024→256) lowers shared memory per block but the loss of warp-level parallelism (32→8 warps) can dominate performance in two-phase kernels separated by __syncthreads() — the occupancy gain from less shared memory is negated by insufficient latency-hiding capacity
- Block size tuning must account for the total warp count available to the scheduler: for memory-bound two-phase kernels, having enough warps to overlap memory latency in both phases is critical, and shared memory savings alone do not compensate for reduced parallelism
