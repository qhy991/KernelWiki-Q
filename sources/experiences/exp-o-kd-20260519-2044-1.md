## Solution

Replace hardcoded VEC_SIZE with runtime-calculated vector size using std::gcd(16/sizeof(scalar_t), hidden_size) to find the largest vector size that fits within 128-bit memory transactions and evenly divides hidden_size, then dispatch as a compile-time template parameter via VLLM_DISPATCH_VEC_SIZE for full loop unrolling

## Key Lessons

- Use std::gcd(16/sizeof(scalar_t), hidden_size) to compute the largest vector size that both fits in 128-bit memory transactions and evenly divides hidden_size — ensures correctness for all tensor dimensions while maximizing memory throughput
- Dispatch runtime-computed vector sizes as compile-time template parameters via macro dispatch (e.g., VLLM_DISPATCH_VEC_SIZE) to preserve constexpr unrolling benefits — the host selects the value but the device code sees it as a compile-time constant
- Adaptive block sizing (std::min(hidden_size/calculated_vec_size, max_block_size)) paired with dynamic vector size dramatically increases SM concurrency for large batch workloads where wider vectors reduce per-block work and allow more blocks to run simultaneously
