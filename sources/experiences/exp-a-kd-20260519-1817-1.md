## Challenge

Replacing an async 4-stage copy-compute pipeline with synchronous reinterpret_cast loads forces a serial copy-then-compute ordering per tile, requiring two __syncthreads() barriers per iteration that stall every thread until the slowest arrives, amplifying global memory read latency on single-SM launches


## Solution

Use PTX cp.async.cg.shared.global instructions with a 4-stage pipeline and cp_async_fence/cp_async_wait group management to overlap global-to-shared async transfers with computation, allowing tile N+1 copies to proceed in hardware while threads compute on tile N-2 data without __syncthreads() barriers in the pipeline loop

## Key Lessons

- Use PTX cp.async.cg.shared.global with multi-stage shared memory buffers and cp_async_fence/cp_async_wait group management to overlap global-to-shared transfers with compute — eliminates __syncthreads() barriers between copy and compute phases and hides global memory latency behind useful work
- Avoid replacing async copy pipelines with synchronous reinterpret_cast loads on sm80+ hardware — losing copy-compute overlap and adding per-tile __syncthreads() barriers causes 2.4x regression due to full-block stalls that amplify global read latency
