## Solution

Store key cache in FP8-E5M2 format (1 byte/element) and use __nv_cvt_fp8x2_to_halfraw2 hardware intrinsic for vectorized fp8x2-to-half2 dequantization at load time, halving memory bandwidth requirements for key cache reads

## Key Lessons

- FP8 dequantization intrinsics like __nv_cvt_fp8x2_to_halfraw2 are hardware-accelerated only on sm_89+ (Ada Lovelace); on sm_80 they are emulated in software with very high overhead that can completely negate memory bandwidth savings from reduced data size
- Memory bandwidth reduction via lower-precision storage only improves performance when the compute overhead of dequantization is negligible — always verify the target architecture supports the required conversion intrinsics natively before deploying FP8 cache compression
- FP8-E5M2 key cache compression halves memory traffic but the tradeoff depends critically on the arithmetic intensity of the surrounding computation; for pure load-bound kernels without significant compute, emulation overhead dominates
