## Solution

Fuse BF16→float conversion directly into the softmax kernel by calling __bfloat162float inline within each of the three softmax loops (max-reduction, exp-sum, output-write), eliminating the separate conversion kernel, its intermediate float buffer allocation, and one full global memory round-trip

## Key Lessons

- Fusing a lightweight type-conversion kernel into the downstream compute kernel eliminates one kernel launch and a full global memory round-trip — especially beneficial when per-row work is small (e.g., 8 experts) so launch overhead and intermediate memory traffic dominate total time
- Inline __bfloat162float calls within existing compute loops add negligible register pressure since the converted value is consumed immediately, but they remove the need for a separate intermediate float buffer allocation in device memory
- Kernel fusion pays off even when the fused computation is repeated (BF16→float conversion occurs 3 times in the three softmax passes) because reading BF16 from global memory 3 times is cheaper than writing and reading a full float intermediate buffer once
