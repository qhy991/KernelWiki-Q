## Challenge

Router GEMM kernel passes num_tokens as a runtime parameter, forcing the compiler to conservatively allocate a 16-element accumulator array and emit a token loop with runtime-bound check on every iteration, wasting registers and adding loop-control overhead


## Solution

Promote num_tokens to a compile-time template parameter kNumTokens so the NVCC compiler fully unrolls the token loop, allocates exactly kNumTokens accumulator registers instead of a worst-case 16, and add __launch_bounds__(128,1) to minimize register pressure for the known single-accumulator case

## Key Lessons

- Promote runtime loop bounds (especially small fixed values like token count) to compile-time template parameters so NVCC can fully unroll loops, allocate exact registers, and eliminate loop-control overhead — yields significant speedup when the bound is small and known at kernel launch time
- Adding __launch_bounds__(blockSize, minBlocksPerSM) alongside compile-time specialization hints to the compiler that register pressure is low, enabling more aggressive optimization and potentially higher occupancy for small-M GEMM kernels
- Shared memory and accumulator arrays sized by compile-time constants (kNumTokens vs hardcoded 16) allow the compiler to eliminate unused slots and generate straight-line code instead of looped access patterns
