## Solution

Replace scalar f32 loads with vectorized float4 loads via the FLOAT4 macro (reinterpret_cast<float4*>), so each thread loads 4 consecutive floats from both A and x in a single coalesced transaction, reducing total global memory transactions by 4x and inner loop iterations by 4x

## Key Lessons

- Vectorized float4 loads reduce global memory transaction count by 4x and cut loop iterations by 4x, but the benefit is only measurable when the memory-bound portion dominates kernel latency — at small problem sizes (M=512, K=128), kernel launch overhead and the tiny compute workload mask the bandwidth improvement
- When applying vectorized loads to GEMV/inner-product kernels, the stride multiplier on the k-index (e.g., * 4 for float4) must match the vector width so threads still access contiguous 128-byte segments for coalescing
