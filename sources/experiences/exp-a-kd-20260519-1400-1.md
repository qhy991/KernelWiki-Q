## Solution

Reorder pipeline stages to issue matmul compute before the cp_async_wait<0>() barrier, allowing arithmetic work to overlap with pending async copy completion — the Marlin 24 pattern of fetch -> matmul -> wait -> load_regs

## Key Lessons

- Reordering pipeline stages (matmul before wait) to overlap compute with async copy completion requires sufficient compute depth to be effective — scalar FMA loops complete too quickly relative to cp.async wait latency, providing no overlap benefit and potentially increasing register pressure by keeping partial sums live across the barrier
- The Marlin 24 pipeline reorder yields ~25% improvement on real tensor-core MMA instructions because they have multi-cycle issue latency and much higher throughput, providing enough compute work to meaningfully overlap with memory synchronization — apply this pattern only when compute kernels have substantial arithmetic intensity
- Pipeline stage reordering that keeps matmul partial sums live across the barrier may increase register pressure; the nvcc compiler may not optimally schedule FMA instructions across the wait_group boundary without explicit scheduling hints, which can negate the overlap benefit on lightweight compute kernels
