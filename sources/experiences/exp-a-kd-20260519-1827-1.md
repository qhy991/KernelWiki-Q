## Challenge

Host code launches a kernel using ~8KB of dynamic shared memory without calling cudaFuncSetAttribute to explicitly declare the maximum dynamic shared memory size, raising the question of whether the driver can optimize shared memory allocation when the attribute is explicitly set even when usage is well within the default 48KB limit


## Solution

Add cudaFuncSetAttribute with cudaFuncAttributeMaxDynamicSharedMemorySize before the kernel launch to explicitly declare the dynamic shared memory requirement, but since the ~8KB usage is well within the default 48KB limit on sm_80, the call has no effect on allocation or execution and produces zero measurable performance change

## Key Lessons

- cudaFuncSetAttribute with cudaFuncAttributeMaxDynamicSharedMemorySize only matters when shared memory exceeds the default 48KB limit — when usage is well within default limits, the call is a no-op with zero performance impact
- Explicit shared memory size declaration via cudaFuncSetAttribute is a launch-time hint for the driver, not a kernel-code optimization — it affects resource allocation heuristics but only produces measurable effects near or beyond architectural defaults
