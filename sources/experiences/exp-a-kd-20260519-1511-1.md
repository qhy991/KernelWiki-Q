## Solution

Cooperative shared memory loading replaces per-thread register arrays with __shared__ float q_vecs[2][64], where thread groups collaboratively load query vectors once and share them — however the L2 cache already absorbs the redundant global reads, while shared memory introduces __syncthreads() barrier overhead, ~5-cycle read latency per inner-loop access, and 2-way bank conflicts when tg_off=0 and tg_off=1 access the same bank index simultaneously

## Key Lessons

- When the working set fits comfortably in L2 cache (~512 bytes for 128 floats here), redundant global memory reads are already absorbed — shared memory optimization provides no benefit and adds barrier/latency overhead; applies when query or key footprints are small relative to L2 cache size
- Register access (1-cycle latency) is significantly faster than shared memory access (~5 cycles) for hot inner-loop data — moving data from registers to shared memory is counterproductive when register pressure is manageable; applies to any kernel where inner-loop operands fit in registers without spilling
- 2-way bank conflicts arise when __shared__ float arr[2][N] has N as a multiple of 32, because both rows map banks identically (bank = index % 32) — use padding or restructure the access pattern to avoid; applies to small 2D shared memory arrays where the row width is a multiple of the bank count
