## Solution

Replace the 3-phase single-block count-prefix-sum-scatter algorithm with a multi-block kernel where each thread simply reads topk_ids[i], atomically claims a slot via atomicAdd on the pre-initialized cumsum_buffer[expert_id], and writes the token index — eliminating the count matrix, prefix-sum, and barrier synchronization entirely

## Key Lessons

- Replacing a multi-phase count-prefix-sum-scatter with atomicAdd on a small buffer (256 entries) eliminates large intermediate global memory allocations (65792 entries), serial prefix-sum bottlenecks, and __syncthreads barriers — applies when the output positions can be claimed atomically from a known starting offset
- Multi-block kernel launch (16 blocks × 256 threads) over a single-block kernel (1 block × 256 threads) dramatically improves SM utilization for data-parallel workloads that do not require cross-block synchronization
- For MoE token routing with a moderate number of experts (≤256), global memory atomics on the cumsum buffer have negligible contention because tokens distribute across experts and each atomic operates on a different expert slot — the atomic approach scales well as long as the expert count is small relative to token count
