## Solution

Launch 320 total threads (256 fill_threads + 64 working threads) in a single block: fill_threads init sorted_token_ids at stride=256 (32 iterations each, 4x fewer) while working threads simultaneously perform shared-memory counting/prefix-sum; fill_threads wait at three __syncthreads barriers to stay synchronized before returning, overlapping global-memory writes with shared-memory computation to reduce total latency to max(init, count)+post

## Key Lessons

- Within a single-thread block, split threads into role-specific groups (fill vs compute) that work on independent memory subsystems (global memory writes vs shared memory ops) concurrently — global-memory init latency hides behind shared-memory counting when both execute before the first __syncthreads
- Increasing total thread count beyond the minimum (64→320) to create dedicated fill threads is worthwhile when the overlapped global-memory init phase takes comparable time to the counting phase and shared-memory capacity is not a constraint
- Use triple __syncthreads barriers in filler threads to match the working threads' barrier sequence, ensuring filler threads do not exit before working threads reach the phase that reads the initialized buffer
