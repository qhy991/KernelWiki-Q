## Challenge

Elementwise BF16 scale-and-add kernel uses cutlass::bfloat16_t overloaded operator* which internally wraps scalar __hmul, preventing nvcc from vectorizing adjacent multiplies into the 2-element __hmul2 intrinsic and doubling arithmetic throughput


## Solution

Replace cutlass::bfloat16_t with native nv_bfloat16 type and explicitly pack adjacent elements into __nv_bfloat162 pairs, enabling __hmul2/__hadd2 vectorized 2-element BF16 intrinsics that halve the instruction count for the multiply-add arithmetic

## Key Lessons

- Vectorized BF16 intrinsics (__hmul2/__hadd2) halve arithmetic instruction count but yield no measurable speedup when the kernel is memory-bandwidth-bound on small workloads — the optimization only helps in register-resident compute loops (e.g., WGMMA inner loops) where arithmetic throughput is the bottleneck
- cutlass::bfloat16_t overloaded operator* wraps scalar __hmul via a function call abstraction that blocks nvcc auto-vectorization; switching to native nv_bfloat16 with explicit __nv_bfloat162 packing gives the compiler visibility needed for vectorization
- Memory-bound kernels processing ~128KB arrays on HBM2e-capable GPUs are dominated by launch overhead and timing noise; microbenchmarks at this scale cannot reliably measure arithmetic-only optimizations
