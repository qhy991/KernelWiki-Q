## Solution

Switch to MQA layout by reducing NKV from 8 to 1 and setting head_mapping to all zeros so every query head shares the single KV head, cutting V-cache memory by 8x while the kernel inner loop (logit*v accumulation over 128 blocks × 16 tokens) remains identical in instruction count and memory access volume

## Key Lessons

- MQA V-cache layout reduces total KV cache memory by num_query_heads/num_kv_heads factor (8x here) without requiring any kernel code changes — only host-side block_stride, NKV, and head_mapping differ, since the kernel's per-head stride arithmetic compiles to the same IMAD instruction pattern regardless of stride value
- Per-kernel latency is unchanged when switching MHA→MQA because the dominant cost is the inner loop's logit×v accumulation over all tokens, which has identical memory access volume and arithmetic intensity whether block_stride is 8192 or 1024
- MQA's real performance benefit manifests at the model level: reduced aggregate KV cache memory across all layers and sequences enables higher batch sizes and longer context lengths within the same GPU memory budget
