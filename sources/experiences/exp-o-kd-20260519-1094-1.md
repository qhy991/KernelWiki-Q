## Challenge

CUB BlockReduce-based MoE softmax kernel launches 256 threads per block with shared-memory TempStorage and two __syncthreads() barriers, but only 8 of 256 threads carry real data per row (num_experts=8) — 248 threads carry -FLT_MAX or zero, wasting shared-memory bandwidth and synchronization cycles


## Solution

Pack all 8 elements per row into a single thread via VPT=8 (THREADS_PER_ROW=1), making max/sum/exp entirely thread-local with zero synchronization — the warp butterfly shfl_xor reduction loops become no-ops since THREADS_PER_ROW/2=0, and 32 rows process independently within each warp

## Key Lessons

- When elements per row (e.g., num_experts) fit in registers, use VPT packing so each thread owns an entire row — eliminates shared-memory TempStorage, __syncthreads() barriers, and inter-warp coordination entirely
- CUB BlockReduce allocates shared-memory TempStorage and forces __syncthreads() even for trivial reduction sizes — for small power-of-2 reductions (≤32 elements), thread-local computation or warp shuffle is significantly cheaper
- Parameterized warp butterfly shfl_xor reduction (mask from THREADS_PER_ROW/2 down to 1) gracefully degenerates to a no-op when THREADS_PER_ROW=1, providing a single unified code path that scales from single-thread to multi-thread reductions
