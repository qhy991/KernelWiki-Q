## Challenge

Shared memory struct reserves 49,152 bytes (48 KB) per thread block via a large preload padding array, consuming nearly the entire per-SM shared memory budget and severely limiting the number of concurrent thread blocks that can resident on each SM


## Solution

Halve the shared memory allocation from 48 KB to 24 KB by reducing the padding budget constant from 49,152 to 24,576, keeping the active computation tile unchanged — this doubles the number of active thread blocks per SM through improved shared memory resource fit

## Key Lessons

- Reducing per-block shared memory footprint directly increases SM occupancy — when a kernel's active data fits in less shared memory than allocated, trimming unused padding or switching to a smaller tile specialization (e.g., 128x128 → 64x64) can yield multiplicative speedups from higher block-level concurrency
- Shared memory occupancy is a first-order performance factor for shared-memory-heavy kernels — always query cudaOccupancyMaxActiveBlocksPerMultiprocessor to verify that shared memory allocation is not the bottleneck for block-level parallelism
