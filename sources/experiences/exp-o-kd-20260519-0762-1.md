## Solution

Cache input data in dynamically-allocated shared memory during Pass-1 with 32-byte bank-conflict-padded stride per warp, then read from shared memory in Pass-2 via __syncwarp() barrier, eliminating the redundant global load and exploiting shared memory's ~5TB/s bandwidth versus ~50-80GB/s from HBM

## Key Lessons

- When a kernel makes multiple passes over the same input data, cache the first pass's global loads in shared memory to avoid redundant HBM traffic — applies when per-CTA data fits in shared memory (~48KB) and passes are within the same kernel launch
- Use 32-byte-aligned bank-conflict-padded strides when multiple warps share a single shared memory allocation to prevent inter-warp bank conflicts at boundary elements — the padding ensures each warp's region starts on a clean bank boundary
