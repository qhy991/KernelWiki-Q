## Challenge

Increasing CUDA block size from 128 to 256 threads for a per-token BF16 quantization kernel (hidden_dim=4096, vec_size=8) regresses performance because each thread processes fewer vector-load iterations (4→2) while blockReduceMax must coordinate twice as many warps (4→8), increasing shared-memory round-trips and __syncthreads barrier cost


## Solution

Doubling block size to 256 threads reduces per-thread iteration count from 4 to 2 for hidden_dim=4096, failing to amortize the increased cross-warp reduction and barrier synchronization overhead — the correct tuning direction for this workload size is to keep fewer threads with more work per thread

## Key Lessons

- Block size tuning must account for per-thread iteration count: doubling threads halves iterations per thread, and if iterations drop below ~4 the thread coordination overhead (cross-warp reduction, __syncthreads) dominates — prefer more iterations per thread for moderate workloads
- blockReduceMax shared-memory round-trips scale with warp count (blockDim.x/32): going from 4 to 8 warps doubles the synchronization work in the second reduction stage, which is costly when each thread has little computational work to amortize it
- Larger block sizes may benefit larger hidden_dim values (e.g., 8192+) where each thread retains enough iterations to amortize barrier overhead — block size is not universally better when increased
