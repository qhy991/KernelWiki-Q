## Solution

Refactor to SFINAE-constrained quant_type_max<T> and min_scaling_factor<T> templates using enable_if_t disjunctions over three types (Float8_e4m3fn, Float8_e4m3fnuz, int8_t), extracting min_scaling_factor into a templated struct with __forceinline__ for explicit inlining control across all quantization types

## Key Lessons

- SFINAE-constrained template constants (quant_type_max<T>, min_scaling_factor<T>) generalize quantization code across numeric types (Float8_e4m3fn, Float8_e4m3fnuz, int8_t) via enable_if_t disjunctions without changing runtime values — applies when extending quantization support to new numeric formats
- At sub-10μs kernel durations, 10-15% timing variance is within measurement noise dominated by kernel launch overhead and event resolution; benchmark conclusions at microsecond scales should verify with increased iteration counts or larger workload sizes
- __forceinline__ on template helper functions (min_scaling_factor<T>::val()) provides explicit inlining control that compiler heuristics may not apply uniformly across nvcc versions — useful when template instantiation depth could inhibit automatic inlining decisions
