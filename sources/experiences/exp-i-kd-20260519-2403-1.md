## Challenge

Skeleton GEMM kernel provides only TODO placeholders for global/shared/register tensor construction, TiledCopy/TiledMMA thread partitioning, and a double-buffered pipeline that must overlap global memory loads with MMA computation across K-tiles


## Solution

Construct global tensors via make_tensor(make_gmem_ptr, shape, stride), tile per-CTA with local_tile using Step selectors, partition copies via ThrCopy slices into register fragments (make_fragment_like), partition MMA via ThrMMA slices, and implement a double-buffered pipeline where register-to-shared copy, next-tile global-to-register prefetch, and shared-to-accumulator GEMM overlap across K-tiles

## Key Lessons

- Use select<dim0,dim1>(shape_MNK) with Step selectors to project the 3D problem shape onto the correct 2D tensor for each matrix — select<0,2> gives (M,K) for A, select<1,2> gives (N,K) for B — keeping stride consistency with the CUTLASS cute tensor abstraction
- Stage global-to-shared copies through register fragments (make_fragment_like) to enable double-buffering: while threads compute gemm on shared memory tile k, they simultaneously prefetch tile k+1 into registers, hiding global memory latency behind MMA arithmetic
- Guard the prefetch with a conditional k_tile_next clamp to avoid out-of-bounds reads on the final iteration when no next tile exists — reuse the current tile's data as a safe no-op
