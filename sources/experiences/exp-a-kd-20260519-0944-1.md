## Challenge

RoPE kernel writes rotated K to k_rope in global memory; a separate kv_save_kernel re-reads k_rope and saves both K and V to the paged KV cache, incurring a redundant global memory round-trip for K and a full extra kernel launch with separate V load


## Solution

Fuse rope_only_kernel and kv_save_kernel into a single fused_rope_kv_kernel that loads V once, computes RoPE into register-resident k_out, then writes k_out to both k_rope and k_buffer without any global memory round-trip

## Key Lessons

- Fusing a producer kernel that writes to global memory with a consumer kernel that immediately reads the same data eliminates the redundant global memory round-trip — the producer's register output can be reused directly by the consumer's store operations, which is especially valuable when the data (e.g., k_out after RoPE) is computed per-element and would otherwise require a full global load/store cycle
- Kernel fusion reduces total global memory traffic proportionally to the intermediate tensor size — here the eliminated k_rope round-trip saves 2 × nnz × num_kv_heads × head_dim × sizeof(float) bytes per iteration, yielding speedups (1.46x) that persist even at medium batch sizes where launch overhead alone is negligible
- Load data that will be consumed by a later store (V in this case) as early as possible within the fused kernel's head loop so the compiler can overlap the load latency with the compute-heavy RoPE section
