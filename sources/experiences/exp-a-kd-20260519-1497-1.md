## Solution

Replace the hand-written sequential-halving blockReduceMax with CUB BlockReduce, passing blockDim.x (256) as the num_valid argument so CUB knows only 256 threads have valid data, enabling it to use warp-shuffle instructions for intra-warp reduction and skip unnecessary comparisons with uninitialized shared-memory slots

## Key Lessons

- Replace hand-written shared-memory reductions with CUB BlockReduce, especially for partial-block scenarios where num_valid < blockDim.x — CUB's num_valid parameter avoids wasted comparisons with uninitialized slots and reduces shared-memory writes and __syncthreads() barrier overhead
- CUB BlockReduce uses warp-shuffle intrinsics for intra-warp reduction, eliminating shared-memory traffic and one __syncthreads() per warp-level stage — this is particularly beneficial when many threads in the block are idle (partial participation)
- For block-level reductions where the active thread count is smaller than blockDim.x, passing the actual valid count to CUB's Reduce() is critical for both correctness and performance — the library handles the partial-block edge case internally without requiring manual masking or padding
