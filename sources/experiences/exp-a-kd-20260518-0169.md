## Solution

Add a (sm_occupancy > 1) guard before the remapping logic so that when only one wavefront fits per SM, the entire FastDivmod computation and branch are skipped, eliminating unnecessary device-side overhead

## Key Lessons

- Guard occupancy-dependent remapping with (sm_occupancy > 1) to skip no-op FastDivmod and branch when only one wavefront fits per SM → eliminates wasted arithmetic and divergence → applies to any kernel launch with variable SM occupancy where single-occupancy is a common case
- FastDivmod is a non-trivial device-side operation involving float multiply, integer subtract, and conditional correction; when the divisor is known to produce an identity mapping, hoisting the guard outside is strictly better even if measured speedup is negligible in microbenchmarks
