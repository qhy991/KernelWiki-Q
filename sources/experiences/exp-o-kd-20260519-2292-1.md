## Solution

Replace per-thread local counting + manual prefix-sum with warp-partitioned atomicAdd into shared memory: 1024 threads stride-iterate over all tokens and atomically increment the per-expert counter in their assigned warp's slot, eliminating the prefix-sum phase, two __syncthreads() barriers, and reducing shared memory from ~16 KB to 256 bytes (num_warps × 32 ints)

## Key Lessons

- Replace per-thread local counting followed by manual prefix-sum with shared-memory atomicAdd when the reduction domain (expert count) is small enough to fit in shared memory — eliminates two __syncthreads() barriers and the per-thread count matrix, reducing shared memory footprint by ~60x
- Stride-based iteration with 1024 threads over per-thread sharding with 64 threads improves memory coalescing and occupancy when the work item count (tokens) far exceeds the original thread count — applies to histogram/counting kernels where contention on the accumulator set is moderate (≤2 warps per bucket set)
- Warp-partitioned shared memory layout (each warp owns experts_per_warp contiguous slots) avoids bank conflicts on atomicAdd because threads within a warp hash to different expert offsets within their warp's partition — applies when the domain can be evenly divided by WARP_SIZE
