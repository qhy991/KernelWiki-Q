## Challenge

Fused Q+K RMSNorm kernel processes both query and key normalization per CTA, requiring 6 vec_t register arrays (v_q, v_k, v_q_weight, v_k_weight, v_q_out, v_k_out) and 64-float shared memory with dual warp/CTA reductions, increasing per-thread register footprint and SMEM allocation


## Solution

Split Q and K normalization into separate CTAs via blockIdx.y (dim3 grid with y=2), reducing per-CTA state to 2 vec_t registers and 32-float shared memory with a single reduction, using a runtime is_q branch for data/weight pointer selection

## Key Lessons

- Splitting fused Q+K normalization into separate CTAs via blockIdx.y reduces per-CTA register pressure (6→2 vec_t arrays) and shared memory (64→32 floats), but when occupancy is already sufficient (hidden_size=4096, 512 threads/CTA on A800 with 108 SMs), register savings alone do not translate to speedup
- Total reduction work is invariant under CTA splitting: 512 fused CTAs × 2 reductions = 1024 split CTAs × 1 reduction = 1024, so the split does not reduce reduction overhead — the per-CTA amortization gain in the fused version is offset by the doubled CTA count in the split version
- Runtime branch-based pointer selection (is_q ? q : k) and dynamic dispatch add minor instruction overhead that can negate register savings when the architectural change does not unlock additional SM-level parallelism
