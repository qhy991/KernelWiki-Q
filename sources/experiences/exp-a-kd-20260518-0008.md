## Challenge

Passing v_in[i] directly to a read-only VecOp in a read-only kernel can cause NVCC to issue multiple narrow scalar loads (16 separate LDG.2) instead of one wide LDG.128, when the callable iterates over the internal scalar array and the compiler cannot prove the whole pack is consumed atomically.


## Solution

Materialize the vector pack into vin_t src = v_in[i] before passing to VecOp. This forces NVCC to load the entire 32-byte pack from global memory in one wide transaction (LDG.128 or LDG.32) rather than potentially scalarizing into 16 individual loads. For memory-bound read-only kernels, reducing memory transactions improves latency.

## Results

- **speedup**: 1.100x (9.1% latency reduction)


## Key Lessons

- Materializing vector packs to locals before passing to read-only callables helps when the callable accesses elements sequentially via scalar array indexing — the compiler may not prove all elements are consumed and might scalarize loads. The local copy provides a single-def point the compiler recognizes as needing one wide load. Applies to read-only memory-bound kernels with vec width ≥16 bytes.
- This is the dual of the regression case (sample 5): local copy helps for read-only VecOp where the compiler can't inline-prove all-element consumption, but hurts for __forceinline__ VecOp where the compiler already sees the full dataflow. The key distinction is whether the VecOp is transparent to the compiler's load scheduling analysis.
