## Challenge

Five separate small kernel launches for metadata copy operations (cache_seqlens, cu_seqlens_k, page_indices, nsa_cache_seqlens, nsa_cu_seqlens_k) each execute serially on the GPU with minimal work — the 1D copies handle only 128 elements each, severely underutilizing SMs, while each launch incurs ~5us overhead and implicit serialization between kernels


## Solution

Fuse all 5 metadata copy operations into a single kernel launch using a parameter struct (FusedMetadataCopyParams) with __grid_constant__ qualifier, consolidating the 5 serial underutilized launches into one grid that fully utilizes available threads across all data sections

## Key Lessons

- Fuse multiple small independent data-copy kernels into a single kernel launch to eliminate per-launch overhead (~5us per launch) and avoid serial underutilization of SMs — applies whenever several copy/elementwise operations execute sequentially on small data batches
- Use __grid_constant__ parameter qualifier for fused kernels with large parameter structs to keep the struct in constant memory for the kernel's lifetime, avoiding repeated loads from global memory
- Use #pragma unroll on sequential copy loops within a fused kernel to encourage the compiler to unroll small iteration-count loops, reducing loop overhead when each copy section is short
