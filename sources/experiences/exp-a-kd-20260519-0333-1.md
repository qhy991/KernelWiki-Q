## Solution

Reduce the shared-memory reservation from 49,152 bytes to 24,576 bytes by halving the padding budget (kPaddingElems from 11264 to 5120), which cuts per-block shared memory in half and allows the SM to fit 2 concurrent blocks instead of 1, doubling occupancy and yielding a 2x speedup without any change to the computation logic

## Key Lessons

- Reducing per-block shared-memory footprint (e.g. by switching from a 128x128 tile with preload staging to a 64x64 specialization without preload) doubles occupancy when the original allocation was near the single-block occupancy cliff — applies whenever padding or staging arrays inflate shared memory beyond the threshold for multiple concurrent blocks
- Occupancy-driven speedups from shared-memory reduction are orthogonal to arithmetic intensity: the computation loop is identical before and after, so the 2x gain comes purely from better SM utilization, not from any instruction-level optimization
