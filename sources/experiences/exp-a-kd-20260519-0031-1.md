## Challenge

When fused reduction (atomic accumulation) is active inside the main kernel, a separate reduction kernel is still unconditionally launched whenever split_kv > 1, causing double reduction that overwrites correct accumulated results and wastes GPU resources


## Solution

Add !is_fused_reduction guard to the reduction kernel launch condition so the separate reduction is only invoked when fused atomic accumulation is not already handling it, eliminating redundant work and fixing correctness

## Key Lessons

- Guard reduction kernel launches with the fused_reduction flag when split-KV accumulation is already performed atomically inside the main kernel — the two reduction paths are mutually exclusive and launching both causes double reduction and incorrect results
- Eliminating a redundant kernel launch saves not only the compute cost of the kernel itself but also the launch overhead, extra global memory reads, and an additional synchronization point — the cumulative effect can yield significant latency reduction
