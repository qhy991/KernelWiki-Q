## Solution

Replace scalar bf16 load/store loop with vectorized float4-width (8 bf16 elements, 16 bytes) aligned load/store via reinterpret_cast of an __align__(16) vec_n_t struct, reducing per-thread instruction count from 9 scalar pairs to 1-2 vector pairs

## Key Lessons

- Vectorizing memory access (float4-width load/store) reduces instruction count even when the scalar pattern is already warp-coalesced — the benefit comes from fewer LDG/STG instructions, not improved coalescing — applies when per-thread iteration count is high relative to the vector width
- Warp-level coalescing (consecutive threads accessing consecutive addresses with stride equal to CTA_SIZE) already provides near-optimal memory throughput for contiguous bf16 patterns — vectorization on top yields only marginal gains (~1-2%) unless combined with other optimizations that reshape the kernel execution pattern
- Use vec_n_t<T, N> with __align__(N*sizeof(T)) and reinterpret_cast to express vectorized load/store without explicit SIMD intrinsics — works for any element type where N elements fit a hardware vector width (float4=16 bytes for bf16×8)
