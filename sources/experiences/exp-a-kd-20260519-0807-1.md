## Challenge

The all-reduce launch configuration uses two iterative while-loops to find valid block and thread counts, which is unnecessarily complex, hard to reason about, and may produce suboptimal grid shapes that don't evenly divide the workload


## Solution

Replace the iterative while-loop block/thread search with direct std::min/divUp arithmetic: cap threads_per_block at min(DEFAULT_BLOCK_SIZE, total_threads) and blocks_per_grid at min(MAX_ALL_REDUCE_BLOCKS, divUp(total_threads, threads_per_block)), also lowering MAX_ALL_REDUCE_BLOCKS from 36 to 32 for a power-of-two block count

## Key Lessons

- Prefer closed-form min/divUp expressions over iterative while-loops for launch config computation — the O(n) search adds host-side overhead and complexity with no GPU-side benefit, since the kernel's bottleneck is memory-bound reduction work unaffected by minor grid shape differences
- Using power-of-two MAX_ALL_REDUCE_BLOCKS (32 vs 36) aligns with GPU warp granularity and simplifies reasoning about block distribution, though the performance impact is negligible for memory-bound reduction kernels
