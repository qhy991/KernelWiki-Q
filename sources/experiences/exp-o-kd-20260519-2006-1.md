## Solution

Add runtime M-dimension-based dispatch that selects one of three CUTLASS tile shape configurations based on next-power-of-2 of M: TileShape 64x64x256 (ClusterShape 1x8x1) for M<=64, TileShape 128x128x64 (ClusterShape 2x2x1) for M in (64,128], and the original TileShape 256x128x64 for M>128, reducing wasted compute from padding and improving occupancy for small-batch GEMM

## Key Lessons

- Select CUTLASS tile shapes proportional to the M dimension via runtime dispatch (using next_pow_2 bucketing) rather than using a single large tile for all problem sizes — smaller tiles reduce wasted padding compute and improve occupancy for small-batch inference workloads
- Use next_pow_2 of M with a floor of 64 to bucket into tile shape classes, providing a cheap O(1) dispatch that avoids per-tile-shape autotuning overhead while still capturing the dominant size regimes in LLM inference decoding
