## Solution

Reduce cluster from 4x4 (16 SMs) to 2x1 (2 SMs) and shrink the N tile from 256 to 128, cutting inter-SM synchronization cost while providing sufficient tile parallelism for medium M=128 problems

## Key Lessons

- For medium M dimensions (16 < M <= 256), reduce cluster width and N tile size to match actual work available — over-provisioning SMs in the cluster increases synchronization overhead without useful parallelism when M underutilizes the M tile
- Simplify 2D cluster mapping to 1D when CLUSTER_N=1: eliminate the local_id / sub-tile offset arithmetic and let blockIdx.y directly index tile rows, reducing address computation overhead
- Match tile/cluster configuration to problem dimensions: a 4x4 cluster with 256x256 tiles wastes resources on M=128 but is appropriate for M>=512, so use M-based config selection to pick the right tiling strategy
