## Challenge

B-tile global-to-shared memory copies in the tiled SGEMM main loop are synchronous (FLOAT4 assignment), blocking the thread until the transfer completes and preventing overlap with the 64-FMA compute body per BK step


## Solution

Replace synchronous FLOAT4 B-tile global-to-shared copies with cp.async.ca (asynchronous copy) via inline PTX, issuing async transfers before compute and waiting only after the 16-step × 64-FMA compute loop completes, allowing the hardware to overlap memory transfer with arithmetic

## Key Lessons

- Replace synchronous global-to-shared FLOAT4 copies with cp.async.ca to decouple memory transfers from compute — the hardware issues the async DMA while the warp executes FMAs, providing speedup when the compute body per tile (BK × TM × TN FMAs) is large enough to hide transfer latency
- Issue cp.async copies for the next tile early, then do synchronous register loads and compute, then CP_ASYNC_WAIT_GROUP(0) at the barrier — this ordering maximizes the overlap window between async transfer and compute
- Use __cvta_generic_to_shared to obtain the shared-memory pointer required by cp.async PTX instructions, and commit groups to signal transfer completion to the hardware before entering compute
