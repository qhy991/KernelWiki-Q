## Challenge

Scalar FP16 dot product kernel processes one half element per thread with 256 threads per block, using individual __hmul per element and scalar half loads from global memory, underutilizing available memory bandwidth for a memory-bound reduction workload


## Solution

Vectorize memory access using half2 loads (2 half elements per thread via reinterpret_cast), performing two __hmul operations and one __hadd per thread, but halving thread count from 256 to 128 — the occupancy reduction negates the memory bandwidth benefit in this memory-bound kernel

## Key Lessons

- Vectorizing scalar half loads to half2 while proportionally reducing thread count can regress performance in memory-bound kernels — the halved occupancy (256→128 threads) reduces the scheduler's ability to hide memory latency, offsetting the wider load benefit
- For memory-bound reductions where global memory access dominates total execution time, half2 vectorization must preserve or increase total thread count (e.g., keep 256 threads each doing half2 loads) to realize bandwidth gains — wider loads alone do not help when total data volume is unchanged and concurrency drops
- When the cross-warp reduction overhead is small relative to global memory access cost (4 warps vs 8 warps), reducing warps via vectorization provides negligible savings compared to the latency-hiding capacity lost from lower occupancy
