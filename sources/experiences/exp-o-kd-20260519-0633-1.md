## Solution

Replace serial global-memory scan with cooperative shared-memory load where threads load int4 chunks in parallel, then use the last warp to count valid tokens in parallel and reduce via __shfl_down_sync, converting O(n) serial latency into O(1) parallel load plus O(log warpSize) reduction that stays constant regardless of block size

## Key Lessons

- Serial global-memory scan loops whose latency scales with block size should be replaced with cooperative shared-memory loads where threads load in parallel — the serial latency grows linearly with block size while parallel load latency stays near-constant
- Using the last warp (threads THREADS-32 to THREADS-1) for warp-level reduction guarantees a full active warp for __shfl_down_sync without divergence concerns, while keeping the rest of the block available for the cooperative load
- Warp shuffle reduction via __shfl_down_sync converts an O(n) serial count into O(log warpSize) parallel reduction, and the cost stays fixed at 5 shuffle stages regardless of how many elements each lane processes
