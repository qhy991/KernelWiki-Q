## Challenge

Scalar bfloat16 load/store instructions in the concat loop body transfer only 2 bytes per memory transaction, severely underutilizing available global memory bandwidth for bulk tensor concatenation of 512-element nope and 64-element rope segments


## Solution

Use int4 (128-bit = 8 bfloat16) vector loads via inline PTX asm with cache-streaming (.cs) hints for the 512-element nope segment, where each warp collectively issues 32 vector loads covering 256 bytes per iteration; the 64-element rope segment uses scalar int (4-byte) PTX loads since it fits in a single warp-wide transaction

## Key Lessons

- Use int4 (128-bit) vector loads via PTX inline asm for bulk memory copies where source and destination have known contiguous inner dimensions — each int4 carries 8 bfloat16 values (16 bytes), giving 8x throughput over scalar loads with no additional register pressure per element
- The .cs (cache-streaming) PTX cache hint on ld.global.cs / st.global.cs marks data as streaming (read-once), preventing pollution of the L2 cache during bulk tensor transfers that will not be reused — apply when copying data that is consumed downstream without being re-read locally
