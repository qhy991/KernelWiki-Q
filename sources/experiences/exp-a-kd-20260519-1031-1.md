## Solution

Fuse the two-kernel pipeline into a single kernel using per-thread counting arrays in shared memory: each of 32 threads counts its own slice of tokens without atomicAdd, then a parallel prefix-sum across threads produces the global offset, enabling direct sorted_token_ids writes from the same kernel

## Key Lessons

- For small-batch MOE routing (≤256 tokens, ≤8 experts), fusing align+sort into one kernel with per-thread counting eliminates all atomicAdd contention and saves one kernel launch — the per-thread prefix-sum over shared memory is cheaper than global-memory atomics at this scale
- Per-thread counting arrays indexed as tokens_cnts[(threadIdx.x+1)*num_experts + expert] give each thread an exclusive write region, avoiding atomics entirely; the +1 offset leaves a zero-initialized row at index 0 for clean prefix-sum seeding
- Reducing thread count from 1024+256 to 32 for small workloads cuts shared-memory initialization overhead and improves cache locality — each thread processes ~8 elements, which amortizes the per-thread setup cost
