## Challenge

The bgmv shrink kernel uses cuda::memcpy_async pipeline to copy W and X into shared memory before computing, but for feat_in==feat_out=256 the single tile covers the entire dimension, so the pipeline never overlaps load with compute — making the pipeline infrastructure (producer_acquire/commit, consumer_wait/release) pure overhead


## Solution

Replace the cuda::memcpy_async pipeline-based shared memory loads with direct global-memory vector loads via float4 reinterpret_cast into registers, eliminating all pipeline overhead (producer_acquire/commit, consumer_wait/release), shared memory allocation (~8KB), and the tiled loop structure for the single-tile case

## Key Lessons

- When the problem dimension fits in a single tile, cuda::memcpy_async pipeline infrastructure adds pure overhead — direct global-memory vector loads via float4 reinterpret_cast into registers are faster because they eliminate pipeline synchronization stages and shared memory allocation
- Prefer register-based vector loads (float4 reinterpret_cast) over shared-memory staging when each thread's working set is small enough to fit in registers, avoiding shared memory capacity pressure and block.sync() barriers
- Use cg::thread_block_tile reduction with g.shfl_down/g.shfl over raw __shfl_down_sync + shared-array cross-warp reduction to avoid shared memory and extra block.sync() barriers for small thread groups
