## Challenge

The static_scaled_int8_quant function accepted scale as a float host scalar, forcing callers with GPU-resident scale tensors to invoke .item() which triggers a device-to-host cudaMemcpy synchronization, then the kernel launch re-sends that scalar back to the device via argument registers — an unnecessary CPU round-trip that stalls the CUDA stream pipeline


## Solution

Changed the scale parameter from float (host scalar) to torch::Tensor const& (device tensor pointer), so the kernel receives a device pointer (scale_type* scale_ptr) and reads the scale with *scale_ptr inside the kernel, eliminating the D2H sync and H2D re-transfer entirely

## Key Lessons

- Pass GPU-resident scalars as torch::Tensor const& (device pointer) instead of extracting them as host float via .item() — eliminates per-invocation cudaMemcpy D2H synchronization that stalls the asynchronous CUDA stream pipeline, especially impactful in tight inference loops
- Avoid .item() on device tensors in performance-critical paths — it forces a full device-to-host sync and breaks the asynchronous execution model, causing the measured speedup to be dominated by synchronization overhead rather than compute
