## Challenge

A transpose kernel is launched to remap a 128x128 scale tensor before a sum reduction, adding a full grid dispatch and 16K strided writes that dominate total execution time for such lightweight downstream computation


## Solution

Eliminate the transpose kernel entirely by computing the sum reduction directly on the original scale tensor, since Sm90BlockwiseScaleConfig now supports explicit GMMA major-mode parameters making the data layout conversion unnecessary

## Key Lessons

- Eliminating an unnecessary data layout transformation (transpose) when the consumer can operate on the original layout removes both a kernel launch overhead and a full-buffer write — especially beneficial when the downstream computation is lightweight relative to the copy cost
- Hardware features like GMMA major-mode parameters in Sm90BlockwiseScaleConfig can make previously required transpose operations redundant — always re-evaluate whether layout conversions are still necessary after hardware/software capability upgrades
