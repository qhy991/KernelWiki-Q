## Challenge

The after kernel early-exits for 224 out of 256 tokens (87.5% suffix-only), meaning most threads take the cheap copy path (one 128b pack load + one 128b pack store + one scalar LSE read) inste


## Solution

Replace synchronous FLOAT4 global-to-shared stores for B with cp.async.ca asynchronous copies: issue 2 CP_ASYNC_CA calls (16 bytes each) via __cvta_generic_to_shared pointer, commit the async group, then perform A-tile loading and transpose while B transfers are in flight, with CP_ASYNC_WAIT_GROUP(0) ensuring completion before proceeding

## Results

- **speedup**: 1.164x (improved)


## Key Lessons

- Use cp.async.ca to replace synchronous global-to-shared FLOAT4 stores for one tile (B), then commit the async group and load the other tile (A) synchronously — the A-loading work hides the B-transfer latency in the prologue
- Obtain the shared-memory pointer via __cvta_generic_to_shared before issuing cp.async, and use byte-offset addressing (ptr + i * 4 for 16-byte copies at stride 4 floats) to match the original vectorized access pattern
- CP_ASYNC_COMMIT_GROUP followed by CP_ASYNC_WAIT_GROUP(0) forms a full fence: commit launches pending async copies, wait(0) blocks until all committed groups complete — place useful independent work between them to amortize the latency
