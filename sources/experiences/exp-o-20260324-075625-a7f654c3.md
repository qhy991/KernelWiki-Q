## Solution

Single 256-thread kernel with __launch_bounds__(256, 4) for better occupancy (4 blocks/SM vs 2 blocks/SM with 512 threads), #pragma unroll 4 on inner loops, float4 vectorization, __ldg cache hints, and __forceinline__ warp reduction.

## Key Lessons

- For rms_norm with hidden_size=7168, 256 threads with __launch_bounds__(256,4) achieves better occupancy than 512 threads
- Removing the dual-kernel dispatch overhead and using a single well-tuned kernel is simpler and faster
- #pragma unroll 4 on the float4 reduction loop provides meaningful speedup
- The baseline's runtime kernel selection (512 vs 256 threads) adds overhead without benefit when 256 threads is optimal for all batch sizes tested
