## Challenge

A pipelined reduction kernel allocates separate shared memory regions for Phase 1 (tile buffer, 16KB) and Phase 2 (reduction buffer, 12KB), totaling 28800 bytes per block including metadata, which limits occupancy to only 5 blocks per SM on shared-memory capacity


## Solution

Overlap the Phase 1 tile buffer (sh_b) and Phase 2 reduction buffer (sh_red) at the same shared memory offset since they are used in strictly sequential phases separated by __syncthreads() barriers, reducing per-block shared memory to 16448 bytes and boosting occupancy from 5 to 8 blocks per SM

## Key Lessons

- When kernel phases use shared memory buffers sequentially (separated by __syncthreads()), overlap them at the same offset to reduce total shared memory per block — this directly increases occupancy when the kernel is shared-memory-limited
- Reducing per-block shared memory from 28800 to 16448 bytes can lift occupancy from 5 to 8 blocks/SM on GPUs where the shared memory ceiling is the occupancy bottleneck, yielding throughput gains proportional to the occupancy increase
- Use alias pointers (sh_b and sh_red pointing to the same offset) rather than union types for shared memory phase overlap — the code reads naturally and the compiler emits identical addressing arithmetic
