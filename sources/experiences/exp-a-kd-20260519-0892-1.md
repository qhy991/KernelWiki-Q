## Challenge

RMSNorm kernel launches 8192 blocks (one per token row) with only 32 threads each, exceeding GPU concurrent block capacity and causing excessive block scheduling overhead with poor SM occupancy


## Solution

Replace one-block-per-row launch with occupancy-based grid sizing (cudaOccupancyMaxActiveBlocksPerMultiprocessor) and a grid-strided loop, so each persistent block processes multiple rows — amortizing block setup cost and improving L2 cache locality for sequential row accesses

## Key Lessons

- Use cudaOccupancyMaxActiveBlocksPerMultiprocessor to compute the optimal block count and launch fewer persistent blocks with a grid-strided loop instead of one block per work item — this amortizes block scheduling overhead across multiple items and keeps SMs fully occupied
- Grid-strided loops improve L2 cache locality when processing sequential rows, since each SM reuses its local cache partition for consecutive row accesses rather than fetching from disjoint memory regions across many short-lived blocks
- The speedup from occupancy-based grid sizing diminishes as per-row computation increases (8 elems/thread vs 4), because scheduling overhead becomes a smaller fraction of total latency — the technique is most impactful for lightweight kernels
