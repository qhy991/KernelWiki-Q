## Challenge

The radix-based topk kernel hard-codes all input accesses as input[idx], assuming each batch row starts at offset 0 in the flattened input buffer, preventing the kernel from operating on non-contiguous or variable-offset input layouts


## Solution

Add a row_start parameter to fast_topk_cuda_tl and change all five input access sites from input[idx] to input[idx + row_start]; the topk_kernel resolves row_start from a per-batch row_starts array with a nullptr fallback to 0, maintaining backward compatibility for contiguous data while enabling variable-offset access

## Key Lessons

- Adding a runtime offset parameter to generalize kernel addressing costs at most one integer ALU per access — negligible relative to the dominant shared-memory atomics and radix sort in topk, even when the compiler cannot eliminate the addition for a runtime variable
- The nullptr fallback pattern (row_starts == nullptr ? 0 : row_starts[bid]) preserves backward compatibility for contiguous data at zero performance cost while extending the API to non-contiguous layouts
