## Solution

Fuse gather and dequantize into a single kernel that reads FP8 data directly from paged cache blocks and writes converted BF16 output in the same loop iteration, using 576 threads matching the per-token work (512 dequant + 64 RoPE copy), eliminating the intermediate buffer allocation, the second kernel launch overhead, and the O(batch) linear scan by using blockIdx.x as the batch identifier

## Key Lessons

- Fusing a gather (data movement) kernel with a compute (dequantize) kernel into a single pass eliminates intermediate buffer allocation and the associated write-then-read round trip — applies whenever the second kernel reads data produced by the first with no inter-token dependencies
- Matching thread count to actual per-element work (576 = 512 dequant + 64 RoPE copy) instead of over-provisioning with a generic 1024-thread config avoids wasted threads and improves SM resource utilization — applies when kernel work decomposes into fixed-size per-element tasks
- A fused paged-cache kernel can use blockIdx-based batch identification instead of a per-token O(batch) linear scan over workspace_starts, converting the lookup to O(1) — applies when the grid dimension already encodes the batch index
