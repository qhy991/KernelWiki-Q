## Challenge

AttentionBackwardKernel was always instantiated with unbounded kMaxK, forcing kBlockSizeI=128 and 8 warps per block even when small K/V dimensions (<=64) only needed a 64x64 tile, wasting threads, shared memory, and scheduling overhead


## Solution

Add a DISPATCH_MAXK macro that computes max(query_dim, value_dim) at runtime, buckets it into compile-time kMaxK values {64, 128, inf}, and passes the bound into AttentionBackwardKernel so the kernel traits resolve to a smaller 64x64 tile (4 warps) when maxK<=64 instead of always using the 64x128 path (8 warps)

## Key Lessons

- Bucket runtime dimensions into compile-time constants via a dispatch macro so kernel templates can specialize tile sizes — for attention backward, passing kMaxK<=64 selects a 64x64 tile (4 warps) instead of the generic 64x128 tile (8 warps), cutting thread count and shared memory in half for small head dimensions
- Compile-time dimension bounds enable the compiler to eliminate unused tile columns/rows entirely, producing tighter inner loops that cannot be achieved with runtime-only dimensions even when the runtime value is small
