## Challenge

GEMM kernel uses BM=128 tile with 16KB shared memory per block (128*32+32*128 half elements), limiting the number of concurrent blocks per SM and reducing occupancy even when M=128 perfectly fits the tile


## Solution

Reduce tile BM from 128 to 64 (and TM from 4 to 2) to cut shared memory per block from 16KB to 12KB, enabling more concurrent blocks per SM and better occupancy while launching 2 blocks in y instead of 1 to distribute work across more parallel resources

## Key Lessons

- Reducing shared memory per block (BM 128→64 cuts smem from 16KB to 12KB) increases occupancy by allowing more concurrent blocks per SM — applies whenever shared memory is the occupancy bottleneck rather than thread count
- Choosing a smaller BM tile that still divides evenly into M (BM=64 splits M=128 into 2 full blocks) can outperform a larger tile that fits exactly (BM=128 = 1 block), because the occupancy gain from reduced smem outweighs the cost of launching more blocks
- Tile tuning should target occupancy: when THREADS_PER_BLOCK is fixed at 1024, reducing BM/TM (128/4→64/2) keeps thread count constant while freeing shared memory, which is the key lever for SM-level parallelism
