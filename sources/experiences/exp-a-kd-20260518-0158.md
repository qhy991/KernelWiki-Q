## Challenge

In the before source, KCOUNT_BEFORE=8 with TILE_COLS=256 means only 1 grid block covers 256 columns (for cols=256), so all 32 threads along X each handle 8 floats sequentially per partition


## Solution

Fix kCount to 128-bit / 32-bit (float) = 4 instead of 128-bit / 16-bit (half_t) = 8, matching workspace element type; this halves TILE_COLS to 128, doubling grid blocks in Y for higher occupancy, reducing per-thread registers from 8 to 4 accumulators, and enabling properly coalesced 128-bit vector loads

## Results

- **speedup**: 1.057x (improved)


## Key Lessons

- Compute vector access width (kCount) as vector_bits / workspace_element_bits, not output_element_bits — mismatching to output type causes threads to stride past valid data per aligned segment and reduces grid parallelism
- Smaller kCount increases grid Y-dimension (TILE_COLS shrinks, more blocks launched), improving GPU occupancy for memory-bound reduction kernels — applies whenever per-thread vector width can be reduced without losing coalescing
- Reducing per-thread accumulator array size (8→4 floats) lowers register pressure, allowing more warps per SM and higher occupancy when the kernel is register-constrained
