## Solution

Fuse zero-initialization into the kernel by expanding the grid to cover all padding groups (rounding num_groups up to TMA-aligned padded count), so padding threads write zero bytes directly via conflict-free byte stores, eliminating the separate cudaMemsetAsync call and replacing atomicOr with direct reinterpret_cast<uint8_t*> writes

## Key Lessons

- Fusing buffer zero-initialization into the compute kernel by expanding the grid to cover padding elements eliminates a separate cudaMemsetAsync call — applies whenever output positions can be partitioned into valid and padding groups processed by the same kernel
- Replacing atomicOr with direct byte writes becomes safe when the expanded grid guarantees each output byte position is written exactly once by exactly one thread — eliminates atomic contention, reduces instruction count, and avoids the need for pre-zeroing
- The speedup from eliminating a separate memset scales with the ratio of memset cost to kernel cost; even for small buffers where memset is a small fraction of total time, removing the extra kernel launch overhead still yields measurable improvement
