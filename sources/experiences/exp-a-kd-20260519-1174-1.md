## Challenge

FP8 quantization uses a GPU segmented max-reduction kernel to compute the scale factor, requiring a full pass over input data with shared-memory halving reduction and atomicMaxFloat across 256 blocks, followed by a second full pass for quantization — doubling memory bandwidth consumption


## Solution

Replace the GPU-based segmented max reduction with a CPU pre-computed scale from host data (single linear scan for max-abs), eliminating the entire reduction kernel launch, its shared-memory reduction stages, and atomicMaxFloat contention — reducing GPU work to a single quantization pass

## Key Lessons

- When data is already available on the host, compute scalar aggregates (max, sum) on CPU instead of launching a dedicated GPU reduction kernel — eliminates an entire GPU pass, shared-memory reduction overhead, and atomic contention, effective for datasets that fit in host memory
- At small data sizes (~1MB, 256K elements), the data traversal cost dominates kernel execution time; removing a redundant full-data pass nearly halves total latency, as shown by the ~2.0x speedup
- Pre-computing static scales on host and uploading once avoids per-iteration scale reinitialization (cudaMemcpy of -1e30f before each reduction), further reducing launch overhead in benchmark loops
