## Challenge

E8M0 scale factor dequantization uses four exp2f calls and four __float2bfloat16 conversions per int32 element (8 FP function calls total), routing through the FP32 arithmetic pipeline with multiple micro-ops per call and creating heavy instruction count and FP pipeline pressure overhead


## Solution

Replace all eight FP function calls per int32 element with three pure bitwise integer operations that exploit the direct E8M0-to-BF16 bit pattern mapping: mask odd-position bytes and shift right by 1 for Out1, left-shift q by 7 then mask for Out2 — these map to single-cycle PTX SHL/SHR/LOP3 instructions on the integer ALU, entirely bypassing the FP pipeline

## Key Lessons

- E8M0 exponent byte e maps directly to BF16 bit pattern (sign=0, exponent=e, mantissa=0, bits = e<<7), enabling pure bitwise dequantization without any FP arithmetic — applies to all E8M0-to-BF16 scale factor conversions in FP8 block-scaled formats
- When a mathematical transformation has a known closed-form bit-level representation, prefer integer shift/mask over FP intrinsics (exp2f, __float2bfloat16) to reduce instruction count from 8+ FP ops to 3 single-cycle integer ops and eliminate FP pipeline contention
- Operating on two BF16 values packed in a 32-bit register simultaneously via masks like 0xFF00FF00 and 0x7F807F80 avoids per-element extraction and reconstruction — the key to making bitwise conversion work at int32 granularity
