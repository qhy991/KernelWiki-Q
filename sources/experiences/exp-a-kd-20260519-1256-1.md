## Challenge

When group_blocks=2 and thread_k_blocks is small (≤4) in the Marlin FE2M1f scale fragment loading path, consecutive k-iterations (k=0, k=1) share the same cur_group_id and the single-branch index formula always reads from the same int2 slot, selecting the wrong sub-element from the packed FE4M3fn scale pair


## Solution

Split the FE2M1f scale fragment index computation into two branches: when group_blocks==1 or thread_k_blocks > 4, use the original formula; otherwise add a k%2 offset so that k=0 reads slot 0 and k=1 reads slot 1 within the packed FE4M3fn scale pair, ensuring correct sub-element selection for the MXFP4 group_blocks=2 case

## Key Lessons

- When packed quantization formats (FE4M3fn/MXFP4) store multiple sub-elements per int2 slot, the scale index formula must account for k-iteration offset within the same group — a missing k%2 offset causes both k-iterations to read the same sub-element instead of alternating slots
- Correctness fixes for quantization scale indexing can be neutral on performance when both branches involve the same number of arithmetic operations — the change is about selecting the right data, not about reducing instruction count
