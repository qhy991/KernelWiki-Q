## Challenge

GEMM tiled kernel loads B matrix tiles from global to shared memory using synchronous FLOAT4 copies within the same loop as A matrix loads, blocking the compute pipeline during each tile transfer with no overlap between memory loads and independent work


## Solution

Replace synchronous B-tile global-to-shared FLOAT4 copies with cp.async asynchronous copy instructions (CP_ASYNC_CA macro using cp.async.ca.shared.global), separating B loads into an independent async pipeline that overlaps with A-matrix FLOAT4 loads and shared-memory transpose writes, managed by CP_ASYNC_COMMIT_GROUP and CP_ASYNC_WAIT_GROUP(0) for synchronization

## Key Lessons

- cp.async (CP_ASYNC_CA) allows global-to-shared memory transfers to proceed asynchronously, overlapping with independent work such as A-matrix loads and shared-memory transpose writes — applies whenever shared memory tiles are loaded from global memory and there is adjacent work to hide the transfer latency
- For small problem sizes (M=N=K=512) with short kernel runtimes (~0.09ms), cp.async overlap savings are proportionally small relative to total execution time and may fall within measurement noise — the technique yields larger gains at bigger K dimensions where load latency hides more computation across more BK iterations
- Separating B-tile async loads from A-tile synchronous loads into distinct loops enables the async pipeline to transfer B data while the thread executes A loads and transpose, using CP_ASYNC_COMMIT_GROUP to launch the group and CP_ASYNC_WAIT_GROUP(0) to fence before consumption
