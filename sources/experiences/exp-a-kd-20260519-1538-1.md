## Solution

Swap tile dimensions to {TILE_K=128, TILE_N=64} which produces 2 N-tiles × 32 K-tiles = same 128 total sync barriers, adding overhead from the extra N-tile loop iteration (redundant shared memory loads, accumulator resets, outer loop branch) without reducing synchronization cost

## Key Lessons

- Swapping TILE_K and TILE_N dimensions does not reduce total sync barrier count when num_n_tiles × num_k_tiles remains constant — verify that the tile reconfiguration actually changes the synchronization product before expecting gains
- Larger TILE_K only benefits kernels with per-tile fixed overhead (e.g., Marlin's quantize-dequant pipeline); in plain FP16 GEMM without per-tile dequantization, the larger K-tile adds shared memory traffic without amortizing any fixed cost
- Adding N-tile iterations introduces redundant shared memory loads, accumulator resets, and outer-loop branching overhead that degrades performance when thread count and total work are unchanged
