## Challenge

SF (scale factor) offset function recomputes numKTiles=(numCols+63)/64 on every call and uses arithmetic division/modulo (/128, /4, %32, %128/32, %4) with separate named stride variables, adding instruction overhead in the inner loop of a quantization kernel


## Solution

Precompute numKTiles once at kernel entry and pass it as parameter, replace all arithmetic division/modulo with bitwise shift/AND (>>7, >>2, &31, >>5/&3, &3), and fold the separate stride-multiply expression into a compact bitwise OR formula ((mTileIdx*numKTiles+kTileIdx)<<9 | outerMIdx<<4 | innerMIdx<<2 | innerKIdx)

## Key Lessons

- Manual arithmetic-to-bitwise conversion of power-of-2 divisions and modulos is redundant when compiling at -O3, because nvcc already emits equivalent shift/AND instructions → verify with SASS/PTX inspection before hand-optimizing integer arithmetic
- In memory-bound kernels, per-thread savings of 5-8 arithmetic instructions are negligible relative to memory latency and FP conversion overhead → prioritize memory access pattern optimization over arithmetic simplification for measurable gains
- Precomputing loop-invariant values at kernel entry and marking helper functions __forceinline__ improves readability but does not change performance when the compiler already hoists invariants and inlines small functions automatically
