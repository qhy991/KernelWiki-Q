## Solution

Reduce pipeline stages from 4 to 2 and propagate stages as a runtime parameter through get_scales_cache_size and get_kernel_cache_size, halving all stage-dependent shared memory buffers (sh_a_size, sh_b_size, sh_s_size, sh_g_idx_size) and bringing total shared memory down to 39040 bytes — well within Turing's 64KB limit

## Key Lessons

- Reducing pipeline stages (pipe_stages) is the most effective shared memory tuning knob for quantized GEMM — it linearly scales all stage-dependent buffers (sh_a_size, sh_b_size, sh_s_size, sh_g_idx_size) while non-stage-dependent components (b_size, sh_block_meta_size, tmp_size) remain constant, so the speedup ratio depends on the fixed-overhead fraction
- Propagating pipeline stages as a runtime parameter (stages_param) instead of a hardcoded global enables per-GPU tuning: fewer stages for shared-memory-constrained architectures (Turing 64KB), more stages for architectures with larger shared memory (Ampere 100KB+, Hopper 228KB+)
- W8A8 quantization with pack_factor=4 produces larger b_size than 4-bit (pack_factor=8), meaning the fixed-cost fraction is higher — the relative benefit of reducing stages is lower than for 4-bit configs because the non-reducible b_size dominates a larger share of total shared memory
