## Challenge

Uniform kChunkSize=512 chunk iteration requires 6 chunks for seqlen=3000, incurring excessive per-chunk overhead (boundary index arithmetic repeated per chunk) and poor ILP when processing small uniform chunks instead of fewer larger variable-length ones


## Solution

Replace uniform kChunkSize iteration with variable-length chunks read from a pre-computed cumulative-offset array (cu_chunk_seqlen), using a running tokens_processed counter instead of chunk*kChunkSize arithmetic, halving the chunk count (6→3) and enabling a large middle chunk (2048 tokens) with better ILP

## Key Lessons

- Pre-computed cumulative-offset arrays (cu_chunk_seqlen) enable variable-length chunk iteration that reduces chunk count and per-chunk overhead — applies when sequence boundaries are misaligned with fixed chunk sizes (e.g., prefix caching, varlen batching)
- A running tokens_processed counter eliminates repeated chunk*chunkSize multiplication and naturally supports variable chunk boundaries — applies whenever chunk sizes vary at runtime
- Larger contiguous chunks (2048 vs 512 tokens) improve ILP within each thread's inner loop by giving the scheduler more independent operations to overlap — applies when chunk boundaries are flexible and can be coalesced
