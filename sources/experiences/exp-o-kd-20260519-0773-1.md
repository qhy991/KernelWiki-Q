## Challenge

MoE token routing for small batches uses two separate kernels (align kernel with 1024 threads + sort kernel with 256 threads) both relying on atomicAdd—shared-memory atomicAdd for counting and global-memory atomicAdd for sorting—while most threads are idle when numel is only 64


## Solution

Fuse the two kernels into a single kernel launched with only 32 threads (one warp), where each thread maintains private per-expert counts via a 2D shared-memory array indexed by (threadId, expert), then computes per-expert prefix-sum across threads and directly writes sorted positions without any atomicAdd

## Key Lessons

- For small-batch MoE routing, fuse align+sort into a single kernel launched with one warp (32 threads) — eliminates kernel launch overhead, cudaMemset, and ensures every thread has real work
- Replace atomicAdd with per-thread private counting in shared memory (tokens_cnts[threadId*experts+expert]) plus a simple prefix-sum — eliminates all atomics when thread count is small and data fits in shared memory
- Right-size thread count to the workload: 32 fully-utilized threads outperform 1280 mostly-idle threads when numel=64, because idle threads still consume scheduling overhead and atomics on mostly-empty bins are pure waste
