## Challenge

The Marlin quantized GEMM kernel uses a compile-time constant pipe_stages=4 for shared memory sizing across all GPU architectures. On Turing GPUs (SM75) with only 64KB shared memory per SM versus Ampere's 164KB, the 4-stage pipeline causes A-matrix, B-matrix, scales, and g_idx buffers to exceed shared memory limits, preventing most thread configurations from passing the is_valid_config check.


## Solution

Replace the compile-time pipe_stages=4 constant with a runtime stages parameter: on Ampere+ (SM80+) stages remains 4, on Turing (SM75) stages is set to 2. This halves shared memory consumption for all pipeline-stage-dependent buffers on Turing, enabling thread configs like {128,128,256} and {64,128,128} to fit within the 64KB shared memory limit while maintaining a 2-stage pipeline for latency hiding.

## Key Lessons

- Replace compile-time pipeline-stage constants with runtime parameters gated on SM architecture version when shared memory budgets differ across target GPUs — this allows the same kernel binary to adapt its shared memory footprint per-architecture rather than over-provisioning for the smallest target
- Even a 2-stage pipeline provides meaningful latency hiding through global-to-shared prefetching; reducing from 4 to 2 stages is preferable to failing shared memory validation entirely on memory-constrained architectures like Turing SM75
- Thread config selection (tile dimensions) is tightly coupled to shared memory sizing — making pipeline stages runtime-configurable re-enables otherwise-valid tile sizes that were rejected solely due to excessive pipeline buffer overhead
