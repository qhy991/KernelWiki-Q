## Solution

Replace 8 scalar __hadd calls with 4 packed __hadd2 calls using __halves2half2 packing and __low2half/__high2half unpacking, halving instruction count from 1600 to 800 per thread over 200 iterations

## Key Lessons

- On Ampere (sm_80), scalar __hadd is already a single native FP16 ALU instruction with the same per-instruction issue rate as __hadd2 — packed FP16 operations provide no throughput advantage when the kernel is not instruction-issue-bound and the FP16 ALU pipeline has surplus capacity
- Packing (__halves2half2) and unpacking (__low2half/__high2half) introduce register manipulation overhead that can fully offset the instruction count reduction from __hadd2, producing net-neutral performance for FP16 arithmetic-intensive kernels
- Packed __hadd2 is transformative for BF16 where scalar add requires a costly convert-float/add/convert-back sequence, but for FP16 where scalar add is already native, the optimization is unnecessary — always profile the scalar baseline before assuming packed intrinsics will help
