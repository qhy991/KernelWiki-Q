## Challenge

Generic NHWC channel-padding kernel processes one scalar float per thread with per-element modular index decomposition (idx%c_out, cascaded divisions for w/h/n), causing uncoalesced 4-byte global stores and expensive integer arithmetic on every element


## Solution

Replace scalar per-element processing with shared-memory tiling: 192 threads cooperatively load float4 vectors into shared memory, then each thread assembles one float4 output (3 data floats + 1 zero padding) and writes it as a single 128-bit coalesced store, eliminating per-element index decomposition and reducing global memory transactions by 4x

## Key Lessons

- For fixed-ratio channel padding (e.g., 3→4), use float4 vectorized I/O so that each 128-bit store naturally contains exactly the right number of data elements plus padding zeros — reduces global memory transactions by 4x and eliminates per-element modular index arithmetic
- Shared-memory cooperative loading followed by per-thread assembly decouples the input read pattern from the output write pattern, enabling both to be coalesced even when the input/output channel counts differ
- Specializing a kernel for a known channel ratio (3→4) allows compile-time-constant logic ((k+1)%4==0 for padding slot) that replaces the generic runtime c_idx >= c_in branch, removing divergent control flow
