## Challenge

Three separate CUDA kernels (two independent GEMM dot-product kernels plus a SiLU-multiply elementwise kernel) are launched sequentially to compute a dual-GEMM with SiLU gate, incurring two extra kernel launch overheads and forcing intermediate GEMM results to be written to and read back from global memory between stages


## Solution

Fusing all three kernels into a single fused_dual_gemm_kernel that computes both dot products (sharing the x vector load) and the SiLU*multiply epilogue in one pass — however, each thread must now hold two full I-length dot-product accumulators simultaneously, increasing register pressure and reducing occupancy enough to regress performance on this problem size (B=128, I=512, H=512)

## Key Lessons

- Kernel fusion that merges multiple independent dot-product loops into one thread doubles the per-thread register footprint (two accumulators live simultaneously) and can reduce occupancy enough to regress performance — always profile fused kernels against multi-launch baselines on the target problem size
- Sharing an input vector load (x[row*I+k] reused for both acc0 and acc1) is a genuine fusion benefit that halves x reads, but the saving is only profitable when the register-pressure cost does not dominate occupancy on the given SM architecture and tile dimensions
- Naive element-wise GEMM kernels (one output element per thread, no shared-memory tiling) are already register-light; fusing them removes headroom and makes register spilling more likely, whereas tiled GEMM implementations with shared memory are better candidates for fusion because their register budgets are already managed
