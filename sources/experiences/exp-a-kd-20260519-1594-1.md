## Solution

Replace the sort-based pipeline with a single-pass atomic scatter: compute_arg_sorts uses atomicAdd on per-expert counters to assign each token to its expert's contiguous output slot in O(N), eliminating CUB sort, init_indices, and all auxiliary sort buffers (sorted_ids, sorted_orig_idx, orig_idx_in, temp_storage)

## Key Lessons

- Replacing O(N log N) sort-based expert permutation with O(N) atomic-scatter eliminates 4 radix-sort passes and all auxiliary buffers (sorted_ids, sorted_orig_idx, orig_idx_in, temp_storage), yielding 2.15x speedup — applies whenever the task is binning/scattering elements into per-group contiguous slots rather than producing a fully sorted order
- atomicAdd on a small number of per-expert counters is cheap despite contention because the counters fit in a single cache line and the atomic volume is bounded by num_experts (8), far lower than the N-element sort traffic it replaces
- For MoE expert routing with few experts and high per-expert token density, the sort is overkill: the real requirement is partitioning tokens by expert ID, not establishing a total order — a scatter with pre-computed offsets suffices
