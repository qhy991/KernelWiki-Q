## Challenge

Two separate GEMM kernels computing D = (A * B0) * B1 materialize the intermediate result C_int to global memory, requiring a full write+read round-trip that wastes memory bandwidth and adds kernel launch overhead between the two matmuls


## Solution

Fuse both GEMMs into a single kernel so accum0 (A*B0 result) stays in registers and is consumed directly by the second GEMM — however, the simplified SIMT implementation incorrectly maps accumulator elements to the second GEMM's K-slices by reusing accum0[ri*RTN+ci] across all K offsets, introducing redundant computation and causing a 0.875x regression instead of the expected improvement

## Key Lessons

- B2B GEMM fusion requires a correct accumulator-to-fragment mapping (CUTLASS FragmentIteratorA1) that extracts the right accumulator elements for each K-slice of the second GEMM — naively reusing accum0[ri*RTN+ci] for all K offsets produces redundant computation and incorrect results
- Register accumulator reuse in fused GEMM is a real optimization (avoids one shared-memory write+read round-trip per tile and reduces smem pressure), but capturing the benefit requires warp-level MMA and proper fragment iteration infrastructure, not a simplified SIMT loop
- Separate kernel launches for chained GEMMs incur not only global memory bandwidth cost for the intermediate but also kernel launch overhead and L2 cache eviction between launches — fusion eliminates all three, but only when the accumulator layout is correctly handled
