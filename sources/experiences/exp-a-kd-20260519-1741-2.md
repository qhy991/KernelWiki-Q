## Challenge

With BLOCK_SIZE=16, NUM_V_VECS_PER_ROW=2 limits each warp to only 2 threads collaborating per value cache row, restricting parallelism across the HEAD_SIZE=128 dimension to NUM_ROWS_PER_ITER=16 rows with 8 rows per thread, and the inline Float8_ construction from uint4 requires four separate __half22float2 calls


## Solution

BLOCK_SIZE=32 raises NUM_V_VECS_PER_ROW from 2 to 4, spreading each warp's 32 threads across NUM_ROWS_PER_ITER=8 value rows with 16 rows per thread, increasing per-thread work and improving ILP; the cast_to_float8 helper encapsulates the uint4-to-Float8_ conversion, though the per-row warp reduction now requires more shuffle steps (4→2→1 vs 2→1)

## Key Lessons

- Increasing BLOCK_SIZE in paged attention value accumulation raises NUM_V_VECS_PER_ROW, distributing warp threads across more value rows per iteration — this increases per-thread accumulator count (8→16) which improves instruction-level parallelism at the cost of more shuffle reduction steps per row
- Encapsulating uint4-to-Float8_ half-to-float conversion in a cast_to_float8 helper reduces code duplication and matches the compiler's ability to optimize the four __half22float2 conversions as a group
