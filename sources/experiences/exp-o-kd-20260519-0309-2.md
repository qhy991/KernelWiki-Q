## Solution

Replace the 1D strided loop with a 2D grid where blockIdx.x maps to row-groups and blockIdx.y × threadIdx.y maps to a dedicated float4 column index, so each thread owns exactly one vector element; additionally, a compile-time template parameter kProcessRows lets each block handle 1/2/4 tokens selected at runtime based on total work, amortizing expert-lookup overhead and improving occupancy for large problems

## Key Lessons

- A 2D grid where one dimension maps to row-groups and the other to column vectors eliminates the strided inner loop, giving each thread a dedicated (row_group, vec_idx) slot — improves memory coalescing because adjacent threadIdx.y values access adjacent float4 elements
- Adaptive multi-row processing via a compile-time template parameter selected at runtime lets small workloads use kProcessRows=1 for maximum grid blocks and occupancy, while large workloads use kProcessRows=4 to amortize per-row expert-lookup and parameter loads across multiple tokens in the same thread
