## Challenge

The same division-to-multiplication replacement applies as the large case. The speedup is smaller (~6%) because with fewer total eleme


## Solution

Replace per-thread scalar half load/store with half2 (32-bit) vectorized access so each thread processes two adjacent half values in a single coalesced transaction, then unpack into .x/.y lanes for element-wise GELU computation since no native half2 tanh exists

## Results

- **speedup**: 1.060x (improved)


## Key Lessons

- Use half2 vectorized load/store instead of scalar half when processing fp16 elementwise kernels — halves fill 32-bit coalesced segments fully and cut memory transactions by ~2x, applicable to any memory-bound fp16 kernel
- When no native half2 math intrinsic exists (e.g., tanh for GELU), load as half2 for bandwidth but unpack into .x/.y lanes for scalar computation — still gains the memory throughput benefit without requiring vectorized math support
