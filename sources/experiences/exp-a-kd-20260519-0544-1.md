## Challenge

Scalar HGEMV kernel processes one half element per thread per iteration, issuing 64-byte memory transactions per warp and requiring 32 loop iterations for K=1024, resulting in high branch overhead and underutilized memory bandwidth


## Solution

Replace scalar half loads with half2 vectorized loads (f16x4 pattern) using reinterpret_cast, processing 4 half values per thread per iteration through half2 registers, reducing loop iterations from 32 to 8 and doubling per-warp effective bandwidth from 64 to 128 bytes per iteration

## Key Lessons

- Use half2 vectorized loads (f16x4 pattern) in HGEMV kernels to double per-warp memory bandwidth and reduce loop iterations by 4x — applies whenever K is a multiple of 4 and memory bandwidth is the bottleneck
- Reducing loop iterations in memory-bound kernels decreases branch overhead and index computation cost proportionally — the instruction count reduction (96→64 ops) directly contributes to latency improvement
- The HALF2 macro via reinterpret_cast<half2*> enables vectorized loads from scalar half arrays without requiring data layout changes — use this pattern when transitioning scalar kernels to vectorized access
