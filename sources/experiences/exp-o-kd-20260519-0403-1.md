## Challenge

FP32 RMS norm kernel uses scalar 32-bit loads and stores per thread, issuing four separate memory transactions for every four elements and launching 256 threads per row, which underutilizes memory bandwidth and increases warp scheduling and barrier synchronization overhead in block_reduce_sum_f32


## Solution

Replace scalar float loads/stores with float4 reinterpret_cast vectorized 128-bit transactions, reducing threads per row from 256 to 64; each thread accumulates four squared values locally before entering block reduction, shortening the warp-level and cross-warp reduction chains

## Key Lessons

- Use float4 reinterpret_cast vectorized loads/stores to coalesce four contiguous FP32 elements into a single 128-bit memory transaction, quadrupling effective memory bandwidth per thread — applies whenever data is contiguous and 16-byte aligned
- Reducing thread count by a vectorization factor (256→64) shrinks the block reduction tree, lowering both warp scheduling overhead and __syncthreads barrier cost — the local accumulation of N elements per thread before reduction shortens the reduction chain by a factor of N
- The FLOAT4 macro via reinterpret_cast<float4*> is the idiomatic zero-overhead way to issue wide loads on already-allocated float arrays — avoids separate allocation or copy into float4 buffers
