## Solution

The after kernel wraps all loads and stores in inline PTX asm with L1::no_allocate hints (ld.global.nc.L1::no_allocate and st.global.L1::no_allocate), intending to save L1 cache capacity for other data, but this strategy regresses on Ampere because the streaming copy data benefits from L1 hits after recent writes and there is no competing data to justify preserving L1 capacity

## Key Lessons

- L1::no_allocate cache hints hurt streaming copy kernels when input data has temporal locality from recent writes — standard loads/stores that allow L1 caching can deliver ~30-cycle hits versus ~200-cycle L2-only access
- st.global.L1::no_allocate only saves L1 capacity for future use, providing no store throughput benefit — only apply it when other code in the same kernel actively competes for L1 space
- Inline PTX asm wrappers for loads/stores add instruction overhead (function call + memory barriers) compared to plain pointer dereference — the cost is only justified when cache hints produce measurable latency reduction
