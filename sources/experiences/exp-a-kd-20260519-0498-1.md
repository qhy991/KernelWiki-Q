## Challenge

The kStage=1 Flash Attention kernel blocks after each Q/K tile load with CP_ASYNC_WAIT_GROUP(0)+__syncthreads(), stalling the MMA pipeline while waiting for shared memory to be ready before computation can proceed


## Solution

Switch to kStage=2 double-buffered async prefetch using CP_ASYNC_CG into alternate smem buffers (smem_sel / smem_sel_next), prefetching the next Q/K tile while the current tile's HMMA16816 MMA computation proceeds, and only synchronizing with CP_ASYNC_WAIT_GROUP(kStage-2)+__syncthreads() after MMA completes

## Key Lessons

- Double-buffered async prefetch (kStage=2) may not benefit small problem sizes where the inner loop has few iterations (only 4 here with kHeadDim/kMmaAtomK=4), as the compute-per-iteration window is too short to overlap meaningful prefetch work
- Software pipelining overhead (stage-selection logic, extra CP_ASYNC_COMMIT_GROUP calls, initial kStage-1 prologue section) can outweigh latency-hiding gains when the kernel runtime is very short (~0.036ms) and the data already fits in L2 cache (~256KB total for Q/K/V tiles)
- Doubling shared memory footprint for double buffering (kStage=2 uses 2x smem) may reduce concurrent blocks per SM and hurt occupancy, especially problematic when the original smem usage is already significant
