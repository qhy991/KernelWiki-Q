## Solution

Replace the scalar stride loop with a generic vectorized_process<float4> template that loads 4 floats per LDG, decomposes via a union, and invokes a lambda for each element — but the alignment-skip computation, union decomposition loop, and remainder-handling branches add more instruction overhead than the bandwidth savings from explicit float4 loads at 128K (~512KB) data sizes where L2 cache already captures most accesses

## Key Lessons

- For workloads under ~1MB where data fits in L2 cache, the GPU memory controller coalesces scalar float accesses into wide transactions automatically — explicit float4 vectorization adds instruction overhead (alignment skip, union decomposition, remainder branches) that exceeds any bandwidth benefit
- Avoid generic vectorized-load templates with alignment handling on small datasets: the reinterpret_cast-based alignment computation, remainder-element boundary processing, and warp-divergent skip/remainder branches cost more instructions than a simple coalesced stride loop
- When each element already requires an atomicAdd to shared memory, vectorized loads do not reduce atomic contention — the bottleneck shifts from memory bandwidth to shared-memory atomics, so vectorization provides diminishing returns
