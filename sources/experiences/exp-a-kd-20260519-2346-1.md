## Challenge

MoE grouped GEMM stride initialization dispatches 3 separate fill_stride_kernel launches plus 1 init_offsets_kernel launch (4 total kernel launches), each incurring full CUDA driver dispatch overhead (~5us) for trivial constant-value scalar writes across only 256 elements


## Solution

Fuse the three separate stride fill kernels into the existing init_offsets_kernel by adding stride output pointers and constant stride values as kernel parameters, enabling each thread to write its expert's stride slot alongside the existing offset computation in a single kernel launch, reducing total launches from 4 to 1

## Key Lessons

- Fuse trivial fill/initialization kernels into adjacent computation kernels that already iterate over the same thread indices to eliminate per-launch driver dispatch overhead (~5us each) — applies when fill values are constant or cheaply computable per-element within the fused kernel
- Kernel fusion yield is proportional to the ratio of eliminated launch overhead to total pipeline time — when the computation kernel dominates latency (~12us), removing three ~5us fill launches produces only a 1.6% improvement within measurement noise, making the optimization correct but marginal for this workload scale
