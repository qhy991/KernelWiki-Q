## Solution

Refactor the kernel into a CUTLASS-style functor pattern with __launch_bounds__(256, 2) to cap registers at 64 per thread (targeting 2 blocks/SM) and __grid_constant__ to pass the parameter struct through constant cache instead of by-value copy, though NVCC -O3 already manages this register budget effectively

## Key Lessons

- __launch_bounds__(maxThreads, minBlocks) caps per-thread register count to guarantee target occupancy, but when NVCC -O3 already fits the working set within the resulting register budget, no spilling reduction occurs and the hint yields no measurable speedup
- __grid_constant__ passes kernel arguments via constant cache (avoiding per-thread local-memory copies of large parameter structs), but the benefit is negligible when the struct is small enough to reside in registers naturally
- For global-memory-bandwidth-bound kernels (one load per element in the inner loop), register-pressure optimizations and parameter-passing techniques have limited impact — the bottleneck is DRAM throughput, not register allocation or parameter access latency
