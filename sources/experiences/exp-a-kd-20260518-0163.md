## Challenge

The before code chains 6 vector-operation pairs (main_work_kernel + tail_kernel) with no early launch signal. The after code records a work-done event after the main kernel and before the tail kernel, with the next chain iteration waiting on it. This simulates the launch_depend


## Solution

Insert an early work-done event recorded after the main kernel but before the tail kernel, with the successor chain iteration waiting on the previous chain's work-done event, simulating SM90a launch_dependent_grids() which signals the successor at work-exhaustion rather than full-completion to enable overlap of successor main work with predecessor tail work

## Results

- **speedup**: 1.000x (neutral)


## Key Lessons

- Event-based PDL simulation on pre-SM90a hardware cannot produce real kernel overlap because cudaStreamWaitEvent on the same stream is semantically equivalent to default serialization → real speedup requires SM90a+ with griddepcontrol.launch_dependents hardware support → event-based patterns serve only as correctness validation, not performance improvement, on older architectures
- Programmatic Dependent Launch (PDL) signals successor grids at work-exhaustion (after all useful output is written) rather than full-completion (after tail/epilogue) → allows successor main work to overlap with predecessor epilogue → most beneficial when tail/epilogue work is a significant fraction of per-chain latency in a long dependency chain
