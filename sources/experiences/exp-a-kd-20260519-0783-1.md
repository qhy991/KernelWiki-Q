## Solution

Reorganize from 1-token-per-CTA (256 threads, blockReduceMax + __syncthreads) to 8-tokens-per-CTA (1 warp=32 threads per token, warpReduceMax only). Each thread loops 8x more iterations (stride 32 vs 256), eliminating cross-warp sync overhead but reducing memory-level parallelism from 256 to 32 concurrent load requests per token.

## Key Lessons

- Reducing threads-per-token from 256 to 32 cuts concurrent memory load requests by 8x even though total traffic is identical — fewer in-flight loads means lower memory-level parallelism and worse throughput when the workload is memory-bandwidth-bound
- Eliminating __syncthreads() via warp-only reduction does not compensate for reduced MLP when the GPU already has enough SMs to schedule all blocks without occupancy pressure — the sync savings are negligible compared to the bandwidth utilization loss
- CTA granularity reorganization is hardware-specific: warp-per-token designs may benefit GPUs with higher warp throughput or different memory subsystems (e.g., H100) but regress on architectures like A800/sm_80 where the original block-per-token approach already saturates memory bandwidth efficiently
