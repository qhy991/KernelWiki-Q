## Solution

The after code adds a full ThreadMap coordinate decomposition (row_idx, group_idx, cluster_idx via modulo/division chains) and a per-row conditional fetch guard (kOobCheck) intended for correctness on boundary tiles, which regresses performance on aligned tiles due to extra integer arithmetic and data-dependent branching in the inner loop

## Key Lessons

- OOB guard (kOobCheck) in CUTLASS ThreadMap is a correctness mechanism for boundary tiles only, not a performance optimization — it defaults to false and should only be enabled when tile rows may exceed problem bounds, as the coordinate decomposition and conditional branch add significant overhead on aligned tiles
- ThreadMap coordinate decomposition (modulo, division, multiply per row iteration) introduces substantial integer arithmetic overhead inside tight reduction loops — keep inner loops branch-free when tiles are known to be fully in-bounds
- Data-dependent conditional branches inside reduction inner loops prevent the compiler from fusing load-and-accumulate into a single dependent chain and may inhibit memory coalescing, even when all threads uniformly take the same branch path
