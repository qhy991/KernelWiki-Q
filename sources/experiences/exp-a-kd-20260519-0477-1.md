## Challenge

ELU kernel loads and stores eight half elements via four separate half2 operations with four independent boundary checks, increasing instruction count and preventing the compiler from coalescing memory accesses into a single wide transaction


## Solution

Replace four separate half2 loads/stores with a single LDST128BITS (float4 reinterpret cast) 128-bit wide load/store and use a #pragma unroll loop for the element-wise ELU computation, consolidating four boundary checks into one

## Key Lessons

- Consolidating four half2 loads/stores into a single LDST128BITS (float4) 128-bit wide transaction reduces memory instruction count from 8 to 2, but the benefit may be invisible at moderate input sizes where kernel launch overhead dominates total latency
- Wide load/store optimizations (half2 → float4/LDST128BITS) are most impactful in memory-bound kernels processing large arrays; at small problem sizes the per-thread instruction savings are negligible relative to launch and synchronization overhead
- Four independent boundary checks versus one are well-predicted by the GPU branch predictor when only the tail warp diverges — reducing branch count alone does not guarantee measurable speedup
