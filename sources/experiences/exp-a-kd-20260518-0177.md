## Solution

Replace synchronous LDG+STS with cp.async.ca.shared.global inline PTX for asynchronous global-to-shared copy, followed by commit_group and wait_all — but the immediate fence and wait eliminate any overlap opportunity, resulting in marginal overhead with no throughput gain for this single-stage 256-byte transfer

## Key Lessons

- cp.async requires multi-stage pipeline overlap (stage N async load overlapping with stage N-1 computation) to deliver benefit → issuing commit_group followed immediately by wait_all in a single-stage kernel eliminates all overlap opportunity → apply cp.async only when you can interleave async copies with independent computation work
- Small data transfers (e.g., 256 bytes for 32 doubles) cannot saturate memory bandwidth → async copy mechanisms add instruction overhead (cvta, commit_group, wait_all) without throughput gain → async copy is beneficial only when transfer size is large enough that pipelining hides latency
- cp.async.ca.shared.global with the 8-byte atom mirrors CuTe's SM80_CP_ASYNC_CACHEALWAYS copy atom — it is the correct primitive for double-precision async shared memory loads, but the programming model demands grouped commits and asynchronous waits that are spaced apart from actual computation to realize any speedup
