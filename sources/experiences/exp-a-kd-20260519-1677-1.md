## Challenge

Top-K selection uses CUB DeviceRadixSort to fully sort all N=16384 elements in O(N log N) time with heavy global memory traffic, then discards all but the first K=2048 sorted results, wasting the majority of the sort computation


## Solution

Replace full O(N log N) sort with multi-pass radix selection: a coarse 256-bin FP16 histogram identifies a threshold bin in shared memory, elements clearly above the threshold are written directly to output, elements below are discarded, and only the small ambiguous threshold-bin subset is refined across up to 4 byte-level passes using FP32 key bits, achieving O(N) expected work proportional to the candidate set rather than the full sequence

## Key Lessons

- Replace full sort with radix selection for top-K — a coarse histogram identifies the threshold bin, and only the ambiguous candidates within that bin need refinement, reducing work from O(N log N) to O(N) expected; applies when K << N and a histogram-able key representation exists
- Use FP16-precision coarse histogram (8-bit bin) followed by FP32 byte-level refinement passes for floating-point top-K — the FP16 coarse pass filters most elements in one pass, and up to 4 FP32 byte passes resolve ties within the threshold bin at full precision
- Early termination when remaining_k reaches zero avoids unnecessary refinement passes — after each histogram pass, check if all required elements have been output and skip remaining passes
