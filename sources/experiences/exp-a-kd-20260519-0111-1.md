## Challenge

Multiple warps (8 warps for 32 rows) concurrently atomicAdd partial sums into the same shared s_prime entries, causing quadratic contention overhead that grows with warp count as competing threads serialize on the same addresses


## Solution

Replace atomicAdd with warp-partitioned addition_storage where each warp writes its partial sum to a dedicated slot [row + num_rows * warp_id], then a subset of threads performs an explicit unrolled loop to sum across all warp contributions, eliminating atomic contention entirely

## Key Lessons

- Replace multi-warp atomicAdd reductions with per-warp storage slots plus an explicit unrolled reduction loop — atomic contention grows quadratically with warp count while shared-memory loads scale linearly, making the deterministic approach faster beyond a small warp count threshold
- Warp-partitioned addition_storage indexed as [row + num_rows * warp_id] provides conflict-free writes since each warp owns its own slot, then a second phase with lane_id < kLinesPerWarp distributes the cross-warp summation across threads for bank-conflict-free reads
