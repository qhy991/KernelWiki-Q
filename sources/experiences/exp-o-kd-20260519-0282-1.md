## Challenge

Split-K reduction kernel sizes its vector-access width (kCount=8) based on the output element type (half, 16-bit) rather than the workspace element type (float, 32-bit), causing each thread to accumulate 8 floats with only a single grid block covering all 256 columns, starving the GPU of concurrency and wasting registers


## Solution

Resize kCount from 8 to 4 (128 bits / sizeof(float)) to match the workspace element type, which doubles the number of thread blocks in the Y dimension for higher occupancy, halves register pressure per thread (4 vs 8 accumulator floats), and aligns each thread's access stride to a natural 128-bit vector load for coalesced float transactions

## Key Lessons

- Size vector-access width (kCount) based on the workspace/intermediate element type, not the output type — when workspace stores 32-bit floats, kCount = 128/32 = 4 yields correct vectorized access; using the output type's 16-bit size inflates kCount to 8 and misaligns memory transactions
- Smaller per-thread work tiles that increase grid block count can improve overall throughput by raising occupancy, even when total work per thread decreases — the trade-off favors more blocks when register pressure drops and memory access becomes naturally coalesced
