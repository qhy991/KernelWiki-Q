## Challenge

RMS normalization kernel assumes contiguous input layout where each token occupies exactly hidden_size contiguous positions; when input rows are wider (e.g., packed QKV projection output where only a subset needs normalization), a separate copy kernel must first extract the relevant elements into a contiguous buffer, adding an extra kernel launch and memory round-trip


## Solution

Introduce an input_stride kernel parameter extracted from input.stride(-2) at the host level, replacing the hardcoded blockIdx.x * hidden_size indexing with blockIdx.x * input_stride when reading the input tensor, while keeping output indexing contiguous since the output is always laid out densely

## Key Lessons

- Replace hardcoded dimension-based strides with a configurable stride parameter extracted from the tensor's actual stride metadata — eliminates intermediate copy kernels when operating on non-contiguous views of packed tensors (e.g., QKV outputs) and has zero overhead when stride equals the dimension
- Decouple input stride from output stride: input may be non-contiguous (wider packed rows) while output remains contiguous, so input indexing uses input_stride but output indexing stays blockIdx.x * hidden_size
