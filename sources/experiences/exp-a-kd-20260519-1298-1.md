## Challenge

Two-step gather-then-dequantize pipeline for FP8 KV-cache uses an intermediate buffer: gather_fp8_cache writes 656 bytes per token to intermediate memory with 1024 threads (most idle), then dequant_fp8_to_bf16 reads it back in a second kernel launch, doubling memory traffic and adding kernel launch overhead


## Solution

Fuse gather and dequantize into a single kernel (cp_gather_and_upconvert_fp8_kv_cache) that reads FP8 data directly from paged cache blocks and writes BF16 output in one pass, eliminating the intermediate buffer allocation and the second kernel launch while using 576 threads matched to per-token work dimensions

## Key Lessons

- Fuse gather and compute kernels that share the same data flow (read source → transform → write destination) into a single kernel to eliminate intermediate buffer allocation and redundant memory traffic — applies whenever an intermediate buffer is only read by the immediately following kernel
- Match thread count to actual per-element work dimensions (576 for 512+64 head dim) rather than using a generic power-of-2 (1024) to avoid idle threads and improve occupancy
- Fusing a paged-cache gather with its consumer kernel avoids writing 656 bytes/token to intermediate memory and immediately reading them back, effectively halving global memory traffic for the gather/dequant pair
