## Solution

Swap problem dimensions from (M, N) to (N, M) so blockIdx.x maps to original columns and threadIdx.x maps to original rows, making warp-wide accesses contiguous at stride 1 in column-major layout — each warp coalesces into 1-2 cache line requests instead of 32

## Key Lessons

- Swap grid/thread dimension interpretation (blockIdx.x ↔ threadIdx.x mapping) to align threadIdx.x with the contiguous dimension of column-major storage — warp threads then access consecutive addresses and coalesce into minimal cache line requests
- In column-major layout with LDC=M, accessing data[row + col*LDC] where threadIdx.x varies col causes stride-LDC access; swapping to col + row*LDC where threadIdx.x varies col makes stride-1 (coalesced) — the fix is purely a dimension reassignment with no algorithmic change
- CUTLASS split-K reduction epilogue must pass GemmCoord(N, M, K).mn() (swapped dimensions) rather than GemmCoord(M, N, K).mn() to the reduction kernel when workspace is in column-major order — a common pitfall in asymmetric M×N GEMM configurations
