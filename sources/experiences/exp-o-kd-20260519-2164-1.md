## Challenge

Fixed CUTLASS 2.x tile configuration (TileShape<128,128,64> with 5 stages) is used uniformly for all GEMM problem shapes regardless of M and N dimensions, which is suboptimal for medium M=128 with N=4096 where a wider K-dimension tile with fewer stages would better utilize shared memory


## Solution

Introduce shape-aware tile configuration via select_tile_config() that maps M/N power-of-2 buckets to optimal TileShape and stage counts — for M∈(64,128] and N≤8192 it selects TileShape<64,128,128> with 3 stages, using a wider K-dimension tile to increase compute density while reducing shared memory staging overhead

## Key Lessons

- Shape-aware tile configuration (mapping M/N power-of-2 buckets to optimal TileShape and stage counts) can outperform fixed tiling — wider K-dimension tiles with fewer stages better utilize shared memory for medium-M problems where the M dimension does not fully occupy a large M-tile
- cuBLAS with CUBLAS_GEMM_DEFAULT_TENSOR_OP applies its own internal tiling heuristics independently of application-level tile config parameters — measured speedups from tile selection logic not actually passed to the GEMM call may reflect run-to-run variation rather than genuine optimization and should be validated with a CUTLASS API that directly consumes tile parameters
