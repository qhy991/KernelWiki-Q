## Challenge

Fused add RMSNorm kernel uses a runtime conditional (if num_tokens < 256) to select between two instantiations of a hand-rolled blockReduceSum template, adding code complexity without improving performance since all threads in a block take the same branch and the reduction is not the bottleneck


## Solution

Replace the runtime-conditional hand-rolled blockReduceSum with a single cub::BlockReduce<float, 1024> call passing blockDim.x as valid-count, which internally handles arbitrary thread counts and eliminates the branching code while using equivalent warp-shuffle primitives

## Key Lessons

- Replacing hand-rolled warp-shuffle reductions with cub::BlockReduce simplifies code and eliminates runtime conditionals without regressing performance when the reduction phase is a negligible fraction of total kernel time in a memory-bound workload
- Runtime conditionals on uniform values (same for all threads in a block) do not cause warp divergence, so replacing them with a library call is a code-clarity improvement rather than a performance optimization
- For memory-bound kernels like RMSNorm with hidden_size=4096, the reduction over a few hundred elements is too small a fraction of execution time for reduction implementation details to affect overall throughput
