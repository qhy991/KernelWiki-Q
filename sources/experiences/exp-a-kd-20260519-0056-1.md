## Solution

Add a float-specific template specialization setting kFragmentsPerIteration=2, which processes two accumulator fragments per iteration to halve the number of __syncthreads() barriers; however, this doubles shared memory from 34KB to 68KB per block, reducing occupancy from 4 to 2 blocks per SM on A800 (164KB shared memory), and the occupancy loss dominates in this microbenchmark because the trivial float copy does too little work per barrier for the sync reduction to compensate

## Key Lessons

- Doubling fragments-per-iteration to reduce __syncthreads() count is only beneficial when the work done per barrier is substantial enough to amortize the sync cost — in a trivial float-copy epilogue the barrier overhead is minimal and the resulting occupancy loss from doubled shared memory dominates performance
- Shared memory occupancy cliffs must be checked before increasing per-block smem: on A800 (164KB/SM), going from 34KB to 68KB halves blocks-per-SM from 4 to 2, a 50% occupancy reduction that no amount of barrier elimination can overcome in lightweight kernels
- Microbenchmark results for epilogue optimizations may not reflect real pipeline behavior — in the full CUTLASS GEMM pipeline the warp-level fragment iterator performs complex strided loads, bank-conflict-aware layouts, and type conversion, making barrier cost proportionally higher and kFPI>1 genuinely beneficial
