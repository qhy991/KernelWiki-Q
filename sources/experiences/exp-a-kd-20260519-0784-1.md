## Solution

Replace 1-token-per-CTA blockReduceMax (256 threads, __syncthreads) with 8-tokens-per-CTA warpReduceMax (32 threads per token) to eliminate synchronization and reduce total CTA count; however this regresses on A800 (sm_80) because the stride reduction from 256 to 32 drops concurrent memory requests per token from 256 to 32, forcing each thread into 8 sequential loads that underutilize the memory subsystem

## Key Lessons

- Warp-per-token restructuring (blockReduceMax→warpReduceMax) eliminates __syncthreads but reduces memory-level parallelism by N× where N=blockSize/warpSize — regresses on GPUs where memory bandwidth (not synchronization) is the bottleneck, such as A800 (sm_80)
- Optimization strategies are hardware-specific: the warp-per-token pattern was tuned for H100 (sm_90, 132 SMs) where reduced CTA count improves scheduler occupancy and the memory subsystem handles stride-32 loops efficiently, but the same pattern regresses 24% on A800 due to fewer concurrent memory requests
- When reducing thread stride per token (e.g., 256→32), each thread must iterate proportionally more times (1→8 iterations), serializing memory accesses that were previously concurrent — this is the key trade-off between synchronization elimination and memory parallelism
