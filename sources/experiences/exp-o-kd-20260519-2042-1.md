## Solution

Cap block size to 256 threads instead of 1024, allowing 4x more blocks to reside concurrently on each SM; the smaller block size dramatically improves memory latency hiding for the bandwidth-bound RMS norm kernel by increasing SM occupancy

## Key Lessons

- Reducing threads per block from 1024 to 256 for bandwidth-bound kernels allows 4x more concurrent blocks per SM, trading per-block parallelism for higher occupancy and better memory latency hiding — applies whenever register/shared memory pressure prevents multiple large blocks from co-residing
- Use a cap of 256 threads per block as a starting point for memory-bandwidth-bound kernels; larger block sizes only help when the kernel has significant compute or needs many threads for intra-block reductions
