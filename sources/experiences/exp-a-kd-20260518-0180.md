## Challenge

The before kernel passes the LargeParams struct (~280 bytes) by value, which NVCC on CUDA 12.8 with -O3 already places into the read-only data cache via its own parameter-passing ABI for kernel launches. The after kernel adds __grid_constant__ to explicitly request this caching. On CUDA 1


## Solution

Annotate the kernel parameter with __grid_constant__ and const to explicitly request that the large struct be cached in the read-only data cache rather than copied to each thread's local memory, ensuring uniform access across all threads in a block

## Results

- **speedup**: 1.001x (neutral)


## Key Lessons

- On CUDA 12.8+ with -O3, NVCC already places const-qualified kernel parameters into the read-only data cache via its parameter-passing ABI, making __grid_constant__ a no-op for cases the compiler already handles → explicit annotation is redundant when the compiler can prove read-only semantics → only beneficial on older toolchains or when compiler cannot infer const-ness
- When a kernel is memory-bandwidth-bound on global memory loads, optimizing parameter access latency yields negligible improvement → the dominant cost of global memory transactions hides any parameter caching benefit → profile before applying parameter-path optimizations to ensure the bottleneck is actually on parameter access
