## Solution

Replace per-thread sharded counting plus prefix-sum with stride-based atomicAdd on a flat shared memory array organized by warp, eliminating the inter-thread prefix-sum phase and one __syncthreads() barrier while scaling from 64 to 1024 threads for better parallelism

## Key Lessons

- Replacing per-thread sharded counting with stride-based atomicAdd eliminates the inter-thread prefix-sum phase and one __syncthreads() barrier, yielding major latency reduction when the expert count fits comfortably in shared memory — applies to histogram/counting kernels where moderate atomic contention is acceptable
- Scaling thread count from the minimum (64, matching num_experts) to the hardware maximum (1024) with stride-based iteration amortizes kernel launch overhead and improves occupancy for counting kernels where work is distributed across many input elements
- Reducing shared memory from (num_threads+1)*num_experts to padded_num_experts entries frees SM resources for higher occupancy — relevant when the counting data structure can be collapsed from a per-thread matrix to a flat per-expert array using atomics
