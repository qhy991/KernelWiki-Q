## Challenge

Fused RMS norm + dynamic per-token quantization kernels assume contiguous input layout, using blockIdx.x * hidden_size as the token offset. This breaks when torch.compile produces non-contiguous tensors after fusion or layout transformations, forcing a separate strided-to-contiguous copy before the kernel can run correctly.


## Solution

Add an input_stride parameter to the kernel function and propagate it through all internal compute helpers (compute_rms, compute_dynamic_per_token_scales, norm_and_quant), replacing the hardcoded hidden_size offset with blockIdx.x * input_stride for correct token addressing in both contiguous and non-contiguous layouts.

## Key Lessons

- Add a stride parameter alongside the logical dimension size in fused kernels — when input_stride equals hidden_size the contiguous path is identical, but non-contiguous inputs from torch.compile fusion passes are handled without an extra copy → applies to any kernel that indexes tokens via blockIdx.x * hidden_size
- Eliminating an intermediate copy kernel for non-contiguous-to-contiguous conversion yields up to 1.56x end-to-end speedup because the copy overhead plus the original kernel launch is replaced by a single stride-aware kernel launch
