## Challenge

GELU activation kernel processes only 2 half values per thread (one half2 register), limiting instruction-level parallelism and underutilizing each thread's compute capacity on GPU hardware


## Solution

Increase per-thread work from 2 to 8 half values by unpacking into 4 separate half2 registers (4x ILP), reducing total thread count and potentially hiding instruction latency through more independent operations per thread

## Key Lessons

- Increasing per-thread ILP (2→8 elements) does not always improve performance for memory-bandwidth-bound kernels at small problem sizes where both variants already saturate available bandwidth — the added instruction overhead and branch cost from managing multiple register groups can outweigh the reduced thread-count scheduling benefit
- Unpacking scalar half operations across multiple half2 registers (.x/.y per register) negates the benefit of vectorized half2 arithmetic — a true vectorized approach using half2-native intrinsics would be more effective than replicated scalar ops on unpacked fields
- Boundary-check branching scales with per-thread element count: 1 branch for 2 elements vs 4 branches for 8 elements — at small N this branch overhead is proportionally significant
