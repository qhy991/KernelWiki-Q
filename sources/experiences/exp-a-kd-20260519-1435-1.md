## Challenge

The sortAndScanExpert function accepts a non-const int* for expert IDs, forcing callers to clone the entire topk_ids array via a 128 KB device-to-device cudaMemcpy on every invocation even though CUB radix sort only reads the keys input and never mutates it


## Solution

Add const-correctness to sortAndScanExpert and CubKeyValueSorter::run by changing the read-only expert_for_source_row parameter from int* to const int*, enabling callers to pass topk_ids directly without cloning and eliminating the per-iteration 128 KB device-to-device memcpy

## Key Lessons

- Add const qualifiers to read-only pointer parameters in GPU APIs to allow callers to pass immutable buffers directly — applies whenever a downstream library (e.g., CUB radix sort) only reads the input but the intermediate API fails to express this via const
- Even modest D2D memcpy sizes (128 KB) can cost ~8-9% latency when competing for the same memory bus as the core compute operation — always verify whether input mutation is actually required before defensively cloning device buffers
