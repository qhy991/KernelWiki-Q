## Challenge

Grouped FP8 GEMM dispatch selects CUTLASS Pingpong schedule (MmaConfig0: tile 64x128x128, cluster 2x1x1) based solely on K dimension > 128, which underutilizes the 132 SMs on H100/H200/H800 GPUs — Pingpong pairs 2 SMs per tile with smaller tile shapes, leaving SMs idle on high-SM-count hardware


## Solution

Add a hardware-aware guard checking multiProcessorCount == 78 to identify H20 GPUs, restricting Pingpong schedule to H20 with K > 128 while using Cooperative schedule (MmaConfig1: tile 128x128x128, cluster 1x2x1) on all other SM90 GPUs (H100/H200/H800) to leverage larger tiles and better SM utilization across 132 available SMs

## Key Lessons

- Hardware-aware schedule dispatch using multiProcessorCount enables optimal CUTLASS GEMM schedule selection per GPU variant — Cooperative schedule (larger tiles, cluster 1x2x1) better utilizes high-SM-count GPUs (132 SMs on H100/H200/H800) while Pingpong schedule (smaller tiles, cluster 2x1x1) remains better for resource-constrained GPUs (78 SMs on H20)
- Pure problem-dimension thresholds (e.g., K > 128) are insufficient for CUTLASS schedule selection on heterogeneous GPU fleets — the SM count and shared memory per SM fundamentally affect which tile shape and cluster configuration delivers best throughput
- Grouped MoE GEMM workloads benefit disproportionately from Cooperative schedule on high-SM-count GPUs because the total M dimension across experts provides enough parallelism to fill all SMs with larger tiles, whereas Pingpong's 2-SM-per-tile pairing wastes SM capacity
