## Challenge

Scalar swish kernel processes one float per thread, issuing separate 32-bit load and store instructions per element, which underutilizes memory bus bandwidth for a memory-bound elementwise operation at N=1M


## Solution

Replace scalar per-thread load/store with float4 vectorized access, processing 4 floats per thread via a single 128-bit coalesced transaction, reducing thread count by 4x and better saturating the memory bus

## Key Lessons

- Vectorized float4 load/store in elementwise kernels replaces 4 separate 32-bit transactions with one 128-bit coalesced transaction, improving memory throughput — applies to any memory-bound elementwise operation over contiguous fp32 arrays
- Scaling thread index by the vector width (* 4) and adjusting the grid size accordingly reduces total thread count, lowering scheduling overhead while keeping work distribution even
