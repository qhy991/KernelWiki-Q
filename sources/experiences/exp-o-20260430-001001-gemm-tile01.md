Different GEMM shapes require different CTA tile and warp tile configurations for optimal performance

## Challenge

GEMM performance is highly sensitive to tile size selection. Using a fixed tile (e.g. 128x128x32) for all shapes leads to poor occupancy for small M, wasted shared memory for large tiles, and underutilization of SMs. The optimal tile depends on M, N, K dimensions and SM architecture shared memory budget.

## Solution

M-bucket tile selection strategy with architecture-specific configuration

Classify dynamic M dimension into buckets (decode_s: 1-1, decode_b: 2-8, small: 9-32, medium_s: 33-128, medium_l: 129-512, large: 513-2048, huge: 2049-8192, ultra: 8193-32768) and assign optimal tile per bucket. Small M (<=16) uses minimal tile 16x64x64 with 7 stages to maximize pipeline depth. Large M uses 128x128x32 with 3-4 stages for balanced compute/memory.

## Results

- **small_M_vs_cublas**: 2-3x speedup (M<=16)
- **large_M_vs_cublas**: competitive with cuBLAS (110+ TFLOPS on L20 SM89)

## Key Lessons

- Small-M GEMM (M<=16, decode/small batch) needs specialized minimal tiles (16x64x64) with max pipeline stages (7) to achieve 2-3x over cuBLAS
- Large-M GEMM should use 128x128x32 CTA tile with 3-4 stages for balanced shared memory usage and compute throughput
- Warp tile should be roughly half CTA tile in each dimension (e.g. CTA 128x128 -> Warp 64x64) for 4-warps-per-CTA configuration
- For 8-warps-per-CTA (SM87 with few SMs), use CTA 128x256x64 with Warp 64x64x64 to reduce CTA count and improve SM utilization
- Stage count is inversely correlated with tile size: smaller tiles can afford more stages within shared memory budget
- Tile selection must respect shared memory budget: estimate = (TileM*TileK + TileN*TileK) * Stages * ElementSize
