## Challenge

The compute_expert_offsets_w4a8 kernel was launched <<<1,1>>> with a single thread sequentially iterating over all num_experts to compute cumulative sums of per-expert row counts, creating a fully serial bottleneck especially for 256+ experts in MoE models like DeepSeek-V3


## Solution

Replace the single-thread sequential loop with a parallel CUB BlockScan InclusiveSum using a multi-chunk approach: each chunk processes BLOCK_SIZE elements cooperatively within a block, propagates the running prefix sum across chunks via shared memory broadcast, with an adaptive block size (32–1024) selected at host level based on num_experts

## Key Lessons

- Replace <<<1,1>>> single-thread kernels with parallel block-level primitives (CUB BlockScan) when computing prefix sums — the serial loop becomes a cooperative block operation, yielding near-2x speedup for 256+ elements
- Use a multi-chunk approach with shared memory broadcast to extend CUB BlockScan beyond a single block's worth of elements — the last thread of each chunk writes the partial sum to shared memory, all threads synchronize, then the next chunk adds this base prefix
- Select adaptive block sizes (32/64/128/256/512/1024) at host level based on problem size to match occupancy without over-allocating shared memory for small inputs
