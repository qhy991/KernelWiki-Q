## Challenge

LSE values are stored in transposed [num_heads, num_tokens] layout, causing strided memory access with 2048-byte stride between consecutive heads for the same token, which prevents coalesced reads and wastes DRAM bandwidth on partially-used cache lines


## Solution

Reorganize LSE storage from transposed [num_heads, num_tokens] to row-major [num_tokens, num_heads] layout, so threads in a warp accessing consecutive head_idx values for the same token_idx read from contiguous 256-byte regions (2 cache lines), enabling near-coalesced access and reducing L2 cache pressure

## Key Lessons

- Reorganize small per-element metadata arrays (LSE, bias) from transposed to row-major layout when threads are parallelized over the inner dimension — warp threads reading consecutive inner indices get coalesced access when the inner dimension is the stride-1 axis
- For num_heads=64 with float LSE, row-major packs all heads per token into 256 bytes (2 cache lines), while transposed scatters them across 64 rows with 2048-byte stride — the coalescing improvement applies whenever the inner parallel dimension fits within a few cache lines
