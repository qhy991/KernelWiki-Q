## Solution

Launch two threadblocks instead of one: block 1 handles the sorted_token_ids global-memory initialization while block 0 immediately proceeds to expert counting via shared-memory atomicAdds, overlapping the two independent phases without any inter-block synchronization since block 0 never reads sorted_token_ids

## Key Lessons

- Overlap independent global-memory initialization with shared-memory computation by assigning them to separate threadblocks — this hides init latency behind compute when the compute block does not depend on the initialized buffer
- Use threadblock-level task parallelism (blockIdx dispatch) to decouple independent work phases that share no data dependencies, transforming serial init+compute into concurrent execution across multiple SMs
- A simple blockIdx.x guard with early return is sufficient for inter-block task dispatch when blocks are truly independent — no inter-block synchronization barrier is needed when data flows are non-overlapping
