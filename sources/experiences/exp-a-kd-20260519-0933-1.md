## Solution

Move all arithmetic to float32 domain: cast both gate and up inputs to float32 explicitly in the kernel loop, compute SiLU and multiply entirely in float32, and cast back to half only at the final store — improving numerical precision at neutral performance cost since the kernel is memory-bound with 128-bit vector loads dominating runtime

## Key Lessons

- For memory-bound kernels with 128-bit vectorized loads/stores, moving arithmetic from half to float32 domain has negligible performance impact because the bottleneck is memory bandwidth, not compute throughput
- Computing fused activation-and-gate multiply entirely in float32 improves numerical accuracy at no measurable performance cost when the kernel is memory-bound, making it strictly preferable over half-domain intermediate computation
- Avoid half-typed activation helpers that perform hidden round-trip casts (half→float→compute→float→half) — instead, keep the activation function in float32 and manage casts explicitly at the kernel loop level for clearer precision semantics
