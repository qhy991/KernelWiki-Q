## Solution

Replace scalar half loads/stores with half2 vectorized 32-bit memory transactions via reinterpret_cast<half2>, halving thread count per row from 256 to 128 so each thread processes two elements — this reduces memory transaction count, warp scheduling overhead, and barrier synchronization cost while also cutting the number of partial values entering the block reduction from 256 to 128

## Key Lessons

- Use half2 vectorized loads/stores via reinterpret_cast<half2&> to coalesce two 16-bit scalar memory transactions into one 32-bit transaction — applies whenever adjacent fp16 elements share the same compute path
- Vectorizing N elements per thread reduces thread count by N×, which cascades into fewer warps, lighter barrier synchronization in block reductions, and fewer partial values entering the reduction chain — compound speedup beyond memory bandwidth alone
