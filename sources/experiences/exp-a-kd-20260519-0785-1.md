## Challenge

Cutlass int8 GEMM uses small TileShape<32,64,128> and WarpShape<16,64,64>, launching many threadblocks that each compute only 2048 output elements (32x64), resulting in high scheduling overhead and poor per-warp tensor core utilization for M=256 N=8192 K=4096


## Solution

Enlarge TileShape to <128,128,64> and WarpShape to <64,64,64>: each threadblock computes 8x more output elements (16384 vs 2048) reducing block count and scheduling overhead, each warp gets 4x more MMA work along M for better tensor core utilization, and halving the K tile dimension from 128 to 64 cuts shared memory per block to improve SM occupancy

## Key Lessons

- Larger TileShape (128x128 vs 32x64) amortizes threadblock launch overhead and improves L2 cache reuse by computing more output elements per block — applies when problem dimensions are large enough to saturate the larger tile
- Increasing WarpShape M dimension (64 vs 16) gives each warp 4x more MMA instructions per iteration, improving tensor core utilization without increasing synchronization cost — effective when register pressure permits
- Reducing TileShape K dimension (64 vs 128) cuts shared memory per block roughly in half, allowing more concurrent threadblocks per SM and higher occupancy — trade more K-loop iterations for better SM-level parallelism
