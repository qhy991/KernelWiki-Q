## Challenge

Batch-split grid with 1024 threads per CTA for paged KV-cache gather suffers severe thread underutilization (576 bf16 elements per entry / 1024 threads < 1 iteration each), complex split boundary and partial-block conditionals causing divergent branching, and low SM occupancy from oversized CTAs in a memory-bound workload


## Solution

Replace batch-split 2D grid (batch × num_splits, 1024 threads/CTA) with per-token 1D grid (one CTA per token, 64 threads), giving each thread 9 meaningful copy iterations (576/64), eliminating split boundary and partial-block conditionals, and enabling more concurrent CTAs per SM for improved occupancy on the memory-bound workload

## Key Lessons

- Match CTA size to the work per work-unit: choosing 64 threads for 576-element entries yields 9 iterations per thread (576/64), while 1024 threads yields <1 iteration — the right CTA size transforms thread underutilization into full utilization
- Per-token grid mapping with a token_to_seq lookup table eliminates complex split boundary arithmetic and partial-block conditionals, reducing divergent branching and address computation overhead for paged cache gather patterns
- Smaller CTAs (64 vs 1024 threads) enable more concurrent CTAs per SM, improving occupancy and hiding memory latency for memory-bound workloads like paged KV-cache gather
