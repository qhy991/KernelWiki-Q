## Solution

Replace the single M16 config with two fine-grained configs (M4, M64) using swap_ab=true to transpose the GEMM from (M,K)x(K,N) to (N,K)x(K,M), so the large N dimension fills the 128-wide tile efficiently while the small M matches the 16-tall tile, dramatically reducing padding waste for MoE batch-1/4 inference

## Key Lessons

- For small-M grouped GEMM (MoE inference), swap A/B operands so the large N dimension becomes the tiled M dimension — this transposes (M,K)x(K,N) to (N,K)x(K,M), enabling efficient tile utilization when M << N
- Split tile shape configs by M range (e.g., M<=4 vs M<=64) with progressively larger K tiles and cluster widths — finer-grained M ranges allow tile shapes that minimize padding waste per range
- When swap_ab=true, increase the first tile dimension to match the now-large effective M (originally N) and shrink the second tile dimension to match the now-small effective N (originally M) — tile shape <128,16,K> replaces <64,64,K> for the transposed problem
