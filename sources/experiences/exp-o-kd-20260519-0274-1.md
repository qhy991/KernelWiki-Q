## Challenge

In TiledCopy construction for TMA copies, when CTA_Tile has a hierarchical multi-mode shape (e.g., from GETT-style tiles) whose rank structure does not match SLayout's hierarchy, passing cta_tile directly causes the internal tiling logic to misinterpret mode hierarchy, producing inconsistent CTA-to-SMem access patterns or compilation failures


## Solution

Add a compile-time compatibility guard using compatible(shape(CTA_Tile{}), shape(SLayout{})): when shapes match, use cta_tile as-is; when they differ, apply product_each(cta_tile) to flatten each hierarchical mode into its product, producing a shape whose rank aligns with the already-flattened SLayout so TiledCopy partitioning remains correct

## Key Lessons

- When constructing TiledCopy objects with hierarchical CTA_Tile shapes (e.g., GETT multi-mode tiles), validate that shape(CTA_Tile) is compatible with shape(SLayout) at the mode-structure level — a size-level check (product_each equality) is insufficient because mode hierarchy mismatch still produces incorrect tiling
- product_each() flattens each mode of a hierarchical shape into its scalar product, converting a multi-mode tile into a flat shape that is compatible with SLayout's hierarchy — use it as a compatibility normalization step when CTA_Tile and SLayout have different rank structures
