## Challenge

cutlass::bfloat16_t overloaded operator* wraps scalar __hmul intrinsics element-by-element, preventing nvcc from vectorizing adjacent multiplies into __hmul2 (2-element BF16 multiply), leaving arithmetic throughput underutilized in the BF16 scale-and-add dequantization loop


## Solution

Replace cutlass::bfloat16_t scalar operator* with native nv_bfloat16 type and explicit __hmul2/__hadd2 vectorized intrinsics, packing element pairs into __nv_bfloat162 for dual-element BF16 processing that doubles theoretical arithmetic throughput

## Key Lessons

- Vectorized BF16 intrinsics (__hmul2/__hadd2) double arithmetic throughput but only help when the kernel is arithmetic-bound; in memory-bound kernels where global memory load/store dominates, the vectorization gain is negligible and pack/unpack overhead may offset it entirely
- cutlass::bfloat16_t overloaded operator* wraps scalar __hmul, preventing nvcc auto-vectorization — recasting to native nv_bfloat16 type allows the compiler to emit __hmul2 vectorized instructions
- BF16 vectorized multiply optimization is most impactful in register-resident loops feeding WGMMA tensor cores (e.g., Machete's smem-to-rmem transfer in the WGMMA data path), not in global-memory-bound microbenchmarks
