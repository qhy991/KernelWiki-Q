## Challenge

Split-K GEMM reduction uses atomicAdd across 4 concurrent thread blocks writing to the same 16384-element global output buffer, creating potential atomic contention at overlapping memory addresses


## Solution

Replace single-kernel atomicAdd with two kernels: per-shard plain writes to non-overlapping slots (eliminating atomics), then shared-memory float4 vectorized accumulation across all shards — designed for AMD global-to-LDS hardware acceleration but regresses on NVIDIA where atomicAdd resolves at L2 cache with lower overhead

## Key Lessons

- Replacing atomicAdd with multi-kernel shared-memory staging regresses on NVIDIA GPUs where global atomicAdd resolves at L2 cache with low contention — the two-kernel launch overhead and __syncthreads barriers exceed the atomic contention cost
- Platform-specific hardware capabilities (AMD __builtin_amdgcn_global_load_lds for hardware-accelerated global-to-shared loads) can make an optimization beneficial on one architecture but detrimental on another — always validate on the target platform
- For small-to-medium reduction sizes (16K elements), a single-kernel atomicAdd approach can outperform multi-kernel approaches due to kernel launch overhead dominating total execution time
