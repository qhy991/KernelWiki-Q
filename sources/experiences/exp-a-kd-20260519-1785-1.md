## Challenge

Tiled GEMM with M<=64 uses a 32x32x64 tile configuration (TN=32, TK=64), producing only 32 thread blocks along N for N=1024 and requiring 64 k_tile iterations for K=4096, underutilizing the GPU and suffering high __syncthreads() and shared-memory load overhead per unit of useful FMA work


## Solution

Retune tile configuration from 32x32x64 to 32x8x128: narrowing TN from 32 to 8 quadruples thread block count along N (32→128) for better GPU occupancy, while doubling TK from 64 to 128 halves the k_tile iteration count (64→32) and amortizes each __syncthreads() and shared-memory load phase across twice as many FMA operations, increasing arithmetic intensity

## Key Lessons

- For small-M GEMMs, use a narrower N-tile (smaller TN) to increase thread block count along the N dimension, improving GPU occupancy — applies whenever M alone cannot generate enough blocks to fill the GPU
- Deepening the K-tile (larger TK) halves the number of k_tile iterations and amortizes per-iteration overhead (__syncthreads + shared-memory loads) across more FMA operations, increasing arithmetic intensity — beneficial when shared memory and register budgets permit
- Tile configuration tuning (TM×TN×TK) is a zero-code-change optimization for parameterized tiled GEMM templates: total FLOPs are unchanged but the distribution across blocks and k-iterations controls occupancy, sync overhead, and arithmetic intensity
