## Challenge

Single-block (1024 threads) MoE token sort kernel forces all 1024 threads to share a 256-entry shared-memory local_offsets array via atomicAdd, creating severe contention and limiting sort throughput to a single SM


## Solution

Split the monolithic single-block kernel into an align kernel (counting/cumsum, still single-block) and a separate multi-block sort kernel (128 blocks x 256 threads) that uses global-memory atomicAdd on cumsum_buffer via grid-stride loops, distributing the sort phase across all SMs and leveraging L2 cache for the small 256-entry buffer

## Key Lessons

- Extracting the data-parallel sort phase from a single-block monolithic kernel into a separate multi-block kernel lets it scale across all SMs via grid-stride loops — applies whenever a kernel has a sequential reduction phase followed by an independent parallel scatter phase
- Global-memory atomicAdd on a small hot buffer (256 int32 entries) stays L2-cache-resident and avoids shared-memory bank conflicts and inter-warp contention — prefer over shared-memory atomicAdd when the target set is small and read-heavy after the initial atomic write
- Kernel decomposition that separates counting/prefix-sum (requires single-block for correctness) from scatter/sort (embarrassingly parallel) lets each phase use its optimal launch configuration rather than forcing both into a one-size-fits-all grid
