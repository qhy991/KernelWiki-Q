## Challenge

The silu_and_mul_kernel hardcodes a direct call to the silu() __device__ function, making the kernel non-reusable for other activation functions (gelu, relu, etc.) without duplicating the entire kernel body


## Solution

Replace the hardcoded silu() call with a non-type template function-pointer parameter ACT_FN, generalizing the kernel into act_and_mul_kernel that accepts any activation function at compile time while producing identical SASS code through NVCC's compile-time resolution and inlining

## Key Lessons

- Passing activation functions as non-type template function-pointer parameters (e.g., scalar_t (*ACT_FN)(const scalar_t&)) enables kernel generality with zero runtime overhead — NVCC resolves and inlines the function pointer at compile time, producing identical SASS to a direct call
- When comparing memory-bound kernels where the only difference is a compile-time-indirectable function call, expect neutral performance; the bottleneck is global memory bandwidth, not instruction selection within the arithmetic pipeline
