## Solution

Keep the scale value on the GPU and pass its device pointer directly to the second kernel, which reads it via *d_scale_ptr — both kernels are enqueued back-to-back with zero host blocking, eliminating the cudaMemcpy D2H synchronization point entirely

## Key Lessons

- Keep scalar parameters (scales, biases, thresholds) on the GPU and pass device pointers to downstream kernels instead of copying them to host via cudaMemcpy D2H — eliminates an implicit stream-drain sync that blocks the host until the preceding kernel completes, which is especially devastating when kernel execution time is short relative to the fixed D2H sync overhead (~2.4ms)
- Reading a device-side scalar via pointer dereference (*d_ptr) inside a kernel is effectively free (single global load cached in L1/L2) and enables back-to-back kernel enqueue on the same stream without any host involvement between stages
- cudaMemcpy D2H is an implicit synchronization barrier even for small transfers — treat any D2H copy between kernel pipeline stages as a performance hazard and restructure the pipeline to keep all intermediate data on-device
