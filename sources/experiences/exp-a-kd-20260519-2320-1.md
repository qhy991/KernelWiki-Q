## Solution

Switch k_scale/v_scale from scalar float pass-by-value to const float* pointers with dereference inside the kernel — both approaches are effectively free (parameter-buffer read vs one cached global load) compared to the 1024-iteration inner loop, so the measured 2.4x speedup is a microbenchmark noise artifact

## Key Lessons

- Scalar pass-by-value vs pointer dereference for small constants (scales, biases) has negligible throughput impact — both resolve to a single register read or one cached global load that is hidden behind the kernel's compute loop, so this is not a meaningful optimization lever
- Single-token microbenchmarks (1 thread block) are dominated by kernel launch overhead and event-timing granularity, producing misleading speedup/regression numbers that do not reflect genuine kernel-level throughput differences — always validate with multi-token or batched workloads
- Contradictory benchmark results across different batch sizes (e.g., 2.4x speedup on single token vs 2.2x regression on batch) are a strong signal that the measured delta is noise rather than a real performance change — treat such contradictions as a red flag for benchmark validity
