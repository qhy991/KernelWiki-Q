## Challenge

GEMM routing for expert models uses a simple M-dimension threshold (m < 64) to decide whether to swap A/B matrices, which misroutes cases where M is moderate but arithmetic intensity is high, or where M is large but arithmetic intensity is low, leading to suboptimal downstream GEMM throughput


## Solution

Replace the single M-dimension threshold with an arithmetic-intensity-aware routing formula: compute AI = 2mnk / (mk + kn + 2mn) per expert and swap A/B when m <= 32 or AI < 70, enabling routing decisions based on actual computational efficiency rather than a single dimension heuristic

## Key Lessons

- Arithmetic intensity (AI = FLOPs / bytes) is a more accurate routing criterion than any single matrix dimension for GEMM dispatch — use AI < threshold to identify memory-bound problems that benefit from A/B swapping, applicable whenever multiple GEMM problem sizes share a dispatch path
- Adding ~10 float instructions to a lightweight routing kernel is negligible when the kernel is already dominated by global memory reads/writes (6 int reads + 3 int writes per thread), so per-thread compute cost should not deter better routing logic
- Downstream compute throughput gains from improved routing can outweigh any overhead in the routing kernel itself — always evaluate routing heuristics by end-to-end GEMM throughput, not the routing kernel's latency
