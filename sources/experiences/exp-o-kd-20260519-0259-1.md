## Challenge

FMHA kernel uses a 4-stage shared-memory pipeline (kStages=4) for medium head sizes where head_size_v (128) equals kKeysPerBlock (128), allocating 33792 bytes of shared memory per block and requiring an opt-in to 163840 bytes via cudaFuncSetAttribute, which reduces L1 cache capacity and limits SM occupancy while also issuing 4 compute iterations with 4 __syncthreads barriers


## Solution

Reduce kStages from 4 to 1 (since ceil(128/128)=1 stage suffices when head_size_v equals kKeysPerBlock), cutting shared memory from 33792 to 8448 bytes per block so the kernel fits within the default 48KB allocation without cudaFuncSetAttribute, preserving full L1 cache capacity and SM occupancy while eliminating 3 redundant compute iterations and 3 __syncthreads barriers

## Key Lessons

- When kKeysPerBlock >= head_size_v, only 1 pipeline stage is needed (kStages = min(default_stages, ceil(head_size_v / kKeysPerBlock))) — over-pipelining wastes shared memory and adds unnecessary sync barriers
- Keeping shared memory per block under 48KB avoids cudaFuncSetAttribute opt-in, preserving full L1 cache capacity and maximizing SM occupancy — the occupancy gain alone can dominate instruction-level savings
- Reducing __syncthreads barriers from N to 1 in a staged pipeline directly cuts barrier stall time by ~Nx, compounding with the occupancy improvement from smaller smem footprint
