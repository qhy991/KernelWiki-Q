## Challenge

Scaling factor is applied inside the atomicMaxFloat call as max_val * scaling, introducing an extra FMUL on every atomicCAS retry iteration in the contended critical section, and the kLog2e multiply is redundantly applied to both the fragment values and the max index mi in separate steps


## Solution

Pre-scale the fragment with scaling * kLog2e before the max-reduction so atomicMaxFloat receives the already-scaled value directly, eliminating the per-retry multiply from the atomicCAS loop and removing the redundant kLog2e multiply on mi in the exp2f stage

## Key Lessons

- Pre-scaling values before an atomicCAS-based reduction eliminates the per-retry multiply from the contended critical path — applies whenever a scaling factor can be absorbed into the input before an atomic operation
- Fusing scaling * kLog2e into the fragment pre-computation unifies the scaling across both the max-reduction and exp2f stages, removing a redundant multiply on the shared max index — applies to online softmax and flash-attention kernels where the log2e factor appears in both paths
