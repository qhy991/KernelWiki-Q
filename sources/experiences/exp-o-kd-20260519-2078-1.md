## Challenge

Paged KV-cache gather uses a single CTA that iterates sequentially over all blocks in a nested loop (outer: blocks, inner: tokens per block), performing repeated block_table lookups per block and divergent conditional branches for partial-block boundary handling, resulting in extreme underutilization of GPU parallelism


## Solution

Replace the sequential single-CTA block-level loop with a parallel per-token direct addressing kernel that launches one CTA per token, computing each token's source address via a single block_table lookup using token_to_seq mapping (batch_offset/block_size for block index, batch_offset%block_size for slot), eliminating nested loops and divergent partial-block branches while exposing massive parallelism across all tokens

## Key Lessons

- Replace sequential single-CTA block-level loops with parallel per-token CTAs when each token's work is independent — reduces block_table lookups from O(blocks) per CTA to O(1) per token and eliminates divergent partial-block boundary branches
- For paged KV-cache gather, compute per-token source addresses directly via division/modulus (batch_offset/block_size for block index, batch_offset%block_size for slot) rather than iterating through blocks — the token_to_seq mapping enables this direct computation across batches
- Launching one CTA per token (gridDim = num_tokens) instead of one CTA for all tokens exposes the full parallelism of the problem, yielding orders-of-magnitude speedup when the kernel is memory-bandwidth-bound and each token's copy is independent
