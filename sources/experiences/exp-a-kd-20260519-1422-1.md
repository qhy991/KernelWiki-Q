## Challenge

Scalar rms_norm kernel processes hidden_size=4096 elements using 1024 threads with element-wise half-precision loads from input and weight and stores to output per stride iteration, leaving potential memory bandwidth optimization opportunity


## Solution

Replace scalar element-wise loads/stores with vec_n_t<half,8> 128-bit packed vector loads/stores across both variance and normalization phases, but the accompanying thread count reduction from 1024 to 512 (hidden_size/VEC_SIZE) halves occupancy and memory-level parallelism, causing a net regression

## Key Lessons

- Vectorizing loads/stores with wider (128-bit) access while proportionally reducing thread count (1024 to 512) can regress performance because the 2x drop in occupancy reduces in-flight memory requests, and the bandwidth gain from wider accesses does not compensate — vectorization must preserve or increase thread count to net positive
- For kernels with a block reduction phase (like rms_norm variance), reducing thread count directly slows the reduction because fewer threads participate, and this overhead is not offset by fewer loop iterations in the vectorized path
- Naive vectorization that substitutes VEC_SIZE elements per thread and divides blockDim.x by VEC_SIZE is a net-negative transformation when the original thread count is already modest (1024) and the workload is memory-latency-bound rather than bandwidth-saturated
