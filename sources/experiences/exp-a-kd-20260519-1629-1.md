## Challenge

Inline fp8-to-half dequantization loads and converts each 8-byte chunk within the same loop iteration, interleaving memory loads with arithmetic conversions and preventing the compiler and hardware scheduler from fully overlapping independent operations


## Solution

Split the fused load+convert loop into two separate phases: Phase 1 loads all raw fp8 data into a compact uint8_t buffer (Klocalb8[KHELOOP][8]), then Phase 2 batch-converts all chunks to half in a separate loop, decoupling memory accesses from arithmetic and giving the scheduler more freedom to overlap independent operations

## Key Lessons

- Separating pure memory loads from arithmetic conversions into distinct phases can improve instruction-level parallelism by decoupling load latency from compute latency, but the benefit is marginal when working sets are small enough (HEAD_SIZE=64, 8 chunks) for the compiler to auto-schedule the inline pattern effectively
- Deferred dequantization increases register pressure by requiring an intermediate buffer (Klocalb8) alongside the output buffer (Klocal), so the tradeoff favors this pattern at larger head sizes where the scheduling flexibility outweighs the register cost
