## Solution

Add an `else if (block_size == 32)` dispatch branch calling `multi_query_cached_kv_attention_launcher<uint16_t, 32>()`, leveraging the existing template's compile-time constants (THREAD_GROUP_SIZE=1, NUM_V_VECS_PER_ROW=2 for half/BLOCK_SIZE=32) to automatically adapt value-loading and reduction logic for the larger block size

## Key Lessons

- Extending a block-size dispatch table with a new BLOCK_SIZE template instantiation is low-risk when the kernel template already encodes all block-size-dependent constants (THREAD_GROUP_SIZE, NUM_V_VECS_PER_ROW, NUM_ROWS_PER_ITER) — the compiler generates the correct variant automatically
- For half-precision attention with BLOCK_SIZE=32, NUM_V_VECS_PER_ROW=32/vec_size determines how many vector loads each thread performs per row; changing block size shifts the work distribution across warps without altering the algorithmic structure
- Multi-query attention shares the same THREAD_GROUP_SIZE=1 mechanism as single-query for large block sizes, meaning each thread computes its own QK dot product independently — no group-level reduction is needed, simplifying the dispatch extension
