## Solution

Replace GemmIdentityThreadblockSwizzle with ThreadblockSwizzleStreamK, which splits K-dimension work of each output tile across multiple threadblocks mapped to different SMs, filling idle SMs; a fused epilogue accumulates partial products from K-subdivisions in shared memory before writing final output, preserving correctness while enabling higher throughput

## Key Lessons

- Use ThreadblockSwizzleStreamK instead of GemmIdentityThreadblockSwizzle when the number of output tiles (M/128 * N/128) is less than the SM count, so that K-dimension splitting can engage idle SMs — applies to any GEMM where large tile sizes produce too few tiles for full GPU occupancy
- Set avail_sms=-1 in GemmUniversal Arguments to let the Stream-K runtime use all available SMs for K-dimension work distribution, maximizing the occupancy benefit of the Stream-K decomposition
- Stream-K's fused epilogue accumulates partial products in shared memory across K-subdivisions before writing the final tile — this preserves numerical correctness of the GEMM result without requiring a separate reduction kernel
