## Challenge

FP8 quantization kernel wraps conversion in a class (Fp8Wrapper) with conditional code paths and routes through a __noinline__ function pointer trampoline, forcing a per-element function call boundary that spills registers and prevents the compiler from fusing the conversion into the stride loop


## Solution

Replace the class-wrapped Fp8Wrapper + __noinline__ trampoline with a direct __forceinline__ function (cvt_float_to_fp8) that performs the same bit manipulation inline, eliminating the per-element call boundary and enabling the compiler to fuse the conversion into the kernel's stride loop

## Key Lessons

- Class wrappers with conditional code paths (e.g., #ifdef __HIP__MI300__ asm vs software fallback) and __noinline__ trampolines create per-element call boundaries that prevent register fusion into inner loops — replace with __forceinline__ direct functions for type-conversion operations in hot paths
- On NVIDIA hardware without native fp8 support, both class-wrapped and direct software conversion produce identical SASS bit manipulation, so the overhead is purely the noinline call boundary (~0.7%) — the real benefit of the direct intrinsic pattern is on AMD ROCm where __hip_cvt_float_to_fp8 maps unconditionally to v_cvt_f32_fp8 hardware instruction regardless of GPU variant
