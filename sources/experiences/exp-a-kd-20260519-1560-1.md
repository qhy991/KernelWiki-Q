## Challenge

The kernel assumes contiguous query layout and requires an expensive cudaMemcpy2D device-to-device strided copy to extract the Q portion from the non-contiguous QKV buffer (stride=3072 floats) into a separate contiguous buffer (stride=1024 floats) before every kernel invocation, moving 65,536 floats across GPU memory per iteration


## Solution

Replace the hardcoded contiguous offset (seq_idx * num_heads * HEAD_SIZE) with a runtime q_stride parameter (seq_idx * q_stride), allowing the kernel to compute correct per-row addresses directly on the original non-contiguous QKV buffer and eliminating the intermediate cudaMemcpy2D copy and its dedicated contiguous buffer allocation entirely

## Key Lessons

- Accept a runtime stride parameter and compute row addresses with pointer arithmetic on the original non-contiguous buffer instead of copying to a contiguous intermediate — eliminates cudaMemcpy2D overhead and extra buffer allocation whenever the kernel reads only contiguous sub-ranges within strided rows
- Within each (seq_idx, head_idx) pair the kernel reads HEAD_SIZE=128 contiguous floats (512 bytes) as a single sequential access, so the inter-row stride does not penalize per-row memory throughput — non-contiguous layout is zero-cost when per-row accesses are contiguous
- Pointer arithmetic cost is identical between hardcoded and stride-based offset (one multiply-add), so making a kernel stride-aware has zero computational overhead and purely removes memory copy latency
