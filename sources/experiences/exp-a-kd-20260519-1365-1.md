## Solution

Apply swapAB heuristic: when M<=64 or M%4!=0, transpose the GEMM by swapping A/B with CUBLAS_OP_T so the smaller dimension becomes N, enabling a different tile configuration (128x32x128) with potentially better alignment

## Key Lessons

- swapAB operand transposition in cuBLAS adds overhead on SM80 because TMA hardware cannot accelerate transposed tile loads, so the tile utilization benefit is outweighed by uncoalesced access penalties — this trade-off flips on SM120 where TMA natively handles transposes
- For M values close to a tile boundary (M=65 vs TILE_M=64), the partial-tile penalty is modest compared to very small M (M=3), so the swapAB optimization should be gated on whether the tile utilization gain exceeds the transpose access cost for the specific hardware generation
