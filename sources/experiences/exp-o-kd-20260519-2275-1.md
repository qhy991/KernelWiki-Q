## Challenge

Scalar kernel computes per-element division and modulo (head_idx = i / head_size, head_offset = i % head_size) for strided KV-cache writes, issuing single 2-byte stores per element and leaving 128-bit memory bandwidth underutilized across the 16384-element key/value copy per token


## Solution

Recognize that NHD layout (head_stride == head_size) causes the multi-dimensional target index head_idx*head_stride + head_offset to collapse to flat offset i, enabling pre-computation of source/destination base pointers once per block and a contiguous vectorized copy using 128-bit (VEC_SIZE=8) loads/stores that eliminates all per-element div/mod and achieves ~4x fewer memory transactions

## Key Lessons

- When head_stride equals head_size (NHD layout), the multi-dimensional index head_idx * head_stride + head_offset simplifies to just i — recognize this algebraic identity to convert strided scatter/gather into a contiguous flat copy
- Use VEC_SIZE=8 (128-bit) vectorized loads/stores for fp16/bf16 memory-bound kernels to issue 16 bytes per transaction instead of 2 bytes, achieving up to 8x higher effective memory throughput per warp
- Pre-compute base pointers outside the thread loop to enable vectorization — moving address arithmetic from per-element to per-block eliminates both the div/mod overhead and the dependency chain that prevents the compiler from generating vectorized memory instructions
