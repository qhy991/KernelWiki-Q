## Solution

Replace permutation loop with direct threadIdx.y-based indexing using a threadIdx.x guard (threadIdx.x < n_per_r/QK8_1) and pre-loaded token offsets, which enables MoE expert-routing via token_offs[threadIdx.y]/top_k lookups but causes severe warp divergence with only 25% of threads active (8 of 32), resulting in a 2.3x performance regression

## Key Lessons

- A threadIdx.x guard that activates only 25% of warp threads (8 of 32) causes severe warp divergence that outweighs any arithmetic simplification — avoid partial-warp guards in hot loading paths when all-thread alternatives exist
- Direct threadIdx.y-based indexing enables per-warp token-offset lookups (token_offs[threadIdx.y]) for MoE expert routing that are infeasible with modulo-scatter patterns — the architectural benefit can justify a localized regression when it unlocks a higher-level optimization
- Permutation loops with modulo scatter keep all threads active and avoid warp divergence, but couple the loading logic to a fixed column mapping — prefer when the indexing pattern is known at compile time and MoE-style indirection is not needed
