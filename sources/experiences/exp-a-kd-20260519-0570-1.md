## Solution

Replace synchronous FLOAT4 B-tile loads with cp.async.ca.shared.global (CP_ASYNC_CA) intrinsics that issue non-blocking global-to-shared transfers, commit them as a group, then proceed with A loading, BK compute, and A transpose while the B async transfers are still in flight; a CP_ASYNC_WAIT_GROUP(0) barrier before __syncthreads ensures completion

## Key Lessons

- Use cp.async.ca.shared.global (CP_ASYNC_CA) to issue non-blocking global-to-shared transfers for the next data tile, then commit the group so the hardware can prefetch while the warp executes compute on the current tile — applies to any tiled GEMM/conv kernel with a double-buffered shared memory scheme
- Decouple B-tile async loads from A-tile synchronous loads by issuing CP_ASYNC_COMMIT_GROUP after the B transfers, allowing B prefetching to proceed in the background while A is loaded via normal FLOAT4 instructions — the two memory paths do not block each other
- Place CP_ASYNC_WAIT_GROUP(0) at the end of the loop iteration (after compute and A transpose) rather than before compute, maximizing the overlap window between async memory transfers and arithmetic operations
