## Solution

Swap operands A and B and apply CUBLAS_OP_T on both, reformulating the GEMM as (N,K)x(K,M) so the large dimension N=4096 becomes the kernel's effective M, enabling full tile utilization; on SM80 cuBLAS the speedup is neutral (1.001x) because cuBLAS already dispatches a specialized low-M vector-matrix path regardless of operand ordering

## Key Lessons

- swapAB (transpose both operands to swap M/N roles) is neutral on SM80 cuBLAS which already uses a specialized low-M path for vector-matrix multiply, but is essential on SM120 CUTLASS where compile-time fixed tile shapes like 64x128x128 waste nearly all TILE_M rows when M=1 — the swapped formulation lets a 128x32x128 tile fully utilize TILE_M=128 across the large dimension
- For tiny-M GEMM workloads, profile the optimization on the target architecture — an optimization that is neutral on one GPU generation (SM80 cuBLAS) may be critical on another (SM120 CUTLASS) due to differences in kernel dispatch strategy and tile shape constraints
