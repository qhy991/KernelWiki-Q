## Challenge

Warp index is computed via __shfl_sync(0xffffffff, threadIdx.x / 32, 0) inside a tight unrolled loop, forcing an implicit warp-level barrier on every iteration that prevents the compiler from hoisting the lane-constant threadIdx.x / 32 expression out of the loop or applying common subexpression elimination


## Solution

Replace __shfl_sync-based warp index broadcast with direct arithmetic (threadIdx.x / 32), which has no synchronization semantics, allowing the compiler to compute the warp index once, hoist it out of the loop, and reuse the register across all 10000 iterations

## Key Lessons

- Never use __shfl_sync to broadcast a warp-uniform value that can be computed from threadIdx/blockIdx via pure arithmetic — __shfl_sync injects an implicit barrier that blocks loop-invariant code motion and common subexpression elimination, even when every lane computes the identical value
- threadIdx.x and blockIdx.x are read-only special registers constant across the kernel lifetime; expressions derived solely from them (e.g., threadIdx.x / 32 for warp index) are loop-invariant and will be hoisted by the optimizer once freed from synchronization barriers
- When an operation is inside a hot unrolled loop, the compound speedup from enabling compiler optimizations (hoisting + CSE + register reuse) can be an order of magnitude larger than the cost of a single eliminated instruction — the 11.5x speedup here comes from removing 10000 SHFL barriers, not one
