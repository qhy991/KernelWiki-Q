## Challenge

Tile offset pointer arithmetic in shared memory navigation omits the tile height multiplier (kRow) from the row component, causing the pointer to land at the wrong row when computing addresses for tiles that are multiple rows away from the origin


## Solution

Insert the missing kRow multiplier into the row offset computation so the pointer advances by offset_row * kRow rows instead of just offset_row rows, producing the correct tile address for multi-row tile navigation

## Key Lessons

- When computing byte offsets for tile-based shared memory navigation, the row component must include the tile height multiplier (offset_row * kRow * stride) — omitting it causes the pointer to skip only individual rows instead of full tile-height rows, producing silently incorrect results
- A single-ALU correctness fix (adding one multiply) in an address computation is entirely hidden in memory-bound kernels — the performance delta is within measurement noise, confirming that pointer arithmetic cost is negligible compared to shared memory access latency
