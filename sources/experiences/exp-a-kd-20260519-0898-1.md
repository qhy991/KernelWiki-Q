## Challenge

AllReduce requires a host-side cudaMemcpyAsync to copy input into the shared communication buffer before launching the reduction kernel, incurring a full separate D2D copy operation plus kernel launch overhead serialized on the same stream


## Solution

Merge the D2D copy into the reduction kernel using a compile-time COPY_INPUT template parameter: each thread first copies its chunk of input to the local shared buffer via vectorized int4 loads, then proceeds to the reduction loop — eliminating the separate cudaMemcpyAsync and its kernel launch overhead, and allowing copy and reduce to pipeline within a single execution context

## Key Lessons

- Merging a pre-launch D2D copy (cudaMemcpyAsync) into the kernel as an in-kernel copy phase eliminates one full kernel launch overhead and allows the copy and compute to pipeline within the same execution context — applies whenever a kernel's input copy can be distributed across the same thread grid as the compute work
- Use a compile-time bool template parameter (e.g., COPY_INPUT) guarded by if constexpr to conditionally include the in-kernel copy path with zero runtime overhead when disabled, enabling a single kernel binary to serve both capturing and non-capturing modes
- Distribute the in-kernel copy across the same thread grid using vectorized int4 loads/stores so each thread copies its own chunk — the copy leverages the same memory parallelism as the reduction and adds negligible latency on top of the compute phase
