## Challenge

The before kernel iterates globally across all rows with `for (globalIdx = tid; globalIdx < numRows * colsPerRow; globalIdx += stride)` and for each element performs a register-chunk search through 64 offset entries, loading 16 uint32 offsets per chunk into local_offsets[17] and doing up to 64 co...


## Solution

The f16x8 variant unrolls per-thread processing from 1 half2 to 4 half2 loads/computes/stores with separate boundary checks for each pair, while keeping the same total elements-per-block (256) but reducing thread count 4x — this is an occupancy-destroying over-vectorization that cannot be offset by instruction-level parallelism alone

## Results

- **speedup**: 1.412x (improved)


## Key Lessons

- Increasing per-thread vectorization without maintaining sufficient threads-per-block destroys occupancy — each block must contain enough warps (ideally ≥4) for the SM scheduler to hide memory latency through warp-level context switching
- Over-vectorization to f16x8 (4 half2 per thread) in bandwidth-bound elementwise kernels offers no ILP benefit when the bottleneck is memory throughput, not compute — the added register pressure and boundary-check branches only add overhead
- When increasing vector width, keep blockSize ≥ 128 threads by scaling gridSize proportionally rather than dividing blockSize by the vectorization factor — this preserves occupancy while still reducing total thread count
