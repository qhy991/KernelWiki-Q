## Challenge

Synchronous global-to-shared load (smem[tid] = g_in[tid]) issues blocking LDG+STS for each thread; replacing it with cp.async.ca.shared.global in a single-stage kernel with immediate commit_group and wait_all adds instruction overhead without enabling any pipeline overlap


## Solution

Replace synchronous LDG+STS with cp.async.ca.shared.global inline PTX to issue asynchronous global-to-shared copies, followed by cp.async.commit_group and cp.async.wait_all to fence — but in a single-stage kernel this eliminates no overlap opportunity and the async path overhead causes marginal slowdown

## Key Lessons

- cp.async only benefits multi-stage pipelines where async loads for stage N+1 can overlap with computation on stage N — in a single-stage copy-and-compute kernel, immediate commit_group+wait_all negates any overlap advantage and the extra PTX instructions add overhead
- Small data transfers (here 32×8B = 256 bytes) cannot saturate memory bandwidth regardless of sync vs async; async copy advantages require enough data volume and pipeline depth to hide latency
- Using __cvta_generic_to_shared to obtain a shared-memory pointer for cp.async PTX is correct but adds a pointer-conversion instruction per thread — worthwhile only when the async overlap yields measurable gains
