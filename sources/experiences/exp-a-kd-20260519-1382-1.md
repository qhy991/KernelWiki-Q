## Solution

Replace scalar element-wise access with vec4_t<__half> (8-byte) and float4_t (16-byte) struct reinterpretation, processing 4 elements per loop iteration with #pragma unroll 4, plus a scalar tail-loop for remainder elements

## Key Lessons

- NVCC -O3 already coalesces scalar half-word reads across a warp into efficient wide HBM transactions when consecutive threads access consecutive addresses — manual per-thread vectorization via reinterpret_cast to vec4_t does not increase warp-level memory throughput because the same 32 threads cover the same address range per stride step
- Per-thread vectorization with struct reinterpretation can introduce overhead: reinterpret_cast pointer manipulation, stride index changes (scalar vs vec-element indexing), added tail-loop for remainder elements, and potential register pressure increase from holding 4-element structs — these costs can negate any memory throughput gains when the compiler already coalesces accesses
- Before applying manual vectorized loads/stores, profile the scalar version first; if the compiler already generates coalesced memory transactions across the warp, vectorization at the thread level is unlikely to help and may slightly hurt performance
