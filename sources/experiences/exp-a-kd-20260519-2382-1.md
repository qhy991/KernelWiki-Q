## Solution

Replace the single-block 3-phase shared-memory kernel with a multi-block (8 blocks x 256 threads) single-pass kernel that uses atomicAdd on the pre-computed global cumsum buffer to claim output positions, eliminating all __syncthreads barriers and shared-memory management overhead while scaling parallelism from 64 to 2048 threads

## Key Lessons

- For token-to-expert sorting in MoE routing, a single-pass atomicAdd on a pre-computed global cumsum buffer outperforms a multi-phase shared-memory approach (init+count, prefix-sum, scatter) because it eliminates __syncthreads barriers and scales to multi-block parallelism — applies when the number of atomics targets (num_experts) is small enough for low contention
- When the workload size (numel) significantly exceeds the thread count of a single block, launching multiple blocks with a grid-stride loop provides better GPU occupancy than cramming all work into one block with per-thread batching
- atomicAdd contention on a small number of global counters (e.g., 64 expert slots) is negligible in practice when numel is much larger than num_experts, because tokens distribute across experts and serializations are rare
