## Challenge

In SSM selective_scan_fwd_kernel with kNRows > 1, the shared memory write for the running prefix uses index smem_rp[si] without the row offset r * MAX_DSTATE_VAL, while the corresponding read correctly uses smem_rp[si + r * MAX_DSTATE_VAL], causing row 1 to overwrite row 0's slot and leaving row 1's own slot unwritten (still at initial (1.0, 0.0))


## Solution

Align the shared memory write index with the read index by adding the row offset r * MAX_DSTATE_VAL, so each row stores its running prefix in its own dedicated shared memory slot and cross-row state does not alias

## Key Lessons

- When multi-row kernels share a flat shared memory array indexed per-row, verify that every read and write path uses the same row-stride offset (r * MAX_DSTATE) — a mismatched write index silently corrupts other rows' state without affecting performance
- Shared memory index aliasing bugs in scan/prefix-sum kernels are latency-neutral but correctness-critical: the before version compiles and runs at full speed yet produces wrong output for all rows beyond the first on multi-chunk sequences
