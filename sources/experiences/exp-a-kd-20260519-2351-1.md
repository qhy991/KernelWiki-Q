## Challenge

The async-only wrapper checks only cudaGetLastError after kernel launch, missing runtime errors (e.g., illegal memory access) that surface only after kernel completion — errors go undetected in production use


## Solution

Embed cudaDeviceSynchronize and printf-based error diagnostics inside the wrapper to catch both launch and runtime errors, accepting a ~12% latency regression on small payloads where the full device-wide barrier cost dominates the microseconds-long kernel execution

## Key Lessons

- cudaDeviceSynchronize inside a wrapper forces a full device-wide barrier per call — for small payloads where kernel time is microseconds, the driver overhead of sync dominates and can regress throughput by 10-15%
- Async-only error checking (cudaGetLastError) catches launch failures but misses asynchronous runtime errors (illegal memory access, out-of-bounds); prefer cudaStreamSynchronize on a dedicated stream for targeted error detection without the full-device barrier cost
- Printf-based conditional error branches in hot-path wrappers add host-side instruction overhead even in the no-error case — for latency-sensitive small-kernel wrappers, consider error reporting via return codes only and defer diagnostics to a separate debug build path
