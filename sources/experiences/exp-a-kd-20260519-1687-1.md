## Challenge

MoE routing pipeline launches three separate kernels — compute_problem_sizes (64×512 threads for per-expert counting), compute_expert_offsets (single-thread serial prefix sum), and compute_arg_sorts (64×512 threads with atomicAdd sorting) — incurring two extra kernel launch overheads and redundant atomic operations for the offset and sort stages


## Solution

Eliminate the compute_expert_offsets and compute_arg_sorts kernel launches entirely, retaining only compute_problem_sizes which is the sole required stage when downstream consumers only need problem_sizes1/problem_sizes2; this removes two kernel launch overheads plus the atomicAdd-based sorting work across 64 blocks of 512 threads

## Key Lessons

- Removing unnecessary pipeline stages (kernel launch elimination) yields significant speedups when downstream kernels produce data that is not consumed by subsequent operations — verify dataflow before adding stages
- Kernel launch overhead is non-trivial for short-running kernels: compute_expert_offsets runs a single-thread serial prefix sum over 64 elements, so its fixed launch cost dominates execution time
- Atomic-free paths are preferable when the output of atomicAdd-based sorting (compute_arg_sorts) is not needed by the next pipeline stage — atomic contention across 64×512 threads wastes GPU resources unnecessarily
