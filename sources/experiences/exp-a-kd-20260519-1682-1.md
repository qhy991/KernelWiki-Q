## Solution

Replace FP32-based 32-bit sign-flip and shift-mask with convert_to_uint8 which converts FP32 to FP16 via __float2half_rn and extracts the high byte with a cheaper 16-bit sign-flip, reducing per-element arithmetic in the coarse histogram pass

## Key Lessons

- Replacing FP32 bit-manipulation with FP16-based bin extraction in a memory-bound radix top-K coarse pass yields negligible speedup because the per-element arithmetic is hidden by memory latency — arithmetic simplification only matters when the kernel is compute-bound
- FP16 high-byte extraction collapses more FP32 values into the same 256 bins (FP16 has 10 mantissa bits vs FP32's 23), changing bin boundaries; this is acceptable for a coarse pruning pass but the resulting histogram distribution differs from FP32-based binning
- A two-tier radix top-K (FP16 coarse pass + FP32 refinement passes) is architecturally sound for separating cheap coarse pruning from precise refinement, but the coarse conversion method choice is performance-irrelevant at seq_len=8192 where the 32KB logit read dominates
