## Solution

Switch accumulation from native fp16 (__hadd/__hmul) to fp32 by converting shared memory tile values to float before multiply-add, trading higher precision for increased conversion instruction count and register pressure

## Key Lessons

- fp32 accumulation via __half2float conversion improves GEMM numeric precision over native fp16 __hadd/__hmul but increases instruction count and register pressure — the per-kernel latency regression is expected when conversion overhead exceeds the benefit of wider compute
- Native fp16 arithmetic (__hadd/__hmul) is lower-latency than fp32 accumulation with explicit conversions for small tiled GEMM where smem loads dominate — use fp16 accumulation when precision tolerance allows
