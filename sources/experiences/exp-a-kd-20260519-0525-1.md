## Challenge

cuBLAS GEMM wrapper uses a global singleton handle with lazy initialization (per-call null check + init_cublas_handle) and static half alpha/beta variables, introducing hidden mutable state and an unnecessary host-side branch on every invocation


## Solution

Refactor to pass cublasHandle_t as an explicit parameter and use local alpha/beta variables instead of static globals, eliminating hidden state and per-call null checks — the resulting host-side overhead reduction is negligible because cuBLAS GEMM kernel execution dominates at this problem size

## Key Lessons

- Host-side micro-optimizations (removing a null check, static-to-local variable changes) are negligible when GPU kernel execution dominates — cuBLAS GEMM at large problem sizes (>18 ms per call) completely masks nanosecond-scale host overhead, so apparent deltas under ~5% are run-to-run variance
- Passing library handles as explicit parameters is a better software engineering practice (testability, no hidden global state) even when it produces no measurable GPU performance difference — the refactoring is justified by code quality, not speedup
- static half and local half compile to identical register/materialized constants in device code, so changing storage duration of scalar GEMM coefficients has zero effect on the launched kernel
