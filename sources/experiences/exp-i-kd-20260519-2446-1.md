## Solution

Use CUTLASS's GemmGroupedProblemVisitor to manage shared storage and iterate tiles via next_tile() with grid-stride advancement; compute grid shape per problem and decompose linear threadblock_idx into 2D tile coordinates (M, N) for threadblock-level tiling

## Key Lessons

- Use GemmGroupedProblemVisitor with shared storage to handle multi-problem iteration in a single kernel launch — the visitor manages problem boundaries and tile tracking so threadblocks seamlessly cross between different GEMM problems
- Compute 2D threadblock tile indices from linear threadblock_idx via grid_shape decomposition: tile_m = idx / grid_shape.n(), tile_n = idx % grid_shape.n() — this maps 1D threadblock scheduling to 2D tile space for M and N dimensions
- Use grid-stride advancement (problem_visitor.advance(gridDim.x)) so fewer threadblocks than total tiles can cover all work — each threadblock iterates across multiple tiles, improving GPU utilization for irregular problem sizes
- Configure DefaultGemmGrouped template with GemmShape<128,128,32> threadblock tile, GemmShape<64,64,32> warp tile, and GemmShape<16,8,16> MMA instruction for SM 80 tensor core half_t GEMM, adjusting kAlignment (8 for half_t, 1 for SIMT) to match the element access pattern
