## Challenge

Persistent tile scheduler assigns whole tiles to SMs via cooperative kernel launch with grid_size=min(tiles, SMs). For tall-skinny GEMM (M=2048, N=256, K=8192), only 16×2=32 tiles exist but 108 SMs are available, leaving 76 SMs completely idle and achieving only 29.6% SM utilization


## Solution

Replace persistent whole-tile scheduler with StreamK-style decomposition: split each tile's K dimension into K-iteration work units (total_k_iters = tiles × ceil(K/TK)), then distribute all 8192 K-iterations across all 108 SMs via strided grid-rank loop with atomicAdd for partial result accumulation, achieving full SM utilization

## Key Lessons

- StreamK decomposition splits GEMM tiles along K into fine-grained K-iteration work units distributed across all SMs — eliminates idle-SM tail for tall-skinny problems (tiles < SMs) by creating enough parallel work units to saturate the GPU
- When decomposing tiles across SMs, each SM produces a partial result for shared output elements — use atomicAdd for accumulation since multiple SMs may write to the same C[row,col] when a tile's K dimension is split
- Cooperative kernel launch with grid_size=min(tiles,SMs) underutilizes hardware when problem dimensions produce few tiles (tall-skinny or small-batch GEMM) — prefer launching grid_size=num_SMs and decomposing work at a finer granularity
