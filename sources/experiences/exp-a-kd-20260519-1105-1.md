## Challenge

Half2 accumulation via __hfma2 overflows when unscaled 8-bit weights occupy the -128..127 range, because the product of two half values in that range exceeds the representable range of FP16 during intermediate accumulation


## Solution

Decompose each half2 weight pair into individual FP32 values via __low2float/__high2float, convert activations to FP32, and accumulate with scalar FP32 fma calls to prevent overflow — the resulting ~3x instruction increase is hidden by memory-bound latency in quantized GEMM contexts

## Key Lessons

- For 8-bit quantized weight dot products where unscaled weights span -128..127, FP16 (half2) accumulation risks overflow — switch to FP32 scalar accumulation via __low2float/__high2float decomposition for correctness
- In memory-bound quantized kernels, replacing 4 packed __hfma2 instructions with 8 scalar FP32 fma calls incurs negligible performance cost (~1%) because arithmetic latency is hidden behind global memory access latency
- __hfma2 provides throughput but limited dynamic range for accumulation; when operand magnitudes exceed ~128 (8-bit range), the FP16 accumulator saturates and produces incorrect results — prefer FP32 accumulation for bit-widths >= 8
