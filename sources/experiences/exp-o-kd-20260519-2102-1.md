## Challenge

The dequantize_s4_to_fp16x2 kernel guards its optimized lop3.b32 / fma.rn.f16x2 dequantization path with __CUDA_ARCH__ < 800, causing Turing GPUs (sm_75) that fully support these PTX instructions to hit assert(false) at runtime instead of using the fast path


## Solution

Lower the preprocessor arch guard from __CUDA_ARCH__ < 800 to __CUDA_ARCH__ < 750, allowing sm_75 (Turing) GPUs to execute the optimized lop3-based dequantization path since lop3.b32 and f16x2 arithmetic are both available on sm_75+

## Key Lessons

- When guarding PTX instruction paths by __CUDA_ARCH__, check the minimum architecture that actually supports each required instruction — lop3.b32 was introduced in sm_70 and f16x2 arithmetic in sm_53, so sm_75 is sufficient even though sm_80 is a common default guard threshold
- A too-conservative arch guard is functionally equivalent to a missing fast path on excluded architectures — validate that guards match actual ISA requirements rather than using the latest architecture as a blanket threshold
