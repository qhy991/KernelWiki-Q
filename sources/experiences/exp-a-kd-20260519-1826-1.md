## Challenge

W8A8 GEMM kernel uses pipe_stages=4 for double-buffered shared memory, consuming 64640 bytes which exceeds Turing's 64KB shared memory limit (after 512-byte reservation), preventing valid kernel launch


## Solution

Reduce pipeline stages from 4 to 2 and thread the stage count as a runtime parameter (stages_param) through the shared memory sizing functions, halving all stages-dependent buffers (sh_a_size, sh_b_size, sh_s_size, sh_g_idx_size) to bring total shared memory from 64640 bytes down to 39040 bytes, fitting comfortably within 64KB

## Key Lessons

- Reducing pipeline stages (e.g., 4→2) is an effective way to cut shared memory consumption when total smem exceeds hardware limits — the resulting kernel launchability gain far outweighs the reduced software pipelining depth
- Thread the stage count as a function parameter rather than using a global constant, enabling per-architecture tuning (e.g., stages=4 on Ampere with 164KB smem vs stages=2 on Turing with 64KB smem) without code duplication
- When fixed-size shared memory components (like b_size with pack_factor=4 for W8A8) dominate, the speedup from halving stages is proportionally less than 2x because only the stages-dependent fraction shrinks
