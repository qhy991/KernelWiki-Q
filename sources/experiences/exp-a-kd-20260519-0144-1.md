## Solution

Replace SM80 FP16 mma.sync m16n8k16 with SM89 FP8 mma.sync m16n8k32, which doubles the K dimension per instruction from 16 to 32 and halves operand memory footprint (FP8 is 1 byte vs FP16's 2 bytes), achieving 8192 FLOPs per MMA atom — on SM89 hardware this doubles throughput at similar latency; on SM80 hardware a fallback issues two sequential K=16 FP16 MMA ops

## Key Lessons

- SM89 FP8 MMA (m16n8k32) doubles the K dimension per instruction compared to SM80 FP16 MMA (m16n8k16) — on native SM89 hardware this yields ~2x throughput gain at similar latency, but on SM80 hardware the fallback path issues two sequential FP16 MMA ops, so benchmark results on mismatched hardware will show apparent regression
- FP8 (e4m3) operands are 1 byte vs FP16's 2 bytes, halving register and shared memory footprint for matrix operands — this improves data movement efficiency and enables larger tile sizes within the same register budget
- When benchmarking architecture-specific instructions like SM89 FP8 MMA on older hardware (SM80), validate the hardware capability match — a regression measured on SM80 does not reflect the true performance on the target SM89 platform
