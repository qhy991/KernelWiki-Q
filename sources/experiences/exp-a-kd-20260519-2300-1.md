## Solution

Fuse all three per-query tensor updates (token copy, position/length increment, slot mapping computation) into a single advance_step_kernel so that next_input_pos stays in registers instead of being written to and read back from global memory, eliminating 2 kernel launch overheads

## Key Lessons

- Fuse multiple trivial kernels that operate on the same element range with 1:1 thread-to-element mapping into a single kernel — eliminates per-launch driver overhead (argument marshaling, command submission) which dominates when per-thread work is minimal
- Kernel fusion keeps intermediate results (next_input_pos) in registers instead of writing to global memory and reading back in the next kernel, avoiding redundant global memory round-trips for shared data
