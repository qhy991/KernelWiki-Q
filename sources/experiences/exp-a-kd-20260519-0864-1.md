## Challenge

MLA K-concat kernel uses a two-phase approach—first loading all 16 nope head rows and the rope row into registers, then storing them in a second loop—with default cache policy, which neither overlaps load latency with store work nor prevents stream-once data from polluting L1 cache


## Solution

Replace two-phase buffered load-then-store with a software-pipelined loop that prefetches the next iteration's nope data into L2 while writing the current iteration, combined with L1::no_allocate cache policy on loads and stores to avoid evicting hot cache lines with stream-once data

## Key Lessons

- Software pipelining (prefetch next iteration's data while writing current iteration) overlaps memory latency with useful store work — applies to any loop body with independent load and store phases across iterations
- L1::no_allocate cache policy prevents stream-once data from evicting hot cache lines in L1 — applies when data is read or written exactly once without temporal reuse within the kernel
- At small kernel scales (fewer than ~1000 warps), kernel launch overhead and synchronization dominate runtime, making memory subsystem optimizations like cache policy and prefetch indistinguishable from measurement noise
