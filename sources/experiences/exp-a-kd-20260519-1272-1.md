## Solution

Cap the grid size to 36 blocks via a BLOCK_LIMIT constant, reducing the number of atomicAdd contenders from ~2M to ~18K while each thread still strides over the full input, eliminating atomic serialization without changing the kernel code

## Key Lessons

- Cap grid size to a small multiple of SM count (e.g., 1–2 blocks per SM) for kernels that use global atomicAdd on a single address — excess blocks only increase atomic contention without improving throughput
- Grid-stride loops decouple data coverage from block count: fewer blocks still cover the full dataset by iterating with a larger stride, so reducing block count does not sacrifice correctness
