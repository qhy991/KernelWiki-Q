## Solution

Split the key range into 64 chunks, launch one CTA per chunk so 64×256=16 384 threads work in parallel, then combine partial sums with atomicAdd into a single output

## Key Lessons

- When a reduction kernel processes a large array with a single CTA, GPU occupancy is near zero — split the data range across multiple CTAs and combine partial sums with atomicAdd to saturate SM parallelism
- atomicAdd on a single output word is an acceptable synchronization cost for reduction epilogues when the number of CTAs is modest (e.g., 64), since contention is low and the alternative (a second kernel launch) has higher overhead
