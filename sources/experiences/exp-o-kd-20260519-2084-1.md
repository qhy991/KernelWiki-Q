## Solution

Replace custom shared-memory halving reduction with CUB BlockReduce, passing blockDim.x as num_valid so CUB knows only 256 threads carry valid data — CUB internally uses warp-shuffle intrinsics that reduce shared-memory traffic and skip unnecessary comparisons against uninitialized slots from idle threads

## Key Lessons

- Use CUB BlockReduce with the num_valid parameter for partial-block participation scenarios — CUB's warp-shuffle path avoids shared-memory writes from idle threads and skips unnecessary comparisons, yielding significant speedup when active thread count is much smaller than blockDim.x
- Custom shared-memory reductions that allocate slots for all threads and perform full halving iterations waste bandwidth when only a subset of threads carry valid data — a library reduction with an explicit valid-count parameter eliminates this overhead entirely
