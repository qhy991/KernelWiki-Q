## Challenge

Fixed dim3(16,16) block tile used for all GEMM problem sizes regardless of M, N, K dimensions, preventing adaptation to workload shapes that benefit from non-default tiling (e.g., small M needs fewer threads in the M-dimension)


## Solution

Introduce problem-size-aware host-side dispatch that selects block dimensions based on M, N, K thresholds (following CUTLASS 3.x grouped GEMM tiling strategy), choosing wider/smaller configs for extreme aspect ratios while falling back to default for balanced sizes

## Key Lessons

- Problem-size-aware tile dispatch adds zero GPU overhead because branch selection runs on the host before kernel launch — the kernel itself is unchanged, only grid/block parameters differ
- For balanced problem sizes (M=128, N=4096, K=4096), the default tile already matches the dispatch selection, so adaptive tiling provides no benefit — the value appears in edge-case aspect ratios like M<=16 where a smaller M-dim block reduces wasted threads
