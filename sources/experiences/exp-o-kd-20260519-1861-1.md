## Solution

Replace hardcoded float-arithmetic zero-point construction with runtime-parameterized IEEE 754 bit-level manipulation: half_uint16(0xe400 | zero) directly ORs the runtime zero value into the known bit pattern for -1024.0f (0xe400 in half-precision), bypassing all float arithmetic and conversion; __hsub(__int2half_rn(-64), __int2half_rn(zero)) replaces float division with integer-based half subtraction

## Key Lessons

- Use IEEE 754 bit-level manipulation (integer OR into known bit patterns like 0xe400 for -1024.0f in half) instead of __float2half_rn for half-precision constant construction when the base value has a known representation — eliminates float subtraction, division, and the full float-to-half conversion pipeline
- Prefer __half2half2(val) over __halves2half2(x, x) for broadcasting a single half into both lanes of a half2 — it communicates broadcast intent explicitly and may avoid redundant register allocation
- Parameterize hardcoded constants as runtime values (e.g., zero-point as a kernel argument) when doing bit-level construction — the integer OR approach makes parameterization essentially free compared to the float-arithmetic equivalent
