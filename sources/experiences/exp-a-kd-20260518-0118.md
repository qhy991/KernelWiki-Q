## Challenge

The before source allocates 4 shared-memory stage buffers of 32x128 half elements each (4*32*128*2=32768 bytes per block), while the after source allocates only 1 buffer (32*128*2=8192 bytes). On A800 (sm_80, 108 SMs), the 4-stage version with 32768 bytes/block allows at most floor(163840/32768


## Solution

Reduce pipeline stages from 4 to 1 by communicating kMaxK=128 to MakeCustomMma so it computes kStages=min(4, ceil(128/128))=1, cutting SMEM per block from 32KB to 8KB and increasing occupancy from 5 to 20 blocks/SM on sm_80 while also eliminating 3 redundant pipeline loop iterations and their __syncthreads() barriers

## Results

- **speedup**: 15.939x (improved)


## Key Lessons

- Reduce CUTLASS pipeline stages to match actual K-dimension iterations when kMaxK fits in a single tile (kMaxK <= kKeysPerBlock) — extra stages waste SMEM proportional to stage count and directly suppress occupancy
- SMEM-driven occupancy limits dominate kernel throughput when per-block SMEM exceeds ~16KB on sm_80 (164KB shared memory / 32 blocks max) — cutting SMEM per block is often more impactful than reducing instruction count
- Pass accurate kMaxK to MakeCustomMma so it can compute kStages=min(maxStages, ceil(kMaxK/kN)) instead of always defaulting to maxStages — this is a compile-time SMEM tuning lever specific to CUTLASS attention kernels
