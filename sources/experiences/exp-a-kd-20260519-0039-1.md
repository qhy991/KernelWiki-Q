## Solution

Fuse both GEMMs into a single CUTLASS B2bGemm kernel where GEMM0's output stays on-chip in the register file (RF residency) and is directly consumed by GEMM1's MMA instructions without ever writing to global memory, eliminating ~20MB of DRAM traffic and one full kernel launch per iteration

## Key Lessons

- Fuse dependent back-to-back GEMMs into a single B2bGemm kernel to keep the intermediate result in the register file rather than round-tripping through global memory — eliminates DRAM traffic proportional to the intermediate tensor size and removes one kernel launch, applying whenever two GEMMs are chained (D0 = A0*B0, D1 = D0*B1)
- When fusing B2b GEMMs, the first GEMM's threadblock/warp tiles must be reduced to fit both operations' working sets on-chip simultaneously (here ThreadblockShape0 shrank from 128×64 to 64×64, WarpShape0 from 64×64 to 16×64) — the tile reduction is a necessary tradeoff for the DRAM traffic savings
- B2b GEMM fusion provides diminishing speedup at larger problem sizes where compute already dominates over memory traffic (1.48x for M=81920 vs 2.15x for smaller M), so the technique is most impactful when the intermediate tensor is large relative to total compute
