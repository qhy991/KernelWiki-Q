## Challenge

Deferred-write software-pipeline pattern holds each thread's 8 half values (16 bytes) in local registers across iterations, attempting to overlap global stores with next-iteration computation, but the deferred data volume is far too small to meaningfully hide memory latency relative to the heavy per-iteration cross-warp reduction workload over 4096 elements


## Solution

Remove the deferred-write pattern entirely: eliminate the register array my_out[], the null-pointer conditional branch, and the post-loop flush, writing output directly to global memory within each iteration — simplifies control flow and reduces register pressure with no measurable performance change

## Key Lessons

- Deferred-write software pipelining only hides latency when the deferred data volume is substantial relative to per-iteration compute — deferring only 16 bytes (8 half values) per thread is negligible when each iteration involves a full 4096-element cross-warp shared-memory reduction
- Eliminating unhelpful software pipelining simplifies the kernel (no null-pointer branch, no register array, no post-loop flush) without performance loss — prefer the simpler immediate-store pattern when deferred writes cannot overlap enough memory traffic to matter
- The latency-hiding benefit of deferred writes scales with the amount of deferred data per thread, not with the total computation per iteration — a small per-thread output footprint makes deferred writes structurally ineffective regardless of how much compute surrounds it
