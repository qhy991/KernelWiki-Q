## Challenge

Scalar FP32 dot product kernel loads one element per thread from global memory via individual 32-bit load instructions, underutilizing available memory bandwidth and instruction throughput for the memory-bound reduction workload


## Solution

Replace scalar float loads with float4 vectorized loads (128-bit transactions) so each thread processes 4 element pairs, reducing total load instructions by 4x — however the proportional thread count reduction (256→64) lowers occupancy from 8 warps to 2 warps, diminishing latency-hiding capability for memory stalls and yielding only neutral net performance

## Key Lessons

- Vectorized float4 loads reduce load instruction count by 4x for memory-bound kernels, but when thread count is reduced proportionally (256→64), the resulting lower occupancy (8 warps→2 warps) reduces latency-hiding capability for memory stalls — for bandwidth-bound workloads where total data volume is unchanged, the net effect can be neutral
- When applying vectorization that reduces thread count, keep total thread count constant by maintaining the same grid dimensions and thread count per block — process more elements per thread without shrinking the block, so occupancy is preserved while instruction count drops
- Cross-warp reduction overhead savings from fewer warps (8→2) are marginal compared to dominant memory access costs in bandwidth-bound kernels — do not rely on reduction savings to justify occupancy reductions
