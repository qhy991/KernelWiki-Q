## Solution

Fuse the two separate kernels into a single advance_step_flashinfer_kernel that reads seq_len once into a register, derives all six output values (token, seq_len, position, slot_mapping, paged_kv_last_page_len, block_table_bound) from that single value and one block_tables lookup, writing all results in one pass to eliminate the second kernel launch overhead and redundant global-memory reads

## Key Lessons

- Fuse lightweight metadata-update kernels that share input data (seq_lens, block_tables) into a single kernel to eliminate redundant global-memory reads and kernel launch overhead — applies whenever two kernels pass data through global memory that could instead stay in registers
- Kernel launch driver overhead (~5-10us per launch) becomes a significant fraction of total execution time for small kernels operating on few queries (32 in this case), making fusion especially impactful for decode-phase operations
- When one kernel writes values that the next kernel immediately re-reads (e.g., seq_lens written by kernel 1 then read by kernel 2), fusing them keeps the intermediate value in a register instead of round-tripping through global memory
