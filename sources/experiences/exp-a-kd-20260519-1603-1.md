## Challenge

Vectorized RMS norm compute function uses hidden_size as the stride multiplier for the vec4_t base pointer, producing incorrect addresses when input_stride differs from hidden_size in padded or strided input layouts


## Solution

Add an explicit input_stride parameter and compute input_token_offset separately using input_stride for the vec4_t base pointer, while keeping token_offset (via hidden_size) for other uses — this correctly handles both contiguous and padded input layouts

## Key Lessons

- Separate input_stride from hidden_size in vectorized kernels — when input_stride != hidden_size (padded KV caches, batched sequences), using hidden_size as the stride produces wrong addresses; when they are equal NVCC compiles both to identical code, so the fix is zero-cost in the contiguous case
- Compute address offsets from stride parameters, not from logical dimensions — input_token_offset = blockIdx.x * input_stride is the correct general formula; token_offset = blockIdx.x * hidden_size only works by coincidence when stride equals hidden_size
