## Solution

The after code attempts to accelerate the streaming copy by (1) using PTX-level L1::no_allocate cache hints on loads and stores to bypass L1, (2) adding software L2 prefetch instructions to hide load latency, and (3) restructuring the two-phase load-then-store loops into a single software-pipelined loop that interleaves next-iteration loads with current-iteration stores. However, this regresses on Ampere because the data was recently written by the fill kernel and is already resident in L2/L1, so bypassing L1 forces reads through the slower L2 path (~200 cycles vs ~30 cycles for L1 hits), and the prefetches add redundant memory traffic.

## Key Lessons

- L1::no_allocate cache hints on loads (ld.global.nc.L1::no_allocate) bypass L1 and force reads through L2 (~200 cycle latency), which is a net regression when input data is recently written and likely L1-resident — only use L1-bypass when L1 pollution from streaming data would evict hotter data that is actually re-read
- Software L2 prefetch (prefetch.global.L2) adds redundant memory traffic and instruction overhead when data is already in L2 from a prior kernel's writes — prefetch only helps when there is a genuine cache miss to hide and sufficient distance between prefetch and consumption
- Two-phase register-buffered copy (load all into registers, then store all) can outperform software-pipelined interleaving on Ampere for small iteration counts (16 iterations) because the simpler control flow and full register reuse amortize pointer arithmetic overhead better than per-iteration conditional prefetch logic
