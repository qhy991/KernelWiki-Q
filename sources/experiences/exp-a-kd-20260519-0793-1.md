## Solution

After code replaces the two-kernel pipeline with a single cooperative multi-block kernel using cudaLaunchCooperativeKernel and cooperative_groups::grid.sync() for inter-block synchronization, fusing counting, alignment, and sorting into one kernel with a four-stage pipeline (local cumsum → cross-block reduction via global atomics/int4 loads → block-size alignment → store results)

## Key Lessons

- Cooperative kernel launches with grid.sync() barriers add synchronization overhead that negates parallelism gains for small-to-medium workloads (8K tokens, 256 experts) — the multi-block reduction via global atomics and int4 transfers costs more than the single-block serialization it replaces
- A simple two-kernel pipeline with a single-block counting stage can outperform a fused cooperative kernel when the workload fits in one block and the second kernel already parallelizes well across many blocks
- cudaLaunchCooperativeKernel requires all blocks to be resident simultaneously (max_blocks_per_sm × num_SMs), constraining scalability and adding launch overhead compared to regular kernel launches
