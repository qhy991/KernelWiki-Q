## Solution

Replace float-arithmetic constant construction with IEEE 754 bit-level manipulation for z1 (half_uint16(0xe400 | zero) directly ORs the runtime zero into the known bit pattern for -1024.0f) and __int2half_rn plus __hsub for z16, then use __half2half2 broadcast intrinsic instead of __halves2half2

## Key Lessons

- Use bit-level IEEE 754 manipulation (integer OR into known bit patterns like 0xe400 for -1024.0f) instead of __float2half_rn for half-precision constant construction when the base value has a known representation — eliminates float subtraction/division and the full float-to-half conversion pipeline
- Prefer __int2half_rn combined with __hsub over __float2half_rn for constructing half constants from integer offsets — the integer-to-half path uses a more compact hardware mapping
- Use __half2half2(val) instead of __halves2half2(x, x) when broadcasting a single half to both lanes — it expresses broadcast intent explicitly and may use a dedicated hardware instruction
