## Challenge

The dequant_kU4B8 function returns a FragB struct (containing half2 data[2]) by value, requiring the compiler to potentially generate a struct copy during return; in heavily templated kernels with high register pressure, the compiler may fail to apply Named Return Value Optimization (NRVO), causing redundant register-to-register moves


## Solution

Change dequant_kU4B8 from return-by-value (FragB struct) to output-pointer (half2* frag_b), eliminating the struct return entirely by writing results directly to a caller-provided half2[2] array, which guarantees register-equivalent addressing without depending on compiler NRVO

## Key Lessons

- Replacing return-by-value with output-pointer for small structs (2×half2) shows no measurable difference in isolated microbenchmarks because NVCC successfully applies NRVO for simple functions — the optimization only matters in complex kernel contexts with heavy template nesting and high register pressure where the compiler cannot consistently apply NRVO
- For small device-side helper functions returning 2–4 element structs, an output-pointer parameter is a defensive coding practice that eliminates reliance on compiler NRVO without introducing overhead, and becomes worthwhile in production kernels with deep template instantiation stacks
