## Challenge

Rotary embedding kernel assumes contiguous head layout (head_size stride), forcing padded tensors to be copied to contiguous buffers via four extra device-to-device copy kernels before the rotary kernel can run, then copied back — doubling memory traffic and adding 4 extra kernel launches


## Solution

Replace hardcoded head_size stride with a configurable int64_t head_stride parameter, allowing the kernel to index directly into padded tensors and eliminating all four copy kernels plus their associated memory traffic

## Key Lessons

- Accept a configurable stride parameter (head_stride) instead of assuming contiguous layout — eliminates the copy-to-contiguous + copy-back pattern whenever tensors have padded/non-contiguous memory layouts, which is common in LLM serving frameworks like vLLM
- Eliminating 4 extra copy kernel launches and 2x memory traffic (read+write per copy) far outweighs the slight cache-unfriendliness of strided access within the compute kernel — the dominant cost was the copy overhead, not the access pattern within the rotary math
