## Challenge

For narrow vec_n_t<half,4> (8 bytes) with read-only VecOp, adding a local copy prevented NVCC from fusing the global load directly into the inline expansion of the callable. NVCC already emits efficient LDG.8/LDG.16 for this narrow width.


## Solution

The local copy forced a separate load-then-copy step, preventing load-use fusion in the short kernel loop. For narrow packs (≤8 bytes), pass v_in[i] directly to allow the compiler to fuse the load into the immediate consumer. The regression confirms there's a minimum pack width threshold for this optimization.

## Results

- **speedup**: 0.872x (14.6% regression from preventing load-use fusion)


## Key Lessons

- There is a minimum vector width threshold for the local-copy optimization: below ~16 bytes, NVCC already emits efficient loads and adding a local copy prevents load-use fusion. Above ~16 bytes and in non-inlinable contexts, it can help. Test at multiple widths before applying broadly.
- Load-use fusion (where LDG feeds directly into an ALU operation without intermediate register copy) is valuable for short kernel loops. Adding explicit locals can block this optimization. When the compiler emits LDG.8/LDG.16 natively, don't intervene.
