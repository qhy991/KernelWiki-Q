## Challenge

Issuing 16 individual cudaMemcpyAsync calls per iteration (8 pairs × key + value) from the CPU side incurs high per-call dispatch overhead that dominates actual transfer time for small 1KB blocks (256 floats), resulting in poor GPU utilization and serialized copies


## Solution

Replace the CPU-side loop of individual cudaMemcpyAsync calls with a single CUDA kernel launch using a 2D grid (blockIdx.x = layer, blockIdx.y = pair) where each thread block copies both key and value cache blocks via a strided loop, eliminating per-copy CPU dispatch overhead and enabling the GPU scheduler to process all pairs concurrently

## Key Lessons

- Replacing a CPU-side loop of many small cudaMemcpyAsync calls with a single kernel launch eliminates per-call dispatch overhead and enables GPU-side parallelism — especially effective when individual copy sizes are small (e.g., 1KB) where API overhead dominates transfer time
- Use a 2D grid (blockIdx.x = layer, blockIdx.y = pair) to map independent copy operations onto thread blocks, allowing the GPU scheduler to process all pairs concurrently rather than sequentially from the CPU
- Pass cache pointers via a device-side int64_t pointer array so the kernel can dereference the correct layer's cache without CPU re-dispatch, consolidating the dispatch from N API calls to a single kernel launch
