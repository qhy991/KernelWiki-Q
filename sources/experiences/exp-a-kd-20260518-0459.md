## Challenge

The after kernel accumulates in double (opmath_t<double>=double), preserving all 53 mantissa bits during the topk sum and scale multiplication. The before kernel truncates each double input to float via static_cast<float>


## Solution

, losing roughly 29 bits of mantissa per element, which explains the before max_error of ~1e-7 vs after max_error of exactly 0. On A800 (sm_80), double arithmetic uses the 64-bit FP64 datapath; for the small problem size

## Results

- **speedup**: 1.011x (neutral)


## Key Lessons

- On A800 (sm_80), double arithmetic uses the 64-bit FP64 datapath; for the small problem size (256 tokens, 512 hidden_dim), both kernels exhibit similar latency since the dominant cost is memory access rather than compute throughput.
- The opmath_t pattern's primary benefit here is precision preservation, not speed.
