## Challenge

On Turing GPUs with 64KB shared memory, the quantized GEMM kernel hardcodes 2 bytes per A-matrix element (FP16 size) regardless of activation type and uses a fixed 4-stage pipeline, causing INT8 activations to over-allocate shared memory and preventing optimal thread configurations from fitting within the shared memory budget


## Solution

Parameterize both element size (is_a_8bit ? 1 : 2) and pipeline depth (stages) so INT8 activations use 1 byte per element with 2 pipeline stages, reducing A-matrix shared memory from 16384 to 4096 bytes (4x) and enabling high-occupancy thread configurations that previously exceeded the 64KB limit

## Key Lessons

- Parameterize shared memory element size based on activation data type (is_a_8bit ? 1 : 2) rather than hardcoding FP16 sizing — INT8 activations need only 1 byte per element, so hardcoding 2 bytes wastes 50% of A-matrix shared memory on architectures with tight shared memory budgets like Turing
- Parameterize pipeline depth instead of hardcoding it — reducing from 4 to 2 stages halves both A-matrix and B-matrix shared memory usage, and on memory-constrained architectures this alone can determine whether optimal thread configurations fit within the shared memory limit
- Per-element footprint reduction and pipeline depth reduction multiply together in shared memory budget calculations — halving element size (2→1 byte) and halving stages (4→2) yields a 4x reduction in the dominant shared memory allocation, potentially unlocking thread configurations with higher occupancy
