## Challenge

Persistent cooperative GEMM kernel assigns whole output tiles to grid-threads with cg::grid_group synchronization between tile iterations, creating significant sync overhead and poor load balancing in the tail wavefront where the last 40 tiles are unevenly spread across 108 SMs


## Solution

StreamK decomposition breaks each tile's K-dimension into independent K-iteration units (256 tiles × 64 K-iters = 16384 fine-grained work items), distributing them across all SMs via strided grid loop with atomicAdd for partial-result accumulation, eliminating per-tile grid synchronization and balancing the tail wavefront

## Key Lessons

- Decompose tile work into fine-grained K-iteration units (tiles × k_iters) instead of whole-tile assignment — multiplies the work-item count by the K-iteration factor, eliminating load imbalance in the tail wavefront when tiles ≫ SMs
- atomicAdd for partial-result accumulation is a low-cost alternative to grid-sync between tile iterations when the computation-to-atomics ratio is high (here K=2048 multiply-adds per atomicAdd), making the synchronization cost negligible
- cg::grid_group sync between persistent kernel tile iterations is a major overhead source — replacing it with a stateless strided grid loop over decomposed work items removes the barrier entirely and enables better SM utilization
