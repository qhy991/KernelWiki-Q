## Challenge

The swish activation kernel processes one float element per thread using scalar load and store, issuing separate 32-bit memory transactions per element and underutilizing available memory bandwidth for a memory-bound elementwise operation


## Solution

Vectorize the kernel with float4 loads/stores so each thread processes 4 elements via a single 128-bit coalesced memory transaction, reducing total thread count by 4x and better saturating the memory bus for this memory-bound elementwise operation

## Key Lessons

- Use float4 vectorized loads/stores in memory-bound elementwise kernels to replace 4 separate 32-bit transactions with one 128-bit coalesced transaction — applies whenever elements are contiguous, type-aligned, and the kernel is memory-bound
- Scale the global thread index by the vector width (idx * 4) and adjust the grid calculation accordingly (divide total elements by BLOCK * 4) when vectorizing an elementwise kernel
