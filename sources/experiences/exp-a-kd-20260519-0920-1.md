## Challenge

Per-token FP8 quantization kernel uses a custom FP8Wrapper class with hand-written fp8e4m3fn_from_fp32_value bit-manipulation conversion, which prevents leveraging native hardware FP8 instructions on GPUs that support them (sm_89+) and adds maintenance burden for the software conversion path


## Solution

Replace the custom FP8Wrapper type with NVCC's native __nv_fp8_e4m3 type as a template parameter DST_DTYPE, using static_cast<DST_DTYPE>(val) for conversion which enables hardware cvt instructions on sm_89+ but invokes NVCC's internal software emulation on sm_80, causing 25.54% regression due to more instructions per element than the hand-tuned bit manipulation

## Key Lessons

- Native __nv_fp8_e4m3 types only accelerate FP8 conversion on sm_89+ (Hopper) where hardware cvt.rna.sat.f32.f8_e4m3 exists; on sm_80 (Ampere), NVCC's software emulation generates more instructions than hand-tuned bit-manipulation, causing regression — always guard FP8 native-type usage with SM version checks
- Hand-tuned IEEE 754 bit-manipulation for FP8 conversion (exponent bias, mantissa rounding, denorm handling) can produce a more compact instruction sequence than compiler-generated software emulation, especially when the custom path is inlined and optimized by NVCC at the function level
- Conversion overhead per element does not fully amortize with larger hidden dimensions — at hidden_dim=4096 the regression is 25.54% vs worse at smaller dimensions, indicating the per-element instruction count difference dominates regardless of loop iteration count
