## Challenge

KV-cache reshape kernel needs to support multiple cache data types (fp16 native, fp8 E4M3, fp8 E5M2) without introducing runtime branching overhead on the existing fp16-only path


## Solution

Generalize the kernel to a triple-template design (scalar_t, cache_t, kv_dt enum) with if constexpr dispatch — the kAuto branch performs identical direct assignment, and the compiler eliminates all other branches at compile time, yielding zero-overhead template parameterization

## Key Lessons

- Use compile-time enum template parameters with if constexpr dispatch to extend kernels for multiple data formats (fp16/fp8) without any runtime cost — the compiler resolves the branch and eliminates dead code, confirmed by exact 1.000x speedup
- Introduce a separate cache_t template parameter distinct from scalar_t so the source and cache types can diverge (e.g., fp16 input → fp8 cache), enabling future quantization paths without touching the existing same-type fast path
