## Solution

Replace float-arithmetic constant construction with IEEE 754 bit-level manipulation: half_uint16(0xe400 | zero) directly ORs the runtime zero into the known bit pattern for -1024.0f, bypassing float arithmetic entirely; __hsub(__int2half_rn(-64), __int2half_rn(zero)) replaces float division with hardware integer-to-half lookup plus half subtract; __half2half2 replaces __halves2half2 for dedicated single-value lane broadcast

## Key Lessons

- Use bit-level IEEE 754 manipulation (integer OR into known bit patterns) instead of __float2half_rn for half-precision constant construction when the base value has a known representation — eliminates float subtraction/division and the full float-to-half conversion pipeline
- Prefer __hsub with __int2half_rn over __float2half_rn(float_expr) when both operands are integers — __int2half_rn maps to hardware/table lookup and avoids the float arithmetic pipeline entirely
- __half2half2(val) is the dedicated broadcast intrinsic for duplicating a single half into both lanes — prefer it over __halves2half2(x, x) for explicit broadcast intent and potentially fewer register moves
