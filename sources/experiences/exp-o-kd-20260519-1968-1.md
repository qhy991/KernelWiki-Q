## Challenge

Int4 weight nibble remapping runs on the CPU via a host-side round-trip: cudaMemcpy copies ~144 MB from device to host over PCIe, a serial CPU loop processes every byte, then another cudaMemcpy copies the result back to the device — two synchronous PCIe transfers plus CPU serialization dominate runtime at ~135 ms


## Solution

Replace the host-side round-trip with an on-device GPU kernel that uses a 256-byte LUT uploaded to __constant__ memory once and uint4 (16-byte) vectorized loads/stores to process all nibble remapping in-place at HBM bandwidth (~900 GB/s), eliminating both PCIe transfers and CPU serialization

## Key Lessons

- Eliminate GPU↔CPU round-trips by moving per-element transformation kernels onto the device — two ~144 MB PCIe transfers at ~12 GB/s cost ~12 ms each, while the same work on-device at ~900 GB/s HBM bandwidth completes in ~1 ms
- For byte-wise lookup-table transformations on GPU, upload a 256-byte LUT to __constant__ memory once and index it directly from the kernel — constant cache serves lookups at register-file latency without consuming shared memory or L1 capacity
- Use uint4 (16-byte) vectorized loads and stores for elementwise byte transformations to coalesce 16 bytes per memory transaction, maximizing HBM throughput and reducing address calculation overhead by 16x compared to single-byte access
