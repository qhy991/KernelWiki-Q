## Challenge

Warp index is broadcast via __shfl_sync(0xffffffff, threadIdx.x / 32, 0) inside the per-tile loop, adding an unnecessary SHFL barrier instruction per iteration to broadcast a value that is already deterministic from threadIdx.x


## Solution

Replace __shfl_sync warp-index broadcast with a __forceinline__ device function canonical_warp_idx() that computes threadIdx.x / 32 via plain integer division, eliminating the per-tile SHFL instruction since the value is already uniform across all threads in a warp

## Key Lessons

- Using __shfl_sync to broadcast a value derived solely from threadIdx.x is unnecessary — each thread can compute the same result independently via plain integer arithmetic, avoiding the SHFL instruction entirely
- Per-iteration shuffle overhead is negligible when the loop body is dominated by global-memory loads (32 loads per tile vs 1 SHFL), so removing __shfl_sync in memory-bound kernels yields at most a few percent improvement within measurement noise
- CUTLASS 3.0 replaced the 2.x __shfl_sync warp_idx broadcast pattern with canonical_warp_idx() (plain threadIdx.x / 32) as part of simplifying the warp-level indexing abstraction — this is a code-clarity improvement rather than a significant performance optimization
