## Solution

Cache the expert_map into shared memory via cooperative strided loading with __syncthreads() barrier, then read from smem_expert_map instead of global memory — but the optimization is neutral because the 256-byte table already resides in L1 cache on sm_80+ with near-zero latency, and the added load loop plus barrier overhead fully offsets any savings

## Key Lessons

- Small read-only lookup tables (≤256 bytes, e.g. 64 int32 entries) already fit in L1 cache on sm_80+ (128KB per SM), so shared memory caching adds overhead without benefit — apply shared memory caching only when the table exceeds L1 capacity or is accessed with heavy bank-conflict patterns
- When a kernel's execution time is dominated by one memory traffic pattern (here: 2×32KB topk_id_ptr read-modify-write per block for 262K elements), optimizing a lightweight secondary lookup that is a negligible fraction of total latency yields no measurable improvement
