## Solution

Introduce double-buffered async copy pipeline (kStage=2): V tile is prefetched to smem via non-blocking cp.async during Q@K^T computation, and next-iteration K tile is prefetched during P@V, using CP_ASYNC_WAIT_GROUP(1) to allow one in-flight group while consuming the other, thereby overlapping gmem transfers with MMA work

## Key Lessons

- Double-buffered async copy pipeline (kStage=2) overlaps gmem loads with computation but requires sufficient compute per tile to hide the transfer latency — if Bc is halved (64→32), the MMA phase is too short to fully overlap, and the pipeline overhead (extra smem partitioning, more sync points) can negate the benefit
- Reducing tile size (Bc) alongside staging increases total tile count and loop overhead (Tc doubles from 32 to 64), which counteracts the latency-hiding benefit of double buffering — always verify that the compute-to-load ratio per tile remains favorable after tile-size adjustments
- CP_ASYNC_WAIT_GROUP(N) allows N groups to remain in flight; using WAIT_GROUP(1) while consuming the ready group enables true pipeline overlap where the next gmem transfer hides behind current MMA work — but this requires that the MMA work duration exceeds the gmem transfer time
