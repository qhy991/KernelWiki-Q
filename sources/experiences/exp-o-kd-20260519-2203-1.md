## Challenge

Single-query cached KV attention dispatch only handles block_size 8 and 16 via explicit template instantiation; requests with block_size=32 fall through to assert(false), preventing use of larger KV cache blocks that reduce block-table indirection overhead for longer sequences


## Solution

Add an `else if (block_size == 32)` branch calling `single_query_cached_kv_attention_launcher<uint16_t, 32>()`, which instantiates the existing kernel template with BLOCK_SIZE=32 where THREAD_GROUP_SIZE=1 so each thread independently handles one token's QK computation without intra-group shuffle reduction

## Key Lessons

- When a kernel template is parameterized by block size and already handles multiple sizes via dispatch branches, extending support to a new block size requires only adding a dispatch branch — the template internals adapt automatically via compile-time constants
- Larger KV cache blocks (BLOCK_SIZE=32) reduce block-table lookup indirection and eliminate intra-group QK shuffle reduction (THREAD_GROUP_SIZE=1), yielding greater speedup on medium-to-long sequences (1.25-1.29x) than short sequences (1.07x)
