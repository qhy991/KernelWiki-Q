## Challenge

When K=0 (k_tile_count=0), the GEMM pipeline kernel unconditionally increments producer_state after the MMA loop, causing pipeline state drift from 0 to 1 with no actual computation performed — in real CUTLASS warp-specialized pipelines this breaks producer_tail/consumer_wait synchronization and accumulator buffer selection


## Solution

Guard both the MMA loop and the pipeline state increment with `if (k_tile_count > 0)`, ensuring that when K=0 the producer_state remains at 0 and no spurious state transitions occur in the warp-specialized pipeline synchronization machinery

## Key Lessons

- Guard all pipeline state mutations (producer_state, barrier signals, accumulator buffer selectors) behind a k_tile_count > 0 check — unconditional increments on zero-iteration paths cause state drift that breaks warp-specialized producer/consumer synchronization in CUTLASS pipelines
- Edge-case K=0 is a correctness hazard, not a performance hazard: both versions produce identical GEMM output (0.0f), but the before version corrupts internal pipeline state that manifests as mis-synchronization in real warp-specialized kernels
- Pipeline state invariants must hold for all valid problem sizes, including degenerate dimensions (M=0, N=0, K=0) — test state variables directly, not just final numerical output, since the numerical result may be accidentally correct while the pipeline state is wrong
