## Challenge

The naive dot product for Float4_ (two float2 pairs) computes mul(a.x,b.x) and mul(a.y,b.y) as independent float2 products, stores two intermediates p0 and p1, then calls add(p0,p1) to combine them — a mul-then-add pattern that requires 4 FP32 muls + 2 FP32 adds + 1 scalar add = 7 total FP32 operations


## Solution

Replace the mul-then-add pattern with FMA-accumulated chaining: compute mul(a.x,b.x) into a single accumulator, then fma(a.y,b.y,acc) folds the second product into it, reducing to 2 FP32 muls + 2 FP32 FMAs + 1 scalar add = 5 equivalent operations with a single accumulation path

## Key Lessons

- FMA-accumulated dot product restructuring yields no measurable benefit when the dot has only 2 operand pairs — the serial FMA dependency chain cancels the instruction count savings, and intra-thread ILP in the naive parallel-mul-then-add pattern provides equivalent throughput
- nvcc -O3 may auto-fuse mul+add sequences into implicit FMAs, so explicit FMA restructuring at the source level provides no additional hardware-level advantage when the compiler already performs the fusion
- FMA chaining benefits grow with the number of accumulation pairs (4+ pairs); for 2-pair dot products the overhead of restructuring outweighs any theoretical instruction reduction
