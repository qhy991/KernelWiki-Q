## Challenge

Fixed CUTLASS 2.x tile configuration (TileShape<128,128,64>, 5 stages) does not adapt to varying GEMM problem shapes, potentially leaving performance on the table for non-square or batched workloads


## Solution

Added shape-aware tile selection logic (select_tile_config) that picks TileShape<64,128,128> with 3 stages for M<=128/N<=8192, but the tile config is only used for printf reporting — the actual cublasGemmEx call remains identical to the before version, making the optimization cosmetic-only since cuBLAS applies its own internal tiling heuristics independently

## Key Lessons

- cuBLAS's CUBLAS_GEMM_DEFAULT_TENSOR_OP applies its own internal tiling heuristics regardless of any CUTLASS-style tile parameters computed in user code — adding tile selection logic without passing it to cuBLAS via a custom algorithm descriptor has no effect on execution
- When evaluating GEMM optimizations, verify that the tile/configuration changes actually reach the kernel launch path — cosmetic config selection that only appears in logging output does not constitute a real optimization
- cuBLAS run-to-run timing variance (from internal heuristic warming or cache effects) can produce apparent speedups of 10-20% that are not reproducible — always validate with multiple seeded runs and statistical testing
