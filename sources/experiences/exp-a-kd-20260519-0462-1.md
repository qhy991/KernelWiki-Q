## Solution

Increase per-thread vectorization to 8 half elements using 128-bit LDST128BITS (float4) loads and __hmul2 for packed half2 multiply-accumulate, reducing threads per block from 128 to 32. However, the resulting single-warp block cannot hide memory latency through warp interleaving, neutralizing the wider-load benefit in this memory-bound workload.

## Key Lessons

- Wider vectorized loads (128-bit LDST128BITS vs 64-bit half2) do not improve throughput when the reduced thread count eliminates warp-level latency hiding — the memory-bound bottleneck shifts from per-thread load width to insufficient warps to overlap loads with compute.
- In memory-bound reductions with fixed total data volume, the number of active warps per block is often more critical than per-thread vectorization width; a single-warp block (32 threads) lacks the interleaving needed to saturate memory bandwidth.
