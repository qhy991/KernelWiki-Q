## Challenge

Scalar half swish kernel processes one element per thread, issuing separate load and store for each element. Attempting half2 vectorization to reduce memory instruction count by processing two elements per thread.


## Solution

Replace scalar half loads/stores with half2 vectorized access via reinterpret_cast, processing two elements per thread and halving the grid dimension to reduce total memory instructions

## Key Lessons

- Half2 vectorization halves the thread count (4096→2048 threads at N=1M); when the workload is already too small to saturate the GPU, reducing parallelism worsens occupancy and the benefit of fewer memory instructions is negated by reduced latency hiding
- Vectorized memory access only pays off when the data volume is large enough (typically N≥4M for half2) to keep sufficient warps active; at small N the kernel launch overhead and underutilization dominate
- Negative-result lesson: always profile vectorization at realistic problem sizes — a transformation that helps at large N can regress at small N due to occupancy collapse
