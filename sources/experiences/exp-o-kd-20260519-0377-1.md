## Solution

Change kOStorageAccFloat32 from 0 to 1 so the accumulator register R_D is stored as FP32 (4 uint32_t per element instead of 2), replacing the __half2float+__fmaf_rn+__float2half_rn conversion chain with a single __fmaf_rn(rescale_factor, existing_float, new_float) per element, eliminating all dtype conversion overhead in both the P@V rescaling and final normalization steps

## Key Lessons

- Store MMA output accumulators in FP32 rather than FP16 when the computation involves repeated online rescaling (e.g., FlashAttention O-update), because each rescale step would otherwise require a __half2float read + __float2half_rn write — the eliminated conversions multiply by the number of K-tile iterations
- FP16 accumulator storage is only beneficial when register pressure is the binding constraint and rescaling frequency is low; for moderate sequence lengths where Tc >= 2, the FP32 path wins despite doubling the accumulator register footprint
