## Solution

Remove the redundant epilogue __syncthreads() (and in real CUTLASS: cp_async_fence + cp_async_wait<0> + __syncthreads) after the mainloop, since the last iteration's __syncthreads() already guarantees all shared memory writes are visible before the epilogue writes to global memory

## Key Lessons

- Removing a post-mainloop __syncthreads() epilogue barrier is safe when the mainloop already ends with __syncthreads(), because that last barrier already ensures all shared memory writes are visible — the epilogue only reads accumulators (registers) and writes to global memory, not shared memory
- An extra __syncthreads() at kernel exit is partially hidden by CUDA's implicit threadblock termination barrier, so microbenchmarks with very short kernels (K=32, ~10us) may not measure the difference — the real benefit appears in larger kernels and when cp_async_wait<0> is also removed
- In CUTLASS-style staged pipelines, when SharedMemoryClear is kZfill (async copy with zero-fill), the cp_async_fence + cp_async_wait<0> epilogue sync is only needed if async copies are still in-flight after the mainloop — if the mainloop's final iteration already drains the pipeline, the epilogue sync is redundant
