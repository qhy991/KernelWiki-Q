## Solution

Track top-2 elements simultaneously per thread using a TopKPair struct (max + secondMax), then reduce across the block in a single BlockReduce call using a custom TopKPairArgMax reducer, halving the number of reduction rounds from k to ceil(k/2)

## Key Lessons

- Reducing BlockReduce iteration count from k to ceil(k/2) by tracking top-2 elements per thread provides no measurable speedup when num_experts << BLOCK_SIZE (64 vs 256 here), because 75% of threads carry idle sentinel values and the shared-memory reduction overhead dominates regardless of iteration count — the optimization targets the wrong bottleneck
- Adding per-thread branching for top-2 tracking (two comparisons + conditional assignment per element) and a custom reducer with multi-branch logic (TopKPairArgMax) can offset the savings from fewer reduction iterations when the reduction payload grows from a single KeyValuePair to a two-element pair struct
