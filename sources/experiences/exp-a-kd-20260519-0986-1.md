## Challenge

Scalar per-thread bf16 kernel uses 256 threads, each processing one element with native __hadd/__hmul accumulation, which works but leaves vectorized memory bandwidth underutilized for the inner reduction over topk=8 rows in a small MoE weighted-sum workload (m=256, row_stride=256)


## Solution

Replaced 256 scalar bf16 threads with 32 vectorized threads using flashinfer::vec_t (8 bf16 elements per thread via cast_load/cast_store) and switched from native bf16 __hadd/__hmul accumulation to float accumulation, but this caused a 13.4% regression because the single-warp occupancy cannot hide memory latency, register pressure rises (8 float accumulators per thread vs 1 bf16), and bf16-to-float conversion overhead negates native HMMA benefits on Ampere

## Key Lessons

- Vectorizing memory access by increasing elements-per-thread while proportionally reducing thread count can destroy occupancy — a single warp (32 threads) leaves the SM unable to overlap memory latency with computation from other warps, causing net slowdown even with higher per-thread throughput
- Native bf16 HMMA instructions (__hadd/__hmul) on Ampere GPUs can outperform explicit bf16-to-float conversion plus float arithmetic for small accumulation counts (topk=8), since the conversion overhead and increased register pressure (8 float accumulators vs 1 bf16) negate vectorization benefits
- For small workloads where the working set fits in L2 cache (256 rows × 256 columns × 2 bytes = 128 KB), scalar access patterns are already effectively cache-friendly, so vectorized loads provide marginal bandwidth gains while the occupancy penalty dominates
