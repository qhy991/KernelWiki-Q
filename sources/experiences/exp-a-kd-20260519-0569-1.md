## Solution

Replace synchronous FLOAT4 B-matrix loads with cp.async (CP_ASYNC_CA) instructions that issue non-blocking global-to-shared copies, commit the async group, then proceed to A loads and computation while B transfers are in flight, synchronizing with CP_ASYNC_WAIT_GROUP(0) only when the data is needed

## Key Lessons

- Use cp.async (CP_ASYNC_CA) for global-to-shared memory copies to enable non-blocking transfers that overlap with subsequent instructions — the commit/wait_group mechanism provides fine-grained synchronization, allowing computation and A-matrix loads to proceed while B transfers are in flight
- In double-buffered tiled GEMM loops, applying cp.async to the B-matrix load path (with commit before compute and wait after) accumulates latency hiding across all K/BK iterations, yielding measurable speedup at larger problem sizes where the per-iteration transfer latency is significant
