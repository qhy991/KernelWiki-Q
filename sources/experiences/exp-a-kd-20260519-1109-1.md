## Challenge

A vec_n_t<half,16> pack (32 bytes) is passed by const reference directly from global memory into an inlined VecOp callable, raising concern that NVCC may scalarize the 16-element load into multiple narrow loads instead of emitting a single wide LD


## Solution

Materialize the global-memory vector pack into a local variable (vin_t src = v_in[i]) before passing it to the inlined VecOp, intending to hint the compiler to issue a single wide vector load rather than multiple scalar loads

## Key Lessons

- Adding a local copy of a wide vector before passing it to an inlined callable can regress performance when NVCC -O3 already traces the full dataflow through inlining and emits a single wide LD — the local copy then becomes a redundant register-to-register move of the entire pack size
- The local-copy materialization pattern is context-dependent: it helps when the compiler scalarizes loads in complex calling contexts, but adds overhead when inlining already exposes enough information for wide load generation
- Do not assume that explicit local copies are always beneficial for vectorized load promotion — verify with profiling (ncu/nsight) rather than applying as a blanket pattern
