## Solution

Replace scalar bf16 load/store with int32 vectorized load/store so each thread handles 2 bf16 elements via a single 32-bit memory transaction, halving the thread count needed for RoPE from 64 to 32 threads

## Key Lessons

- Use int32 reinterpret_cast to load/store 2 adjacent bf16 values in a single 32-bit transaction instead of scalar 16-bit operations — doubles throughput for simple copy kernels and reduces instruction count by half
- Vectorizing RoPE/copy paths is often overlooked in favor of optimizing compute-heavy paths, but these memory-bound copy operations benefit equally from wider memory transactions
