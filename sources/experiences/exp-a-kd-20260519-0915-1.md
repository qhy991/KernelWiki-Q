## Challenge

Per-tensor FP8 quantization kernel uses a software-emulated FP8_TYPE_WRAPPER struct (with fp8e4m3fn_from_fp32_value bit-manipulation constructor) for fp32-to-fp8 conversion, which was expected to be slower than native hardware instructions and was replaced with __nv_fp8_e4m3


## Solution

Replace FP8_TYPE_WRAPPER with __nv_fp8_e4m3 native CUDA type, expecting NVCC to emit PTX cvt.rn.satfinite.e4m3x2.f32 hardware conversion instructions instead of software bit-manipulation; however on SM 8.0 (Ampere) this native type lacks hardware support and falls back to a slower runtime library emulation path

## Key Lessons

- On SM 8.0 (Ampere), __nv_fp8_e4m3 lacks native PTX cvt.rn.satfinite.e4m3x2.f32 support and must emulate through a higher-latency runtime library path — software bit-manipulation conversion (fp8e4m3fn_from_fp32_value) compiles to more efficient ALU instructions on this architecture
- Native FP8 types (__nv_fp8_e4m3) only deliver hardware-accelerated conversion on SM 8.9+ (Ada/Hopper); on older architectures, a hand-rolled integer bit-manipulation path can outperform the compiler's emulation fallback
- When profiling fp8 quantization kernels, always validate on the target SM architecture — the relative performance of software vs. native fp8 conversion inverts between SM 8.0 and SM 8.9+
