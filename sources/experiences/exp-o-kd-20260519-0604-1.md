## Challenge

Resolving future token IDs requires three separate kernel launches (negate+clamp indices, gather from map, conditional write), writing 64KB of intermediate results to global memory (neg_indices and gathered arrays) that are immediately consumed by the next kernel, plus incurring three times the driver launch overhead


## Solution

Fuse all three kernels into a single resolve_future_token_ids_kernel that reads input_ids once, performs the negate-clamp-gather-select logic per element in-place, and writes directly back to input_ids — eliminating 64KB of intermediate global memory traffic and reducing kernel launch overhead from 3 to 1

## Key Lessons

- Fuse sequential kernels where one kernel's output is immediately consumed by the next into a single kernel — eliminates intermediate global memory round-trips and reduces per-launch driver overhead, especially impactful for small data sizes (e.g., 4096 elements) where fixed costs dominate
- When each element's computation is independent and intermediate results are never reused across threads, fusion is always safe and effective — the per-element logic can be composed into a single thread body without shared memory or synchronization concerns
- Removing temporary device allocations (neg_indices, gathered buffers) also reduces peak GPU memory usage and simplifies resource management
