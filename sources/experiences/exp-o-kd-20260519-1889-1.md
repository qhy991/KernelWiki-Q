## Challenge

CPU-side loop issues individual cudaMemcpyAsync calls per src-to-dst block pair for KV cache copying during beam search; each call incurs CPU-side dispatch overhead and the sequential loop cannot exploit GPU parallelism across layers or block pairs


## Solution

Replace the CPU-loop cudaMemcpyAsync approach with a single-launch CUDA kernel (copy_blocks_kernel) using a 2D grid where blockIdx.x maps to layer_idx and blockIdx.y maps to pair_idx, so each thread block handles one (layer, pair) combination and all copies execute in parallel on the GPU without per-copy CPU dispatch overhead

## Key Lessons

- Replacing a CPU-side loop of individual cudaMemcpyAsync calls with a single CUDA kernel launch eliminates per-copy CPU dispatch overhead and enables GPU-side parallelism across all work items — the speedup scales with the number of independent copy operations (7.43x for 8 pairs, 28.09x for 4 layers × 8 pairs)
- A 2D grid (blockIdx.x = layer, blockIdx.y = pair) is an effective pattern for batching independent copy operations across multiple tensor arrays and mapping entries into a single kernel launch, allowing the GPU scheduler to run all blocks concurrently
