## Challenge

Scalar row shuffle kernel copies one half element per thread per iteration, issuing narrow 16-bit load/store transactions that underutilize available memory bandwidth for contiguous row data


## Solution

Replace scalar half-element loads with 128-bit vectorized loads via cutlass::Array<half,8>, packing 8 elements per transaction to improve memory throughput; however, with num_cols=256 and 8 elements per thread the iteration space shrinks to 32, leaving threads 32-255 idle and limiting net gain to ~4%

## Key Lessons

- Vectorized 128-bit loads via cutlass::Array<T,N> widen each memory transaction and reduce loop iterations, but when the total iteration count (num_cols/ELEM_PER_THREAD) falls below blockDim.x, most threads sit idle — the bandwidth gain is offset by poor thread utilization
- When applying vectorized loads to small rows, scale blockDim.x proportionally to the reduced iteration space (e.g., use 32 threads instead of 256 for 32 vector elements) to maintain full warp occupancy
- For half-precision row copies at 512 bytes/row, the data volume per row is already small enough that scalar loads saturate the memory pipeline; vectorization yields measurable gains only when row width or batch size is significantly larger
