## Challenge

Expert offset lookup performs linear scan with direct scalar global memory reads for each offset without cache hints, potentially underutilizing L1 cache persistence for the frequently accessed offset array during per-row expert assignment


## Solution

Replaced direct global memory reads with __ldca cache-all allocation hints to force L1 persistence of expert offsets, and added a runtime branch selecting between linear scan with __ldca for small expert counts and vectorized int4 loads into register arrays for large expert counts, aiming to reduce global memory latency through improved cache behavior

## Key Lessons

- __ldca cache-all hints add per-load PTX instruction overhead that exceeds any L1 persistence benefit when the data already fits naturally in L1 cache — avoid __ldca for small arrays (e.g., 9 uint32_t elements for 8 experts) that are hot in cache from regular access patterns
- Adding a runtime branch inside a hot kernel loop to select between optimization strategies introduces divergent-path overhead and extra register pressure — the cost of branching can negate the benefit when the fallback path is already efficient
- For short-lived linear scans averaging ~4 iterations on tiny lookup tables, simple scalar global reads outperform cache-optimized variants because the baseline hardware caching is sufficient and each extra instruction per iteration compounds across all threads
