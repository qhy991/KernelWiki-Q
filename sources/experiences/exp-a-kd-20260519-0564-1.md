## Challenge

Naive SGEMM kernel computes one output element per thread, reading an entire row of A and column of B directly from global memory across all K iterations, causing massive redundant global memory traffic and making the kernel bottlenecked on global memory bandwidth (~900 GB/s)


## Solution

Apply block tiling with K-dimension slicing: each 32x32 thread block cooperatively loads BMxBK and BKxBN tiles of A and B into shared memory per BK step, then all threads reuse that shared data for the partial dot product — shifting the bottleneck from global memory (~900 GB/s) to shared memory (~19 TB/s)

## Key Lessons

- Block tiling with shared memory reduces global memory accesses by a factor of BM and BN — each tile loaded once from global memory is reused by all threads in the block across BK iterations of the compute loop, amortizing global memory traffic
- Slice the K dimension into BK-sized chunks and load cooperative tiles per step — this transforms the memory access pattern from O(K) global reads per thread to O(K/BK) global reads with BK shared memory reads, shifting the bottleneck from global bandwidth to shared memory bandwidth
- __syncthreads() barriers are required both after loading shared memory tiles and after computing with them to prevent data races between consecutive BK steps
