## Challenge

The before kernel maps each thread to exactly one paged_kv_indices entry via a flat idx/stride decomposition into (row, col), wasting thread capacity when the total element count is small and producing underutilization for large inputs where stride-based row mapping limits coverage


## Solution

Replace the one-element-per-thread idx/stride decomposition with a grid-stride loop where each thread iterates over the entire flattened block_tables array (num_seqs * max_num_blocks_per_seq) using a stride of gridDim.x * blockDim.x, enabling automatic work distribution regardless of problem size

## Key Lessons

- Grid-stride loops provide scalability for variable workloads but show neutral performance when the total element count (num_seqs * stride) is small enough that both approaches execute the same number of memory operations per thread — launch overhead and memory latency dominate for lightweight kernels
- Replacing a hardcoded num_threads kernel parameter with the idiomatic blockDim.x * gridDim.x grid-stride pattern is a correctness and maintainability improvement that scales to arbitrary input sizes, even when current benchmarks show no speedup
