## Challenge

Static __shared__ declarations force NVCC to align each array independently, inserting padding gaps between int16_t and int32_t arrays (e.g., padding the 2048-byte s_lru_slots_out to a 4-byte boundary before placing s_hash_keys), wasting shared memory bytes and worsening bank-conflict patterns during hash-table probe and prefix-sum phases


## Solution

Replace static __shared__ declarations with a single extern __shared__ buffer managed by a SmemLayout struct that packs all int32_t arrays contiguously (1040 bytes) followed by int16_t arrays (2304 bytes), eliminating inter-type alignment padding and producing a more uniform bank-mapping layout that reduces bank-conflict stalls in the buffer-scan and prefix-sum loops

## Key Lessons

- Use extern __shared__ with a typed layout struct (SmemLayout) to pack shared memory arrays contiguously by type — eliminates compiler-inserted alignment padding between arrays of different element sizes, reducing total smem consumption and bank-conflict overlap → applies whenever a kernel uses multiple __shared__ arrays of mixed int32_t/int16_t types
- Separate mixed-type shared memory into two contiguous regions (all int32_t first, then all int16_t) rather than interleaving declarations — the NVCC compiler pads each static __shared__ variable to its natural alignment boundary, so static int16_t followed by static int32_t wastes bytes → applies to all kernels with mixed-width shared arrays
- Pass computed smem size via the third launch argument (<<<grid, block, smem_bytes>>>) when using extern __shared__, allowing the SmemLayout::BYTES constant to centralize size calculation and prevent mismatches between declaration and allocation → applies to all dynamic shared memory kernels
