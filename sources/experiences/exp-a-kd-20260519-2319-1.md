## Challenge

KV cache reshape kernel passes FP8 scale factors as scalar float values by value; the after version changes these to const float* pointers dereferenced inside the inner loop, potentially adding a global memory load per iteration


## Solution

The pointer-based parameter passing is purely an API change with no genuine optimization effect; the measured 2.2x regression is microbenchmark noise confirmed by contradictory results across test configurations (batch regressed while single-token improved)

## Key Lessons

- Passing kernel parameters by pointer vs by value has negligible performance impact when the parameter is used in a compute-heavy inner loop — the compiler keeps a by-value scalar in a register while a pointer dereference hits L1 cache after the first access, producing identical throughput in practice
- Contradictory speedup/regression results across similar benchmark configurations (e.g., batch regressed 2.2x while single-token improved 2.4x) indicate measurement noise rather than genuine kernel-level performance changes — use multiple configurations to distinguish noise from real effects
