## Solution

Introduce a dedicated out_rescale shared-memory array to hold the rescale factor, allowing m_prime to retain its role as the running maximum; the rescale factor is computed into out_rescale before the existing __syncthreads(), so frag_o can safely read it afterward without an additional barrier

## Key Lessons

- Use a dedicated shared-memory variable for the rescale factor instead of repurposing a live variable (m_prime) — this eliminates one __syncthreads() from the hot loop by ensuring the rescale data is written before the existing barrier and read after it
- In microbenchmarks with small data sizes and few iterations, __syncthreads() overhead (~20-30 cycles on SM80) is negligible, but in production fMHA kernels with many key-block iterations the compounding savings from removing a barrier per iteration can be measurable
- Separating the semantic roles of shared variables (running max vs. rescale factor) improves code clarity and enables warp-distributed writes that avoid bank conflicts and reduce the number of threads that need to participate in the write
