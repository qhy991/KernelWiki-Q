## Challenge

MoE token-alignment kernel for 256 experts uses a per-thread 2D token-counts matrix in global memory ((blockDim.x+1)*num_experts*4 ≈ 262KB), causing high-latency global stores on every count increment and requiring cudaMemset to clear the entire buffer each invocation


## Solution

Replace the 262KB global-memory token-counts matrix with warp-partitioned shared_counts[32][8] (1KB, each of 32 warps owns 8 experts via atomicAdd) and local_offsets[256] (1KB) for scatter offsets, totaling ~2KB shared memory — well within device limits — eliminating all global-memory traffic for intermediate counts and the cudaMemset overhead

## Key Lessons

- When a per-thread count matrix (N_experts × blockDim) exceeds shared memory capacity, partition experts across warps (shared_counts[num_warps][experts_per_warp]) and use atomicAdd — this reduces shared memory from O(N×blockDim) to O(N) while maintaining lock-free concurrent counting
- Replacing global-memory intermediate buffers with shared memory eliminates both the per-invocation cudaMemset clearing cost and high-latency global stores, yielding order-of-magnitude speedups when intermediate data is reused within a single kernel launch
- Use shared-memory atomicAdd on local_offsets for scatter-phase rank assignment instead of per-thread offset tracking in global memory — atomicAdd on shared memory is low-latency (~tens of cycles) and avoids the multi-pass prefix-sum required by the per-thread approach
