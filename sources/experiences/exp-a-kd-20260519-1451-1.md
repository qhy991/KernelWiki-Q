## Challenge

MoE block-alignment kernel stores tokens_cnts and cumsum arrays in global memory because the int32_t shared memory requirement (264KB) exceeds the A800's 164KB per-block limit, causing every token-count and prefix-sum access to incur ~400-800 cycle global memory latency


## Solution

Reduce token count type from int32_t to uint16_t and relocate both tokens_cnts and cumsum from global memory into shared memory via extern __shared__ with a compact layout (cumsum at base, uint16_t tokens_cnts at offset), cutting shared memory from 264KB to 133KB and fitting within the 164KB per-block limit

## Key Lessons

- Downcast intermediate count arrays (e.g., int32_t → uint16_t) to halve shared memory footprint when count values are bounded by problem size — this alone can shift a kernel from global-memory fallback to shared-memory operation with ~20x latency reduction per access
- Use a compact shared memory layout (widely-used cumsum at base with narrower type overlay after) to fit heterogeneous data structures within device shared memory limits, combined with cudaFuncAttributeMaxDynamicSharedMemorySize to request beyond the default 48KB
- When shared memory is too large with int32_t counts, verify the maximum possible count value per thread-expert pair; if it fits in uint16 (≤65535), the type reduction is lossless and the full prefix-sum/cumsum pipeline remains correct with only narrow→wide casts at arithmetic boundaries
