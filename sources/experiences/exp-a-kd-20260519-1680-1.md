## Challenge

Multi-pass radix selection top-K kernel allocates 128KB shared memory per block (kSmem=32*1024*sizeof(uint32_t)), which saturates the 128KB per-SM shared memory capacity on GPUs like A100, limiting occupancy to only 1 block per SM and preventing latency hiding through concurrent warps


## Solution

Reduce shared memory allocation from 128KB to 32KB (kSmem=8*1024*sizeof(uint32_t)), leveraging the existing ping-pong double buffering scheme (alternating src_buffer/dst_buffer across passes) to maintain correctness with smaller per-pass buffer capacity, while allowing up to 4 blocks per SM for dramatically higher occupancy

## Key Lessons

- Reducing shared memory footprint to enable multiple blocks per SM can yield net speedups even when per-block buffer capacity shrinks — the latency hiding from higher occupancy outweighs the cost of processing fewer buffered items per radix pass
- Ping-pong double buffering (alternating src/dst buffers across passes) allows a multi-pass radix algorithm to work correctly with smaller per-buffer capacity by reusing the same shared memory region across successive refinement passes
- On GPUs with 128KB shared memory per SM (e.g. A100), 128KB per block limits to 1 block/SM; shrinking to 32KB allows 4 blocks/SM — a 4x occupancy increase that directly translates to better warp-level parallelism
