## Solution

Split the 320 total threads into 256 dedicated fill_threads that initialize sorted_token_ids with stride=256 (fewer iterations per thread) and 64 working threads that simultaneously perform the shared-memory counting phase, overlapping the previously-serial init with computation via barrier synchronization so global-memory writes no longer block counting

## Key Lessons

- When a single-block kernel has a large global-memory initialization phase followed by a shared-memory computation phase, dedicate extra threads to init and overlap it with computation via __syncthreads barriers — the init threads finish and wait while working threads proceed, eliminating serial overhead
- Increasing total threads for init (e.g., 256 fill_threads with stride=256 vs 64 threads with stride=64) reduces per-thread iteration count and improves global-memory throughput through wider parallelism within the single block
- Use a compile-time template parameter (int32_t fill_threads) for the thread-role split threshold so the compiler can eliminate the branch and optimize each path independently
