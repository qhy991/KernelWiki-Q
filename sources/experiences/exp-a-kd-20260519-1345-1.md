## Challenge

When the input tensor has a non-contiguous stride (stride=512 vs hidden_size=256), a separate copy kernel must be launched to compact the data into a contiguous buffer before the vectorized fused_add_rms_norm kernel can process it, adding a full kernel launch and 64KB of redundant global memory read+write traffic


## Solution

Eliminate the separate contiguous-copy kernel by adding an input_stride parameter to the main kernel and computing strided_id = blockIdx.x * vec_input_stride + idx for all input loads and stores, while residual/weight indexing remains contiguous — the 128-bit vector loads at stride-512 offsets still hit contiguous cache-line segments within each row

## Key Lessons

- Add stride parameters to vectorized kernels instead of forcing a separate contiguous-copy kernel — the stride-aware indexing adds only one multiply per load/store while eliminating an entire kernel launch and its redundant data movement
- 128-bit vector loads (e.g., HalfVec8 = 4×__half2) at stride-aligned offsets still hit contiguous cache-line segments within each row, so vectorized memory throughput is preserved even with non-unit strides
- Separate copy-then-compute patterns (simulating .contiguous()) are a common hidden cost when tensors have non-contiguous memory layouts — fusing the stride logic into the compute kernel avoids the extra pass over global memory
