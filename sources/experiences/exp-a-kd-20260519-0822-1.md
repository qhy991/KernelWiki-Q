## Challenge

Warp-cooperative memory copy issues one 64-bit load+store per lane per iteration using scalar ld.global.nc.b64 / st.global.cg.b64 PTX instructions, requiring all 32 lanes to each issue separate 8-byte memory transactions to copy a 256-byte item


## Solution

Replace scalar 64-bit PTX load/store with vectorized 128-bit .v2.b64 variants that load and store two uint64_t values per instruction, halving the number of memory instructions from 32 to 16 per warp per item, with a tail-handling branch for sizes not divisible by 16 bytes

## Key Lessons

- Vectorizing PTX memory instructions from .b64 to .v2.b64 halves instruction count but does not guarantee speedup when hardware coalescing already merges adjacent per-lane 64-bit accesses into 128-bit transactions on Ampere — the memory bus sees the same total bytes either way
- When item_size_bytes is a multiple of WARP_SIZE × element_size, all 32 lanes are already fully utilized with scalar accesses; vectorization helps more when the loop has multiple iterations per lane or when fewer lanes are active per iteration
- Always include a tail-handling path when switching to wider vectorized accesses to maintain correctness for sizes not divisible by the new vector width (16 bytes for .v2.b64)
