## Solution

Restructure thread mapping from strided scalar loop (512 threads iterating all elements) to per-head-block assignment (num_heads*head_size/x=128 threads, each handling one x-element block), using vectorize_with_alignment<8> for contiguous key writes producing 128-bit vector stores and an unrolled scalar loop for strided value writes, but the reduced thread count fails to hide memory latency for fp16 elements

## Key Lessons

- Reducing thread count from 512 to num_heads*(head_size/x) while adding vectorized stores can regress fp16 kernels because the original high-occupancy scalar approach already hides 2-byte store latency effectively — fewer threads cannot overlap memory latency as well even with wider per-thread transactions
- Vectorized 128-bit stores for fp16 (8x2B=16 bytes per transaction) have identical byte width to fp32 vectorized stores (4x4B=16 bytes) but concentrate the same total work into fewer threads, making fp16 more sensitive to occupancy-loss regression than fp16-to-vectorize
- When per-thread work is short (only x=8 elements per thread), the kernel becomes latency-bound rather than throughput-bound — increasing vector width cannot compensate because the bottleneck is memory latency hiding, not memory bandwidth
