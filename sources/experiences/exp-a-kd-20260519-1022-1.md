## Challenge

Host-side runtime cc_major detection via cudaGetDeviceProperties and a ternary branch selects between 16-byte and 32-byte vector kernel template instantiations at every kernel launch, incurring unnecessary host-side branching and potentially instantiating dead template specializations


## Solution

Replace runtime cc_major detection and ternary dispatch with a compile-time constexpr int kMaxVecBytes constant, allowing the compiler to resolve the vector size at compile time and instantiate only the needed template specialization, eliminating the host-side branch entirely

## Key Lessons

- Replacing runtime cc_major detection with compile-time constexpr for architecture-dependent constants eliminates dead template instantiations and host-side branches, but yields no measurable runtime improvement when the runtime path already resolves to the same value on the target hardware (both paths select 16-byte vectors on sm_80)
- Compile-time constexpr is architecturally sound for code hygiene (smaller binary, clearer intent) even when runtime latency is unchanged — it prevents accidental instantiation of unused template specializations and removes cudaGetDeviceProperties overhead from the launch path
