## Challenge

Replacing custom inline FP8 conversion (fp8e4m3fn_from_fp32_value bit-manipulation) with __nv_fp8_e4m3 native type causes 2.8x performance regression on sm_80, because NVCC's internal software emulation for __nv_fp8_e4m3 is heavier than the hand-optimized compact integer bit-manipulation path


## Solution

The analysis reveals that __nv_fp8_e4m3 with static_cast only benefits GPUs with native FP8 hardware (sm_89 Ada Lovelace or sm_90 Hopper) where it emits a single cvt.rna.sat PTX instruction; on sm_80 without hardware FP8, NVCC's internal software emulation is heavier than the custom compact bit-manipulation, making the hand-optimized path strictly superior

## Key Lessons

- On GPUs without native FP8 hardware (sm_80 and below), custom inline bit-manipulation FP8 conversion (fp8e4m3fn_from_fp32_value) outperforms NVCC's __nv_fp8_e4m3 software emulation — use __nv_fp8_e4m3 only on sm_89+ where it maps to a single hardware cvt.rna.sat instruction
- Per-element conversion overhead in quantization kernels compounds with hidden dimension size — a small per-element cost increase amplifies linearly across large token dimensions (e.g., 4096 elements per token)
