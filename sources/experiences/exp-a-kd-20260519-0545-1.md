## Solution

Use int4 vectorized loads (128-bit) to read 4 integers per thread in a single memory transaction, reducing global read transactions from 1M to 256K; however the kernel remains atomics-bound since all 1M atomicAdd operations on the 257-bin histogram still serialize, making the read-bandwidth improvement negligible at this scale

## Key Lessons

- Vectorizing loads with int4 reduces global memory read transactions by 4x, but when the kernel bottleneck is atomic contention on output bins rather than input memory bandwidth, the optimization has no measurable latency impact — profile before optimizing to identify the true bottleneck
- Histogram kernels with few bins (e.g., 257) and many elements (~4K updates per bin) are atomics-bound; effective optimization requires reducing atomic contention (e.g., per-block private histograms, warp aggregation, or shared-memory staging) rather than just improving load throughput
- Reducing threads per block from 256 to 64 while keeping the same total elements-per-block (256) changes occupancy characteristics but does not compensate when the dominant cost is serialized atomic operations on a small number of histogram bins
