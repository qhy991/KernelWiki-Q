## Challenge

Both the before and after kernels execute identical scalar shared-memory stores on SM80 hardware. The after kernel's SM90_U32x4_STSM_N::copy() method takes the #else fallback branch at compile time because __CUDA_ARCH__ < 900, wri


## Solution

Replace four scalar st.shared instructions with the SM90 stmatrix.sync.aligned.x4.m8n8.shared.b16 PTX instruction via a wrapper struct, which issues a single warp-cooperative matrix store that avoids bank conflicts on SM90+ hardware, with scalar fallback for pre-SM90 architectures

## Results

- **speedup**: 0.998x (neutral)


## Key Lessons

- stmatrix.sync.aligned.x4.m8n8.shared.b16 is an SM90-only PTX instruction that requires hardware support to issue a warp-cooperative matrix store — on pre-SM90 GPUs the #else fallback compiles to identical scalar stores, producing zero performance difference
- Guard architecture-specific intrinsics with #if defined(__CUDA_ARCH__) && (__CUDA_ARCH__ >= target) and provide a correct scalar fallback so code remains portable and functionally correct across GPU generations
- Warp-cooperative matrix stores (stmatrix) can eliminate shared-memory bank conflicts from column-stride scalar access patterns, but the benefit only materializes on hardware that natively supports the instruction
