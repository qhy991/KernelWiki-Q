## Challenge

Per-token-group quantization kernel is hardcoded to FP8_TYPE (c10::Float8_e4m3fn), preventing reuse of its efficient vectorized load + warp-shuffle reduction pattern for int8 quantization, which had no optimized CUDA implementation


## Solution

Replace the hardcoded FP8_TYPE alias with a DST_DTYPE template parameter on the kernel, renaming fp8_min/fp8_max bounds to min_8bit/max_8bit and converting GROUPS_PER_BLOCK from a compile-time constant to a runtime parameter, so the same vectorized warp-cooperative kernel handles both int8_t and fp8 quantization through a single dispatch

## Key Lessons

- Parameterize the output dtype as a template parameter rather than hardcoding a type alias — allows a single optimized kernel (vectorized loads, warp-shuffle reduction, two-pass quantize+store) to serve multiple quantization targets (fp8, int8) without code duplication
- Move compile-time constants (GROUPS_PER_BLOCK) to runtime kernel parameters when generalizing a kernel across data types — different output types may benefit from different tiling factors, and runtime dispatch is cheaper than maintaining multiple template instantiations
- Renaming domain-specific bounds (fp8_min/fp8_max → min_8bit/max_8bit) signals the generalized contract and avoids confusion when the same clamp logic applies to multiple 8-bit output types
