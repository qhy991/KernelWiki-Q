## Challenge

MoE top-k selection and weight renormalization are split across two separate kernel launches: topkKernel writes unnormalized weights to global memory, then renormKernel reads them back (64KB read + 64KB write for 8192×2 floats), computes row-wise sums via atomicAdd into shared memory, and rewrites normalized weights — the separate kernel launch overhead and redundant global memory traffic dominate cost for small k=2


## Solution

Fuse topk selection and renormalize into a single fusedTopkRenormKernel: accumulate selected_sum in a shared variable during the iterative argmax loop (one addition per selected weight by thread 0), then divide all k weights by selected_sum in a short serial epilogue — eliminating the extra global memory round-trip and one full kernel launch

## Key Lessons

- Fuse a lightweight post-processing kernel (renormalize, bias-add, etc.) into the preceding compute kernel when the post-process data volume is small relative to launch overhead — the eliminated kernel launch and avoided global memory round-trip dwarf any extra register/shared-memory cost
- Accumulate a running sum in shared memory during an iterative selection loop (argmax/topk) to avoid a separate reduction pass — works when the selection loop already uses shared memory for coordination and the per-iteration cost of one float addition is negligible
- When k is small (k<=4), serial normalization by a single thread (threadIdx.x==0) is faster than a parallel reduction + parallel divide because the synchronization and atomic overhead of the parallel path exceeds the cost of k serial divides
