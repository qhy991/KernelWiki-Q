## Challenge

Similar to the large_data case, the before code launches 4096 blocks with all threads calling atomicAdd on one output address. The 4096 blocks produce ~2M atomic operations that serialize on the same memory location


## Solution

Switch accumulator storage from FP16 (2 regs/element) to FP32 (4 regs/element) by setting kOStorageAccFloat32=1, eliminating all per-iteration __half2float/__float2half_rn conversions and performing direct __fmaf_rn on float values, with a single float-to-half conversion deferred to the final output store

## Results

- **speedup**: 96.532x (improved)


## Key Lessons

- Use FP32 accumulator storage for online-softmax attention output accumulators instead of FP16 — the per-iteration __half2float/__float2half_rn conversion overhead compounds across all K-tile iterations, while the extra register cost (2x) is amortized by eliminating conversion instructions
- Defer the single FP32-to-FP16 conversion to the final output store rather than converting at every rescaling step — this trades register pressure for instruction count reduction, and is beneficial when the number of K-tile iterations exceeds the register overhead ratio
