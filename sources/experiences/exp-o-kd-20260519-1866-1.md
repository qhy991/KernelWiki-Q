## Solution

Round m up to the next multiple of 128 via computeEffectiveRows(m) before computing the grid dimension, ensuring at least 128 blocks are launched for small m values; the kernel's grid-stride loop distributes the extra padding rows harmlessly across blocks (out-of-range threads write minimal SF zeros)

## Key Lessons

- When a kernel uses grid-stride loops (stride by gridDim.x), padding the grid launch count via round_up(m, TILE_SIZE) is safe — threads assigned to rows beyond m perform only trivial padding work, but the extra blocks fill SMs that would otherwise sit idle
- Small-batch grid underutilization is a common oversight: always check whether grid_dim = min(m, max_blocks) can produce fewer blocks than the SM count when m is small, and consider rounding up m to a multiple of a tile size
