## Solution

Replace B smem padding (B_PAD=8) with XOR-permuted swizzle addressing (swizzle_permuted_B_j) that scatters column offsets across all 32 banks without any padding, keeping B rows compact at exactly BK=16 half-values to eliminate structural bank conflicts and double effective smem data density for the async copy pipeline

## Key Lessons

- XOR-permuted swizzle addressing eliminates shared memory bank conflicts without padding by scattering column offsets across all 32 banks — applies whenever tile row width in half-values is not co-prime with 32
- Padding-based bank-conflict avoidance wastes shared memory proportionally to the pad width; swizzle-based approaches achieve identical conflict elimination at zero memory overhead, critical when smem budget limits occupancy or pipeline depth
- Shared memory data density directly impacts async copy + ldmatrix pipeline throughput — 50% smem waste per B stage halves effective tile capacity and starves the compute pipeline at large problem sizes
