## Challenge

Naive RMSNorm kernel launches one 32-thread block per row (16384 blocks for 16384 tokens), far exceeding the GPU's concurrent block capacity (~6912 on A800), causing serial block rotation, frequent block setup/teardown overhead, and L2 cache thrashing as each block's tiny 128-byte row data evicts the previous block's data


## Solution

Replace one-block-per-row launch with occupancy-optimal grid sizing (cudaOccupancyMaxActiveBlocksPerMultiprocessor to compute max concurrent blocks) combined with a grid-strided loop (row += gridDim.x) so each block processes multiple rows sequentially while staying resident on its SM, eliminating scheduling overhead and improving L2 cache locality

## Key Lessons

- Use occupancy-based grid sizing (cudaOccupancyMaxActiveBlocksPerMultiprocessor) instead of launching one block per work item when block count far exceeds SM capacity — over-subscription causes serial scheduling overhead that dominates lightweight kernels
- Grid-strided loops (for row = blockIdx.x; row < N; row += gridDim.x) keep blocks resident on SMs and amortize block setup/teardown cost across multiple work items, especially beneficial for small-dimension kernels where per-row computation is lightweight
- For lightweight kernels with small per-block data footprints (e.g., kDim=128 in fp16 = 256 bytes), occupancy-optimal launches improve L2 cache locality by keeping sequentially-accessed rows within SM-local cache partitions rather than thrashing across SM rotations
