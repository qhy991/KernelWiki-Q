## Solution

Replace the hardcoded pipe_stages=4 constant with a runtime stages parameter set to 2 for Turing (sm75), threading it through get_scales_cache_size and get_kernel_cache_size so that all shared memory buffers (sh_a, sh_b, sh_s, sh_g_idx) are sized proportionally to the actual pipeline depth required by the architecture

## Key Lessons

- Hardcoded pipeline depth constants waste shared memory on architectures that support fewer stages — making stages a runtime parameter tuned per-SM-version reduces smem footprint without changing the computation logic
- Shared memory budget is a direct occupancy limiter on GPUs; halving the pipeline stages from 4→2 on Turing reduces total smem by ~1.78x, which can enable more concurrent threadblocks and higher occupancy
- When shared memory sizing depends on pipeline depth (sh_a, sh_b, sh_s, sh_g_idx all scale linearly with stages), the parameter must be threaded through all helper functions to avoid stale global references
