## Challenge

Matrix transpose kernel uses hardcoded stride values (K for input, M for output) derived from layout permutation, which assumes row-major layout and cannot correctly handle non-standard strides that arise in CUTLASS sparse GEMM C^T = B^T * A^T computations where the transposed tensor may have non-trivial stride ordering


## Solution

Replace hardcoded stride constants with explicit int64_t stride parameters (input_stride, output_stride) passed from the caller's tensor metadata, enabling correct addressing for non-standard or transposed layouts in CUTLASS sparse GEMM dispatch while producing identical SASS code when strides happen to equal the compile-time constants

## Key Lessons

- Use explicit stride parameters from tensor metadata instead of hardcoded dimension-derived values — the compiler constant-folds them to identical SASS when they match, but the parameterized form correctly handles non-standard layouts (e.g., transposed tensors in sparse GEMM) that layout permutation cannot
- Layout permutation (permute_layout<1,0,2>) vs direct Stride construction is a CUTLASS template dispatch concern affecting stride interpretation correctness, not a runtime memory access pattern change — the real benefit is correctness for non-trivial tensor layouts, not performance
