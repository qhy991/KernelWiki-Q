## Solution

Fuse all three operations into a single resolve_future_token_ids_kernel that reads each element once, checks the sign condition, negates and clamps in-register, performs the future_map lookup, and writes the result in-place — eliminating two intermediate buffers and two extra global memory round-trips, and reducing three kernel launches to one

## Key Lessons

- Fuse sequential element-wise kernels that pass data through intermediate global memory buffers into a single kernel — each intermediate buffer adds one global write + one global read per element, and each extra kernel launch adds ~2-5us of driver overhead that dominates runtime for small problem sizes
- For small tensor operations (e.g. 256 elements), kernel launch overhead can exceed actual compute time — multi-kernel pipelines suffer disproportionately, making single-pass fusion especially valuable at small scales
- In-place output (writing directly back to input_ids instead of a separate buffer) eliminates one allocation and one global memory round-trip when the input is not needed after the operation
