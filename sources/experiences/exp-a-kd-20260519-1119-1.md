## Solution

Replace raw m with effectiveRows=round_up(m,128) in the grid size calculation, distributing 128 virtual rows across up to 128 concurrent blocks on many SMs; the existing grid-stride loop ensures each block handles ~1 row, with extra blocks skipping real work but sharing the padding workload

## Key Lessons

- For kernels that combine grid-stride loops over real rows with padding-row work, use round_up(numRows, tile_size) as grid size instead of raw numRows — this launches enough blocks to saturate SMs even for tiny batch sizes, yielding up to ~5x speedup
- Grid-stride loops (rowIdx += gridDim.x) naturally tolerate excess blocks without correctness issues: blocks whose starting index exceeds numRows skip the main computation loop while still sharing the padding workload, so launching more blocks than real rows is safe
