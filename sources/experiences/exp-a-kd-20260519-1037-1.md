## Challenge

Scalar int32 fill kernel writes one element per store instruction to the sorted_token_ids buffer, issuing 4x more memory transactions than necessary for a simple sentinel fill operation


## Solution

Replace scalar int32 stores with int4 vectorized stores: pack the sentinel into a 128-bit Vec (4 × int32) and write four elements per store instruction, reducing total memory transactions by 4x

## Key Lessons

- Vectorized int4 stores reduce memory transaction count 4x for fill operations, but the speedup is only measurable when the memory write time dominates kernel execution — for small buffers (≤64K elements), kernel launch overhead and cudaEvent synchronization noise dominate the measured latency
- reinterpret_cast<Vec*>(ptr) enables vectorized access to a scalar-typed buffer without requiring allocation changes — applies whenever the buffer is 16-byte aligned (guaranteed by cudaMalloc)
