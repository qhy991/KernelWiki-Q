## Solution

Eliminate the separate transpose kernel by directly indexing the original scale tensor with row-major layout (bn * nbk + bk) inside the block-scale kernel, leveraging SM90 CUTLASS GMMA major-mode parameters that make the transpose unnecessary

## Key Lessons

- When a small auxiliary tensor (e.g., block-scale factors) requires transposition for layout compatibility, prefer adjusting the indexing formula to match the original layout instead of launching a separate transpose kernel — eliminates an extra kernel launch, a temporary buffer, and simplifies the pipeline
- The real-world benefit of eliminating a transpose on a tiny tensor (4KB) is not measurable kernel-level speedup (~2% noise) but reduced host-side memory allocation overhead during model loading and simpler code paths — optimize for engineering clarity and loading cost, not just inference microseconds
- SM90 CUTLASS Sm90BlockwiseScaleConfig with explicit GMMA major-mode parameters (cute::GMMA::Major::MN, cute::GMMA::Major::K) enables the consumer to handle both MN-major and K-major scale layouts without requiring a host-side .T.contiguous() transpose
