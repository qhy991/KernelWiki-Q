## Solution

Parameterize warpReduceSum with a numLanes template to reduce shuffle rounds to log2(activeLanes), and refactor blockReduceSum with a maxBlockSize template and _nextPow2 helper so the final reduction uses only the necessary lane count (e.g., 16 lanes for 16 warps → 4 rounds instead of 5)

## Key Lessons

- Parameterizing warpReduceSum with numLanes to use only log2(activeWarps) shuffle rounds instead of always log2(32) is architecturally correct but the single-instruction saving is below the noise floor when the kernel is dominated by global memory access and shared-memory barriers
- Compile-time template parameters (numLanes, maxBlockSize) combined with static_assert and if constexpr enable zero-overhead lane narrowing — the compiler unrolls only the needed shuffle rounds without runtime branching
- Reducing shared memory from WARP_SIZE to maxActiveLanes (16 vs 32 elements) saves a negligible amount of shared memory; the real benefit is structural correctness and enabling the compile-time numLanes specialization
