## Challenge

FP8 quantization dynamically computes the per-tensor scale on GPU via a dedicated segmented_max_reduction kernel with 4KB shared memory, a 1024-thread halving reduction tree, and atomicMaxFloat merges, requiring a full extra kernel launch and data traversal before the quant kernel can run


## Solution

Eliminate the GPU-side segmented_max_reduction kernel entirely by pre-computing the per-tensor max-abs scale on the CPU host from the already-available input buffer, uploading the static scale once, and launching only the quant kernel — removing one full kernel launch, 4KB shared memory allocation, and an entire data traversal

## Key Lessons

- When input data is already on the host (or host-visible), pre-computing scalar reductions like per-tensor max-abs on the CPU avoids an entire GPU kernel launch, shared-memory allocation, and reduction tree — the host-side sequential scan is negligible compared to kernel launch overhead for small tensors
- For small tensor sizes (e.g., 4K elements), kernel launch overhead dominates execution time; eliminating even one kernel launch can yield near-2x speedups, making static/pre-computed parameters strongly preferable over dynamic GPU-side computation
- Eliminating shared-memory reductions (cache[1024] + __syncthreads barriers + atomicMaxFloat merges) is especially beneficial at small scales where the reduction overhead is amortized poorly across few elements
