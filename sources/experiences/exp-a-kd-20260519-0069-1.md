## Challenge

The GEMM epilogue processes 4 tile fragments through shared memory in separate iterations, each requiring a load-sync-store-sync cycle totaling 8 __syncthreads() barriers over a small 8.7KB shared memory buffer


## Solution

Batch all 4 fragments into a single 34.8KB shared memory buffer (4x larger) to collapse the 8 __syncthreads() barriers into 1, writing all fragments before one sync then reading all fragments after; however the 34.8KB per-block allocation severely reduces occupancy on A800 (164KB shared/SM), causing a net 3.7x regression for this lightweight kernel

## Key Lessons

- Batching fragment iterations to reduce __syncthreads() count can cause severe occupancy loss when the resulting per-block shared memory allocation approaches a large fraction of per-SM capacity — the occupancy reduction (4x fewer concurrent blocks on A800 with 164KB/SM) can dominate the barrier savings
- Lightweight kernels with low compute density per tile (scalar element-wise copy without vectorized loads/stores) are especially vulnerable to occupancy-driven regression from larger shared memory allocations — the barrier reduction benefit scales with work per barrier, not barrier count alone
- In real CUTLASS epilogues, the accumulator-to-smem path uses warp-tile iterators with vectorized loads and type conversions, making work density much higher per fragment; a microbenchmark with simple scalar indexing lacks that density and may mispredict the tradeoff
