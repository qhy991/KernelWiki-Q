## Challenge

The MoE token-sorting kernel for 256 experts uses a per-thread global-memory counts matrix of (blockDim.x+1)*256 int32 entries (~262KB), exceeding shared-memory capacity and forcing all counting, prefix-sum, and sorted-placement operations through high-latency global memory, plus requiring a cudaMemset before each kernel invocation


## Solution

Replace the large per-thread global-memory counts matrix with warp-partitioned shared_counts[32][8] (1KB) plus local_offsets[256] (1KB) in shared memory, where each warp owns 8 experts and counts via atomicAdd on shared memory, and sorted tokens are placed via atomicAdd on local_offsets, eliminating all global-memory traffic for counting and sorting as well as the cudaMemset clearing cost

## Key Lessons

- Partition data by warp instead of per-thread to compress working sets into shared memory — reduces footprint by warpSize factor — applies when each thread tracks independent counts that can be merged via atomics
- atomicAdd on shared memory is far cheaper than global-memory writes for counting and scatter patterns — use warp-partitioned shared atomics when the bin counts fit in shared memory after repartitioning
- Moving working buffers from global to shared memory eliminates both global-memory access latency and the cudaMemset clearing cost before each kernel invocation — applies when the buffer fits in shared memory after restructuring
