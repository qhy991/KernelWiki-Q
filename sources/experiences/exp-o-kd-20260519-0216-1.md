## Solution

Add !params.fmha_params.is_fused_reduction to the guard condition so the separate reduction kernel is only launched when fused reduction is not active, eliminating both the double-reduction correctness bug and the redundant kernel launch overhead

## Key Lessons

- When a kernel supports a fused reduction path (atomic accumulation inside the main kernel), the separate reduction kernel launch must be guarded by !is_fused_reduction — omitting this guard causes double reduction and incorrect output, not just wasted work
- Fused reduction modes change the correctness contract of downstream kernels: code that was correct for the non-fused path (where a separate reduction pass is required) becomes incorrect when fusion is enabled, so guard conditions must reflect the active reduction strategy
