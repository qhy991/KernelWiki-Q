## Challenge

Top-K selection and per-row offset addition are split across two separate kernel launches: topk_select_kernel writes raw indices to global memory, then add_offset_kernel reads them back, adds offsets, and writes final indices, incurring an extra kernel launch overhead and a redundant 32KB global-memory round-trip for the intermediate indices


## Solution

Fuse topk selection and offset addition into a single kernel (topk_fused_ragged_kernel) that routes scatter writes through shared memory (s_indices[TopK]) instead of global memory, adds the per-row offset inline before writing to global memory, and eliminates the second kernel launch entirely

## Key Lessons

- Fusing a scatter-write kernel with a trivial post-processing kernel (offset addition) into a single launch eliminates one kernel launch overhead and a full global-memory round-trip — applies whenever a producer kernel writes small intermediate results that a consumer kernel immediately reads and transforms
- Route scatter writes through shared memory instead of global memory when the intermediate data (e.g., TopK indices) fits in shared memory and is consumed by the same thread block — avoids uncoalesced global writes during radix refinement stages of topk algorithms
- The speedup from kernel fusion is proportionally smaller when the dominant compute phase (e.g., topk scan over long inputs) is much larger than the fixed-cost savings; the technique is most impactful when the fused operations represent a significant fraction of total latency
