## Challenge

Kernels that accumulate in fp32 but expose fp16 output reuse the published output tensor as accumulator scratch storage, coupling accumulation format to storage format and preventing a cheaper cast-only writeback path


## Solution

Conditionally allocate a dedicated fp32 accumulation buffer (output_accum) when Kernel::kNeedsOutputAccumulatorBuffer is true, letting the kernel accumulate in the wider type and cast to the final output dtype only at writeback

## Key Lessons

- Decouple accumulation type from output type by allocating a dedicated wider accumulator buffer — the kernel accumulates in fp32 and casts to fp16 only on final writeback, improving numerical stability and avoiding format-coupled scratch storage
- Use a compile-time trait (e.g., kNeedsOutputAccumulatorBuffer) to conditionally allocate the accumulator buffer only for kernel variants that need it, avoiding extra memory overhead for kernels that already match output dtype
