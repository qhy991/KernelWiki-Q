## Challenge

Per-token FP8 quantization kernel uses type-dependent vec_size=8 (16/sizeof(__half)) and writes quantized output bytes via an unrolled scalar store loop, generating 8 individual byte-width memory transactions per iteration that underutilize available store bandwidth on the narrow uint8 output path


## Solution

Double the vector processing width to VEC_SIZE=16 and replace the 8-iteration scalar byte store loop with a single 128-bit uint4 vector store via a union-based OutputStorage, consolidating all output writes into one coalesced transaction that halves the number of quantize-loop iterations and maximizes DRAM write bandwidth utilization on the narrow uint8 output path

## Key Lessons

- Replace scalar byte-width store loops with a single uint4 (128-bit) vector store via a union overlay when accumulating quantized uint8 outputs — the union provides both byte-level indexing for computation and 16-byte alignment for the vector store, eliminating multiple narrow STG instructions in favor of one coalesced transaction
- Use a fixed VEC_SIZE=16 rather than type-dependent vec_size (16/sizeof(T)) in quantization kernels to ensure the output vector is always 16 bytes regardless of input precision, enabling a single uint4 store for the uint8 output path
- Doubling the vector width halves the loop iteration count, which reduces loop overhead and improves instruction-level parallelism — especially beneficial when the store path is narrower than the load path (1-byte output vs 2-byte fp16 input)
