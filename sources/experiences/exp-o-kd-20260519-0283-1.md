## Solution

Change the template parameter from sizeof_bits<ElementOutput> to sizeof_bits<ElementWorkspace> so kCount = 128/32 = 4, matching the actual float workspace buffer layout and ensuring correct vector load/store alignment and reduction tile shape for parallel split-K

## Key Lessons

- When computing vector-access width for a CUTLASS epilogue that reads from a split-K workspace, use sizeof_bits<ElementWorkspace> (the actual buffer element type) not sizeof_bits<ElementOutput> (the final output type) — the workspace layout determines the correct vector width, not the destination type
- In parallel split-K mode the epilogue's kCount directly feeds into ReduceSplitK tile shape (MatrixShape<4, 32*kCount>), so a wrong kCount propagates to out-of-bounds memory accesses beyond just alignment — correctness depends on matching the workspace element size
