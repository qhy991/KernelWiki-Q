## Solution

Switch from type-based config selection to M-dimension-based selection for medium M (16<M<=256): reduce cluster from 4x4 to 2x1 (2 SMs instead of 16), shrink N tile from 256 to 128, and simplify cluster mapping to a 1D layout, cutting inter-SM synchronization cost while retaining sufficient tile parallelism

## Key Lessons

- Select tile/cluster shapes based on the M dimension rather than only the output data type — medium M problems (16<M<=256) benefit from smaller clusters (2x1 vs 4x4) that reduce inter-SM synchronization overhead while still covering the working set
- Over-provisioning SMs via large cluster shapes (e.g., 4x4=16 SMs) wastes resources when the problem dimension is small relative to the tile size, because many SMs have low occupancy and spend cycles on barrier synchronization
- Simplify cluster mapping from 2D (cluster_id_x/y + local_id_x/y) to 1D (single cluster_id) when one cluster dimension is 1 — reduces address arithmetic and makes the launch grid more straightforward
