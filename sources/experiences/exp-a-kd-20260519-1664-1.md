## Solution

Replace memcpy-based __nv_fp8_e4m3 construction with direct assignment to the __x member field, allowing the compiler to use native FP8 conversion instructions on MI300+ hardware; on NVIDIA both paths compile to identical PTX so no regression occurs

## Key Lessons

- Direct __x member assignment on __nv_fp8_e4m3 (or fp8_type on ROCm) replaces memcpy bit-cast for constructing FP8 values from raw bits — the compiler recognizes __x assignment as a trivial store and can emit native conversion instructions, whereas memcpy may force a software emulation path on some platforms
- On NVIDIA hardware, memcpy-based and __x-assignment-based FP8 construction compile to identical PTX, making this a portability improvement (critical for ROCm MI300+) with zero regression risk on CUDA
