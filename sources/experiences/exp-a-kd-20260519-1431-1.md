## Solution

Use adaptive block sizing that selects smaller blocks (256 threads) for large batches (num_tokens >= 256), allowing each SM to host more concurrent blocks and increasing warp-level concurrency to hide memory latency in this memory-bound kernel

## Key Lessons

- Smaller block sizes (256 vs 1024) increase SM occupancy when launching many blocks — each SM can run more concurrent blocks within the 2048-thread/SM cap, providing more warps to hide memory latency via context switching
- Adaptive block sizing based on grid size (num_tokens) is critical for memory-bound kernels: large batches benefit from smaller blocks for higher concurrency, while small batches need larger blocks to amortize per-block overhead
- RMS norm is memory-latency-bound due to two passes over the input row plus weight reads — optimizing occupancy through block-size tuning is more effective than any compute-side optimization for this workload
