## Challenge

For tall-and-skinny GEMM shapes (M=512, N=64, K=4096 where K=64*N far exceeds 3*N), the persistent tile model launches only total_tiles=8 blocks on a 108-SM A800 GPU, leaving ~100 SMs idle and forcing each block to compute hundreds of output elements with the full K=4096 dot product


## Solution

Introduce a K > 3*N heuristic to detect tall-and-skinny shapes where persistent tiles under-occupy the GPU, and dispatch to a StreamK kernel that distributes all M*N output elements evenly across num_sms blocks, achieving full SM utilization

## Key Lessons

- Use a shape heuristic (K > 3*N) to detect tall-and-skinny GEMM where persistent tile scheduling produces fewer blocks than SMs, and switch to StreamK-style element distribution that fills all SMs — applies whenever tile_count << num_sms
- StreamK work distribution assigns output elements in contiguous chunks across all blocks via (total_elements + num_sms - 1) / num_sms, ensuring balanced load and full GPU occupancy regardless of tile geometry
