## Challenge

Implement a persistent tile scheduler for SM90 Stream K GEMM that distributes tile-level work across thread blocks with cluster awareness, requiring correct initialization, tile offset computation across split-K slices, and atomic visit-counter tracking to verify full output coverage


## Solution

Initialize PersistentTileSchedulerSm90StreamK from params, then loop over valid work tiles via get_current_work()/is_valid(); for each tile compute a flattened offset using output_tile_index combined with divmod_tiles_per_output_tile_ and K_idx to account for Stream K's K-dimension splitting; increment visit counters with atomicAdd across k_tile_count slices; use continue_current_work() to decide whether to keep processing the same logical tile or advance to the next

## Key Lessons

- Stream K schedulers split a single output tile across multiple K-slices (K_idx), so the visit-counter offset must be computed as tile_idx * divmod_tiles_per_output_tile_.divisor + K_idx to correctly index into a 2D counter array flattened by the number of K-slices per tile
- Use atomicAdd for shared visit counters in persistent schedulers because multiple thread blocks may be assigned overlapping tiles in Stream K mode — atomic increments capture both visits to detect over-coverage bugs
- Call continue_current_work() before advancing to check if the current tile needs more processing (e.g., remaining K slices); only call advance_to_next_work() + get_current_work() when the tile is fully complete
