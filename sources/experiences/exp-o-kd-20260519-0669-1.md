## Solution

Fuse the sorted_token_ids padding logic directly into both moe_align_block_size_kernel variants via a conditional pad_sorted_token_ids flag, using vectorized Vec stores to fill the entire array with sentinel values within the same kernel, eliminating the separate kernel launch and its associated overhead

## Key Lessons

- Fuse lightweight padding/fill operations into an existing kernel via a conditional flag rather than launching a separate kernel — eliminates ~2.5–5 microseconds of launch overhead per invocation, which is significant for MoE models where expert routing happens on every token
- Use vectorized struct-based stores (Vec with VEC_SIZE elements per write) instead of scalar stores for bulk fill operations — reduces the number of store instructions by the vector width and improves memory throughput
- When fusing a fill-then-use pattern, insert a __syncthreads() barrier (as done in the small-batch variant) between the fill and the consumer loop to ensure all threads complete the overwrite before any thread reads the filled data
