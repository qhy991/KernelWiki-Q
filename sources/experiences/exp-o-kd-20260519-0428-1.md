## Challenge

Tiled SGEMM kernel loads B-matrix tiles from global to shared memory using synchronous FLOAT4 assignments, blocking the warp until each transfer completes and preventing overlap of memory transfers with ongoing computation on the SM


## Solution

Replace synchronous B-matrix global-to-shared FLOAT4 loads with cp.async.ca.shared.global PTX instructions that leverage the hardware async copy engine, issuing cp.async.commit_group after each tile load and cp.async.wait_group(0) before consuming the data, allowing B-matrix loads to proceed in parallel with A-matrix loads and the compute inner loop

## Key Lessons

- Use cp.async.ca.shared.global PTX instructions to replace synchronous global-to-shared FLOAT4 transfers in tiled kernels → the async copy engine operates independently of SM pipelines, allowing load latency to overlap with compute → applies to double-buffered tiled GEMM/convolution kernels where next-tile loads can proceed concurrently with current-tile computation
- Issue cp.async.commit_group after launching async copies and cp.async.wait_group(n) before consuming the data to provide fine-grained load synchronization → avoids stalling the SM on each individual transfer → applies when multiple async copies are batched into a group that can complete in parallel
