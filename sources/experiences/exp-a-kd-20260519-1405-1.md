## Challenge

expandInputRowsKernel performs two scalar global stores per block (reverse_map and permuted_idx) as side-effect bookkeeping alongside the dominant vectorized row copy, adding unnecessary parameter complexity and extra memory allocations that are invisible in isolated timing but create overhead in the full pipeline


## Solution

Remove side-effect writes (reverse_map, permuted_idx) from the kernel entirely, simplifying it to a pure row-shuffle copy that only reads the dst2src map and copies data — the reverse_map/permuted_idx bookkeeping is moved to the caller or eliminated by restructuring the downstream unpermute logic

## Key Lessons

- Scalar side-effect stores (~32KB total) are invisible in microbenchmarks when the dominant path is a ~56MB vectorized copy — the real pipeline benefit comes from eliminating separate memory allocations and downstream kernel lookups, not from kernel-level throughput
- When evaluating kernel optimizations, consider end-to-end pipeline impact (allocation overhead, kernel launch count, downstream consumer simplification) rather than isolated microbenchmark timing — the copy loop dominates this kernel regardless of side-effect removal
- Separating bookkeeping writes from pure data movement kernels simplifies the interface and enables the caller to manage indexing metadata more efficiently, even when the microbenchmark shows neutral performance
