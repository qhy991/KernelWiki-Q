## Solution

The after kernel attempts to improve output throughput by using a fixed VEC_SIZE=16 and replacing 4 scalar stores with a single uint4 coalesced 128-bit store, but the larger vector size reduces total vector iterations below the grid stride, leaving most threads idle and adding per-thread overhead from 4×float4 loads and a 16-element conversion loop

## Key Lessons

- Increasing VEC_SIZE to enable wider vectorized stores can regress performance when the resulting iteration count falls below the total number of GPU threads (blockDim×gridDim) — always verify that num_elements/vec_size ≥ grid_stride to keep all threads occupied
- Wider vector processing increases per-thread register pressure and instruction count (4×float4 loads + 16-element loop vs 1×float4 + 4-element loop); the overhead must be amortized by sufficient per-thread work that keeps all threads active
- A uint4 coalesced store replacing scalar writes only benefits throughput when the kernel is memory-store-bound and all threads participate — here the output of fp8 quant (1 byte/element) is small enough that store bandwidth was not the bottleneck
