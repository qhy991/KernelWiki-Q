## Solution

Scale up to TileShape<128,128,64> and WarpShape<64,64,64>, computing 16384 output elements per threadblock with 16384 int8 global memory fetches (~1:1 arithmetic intensity vs the prior ~0.17), and use K=64 instead of K=128 to reduce shared memory pressure while keeping tensor cores fully utilized via larger warp-level MMA tiles

## Key Lessons

- For large-M GEMM workloads, prefer large tile shapes (e.g. <128,128,64>) over small tiles (e.g. <32,64,128>) to maximize arithmetic intensity — the ratio of output elements to global memory fetches should approach 1:1 rather than being dominated by data movement
- Reducing the K dimension of the ThreadblockShape (128→64) while increasing M and N (32→128, 64→128) lowers shared memory pressure per stage and fits better within SM80 shared memory limits without sacrificing tensor core utilization
- WarpShape must scale proportionally with ThreadblockShape — WarpShape<16,64,64> leaves tensor cores underutilized in small-M warps, while WarpShape<64,64,64> gives each warp enough MMA rows for full occupancy on large-M problems
