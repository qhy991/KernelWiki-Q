## Challenge

On CUDA (NVIDIA), fp8_e4m3_adjusted_max_v<fp8_e4m3fn_t> resolves to 448.0f, identical to the hardcoded FP8_E4M3_MAX constant used in the before version. The NVCC compiler produces equivalent machine code for both the macro-based and template-variable paths, resulting in no measurable performance ...


## Solution

The optimization's value is correctness on ROCm where e4m3fnuz uses 224.0f instead of 448.0f, not a CUDA latency improvement.
