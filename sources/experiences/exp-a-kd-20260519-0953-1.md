## Challenge

The RMSNorm kernel uses a plain device_vec<half2,2> struct with no alignment annotation, relying on manual static_cast pointer indexing by lane_id to load/store packed half2 data, which obscures the intended memory access pattern and forces the compiler to infer alignment from context rather than from explicit type guarantees


## Solution

Replace device_vec with aligned_vector featuring alignas(sizeof(T)*N) on its storage member, and replace manual static_cast pointer indexing with warp_ns::load/store wrappers — explicitly guaranteeing alignment at the type level so the compiler does not need to infer it from cudaMalloc guarantees and stride arithmetic

## Key Lessons

- Explicit alignas annotations on vector types do not improve emitted LD/ST instruction width when the compiler (nvcc -O3) can already infer alignment from cudaMalloc's 256-byte return guarantee and stride arithmetic — the annotation is a correctness/clarity contract, not a performance knob in this scenario
- Warp-level load/store wrappers (warp_ns::load/store) that encapsulate static_cast pointer indexing by lane_id improve code readability and centralize the memory access pattern, but produce identical PTX to the manual pattern
- Microbenchmark differences under ~2% on sub-0.1ms kernels should be treated as noise rather than a meaningful speedup, especially when the instruction-level diff shows no change in load/store width
