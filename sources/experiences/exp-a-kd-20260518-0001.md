## Challenge

SSM scan kernel with multiple rows (kNRows=2) had a shared memory write at index smem_rp[si] without row offset r*MAX_DSTATE_VAL, while reads correctly used smem_rp[si + r*MAX_DSTATE_VAL]. Row 1 overwrites row 0's running prefix, and row 1 reads from its own slot which was never written — producing wrong outputs for rows > 0 on chunks > 0.


## Solution

Align shared memory write index with read index by adding r * MAX_DSTATE_VAL to the write, giving each row its own dedicated shared memory slot. A single addition with zero performance impact — purely a correctness fix.

## Results

- **speedup**: 1.000x (neutral — no measurable performance difference)


## Key Lessons

- When using shared memory with multi-row prefix state in scan kernels, always index by (state_idx + row * ROW_STRIDE) for both reads AND writes. A mismatch where read uses row offset but write does not produces silent data corruption that passes single-row tests. Applies to any multi-row prefix scan or stateful kernel.
- Bugs that are purely index-mismatch (adding an integer offset) have zero measurable performance impact. The fix doesn't change instruction count, memory access width, or computation flow — only correctness. Benchmarking alone would not catch this; reference-based validation is essential.
