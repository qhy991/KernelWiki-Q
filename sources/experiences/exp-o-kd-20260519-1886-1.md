## Challenge

The group iteration loop in group_idx_and_topk_idx_kernel uses a runtime variable n_group as the loop bound, preventing the compiler from unrolling even though n_group typically takes common small values (4, 8, 16, 32) in production models, incurring unnecessary loop overhead and suboptimal instruction scheduling


## Solution

Add a compile-time NGroup template parameter to the kernel; when NGroup > 0, wrap the loop body in a lambda and dispatch it under #pragma unroll with the static bound, while a host-side launch helper uses a switch on runtime n_group (4,8,16,32) to instantiate the correct specialization

## Key Lessons

- Make loop bounds compile-time constants via template parameters when the runtime value belongs to a small known set (4,8,16,32) — this enables #pragma unroll which eliminates branch overhead and lets the compiler interleave iterations for better instruction scheduling
- Use a host-side switch dispatch to map runtime values to compile-time template specializations, preserving backward compatibility with a fallback default for uncommon values
- Loop unrolling benefits are workload-dependent: larger loop bodies (NGroup=16) benefit from reduced overhead, while small loops (NGroup=8) may regress due to increased register pressure
