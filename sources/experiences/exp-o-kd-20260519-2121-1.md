## Solution

Add a q_stride kernel parameter set to query.stride(0) from the host launcher, replacing the hardcoded num_heads*HEAD_SIZE factor with the actual stride so the kernel reads correct data directly from both contiguous and non-contiguous query tensors

## Key Lessons

- Use stride-based pointer arithmetic instead of hardcoded dimension offsets to support both contiguous and non-contiguous tensor layouts without requiring a host-side contiguous copy — applies whenever a kernel receives a tensor that may be a slice from a larger concatenated tensor (e.g., q split from qkv)
- When a sub-tensor is sliced from a concatenated tensor, its stride(0) reflects the parent layout (e.g., 3*num_heads*HEAD_SIZE for a qkv slice) — passing the actual stride as a kernel parameter allows direct GPU access and eliminates the cudaMemcpy2D or torch.contiguous() overhead
