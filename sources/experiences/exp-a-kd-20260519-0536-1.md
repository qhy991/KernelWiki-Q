## Solution

Changing BLOCK_SWIZZLE from false to true with stride=2048 switches from a 2D grid (dim3(8,8)) to a 3D grid (dim3(8,8,1)), intended to interleave block scheduling for better L2 cache reuse, but at M=N=K=1024 the stride exceeds the matrix dimension so N_SWIZZLE=1 and no interleaving occurs — the 3D grid shape instead disrupts the GigaThread Engine's natural block dispatch order

## Key Lessons

- Block swizzle with a stride larger than or equal to the problem dimension produces N_SWIZZLE=1, providing zero interleaving benefit — always verify that (N+stride-1)/stride >= 2 before enabling swizzle so the 3D grid actually has multiple z-slices to interleave
- A 3D grid with grid.z=1 can disrupt the GPU GigaThread Engine's block dispatch order compared to an equivalent 2D grid, causing severe regression even when the mathematical mapping (bx=blockIdx.x) is identical — the hardware scheduling path differs between dim3(x,y) and dim3(x,y,1)
- Block swizzle is a large-problem optimization: at small matrix sizes (1024x1024), the baseline 2D grid already achieves good L2 behavior due to sufficient block count, and swizzle overhead is pure waste
