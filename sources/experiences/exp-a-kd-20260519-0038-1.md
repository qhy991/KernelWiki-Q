## Challenge

Two separate FP16 GEMM kernels (D0=ReLU(A0*B0+C0), D1=ReLU(D0*B1+C1)) execute back-to-back, writing the intermediate D0 tensor (128 KB) to global memory and reading it back, plus incurring double kernel launch overhead


## Solution

Fuse both GEMMs into a single B2bGemm kernel using CUTLASS's back-to-back GEMM API, where GEMM0 output stays in the register file and is consumed directly by GEMM1, eliminating the intermediate DRAM write/read and reducing threadblock shapes (128→64 M-dim) to keep the accumulator resident in RF

## Key Lessons

- Fuse back-to-back GEMMs where one's output feeds the next's input via CUTLASS B2bGemm — the intermediate stays in the register file, eliminating a full DRAM round-trip (2×M×K×sizeof(element)) and one kernel launch boundary
- Reduce threadblock tile M-dimension (128→64) and warp M-dimension (64→16) when fusing GEMMs to keep GEMM0 accumulators within RF capacity, since the epilogue output must reside on-chip for GEMM1 to consume without spilling to shared memory
- For small-to-medium problem sizes, kernel launch overhead and intermediate DRAM traffic dominate total latency — a fused kernel can halve end-to-end time even when compute throughput per-kernel is unchanged
