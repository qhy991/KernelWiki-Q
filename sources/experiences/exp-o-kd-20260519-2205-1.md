## Solution

Replace fixed-size kChunkSize=512 chunking with variable-length chunks read from a device-side cu_chunk_seqlen cumulative-offset array using block_size=2048, reducing chunk count from 8 to 2 for seqlen=4096. The tokens_processed accumulator replaces repeated chunk*kChunkSize multiplication, and the larger per-chunk body improves instruction-level parallelism within each thread's inner loop.

## Key Lessons

- Increase chunk/block size to reduce per-chunk loop overhead (bookkeeping, boundary checks, address recomputation) when the computation per element is lightweight — a 4x chunk-size increase yielded 1.68x speedup by cutting chunk count from 8 to 2
- Prefer a running offset accumulator (tokens_processed) over recomputing chunk*kChunkSize each iteration — eliminates one integer multiply per index calculation and provides natural support for variable-length chunks
- Use a cumulative-offset array (cu_chunk_seqlen) for variable-length chunk metadata instead of embedding a fixed chunk size as a compile-time constant — decouples chunk boundaries from thread geometry and enables runtime flexibility without code changes
