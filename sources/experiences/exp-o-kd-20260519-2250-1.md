## Solution

Replace the hardcoded head_size stride parameter with a configurable int64_t head_stride, allowing the kernel to address heads at arbitrary offsets within the padded tensor directly — this eliminates all four copy-to-contiguous / copy-from-contiguous kernels and their associated launch overhead and redundant memory traffic

## Key Lessons

- Parametrize stride dimensions (head_stride) rather than hardcoding structural sizes (head_size) so the kernel can operate directly on non-contiguous padded layouts — eliminates copy-to-contiguous and copy-from-contiguous overhead entirely, which dominates when copy kernels add 4 extra launches and 2x memory traffic
- Copy-to-contiguous + kernel + copy-back patterns for padded tensors are a major performance anti-pattern because the redundant memory round-trip cost far exceeds the slightly less cache-friendly strided access within the padded layout
- Use int64_t for stride parameters to avoid truncation when tensor strides exceed 2^31 elements — critical for large-batch or high-dimensional layouts
