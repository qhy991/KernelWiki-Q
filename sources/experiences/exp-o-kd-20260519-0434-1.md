## Challenge

In an 8x16 SGEMM prologue (BM=128, BN=256, 256 threads), each thread loads 16 float values for the B tile from global to shared memory using synchronous FLOAT4 stores, blocking the warp while A-tile loading and online transpose must wait


## Solution

Replace synchronous FLOAT4 global-to-shared stores for B with cp.async.ca asynchronous copies (4 issues of 16 bytes each, committed as a group), allowing B-tile transfers to overlap with A-tile loading and online transpose, then CP_ASYNC_WAIT_GROUP(0) before __syncthreads ensures completion

## Key Lessons

- Replace synchronous FLOAT4 global-to-shared stores with cp.async.ca to overlap gmem-to-smem transfers with other independent per-thread work — applies when a tile prologue has multiple load stages that can proceed in parallel
- Use CP_ASYNC_COMMIT_GROUP after issuing async copies and CP_ASYNC_WAIT_GROUP(n) before consuming the data to create a pipelined overlap window without manual barrier management
- Larger B tiles (BN=256 vs smaller) benefit more from async copies because the greater global memory traffic for B makes latency hiding more impactful
