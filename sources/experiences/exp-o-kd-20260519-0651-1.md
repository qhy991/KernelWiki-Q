## Solution

Replace ^row with ^(row % 8) in the XOR remapping, which constrains the XOR operand to [0,7] so that the remapped index always stays within the row's valid shared memory range, eliminating the bounds-check clamp and producing a predictable bank-conflict-free access pattern for any tile height

## Key Lessons

- When using XOR-based shared memory index remapping for bank-conflict avoidance, constrain the XOR operand with modulo equal to the warp tile width (e.g., row % 8) to guarantee remapped indices remain within the valid row range — the full row value can exceed the tile width and produce out-of-bounds accesses for tiles beyond the first warp
- Eliminating a runtime bounds-check clamp by construction (making indices always valid) removes branch divergence in shared memory access loops and improves both correctness and performance
