## Solution

Replace scalar 1-element-per-thread processing with vectorized int4 loads (8 bf16 per thread via 128-bit transactions), reducing threads from 576 to 96; eliminate shared memory and both __syncthreads() by computing per-tile scales via half-warp shfl_xor reductions (width=16) writing directly to global memory; pack 8 fp8 outputs into aligned uint64_t stores

## Key Lessons

- Vectorized int4 loads (128-bit) process 8 bf16 elements per thread, reducing total threads from 576 to 96 and increasing concurrent blocks per SM — applies whenever input data is contiguous and 16-byte aligned
- Eliminating shared memory and __syncthreads() via half-warp shfl_xor reductions (width=16) removes ~100–200 cycles of barrier stall per invocation — applies when per-tile reductions span only a sub-warp and each thread already computes a local aggregate
- Packing multiple fp8 quantized values into a single uint64_t store (8 bytes per transaction) improves output memory throughput — applies when quantized outputs are byte-aligned and contiguous
