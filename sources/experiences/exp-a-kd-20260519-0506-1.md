## Challenge

FlashAttention kernel uses FP16 accumulator storage (kOStorageAccFloat32=0), requiring explicit __half2float/__float2half_rn conversion instructions around every online-softmax rescaling step, adding per-step instruction overhead to the attention loop


## Solution

Switch the output accumulator storage to FP32 (kOStorageAccFloat32=1) so that the online-softmax rescaling operates directly in float, eliminating the 8 half↔float conversion instructions per rescaling step, though the net speedup is marginal because MMA compute and global memory traffic dominate the kernel's critical path

## Key Lessons

- FP32 accumulator storage in FlashAttention eliminates half↔float conversion overhead during online-softmax rescaling, but the benefit is negligible when the kernel is compute-bound (MMA ops) or memory-bound (Q/K/V loads) — the rescaling conversions are a small fraction of total instruction count
- When benchmarking accumulator precision changes, expect neutral results if the conversion instructions are not on the critical path; the dominant cost in FlashAttention is the QK^T and PV MMA operations plus global memory traffic, not the scalar rescaling arithmetic
