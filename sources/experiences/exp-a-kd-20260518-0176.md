## Challenge

The after kernel replaces synchronous smem[tid]=gmem[tid] loads with cp.async.ca.shared.global PTX instructions followed by commit_group and wait_all. However, the kernel has no compute phase that can overlap with the async copy — the cp_async_wait<0>() immediately blocks until all async copies c...


## Solution

Replace synchronous smem[i]=g_in[i] with cp.async.ca.shared.global PTX instructions followed by commit_group and wait_all to issue asynchronous global-to-shared copies, though the single-stage design with no interleaved compute means the wait_all immediately blocks, yielding no overlap benefit

## Results

- **speedup**: 0.983x (neutral)


## Key Lessons

- cp.async only provides benefit when concurrent computation can overlap with the async copy — issuing cp.async+commit_group+wait_all in a single-stage kernel with no interleaved compute is functionally equivalent to synchronous load with added instruction overhead for barrier primitives
- In pipelined kernels with multiple async stages, cp.async enables issuing global loads for stage N+1 while computing on stage N, but single-stage microbenchmarks cannot exploit this overlap and may see slight slowdown from extra fence/wait instructions
