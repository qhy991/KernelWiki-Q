## Solution

Replace byte-by-byte access with uint4 (16-byte) vectorized load/store: each thread loads 16 bytes at once via uint4, processes all bytes in registers using the same LUT lookup, then stores 16 bytes back — reducing total memory transactions by 16x

## Key Lessons

- Vectorized uint4 load/store reduces memory transactions but also reduces total work items by 16x (1048576 bytes → 65536 vectors), cutting launched thread-blocks from 4096 to 256 — at small data sizes that fit in L2 cache, the occupancy loss outweighs coalescing gains
- When data fits entirely in L2 cache (e.g., 1 MB on a 40 MB L2 A800), byte-by-byte access is already efficiently serviced by the cache, so vectorization provides negligible bandwidth benefit while reducing parallelism — vectorization benefits appear primarily at bandwidth-bound scales exceeding L2 capacity
- Kernel runtime at small data sizes is dominated by launch overhead and cache hits rather than global memory bandwidth; always profile at multiple data sizes before concluding that vectorization is beneficial
