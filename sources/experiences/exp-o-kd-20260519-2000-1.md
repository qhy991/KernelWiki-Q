## Challenge

The vectorized fused_add_rms_norm_kernel computed input vector offsets as `blockIdx.x * vec_hidden_size + idx`, assuming the input tensor's row stride equals hidden_size; for strided input tensors this assumption breaks, causing incorrect memory addressing and forcing a fallback to the slower scalar path or a costly `.contiguous()` copy


## Solution

Introduce vec_input_stride = input_stride / width derived from the element-level input stride, and use a dedicated strided_id = blockIdx.x * vec_input_stride + idx for all input_v reads and writes, while residual continues using the original contiguous id; this separates vector-level addressing for strided input from contiguous residual access

## Key Lessons

- When vectorizing a kernel via reinterpret-cast to wider types, the vector-level stride must be computed as element_stride / vector_width — using hidden_size / width as stride assumes contiguous layout and breaks for strided tensors
- Keep independent strides for tensors with different memory layouts in fused kernels: residual may be contiguous while input is strided, so separate id computations (contiguous id vs strided_id) ensure correct addressing for each tensor without forcing a .contiguous() copy
- Passing an explicit input_stride parameter to fused kernels enables the fast vectorized path to handle non-contiguous inputs directly, eliminating the need for upstream .contiguous() calls that copy data (1.42x speedup in the strided case)
