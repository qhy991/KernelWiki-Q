## Challenge

Paged attention V-cache is stored as fp16 (2 bytes/element), consuming twice the memory bandwidth of FP8 for value vectors that can tolerate reduced precision during the weighted accumulation step


## Solution

Store V-cache as FP8-E5M2 (1 byte/element) and dequantize to fp16 via software bit manipulation (fp8_to_half extracts sign/exponent/mantissa and reconstructs fp16 bits), halving memory traffic — but the per-element multi-instruction conversion overhead plus atomicAdd contention made the kernel 12.6% slower

## Key Lessons

- Software FP8-to-fp16 dequantization via integer bit manipulation generates multiple shift/mask/or instructions per element — this compute overhead can exceed memory bandwidth savings when each converted value is immediately consumed by atomicAdd rather than batched into vectorized operations
- When the kernel bottleneck is shared-memory atomicAdd contention rather than memory bandwidth, reducing V-cache data width (fp16 to fp8) does not help because the critical path is the atomic operation latency, not the load width
- FP8 quantization of V-cache is more effective when hardware FP8 conversion (e.g. __nv_fp8_e5m2 to half on sm_89+) is available, or when values can be loaded and converted in batches to amortize the per-element conversion cost
