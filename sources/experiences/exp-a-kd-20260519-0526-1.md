## Challenge

Default 2D grid scheduling maps thread blocks in row-major order so that adjacent blocks along the N dimension access overlapping L2 cache sets when loading the B matrix, limiting L2 locality for large matrix multiplications


## Solution

Block swizzle remaps thread-block IDs via a 3D grid (gridDim.z = N_SWIZZLE groups) so that consecutive blocks are spread across non-overlapping N-dimension tiles, improving L2 cache reuse for B-matrix tiles when the matrix is large enough to have multiple swizzle groups

## Key Lessons

- Block swizzle via 3D grid remapping only produces measurable L2 locality benefits when N_SWIZZLE > 1 (i.e., N >> SWIZZLE_STRIDE) — at small problem sizes where N < SWIZZLE_STRIDE the 3D grid degenerates to the same mapping as the 2D grid and any timing difference is noise
- The compile-time BLOCK_SWIZZLE flag controls the bx = blockIdx.z * gridDim.x + blockIdx.x remapping at zero runtime cost when false (the multiplication by zero is elided), making it safe to always include in production kernels
