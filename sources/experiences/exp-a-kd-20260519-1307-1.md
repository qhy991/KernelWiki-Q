## Challenge

preprocessTopkId kernel performs per-thread global memory lookups into expert_map (n_experts=8, 32 bytes) for every topk ID element, but this tiny table is already resident in L1 cache after the first access, making a shared-memory caching optimization irrelevant to performance


## Solution

Add cooperative shared-memory loading of expert_map with __syncthreads() barrier before per-thread lookups, so threads read from smem instead of global memory

## Key Lessons

- Shared-memory caching of very small lookup tables (≤32 bytes) provides no measurable benefit because the data is already resident in L1 cache after the first thread access — the hardware cache handles it transparently
- Before optimizing a lookup table with shared memory, verify that the lookup is the actual bottleneck; here the dominant memory traffic is the 32768-element topk_id array read/write, not the 8-entry expert_map lookup
- Adding __syncthreads() for a negligible smem load can slightly hurt performance on tiny kernels by increasing instruction overhead and barrier latency without reducing any real memory stall
