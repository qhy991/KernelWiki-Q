## Solution

Replace the entire 25-line manual bit-manipulation routine with the vendor-provided __nv_fp8_e4m3(float) constructor intrinsic from cuda_fp8.h, which should theoretically map to a single hardware conversion instruction on FP8-capable architectures

## Key Lessons

- On GPUs without native FP8 hardware (SM 8.0/Hopper-), vendor intrinsics like __nv_fp8_e4m3 fall back to opaque software emulation that can be slower than hand-written bit-manipulation — always benchmark on the target SM architecture rather than assuming intrinsics map to efficient hardware
- Manual bit-manipulation FP8 conversion with loop-based exponent search, despite its apparent complexity, can be effectively optimized by the compiler through loop unrolling and branch prediction, outperforming black-box vendor emulation paths
- Vendor FP8 intrinsics should be validated per-SM-generation: they provide clear wins on SM 9.0+ with native FP8 ALUs but may regress on SM 8.0 where the conversion is entirely software-emulated
