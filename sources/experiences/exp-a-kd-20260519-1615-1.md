## Challenge

Naive FP16-to-INT8 conversion processes one element per thread using __half2float + roundf + conditional clamp + scalar cast, requiring multiple FP arithmetic and branching instructions per element and issuing scalar loads/stores


## Solution

Magic-bias technique (arxiv:2406.09904) adds 0x64806480 to FP16 pairs via __hadd2 so the resulting bit pattern directly contains the int8 value in its low byte; prmt.b32 extracts the bytes across two registers, XOR with 0x80808080 flips the sign bit to produce signed int8, processing 4 elements per thread with a single packed int32 store

## Key Lessons

- Magic-bias FP16-to-INT8 (adding 0x64806480 via __hadd2, extracting with prmt.b32, XOR sign-flip) replaces per-element __half2float + roundf + clamp + cast with three register-level operations — applicable when converting bounded FP16 values to INT8 and the overhead of float conversion pipeline matters
- Instruction-level optimizations (eliminating __half2float + roundf) are invisible at small payload sizes (4096 elements) where kernel launch overhead and memory transfer dominate — such techniques require larger workloads or fusion into a larger kernel to show measurable gains
- Processing 4 elements per thread with packed int32 store (vs scalar int8 store) quarters the number of store transactions and threads needed — benefits are proportional to the memory-bound fraction of the kernel
