## Solution

Replace the two-step bf16-GEMM + cast-kernel pipeline with a single cublasGemmEx call that uses CUDA_R_32F as the output type, letting cuBLAS write fp32 output directly from the fp32 accumulator and eliminating the intermediate bf16 buffer and the separate cast kernel entirely

## Key Lessons

- Use cublasGemmEx with CUDA_R_32F output type when the final consumer needs fp32 — cuBLAS with CUBLAS_COMPUTE_32F already accumulates in fp32 internally, so requesting fp32 output avoids an unnecessary bf16 round-trip through global memory
- Eliminating separate dtype-cast kernels after GEMM saves both global-memory bandwidth (one fewer write+read of the output matrix) and kernel launch overhead — the savings compound with larger output matrices
- cublasGemmEx supports mixed input/output types (e.g., CUDA_R_16BF inputs with CUDA_R_32F output) — always check whether the library can produce the desired output type directly before adding a post-GEMM conversion pass
