## Challenge

MLA key concatenation kernel reads and writes individual bf16 values via scalar 2-byte transactions in inner j-loops (NOPE_PER_LANE=4 and ROPE_PER_LANE=2), producing scattered 2-byte sectors across 32 lanes instead of coalesced 128/256-byte transactions aligned to L2 cache line boundaries


## Solution

Replace scalar bf16 j-loops with reinterpret_cast<int2*> (8-byte) reads/writes for NOPE and reinterpret_cast<int*> (4-byte) for ROPE, so each lane issues a single vectorized transaction producing 32-lane x 8-byte = 256-byte or 32-lane x 4-byte = 128-byte coalesced transfers matching L2 cache line granularity

## Key Lessons

- Replace scalar bf16 element loops with reinterpret_cast<int2*> or reinterpret_cast<int*> vectorized transactions so the warp produces one coalesced 128/256-byte transaction instead of many scattered 2-byte sectors — applies whenever each lane accesses contiguous bf16 elements whose total width divides evenly into 4 or 8 bytes
- Use static_assert(sizeof(BufType) == ELEMENTS_PER_LANE * sizeof(element)) to enforce that the vectorized buffer type exactly matches the per-lane data width, catching stride mismatches at compile time
