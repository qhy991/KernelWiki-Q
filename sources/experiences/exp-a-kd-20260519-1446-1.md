## Challenge

The bit-extraction kernel uses hardcoded shift amount (8) and mask value (0xff) for 8-bit element extraction, making the code non-reusable for other bit widths (4-bit, 2-bit, 1-bit) and requiring a separate kernel implementation per quantization granularity


## Solution

Replace hardcoded constants with a compile-time template parameter num_bits, deriving pack_factor=32/num_bits, mask=(1<<num_bits)-1, and tile_ints as constexpr values so the compiler generates identical specialized PTX for each instantiation while enabling code reuse across bit widths

## Key Lessons

- Parameterizing bit-width via a compile-time template parameter (num_bits) and deriving all related constants (pack_factor, mask, tile_ints) as constexpr yields zero-overhead abstraction — the compiler specializes each instantiation to the same PTX as handwritten hardcoded constants
- Code generality through templates does not imply performance cost when all derived values are constexpr and the loop bounds are known at compile time; NVCC fully unrolls and constant-folds them into identical machine code
