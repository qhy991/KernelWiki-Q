## Solution

Replace custom FP8_TYPE_WRAPPER software emulation with CUDA native __nv_fp8_e4m3 type, which maps to a single PTX cvt.rn.satfinite.e4m3x2.f32 instruction on SM 8.9+/9.0+ GPUs but falls back to a less efficient runtime emulation function on SM 8.0, causing a 50% regression

## Key Lessons

- On SM 8.0 (Ampere) without native fp8 hardware, __nv_fp8_e4m3 static_cast dispatches to a CUDA runtime emulation function less efficient than a well-optimized inline integer bit-manipulation path — always benchmark native types on target hardware before replacing custom software emulation
- The fp8e4m3fn_from_fp32_value function compiles to ~5-6 ALU instructions that NVCC -O3 fully inlines into unrolled loops with good ILP — on hardware without native fp8 support, hand-tuned software emulation can outperform the runtime library path
- Native fp8 types (__nv_fp8_e4m3) provide significant speedup only on SM 8.9+ (Ada) and SM 9.0+ (Hopper) with dedicated cvt.rn.satfinite.e4m3x2.f32 PTX instructions — verify target architecture before migrating to native fp8 types
