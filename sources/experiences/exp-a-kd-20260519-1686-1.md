## Challenge

Grouped GEMM offset computation kernel stores expert_offsets as int32_t, causing silent integer overflow and truncation when MoE expert row-counts exceed INT32_MAX (~2.1 billion), producing completely wrong pointer arithmetic for downstream weight/activation matrices


## Solution

Change the expert_offsets parameter type from int32_t* to int64_t* so the full 64-bit offset values are preserved when read by the kernel, ensuring correct pointer arithmetic for all expert sizes; also update the host-side allocation and copy to use int64_t throughout

## Key Lessons

- Use int64_t for tensor offset indices in MoE/grouped-GEMM kernels where accumulated expert row-counts can exceed 2^31 — silent int32 truncation produces wrong pointer arithmetic that is difficult to diagnose because the kernel runs without any visible error
- Prefer matching the index type to the maximum possible value rather than the 'typical' value: even if most workloads stay within int32 range, a correctness bug that only manifests at scale is the most dangerous kind of bug in production serving systems
