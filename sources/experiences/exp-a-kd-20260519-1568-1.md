## Challenge

Naive iterative-selection top-k with K=8 and N=128 performs 8 full strided scans of shared memory, each followed by a 5-step warp shfl_down_sync reduction and __syncthreads — O(N*K) total work, but the tight scan-plus-reduction loop has minimal synchronization overhead


## Solution

WarpSelect replaces iterative max-selection with an incremental top-k buffer: elements are streamed in rounds, filtered by __ballot_sync early-exit against the k-th value, compacted via __popc into shared memory, then merged using warp-level bitonic sort/merge — intended O(N log k) but the per-round __ballot_sync, __popc position computation, shared memory staging, and __syncwarp overhead exceeds the before code's simpler scan-plus-reduction cost at k=8, N=128

## Key Lessons

- For small problem sizes (k=8, N=128), the overhead of sophisticated algorithms like WarpSelect (__ballot_sync + __popc compaction + bitonic sort/merge + __syncwarp per round) can exceed the cost of a simpler iterative scan with warp reduction — the naive O(N*K) approach has fewer synchronization events and no shared memory staging overhead
- WarpSelect-style incremental top-k with early-exit only benefits when the k-th threshold becomes restrictive enough to filter most candidates early; for k=8 with only 4 rounds of 32 elements, the buffer fills roughly once before the threshold kicks in, yielding negligible early-exit savings
- Algorithmic complexity improvements (O(N*K) → O(N log k)) do not guarantee GPU performance gains — constant factors from warp synchronization, shared memory compaction, and multi-stage sorting networks dominate at small N
