## Challenge

A scan/prefix-sum kernel uses a modulo guard (seqlen % (kNItems*kNThreads) != 0) to skip masking when seqlen is evenly divisible by the chunk size; the after code replaces this with an always-executed per-item runtime check (tid*kNItems+i >= chunk_tokens) that the compiler cannot eliminate because chunk_tokens is a runtime variable


## Solution

The after code unconditionally checks chunk_tokens per item for correctness across all seqlen values, but this removes the compile-time-eliminable modulo guard that made the inner masking branch dead code for even-length sequences, resulting in extra per-item comparisons the compiler cannot optimize away

## Key Lessons

- Preserve compile-time-eliminable guards (e.g., modulo checks on known-divisible seqlen) over unconditional runtime checks — when the compiler can prove a branch is dead code it incurs zero cost, but a runtime variable comparison per item cannot be eliminated
- Unifying code paths for generality can introduce measurable overhead in hot loops; prefer template specialization or branch-on-compile-time-constants to maintain both correctness and performance across different input shapes
