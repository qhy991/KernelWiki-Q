## Solution

Replace single-stage blocking copy with 3-stage cp.async pipeline: a prologue issues 2 async copies into multi-buffered shared memory, then the mainloop overlaps cp.async copies for the next tile with per-thread scalar multiply-add on the current tile, attempting to hide global memory latency

## Key Lessons

- Multi-stage cp.async pipelining triples shared memory per stage (16 KB → 48 KB for 3 stages), which reduces SM occupancy — when compute per k-tile is lightweight (per-thread scalar multiply-add), the occupancy loss outweighs any latency-hiding gain
- cp.async pipelining only benefits kernels where the compute phase is long enough to fully overlap with async copy latency; tensor-core warp-level MMA provides sufficient compute density but per-thread scalar loops do not
- Pipelining adds fixed overhead (prologue copies, cp_async_fence, cp_async_wait, stage index bookkeeping) that must be amortized across enough compute work — applying it to low-compute-density kernels is a net negative
