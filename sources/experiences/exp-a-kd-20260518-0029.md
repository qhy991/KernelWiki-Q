## Solution

Add a cap of sm_count/2 on the computed split_kv value so that the split count never exceeds half the SMs, preventing resource oversubscription; in small-K scenarios (split_kv=8 < cap=54) the cap is a no-op, confirming correct guard behavior

## Results

- **speedup**: 0.999x (neutral)


## Key Lessons

- Cap split-KV count at sm_count/2 to prevent excessive parallel splits from oversubscribing SMs and increasing atomic contention — applies to any split-K or split-KV attention pattern where the heuristic can overestimate optimal split count
- When adding a heuristic guard, test with small-K baseline cases where the cap should not activate — identical performance confirms the guard is correctly transparent in no-op scenarios
