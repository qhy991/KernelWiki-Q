## Challenge

AWQ INT4 dequantize kernel applies per-element (q - zero) * scale using a scalar loop of __hsub and __hmul intrinsics over 8 half values, issuing 16 separate scalar half arithmetic instructions where vectorized f16x2 operations could handle two half values per instruction


## Solution

Replace the scalar __hsub/__hmul loop with explicit inline PTX sub.f16x2 and fma.rn.f16x2 instructions operating on the uint4 .x/.y/.z/.w lanes, processing two half values per PTX instruction (8 PTX ops vs 16 scalar ops theoretically)

## Key Lessons

- NVCC at -O3 auto-vectorizes scalar __hsub/__hmul intrinsics into f16x2 PTX instructions, producing nearly identical SASS code — explicit inline PTX provides no measurable advantage when the compiler already performs this transformation
- When a kernel is memory-bandwidth-bound rather than arithmetic-bound, even halving the instruction count for arithmetic yields negligible speedup — profile before optimizing the wrong bottleneck
- A scalar loop over half values with __hsub/__hmul is sufficient and preferable to hand-written PTX when NVCC's auto-vectorization covers the case — reserve explicit PTX for patterns the compiler cannot vectorize (non-uniform access, conditionals, cross-lane operations)
