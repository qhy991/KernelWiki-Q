## Solution

Remove the runtime min_scaling_factor parameter from all kernel and device function signatures (3 signatures, 4 call sites) and replace it with a compile-time constexpr type trait min_scaling_factor<scalar_out_t>::val() accessed inside compute_dynamic_per_token_scales, enabling constant folding and freeing a register in the inner loop

## Key Lessons

- Type-dependent constants (e.g., min_scaling_factor determined by scalar_out_t) should be compile-time constexpr type traits rather than runtime kernel parameters — this enables constant folding, frees register slots in inner loops, and eliminates host-side dtype branching
- Removing a single float kernel parameter can yield measurable speedup (1.127x on h4096) when it reduces register pressure inside tightly-strided inner loops, even though launch overhead for one float is negligible
- Centralizing quantization constants in a dedicated header (quantization/utils.cuh) via constexpr type traits also enables platform-specific specializations (e.g., ROCm Float8_e4m3fnuz max=224.0 vs CUDA 240.0) without polluting kernel code
