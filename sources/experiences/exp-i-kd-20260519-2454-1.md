## Challenge

Need to implement a GPU delay kernel that spins waiting for an atomic release flag, while minimizing resource usage during the wait period and ensuring correct memory ordering semantics


## Solution

Implement a spin-wait loop using cuda::atomic<bool>::load with memory_order_acquire to poll the release flag, and insert __nanosleep(100) guarded by an architecture check (__CUDA_ARCH__ >= 700) to yield the SM during each poll iteration, reducing busy-wait power consumption

## Key Lessons

- Use memory_order_acquire when polling an atomic flag from a waiting thread — it ensures the waiting thread sees all writes from the releasing thread that happened before the flag was set, providing correct synchronization without full sequential consistency overhead
- Guard __nanosleep with __CUDA_ARCH__ >= 700 to maintain backward compatibility — __nanosleep is only available on CUDA 7.0+ (Volta and later), and the preprocessor check ensures the kernel compiles (but spins without sleeping) on older architectures
- A 100ns __nanosleep interval in a spin-wait loop is a practical balance between polling latency and SM resource usage — shorter intervals increase polling overhead while longer intervals increase notification latency
