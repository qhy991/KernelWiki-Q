## Solution

Replace the two while-loops with a direct min/divUp formula: threads_per_block = min(DEFAULT_BLOCK_SIZE, total_threads), blocks_per_grid = min(MAX_ALL_REDUCE_BLOCKS=32, divUp(total_threads, threads_per_block)), and change MAX_ALL_REDUCE_BLOCKS from 36 to 32 (power of 2) for deterministic, O(1) launch configuration

## Key Lessons

- Prefer closed-form min/divUp formulas over iterative search loops for kernel launch configuration — the while-loop approach adds unpredictable host-side overhead and is harder to reason about, even when the resulting block/thread counts may coincide
- Use power-of-2 constants (32 instead of 36) for MAX_ALL_REDUCE_BLOCKS to ensure clean division properties and predictable occupancy across workload sizes
- Simplifying launch configuration logic from O(n) search to O(1) arithmetic improves code maintainability without regressing kernel performance, since the dominant cost is in the kernel itself (global memory access and shared-memory reduction), not in host-side config computation
