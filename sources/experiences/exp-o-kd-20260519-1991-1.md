## Solution

Remove the power-of-2 static_asserts on VPT and NUM_EXPERTS, and promote MAX_BYTES_PER_LDG from a hardcoded constant to a template parameter on topkGatingSoftmaxLauncherHelper. This allows the launcher to specify 8-byte loads (2 floats per ldg) for multiples-of-64 expert counts, keeping THREADS_PER_ROW a power of 2 so the butterfly shuffle reduction and warp-level row partitioning work correctly without falling back to the two-kernel path.

## Key Lessons

- Replacing overly restrictive power-of-2 static_asserts on derived quantities (VPT, NUM_EXPERTS) with constraints on the actual invariant (BYTES_PER_LDG is power-of-2 and <= 16) preserves correctness while expanding compile-time coverage to non-power-of-2 expert counts — applies whenever kernel correctness depends on alignment of the load width rather than the input dimension itself
- Promoting hardcoded tuning constants (MAX_BYTES_PER_LDG) to template parameters enables per-specialization load-width selection, allowing smaller vector loads that keep derived thread counts power-of-2 — critical for kernels using butterfly shuffle reductions where THREADS_PER_ROW must be power-of-2
- Fusing separate softmax+topk kernels into a single kernel eliminates redundant global memory round-trips and one kernel launch overhead — the 2.16x speedup on 192-expert MoE gating confirms that the fallback path was dominated by memory traffic and launch overhead
