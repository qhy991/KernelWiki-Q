## Challenge

The original verify_kernel uses a pure stochastic acceptance condition `coin < acc` with cumulative probability, and the after version replaces it with a combined greedy+stochastic criterion `coin <= acc / threshold_acc || target_prob_single >= threshold_single`, but with default thresholds (1.0, 1.0) the only semantic change is the boundary comparison operator from `<` to `<=`


## Solution

Replace the pure stochastic acceptance `coin < acc` with a parameterized combined criterion `coin <= acc / threshold_acc || target_prob_single >= threshold_single` that supports greedy override and adjustable acceptance threshold, while preserving identical behavior under default thresholds (1.0, 1.0) since floating-point equality coin == acc is vanishingly rare

## Key Lessons

- Changing `<` to `<=` in floating-point acceptance thresholds is neutral in practice because exact floating-point equality (coin == acc) occurs with negligible probability — applies when converting stochastic sampling from strict to non-strict comparison
- Parameterizing acceptance criteria with threshold_single and threshold_acc enables future greedy-override and adjustable-acceptance strategies without changing kernel structure, even though default parameters yield identical behavior
- Introducing a local variable (target_prob_single) for an already-loaded value is a stylistic refactor that aids readability for multi-criterion conditions but has no performance impact since the compiler optimizes the register usage identically
