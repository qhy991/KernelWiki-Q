## Challenge

Warp index is computed via __shfl_sync(threadIdx.x / 32, 0) placed after a divergent branch where even lanes execute extra work and odd lanes skip, forcing odd lanes to stall at the shuffle reconvergence barrier while waiting for even lanes to finish


## Solution

Replace __shfl_sync-based warp index broadcast with direct arithmetic (threadIdx.x / 32), eliminating the implicit warp reconvergence barrier so each thread independently computes its warp index without waiting for divergent lanes to catch up

## Key Lessons

- Prefer direct arithmetic (threadIdx.x / 32) over __shfl_sync for values each thread can compute independently — shuffle intrinsics create implicit reconvergence barriers that stall early-arriving lanes when placed after divergent branches
- __shfl_sync requires all participating lanes to reach the instruction before any can proceed; when the shuffled value is a pure function of per-thread data (like threadIdx.x), eliminating the shuffle removes an unnecessary synchronization point that amplifies divergence overhead
