## Challenge

Kernel uses a hardcoded stride formula (seq_idx * num_heads * HEAD_SIZE) that assumes a contiguous memory layout for the Q tensor, preventing reuse with non-contiguous (strided) tensor layouts such as padded or transposed batches


## Solution

Replace the hardcoded stride expression (seq_idx * num_heads * HEAD_SIZE) with a runtime q_stride parameter, decoupling the kernel from the assumption of contiguous layout while preserving identical pointer arithmetic when q_stride == num_heads * HEAD_SIZE

## Key Lessons

- Parameterizing tensor stride as a runtime kernel argument instead of hardcoding the layout formula (num_heads * HEAD_SIZE) adds generality at zero performance cost — the compiler generates the same multiply-add for a runtime stride as for the constant expression
- Stride parameterization enables a single kernel to handle both contiguous and non-contiguous tensor layouts (e.g., padded batches, transposed views) without code duplication or separate kernel variants
