## Challenge

MMA promotion interval validation uses a hardcoded divisibility-by-4 check (mma_promotion_interval % 4 == 0), which incorrectly assumes every mainloop iteration always issues exactly 4 MMA instructions; in reality the count is size<2>(TileShape{}) / tile_size_mnk<2>(), varying with tile configuration


## Solution

Replace the hardcoded % 4 divisibility check with a dynamically computed divisor (k_tile / mma_k_atom) that correctly reflects the actual number of MMA instructions issued per mainloop iteration for any tile shape configuration

## Key Lessons

- Never hardcode the number of MMA instructions per mainloop iteration — compute it dynamically as size<2>(TileShape{}) / tile_size_mnk<2>() because the actual count varies across tile configurations (2, 4, 6, 8, etc.)
- Validation checks that bake in a fixed divisor based on one tile configuration will silently accept or reject promotion intervals incorrectly for all non-matching configurations — always derive validation parameters from the same tile shape metadata used by the mainloop
