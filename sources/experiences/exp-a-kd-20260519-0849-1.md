## Challenge

One-shot all-reduce performs a single-pass reduce across all ranks, reading from every rank buffer and writing the sum directly to output — efficient on single-GPU but scales poorly on multi-GPU where cross-GPU data movement and barrier synchronization dominate


## Solution

Replace one-shot single-pass reduce with two-shot reduce-then-gather: Phase 1 reduces each rank's responsible portion into a shared buffer (partitioning work across ranks), Phase 2 gathers the reduced portions from all shared buffers into the final output — trading extra memory passes and kernel launches for reduced cross-GPU data movement in multi-GPU setups

## Key Lessons

- Two-shot all-reduce trades single-pass simplicity for reduced cross-GPU data movement by partitioning reduction responsibility across ranks — this regresses on single-GPU (extra memory pass + kernel launch overhead) but benefits multi-GPU where inter-node barrier cost dominates
- All-reduce algorithm selection must match deployment topology: one-shot minimizes memory traffic on single-device, while two-shot (or ring/tree) reduces cross-device synchronization overhead in distributed settings
- Kernel launch overhead and redundant memory passes are hidden costs in multi-phase algorithms — always benchmark end-to-end rather than per-kernel to capture the real performance impact
