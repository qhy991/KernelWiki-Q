## Solution

Replace the scalar strided loop with a per-group float4 vectorized path that processes each group of 32 elements as 8 float4 vectors, calling vec_quant per group in a serial outer loop over 64 groups

## Key Lessons

- Per-group vectorization with small groups (group_n=32, VEC_SIZE=4 → only 8 float4 vectors per group) causes severe thread underutilization (3.1% of 256 threads active per group iteration) that outweighs any memory throughput gain from vectorized loads — applies whenever group_size / VEC_SIZE << blockDim.x
- A scalar strided loop (tid; n < N; n += blockDim.x) already provides naturally coalesced warp-level access for float elements and full thread utilization, making it a hard-to-beat baseline for elementwise operations with per-group scaling
- float4 vectorization for fp32 quantization (4-byte elements) provides only 4x wider transactions versus already-coalesced scalar access; this pattern becomes justified only for narrow types like FP8 (1-byte) where VEC_SIZE=16 yields 16x throughput improvement that can compensate for thread underutilization with larger groups
