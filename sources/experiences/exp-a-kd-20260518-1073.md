## Challenge

The before code uses blockReduceMax with a function-pointer dispatch to detail::_max, performing a warp-level shuffle reduce followed by shared-memory cross-warp aggregation across 32 warps. The after code uses cub::BlockReduce<float,1024>.Reduce with cub::Max functor. The regression is likely cause


## Solution

shared memory for its TempStorage (which includes internal buffers for the multi-stage reduce algorithm across all 1024 threads) compared to the custom implementation's compact shared[maxActiveLanes=32] array. At 1024 threads, CUB's internal algorithm may choose a different reduction tree structure 

## Results

- **speedup**: 0.930x (regression)


## Key Lessons

- The custom implementation's function-pointer overhead for max(a,b) is minimal since nvcc can inline the simple comparison at -O3, so the function-pointer disadvantage does not offset CUB's larger shared-memory footprint and potentially more synchronization stages at this block size.
