## Solution

Convert entry_size and CTA_SIZE into compile-time template parameters (ENTRY_SIZE_T=576, CTA_SIZE_T=64), enabling #pragma unroll to fully unroll the inner copy loop (9 known iterations) and letting the compiler eliminate loop-counter overhead and runtime stride arithmetic, producing straight-line code with better ILP

## Key Lessons

- Convert loop bounds that are effectively constant at deployment into template parameters to give the compiler full unrolling visibility — when ENTRY_SIZE/CTA_SIZE are known #defines, templating them eliminates loop overhead and enables straight-line code generation (1.42x speedup observed for a 576-element copy loop)
- Use a runtime assert(assert(CTA_SIZE_T == blockDim.x)) to guard template-parameter vs actual launch-config mismatches, catching bugs early without affecting the optimized code path
- Loop unrolling is most impactful for short-to-medium fixed-count inner loops (5–20 iterations) where the loop counter and branch overhead represent a significant fraction of total work — for very long loops the code-size increase may hurt instruction cache
