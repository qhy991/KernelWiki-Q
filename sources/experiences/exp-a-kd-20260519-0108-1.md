## Solution

Remove the epilogue synchronization barrier entirely: the last __syncthreads() inside the mainloop already ensures all threads have finished writing shared memory for the final stage, so no additional fence or barrier is needed before writing accumulated results to global memory

## Key Lessons

- When the last iteration of a pipelined mainloop already contains a __syncthreads() that ensures shared memory visibility, any additional epilogue barrier is redundant and can be safely removed — the mainloop barrier subsumes it
- In real CUTLASS code, removing cp_async_wait<0> in addition to __syncthreads() yields meaningful savings because cp_async_wait<0> stalls the async copy pipeline until all pending transactions drain, which is more expensive than a plain barrier
- For very short K dimensions where the kernel runtime is dominated by launch overhead and timer resolution, per-barrier cost differences may fall below measurable noise — benchmark with longer K to expose the savings
