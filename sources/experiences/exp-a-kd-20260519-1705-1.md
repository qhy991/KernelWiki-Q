## Solution

Replace scalar __half loads with 128-bit vectorized _f16Vec<__half,8> loads (8 FP16 values per transaction) combined with packed __half2 arithmetic for add/multiply operations, reducing iteration count from 4096 to 512 and leveraging half2 fused operations for compute throughput while reducing block size from 1024 to 256 threads to improve SM occupancy

## Key Lessons

- Use 128-bit aligned vector types (_f16Vec<T,8>) with reinterpret_cast to coalesce scalar __half loads into single 128-bit global memory transactions, reducing transaction count by the vector width factor — applies to any stride-1 FP16 memory access pattern where data is naturally aligned
- Packed __half2 arithmetic in custom vector struct operators processes two FP16 values per instruction while avoiding per-element float conversion overhead — applies when consecutive half values can be paired into half2 for add, multiply, and sum-of-squares operations
- Reducing block size from 1024 to 256 threads when vectorization reduces per-block work allows 4x more concurrent blocks per SM, improving latency hiding on memory-bound kernels — smaller blocks are preferred when vectorized access already saturates per-thread memory throughput
