## Challenge

MoE block-size alignment kernel launches only 64 threads that must first initialize a 16384-element global-memory array (256 writes per thread at stride 64) before any token counting begins, causing the init phase to dominate execution time as all threads block on global-memory bandwidth before useful shared-memory computation can start


## Solution

Split 320 total threads into 256 fill_threads and 64 working_threads launched in a single block: fill_threads initialize the sorted_token_ids array with wider parallelism (stride 256, 64 iterations per thread instead of 256), while working threads concurrently proceed through shared-memory counting, prefix-sum, and sorting phases, overlapping the previously-serial global-memory init bottleneck with useful computation

## Key Lessons

- Split single-block kernels into dedicated fill threads and working threads to overlap global-memory initialization with shared-memory computation — applies when init writes to a large array independent of counting/sorting logic and the init phase dominates total execution time
- Use wider parallelism (more threads, larger stride) for global-memory fill loops to reduce per-thread iteration count and improve memory-level parallelism, even if those extra threads remain idle after init — the cost of idle warms is negligible compared to overlapping the init bottleneck
