## Solution

Fix the comparison operator from < to > so the clamp correctly caps sh_num_groups when it exceeds the pre-allocated maximum, preventing out-of-bounds shared memory access in the act-order dequant path.

## Key Lessons

- Reversing a comparison operator in a clamp/guard condition (using < where > is intended) silently inverts the safety check — the code runs without crashing on most inputs but produces incorrect results for boundary cases, making the bug hard to detect without targeted testing
- Correctness-only bug fixes (operator typos, off-by-one) do not change instruction count or control flow shape, so expect neutral performance verdicts and focus validation on output correctness rather than latency
