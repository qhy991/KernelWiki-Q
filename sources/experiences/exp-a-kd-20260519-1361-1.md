## Challenge

cuBLAS GEMM with small M=64 dimension calls cublasGemmEx with CUBLAS_OP_N on both operands in a direct (64×K)×(K×N) layout, which may prevent cuBLAS from selecting the optimal internal tile size when the M dimension is much smaller than N


## Solution

Apply swapAB transformation: transpose both operands via CUBLAS_OP_T and swap A/B matrix pointers so that cuBLAS sees the larger N=4096 as the effective M dimension, enabling better tile selection on architectures with fixed compile-time tiles (e.g., SM120 CUTLASS blockwise TMA scheduling forces 128×32×128 instead of 64×128×128)

## Key Lessons

- swapAB (transposing both operands and swapping A/B pointers) can improve tile utilization on architectures with fixed compile-time tile sizes (SM120 CUTLASS), where a small M=64 forces a suboptimal 64×128×128 tile while swapAB enables the larger N dimension to act as M and select a 128×32×128 tile
- cuBLAS on SM80 uses dynamic internal heuristics that already handle small-M problems effectively, so swapAB yields only noise-level improvement (~2%) — the benefit is architecture-dependent and mainly applies to fixed-tile-size backends like CUTLASS
- When benchmarking small-dimension GEMMs, kernel launch overhead can dominate the measured latency, making it difficult to isolate the effect of tile-size optimizations
