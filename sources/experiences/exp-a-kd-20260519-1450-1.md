## Challenge

MoE block-alignment kernel stores per-thread expert token counts in int32_t shared memory, consuming 16900 bytes for a 64-expert configuration — nearly double the necessary footprint since per-thread counts never exceed uint16_t range


## Solution

Template the token count storage type as uint16_t, halving the tokens_cnts shared memory portion from 8448 to 4224 bytes, and reorder shared memory layout to place cumsum first so the uint16_t array starts at a known offset without alignment padding

## Key Lessons

- Reducing shared memory per block only yields speedup when it enables higher occupancy (more concurrent blocks per SM) — for single-block kernels like this <<<1,64>>> launch, halving shared memory has no throughput benefit
- Downsizing shared memory element types (int32_t to uint16_t) can introduce bank conflicts when multiple 16-bit values map into the same 32-bit shared memory bank slot, potentially offsetting any bandwidth gains
- Template parameterization of shared memory types (token_cnts_t) is a good pattern for keeping the optimization switchable and testable, even when the current configuration sees no net benefit
