## Solution

The before kernel's 2D TiledCopy partition computes thread-to-row/col mapping once from threadIdx.x at kernel entry, then uses simple stride-based row*total_K+col addressing for contiguous input reads, whereas the after kernel remaps each output-order position back to input via per-element inverse_interleave_nk calls that cannot be amortized across iterations

## Key Lessons

- Per-element inverse mapping functions that require multiple integer divisions and modulo operations per element are prohibitively expensive for lightweight kernels where arithmetic overhead dominates memory access latency
- 2D thread partitioning where thread-to-row/col mapping is computed once from threadIdx.x and reused across all iterations is cheaper than 1D linear iteration with per-element remapping through inverse functions, even when the 1D approach produces contiguous output writes
- Code clarity refactors targeting generality can regress performance when the original approach exploits compile-time-known thread indices for cheap addressing and stride-based access patterns that avoid per-element division
