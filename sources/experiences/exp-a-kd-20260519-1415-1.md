## Challenge

Dense mma.sync.aligned.m16n8k16 instructions process all 16x8x16 elements including zeros in 2:4 sparse matrices, requiring four separate mma.sync calls (two k-slices across two output columns) to cover the full k=32 dimension and wasting compute on zero entries


## Solution

Replace four dense mma.sync.aligned.m16n8k16 instructions with four sparse mma.sp.sync.aligned.m16n8k32 instructions that leverage Ampere+ 2:4 structured sparsity metadata (frag_m) to skip zero entries, covering k=32 per instruction instead of k=16 and halving the A-matrix operand size from FragA(4 half2) to FragB(2 half2)

## Key Lessons

- mma.sp.sync.aligned.m16n8k32 with 2:4 sparsity metadata doubles effective FLOPs per instruction over dense mma.sync.m16n8k16 at sustained tensor core occupancy, but the advantage requires multi-warp workloads where pipeline fill dominates — single-warp microbenchmarks show negligible improvement due to launch overhead and metadata register pressure
- Sparse MMA reduces A-matrix register footprint (FragA with 4 half2 → FragB with 2 half2) since only non-zero pairs are stored, freeing registers for other purposes in real GEMM tile loops
- The psel parameter in mma.sp selects which partition of the 2:4 sparse matrix to use (0 or 1), requiring two calls per output column to cover both partitions — total instruction count remains the same as dense, but each instruction does twice the useful work on genuinely sparse data
