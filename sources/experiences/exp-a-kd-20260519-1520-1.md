## Solution

Replace the platform-dependent WARP_SIZE=64 macro with a dedicated WARP_SIZE_GGUF=32 hardcoded constant throughout the kernel, halving shared memory per block (tile_y_qs from 4096 to 2048 ints), doubling accumulator rows (sum[2][16] to sum[4][16]), and reducing block thread count from 256 to 128 to match the MMQ algorithm's 32-wide tile operations design

## Key Lessons

- GGUF quantized matmul kernels must use a hardcoded WARP_SIZE_GGUF=32 rather than the platform's WARP_SIZE macro, because the MMQ algorithm's shared memory tiling and accumulator layout are designed around 32-wide operations — using AMD GFX9's native 64-wide warp doubles shared memory and halves accumulator rows without correctness benefit on NVIDIA
- When porting GPU kernels across hardware vendors, any algorithm parameter that encodes hardware warp width must be decoupled from the platform macro and hardcoded to the algorithm's expected value — using the native macro silently changes shared memory footprint, thread count per block, and accumulator geometry
