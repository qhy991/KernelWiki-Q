## Challenge

LSE arrays are stored in transposed [num_heads, num_tokens] layout and accessed as prefix_lse[head_idx * num_tokens + token_idx], producing a stride of num_tokens*4=4096 bytes between adjacent heads for the same token, causing each warp's LSE reads to span 32 disjoint 128-byte cache lines instead of a single coalesced transaction


## Solution

Change LSE memory layout from transposed [num_heads, num_tokens] to row-major [num_tokens, num_heads], accessing via prefix_lse[token_idx * num_heads + head_idx], so that all 32 head entries for a given token occupy a contiguous 128-byte region and a single fully-coalesced warp transaction replaces 32 strided cache-line reads

## Key Lessons

- Transpose array layouts so that the fastest-varying index in memory matches the thread-iteration dimension (head_idx in this kernel) — this collapses strided multi-cache-line reads into a single coalesced transaction, a 5%+ win even for just two float reads per thread
- When threads within a warp iterate over a contiguous dimension (e.g., num_heads=32), ensure the data layout places those elements contiguously (row-major [tokens, heads] not transposed [heads, tokens]) so the 128-byte cache-line granularity is fully utilized
