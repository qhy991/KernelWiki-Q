## Challenge

In the small_batch_64_8exp before code, the two-kernel pipeline launches moe_align_block_size_kernel with 1024 threads (most of which are idle because numel=64) followed by count_and_sort_expert_tokens_kernel with 1 block of 256 threads (also mostly idle). ## Solution

Replace occupancy-rounded waveset formula with fast_max(work_blocks, avail_sms*4), launching only the blocks that perform real work while guaranteeing enough CTAs to saturate all SMs

## Results

- **speedup**: 1.399x (improved)


## Key Lessons

- Rounding grid size up to full gpu_occupancy (sm_occupancy × avail_sms) multiples causes massive over-allocation for multi-occupancy kernels — prefer launching exactly the number of work blocks when it already exceeds a minimum saturation threshold
- Use fast_max(work_blocks, avail_sms * k) with a small constant k to ensure SM saturation without padding to full occupancy multiples — idle CTAs still consume scheduler slots and increase kernel launch latency
- Grid sizing heuristics should treat work_blocks and gpu_occupancy as independent concerns: compute actual work count first, then only floor it to a minimum parallelism target, never round up to arbitrary multiples
