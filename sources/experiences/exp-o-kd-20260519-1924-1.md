## Challenge

Each iteration calls cudaMemcpy with cudaMemcpyDeviceToHost to retrieve a 4-byte scalar scale value from GPU, which synchronizes the CUDA stream and blocks the CPU thread, serializing kernel launches and creating a massive pipeline bubble


## Solution

Pass the scalar as a device pointer (const float* d_scale) and dereference it directly inside the kernel (*d_scale), keeping the value in GPU memory and eliminating the synchronous cudaMemcpy call that was blocking the stream and CPU thread each iteration

## Key Lessons

- Never use cudaMemcpy DeviceToHost to retrieve a scalar that a kernel needs — pass the device pointer and dereference it in-kernel to preserve asynchronous execution and avoid implicit stream synchronization
- cudaMemcpy with cudaMemcpyDeviceToHost acts as an implicit cudaStreamSynchronize, destroying kernel-launch pipelining and serializing the entire host-GPU interaction loop
- Operations like torch::Tensor::item<float>() that trigger device-to-host transfers are extremely expensive in hot loops; restructure kernels to consume device-resident values directly
