## Solution

The after code adds an OOB guard via ThreadMap coordinate decomposition (modulo/division/multiply on row, group, cluster indices) and a per-row conditional fetch check, which ensures correctness on boundary tiles but introduces significant overhead on aligned tiles where no rows are out-of-bounds

## Key Lessons

- OOB guards (ThreadMap coordinate decomposition + conditional fetch) in CUTLASS epilogue reduction are correctness mechanisms for boundary tiles only — enabling them on aligned tiles adds multiple integer operations (modulo, division, multiply) and a data-dependent branch per row with no correctness benefit, degrading performance by over 2x
- CUTLASS's kOobCheck template parameter defaults to false for this reason: the guard overhead is justified only when actual out-of-bounds rows exist (partial tiles at problem boundaries), not as a blanket safety mechanism
