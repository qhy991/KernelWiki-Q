## Challenge

The prefix sum over per-expert padded token counts is computed by a single thread (threadIdx.x == 0) in a serial for-loop iterating over all num_experts, leaving up to 1023 threads idle and creating a serial O(num_experts) bottleneck in the MoE alignment kernel


## Solution

Replace the serial single-thread loop with CUB BlockScan's ExclusiveSum primitive: each of the 1024 threads reads its own expert's padded count, then BlockScan computes the exclusive prefix sum in parallel using a tree-based reduction with O(log(num_experts)) parallel depth, distributing work across all threads instead of one

## Key Lessons

- Replace single-thread serial prefix sum loops with CUB BlockScan ExclusiveSum to distribute work across all block threads — reduces O(n) serial depth to O(log n) parallel depth and eliminates thread idling
- BlockScan infrastructure has non-trivial overhead (shared memory temp storage, tree synchronization) — speedup scales with element count and may be neutral or negative for small counts (e.g., < 16 elements) where the serial loop cost is already negligible
- Move per-element work (CEILDIV padding) out of the cumulative loop and into per-thread pre-processing before the scan — each thread independently computes its padded count, keeping the scan primitive clean and general-purpose
