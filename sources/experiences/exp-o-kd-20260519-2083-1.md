## Solution

Replace the custom shared-memory sequential halving reduction with cub::BlockReduce, which uses warp shuffle instructions (__shfl_down_sync) for intra-warp communication and only falls back to shared memory for the inter-warp final step — eliminating most __syncthreads() barriers and reducing shared-memory footprint

## Key Lessons

- Prefer cub::BlockReduce over hand-written shared-memory sequential halving — it uses warp shuffle instructions for intra-warp steps, avoiding shared-memory traffic and __syncthreads() barriers within warps, which yields significant latency reduction for blocks of 512+ threads
- cub::BlockReduce::TempStorage uses only a small fraction of the shared memory that a full-block shared array (e.g., float[1024]) would require, freeing shared memory for other uses in the kernel
