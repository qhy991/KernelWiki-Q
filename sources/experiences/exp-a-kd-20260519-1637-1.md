## Challenge

Hardcoded FP8_E4M3_MAX constant (448.0f) prevents correct operation on ROCm/AMD where the e4m3fnuz variant uses 224.0f as its maximum representable value, causing incorrect per-token scaling factors in dynamic FP8 quantization


## Solution

Replace the hardcoded FP8_E4M3_MAX macro with a type-dependent fp8_e4m3_adjusted_max_v<fp8_type> template variable that dispatches to the correct max value per FP8 format at compile time (448.0f for e4m3fn on CUDA, 224.0f for e4m3fnuz on ROCm), enabling portable quantization across NVIDIA and AMD platforms with zero runtime overhead

## Key Lessons

- Use type-trait templates (fp8_e4m3_adjusted_max_v<T>) instead of hardcoded macros for format-dependent constants — the same FP8 e4m3 format has different max values across vendors (448.0f for e4m3fn on NVIDIA vs 224.0f for e4m3fnuz on AMD), and compile-time dispatch resolves this at zero runtime cost
- Performance-neutral refactorings that improve cross-platform correctness are still valuable — NVCC produces identical machine code when the template resolves to the same constant, so portability comes free on CUDA while fixing correctness on ROCm
