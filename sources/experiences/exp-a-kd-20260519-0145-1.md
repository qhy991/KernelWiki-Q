## Solution

Replace SM80 FP16 m16n8k16 MMA with SM89 FP8 m16n8k32 MMA that doubles the K dimension per instruction (8192 vs 4096 FLOPs), with an SM80 fallback that decomposes K=32 into two sequential K=16 FP16 MMA ops chaining the accumulator output of the first into the second

## Key Lessons

- SM89 FP8 MMA (m16n8k32) doubles the K dimension per instruction vs SM80 FP16 (m16n8k16), but on SM80 hardware the fallback must issue two sequential FP16 MMA ops — wall-clock latency doubles while per-FLOP throughput stays constant, confirming the MMA pipeline is the bottleneck
- SM89 native FP8 MMA performs both K=16 chunks in a single instruction, yielding approximately 2x throughput improvement over SM80 FP16 for the same 8192 FLOPs — always guard SM89-specific PTX with __CUDA_ARCH__ >= 890 and provide a correct arithmetic fallback for older architectures
- When benchmarking cross-architecture MMA shapes, compare per-FLOP throughput rather than raw latency to isolate hardware capability from instruction count differences
