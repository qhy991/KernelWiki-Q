## Challenge

Batched expert GEMM (M=32, N=3072, K=1024) uses direct (M,K)x(K,N) tiling with tile <64,64,128>, padding M=32 up to 64 and wasting 50% of computed rows, while the large N=3072 dimension incurs heavy per-tile overhead in the 64-wide N tile direction


## Solution

Swap A and B operands (swap_ab) to reinterpret the problem as (N,K)x(K,M) with tile <128,16,256>: the large N=3072 aligns perfectly with 128-wide tiles (24 tiles, zero waste), the small M=32 fits in exactly 2 tiles of width 16, and the larger K_TILE=256 reduces K-loop iterations, cutting padded work from 2x to 1.0x of useful computation

## Key Lessons

- When one GEMM dimension (M) is much smaller than the other (N), swap A and B operands so the large dimension becomes the M-direction of the tile — this eliminates padding waste when the large dimension is a multiple of the tile width, yielding up to 2x reduction in padded computation
- Choose tile dimensions that align with your actual problem sizes: tile width 128 aligns with N=3072 (24 tiles, zero waste) while tile width 64 over-pads M=32 to 64 (50% waste) — always check ceil(dim/tile)*tile vs dim to quantify padding overhead
- A swap_ab strategy (reinterpreting C=A×B as C^T=B^T×A^T) requires only host-side data transposition before/after the kernel, leaving the GPU kernel unchanged — useful when the kernel is already tuned and only the tiling configuration is suboptimal
