## Solution

Replace the shared-memory Blelloch scan with a warp-shuffle (__shfl_up_sync) based two-level exclusive prefix sum: intra-warp scanning uses __shfl_up_sync with no barriers or shared-memory access, and cross-warp aggregation requires only a single warp-0 reduction pass, reducing total __syncthreads() from ~13 to 6 for scan_size=64

## Key Lessons

- Warp shuffle (__shfl_up_sync) based prefix sum eliminates all __syncthreads() barriers within a warp, completing a 32-element inclusive scan in log2(32)=5 shuffle steps with no shared-memory traffic — applies when scan_size fits within a small number of warps (≤32 warps for a single cross-warp reduction pass)
- For scans on data smaller than ~1024 elements (e.g., MoE expert counts of 64–256), a two-level warp-shuffle approach (intra-warp scan → warp-0 cross-warp scan → per-warp offset add) requires only O(log(WARP_SIZE) + num_warps) barriers vs O(2·log(n)) for Blelloch, yielding significant savings when n is small but thread count is large
- Reducing __syncthreads() count directly improves kernel latency because each barrier stalls all threads in the block — for a 1024-thread block doing a 64-element scan, eliminating 7 of 13 barriers (saving each of 1024 threads from waiting) dominates the speedup even when the arithmetic work is unchanged
