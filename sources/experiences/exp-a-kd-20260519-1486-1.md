## Solution

Replace the scalar loop with vectorize_read_with_alignment<16> template that loads 16-float (128-byte) vectors per iteration, reducing per-thread iterations from 16 to 1; however, the DefaultReadVecOp functor unrolls the scalar lambda per element, adding template indirection and register pressure that outweighs the memory-transaction benefit at single-block occupancy

## Key Lessons

- Vectorized loads via template abstraction (vectorize_read_with_alignment<16> wrapping a scalar lambda through DefaultReadVecOp) can degrade performance when the compiler cannot inline the lambda-through-functor chain as effectively as a direct scalar loop — increased register pressure and instruction overhead from template indirection can outweigh the reduced memory-transaction count
- When a kernel runs with a single block and does not saturate memory bandwidth, wider vector loads provide no wall-clock benefit even when coalesced; the baseline scalar stride loop with 256 threads already achieves full coalescing, so the only effect of vectorization is added overhead
- Vectorized reads are beneficial primarily when memory bandwidth is the binding constraint (many blocks, large data volumes); for latency-dominated single-block kernels with modest data sizes, a straightforward scalar loop is often faster due to lower instruction count and better compiler optimization
