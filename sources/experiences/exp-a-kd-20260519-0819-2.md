## Challenge

The launch configuration always uses the token-parallel kernel regardless of actual occupancy, so small batch sizes produce too few thread blocks to fill the GPU, wasting most SM cycles


## Solution

Query runtime occupancy via cudaOccupancyMaxActiveBlocksPerMultiprocessor and cudaDeviceGetAttribute to compute total available CTAs; when token blocks are fewer than available CTAs, switch to a head-parallel kernel that expands the grid along blockIdx.y to fill SMs

## Key Lessons

- Use cudaOccupancyMaxActiveBlocksPerMultiprocessor + cudaDevAttrMultiProcessorCount at runtime to compute total available CTAs and compare against required blocks — when required < available, the kernel is under-utilizing the GPU and an alternative parallelization strategy should be dispatched
- Adaptive dispatch (choosing kernel variants based on occupancy) is especially valuable for operations like RoPE where the work per element is small and the dominant cost is launch overhead and SM starvation rather than compute
