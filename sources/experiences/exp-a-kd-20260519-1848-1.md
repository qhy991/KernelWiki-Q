## Challenge

Float-to-fp8_e4m3 quantization kernel uses a software conversion path (float→half→fp8 via __float2half then __nv_cvt_halfraw_to_fp8), which introduces an extra half-precision intermediary but leverages the high-throughput FP16 ALU pipeline available on SM80+ hardware


## Solution

Replace the two-step software path with the direct __nv_cvt_float_to_fp8 hardware intrinsic, which skips the half intermediary — however, on SM80 the FP16 ALU pipeline servicing __nv_cvt_halfraw_to_fp8 has higher dedicated throughput than the FP32 conversion pipeline used by __nv_cvt_float_to_fp8, causing a regression

## Key Lessons

- Fewer instructions does not guarantee higher performance — the direct __nv_cvt_float_to_fp8 intrinsic routes through the FP32 conversion pipeline which has lower throughput on SM80 than the FP16 ALU pipeline used by the two-step __float2half + __nv_cvt_halfraw_to_fp8 path
- On memory-bound kernels with large payloads, conversion-path throughput differences are hidden by bandwidth saturation; on compute-visible workloads (smaller element counts like 4M), the pipeline disparity becomes the dominant factor — always benchmark on target sizes
- Hardware intrinsic availability does not imply hardware optimization — __nv_cvt_float_to_fp8 may map to a lower-throughput functional unit than the half-to-fp8 conversion, so profiling is essential before replacing an apparent 'software path'
