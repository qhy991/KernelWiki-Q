## Challenge

Two separate kernels (advance_step_base_kernel and advance_step_extra_kernel) are launched sequentially on the same stream to update paged-attention metadata, incurring double kernel launch overhead (~5-10us per launch) and forcing the second kernel to re-read seq_lens_ptr and block_tables_ptr from global memory that the first kernel just wrote


## Solution

Fuse both kernels into a single advance_step_flashinfer_kernel where each thread reads seq_len and block_tables once, then derives all six outputs (tokens, seq_lens, positions, slot_mapping, paged_kv_last_page_len, block_table_bound) from register-held intermediate values, eliminating one kernel launch and all redundant global memory access

## Key Lessons

- Fuse kernels that read overlapping input data and share intermediate computations into a single kernel — eliminates launch overhead and redundant global memory loads, yielding speedups proportional to launch cost relative to compute cost
- When a second kernel re-reads values just written by a first kernel on the same stream, the writes must flush to L2/global memory before the second kernel can read them — fusing keeps values in registers and avoids this unnecessary round-trip through the memory hierarchy
