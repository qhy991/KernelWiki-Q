## Solution

Pad the shared memory array from s_data[32][32] to s_data[32][33], adding one element per row so that each row's base address shifts by one element, changing the bank assignment to (row*33 + col) % 32 which distributes the 32 threads across all 32 banks and eliminates the 32-way bank conflict

## Key Lessons

- Padding shared memory arrays by one element per row (e.g., [32][32] → [32][33]) eliminates bank conflicts when accessing transposed column indices, because the bank assignment (row*33 + col) % 32 distributes accesses across all 32 banks instead of mapping multiple threads to the same bank
- Shared memory bank conflicts cause serialization of memory accesses within a warp — for int elements with 32 banks, a 32-way conflict means all 32 threads in a warp serialize to the same bank, incurring up to 32x slowdown on that memory transaction
