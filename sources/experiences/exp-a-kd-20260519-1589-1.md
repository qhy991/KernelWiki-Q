## Challenge

Fixed 16x16 block size used for all GEMM problem shapes results in only 16 consecutive threads loading from B-matrix columns per block row, limiting memory coalescing width when N is large (e.g., 8192)


## Solution

Problem-size-aware dispatch selects wider block dimensions (32x8) for N>=8192, keeping total threads at 256 but increasing X-dimension to 32 for better B-matrix coalescing; additional dispatch branches handle K>=8192 and small-M cases

## Key Lessons

- Wider X-dimension blocks (32 vs 16) improve coalescing for B-matrix column loads when N is large, but naive GEMM remains fundamentally memory-bound so the speedup is marginal (~2.5%) without shared-memory tiling
- Problem-size-aware block dispatch is a low-effort tuning knob that can be combined with more substantial optimizations (shared memory, register tiling) for compound gains
- Block-dimension tuning that works well with CUTLASS TMA-based async tile loads on SM90 does not translate to equivalent gains in naive kernels on SM80 — the hardware path matters as much as the tile geometry
