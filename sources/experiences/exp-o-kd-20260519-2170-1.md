## Solution

Reduce shared memory allocation from 128KB to 32KB by changing kSmem from 32*1024 to 8*1024, increasing potential occupancy from 1 to 4 blocks per SM; the existing ping-pong double buffering across refinement passes naturally adapts to the smaller per-buffer capacity without algorithm changes

## Key Lessons

- Reducing per-block shared memory from 128KB to 32KB increases SM occupancy from 1 to 4 concurrent blocks on A100-class GPUs, and the resulting higher warp-level parallelism hides memory and instruction latency — applies whenever shared memory usage is the occupancy bottleneck on GPUs with 128KB+ shared memory capacity
- Ping-pong double buffering across radix refinement passes enables reducing per-buffer capacity without algorithm changes, since the alternating src_buffer/dst_buffer pattern naturally adapts to smaller intermediate storage — applies to multi-pass radix or partition algorithms where intermediate state can be split across alternating buffers
- A single constant change (kSmem) can yield measurable speedup when it unlocks occupancy that was previously constrained by shared memory allocation, provided the algorithm already has a buffering strategy that tolerates smaller per-pass capacity
