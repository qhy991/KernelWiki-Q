## Solution

Replacing the two-step half-intermediary path with the single-step __nv_cvt_float_to_fp8 intrinsic, which routes directly through the FP32-to-FP8 hardware pipeline — but this direct path has higher instruction latency and lower throughput on SM80 (A100/A800) compared to the well-optimized FP16 ALU path

## Key Lessons

- The direct __nv_cvt_float_to_fp8 intrinsic is not universally faster than the half-intermediary path — on SM80 (A100/A800) the FP16 execution units have higher throughput and shorter pipeline latency, making the two-step float→half→fp8 path faster despite involving an extra conversion
- Even in memory-bound kernels, instruction latency matters when the conversion pipeline is long enough that it cannot be fully hidden by memory stalls — always benchmark rather than assuming fewer instructions means faster execution
- Architecture-specific performance characteristics dominate conversion-path choices: the same intrinsic that wins on one SM generation may regress on another, so fp8 conversion strategy should be tuned per-target architecture
