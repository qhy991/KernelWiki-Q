## Challenge

Scale-factor offset computation uses integer division and modulo (mIdx/128, mIdx%32, kIdx/4, kIdx%4) with six separate stride variables and five multiply-add terms, producing a long dependency chain of arithmetic instructions per offset call


## Solution

Replace all division/modulo with bitwise shift/AND equivalents (>>7 for /128, &31 for %32, >>5 for /32, >>2 for /4, &3 for %4), fold the six stride constants into a single bit-pack expression using left-shift and OR, and pass precomputed numKTiles as a parameter to avoid recomputing it per call

## Key Lessons

- Manually replacing power-of-2 division/modulo with bitwise shift/AND yields negligible speedup when NVCC -O3 already performs this transformation — profile before optimizing arithmetic that the compiler handles
- Folding stride multiplications into a single bit-pack expression (via left-shift and OR) reduces source-level complexity but does not improve runtime when the original stride computation is memory-bandwidth-bound and fully hidden by latency
- Preferring __forceinline__ and precomputed parameters (numKTiles passed in vs recomputed from numCols) is good practice but only matters when the function is a true hot-path — here the kernel is dominated by global memory accesses to input/output/sf_out buffers
