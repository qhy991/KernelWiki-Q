## Solution

Fuse all four kernels into a single init_offsets_fused_kernel that writes stride values as three additional scalar stores per thread alongside existing offset computation, eliminating 3 kernel launch dispatches whose combined ~15us overhead exceeded the actual GPU compute time

## Key Lessons

- Fuse trivial fill/initialization kernels into existing compute kernels when per-kernel launch overhead (~3-5us driver dispatch) dominates actual GPU work — the added scalar stores cost negligible instruction overhead but eliminate entire kernel launch pipelines
- Kernel launch overhead is a fixed per-dispatch cost independent of grid size, so it disproportionately impacts small kernels (few threads, little compute) — always audit the ratio of launch overhead to compute time for utility kernels in latency-sensitive pipelines
- torch::full() and similar framework convenience functions each translate to a separate GPU kernel launch; in grouped/mixture-of-experts workloads with many small buffers, replacing multiple torch::full() calls with inline stores in an existing kernel can halve end-to-end latency
