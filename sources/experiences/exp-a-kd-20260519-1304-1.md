## Solution

Replace the O(N) sequential scan with O(log N) binary search via findTotalEltsLessThanTarget, exploiting the sorted property of the expert array to locate the boundary index with ~18 random reads instead of up to 262K sequential reads per thread

## Key Lessons

- Replace O(N) linear scans with O(log N) binary search when operating on sorted arrays — the logarithmic reduction in memory reads dominates all other factors, yielding orders-of-magnitude speedup for large N
- When each thread independently searches a sorted array for a different boundary value, binary search avoids the redundant sequential memory traffic that linear scans produce across threads, since each thread reads only ~log2(N) scattered locations instead of sweeping through up to N elements
- Exploiting known data properties (sorted order) to select the right algorithm is far more impactful than micro-optimizing the wrong algorithm — no amount of warp-level tuning can recover from a fundamentally O(N) per-thread approach when O(log N) is available
