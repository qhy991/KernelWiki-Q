## Solution

Replace scalar half load/store with half2 (32-bit) vectorized access so each thread processes 2 adjacent half values, fully filling coalesced 32-bit memory segments and doubling per-thread throughput; computation is unpacked into .x/.y lanes for per-element GELU since no native half2 tanh exists

## Key Lessons

- Use half2 vectorized load/store instead of scalar half access in elementwise kernels operating on fp16 data — halves only fill 16 bits of a 32-bit coalesced segment, so pairing two into half2 fully utilizes each memory transaction and can halve total transactions
- When no native half2 math intrinsic exists (e.g., half2 tanh for GELU), unpack half2 into .x/.y lanes, compute per-half with scalar half intrinsics, then repack — this still gains the memory throughput benefit of vectorized loads/stores
- Scale the thread index by 2 (threadIdx * 2) when switching from scalar to half2 so the same grid covers twice the elements without increasing thread count
