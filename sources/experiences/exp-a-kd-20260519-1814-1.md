## Challenge

The get_kernel_cache_size function hardcodes 2 bytes per A-matrix element via pipe_stages * (tb_m * tb_k) * 2, always assuming FP16/BF16 activations; when INT8 activations are used (1 byte per element), this over-allocates shared memory by 2x for the A-matrix tile, potentially preventing valid thread configurations from passing the shared memory budget check on constrained GPUs


## Solution

Replace the hardcoded 2-byte multiplier with a runtime conditional (is_a_8bit ? 1 : 2) and parameterize pipeline stages as a function argument instead of a global constant, enabling correct shared memory sizing for both FP16 and INT8 activations with zero runtime overhead since the conditional resolves during static shared memory computation

## Key Lessons

- Use runtime conditionals for data-type-dependent shared memory sizing (is_a_8bit ? 1 : 2) instead of hardcoded constants — the ternary has zero cost in host-side shared memory computation and enables correct resource usage across mixed-precision kernels
- Parameterize pipeline stages as a function argument rather than using a global constant — allows the same kernel launch logic to adapt shared memory requirements across GPU architectures with different shared memory budgets (e.g., 4 stages on Ampere vs 2 on Turing)
