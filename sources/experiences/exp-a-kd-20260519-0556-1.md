## Challenge

RoPE kernel uses scalar float loads and stores (two separate 32-bit reads and two writes per thread), resulting in low memory throughput per issued instruction and underutilizing 128-bit memory bus width


## Solution

Replace two scalar float loads and two scalar stores with a single float4 (128-bit) vectorized load/store, processing two consecutive RoPE pairs per thread and halving the number of memory transactions

## Key Lessons

- Using float4 vectorized loads/stores in RoPE kernels halves the number of memory transactions per thread by coalescing four scalar floats into one 128-bit transaction — applies to any elementwise or pairwise kernel where input elements are contiguous in memory
- Processing multiple RoPE pairs per thread via float4 packing also halves total thread count for the same workload, reducing scheduling overhead and increasing work per thread, which is especially beneficial at larger hidden dimensions where memory bandwidth dominates
- Each pair in the float4 still needs its own frequency exponent (exp_f_v, exp_s_v) because adjacent pairs correspond to different dimension indices — do not reuse the same trig values across all four components
