## Challenge

Two separate kernel launches — one for fp8-to-float conversion and another for scale multiplication — force an intermediate global memory write and read of a 4N-byte float buffer, doubling memory traffic and adding kernel launch overhead


## Solution

Fuse the fp8-to-float conversion and scale multiplication into a single kernel where scaled_vec_conversion_fp8_to_float performs the full fp8→half→float chain and immediately multiplies by scale, eliminating one kernel launch, the intermediate 4N-byte global memory buffer, and its associated read/write traffic

## Key Lessons

- Fuse type-conversion and scaling into a single kernel when the intermediate result is never reused — eliminates one kernel launch, one global memory allocation, and one full read/write round-trip (4N bytes saved for float intermediates)
- Per-element arithmetic (fp8→half→float + scale multiply) is identical whether split across two kernels or fused into one; the speedup comes entirely from reduced memory traffic and launch overhead, not from instruction-level optimization
- When a conversion pipeline produces a temporary that is only consumed by a single subsequent elementwise operation, fusing them is almost always a net win on bandwidth-limited workloads
