## Solution

Attempt to vectorize each thread to process 4 consecutive f32 elements (f32x4), reducing total thread count per block from 1024 to 256 in hopes of improving memory throughput — but this reduces occupancy too aggressively, leaving insufficient active warps to hide global memory latency

## Key Lessons

- Reducing thread count to increase per-thread work can regress performance when it drops occupancy below the threshold needed to hide memory latency — always check that the new thread count still provides enough active warps (typically >= 256 threads/block for memory-bound kernels)
- Vectorization by unrolling scalar accesses (output[tx+0..3]) without using actual vectorized load/store instructions (float4/ld.global.v4) wastes the potential benefit: the compiler emits separate load/store transactions anyway, so no memory bandwidth is saved
- For memory-bound embedding lookup kernels where each thread already performs independent coalesced accesses, the scalar 1-thread-1-element mapping is often near-optimal because it maximizes both occupancy and coalescing without register pressure increases
