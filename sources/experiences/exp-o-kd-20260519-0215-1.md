## Challenge

When split_kv > 1, a separate reduction kernel is always launched regardless of whether fused reduction (atomic accumulation inside the main kernel) is already active, causing double reduction that overwrites correct results and wastes a full kernel launch worth of GPU work


## Solution

Guard the separate reduction kernel launch with !is_fused_reduction so it only runs when fused atomic accumulation is not already handling the reduction inside the main kernel, eliminating redundant GPU work and fixing the correctness bug

## Key Lessons

- When a kernel supports fused reduction via atomic accumulation, the separate reduction kernel launch must be mutually exclusive with the fused path — add the fused-reduction flag to the guard condition to avoid double reduction and wasted kernel launches
- Redundant kernel launches that overwrite correct results are both a correctness bug and a performance bottleneck; eliminating them saves launch overhead, GPU compute, and synchronization cost
