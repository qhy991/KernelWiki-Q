## Challenge

Per-token FP8 quantization kernel reads input data twice from global memory (once for max-reduction in pass-1, again for quantization in pass-2), causing redundant global memory traffic when data does not remain cache-resident between the two passes


## Solution

Stage input data to shared memory during pass-1 (alongside max computation), then read from shared memory during pass-2 quantization, eliminating the redundant global memory reload; includes bank-conflict padding and __syncwarp() barrier between passes

## Key Lessons

- Shared memory staging for two-pass kernels provides neutral performance when the working set (4MB for 2048 tokens × 1024 fp16) fits entirely in L2 cache (40MB on A800) — the L2 hit latency approximates shared memory access latency, negating the benefit
- Element-wise shared memory stores (scalar writes per element in pass-1) and __syncwarp() barrier overhead can erase the latency savings from avoiding global reload, especially when vectorized uint4 loads from global memory are replaced by non-vectorized per-element smem writes
- Shared memory staging becomes beneficial when L2 cache cannot hold the working set between passes (e.g., larger hidden dimensions, GPUs with smaller L2 caches, or concurrent kernels competing for L2 capacity) — evaluate cache residency before applying this pattern
