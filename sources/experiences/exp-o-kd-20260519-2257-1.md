## Solution

Fuse all three per-query metadata updates (token copy, seq_len/position increment, slot mapping from block tables) into a single advance_step_kernel that keeps seq_len in a register and reuses next_input_pos for both position write and slot mapping computation, eliminating 2 of 3 kernel launch overheads and removing the inter-kernel global memory round-trip for input_positions

## Key Lessons

- For small query counts where per-thread work is only a few scalar memory ops, kernel launch overhead dominates total latency — fuse adjacent kernels that operate on the same per-element domain into a single kernel to eliminate launch costs
- Fusing kernels that share intermediate values (seq_len → next_input_pos → slot mapping) allows register reuse instead of writing to and re-reading from global memory between kernels, removing inter-kernel data dependencies and global memory round-trips
- When kernel grid size is 1 block (ceil(64/256)=1), each launch pays full driver dispatch cost for nearly zero compute — consolidate such micro-kernels even if the fused kernel has no arithmetic savings
