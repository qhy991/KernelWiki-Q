## Challenge

Warp index computation inside the GEMM iteration loop uses __shfl_sync(0xffffffff, threadIdx.x / 32, 0), which forces an implicit warp convergence barrier before the value is available, adding a SHFL instruction and serialization overhead per iteration even though the warp index is thread-invariant across iterations


## Solution

Replace __shfl_sync-based warp index computation with canonical_warp_idx() (plain threadIdx.x / 32), eliminating the per-iteration SHFL instruction and implicit warp barrier; the compiler can then fold the invariant division into register reuse since the value does not change across iterations

## Key Lessons

- Use plain arithmetic (threadIdx.x / 32) instead of __shfl_sync for warp-invariant values — __shfl_sync forces an implicit warp convergence barrier that serializes execution, while a direct division has no such overhead and the compiler can reuse the register across iterations
- Eliminating a single SHFL per iteration produces negligible speedup when the inner compute loop (K iterations of FMA + memory access) dominates total execution time — measure before and after to confirm the optimization is not hidden by noise
