## Challenge

SiLU activation and elementwise multiply are implemented as two separate CUDA kernels communicating through an intermediate bfloat16 global memory buffer, causing a full global memory round-trip per element (write by kernel 1, read by kernel 2) plus overhead of two kernel launches


## Solution

Fuse SiLU and Mul into a single kernel (kernel_fused_silu_mul_bf16) that reads gate and up vectors once, computes SiLU(gate) * up entirely in registers, and writes the result directly to output, eliminating the intermediate global memory buffer and one kernel launch

## Key Lessons

- Fuse elementwise operations (SiLU + Mul) into a single kernel when they form a producer-consumer chain — intermediate values stay in registers instead of round-tripping through global memory, eliminating both memory traffic and one kernel launch overhead
- Removing an intermediate global memory buffer saves memory allocation cost and DRAM bandwidth; for short producer-consumer kernel chains the saved global memory traffic dominates the speedup
- Use bfloat162 (packed pair) vector loads/stores when operating on bf16 data to halve the number of memory transactions — applies to any elementwise bf16 kernel operating on contiguous aligned data
