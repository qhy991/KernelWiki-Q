## Challenge

Optimization focus: Added vectorized FP16/BF16 kernel specialization for fused_add_rms_norm using _f16Vec struct with 128-bit aligned memory access and packed half2/bfloat162 arithmetic.
Problem before: The original fused_add_rms_norm_kernel accessed global memory one scalar FP16/BF16 element at a t


## Solution

ses into wider vector loads, and packed half2/bfloat162 arithmetic was not exploited, so each add/multiply operated on a single 16-bit value.
Applied change: Introduced _f16Vec<scalar_t,width> (alignas(16) POD struct holding width native half/bfloat16 values) with overloaded +=, *=, *=float, and sum

## Key Lessons

- Each thread issued individual load/store instructions per element, resulting in poor memory bandwidth utilization because each 2-byte load could not saturate the 128-byte memory transaction granularity.
