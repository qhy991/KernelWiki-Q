## Solution

Split-KV factor of 2 distributes 256 KV tokens across two partial attention kernels (each processing a 128-token chunk), launching 128+64=192 total thread blocks versus 64 for single-pass, which better saturates 108 SMs at small batch sizes despite the extra reduction kernel overhead

## Key Lessons

- Split-KV parallelism (distributing KV tokens across multiple thread blocks with a separate reduction) increases total thread block count and improves SM occupancy for small-batch decode workloads where BATCH*HEADS is well below the SM count — the extra reduction kernel overhead is amortized by better hardware utilization
- Removing split-KV in favor of a single-pass kernel reduces launch overhead and code complexity but can regress performance at small scale where the dominant bottleneck is SM under-utilization rather than kernel launch or memory traffic
