## Solution

Widen the per-thread vector size to 16 fp8 elements (128 bits) and replace the scalar output store loop with a single uint4 coalesced store, while loading 4×float4 input chunks to feed the larger vector — enabling one 128-bit STG per iteration instead of four 32-bit stores

## Key Lessons

- Replacing scalar output stores with uint4 coalesced 128-bit stores improves memory throughput for narrow-type (fp8/int4) kernels, but gains are modest (~4%) when the fp8 conversion arithmetic dominates execution time — applies when the compute-to-memory ratio is not heavily memory-bound
- Type-dependent vec_size (16/sizeof(T)) undersizes the output vector for narrow types like fp8 where 16 elements = only 16 bytes = one uint4 transaction — hardcoding VEC_SIZE=16 for the output side maximizes store coalescing regardless of input type
- Scalar stores from consecutive threads targeting adjacent addresses get partially coalesced by the L2 controller at the hardware level, so the gap between manual vectorization and hardware coalescing is smaller than theoretical analysis suggests
