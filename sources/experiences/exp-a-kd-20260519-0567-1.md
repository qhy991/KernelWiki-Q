## Challenge

The single-buffered GEMM kernel uses one shared memory tile per operand, requiring two __syncthreads() barriers per K-tile iteration: one after loading into smem and one after compute, which serializes global memory loads behind computation and leaves the memory controller idle during the FFMA phase


## Solution

Double buffering with a 2-stage software pipeline: allocate s_a[2][BK][BM] and s_b[2][BK][BN] so that the next tile's global load is issued into registers before computing on the current buffer, allowing the GPU memory controller to service the load in parallel with FFMA execution; only one __syncthreads() is needed per iteration since load and compute target different smem buffers

## Key Lessons

- Double buffering (2-stage software pipeline) overlaps global memory loads with compute by issuing the next tile's gmem load into registers before computing on the current smem buffer, letting the memory controller work in parallel with FFMA units — applies to any tiled kernel where compute and memory load are serialized by __syncthreads() barriers
- Using two smem buffers (indexed by bk & 1) eliminates the need for a second __syncthreads() per iteration because the next tile writes to a different buffer than the one being read by compute, reducing barrier synchronization overhead by roughly half
- The prologue/epilogue pattern (load first tile before the loop, compute last tile after the loop) is the standard structure for N-stage software pipelines in CUDA — the loop body handles stage N while the prologue primes stage 1
