## Challenge

In the inner K-loop of a wmma-based GEMM tile, fragments are loaded from shared memory and transformed before issuing mma_sync, meaning the warp must finish all data movement before the Tensor Core can begin computation, serializing copy and compute phases


## Solution

Reorder the loop to issue mma_sync first with the current fragment, then prefetch the next k_block's fragments afterward (software pipelining), allowing asynchronous MMA hardware (Hopper GMMA) to overlap computation with data preparation for the next iteration

## Key Lessons

- Issuing MMA before the next data copy (software pipelining) only yields speedup on hardware with asynchronous tensor core instructions (Hopper GMMA via warpgroup_arrive/commit_batch) — on Ampere where wmma::mma_sync is synchronous and blocks the warp, the reorder has no effect because the warp stalls regardless of instruction order
- The CUTLASS-style dispatch reorder (warpgroup_arrive -> gemm -> commit -> copy -> transform) maps to Hopper's async GMMA pipeline; porting the same pattern to wmma on pre-Hopper architectures produces neutral results since wmma lacks the underlying async execution model
- Software pipelining of MMA + data prefetch requires the MMA instruction to be non-blocking from the warp's perspective — verify the target architecture's MMA execution model before applying this scheduling optimization
