## Solution

Cache all expert offsets into shared memory via cooperative int4-vectorized loading at block start, then replace the linear scan with binary search over the shared-memory array — reducing comparisons from O(n) to O(log n) and each comparison from global memory (~400ns) to shared memory (~5ns) latency

## Key Lessons

- Cache small read-only lookup tables (expert offsets, routing tables) in shared memory with cooperative int4-vectorized loading to eliminate repeated global memory accesses — the one-time __syncthreads cost is amortized across all elements processed by the block
- Replace linear O(n) table scans with binary search O(log n) when the table is sorted — for 64 experts this cuts comparisons from 64 to 6, and when the table is in shared memory each comparison is ~80x faster than global memory
- Use int4 vectorized loads (16 bytes per thread per iteration) for cooperative shared memory population to minimize the number of load transactions and improve memory throughput when copying arrays of 4+ elements
