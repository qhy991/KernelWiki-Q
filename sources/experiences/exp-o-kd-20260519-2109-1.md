## Solution

Add a runtime heuristic in the dispatch function: if k > 3*n, use StreamKScheduler which splits tiles across SMs for better occupancy; otherwise use PersistentScheduler which is optimal when there are enough N-dimension tiles to keep all SMs busy

## Key Lessons

- Use StreamKScheduler over PersistentScheduler when K >> N (specifically k > 3*n) in FP8 blockwise GEMM — StreamK splits tiles across SMs to improve occupancy when there are too few persistent tiles in the N dimension, yielding up to 15x speedup for tall-and-skinny problem shapes
- Problem-shape-aware dispatch with a simple threshold heuristic provides the best of both scheduling strategies with negligible overhead — PersistentScheduler for balanced shapes and StreamK for tall-and-skinny shapes, avoiding one-size-fits-all scheduling
