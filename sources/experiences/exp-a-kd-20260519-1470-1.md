## Solution

Control case: both before and after use the identical GemmShape<128,128,64> tile configuration. For M=256, the tile M dimension of 128 is fully utilized with no padding waste since 256 is an exact multiple of 128, making the default config already near-optimal

## Key Lessons

- For CUTLASS int8 tensor-core GEMM on SM80, when M is a multiple of the tile M dimension (here M=256, tile M=128), the default GemmShape<128,128,64> achieves full tile utilization with zero padding waste — further tile-shape tuning yields no measurable improvement
- CUDA event microbenchmark latency differences below ~1% at problem sizes like M=256,N=4096,K=4096 fall within measurement noise and should not be interpreted as real performance changes
