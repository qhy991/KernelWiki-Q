## Challenge

Integer zero-points are loaded from shared memory via a per-element scalar int loop (N iterations of frag_qzp[k%2][i] = reinterpret_cast<int*>(sh_zp)[zp_sh_rd + i]), requiring N individual shared-memory reads plus a subsequent dequantization step (unpack + type conversion) per thread per k-iteration in the hot matmul loop


## Solution

Add a dedicated is_zp_float code path that loads the entire 16-byte zero-point fragment as a single int4 vectorized load into frag_zpf[k%2] via reinterpret_cast, bypassing the frag_qzp array and the dequant step entirely since fp16 values are already in the correct compute format

## Key Lessons

- When zero-points are already in the target compute format (e.g., fp16), load them directly via int4 vectorized loads into frag_zpf instead of going through an intermediate int array (frag_qzp) and a dequant step — eliminates both shared-memory load instructions and arithmetic in the hot matmul loop, yielding up to 7.8x speedup on the ZP fetch path
- Split zero-point handling by data format (is_zp_float) to exploit format-specific optimizations: float ZPs can skip dequant entirely via a single vectorized load, while integer ZPs retain the existing per-element unpack pipeline
