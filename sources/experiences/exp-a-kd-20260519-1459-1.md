## Challenge

Two-pass softmax requires a separate BF16→float conversion kernel that writes 256KB to an intermediate global memory buffer, followed by a second softmax kernel that reads it back, doubling memory traffic and adding kernel launch overhead that dominates when ncols is small (8 experts)


## Solution

Fuse BF16→float conversion directly into the softmax kernel by calling __bfloat162float() inline in each of the three softmax loops (max-reduction, exp-sum, output-write), eliminating the intermediate buffer allocation and one entire kernel launch

## Key Lessons

- Fusing a type-conversion kernel into its consumer eliminates an intermediate global memory round-trip and one kernel launch — especially beneficial when the per-row work is small (e.g., softmax over 8 experts) so that kernel launch overhead and buffer traffic dominate total time
- Inline __bfloat162float() is essentially free on modern GPUs (single HFMA2 pipeline instruction) and can be repeated in multiple loops without materializing the converted values to global memory
