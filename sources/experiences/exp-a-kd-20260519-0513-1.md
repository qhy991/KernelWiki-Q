## Solution

Attempted f16x8 expansion: each thread loads 4 half2 registers and processes 8 elements to increase per-thread work, but the resulting 32-thread blocks (down from 128) drastically reduce occupancy and introduce 4 separate boundary checks causing warp divergence within warps

## Key Lessons

- Increasing per-thread element count without maintaining sufficient thread count reduces occupancy — for memory-bound elementwise kernels, 128 threads/block (f16x2) provides better latency hiding than 32 threads/block (f16x8)
- Multiple per-element boundary checks (4 separate if-guards in f16x8) introduce warp divergence when some threads in a warp pass different checks — a single consolidated guard is preferable
- f16x8 expansion can benefit compute-bound kernels but hurts memory-bound elementwise activations where the GPU needs many active warps to pipeline global memory accesses
