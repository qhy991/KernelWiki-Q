## Challenge

A vectorized RMS computation loop is left without #pragma unroll, relying on the compiler's default unrolling decisions for a loop that loads vec4 elements and accumulates squared sums


## Solution

Adding #pragma unroll 4 to the vectorized reduction loop has no measurable effect because with hidden_size=4096 and 1024 threads, num_vec_elems=1024 and each thread executes exactly one iteration (stride equals count), leaving the compiler nothing to unroll

## Key Lessons

- #pragma unroll is ineffective when each thread executes at most one loop iteration (stride equals trip count) — unrolling only helps when multiple iterations exist for the compiler to overlap and eliminate branch overhead
- Before applying loop unroll pragmas, verify the iteration-to-thread ratio: when blockDim.x >= num_elements / vector_width, each thread has a single-iteration loop and unrolling cannot help
- Loop unroll optimizations require multiple iterations per thread to eliminate branch overhead and enable instruction-level parallelism across iteration bodies — measure the actual trip count per thread before expecting gains
