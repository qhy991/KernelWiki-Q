## Challenge

Two back-to-back FP16 GEMM kernels execute as separate launches: GEMM0's epilogue writes a 512x64 FP16 intermediate result (64 KB) to global memory, then GEMM1 reads it back as its A operand, incurring two global memory round-trips and a second kernel launch overhead


## Solution

Replace two separate CUTLASS Gemm kernels with a single B2bGemm using register-file (RF) residency: GEMM0's accumulator fragment stays in registers, FragmentIteratorA1 slices it in-place applying EpilogueOutputOp0 (alpha*scale + bias + ReLU) during load, and feeds transformed sub-fragments directly to GEMM1's warp-level MMA without any intermediate global memory write or read

## Key Lessons

- For back-to-back GEMMs where one's output feeds the next's input, fusing into a single B2bGemm kernel with RF residency eliminates intermediate global memory traffic and kernel launch overhead — applies when the intermediate tensor fits in the register file across warps
- Widening WarpShape0 (32x32x32 to 32x64x32) aligns the accumulator layout with GEMM1's K dimension so FragmentIteratorA1 can slice registers in-place without rearrangement — this increases per-warp register pressure but avoids shared-memory staging
- Switching EpilogueOutputOp0 from NoBetaScaling to OnlyAlphaScaling (with beta=0) signals that GEMM0's output has no residual contribution, allowing the fused kernel to skip the C0 read in the epilogue
