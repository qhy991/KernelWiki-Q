## Challenge

Per-tensor scale extraction via cudaMemcpy DeviceToHost between two back-to-back kernel launches forces a full GPU pipeline drain, blocking the host thread and creating a gap on the GPU timeline where no work executes — for small kernels the ~2.4ms sync overhead dominates total latency


## Solution

Keep the scale value on device memory and pass a device pointer (const float* d_scale_ptr) to the downstream kernel, which reads the scalar via pointer dereference inside the kernel body — both kernels are enqueued back-to-back on the same stream with zero host-side blocking

## Key Lessons

- Replace per-tensor scalar extraction (cudaMemcpy D2H / .item<float>()) with device-pointer passthrough so the downstream kernel reads the value directly from device memory — eliminates an implicit stream sync that drains the entire GPU pipeline for a single 4-byte copy
- When chaining small kernels on the same stream, any host-blocking operation (D2H memcpy, cuStreamSynchronize) between launches creates a pipeline gap orders of magnitude larger than the kernel execution itself — keep all intermediate data on device to maintain back-to-back kernel enqueue
- The device-pointer broadcast pattern (kernel reads *d_scale_ptr) mirrors CUTLASS Sm90ColOrScalarBroadcast and is the idiomatic way to handle per-tensor scalars without host synchronization
