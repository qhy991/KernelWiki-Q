## Key Lessons

- The uint16_t change halves the tokens_cnts shared memory portion but since this kernel runs as a single block <<<1, 64, ...>>> with no multi-block occupancy concerns, the reduced shared memory footprint does not translate into higher occupancy or more concurrent work.
- The uint16_t type also introduces type-punning via (uint16_t*)(shared_mem + blockDim.x + 1) which maps two 16-bit values into each 32-bit shared memory bank slot, potentially increasing bank conflicts on the 2D tokens_cnts access pattern compared to the straightforward int32_t layout.
