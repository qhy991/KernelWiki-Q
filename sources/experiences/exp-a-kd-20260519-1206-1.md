## Challenge

Shared memory XOR swizzle pattern uses full row index (col ^ row) for bank-conflict avoidance, but only the lower 3 bits of row affect bank selection for stride-8 accesses on 32-bank shared memory, making the full-row XOR equivalent in effect to a truncated version


## Solution

Replace full row XOR (col ^ row) with truncated row XOR (col ^ (row % 8)) — since stride=8 maps columns across 32 4-byte banks, only the lower 3 bits of row influence bank selection, making both patterns equivalent for bank-conflict avoidance while the after version adds a modulo operation that is negligible at runtime and can be eliminated via precomputation in production kernels like Marlin's transform_a

## Key Lessons

- For stride-8 shared memory layouts on 32-bank architectures, only the lower 3 bits of the row index affect bank selection via XOR swizzle — full row XOR provides the same bank-conflict avoidance as row % 8 XOR, so the truncated form is preferred for clarity
- Runtime modulo operations (row % 8) add ~1 ALU cycle overhead per access, which is negligible compared to shared memory read latency (~1-2 cycles), and is fully eliminated when swizzle indices are precomputed at compile time as in Marlin's transform_a function
