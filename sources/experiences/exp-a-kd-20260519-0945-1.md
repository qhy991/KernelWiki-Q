## Challenge

RoPE kernel writes post-rotation K values to global memory (k_rope), then a separate kv_writeback kernel re-reads those same K values from global memory to copy into k_buffer — causing a redundant global memory round-trip and an extra kernel launch with its argument-setup and enqueue overhead


## Solution

Fuse RoPE and KV write-back into a single kernel: after computing post-RoPE K values (y1, y2), keep them in registers and write directly to both k_rope and k_buffer, then load V and write to v_buffer — eliminating the redundant global memory read of K and removing one kernel launch overhead

## Key Lessons

- Fuse producer-consumer kernel pairs where the consumer re-reads data the producer just wrote to global memory — keeping intermediate results in registers eliminates a full global memory round-trip (read + write) and is especially impactful for small batches where kernel launch overhead is a significant fraction of total latency
- Kernel launch overhead (argument setup, enqueue, stream synchronization) is a fixed cost per launch; for small workloads (few tokens, low occupancy), eliminating one launch can yield 20-40% end-to-end speedup as seen here with nnz=8
- When fusing kernels, merge the consumer's work into the producer's else-branch if they operate on disjoint head groups (here: Q heads vs K/V heads), avoiding warp divergence while keeping both paths in the same kernel
