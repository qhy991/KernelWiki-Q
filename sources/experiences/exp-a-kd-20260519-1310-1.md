## Challenge

MoE routing finalization kernel loads one float at a time from expanded_permuted_rows in the inner k-loop, issuing 8*2048=16384 scalar 4-byte global loads per thread; output stores are similarly scalar, underutilizing memory bandwidth on a bandwidth-bound kernel


## Solution

Reinterpret float pointers as uint4 (128-bit) to load and store 4 floats per instruction, reducing total memory instructions by 4x: loads drop from 8*2048 to 8*512 and stores from 2048 to 512, dramatically improving bandwidth utilization on this memory-bound kernel

## Key Lessons

- Reinterpret float* as uint4* (128-bit) to issue 4-wide vectorized loads/stores — reduces memory instruction count by 4x and significantly improves bandwidth utilization on memory-bound kernels where the working set far exceeds cache capacity
- Vectorize both loads and stores together in bandwidth-bound reduction loops — the 4x reduction applies to both sides of the memory pipeline, compounding the throughput gain
- Use reinterpret_cast<const uint4*>(float_ptr) with cols/4 loop stride for zero-alignment-overhead vectorization when the column count is known to be a multiple of 4 at compile time
