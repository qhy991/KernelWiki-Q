## Challenge

A synchronous int4 global-to-shared bias copy followed by a single __syncthreads() blocks the entire threadblock during the global load, but replacing it with cp.async adds three extra synchronization points (fence + intermediate syncthreads + wait + syncthreads) whose overhead exceeds the benefit when insufficient compute overlaps the async window


## Solution

Replace the synchronous int4 copy with cp.async.ca.shared.global PTX instruction plus cp_async_fence()/cp_async_wait<0>() to issue a non-blocking global-to-shared copy, but in this simplified benchmark the async window contains only one __syncthreads() of intermediate work — insufficient to hide the async barrier overhead, causing a 16.5% regression

## Key Lessons

- cp.async overlap only yields speedup when the async window (between issue and wait) contains enough concurrent compute (e.g., barrier_acquire/release, scale loading, dequantization) to hide the copy latency — with trivial intermediate work the fence/wait barrier overhead dominates and causes regression
- Introducing cp.async converts one __syncthreads() into three synchronization points (fence + intermediate syncthreads + wait + syncthreads); always verify that the overlapped work amortizes this increased synchronization cost
- In real kernels like Marlin where the gap between cp_async issue and wait contains substantial per-column scale loading and dequantization, cp.async provides meaningful overlap; in micro-benchmarks with minimal intermediate work the same pattern regresses — always profile the actual workload, not a simplified surrogate
