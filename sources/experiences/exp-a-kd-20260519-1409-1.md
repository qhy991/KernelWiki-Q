## Solution

Replace inline with __forceinline__ on all four convert method overloads to guarantee inlining regardless of compiler heuristics; however, since each function body is a single hardware intrinsic, nvcc at -O3 already inlines them, producing identical PTX/SASS and yielding no measurable speedup

## Key Lessons

- Replacing inline with __forceinline__ on trivially small single-intrinsic device functions produces no measurable speedup when nvcc -O3 already inlines them — the change only matters in deep template instantiation stacks where the compiler's cost heuristic might suppress inlining across namespace/template boundaries
- Profiling before applying __forceinline__ is essential: if the compiler already generates identical SASS for inline vs __forceinline__, the qualifier adds no value and can hurt compile time and instruction cache pressure in larger codebases
