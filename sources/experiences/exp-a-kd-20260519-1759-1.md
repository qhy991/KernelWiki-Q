## Solution

Derive alignment from element bit width via 128 / (sizeof(ElementAB) * 8), so for __half alignment becomes 128/16=8 elements = 16 bytes = one float4 load/store per work item, exactly matching 128-bit access granularity

## Key Lessons

- Derive vectorized alignment from 128 / sizeof_bits<Element> rather than hardcoding a constant — this ensures the vector width always matches the 128-bit memory access granularity regardless of element type, avoiding redundant memory transactions for sub-32-bit types
- Hardcoded alignment values that are correct for one precision (e.g., fp32 with alignment=4) silently cause correctness or performance issues for other precisions (e.g., fp16 with same alignment=4 means only 8 bytes, underutilizing bandwidth; alignment=16 means 32 bytes, exceeding 128-bit boundary)
