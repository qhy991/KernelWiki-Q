## Challenge

Both before and after versions compile to equivalent register-level code because the modulo and division operations in the before version are on compile-time-constant values derived from constexpr template parameters. ## Solution

Replace modulo/division index computation with direct linear offset (index = iter_idx * MMA_N) and switch fragment layout to row-major so that write (frag[m * MMA_C + n]) and consume (frag[m * MMA_C + n]) use identical stride — however NVCC constant-folds the original constexpr-derived arithmetic, making this a readability refactoring with no measurable speedup

## Results

- **speedup**: 0.998x (neutral)


## Key Lessons

- NVCC constant-folds modulo and division on constexpr-derived values at compile time, producing identical PTX — arithmetic simplifications on compile-time-constant indices do not yield runtime savings and benchmarking is essential to avoid false optimization assumptions
- Register file access is not banked like shared memory, so column-major vs row-major indexing of small register arrays (e.g., 8-element fragments) has no measurable performance penalty regardless of stride pattern
- When refactoring CUTLASS fragment iterators, prioritize readability and maintainability (direct linear offsets, consistent row-major layout) over presumed arithmetic savings — the compiler already handles constexpr arithmetic elimination
