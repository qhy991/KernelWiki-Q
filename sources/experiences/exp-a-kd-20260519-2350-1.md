## Challenge

Host-side wrapper launches the kernel asynchronously and only checks cudaGetLastError without calling cudaDeviceSynchronize, so runtime errors (e.g., illegal memory access) go undetected and the caller must remember to sync separately for correct wall-clock timing


## Solution

Embed cudaDeviceSynchronize inside the wrapper alongside cudaGetLastError, adding printf-based error diagnostics for both launch and runtime errors, and remove the redundant external sync from the benchmark loop so the wrapper provides complete error detection in a single call

## Key Lessons

- Moving cudaDeviceSynchronize from the caller into a wrapper function does not change total wall-clock completion time for compute-bound kernels — the sync overhead is identical regardless of call site, and kernel execution time dominates for sufficiently large payloads
- cudaGetLastError only catches launch-time errors (invalid config, invalid argument); cudaDeviceSynchronize is required to detect asynchronous runtime errors (illegal address, misaligned address) — a production wrapper should check both
- For microbenchmarks at this scale (8 CTAs, ~1.7 ms), ~2% timing variation is within measurement noise; any sync-placement refactoring claiming a speedup below this threshold should be treated as neutral
