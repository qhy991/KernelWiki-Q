## Challenge

Grid launch configuration uses raw row count m to determine the number of blocks; when m=1 (small batch), only a single block is launched on one SM, leaving the remaining SMs idle while that one block must sequentially process all 128 padded SF rows plus the real computation row


## Solution

Replace raw m with effectiveRows=round_up(m,128) in the grid-size calculation so that for small m the launch uses up to 128 blocks (one per padded SF row), distributing the padding work and the real computation across many SMs concurrently via the existing grid-stride loops

## Key Lessons

- Use the padded/effective row count (round_up to the SF group size) instead of the raw row count for grid-size calculation — when the kernel already uses grid-stride loops, launching more blocks than real rows is safe and dramatically improves occupancy for small-batch cases
- Kernels that must write padding rows (e.g., SF alignment zeros beyond numRows) should factor those padding rows into the grid size so the work is parallelized rather than serialized into fewer blocks
- Grid-stride loops (rowIdx+=gridDim.x) make a kernel robust to over-launching: extra blocks simply skip iterations where rowIdx>=numRows, so using a larger grid based on effectiveRows never causes correctness issues
