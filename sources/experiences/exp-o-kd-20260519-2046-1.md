## Solution

Adaptively reduce block size to 256 threads when num_tokens >= 256, allowing each SM to schedule up to 8 concurrent blocks instead of 2, dramatically increasing occupancy and warp-level context switching to hide global memory latency in this memory-bound two-pass RMS norm kernel

## Key Lessons

- Reduce block size (e.g., 256 vs 1024 threads) for large-batch memory-bound kernels to increase SM occupancy and concurrent blocks — more warps hiding latency outperforms fewer warps with more threads each when the kernel is not compute-bound
- Adaptively choose block size based on grid dimension (num_tokens): small batches benefit from larger blocks to amortize per-block overhead, while large batches benefit from smaller blocks to maximize occupancy across many SMs
- For two-pass normalization kernels (variance computation + normalization), the memory-latency-bound nature means occupancy-driven optimizations dominate over per-thread compute throughput
