## Solution

Replace the single-thread serial loop with a CUB BlockScan inclusive prefix sum launched as <<<1,64>>>, distributing the 64 element reads across 64 threads and computing the scan in O(log N) sequential steps via cooperative shared-memory parallelism

## Key Lessons

- Replace single-thread serial prefix-sum kernels with CUB BlockScan even at small scales (64 elements) — the O(log N) parallel scan reduces sequential steps and the shared-memory cooperative approach still yields measurable speedup despite kernel launch overhead
- Use a chunked BlockScan pattern with a shared broadcast variable to handle n > BLOCK_SIZE — each chunk carries the running prefix sum forward via the last thread writing to shared memory and a __syncthreads barrier before the next chunk
- Prefer templated BLOCK_SIZE for BlockScan to match the problem size exactly (e.g., block_size=64 for 64 experts) — this avoids wasted threads and keeps shared memory footprint minimal
