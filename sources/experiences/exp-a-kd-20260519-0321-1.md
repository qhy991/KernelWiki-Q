## Challenge

The copy kernel dispatches between a vectorized float4 path and a scalar fallback based on an alignment check, but the alignment variable is set to stride_last=1 which always evaluates to unaligned, forcing the scalar path even when a more meaningful stride could enable vectorization


## Solution

Replace stride_last=1 with stride_heads=kHeadDim in the alignment check so the dispatch considers the actual head stride (60) rather than the trivial innermost stride, attempting to enable vectorized float4 copies when the memory layout is aligned

## Key Lessons

- Changing the alignment-check variable from stride_last=1 to stride_heads=kHeadDim(60) has no effect when kHeadDim=60 is itself not divisible by the vector alignment requirement (16 bytes = 4 floats), so both before and after take the scalar fallback — verify the new alignment condition actually evaluates to true for the target data layout before expecting a speedup
- When dispatching between vectorized and scalar memory paths, the alignment predicate must reflect the stride that determines whether reinterpret_cast<float4*> is safe for the entire buffer — checking an irrelevant stride (1 vs 60) that is never divisible by 4 yields the same scalar path and no performance gain
