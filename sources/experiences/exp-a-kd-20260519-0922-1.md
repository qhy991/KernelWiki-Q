## Solution

Reduce thread count to 512 and experts_per_warp to 16 (matching num_experts=16) to shrink shared memory from 1024 to 64 bytes, trading fewer threads for more per-thread work (16 vs 8 tokens); however, the doubled per-thread iteration count in the atomicAdd counting loop dominates execution time, causing a net regression

## Key Lessons

- Reducing thread count to save shared memory can regress when the dominant kernel phase is a per-thread sequential loop — doubling per-thread iterations outweighs the shared memory savings and reduced synchronization overhead
- Adaptive thread/experts-per-warp tuning primarily benefits GPU resource availability for concurrent kernels rather than isolated kernel latency; benchmark single-kernel latency separately before accepting such trade-offs
