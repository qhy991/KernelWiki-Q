## Challenge

Broadcast FMA kernel performs scalar * vector + vector by decomposing half2 into separate float lanes via __low2half/__high2half + __half2float, running two separate float-domain fma() calls, then converting back through __floats2half2_rn and re-extracting — 8 conversion operations per iteration create massive overhead in a dependent accumulation loop


## Solution

Eliminate all half↔float conversion round-trips by operating directly on packed f16x2 values using PTX inline assembly: h0_h0 (mov.b32) broadcasts the scalar half into both lanes of a half2, fma.rn.f16x2 computes scalar*a+c for both lanes in one instruction, and add.f16x2 accumulates — keeping all arithmetic in the native half2 domain

## Key Lessons

- Use h0_h0 (PTX mov.b32 {lo,hi}) to broadcast a scalar half into both lanes of a half2 instead of converting to float and back — the PTX instruction is a single-register rearrangement with zero arithmetic overhead
- Replace separate per-lane float-domain fma() with fma.rn.f16x2 PTX inline asm to process both half2 lanes in one instruction, eliminating half↔float conversion round-trips that dominate inner-loop overhead in tight accumulation chains
- Treat half2 values as uint32_t throughout the computation pipeline and use reinterpret_cast for type punning — this avoids compiler-inserted conversion code and keeps all data in registers as packed f16x2
