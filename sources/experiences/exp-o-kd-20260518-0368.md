## Solution

Upgrade from single-stage synchronous K/V loading (kStage=1) to two-stage async pipeline (kStage=2) with double-buffered K tiles: the next K tile is prefetched into an alternate shared memory slot via cp.async while the current iteration's Q@K computation proceeds, hiding global memory latency across the 64-tile KV loop

## Key Lessons

- Increasing pipeline stages from kStage=1 to kStage=2 with double-buffered K shared memory and cp.async prefetch hides global memory latency by overlapping the next tile's load with current computation — applies when the KV loop has enough iterations (Tc>=8) to amortize the extra smem and setup overhead
- CP_ASYNC_WAIT_GROUP(1) before P@V ensures only the current V tile load is synchronized while the next K prefetch continues in background, preserving maximum overlap between memory and compute
- Smem slot alternation (smem_sel/smem_sel_next) prevents read-write hazards on the double-buffered K shared memory across consecutive KV loop iterations without requiring explicit barriers
