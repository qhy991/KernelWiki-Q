## Challenge

Two sequential mma.sync.aligned.m16n8k16 instructions are needed to cover k=32 in the GEMM kernel, requiring double the MMA instruction dispatches, two full fragment-load rounds per k-tile, and extra register pressure from holding four A-fragments and two B-fragments simultaneously


## Solution

Replace two sequential dense mma.sync.m16n8k16 calls with a single mma.sp.sync.aligned.m16n8k32 instruction that covers the full k=32 range in one tensor core dispatch, using a FragM metadata register (Vec<uint,1>) to provide 2:4 sparsity selectors to Ampere's dedicated sparse tensor core datapath

## Key Lessons

- Replace two sequential dense mma.sync.m16n8k16 calls with a single mma.sp.sync.aligned.m16n8k32 when 2:4 sparsity metadata is available — Ampere's sparse tensor core datapath doubles effective K-dimension throughput per instruction, halving MMA dispatch count and reducing register pressure
- The mma.sp instruction requires a FragM metadata register (Vec<uint,1>) encoding the 2:4 sparsity selector pattern, plus a psel parameter controlling which pair of B-matrix elements participate — always pass metadata alongside the sparse instruction for correct hardware dispatch
- Sparse MMA reorganizes fragment roles: the A-matrix supplies two B-fragments (a_frag0, a_frag1) and the B-matrix supplies the larger A-fragment (4 half2 registers for doubled k), so fragment-to-matrix mapping differs from dense MMA — verify register-to-operand correspondence against the PTX spec to avoid silent correctness errors
