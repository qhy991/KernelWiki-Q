## Solution

Add hardware-aware dispatch by gating sequential mode on sm_count == 78 (H20 GPU) combined with K > 128; on GPUs with abundant SMs like H100/H800, the batched cublasGemmStridedBatchedEx processes all 8 expert GEMMs in a single kernel launch, eliminating 7 redundant launches and enabling full SM occupancy across the grouped workload

## Key Lessons

- Hardware-aware dispatch (SM count + problem dimensions) outperforms purely heuristic dimension-based branching — on GPUs with abundant SMs, batched GEMM launches exploit parallelism that sequential per-expert launches cannot, achieving up to 2.65x speedup in MoE workloads
- cublasGemmStridedBatchedEx consolidates multiple independent GEMM problems (MoE experts) into a single kernel launch, reducing launch overhead and enabling the library to distribute work across all available SMs — prefer it over sequential cublasGemmEx loops when the GPU has enough SMs to service all problems concurrently
- Sequential per-expert GEMM calls are only justified on SM-sparse GPUs (e.g., H20 with 78 SMs) where insufficient SMs exist to service all expert problems in parallel; always query runtime SM count via cudaGetDeviceProperties before selecting dispatch strategy
