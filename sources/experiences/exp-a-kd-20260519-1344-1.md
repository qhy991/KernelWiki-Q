## Solution

Add an explicit int64_t input_stride parameter and compute a separate strided_id (blockIdx.x * vec_input_stride + idx) for all input tensor accesses, while keeping residual indexing contiguous with the original id, enabling the kernel to handle arbitrary input strides with zero overhead in the contiguous case where stride equals hidden_size

## Key Lessons

- Decoupling input stride from hidden_size via a runtime parameter generalizes kernels to handle both contiguous and padded layouts — when stride equals hidden_size the compiler resolves both address paths identically, yielding zero performance overhead
- Using int64_t for stride-derived index computations prevents address overflow on large tensors while keeping the per-element vectorized load/store paths (128-bit HalfVec8) unchanged regardless of stride
- Separating input and residual indexing allows the two tensors to have different memory layouts — residual stays contiguous while input can be strided, matching the asymmetric layout patterns common in paged-attention frameworks
