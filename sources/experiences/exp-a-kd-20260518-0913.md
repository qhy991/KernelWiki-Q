## Challenge

The after code uses BM=32 instead of BM=128, reducing the M-dimension tile from 128 rows to 32. For M=16, this cuts padding waste from 112 zero-padded rows (87.5% of the tile) to just 16 rows (50%). The smaller tile also reduces shared memory per block from 16KB (128*32 + 32*128 half values) to 1...


## Solution

in the after config doubles the inner-dimension tile, meaning each shared memory load amortizes its cost over twice as many multiply-accumulate iterations, improving compute-to-memory ratio for this thin M problem.
