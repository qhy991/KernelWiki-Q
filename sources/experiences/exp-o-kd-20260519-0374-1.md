## Challenge

FlashAttention-2 kernel stores the output accumulator R_D in FP16, forcing the online-softmax rescale step to perform half2float→fmaf→float2half round-trips on every K/V tile iteration, incurring ~512 unnecessary format-conversion instructions per thread and pipeline latency from the FP16 conversion unit


## Solution

Promote the R_D accumulator from FP16 (2 uint32 registers per element) to FP32 (4 uint32 registers per element), so the rescale loop reads the accumulator directly as float and performs fmaf with only one half2float for the incoming R_O, deferring the single float2half conversion to the final output write — eliminating 4 float2half + 4 half2float instructions per inner iteration

## Key Lessons

- Use FP32 accumulators for iterative reduction loops (online softmax rescale, running sums) even when MMA output and final storage are FP16 — the eliminated half2float/float2half conversions amortize the doubled register cost across all tile iterations
- Defer precision narrowing to the latest possible point (final output store) rather than performing it inside every iteration of a hot loop — each deferred conversion removes N round-trips where N is the number of K/V tile steps
