## Solution

The layout transpose swaps block_size and head_size in the inner two dimensions, shifting from stride-1 to stride-block_size between consecutive head_offset values — this intentionally sacrifices reshape_and_cache write coalescing to gain read coalescing in the downstream attention kernel where threads access along the head_size dimension

## Key Lessons

- Swapping inner dimensions in a multi-dimensional cache layout directly controls which access pattern gets stride-1 coalescing: [block_size, head_size] gives coalesced writes for reshape_and_cache but scattered reads for attention, while [head_size, block_size] reverses the trade-off
- KV-cache layout selection is a read-vs-write trade-off: the reshape_and_cache write path and the attention read path favor opposite inner-dimension orderings, so the optimal layout depends on which path dominates the overall workload (typically the attention reads)
