## Challenge

merge_attn_states kernel uses scalar per-element half load/store in a strided loop, issuing individual 16-bit memory transactions per element even though each thread processes contiguous head elements that could be packed into wider loads


## Solution

Replace scalar half load/store with 128-bit uint4 packed loads (8 half elements per transaction) using a flat 1D grid where each thread processes one 8-element pack, reducing memory instruction count from 64 individual loads to 8 packed loads per head

## Key Lessons

- 128-bit vectorized loads (uint4) reduce memory instruction count but provide negligible speedup when the working set per thread block is tiny (head_size=64 = 128 bytes fits in one L2 cache line) — scalar loads already coalesce effectively at this scale
- Kernel launch overhead (~7-8 microseconds total) can dominate runtime for small kernels, masking any memory-level optimization — vectorization benefits only materialize when memory throughput is the actual bottleneck
- The uint4 unpack/repack loop (8 scalar __half2float conversions inside a #pragma unroll loop) adds instruction overhead that can offset the wider load benefit when the data volume per thread is small
