## Challenge

Per-thread read-modify-write on self_counter[blockIdx.x][threadIdx.x] += 1 causes cache-line thrashing on 128-byte-aligned arrays, as multiple threads in the same block independently update adjacent slots within the same aligned cache line, and modulo-2 alternation adds further arithmetic and storage overhead


## Solution

Redesign the Signal struct to separate start/end phases into distinct fields and introduce a shared _flag counter read once per block; delegate the single flag write-back to thread 0 only after __syncthreads(), eliminating per-thread read-modify-write and modulo-2 alternation overhead

## Key Lessons

- Replace per-thread read-modify-write on shared counter arrays with a single thread-0-only write-back after __syncthreads() — eliminates cache-line thrashing when multiple threads update adjacent slots within the same 128-byte-aligned line
- Separate barrier phases (start/end) into distinct fields instead of using modulo-2 buffer alternation — removes modulo arithmetic overhead and the need for double-buffered peer_counter arrays
- Reading a shared flag once per block (_flag[blockIdx.x]) and incrementing locally is cheaper than each thread independently incrementing its own counter slot — the single-writer pattern avoids L2 cache contention in multi-thread barrier implementations
