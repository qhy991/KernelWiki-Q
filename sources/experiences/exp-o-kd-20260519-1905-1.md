## Challenge

MLA KV cache write uses three separate kernel launches — two index_select kernels copy selected token rows into intermediate global-memory buffers, then a third cache_write kernel reads those buffers and writes to the paged KV cache, doubling global memory traffic and incurring two extra kernel launch synchronization barriers


## Solution

Fuse all three kernels into a single kernel that reads source kv_c and k_pe tensors directly using selection indices and writes to the paged KV cache in one pass, eliminating intermediate global-memory buffers and reducing memory traffic from 2 source reads + 2 intermediate writes + 2 intermediate reads + 1 cache write per element to just 1 source read + 1 cache write per element

## Key Lessons

- Fuse index_select + cache_write kernels when the intermediate buffers are only consumed once — eliminates a full global memory round-trip per element and removes kernel launch synchronization overhead
- Use a lambda-based copy helper to handle multiple source tensors with different strides writing to contiguous regions of a single destination, reducing code duplication in fused kernels while preserving stride flexibility
- Kernel fusion for data-movement pipelines (gather + scatter) yields the largest speedups when intermediate data does not fit in L2 cache — the 1.9x speedup here comes from halving global memory traffic
