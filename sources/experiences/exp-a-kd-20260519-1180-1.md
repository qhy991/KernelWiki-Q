## Challenge

Replacing half2 scale storage with fp8x2 reduces memory footprint from 4 to 2 bytes per pair, but on sm_80 hardware without native fp8 support the conversion expands into a software emulation chain (fp8x2→half2→float2), adding instruction overhead that negates the bandwidth savings


## Solution

Replace half2 scales with __nv_fp8x2_e4m3 to halve scale memory footprint, converting via (float2)s cast instead of __half22float2 — however on sm_80 the fp8 conversion lacks native hardware and falls back to software emulation (fp8x2→half2→float2), incurring extra instructions that offset the bandwidth reduction

## Key Lessons

- Replacing half2 with fp8x2 for dequant scales reduces memory bandwidth only when working sets exceed L2 cache; for small tensors that cache-reside, the conversion instruction overhead dominates and can cause regression
- Native fp8 hardware support (sm_89+/sm_90+) is required for fp8 conversion to be efficient — on sm_80 the (float2)s cast on __nv_fp8x2_e4m3 expands into a multi-instruction software emulation chain (fp8x2→half2→float2), negating the storage savings
- Lower-precision storage formats only benefit kernels that are genuinely memory-bandwidth-bound; when the compute-to-memory ratio is low and data fits in cache, the extra conversion instructions add latency without measurable bandwidth improvement
