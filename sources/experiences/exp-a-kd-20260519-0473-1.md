## Challenge

An FP16 ELU kernel processes only 2 half-elements per thread (half2), resulting in limited instruction-level parallelism and leaving GPU memory bandwidth underutilized for elementwise activation kernels


## Solution

Increase per-thread processing from 2 to 8 half-elements (4x half2 registers), loading and computing four half2 values per thread to expose more ILP and reduce kernel launch overhead; however, the corresponding thread count drops from 128 to 32 per block, reducing occupancy and offsetting the ILP gains

## Key Lessons

- Increasing vectorization width from x2 to x8 (4x half2 per thread) does not always improve performance because the resulting drop in threads per block (128→32) can reduce occupancy below the threshold needed to hide memory latency — the ILP gains from more per-thread work are offset by fewer concurrent warps
- Per-thread vectorization width must be balanced against minimum occupancy requirements; a moderate increase (e.g., x4) may hit the sweet spot better than an aggressive jump (x8) when the kernel is memory-bound at small problem sizes
