## Challenge

GEMM epilogue processes fragments one at a time (FragmentsPerIteration=1), requiring 4 __syncthreads() barriers across 2 fragment iterations in a write-then-reload loop through shared memory, introducing unnecessary barrier overhead


## Solution

Batch all fragments into a single iteration (FragmentsPerIteration=2) to reduce __syncthreads() from 4 to 2, but this doubles shared memory from 2112 to 4224 floats (~17KB), which halves per-SM occupancy on A100/A800 and causes a net performance regression for lightweight kernels where occupancy matters more than barrier count

## Key Lessons

- Reducing __syncthreads() barriers by batching fragments into one iteration is counterproductive when the doubled shared memory footprint significantly reduces SM occupancy — occupancy loss dominates barrier savings for lightweight kernels
- Shared memory sizing is a first-order performance knob: doubling from ~8.5KB to ~17KB per block can halve concurrent blocks per SM on A100/A800 (164KB shared memory), turning a barrier optimization into a net regression
- FragmentsPerIteration batching in CUTLASS epilogue benefits compute-heavy kernels where shared memory pressure is amortized by arithmetic intensity, but harms memory-bound or lightweight kernels where occupancy is the bottleneck
