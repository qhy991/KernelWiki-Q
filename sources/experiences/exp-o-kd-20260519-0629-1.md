## Challenge

Five separate kernel launches copy small metadata arrays (cache_seqlens, cu_seqlens_k, page_table, nsa_cache_seqlens, nsa_cu_seqlens_k) totaling ~2600 int32 elements for bs=4, max_len=128; kernel launch overhead (~5us per launch) dominates over actual data movement work


## Solution

Fuse all 5 metadata copy operations into a single kernel that iterates over each field sequentially within the same thread grid, eliminating 4 out of 5 kernel launches and using __grid_constant__ to store the parameter struct in constant memory for fast repeated access to bs, stride, and other fields

## Key Lessons

- Fuse multiple small-copy kernel launches into a single kernel when data volume per field is tiny (e.g., bs=4 → 4 elements) and kernel launch overhead (~5us per launch) dominates actual work — the optimization is purely about eliminating launch overhead, not about computation
- Use __grid_constant__ for parameter structs in fused kernels to keep frequently-read values (bs, stride, pointers) in constant memory with zero-copy overhead, avoiding repeated global memory loads for small scalars
- #pragma unroll on the inner loops of a fused small-copy kernel ensures the compiler fully unrolls the tiny iteration counts (bs=4), turning each field copy into a few inline load-store instructions without loop overhead
