## Challenge

A unified thread config table selects {thread_k=64, thread_n=256, num_threads=256} for all batch sizes including small batches (prob_m=1), producing 64 K-slices that each require serial global reduction, and the config is not tuned for the small-batch regime where fewer K-slices would reduce reduction overhead


## Solution

Split the single thread_configs table into batch-size-aware tables (small_batch_thread_configs defaulting to {128,128,256} for prob_m<=16, large_batch_thread_configs retaining {64,256,256}) and add a prob_m<=16 branch in determine_thread_config to halve the number of K-blocks from 64 to 32 for small batches

## Key Lessons

- Reducing thread_n from 256 to 128 while keeping num_threads=256 in a naive per-thread dot-product loop causes half the threads to be idle (threads 128-255 iterate over 0 elements), wasting compute capacity — this only works in real Marlin because tensor core mma.sync instructions require all 256 threads in a fixed warp-group layout regardless of thread_n
- Thread config tuning for tensor-core kernels (MMA instructions) cannot be validated with simplified scalar dot-product microbenchmarks — the thread utilization model is fundamentally different between MMA and naive loops
- Batch-size-aware config selection is a valid strategy for reducing serial global reduction overhead (fewer K-blocks), but the chosen {thread_k, thread_n} pair must ensure full thread utilization in the actual kernel's compute loop
