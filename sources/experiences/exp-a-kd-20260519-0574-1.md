## Challenge

In the tiled SGEMM main loop, B-tile loads from global to shared memory are issued synchronously via FLOAT4 assignments, serializing the global memory transfer with subsequent FMA computation and preventing any overlap between memory and arithmetic units


## Solution

Replace synchronous FLOAT4 B-tile loads with cp.async.ca.shared.global directives, committing the async group before starting the BK FMA compute loop so that B-tile global-to-shared transfers proceed concurrently with arithmetic, then wait on the group before the thread barrier

## Key Lessons

- Use cp.async.ca.shared.global to issue asynchronous global-to-shared copies for tile data, then CP_ASYNC_COMMIT_GROUP before entering the compute loop — the hardware pipelines the transfer alongside FMAs, hiding global memory latency without occupying a thread's compute slots
- Separate B-tile async loads from A-tile synchronous loads: only one operand needs cp.async since the async group wait + __syncthreads barrier must serialize at some point; choose the larger or more latency-bound operand for async transfer
- CP_ASYNC_WAIT_GROUP(0) before __syncthreads ensures all pending async copies complete before threads proceed, maintaining correctness of double-buffered shared memory swaps while still gaining overlap within the loop body
