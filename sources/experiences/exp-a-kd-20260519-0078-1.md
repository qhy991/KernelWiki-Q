## Solution

Replace the 128-iteration element-by-element XOR loop with a 4-iteration word-level XOR on the uint32_t storage words directly, leveraging the bitwise-parallel nature of XOR which produces identical results across all packed bits without any decode/encode overhead

## Key Lessons

- Bitwise operations (AND, OR, XOR, NOT) on packed sub-byte data should always operate directly on storage words rather than extracting individual elements — bitwise operators are inherently parallel across all bits in a word, so a single uint32_t XOR replaces 32 separate 1-bit XOR operations
- Sub-byte element access patterns (extract via shift+mask, insert via conditional read-modify-write) cost ~5-8 instructions per element and introduce warp divergence through per-element branching, making them prohibitively expensive compared to word-level operations for any bitwise logical kernel
