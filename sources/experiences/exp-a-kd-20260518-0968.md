row_shuffle_fp16_7168cols

## Solution

The optimization's benefit in the real vLLM pipeline likely comes from avoiding the allocation and separate kernel launch for the reverse_map/permuted_idx arrays, plus the downstream unpermute kernel being able to skip the reverse-map lookup—benefits that are not captured by this isolated microbe...

## Key Lessons

- Same pattern as the 4096-col case: the before kernel's 2 scalar global stores (reverse_map, permuted_idx) per block amount to ~32KB total (2 x 4 bytes x 4096 rows) against ~56MB of row data (4096 x 7168 x 2 bytes).
