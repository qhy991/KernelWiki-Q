## Challenge

FP32 sigmoid kernel processes one element per thread using scalar float load and store, issuing separate LDG/STG instructions for each element and underutilizing available HBM bandwidth on the memory-bound 64 MB working set


## Solution

Vectorize loads and stores using float4 (16-byte) transactions via reinterpret_cast, so each thread processes 4 elements — reducing total load/store instruction count by 4x and improving coalescing efficiency for the bandwidth-bound kernel

## Key Lessons

- Use float4 vectorized loads/stores (via reinterpret_cast<float4*>) for memory-bandwidth-bound elementwise kernels working on contiguous fp32 arrays — reduces load/store instruction count by 4x and improves coalescing when the working set exceeds L2 cache
- Scale block size down by the vectorization factor (256/4=64 threads) so total logical elements per block stays constant, preserving occupancy and grid coverage
- When converting scalar elementwise kernels to vectorized form, expand per-element computation across the .xyzw lanes of a float4 register and perform a single vectorized store with a bounds check on the first lane index
