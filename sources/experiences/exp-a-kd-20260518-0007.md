## Challenge

For vec_n_t<half,4> (8-byte pack), adding a local variable before passing to VecOp was intended to force wide loads. At this narrow width, NVCC already emits LDG.8 narrow-wide loads regardless.


## Solution

At 8-byte pack width, NVCC -O3 handles load scheduling efficiently with or without local copy. The 2.3% difference is within measurement noise. Neither version has a meaningful advantage — local copy is unnecessary dead code at this width.

## Results

- **speedup**: 0.978x (neutral, within measurement noise)


## Key Lessons

- For narrow vector packs (≤8 bytes), NVCC already emits efficient loads (LDG.8/LDG.16) regardless of whether values are materialized to locals first. Adding local copies for load-width forcing has a threshold: below ~16 bytes, it's dead code; above ~16 bytes, it may help depending on the inlining context.
- When benchmarking shows delta < 3% for a kernel this short, don't draw conclusions. At nanosecond-scale kernel durations, measurement noise from clock resolution and scheduler jitter can produce apparent speedups or regressions that are not reproducible.
