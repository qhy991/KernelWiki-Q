## Challenge

The fused_add_rms_norm kernel always launches with a fixed block_size of min(hidden_size, 1024) regardless of num_tokens, using a blockReduceSum defaulting to maxBlockSize=1024. When num_tokens is large (>=256), the 1024-thread blocks consume excessive registers and shared memory per block, limiting SM occupancy and reducing overall throughput.


## Solution

Introduce adaptive block sizing: when num_tokens < 256, keep block_size=1024 for maximum per-block parallelism; when num_tokens >= 256, reduce to block_size=256 to allow more concurrent blocks per SM and improve occupancy. The kernel dispatches blockReduceSum with the matching maxBlockSize template parameter at runtime, and the launch config selects the appropriate block size accordingly.

## Key Lessons

- Adaptive block sizing based on grid dimensions (num_tokens) can improve SM occupancy for large-batch workloads by reducing per-block resource consumption — the blockReduceSum template parameter must match the launch block size for correct shared memory sizing and warp-level reduction
- This control case confirms the adaptive logic correctly falls back to the original 1024-thread configuration when num_tokens < 256, producing equivalent performance within measurement noise
