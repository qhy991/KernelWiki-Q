## Solution

Replace variable-count qword loop with a fixed 3-operation pattern per lane: load 2 value qwords at lane_id and lane_id+32 into a local staging array, then load 1 tail/scale item (value tail for lanes 0-7, scale for lanes 8-31), and store all three — eliminating loop overhead and enabling better instruction scheduling

## Key Lessons

- Fixed-size load/store patterns (known count at compile time) eliminate loop overhead and enable better compiler instruction scheduling compared to variable-count loops — applies when item size is known and evenly partitions across warp lanes
- Staging loads into a local array (local_items[2]) before storing breaks the immediate load-to-store dependency chain, allowing latency overlap — applies when the transfer pipeline has address dependency chains
