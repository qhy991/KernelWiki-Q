## Challenge

MoE topk ID remapping kernel reads a small expert_map lookup table (64 ints, 256 bytes) directly from global memory on every thread access, prompting an attempt to cache it in shared memory to reduce latency


## Solution

Cache the expert_map into shared memory via a cooperative load loop with __syncthreads() barrier before reading, replacing global memory reads with shared memory reads — but the table is only 256 bytes and already fits in L1 cache, so the smem overhead (barrier + load loop) provides no net benefit

## Key Lessons

- Shared memory caching of a small lookup table (≤256 bytes) provides no latency advantage over L1-cached global reads — the GPU L1 cache already covers such small working sets, and the cooperative load loop + __syncthreads() barrier adds overhead that neutralizes any benefit
- Before applying smem caching, verify that the data exceeds L1 cache capacity and is accessed frequently enough to amortize the barrier synchronization cost — shared memory is a scarce resource and unnecessary use reduces occupancy
- Adding __syncwarp() after per-thread writes that have no cross-warp data dependency is redundant and introduces unnecessary stall cycles
