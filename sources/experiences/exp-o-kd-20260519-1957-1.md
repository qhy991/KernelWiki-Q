## Challenge

GEMM produces bf16 output via cublasGemmEx with CUDA_R_16BF output type, then a separate cast_bf16_to_fp32 kernel converts the result to fp32 — this requires an intermediate bf16 buffer, an extra global-memory round-trip, and a second kernel launch whose fixed overhead dominates for small M dimensions


## Solution

Replace the two-step bf16-GEMM + cast-kernel pattern with a single cublasGemmEx call using CUDA_R_32F output type — cuBLAS internally accumulates in fp32 (CUBLAS_COMPUTE_32F) and writes fp32 directly, eliminating the intermediate bf16 buffer, the separate cast kernel launch, and the extra global-memory read of the bf16 output

## Key Lessons

- cuBLAS supports mixed-type GEMM where input and output types differ — use CUDA_R_32F output with bf16 inputs and CUBLAS_COMPUTE_32F to get fp32 results directly, avoiding a separate type-conversion kernel pass — applies whenever a bf16 GEMM result must be consumed in fp32 (e.g., F.linear(...).to(torch.float32))
- Kernel launch overhead (driver dispatch, grid setup) is fixed-cost and can dominate total runtime for small outputs — fusing the conversion into the library call eliminates this overhead entirely, especially impactful when M is small (e.g., single-token decode)
