## Challenge

Grouped GEMM scheduler hardcodes ASSUMED_SM_COUNT=84 for computing the number of thread blocks and groups-per-block, causing severe load imbalance on devices with more SMs (e.g. A800 has 108 SMs) — too few blocks are launched, each block processes excess groups sequentially, leaving SMs idle


## Solution

Query the actual device SM count at runtime via cudaDeviceGetAttribute(cudaDevAttrMultiProcessorCount) instead of hardcoding it, so that num_blocks scales to the real hardware — on A800 this yields 216 blocks with 10 groups each (vs 168 blocks with 13 groups), spreading work across 48 additional SMs and reducing per-block sequential work by ~23%

## Key Lessons

- Never hardcode SM counts for block scheduling — query cudaDevAttrMultiProcessorCount at runtime so that num_blocks adapts to the actual device, directly increasing SM utilization and reducing sequential work per block
- In grouped/batched kernel launch patterns, groups_per_block = ceil(total_groups / num_blocks) is the critical load-balancing lever: more blocks means fewer groups per block and higher intra-grid parallelism, yielding near-linear speedup until SM saturation is reached
