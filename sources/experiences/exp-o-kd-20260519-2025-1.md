## Challenge

The merge_attn_states kernel always performs the full LSE rescaling merge for every token, even when 87.5% of tokens are suffix-only and have no prefix context, wasting compute on exp, division, and per-element FMA that simply reduces to identity


## Solution

Add a prefix_num_tokens kernel parameter and an early-exit branch: threads handling tokens at or beyond prefix_num_tokens copy suffix output and LSE directly, bypassing the expensive LSE rescaling, exp, division, and per-element FMA merge entirely

## Key Lessons

- Add data-dependent early-exit branches when a large fraction of threads operate on trivially-handled inputs (e.g. suffix-only tokens with no prefix context) — replacing expensive arithmetic with a single copy path yields speedups proportional to the sparsity ratio
- Pass metadata describing data layout (prefix_num_tokens) as a kernel parameter rather than relying on sentinel values (INFINITY in LSE) to detect trivial cases — explicit bounds are cheaper to check than loading and inspecting per-element sentinels
- In prefix-suffix attention merge, tokens beyond the prefix range have no valid prefix contribution, so the mathematically correct merge reduces to a direct copy — exploiting this algebraic identity avoids wasted exp/division/FMA for the majority of tokens
