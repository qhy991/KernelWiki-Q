## Challenge

Inverse scale computation uses float-promoted division (__float2bfloat16_rn(1.f) / y_s) which implicitly promotes the bfloat16 divisor y_s to float, performs float division, then converts back — adding unnecessary type conversion overhead when the entire computation could remain in bfloat16 domain


## Solution

Use native bfloat16 __hdiv(one_bf16, y_s) to compute the inverse entirely within bfloat16 domain, eliminating the implicit float promotion and back-conversion while keeping both operands in the same precision throughout the computation

## Key Lessons

- Prefer native __hdiv(a, b) over implicit float-promoted division (bf16_value_a / bf16_value_b) to avoid unnecessary type promotion and conversion round-trips when both operands are bfloat16
- On Ampere+ GPUs, __hdiv for bfloat16 is sufficiently fast that single division per warp does not bottleneck in memory-bound kernels — the benefit is code clarity and avoiding hidden precision widening rather than raw throughput
- Hoisting constant conversions (e.g., one_bf16 = __float2bfloat16_rn(1.f)) outside loops reduces redundant conversion instructions per iteration
