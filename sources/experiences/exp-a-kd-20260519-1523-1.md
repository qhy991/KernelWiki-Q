## Solution

Replace bare WARP_SIZE with a project-specific WARP_SIZE_GGUF constant hard-coded to 32, ensuring shared-memory tile stride is always 33 regardless of the platform's native warp or wavefront width

## Key Lessons

- Never rely on platform-defined WARP_SIZE for cross-platform CUDA/HIP code — define a project-specific constant (e.g., WARP_SIZE_GGUF=32) to avoid silent macro collisions on AMD ROCm where WARP_SIZE=64 matches the wavefront width
- Macro-name collision correctness bugs are invisible on NVIDIA hardware where WARP_SIZE=32 happens to match the intended value — always validate quantization kernels on both NVIDIA and AMD platforms to catch portability regressions
- Shared-memory bank-conflict padding strides must use a fixed tile width tied to the algorithm's logical tile size, not the hardware's native warp size, to ensure correct addressing across GPU architectures
