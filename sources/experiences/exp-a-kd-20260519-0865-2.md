## Challenge

Software pipelining with explicit L2 prefetch instructions and stride-based pointer arithmetic adds instruction overhead (conditional branches per iteration, extra prefetch loads, pointer math) without latency benefit because the input data is already resident in L2 cache from the fill kernel's writes


## Solution

The after kernel replaces the two-phase load-all-then-store-all pattern with a software-pipelined single loop that prefetches the next iteration's data into L2 while writing the current iteration, but the prefetch is redundant because data already resides in L2 and the pipelining adds conditional logic and stride-based pointer arithmetic overhead that exceeds any latency-hiding benefit

## Key Lessons

- Software pipelining with explicit prefetch.global.L2 is counterproductive when the producer kernel already placed data in L2 — the prefetch only adds redundant memory traffic and instruction overhead without reducing latency
- A two-phase load-all-then-store-all pattern that keeps data in registers (16-element buffer array) can outperform interleaved software pipelining when the working set fits entirely in registers and the loop count is small (HEAD_CHUNK_SIZE=16)
- Stride-based pointer arithmetic (precomputed vector strides via bit-shift) does not save instructions versus direct head_id*stride computation in a fully unrolled loop — the compiler folds constant strides into addressing math for both patterns
