## Solution

Replace 16 separate small kernel launches with a single batched kernel launch covering all 64 tokens, using a per-token cos_sin_cache_offsets array to route each thread block to the correct LoRA adapter group's cache region, eliminating 15 kernel launch overheads and improving GPU occupancy from 4 to 64 concurrent thread blocks

## Key Lessons

- Consolidate multiple small kernel launches into a single batched kernel when processing independent groups of tokens — kernel launch overhead (~5-20μs each) dominates when per-group work is tiny (few thread blocks), and batching eliminates N-1 launches while improving SM occupancy
- Use a per-token offset array (cos_sin_cache_offsets) to generalize a group-based token_offset parameter into a per-block index, enabling arbitrary token-to-adapter-group mapping within a single kernel without changing the inner computation logic
- When GPU utilization is low due to small grid sizes, increasing thread block count by batching independent work items is often more impactful than optimizing the kernel itself — the 16x speedup here comes from eliminating launch overhead, not from faster per-thread computation
