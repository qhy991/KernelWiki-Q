## Challenge

Two-phase candidates computation uses separate kernels: compute_swb_kernel writes sigmoid(scores)+bias to an intermediate global-memory buffer, then candidates_before_kernel reads it back via ternary-pattern access, causing double global-memory traffic for the entire scores_with_bias array and an extra kernel launch


## Solution

Fuse the two separate kernels into a single candidates_after_kernel that reads raw scores and bias directly, computes sigmoid_accurate inline in registers, and performs the max-reduction in one pass — eliminating the intermediate swb buffer allocation, the full global-memory write+read round-trip, and one kernel launch overhead

## Key Lessons

- Fusing a pre-compute kernel into its consumer eliminates one kernel launch and the intermediate global-memory buffer's full write+read cycle — applies whenever the intermediate data is used only once and fits in registers per thread
- Replacing ternary (cond ? val : neg_inf) with an if-block for per-element computation keeps the sigmoid+bias result in registers and avoids reading from a separate global buffer, improving data locality when the per-element work is lightweight (e.g., one SFU instruction)
