## Challenge

Non-contiguous input (input_stride > hidden_size) forces a separate strided_to_contiguous copy kernel launch and an extra contiguous buffer allocation before the RMS norm kernel can run, doubling memory traffic and adding kernel launch overhead


## Solution

Pass input_stride as a kernel parameter and compute input offsets as blockIdx.x * input_stride inside each device function (compute_rms, compute_scales, norm_and_quant), eliminating the separate copy kernel and its contiguous buffer entirely

## Key Lessons

- Accept stride as a kernel parameter and compute per-row offsets as blockIdx.x * stride instead of requiring contiguous input — eliminates an extra copy kernel launch, an extra pass over all input data, and a contiguous buffer allocation, yielding significant end-to-end speedup for non-contiguous tensors
- When a kernel is called on data that must first be copied into contiguous layout, consider whether the kernel can instead accept the stride natively — this pattern applies broadly to any per-row/per-token computation where the input may come from a padded or interleaved layout
