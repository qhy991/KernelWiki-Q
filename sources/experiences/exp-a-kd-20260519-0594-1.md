## Challenge

Shared memory declaration s_data[32][32] causes 32-way bank conflicts on the transposed read path: threads in the same warp read s_data[threadIdx.y][threadIdx.x] where fixed threadIdx.y means all threads access the same bank (bank = (row*32 + col) % 32 yields identical banks across the warp)


## Solution

Pad the shared memory array from s_data[32][32] to s_data[32][33], adding one element per row so the bank assignment becomes (row*33 + col) % 32, which distributes the 32 threads across all 32 banks and eliminates bank conflicts on the transposed read path

## Key Lessons

- Padding shared memory arrays by one element per row (e.g., [32][32] → [32][33]) eliminates bank conflicts in transpose patterns because the bank index shifts from (row*WIDTH + col) % 32 to (row*(WIDTH+1) + col) % 32, distributing accesses across all 32 banks — applies whenever threads read shared memory along the column dimension of a row-major array
- Bank conflicts in shared memory are among the highest-impact bottlenecks for transpose kernels: a single-element padding change yielded 1.84x speedup with no algorithmic change
