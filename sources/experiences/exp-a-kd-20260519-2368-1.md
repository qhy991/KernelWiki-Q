## Challenge

Naive INT4 dequantization extracts packed 4-bit values using two sequential bitwise operations per extraction (AND to mask relevant nibbles, then OR to set fp16 exponent bits), requiring 4 separate instructions for lo and hi extractions where a fused single-instruction approach might reduce ALU overhead


## Solution

Replace the separate AND+OR pattern with PTX lop3.b32 inline assembly using a compile-time lookup table (lut=0xe8 computing (a & b) | c) to fuse both operations into a single instruction. However, this regresses by 11.3% in isolation because nvcc already auto-fuses (a & b) | c into lop3.b32, and the asm volatile template prevents surrounding compiler optimizations including constant folding and register allocation

## Key Lessons

- Explicit LOP3 inline asm can regress vs naive AND+OR because nvcc already recognizes (a & b) | c patterns and auto-emits lop3.b32, while asm volatile prevents surrounding compiler optimizations such as constant propagation and register coalescing
- LOP3's real benefit materializes in production inner-loop contexts (e.g., Marlin GPTQ matmul) where each saved ALU instruction frees an issue slot for tensor core MMA overlap — this instruction-level parallelism advantage is invisible in isolated microbenchmarks
- Microbenchmark results for instruction-fusion optimizations can be misleading when the optimization's value depends on ILP overlap with surrounding heavy compute (tensor cores) rather than raw instruction count reduction
