## Solution

Replace the serial naiveBlockReduceSum with a two-phase warp-shuffle blockReduceSum: (1) each warp reduces its 32 lanes via __shfl_xor_sync in 5 steps, writes the warp sum to shared memory; (2) the first warp reduces 32 warp-level sums via another __shfl_xor_sync pass, achieving O(log2(32)×2) = 10 parallel steps instead of 1023 serial additions

## Key Lessons

- Replace serial single-thread shared-memory reductions with warp-shuffle two-phase blockReduceSum — each warp reduces via __shfl_xor_sync in log2(warpSize) steps, then first warp reduces warp-level sums from shared memory, reducing O(blockDim.x) serial work to O(2·log2(warpSize)) parallel steps
- Warp-shuffle reduction eliminates the shared-memory bottleneck of naive approaches: only 32 floats of shared memory are needed (one per warp) instead of blockDim.x entries, reducing shared-memory footprint and avoiding bank conflicts
- For block sizes that are exact multiples of warpSize (e.g., 1024 = 32×32), the two-phase warp-shuffle pattern maps cleanly with no edge-case handling — each phase uses the same __shfl_xor_sync reduction, differing only in data source (register vs shared memory)
