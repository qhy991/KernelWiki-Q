## Challenge

A two-step pipeline performs bf16×bf16 GEMM producing bf16 output via cublasGemmEx, then launches a separate cast_bf16_to_fp32 kernel to convert the result to fp32, requiring an intermediate bf16 buffer and an extra 32 KB memory round-trip plus kernel launch overhead


## Solution

Use cublasGemmEx with CUDA_R_32F as the output matrix dtype instead of CUDA_R_16BF, allowing the single GEMM call to produce fp32 output directly via CUBLAS_COMPUTE_32F internal fp32 accumulation, eliminating the intermediate bf16 buffer and the separate cast kernel entirely

## Key Lessons

- cublasGemmEx supports mixed input/output dtypes — setting the C matrix type to CUDA_R_32F with bf16 inputs (A,B as CUDA_R_16BF) and CUBLAS_COMPUTE_32F accumulation produces fp32 output directly, avoiding a separate type-conversion kernel
- Eliminating intermediate buffers and extra kernel launches saves memory bandwidth and launch overhead — at small-to-medium problem sizes (e.g. M=64, N=256, K=7168), the cast kernel adds ~20% overhead relative to the GEMM itself
- When using CUBLAS_COMPUTE_32F, the internal accumulation is already in fp32 — outputting to CUDA_R_16BF and then casting to fp32 is redundant since the precision loss to bf16 in the output step is immediately reversed
