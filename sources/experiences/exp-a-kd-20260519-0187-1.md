## Challenge

Warp index computation uses __shfl_sync(0xffffffff, threadIdx.x/32, 0) inside a tight loop, adding a redundant SHFL instruction and an implicit warp convergence barrier every iteration even though threadIdx.x/32 is already warp-uniform across all lanes


## Solution

Replace __shfl_sync(threadIdx.x/32, 0) with a plain threadIdx.x/32 computation via canonical_warp_idx(), eliminating the redundant SHFL instruction and warp barrier per iteration since all threads in the same warp already produce the same quotient

## Key Lessons

- Avoid __shfl_sync for broadcasting warp-uniform values (threadIdx.x/32, blockIdx.x, etc.) — the shuffle instruction adds an implicit convergence barrier and one SHFL per call, which compounds in tight loops
- threadIdx.x / 32 is warp-uniform by construction (all lanes in a warp share the same quotient), so direct arithmetic is both correct and cheaper than a shuffle broadcast
- Wrapping warp-index computation in a __device__ __forceinline__ helper with __CUDA_ARCH__ guards provides a portable, zero-overhead abstraction that the compiler can fully inline and optimize away
