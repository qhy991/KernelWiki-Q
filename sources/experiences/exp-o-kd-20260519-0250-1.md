## Challenge

NHWC channel padding kernel processes elements one-at-a-time with per-element modular index decomposition (idx%c_out, cascading integer divisions for w/h/n), wasting computation on the 5/8 of output elements that are padding zeros and issuing scalar stores that underutilize memory bandwidth


## Solution

Replace the generic scalar kernel with a specialized 3-to-8 channel kernel using shared memory tiling: 192 threads cooperatively load float4 input vectors into shared memory, then each thread assembles a float4 output where odd-indexed threads write all-zero float4 and even-indexed threads write 3 data floats + 1 zero, eliminating per-element index arithmetic and enabling coalesced float4 stores

## Key Lessons

- For channel-padding kernels where a known fraction of output is zeros (e.g. 5/8 for 3→8 padding), specialized shared-memory tiling eliminates expensive per-element modular index decomposition and replaces scalar stores with coalesced float4 vectorized stores → applies whenever channel ratios are known at compile time and the padding fraction is large
- When padding zeros dominate output elements, cooperative shared-memory loading lets threads bypass coordinate recomputation for zero positions — odd/even thread indexing encodes the data-vs-padding decision without branching on channel index
