## Solution

Replace the scalar loop with vectorize_read_with_alignment<16> which loads 128-byte vec_n_t<float,16> vectors per thread iteration and unrolls through DefaultReadVecOp into 16 scalar MinMax updates via a device lambda, but the indirect functor chain increases register pressure and instruction count, causing a regression under single-block low-occupancy conditions

## Key Lessons

- Vectorized reads that reduce memory transactions can still regress performance when GPU occupancy is too low (single block / 256 threads) to saturate memory bandwidth — the extra register pressure and instruction overhead from functor-plus-lambda indirection outweighs the saved memory transactions
- A tight scalar loop may outperform a generic vectorized abstraction when NVCC cannot fully inline the lambda-through-functor call chain, increasing instruction count compared to the direct scalar path
- Vectorization benefits depend on sufficient occupancy to exploit the wider memory transactions; for single-block kernels the scalar path can be the optimal choice
