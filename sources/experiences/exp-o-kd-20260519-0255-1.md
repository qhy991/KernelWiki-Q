## Challenge

Multiple warps concurrently atomicAdd to the same shared-memory s_prime[row] locations, causing lock/retry serialization across 4 contending lane-0 threads per row during softmax row-sum reduction


## Solution

Allocate per-warp disjoint slots in addition_storage[kQueriesPerBlock * kNumWarps] so each warp writes its partial sum without contention, then a single lane per row reads all warp contributions with an unrolled sequential loop — eliminating atomic lock/retry overhead entirely

## Key Lessons

- Replace contended atomicAdd on shared memory with per-warp disjoint slots to eliminate lock/retry serialization — applies whenever multiple warps contribute partial sums to the same output locations
- A deterministic serial reduction by one lane reading all warp slots is faster than N atomic operations when N is small (≤ warp count), because pipelined shared-memory reads have no contention overhead
- Warp-partitioned storage with a strided index (row + num_rows * warp_id) avoids bank conflicts when each warp's slot for the same row maps to a different shared-memory bank
