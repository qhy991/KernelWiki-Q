stride_access

## Solution

The after kernel uses a_gl_stride = lda / 8 = 64, which correctly steps over the full padded row width. The core computation per thread is identical; the only difference is the stride value used in address arithmetic, which determines which memory locations are read.

## Key Lessons

- Using 32 causes the before kernel to read from addresses that skip over padding incorrectly — for row r, it accesses A + r * 32 instead of A + r * 64, effectively reading from adjacent rows' padding regions instead of the intended rows.
