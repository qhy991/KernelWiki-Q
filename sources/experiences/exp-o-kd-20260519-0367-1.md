## Challenge

Flash attention split-KV kernel uses kStage=1 (single-stage synchronous load), forcing each KV-loop iteration to load K and V tiles synchronously via cp.async then block on CP_ASYNC_WAIT_GROUP(1)+syncthreads before starting Q@K^T and P@V MMA computation, serializing global memory transfers with Tensor Core work


## Solution

Upgrade to kStage=2 multi-stage async pipeline: issue current V tile and next K tile asynchronously via cp.async.cg at each iteration start, use smem_sel/smem_sel_next double-buffered K shared memory slots to avoid read-write conflicts on current K tile, and apply selective CP_ASYNC_WAIT_GROUP(1) only before P@V to let next K prefetch continue while current Q@K^T completes

## Key Lessons

- Upgrading from kStage=1 to kStage=2 in MMA-based flash attention enables double-buffered async K/V loading that overlaps global memory latency with Tensor Core computation — effective when the KV loop has enough iterations (>=8-16) to amortize the pipeline startup cost
- Use smem_sel/smem_sel_next alternation for double-buffered shared memory slots so that writing the next K tile into the alternate buffer does not conflict with reading the current K tile during MMA operations
- Apply selective CP_ASYNC_WAIT_GROUP(1) only before the P@V step rather than before all computation — this lets the next K prefetch continue in the background while current Q@K^T MMA proceeds, maximizing compute-memory overlap
