## Challenge

The RMS normalization per-token quantization kernel accepts min_scaling_factor as a runtime parameter, forcing the compiler to reserve a register for it throughout three stride-loop phases (RMS, absmax, quantization), increasing register pressure in a kernel where each thread iterates 4 elements per phase with blockDim=1024


## Solution

Replace the runtime min_scaling_factor kernel parameter with a compile-time constexpr type trait (MinScalingFactor<int8_t>::val()), allowing the compiler to fold the value into an immediate constant in the fmax comparison and free the register previously held by the runtime parameter, reducing register pressure across all stride-loop phases

## Key Lessons

- Replace runtime kernel parameters with compile-time constexpr type traits when the value depends only on the data type — the compiler folds the constant into an immediate, freeing a register across all loop phases and improving occupancy in stride-loop kernels
- Use template specialization (e.g., MinScalingFactor<int8_t>) to provide type-specific constant values at compile time, enabling a single kernel template to handle different quantization types without runtime dispatch or parameter passing
