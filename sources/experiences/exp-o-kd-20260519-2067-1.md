## Challenge

The moeSoftmax fallback kernel (used for non-power-of-2 expert counts) required float input, forcing BF16/FP16 gating output to be converted to float via a separate conversion kernel before calling softmax, wasting one kernel launch and one full global memory write of intermediate float data


## Solution

Template the moeSoftmax kernel on InputType (float, __nv_bfloat16, or __half) and use a toFloat() helper to convert each element inline within the three softmax loops (max-reduction, exp-sum, output-write), fusing the type conversion into existing data processing loops and eliminating the separate conversion kernel

## Key Lessons

- Fuse type conversion (BF16/FP16→float) into existing computation loops via a templated kernel and toFloat() helper rather than running a separate conversion kernel — eliminates one kernel launch and one full global memory round-trip for intermediate data
- Template kernels on InputType to accept float, __nv_bfloat16, or __half transparently, enabling the same kernel logic to handle multiple precisions without code duplication
