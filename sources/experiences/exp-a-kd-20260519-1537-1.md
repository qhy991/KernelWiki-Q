## Solution

Add a {thread_k=128, thread_n=64, num_threads=128} config to the thread config table, enabling a tiled GEMM kernel with 32KB shared memory for B tiles; however, only 64 of 128 threads compute unique outputs (half the warp is idle), and shared memory overhead is not amortized over the tiny M=16, N=64 problem shape, resulting in a 0.778x regression

## Key Lessons

- Tiled shared-memory GEMM can regress on very small problem shapes (M=16, N=64) because the cooperative load/store and __syncthreads overhead per k-tile iteration is not amortized over enough output elements — a naive kernel with direct global reads may be faster for sub-128 output dimensions
- When TILE_N equals only 64 but 128 threads are launched, half the threads perform redundant work (tid % TILE_N aliases), reducing effective occupancy; match thread count to the actual tile dimensions or use a smaller thread block for narrow tiles
- Enabling a previously unsupported shape (prob_n=64 where before had no valid config) is an enabling change even when latency regresses — the real Marlin kernel would TORCH_CHECK-fail without this config, so the regression is acceptable for correctness coverage
