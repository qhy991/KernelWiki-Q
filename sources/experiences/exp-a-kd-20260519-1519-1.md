## Solution

Load topk_ids[i] into a local const register (expert_id), add an expert_id == -1 sentinel branch that assigns output_permutation[i] = num_tokens, and keep the existing expert-matching path behind else-if to decouple the global load from the branch predicate while supporting mixed valid/invalid topk inputs

## Key Lessons

- Explicit local-register caching of global-memory values (topk_ids[i] → expert_id) provides no register-reuse benefit when the compiler at -O3 already materializes the load into a register before the single downstream comparison — only beneficial when the value is consumed multiple times
- Adding a sentinel-value branch (expert_id == -1) that is always predicted false on valid-only data adds negligible instruction overhead (~1 ISEL per iteration) but is irrelevant when the bottleneck is global-memory bandwidth and atomic contention rather than branch divergence
- For low arithmetic-intensity kernels bottlenecked on global-memory reads and atomics, changes to the comparison/branching logic within the loop body are unlikely to produce measurable performance differences regardless of their structural elegance
