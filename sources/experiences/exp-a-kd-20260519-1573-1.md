## Solution

Store key cache in FP8-E5M2 format and dequantize to bf16 on load via __nv_cvt_fp8_to_halfraw intrinsic to halve memory bandwidth for key cache reads; however, on sm_80 (A800) this Hopper intrinsic is unavailable and triggers slow software emulation, causing the dequantization overhead to far exceed the bandwidth savings

## Key Lessons

- __nv_cvt_fp8_to_halfraw is a Hopper (sm_89+) hardware intrinsic — on sm_80 and earlier architectures it triggers slow software emulation that far exceeds the 2× memory bandwidth savings from FP8 storage, causing net regression
- FP8 cache compression must be validated on the target architecture: dequantization overhead from emulated intrinsics can dominate over memory bandwidth savings, especially for bandwidth-bound key cache loads where the compute-to-bytes ratio is low
- The fp8→half→float→bf16 multi-stage conversion chain (as used in vLLM) incurs high instruction count per element — on hardware without native FP8 support, prefer keeping the cache in bf16 or use a fused dequant kernel that avoids the intermediate half/float stages
