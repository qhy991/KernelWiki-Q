## Challenge

FP8-to-half vector conversion and scaling are performed by two separate kernels: first fp8x4_to_half_unscaled_kernel converts fp8x4 to half2x2 and writes to an intermediate global memory buffer, then scale_half2x2_kernel reads that buffer and applies scaling via expensive half→float→multiply→half round-trip per element, costing an extra global memory round-trip and kernel launch overhead


## Solution

Fuse the fp8-to-half conversion and scale multiply into a single device function (scaled_vec_conversion_fp8x2_to_half2) that performs __nv_cvt_fp8x2_to_halfraw2 followed by half_to_float × scale → float_to_half inline, eliminating the intermediate global memory buffer and second kernel launch entirely

## Key Lessons

- Fuse type-conversion and scaling into a single kernel to eliminate intermediate global memory write/read and second kernel launch — the dominant cost is the extra memory round-trip, not per-element arithmetic, yielding ~1.6x speedup
- When a pipeline consists of kernel A writing to a temp buffer and kernel B reading it to apply a per-element operation, inline the per-element operation into kernel A's device function and remove the temp buffer entirely
- Vectorized 4-element (uint2) load/store patterns remain valid after fusion — the fusion optimization applies equally to scalar and vectorized variants without code restructuring
