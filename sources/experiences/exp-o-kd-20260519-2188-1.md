## Challenge

blockReduceSum always allocates shared memory, runs two full-warp warpReduceSum passes, and executes __syncthreads even when the block has only 16 threads — far fewer than one warp — making the shared-memory roundtrip and barrier entirely redundant


## Solution

Add a maxBlockSize template parameter to blockReduceSum and use if constexpr to dispatch: when maxBlockSize <= WARP_SIZE, skip shared memory, __syncthreads, and the second warpReduceSum entirely, using a single parameterized warpReduceSum with the exact lane count (rounded to next power of 2) for minimal shuffle rounds

## Key Lessons

- Use if constexpr with a maxBlockSize template parameter to eliminate shared-memory allocation, __syncthreads barriers, and redundant warp-level reductions when the block size fits within a single warp — the compile-time branch costs zero at runtime
- Parameterize warpReduceSum with the exact active lane count (next power of 2) to reduce shuffle rounds: a 16-thread block needs only 4 rounds instead of the full 5-round WARP_SIZE reduction
- Removing __syncthreads is particularly impactful because it forces a full warp-level stall and drains the instruction pipeline, so any compile-time path that avoids it yields measurable latency savings
