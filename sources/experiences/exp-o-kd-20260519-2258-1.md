## Challenge

A runtime min_scaling_factor kernel parameter occupies a register throughout stride-loop execution across three phases (RMS, absmax, quantization), increasing per-thread register pressure and limiting compiler scheduling flexibility in a kernel where each thread iterates hidden_size/blockDim.x = 4 elements per phase


## Solution

Replace the runtime min_scaling_factor kernel parameter with a compile-time constexpr type trait (MinScalingFactor<int8_t>::val()), allowing the compiler to fold the value into an immediate constant in the fmax comparison and free the register previously holding the runtime value across stride loops

## Key Lessons

- Replace runtime kernel parameters with compile-time constexpr when the value depends only on the quantization type and is known at compile time — this frees a register across stride-loop iterations, improving instruction scheduling and potentially occupancy
- Use type-trait structs with static constexpr val() methods (e.g., MinScalingFactor<T>) to encapsulate type-dependent compile-time constants, enabling the compiler to fold them into immediate values without any runtime cost
- Register pressure matters in stride-loop kernels where each thread iterates multiple elements per phase across several phases — even eliminating a single live register can yield measurable speedup by giving the compiler more scheduling freedom
