## Challenge

Monolithic RMS-norm function interleaves weight global-memory load between norm-factor reduction and output computation, preventing the compiler from issuing independent loads early and potentially stalling on weight fetch latency


## Solution

Decompose the monolithic function into a TileMemWarp struct for cooperative warp load/store and a pure-computation apply_norm_warp function, pre-loading both input_vec and weight_vec into registers before invoking computation so the scheduler can overlap both global loads with reduction arithmetic

## Key Lessons

- Decomposing a monolithic kernel into explicit load/compute/store phases and pre-loading all operands before computation is a sound structural pattern, but NVCC at -O3 can inline and reorder loads in a monolithic function just as effectively when the register footprint is modest (8 half2 ≈ 32 bytes per thread for kLoopCount=4)
- Manual load scheduling (pre-loading weight before reduction) only yields measurable gains when the compiler cannot prove independence or when the register working set is large enough to cause spilling — for small per-thread tile sizes the compiler already overlaps independent loads with arithmetic
- Structuring kernel code with TileMemWarp-style load/store abstractions improves readability and composability even when it does not change the generated instruction stream, making the load/compute/store contract explicit for future maintenance
