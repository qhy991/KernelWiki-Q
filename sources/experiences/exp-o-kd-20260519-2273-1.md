## Challenge

After issuing asynchronous __builtin_amdgcn_global_load_lds instructions for cooperative B[] staging across GrpsShrB groups, the code relied solely on __syncthreads() without an explicit wait for async copy completion, risking stale LDS data reads if the compiler omits the implicit s_waitcnt


## Solution

Insert an explicit asm volatile("s_waitcnt 0") barrier immediately before __syncthreads() to guarantee all pending async global-to-LDS copy operations complete before any thread proceeds past the synchronization point, ensuring staged data is visible to every thread in the workgroup

## Key Lessons

- AMD async global-to-LDS copies (__builtin_amdgcn_global_load_lds) require an explicit s_waitcnt 0 before __syncthreads() to guarantee completion — while compilers typically insert this implicitly, the ISA does not mandate it, so explicit waits ensure correctness across compiler versions and optimization levels
- When cooperative staging uses asynchronous memory copies, always pair the async issue with an explicit completion barrier before synchronization primitives — defensive synchronization prevents intermittent data races that are difficult to reproduce
