## Challenge

Two-pass softmax pipeline uses separate convertBf16ToFloat and softmaxFloat kernels with a 2MB intermediate float buffer (8192×64×4 bytes) in global memory, requiring two kernel launches and doubling memory traffic for the input data path


## Solution

Fuse the two kernels into a single fusedBf16Softmax kernel that reads BF16 input directly and converts each element inline using __bfloat162float at the point of use in all three softmax passes (max-reduction, exp-sum, output-write), eliminating the intermediate float buffer and one kernel launch entirely

## Key Lessons

- Fusing a type-conversion kernel with its downstream compute kernel eliminates the intermediate buffer round-trip to global memory — applies whenever a lightweight conversion (BF16→float, int8→float) feeds into a compute kernel that can absorb the conversion inline
- Inline __bfloat162float at the point of use is essentially free in register space compared to the global memory traffic it saves; the conversion instruction latency is negligible versus the write-then-read bandwidth cost of a separate intermediate buffer
