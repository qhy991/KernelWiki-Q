## Challenge

FlashAttention-2 online softmax rescale stores the output accumulator R_D in FP16, requiring half2float conversions to read the accumulator, fmaf in FP32, then float2half conversions to write back — totaling ~512 conversion instructions per thread across all K/V tile steps, each adding pipeline latency and rounding error


## Solution

Promote the R_D accumulator register array from FP16 (2 uint32 per element) to FP32 storage (4 uint32 per element), so the rescale loop reads the accumulator directly as float and performs fmaf with only one half2float conversion for the incoming R_O value, deferring the single float→half conversion to the final output step

## Key Lessons

- Store MMA output accumulators in FP32 registers when they undergo repeated arithmetic (rescale, accumulation across tiles) in FlashAttention — this eliminates half↔float round-trip conversions inside the tile loop, reducing instruction count and avoiding FP16 conversion pipeline latency
- Deferring the single float→half conversion to the final output step is sufficient — intermediate FP32 precision accumulates without loss, and the one-time conversion cost at the end is negligible compared to per-tile conversion overhead
- The register pressure tradeoff (2x registers for FP32 R_D) is acceptable for head_dim=64 with typical tile sizes — verify occupancy limits when scaling to larger head dimensions or more warp tiles
