## Challenge

In a single-block MOE alignment kernel (<<<1,1024>>>), all 1024 threads first initialize sorted_token_ids (32768 global-memory writes, ~32 iterations per thread) before proceeding to the expert counting phase (32768 atomicAdds to shared memory), serializing two comparable-duration phases that have no data dependency


## Solution

Launch <<<2,1024>>> instead of <<<1,1024>>>, delegating sorted_token_ids initialization to blockIdx.x==1 (which returns immediately after init) while blockIdx.x==0 handles counting/alignment; the two blocks run concurrently on separate SMs with no inter-block synchronization needed since block 0 never reads sorted_token_ids during counting

## Key Lessons

- When a kernel has two independent phases of comparable duration (e.g., memory initialization and computation), overlapping them via multi-block launch converts serialized time init+compute into max(init,compute), yielding up to ~2x speedup on the overlappable portion
- Multi-block concurrency requires no inter-block synchronization when the phases have no data dependency — blockIdx.x branching with early return is a lightweight pattern for dispatching independent work to separate SMs
- The achievable speedup from block-level concurrency is bounded by the non-overlappable fraction of the kernel (prefix-sum and expert_ids-fill in this case), which limits improvement as the overlappable portion shrinks relative to total kernel time
