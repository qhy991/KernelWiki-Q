## Challenge

Single-block token-sorting kernel uses a 257×256 global-memory count matrix (~256 KB) with per-thread rows, a serial prefix-sum across 256 threads via global memory reads/writes, and two __syncthreads() barriers, severely limiting parallelism and saturating global memory bandwidth for what is essentially an expert-assignment scatter operation


## Solution

Replace the single-block prefix-sum architecture with a multi-block kernel where each thread atomically claims a slot per expert via atomicAdd on a 256-entry cumsum_buffer, eliminating the count matrix, prefix-sum phase, and __syncthreads barriers entirely while scaling to 16 blocks for full SM utilization

## Key Lessons

- For scatter/write-pattern problems where each element maps to one of N buckets, atomicAdd on a small N-entry counter array is dramatically faster than building a per-thread count matrix plus prefix-sum — the prefix-sum approach requires O(threads×experts) global memory and serial reduction, while atomics need only O(experts) memory and scale across multiple blocks
- Multi-block grid-stride loops (for i = tid; i < numel; i += stride) unlock full SM utilization compared to single-block launches that leave most SMs idle — applies whenever the algorithm has no inter-block dependencies
- When the working set for atomics fits in L2 cache (e.g., 256 int32 = 1 KB), global atomics are effectively as fast as shared memory atomics, making the prefix-sum-avoidance strategy viable for expert counts up to several thousand
