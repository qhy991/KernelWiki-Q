## Challenge

Per-block shared memory totals 28800 bytes with separate allocations for the Phase 1 load buffer (sh_b, 16KB), Phase 2 reduction buffer (sh_red, 12KB), and double-width metadata (128B), limiting occupancy to only 5 blocks/SM under shared memory capacity constraints


## Solution

Overlap the Phase 1 load buffer (sh_b) and Phase 2 reduction buffer (sh_red) at the same shared memory offset since they are used in sequential phases separated by __syncthreads() barriers, reducing per-block shared memory from 28800 to 16448 bytes and increasing occupancy from 5 to 8 blocks/SM

## Key Lessons

- When pipeline phases access shared memory buffers sequentially (separated by __syncthreads() barriers), their buffers can safely overlap at the same offset — this aliasing eliminates the sum of buffer sizes and replaces it with the max, directly reducing per-block shared memory and increasing occupancy on shared-memory-limited kernels
- Reducing shared memory per block by 43% (28800→16448 bytes) lifted occupancy by 60% (5→8 blocks/SM) and yielded 8.9% wall-clock throughput improvement — when occupancy is shmem-limited, co-locating phase-local buffers is often the highest-impact single change
