## Challenge

Scalar row shuffle kernel copies elements one-by-one using individual float32 load/store instructions, processing only 4 bytes per memory transaction and underutilizing available memory bandwidth per instruction


## Solution

Replace individual float32 loads/stores with 128-bit vectorized loads via cutlass::Array<float,4>, packing four floats per memory transaction to increase per-thread memory throughput

## Key Lessons

- Vectorizing loads without proportionally reducing blockDim.x creates thread utilization imbalance — when work is divided by vector width (4x) but thread count stays the same (256), 75% of threads become idle while still occupying warp slots and SM resources, regressing performance
- For memory-bound row-copy kernels where total traffic is fixed, vectorization only helps when all threads remain active; if blockDim.x > num_cols / ELEM_PER_THREAD, the scalar approach with full thread utilization outperforms vectorized loads with idle threads
- When adopting wider vectorized loads (128-bit / 256-bit), reduce blockDim.x proportionally so that blockDim.x <= num_vectorized_elements_per_row, ensuring every thread contributes useful memory traffic
