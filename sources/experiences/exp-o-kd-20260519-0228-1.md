## Challenge

Epilogue kernel with FragmentsPerPartition=1 issues two __syncthreads() barriers per iteration (one before SMEM store, one before SMEM load), totaling 2*kIterations barriers across all iterations — synchronization latency dominates the per-fragment copy work


## Solution

Increase FragmentsPerPartition from 1 to 2: the outer loop strides by 2, an inner loop stores two fragments into separate SMEM regions before the first barrier, then loads two fragments after the second barrier — halving the total barrier count from 2*kIterations to 2*(kIterations/FPP) while keeping per-fragment copy work identical

## Key Lessons

- Batch multiple fragments into a single barrier pair (FragmentsPerPartition>1) to reduce __syncthreads() count proportionally — applies whenever an epilogue stores multiple independent fragments through SMEM and the SMEM capacity can hold FPP fragments simultaneously
- When increasing FPP, SMEM allocation must scale linearly (smem_stride * FPP) and each fragment uses an offset (p * smem_pointer_offset) to avoid overwriting — the total per-fragment copy work is unchanged but barrier latency is eliminated
