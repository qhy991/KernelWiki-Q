## Challenge

Before kernel uses 32x32x64 tile for M<=64 GEMMs, requiring 64 shared-memory load rounds to cover K=4096 and producing only 32 blocks along N=1024, underutilizing GPU occupancy and wasting computation with the wide N-tile when N is small


## Solution

Reconfigure tile from 32x32x64 to 32x8x128: narrower N-tile (8 vs 32) spawns 128 blocks along N for better occupancy, while doubled K-tile depth (128 vs 64) halves K rounds from 64 to 32, amortizing __syncthreads() and shared-memory load overhead across more FMA operations per round

## Key Lessons

- Narrowing the N-tile dimension for small-M GEMMs increases grid parallelism by spawning more blocks along N, improving GPU occupancy — applies when M is too small to saturate the GPU with blocks along M alone
- Doubling the K-tile depth halves the number of __syncthreads() barriers and shared-memory load phases per K-dimension traversal, increasing arithmetic intensity — applies when shared memory capacity permits the larger tile and register pressure remains manageable
- Tile configuration tuning (TM, TN, TK) is the highest-impact knob for parameterized tiled GEMM kernels: for M=32, K=4096, N=1024 the shift from 32x32x64 to 32x8x128 yields 2.1x speedup with no algorithmic change
