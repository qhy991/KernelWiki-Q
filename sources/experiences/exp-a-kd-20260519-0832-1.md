## Challenge

Five separate kernel launches for metadata copy operations (cache_seqlens, cu_seqlens_k, page_table, nsa variants) each incur ~5us launch overhead, wasting ~20us on overhead alone for only ~20KB of total data — launch overhead dominates actual copy time at this small data volume


## Solution

Fuse all 5 metadata copy operations into a single kernel using a parameter struct (FusedMetadataCopyParams) passed via __grid_constant__, with grid-stride loops and #pragma unroll — eliminating 4 kernel launches and improving SM utilization by consolidating all threads into one kernel

## Key Lessons

- Fuse multiple small metadata-copy kernels into a single kernel to eliminate per-launch overhead (~5us per launch) — applies when total data volume is small (<50KB) and launch overhead dominates actual copy time
- Use __grid_constant__ qualifier for struct parameters in fused kernels to enable constant-cache-backed parameter access without extra register pressure, beneficial when all threads read the same parameter fields repeatedly
- Apply #pragma unroll on grid-stride loops over small iteration counts (e.g., bs=32) to fully unroll and eliminate loop branch overhead when the trip count is bounded and known to be small
