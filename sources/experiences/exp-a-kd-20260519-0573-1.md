## Challenge

In a tiled SGEMM kernel with 8x8 thread tiles and double-buffered shared memory, B-tile loads from global memory use synchronous FLOAT4 copies that stall the warp until the transfer completes, preventing overlap with the compute phase of the previous tile


## Solution

Replace synchronous B-tile global-to-shared FLOAT4 copies with cp.async.ca.shared.global instructions that issue non-blocking asynchronous transfers, then use CP_ASYNC_COMMIT_GROUP and CP_ASYNC_WAIT_GROUP(0) to allow B-tile loads to overlap with the A-tile load and BK compute steps, hiding global memory latency

## Key Lessons

- Use cp.async.ca.shared.global to issue non-blocking global-to-shared transfers that can overlap with computation — replaces synchronous FLOAT4 assignment with an asynchronous copy pipeline, hiding global memory latency when there is sufficient compute work to fill the transfer latency
- Larger thread tiles (8x8 vs 8x4) provide more compute per BK step, creating better overlap opportunity for async copy — the async transfer of the next tile can complete while the current tile's 64 FMAs per thread are executing
- Use __cvta_generic_to_shared to obtain a shared-memory pointer for cp.async destination addresses, and pair CP_ASYNC_COMMIT_GROUP with CP_ASYNC_WAIT_GROUP(n) to manage async copy completion ordering
