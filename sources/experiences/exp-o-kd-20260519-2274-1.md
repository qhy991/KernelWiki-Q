## Challenge

The wvSplitKrc kernel uses a fixed K-split granularity of 512 elements per shard with hardcoded template parameters in a static switch over N_p2. For skinny GEMM problems (small M, large K), the coarse 512-shard split leaves many CUs idle because too few shards exist to occupy the GPU, wasting available compute resources.


## Solution

Add a CHUNKK template parameter (1 or 2) that halves the per-shard K-dimension from 512 to 256 when active, and introduce host-side dynamic dispatch that computes CuNeeded = rndup_cus * GrpsShrB and selects chunkk=2 when CuNeeded*2 <= CuCount, doubling K-shard count to fill idle CUs. Template parameters are simplified to fixed THRDS=64, YTILE=16, WvPrGrp=4 with CHUNKK as the sole adaptive variable.

## Key Lessons

- For split-K GEMM on GPUs with many CUs, make the K-split granularity adaptive rather than fixed: compute the number of CUs that the current shard count would occupy, and halve the shard size (double shard count) when twice the shards still fit within available CUs — this fills idle hardware for skinny GEMM shapes
- Simplify template parameter spaces by fixing tunable dimensions to empirically optimal values (THRDS=64, YTILE=16, WvPrGrp=4) and exposing only one adaptive knob (CHUNKK), reducing compile-time binary bloat while retaining runtime flexibility
- Host-side CU-count awareness enables portable performance across GPU generations: the same kernel binary adapts its parallelism to the target device's compute unit count rather than being tuned for a single GPU model
