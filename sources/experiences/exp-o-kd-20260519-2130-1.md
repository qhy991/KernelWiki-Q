## Challenge

Two-kernel pipeline dequantizes Q4_0 to half in the first kernel, then launches a second kernel to convert the intermediate half buffer to BFloat16, incurring an extra kernel launch overhead and a full redundant read-rewrite pass over the output buffer through global memory


## Solution

Fuse the half-to-bfloat16 conversion into the dequantize kernel by introducing a templated convert_from_half specialization that applies __float2bfloat16 at the store point, eliminating the intermediate half buffer, the second kernel launch, and halving global memory traffic for the output path

## Key Lessons

- Fuse type-conversion into the producer kernel's store operation rather than launching a separate conversion kernel — eliminates one kernel launch, one full global-memory read pass, and the intermediate buffer allocation
- Use template specialization (e.g., convert_from_half<__nv_bfloat16>) to make kernels type-agnostic at compile time, so a single kernel body can output half, bfloat16, or other types without runtime branches
- On SM80+ hardware, __float2bfloat16(__half2float(val)) compiles to a single instruction, making the fused conversion at the store point essentially free compared to the cost of an extra kernel launch and global memory round-trip
