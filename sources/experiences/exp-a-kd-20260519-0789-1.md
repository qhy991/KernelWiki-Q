## Solution

Fuse three separate kernels into a single-pass resolve_future_token_ids_kernel that reads input, checks sign, negates+clamps, looks up future_map, and writes the result in one kernel launch, eliminating two intermediate global memory buffers and two extra kernel launches

## Key Lessons

- Fuse multi-step elementwise pipelines (negate → gather → select) into a single kernel to eliminate kernel launch overhead (~2-5us per launch) and intermediate global memory round-trips — especially beneficial for small problem sizes where launch overhead dominates compute time
- Replacing N kernel launches with one fused kernel removes (N-1) intermediate global memory buffers, saving both allocation cost and the memory bandwidth of writing then reading intermediate tensors
- Kernel fusion provides the largest speedup when per-element work is light and problem size is small; for this 256-element case the 1.9x speedup is primarily from eliminating launch overhead rather than reducing compute
