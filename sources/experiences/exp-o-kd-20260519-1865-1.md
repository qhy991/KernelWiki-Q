## Challenge

When m=8 (small batch), the kernel launch grid uses raw m as the block count, producing only 8 blocks — far fewer than available SMs — resulting in low occupancy and poor hardware utilization while each of those 8 blocks must process ~15 padding rows via grid-stride loops


## Solution

Replace the raw m grid dimension with effectiveRows = round_up(m, 128), launching 128 blocks instead of 8 for the m=8 case, so that padding rows are distributed across many more blocks and SMs are fully occupied

## Key Lessons

- Use round_up(m, alignment) as the grid dimension instead of raw m for kernels that must process aligned padding rows — distributing padding work across more blocks increases SM utilization from ~8 to 108+ active SMs in the small-batch regime
- Grid-stride loops over padding data benefit from more blocks even when real data rows are few — each block's padding burden drops from ~15 rows to ~1, improving throughput by 1.43x
