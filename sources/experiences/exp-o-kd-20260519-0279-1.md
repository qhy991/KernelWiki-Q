## Challenge

Split-K reduction kernel launched with <<<M=64 blocks, N=1024 threads>>> causes stride-LDC=64 memory access per warp thread, producing fully uncoalesced global memory transactions across 8 partition iterations, while 1024-thread blocks limit SM occupancy and only 64 blocks underutilize 108 SMs


## Solution

Swap grid/block dimensions to <<<N=1024 blocks, M=64 threads>>> and remap array indices from row+col*ld to col+row*ld, converting stride-64 access into stride-1 contiguous coalesced access; 64-thread blocks (2 warps) raise SM occupancy while 1024 blocks saturate the GPU's 108 SMs, yielding a compounded 27x speedup amplified by the 8-iteration partition loop

## Key Lessons

- Swapping grid/block dimensions to make consecutive thread indices traverse contiguous memory (stride-1) converts uncoalesced strided access into coalesced access — the effect multiplies with the number of loop iterations over the data, producing superlinear speedup
- Smaller thread blocks (64 threads / 2 warps) raise per-SM occupancy by reducing register and shared resource pressure, allowing more concurrent blocks to hide memory latency — prefer fewer threads per block when the kernel is memory-bound and each thread does little work
- Grid size should approach or exceed the number of SMs to saturate the device — 64 blocks on 108 SMs leaves ~40% of compute resources idle, while 1024 blocks keeps all SMs busy with multiple concurrent blocks
