## Challenge

Decomposing a single m16n8k16 MMA instruction into two m16n8k8 MMA instructions doubles the tensor core instruction count from 100 to 200 per kernel launch, incurring full MMA pipeline overhead for each sub-operation


## Solution

Replace one m16n8k16 MMA with two sequential m16n8k8 MMA calls, feeding a[0..1]/b[0] into the first and a[2..3]/b[1] into the second — a hardware compatibility adaptation for sm75 (Turing) where the m16n8k16 shape is not natively supported

## Key Lessons

- Decomposing a larger MMA shape (m16n8k16) into smaller ones (2× m16n8k8) doubles the tensor core instruction count and causes measurable regression on sm80+ hardware where the larger shape is natively supported — always prefer the largest natively available MMA shape for the target architecture
- MMA instruction decomposition is a hardware compatibility workaround for sm75 (Turing) where m16n8k16 does not exist, not a performance optimization — use conditional compilation (__CUDA_ARCH__) to select the optimal path per architecture
- Each MMA call consumes a full tensor core pipeline slot regardless of operand size, so instruction count is the dominant cost factor for small-shape MMA microbenchmarks
