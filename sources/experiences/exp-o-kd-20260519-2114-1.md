## Solution

Offload the sorted_token_ids initialization loop to a second threadblock (blockIdx.x==1) that runs concurrently with the counting/alignment block (blockIdx.x==0) on a separate SM, changing the launch from <<<1,1024>>> to <<<2,1024>>>; since block 0 never reads sorted_token_ids, no inter-block synchronization is needed and total latency shifts from init_time+count_time toward max(init_time, count_time)

## Key Lessons

- Move independent data-initialization loops to a separate threadblock (blockIdx guard + early return) so the init and compute phases overlap on different SMs — works whenever the compute block never reads the buffer being initialized, eliminating the need for inter-block synchronization
- The achievable speedup from two-block concurrency is bounded by max(init_time, count_time)/total_serial_time, so the fraction that can be overlapped shrinks as non-overlappable phases (prefix-sum, expert_ids fill) take a larger share of the kernel
