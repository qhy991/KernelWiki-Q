## Challenge

Logits bitmask kernel uses scalar 4-byte memory accesses to read/write one float per iteration, underutilizing available memory bandwidth when processing a 32k vocabulary with kBitsPerThread=4 where each thread iterates only once


## Solution

Replace scalar float reads/writes with vectorized float4 (128-bit) loads/stores, adding a fast path that writes PackedNegInf() in a single transaction when all 4 elements in a packed group are masked, and a partial-mask path that loads, modifies individual lanes, and writes back

## Key Lessons

- Vectorized float4 access does not guarantee speedup when per-iteration overhead dominates — with a small vocabulary (32k) and kBitsPerThread=4, each thread iterates only once, so the added vectorized logic cost outweighs the bandwidth benefit
- When the bitmask is random (~87.5% partial-mask rate), the vectorized kernel must load-modify-store all 4 floats per group rather than writing a single -inf scalar, increasing memory traffic compared to the scalar path that skips unmasked positions with a single 4-byte write
- Small problem sizes amplify launch and memcpy overhead in benchmarks, reducing the measurable signal from kernel-level optimizations and making vectorization gains harder to observe
