## Challenge

For small-M GEMM (M=8, K=4096, N=8192), the standard row-thread mapping launches only grid.y=M=8 thread blocks, seemingly underutilizing the GPU by limiting parallelism to ~8 blocks across all SMs


## Solution

Swap A and B operands via transposition (compute C^T = B^T * A^T) so that grid.y becomes N=8192 instead of M=8, creating many more thread blocks to increase parallelism — but the transposed memory layout causes B_swap (A^T) to be accessed column-major, thrashing the L2 cache and overwhelming any occupancy benefit

## Key Lessons

- Increasing thread block count by swapping GEMM operands is counterproductive when the transposed operand (A^T as B_swap, K×M) is accessed with column-major stride M=8, causing poor spatial locality and L2 cache thrashing that dominates any occupancy gains
- For small-M batched decode GEMMs, the original layout with A(M×K) row-major and B(K×N) column-major already provides good cache behavior — the apparent low block count (M=8) is sufficient for A800 occupancy (≥16 blocks per SM × 8 rows = adequate parallelism)
