## Challenge

In the 8x4 SGEMM main loop (BM=64, BN=64, 128 threads), B-tile global-to-shared loading is performed synchronously using FLOAT4 stores interleaved with A-tile loading, blocking threads until global reads complete and preventing overlap with the BK compute iterations of the current tile


## Solution

Issue cp.async.ca instructions for the next B tile at the start of each loop iteration and commit the async group, then proceed with A loading and BK compute while B transfers are in flight; CP_ASYNC_WAIT_GROUP(0) before __syncthreads() ensures B async copies complete before the next iteration reuses the shared-memory buffer

## Key Lessons

- Replace synchronous FLOAT4 global-to-shared stores with cp.async.ca to overlap next-tile data transfer with current-tile compute — applies when a tiled kernel has a predictable next-tile address available before the compute phase begins
- Separate B-tile async copies from A-tile synchronous loads and issue CP_ASYNC_COMMIT_GROUP() before A loading so that async B transfers proceed in parallel with A reads and the BK compute loop
- Use __cvta_generic_to_shared() to obtain a shared-memory pointer for cp.async destination addressing, and pair CP_ASYNC_WAIT_GROUP(0) with __syncthreads() to ensure both async completion and thread synchronization before buffer reuse
