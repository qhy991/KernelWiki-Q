## Challenge

MoE token alignment and sorting uses two separate kernels: an align kernel with 1024 threads performing shared-memory atomicAdd for expert counting, and a sort kernel with global-memory atomicAdd for token ranking, incurring two kernel launch overheads and a cudaMemset between them


## Solution

Merge the two-kernel pipeline into a single kernel with only 32 threads, replacing atomic operations with per-thread exclusive counting slots (tokens_cnts[thrIdx * num_experts + expert]) followed by a shared-memory prefix-sum, so each thread writes its own slot without contention and computes final ranks by adding per-thread offsets to the global cumsum

## Key Lessons

- For small-batch MoE token routing (numel <= warp_size * num_experts), a single-kernel per-thread prefix-sum approach eliminates all atomic operations by giving each thread exclusive counting slots — avoiding both shared-memory and global-memory atomicAdd contention
- Kernel fusion that merges an align kernel and a sort kernel into one eliminates kernel launch overhead and intermediate cudaMemset calls, which is significant for short-running kernels where launch overhead dominates execution time
- Thread count reduction from 1024+256 to 32 (a single warp) reduces shared-memory initialization overhead and is sufficient when the workload per thread is small (~8 elements per thread for 256 tokens across 32 threads)
