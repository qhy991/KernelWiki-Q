## Solution

Split the thread configuration table into small_batch_thread_configs (first entry {128,128,256}) and large_batch_thread_configs (first entry {64,256,256}), selected at runtime based on prob_m<=16. The goal was to halve K-block count (64→32) by doubling thread_k, reducing global reduction iterations, but in the simplified scalar kernel the thread_n reduction from 256→128 leaves half the 256 threads idle

## Key Lessons

- Reducing thread_n below num_threads in a scalar (non-tensor-core) kernel leaves threads idle — all threads must have work assigned via the N-stride loop, so thread_n must equal or exceed num_threads for full utilization
- Batch-size-aware config splitting is valid in principle but the config parameters must match the kernel's thread dispatch model: tensor-core MMA kernels distribute work differently (all threads participate in mma.sync) than scalar compute loops
- When evaluating thread_k vs thread_n tradeoffs, the compute-phase slowdown from underutilized threads can dominate the reduction-phase savings from fewer K-blocks, especially for small M where the reduction cost is already low
