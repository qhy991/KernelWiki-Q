## Solution

Store K-cache in FP8-E5M2 format (1 byte/element) instead of fp16, halving K-cache DRAM traffic from ~4 MB to ~2 MB per invocation; dequantize FP8 pairs to fp16 via software integer bit-manipulation (fp8_e5m2_to_half_bits) before computing QK dot products, keeping the arithmetic path in fp16/fp32

## Key Lessons

- FP8-E5M2 KV-cache quantization halves memory bandwidth for attention QK dot-product kernels — the software integer-based FP8-to-fp16 dequantization cost is negligible compared to DRAM bandwidth savings, especially for long sequences where K-cache traffic scales linearly with block count
- Speedup from cache quantization increases with sequence length because K-cache traffic is proportional to num_logical_blocks while fixed overhead (kernel launch, Q loads, warp reduction) stays constant — the 128-block case shows 1.14x vs smaller speedups on shorter sequences
