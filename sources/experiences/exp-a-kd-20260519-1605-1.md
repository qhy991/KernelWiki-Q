## Challenge

K-cache for 8 query heads uses an MHA layout with 8 KV head slots (block_stride=8192), allocating ~8 MB of cache even though heads 0-3 and 4-7 share identical data patterns via a redundant kv_src indirection, wasting 4x memory and reducing spatial locality


## Solution

Replace MHA layout (NKV=8, identity mapping) with GQA layout (NKV=2, head_mapping={0,0,0,0,1,1,1,1}), compressing the K-cache 4x by storing only 2 unique KV heads and routing 4 query heads to each, which reduces total cache from ~8 MB to ~2 MB and improves spatial locality in the paged block table

## Key Lessons

- GQA KV-head compression (NKV reduction) yields large memory savings (4x) with negligible per-kernel speedup (~1.3%), because the per-call dot-product workload per query head remains identical — the benefit is architectural (smaller KV cache footprint enabling longer sequences or larger batch sizes) rather than computational
- The kernel function (kcache_dot_kernel) is layout-agnostic when parameterized with block_stride, head_stride, and head_mapping — this decoupling allows MHA-to-GQA transitions with zero kernel changes, only host-side layout and mapping table adjustments
- Paged attention block_stride should be computed from NKV (actual KV heads) rather than NH (query heads) to reflect the true memory footprint of the K-cache in GQA models
