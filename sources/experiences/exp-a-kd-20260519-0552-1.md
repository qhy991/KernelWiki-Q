## Challenge

The before kernel processes only 2 half values per thread with 128 threads/block, which underutilizes per-thread ILP; increasing to 8 elements per thread with only 32 threads/block (1 warp) drastically reduces SM occupancy and adds 4x more boundary-check branches and separate half2 load/store instructions


## Solution

The after kernel increases elements-per-thread from 2 to 8 (4 half2 pairs), reducing total thread count by 4x and block size from 128 to 32 threads; however this leaves only 1 warp per block, severely limiting SM occupancy, and the 4 separate half2 load/store pairs plus 4 boundary-check branches add instruction overhead that negates the ILP benefit for L2-cache-resident data

## Key Lessons

- Increasing elements-per-thread without maintaining sufficient block size (>=64 threads) collapses SM occupancy — a single-warp block cannot hide memory latency even with higher per-thread ILP
- For bandwidth-bound kernels operating on L2-cache-resident data, fewer threads with more work per thread adds instruction overhead (multiple boundary checks, separate load/store pairs) that negates any latency-hiding benefit
- When scaling elements-per-thread, keep block size >= 2 warps (64 threads) and use vectorized loads (e.g., float4 reinterpret) instead of multiple separate half2 loads to reduce instruction count per element
