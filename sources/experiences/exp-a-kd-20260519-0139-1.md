## Challenge

Warp-level shared-memory iterator constructor uses a triple-nested unrolled loop with a per-iteration conditional branch (if access_idx == ldsm_vec_num) to compute each thread's smem origin offset, generating unnecessary branch instructions and redundant arithmetic that accumulate across repeated iterator construction in hot loops


## Solution

Replace the triple-nested loop and conditional match with three direct integer division/modulo operations on ldsm_vec_num to decompose it into (access_m_idx, inner_idx, inst_m_idx), eliminating all branch instructions and reducing the arithmetic to a fixed three-division sequence

## Key Lessons

- When a loop enumerates all indices and a conditional selects the matching one, replace it with direct integer division/modulo decomposition of the target index — this eliminates branches and reduces the loop overhead to O(1) arithmetic
- In hot-path constructors that execute every iteration (e.g., warp iterators inside GEMM kernels), even small per-construction savings from replacing a triple-nested loop with 3 divisions accumulate measurably across thousands of iterations
- Unrolled loops with conditional branches inside device constructors may not be fully optimized away by nvcc; explicit O(1) index computation via div/mod is a more reliable way to express the intent and avoid residual instructions
