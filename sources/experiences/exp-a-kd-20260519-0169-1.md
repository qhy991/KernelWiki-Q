## Challenge

When sm_occupancy=1, get_block_idx() still invokes FastDivmod to compute dest_sm and dest_wave and enters the remapping branch for the first wave of blocks, even though remapping is a no-op (dest_sm=block_idx, dest_wave=0), wasting device-side compute cycles on unnecessary integer division and branch evaluation


## Solution

Add a (sm_occupancy > 1) guard to skip all remapping logic when sm_occupancy=1, and replace the generic FastDivmod with simple div-by-2 since the remapping divisor is fixed, eliminating both the FastDivmod object construction and the per-block division overhead in the common single-occupancy case

## Key Lessons

- Guard remapping logic with sm_occupancy > 1 to skip no-op FastDivmod computation and branch evaluation when there is only one wave per SM — avoids per-block integer division overhead in the single-occupancy case
- FastDivmod overhead is negligible compared to kernel workload (here ~128 iters of FMAD per thread), so this optimization is only measurable in very lightweight kernels where index computation is a significant fraction of total work
