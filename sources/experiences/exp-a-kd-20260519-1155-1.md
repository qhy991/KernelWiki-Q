## Solution

Quantize fp16 KV cache entries to fp8 e4m3 via __nv_cvt_float_to_fp8 with per-key/value scaling, writing 1 byte per element to uint8_t cache to halve memory write bandwidth — but on sm_80 the fp8 conversion uses a software emulation path through double precision with multiple conditional branches, causing net regression

## Key Lessons

- FP8 quantization of KV cache halves write bandwidth (1 byte vs 2 bytes per element) but __nv_cvt_float_to_fp8 on sm_80 (__CUDA_ARCH__ < 890) executes a software emulation path through double precision with ~20-30 extra arithmetic and branch instructions per element — the compute overhead dominates the bandwidth savings, producing net regression
- On sm_89+ hardware with native cvt.rn.satfinite.e4m3x2.f32 PTX instructions, fp8 conversion becomes a single hardware instruction, making the bandwidth reduction from fp16→fp8 KV cache quantization likely yield net improvement — always gate fp8 paths on architecture capability
