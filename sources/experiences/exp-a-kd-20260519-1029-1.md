## Challenge

Contiguous-block iteration assigns each thread a CEILDIV-sized chunk with a compound bounds check (i < numel && i < start_idx + tokens_per_thread), adding unnecessary branching overhead and preventing memory-coalesced access across warps when threads process non-adjacent elements


## Solution

Replace contiguous-block iteration with grid-stride loop (tid = threadIdx.x, stride = blockDim.x, i += stride) so threads in the same warp access adjacent elements on each iteration, improving memory coalescing potential and simplifying the loop condition to a single bounds check

## Key Lessons

- Stride-based (grid-stride) iteration provides better memory coalescing than contiguous-block iteration when multiple threads per warp iterate over the same array, because warp lanes access adjacent addresses on each stride — but when numel ≈ blockDim.x each thread processes exactly one element and both patterns produce identical access sequences
- Contiguous-block iteration requires CEILDIV and a compound bounds check (two comparisons per iteration) while stride-based uses a single condition — overhead difference is negligible for single-iteration loops but stride-based is preferred for clarity and scalability to larger inputs
- Shared-memory atomicAdd contention dominates execution time in token-counting kernels with few experts, making iteration-pattern changes invisible when atomics are the bottleneck
