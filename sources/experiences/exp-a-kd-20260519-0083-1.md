## Solution

Replace the C++ if-guarded store with an explicit PTX inline asm predicated store (st.global.v4.u32 with @p predicate) via ptx_global_store_16b wrapper, mirroring cutlass::arch::global_store — but this forces mandatory 64-bit pointer materialization, splits float4 into four uint32 registers, and the asm volatile boundary prevents the compiler from fusing address computation and store scheduling, causing a 44% performance regression

## Key Lessons

- NVCC at -O3 already converts simple if-guarded float4 stores into native predicated stores — explicitly writing inline PTX predicated stores adds overhead via forced register materialization (pointer as 'l' constraint, float4 split into four 'r' register operands) and prevents surrounding compiler optimizations
- asm volatile is an optimization barrier that prevents the compiler from fusing address computation with store scheduling, regressing performance versus letting the compiler schedule the same hardware instructions itself
- The CUTLASS arch::global_store abstraction exists for code consistency and correctness guarantees across access sizes, not for performance improvement over compiler-generated predicated stores — do not hand-replicate it expecting speedup
