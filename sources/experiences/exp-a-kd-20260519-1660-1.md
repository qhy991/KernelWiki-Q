## Challenge

Manual scalar float-to-FP8 E4M3 conversion decomposes float2 vectors into individual elements and invokes a multi-step software routine per element (clamp, sign extract, exponent normalize via loops, mantissa extract), creating significant instruction overhead for packed conversion


## Solution

Replace the manual scalar conversion and decomposition with the vectorized __nv_fp8x2_e4m3(float2) vendor intrinsic to convert float2 to fp8x2 in a single constructor call — however, on SM 8.0 the intrinsic uses software emulation with higher overhead than the manual path, resulting in a performance regression

## Key Lessons

- Vendor FP8 intrinsics (__nv_fp8x2_e4m3, __nv_fp8_e4m3) may fall back to software emulation on GPUs without native FP8 hardware support (SM < 8.9), making them slower than manual bit-level conversion despite cleaner source code — always benchmark on the target architecture
- A manually implemented float-to-FP8 conversion using integer arithmetic (sign extraction, exponent normalization via loops, mantissa rounding) can outperform vendor intrinsics on older architectures because the compiler can optimize the scalar path with native FP32 instructions more effectively than the emulation library's generic code
- When targeting mixed-precision conversion kernels, validate that vendor intrinsics map to hardware instructions on the target SM version before adopting them as replacements for working software implementations
