## Challenge

A paged KV-cache gather kernel processes all blocks and tokens inside a single CTA via nested loops (outer: blocks, inner: tokens per block), serializing all copy work and requiring O(blocks) block_table lookups plus divergent conditional branches for partial-block boundary handling


## Solution

Replace single-CTA nested block/token loop with a per-token-direct kernel where each CTA handles one token via a token_to_seq mapping array, computing the source address with a single block_table lookup using batch_offset/block_size arithmetic and launching NUM_TOKENS independent CTAs for full GPU parallelism

## Key Lessons

- Prefer per-token direct kernels over block-level loop kernels for paged KV-cache gather operations — assigning one CTA per token exposes massive parallelism (4096 CTAs vs 1 CTA) and eliminates serial bottleneck within a single CTA
- Use a token_to_seq mapping array to convert flat token indices to batch IDs in O(1), enabling per-token CTAs to compute paged-cache addresses with a single block_table lookup instead of iterating over all blocks
- Eliminate partial-block conditional branches (e.g., last-block size checks) by computing slot_id = batch_offset % block_size per token — divergent branching within a warp is avoided when each CTA processes exactly one token
