## Challenge

Warp-level RMS norm kernel uses only 32 threads per work item with each thread loading 8 packed bf16x2 pairs, limiting memory-level parallelism because fewer threads issue concurrent loads for the 256 packed pairs in a head_dim=512 row


## Solution

Reorganize from warp-level to CTA-level processing: 64 threads (2 warps) each load 4 bf16x2 pairs instead of 32 threads loading 8 pairs, doubling concurrent memory load threads, then use shared memory two-stage reduction with __syncthreads() barriers for cross-warp sum aggregation

## Key Lessons

- Doubling thread count from 32 to 64 to improve memory-level parallelism yields marginal gains for head_dim=512 because the warp-level approach already efficiently covers 256 packed pairs with manageable register pressure (8 pairs/thread) — the added __syncthreads() barriers offset the MLP gains
- Warp-level shuffle reduction (no shared memory, no sync) is competitive with CTA-level shared-memory reduction for small warp counts (2 warps) because the two __syncthreads() barriers cost ~3–5% of kernel time for this workload size
- Persistent-kernel stride patterns (4 warps/block striding across work items) and one-block-per-work-item patterns have different occupancy implications — the persistent approach amortizes block launch overhead while the CTA approach dedicates a full block per work item
