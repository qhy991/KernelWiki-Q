## Solution

Reorder the INT4 weight layout offline so that all 16 INT4 values per thread become contiguous, enabling a single 64-bit vectorized load (uint64_t) instead of 4 scattered 16-bit loads, reducing shared memory load instructions from 4 to 1

## Key Lessons

- Reordering INT4 weight layout from K-major (column-major) to thread-contiguous enables vectorized 64-bit loads (4× uint16_t → 1× uint64_t), reducing shared memory load instruction count — this matters on the critical path between MMA operations in WGMMA mainloops
- Isolated microbenchmarks can show neutral or negative results for shared memory optimizations when the surrounding computation (manual nibble extraction, sign extension, conditional branches) dominates loop time and masks the memory access benefit
- The volatile qualifier forces actual shared memory loads every iteration, preventing compiler caching — when benchmarking memory patterns, volatile adds equal overhead to both sides and can obscure real differences; in production CUTLASS code, the WGMMA mainloop uses specialized hardware (NumericArrayShuffleConverter) where load count reduction directly impacts throughput
