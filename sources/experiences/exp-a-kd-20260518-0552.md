## Key Lessons

- Increasing elements-per-thread without maintaining sufficient block size can hurt performance — a 32-thread block (1 warp) starves the SM of concurrency and cannot hide memory latency, even when total work is unchanged
- For bandwidth-bound kernels on small L2-cache-resident datasets, occupancy matters more than per-thread work; prefer keeping block size >= 128 threads when increasing vectorization factor
- Multiple separate half2 load/store pairs per thread add instruction overhead and boundary-check branches that negate the benefit of reduced thread count when the memory subsystem is not saturated
