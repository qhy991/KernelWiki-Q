## Solution

Replaced block-level dispatch with warp-per-token dispatch: each warp (32 lanes) is assigned to one token, iterating over hidden_dim with stride gridDim.x*32, using WARPS_PER_BLOCK warps per block to map blockIdx.y to token index via blockIdx.y*WARPS_PER_BLOCK+warp_id — intended to concentrate each warp's memory accesses on a single token for better spatial locality

## Key Lessons

- Warp-per-token dispatch regresses over block-level dispatch when token count is small (e.g., 64) because the smaller 128-thread blocks (4 warps) reduce SM occupancy and L1 cache reuse per token compared to 256-thread blocks where 8 warps per SM share the same token's working set
- Dispatch strategy must match workload dimensions: warp-per-token benefits require enough tokens to saturate the GPU and amortize scheduling overhead — with few tokens the overhead of managing more smaller blocks (2048 vs 1024) dominates
- L1 cache sharing across warps within a block is a key occupancy-side effect: 256-thread blocks processing the same token share L1 across all 8 warps per SM, while 4-warps-per-block warp-per-token splits L1 across 4 different tokens, reducing per-token cache hit rates
