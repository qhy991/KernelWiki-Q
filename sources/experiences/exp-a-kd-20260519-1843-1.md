## Challenge

Scale stride divisor is determined by checking the weight type (b_type == kFE2M1f) rather than the scale type's element size, causing incorrect stride computation when a new 8-bit scale type (e.g., kFE8M0fnu) is paired with a non-kFE2M1f weight type (e.g., kFE4M3fn)


## Solution

Replace hardcoded weight-type check (b_type == kFE2M1f) with a semantic property check based on the scale type's element size (is_8bit_scale = s_type.size_bits() == 8), so the stride divisor correctly reflects how many scale elements pack into a 128-bit vector regardless of the weight format

## Key Lessons

- Base conditional logic on the semantic property of the data being operated on (scale element size) rather than the type tag of a related but different entity (weight type) — prevents silent correctness bugs when new type combinations are introduced
- When stride or packing divisors depend on element bit-width, derive them from the target element's size_bits() property instead of enumerating known types — this generalizes automatically to future formats without code changes
