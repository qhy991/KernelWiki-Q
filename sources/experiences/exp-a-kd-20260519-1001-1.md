## Challenge

Each pipeline stage function (pre_compute, launch_gemm, distpatch) independently calls cudaGetDeviceProperties for device-type detection, cudaStreamCreate/Destroy for a per-stage stream lifecycle, and cudaSetDevice as a CUDAGuard equivalent, incurring ~10-15us per stream create-destroy pair and ~5-10us per device property query — totalling ~37us of redundant driver overhead per pipeline iteration


## Solution

Hoist cudaStream_t creation, cudaGetDeviceProperties query, and cudaSetDevice guard to the pipeline caller (pipeline_after), computing them once and passing stream and is_h20_device as parameters to each stage function, eliminating 3 redundant stream create-destroy cycles, 3 redundant device property queries, and 2 redundant cudaSetDevice calls per pipeline invocation

## Key Lessons

- Hoist CUDA runtime state queries (cudaGetDeviceProperties, cudaSetDevice, cudaStreamCreate) to the outermost caller and pass results as parameters — each cudaStreamCreate/Destroy pair costs ~10-15us and each cudaGetDeviceProperties costs ~5-10us in driver overhead, so redundant per-function calls in multi-stage pipelines compound significantly
- Stream lifecycle (create/synchronize/destroy) should be owned by the pipeline orchestrator, not individual stage functions — stage functions should receive a pre-created stream parameter to avoid driver-level memory allocation and context setup overhead on every invocation
- When stage functions use CUDAGuard or at::cuda::getCurrentCUDAStream() internally, the per-call overhead is hidden but accumulates across pipeline stages — profiling host-side latency (not just kernel time) is essential to detect this class of bottleneck
