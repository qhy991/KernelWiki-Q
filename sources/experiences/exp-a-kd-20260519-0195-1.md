## Solution

Replace __shfl_sync-based warp index broadcast with direct arithmetic (threadIdx.x / NumThreadsPerWarp) inside a canonical_warp_idx() helper, eliminating SHFL instructions and the implicit warp convergence barrier while producing identical results since every thread already has its own threadIdx.x

## Key Lessons

- Use direct arithmetic (threadIdx.x / 32) instead of __shfl_sync for warp index computation — every thread can compute its own warp index independently, so shuffle-based broadcast is pure overhead that forces unnecessary warp convergence and wastes SHFL instruction slots
- SHFL instruction savings are negligible when global memory accesses dominate the critical path — in this benchmark the strided HBM2e loads dwarf the two eliminated SHFL instructions per iteration, yielding only 4.2% improvement below the meaningful threshold
- In real GEMM kernels where warp_idx is computed once per tile (not inside a tight loop), the absolute time saved by removing one SHFL is negligible relative to the full matrix multiply workload — focus optimization effort on memory bandwidth and compute throughput instead
