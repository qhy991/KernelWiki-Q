## Challenge

Warp-cooperative concat kernel uses scalar bf16 load/store in strided loops — 18 scalar instruction pairs per warp (16 iterations for 512-element NOPE portion, 2 iterations for 64-element ROPE portion) — resulting in high instruction dispatch overhead and no cache-streaming hints, wasting L2 capacity on read-once data


## Solution

Replace 18 scalar bf16 load/store pairs with 3 vectorized PTX inline-asm operations: 2 iterations of ld128_cs/st128_cs (128-bit = 8 bf16 per lane) for the 512-element NOPE portion and 1 ld32_cs/st32_cs (32-bit = 2 bf16 per lane) for the 64-element ROPE portion, all with .cs cache-streaming hints to avoid L2 allocation on read-once data

## Key Lessons

- Use inline PTX vectorized loads/stores (ld.global.v4.u32 = 128-bit, 8 bf16 elements) instead of scalar element-wise loops to reduce warp instruction count by 6x — applies when source and destination are contiguous and naturally aligned
- Cache-streaming hints (.cs) on global memory accesses prevent unnecessary L2 cache line allocation for read-once/write-once data, reducing cache pollution overhead that is proportionally significant at sub-bandwidth-saturation scales
- Compute vectorized-loop trip counts at compile time via constexpr (dim * sizeof(type) / (vec_bytes * warp_size)) to let the compiler fully unroll and eliminate loop overhead
