## Solution

Replace scalar half loads with half2 vectorized loads so each thread processes two adjacent half elements via a single 32-bit load, computing two FP16 multiplies and an add per thread while halving thread count from 256 to 128

## Key Lessons

- half2 vectorized loads on memory-bound kernels that already coalesce at the warp level may not improve performance — the 32-bit transaction is still narrow relative to the 128-byte cache line, so bandwidth savings are marginal
- Halving thread count to accommodate vectorized loads reduces warp-level occupancy and available scheduling flexibility, which can offset the minor load consolidation benefit in memory-bound workloads
- Vectorization of memory access is most beneficial when it enables wider transactions (e.g., float4/uint4) that approach the cache line width, or when the kernel is compute-bound and the vectorized arithmetic reduces instruction count
