## Challenge

Flash Attention kernel uses kStage=1 single-buffer synchronous gmem-to-smem transfers: CP_ASYNC_COMMIT_GROUP followed immediately by CP_ASYNC_WAIT_GROUP(0) and __syncthreads() creates a serial load-sync-compute dependency chain, stalling the MMA pipeline every tile_K_d iteration while waiting for Q/K copies to complete


## Solution

Upgrade from kStage=1 single-buffer to kStage=2 double-buffer pipeline: add a prologue that pre-loads (kStage-1)=1 tiles of Q/K before the main loop, then issue async copies for the next tile into smem_sel_next while MMA runs on the current tile in smem_sel, deferring CP_ASYNC_WAIT_GROUP(kStage-2) to after MMA to overlap computation with memory transfer

## Key Lessons

- Double-buffering (kStage=2) doubles the shared memory footprint for Q and K tiles, which reduces the number of concurrent thread blocks per SM and can hurt occupancy enough to negate the latency-hiding benefit — this trade-off is especially pronounced at larger problem sizes where many blocks compete for L2 cache
- CP_ASYNC_WAIT_GROUP(kStage-2) instead of WAIT_GROUP(0) allows the async pipeline to have up to (kStage-1) in-flight groups, overlapping MMA computation with the next tile's gmem-to-smem transfer — but only beneficial when occupancy is not the binding constraint
- Prefetch V tiles before the P@V phase using the same double-buffer pattern reuses Q's smem space (since Q is fully consumed after Q@K^T), avoiding additional smem overhead for the V pipeline
