## Challenge

Optimization focus: FP8-E5M2 dequantization of K-cache vectors in the QK dot-product loop.
Problem before: The paged attention kernel always read K-cache vectors in the same dtype as the query (scalar_t), which limited KV cache to fp16/bf16/fp32 precision. For long sequences, the K-cache consumes si


## Solution

Applied change: The kernel now conditionally reads K-cache as FP8-E5M2 packed vectors (Quant_vec, uint8_t-based types) and converts them to the computation dtype (K_vec, e.g. half2) via fp8_e5m2_unscaled::vec_conversion, which maps to hardware-accelerated __nv_cvt_fp8_to_halfraw / __nv_cvt_fp8x2_to_

## Key Lessons

- Expected impact: FP8 K-cache halves the memory footprint and DRAM traffic for key vectors.
