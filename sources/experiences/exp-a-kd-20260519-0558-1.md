## Challenge

RoPE kernel uses scalar 32-bit loads and stores (two separate float reads and two separate float writes per thread), incurring four memory transactions per pair and leaving bandwidth underutilized at large sequence lengths


## Solution

Pack the RoPE computation to process four floats per thread using float4 (128-bit) vectorized loads and stores, computing two adjacent rotation pairs in a single thread and halving the number of memory transactions

## Key Lessons

- Use float4 (128-bit) vectorized loads/stores to process multiple element pairs per thread, reducing memory transaction count by 4x — applies to any elementwise kernel where input elements are contiguous and 16-byte aligned
- Packing two adjacent RoPE rotation pairs into one float4 thread is safe because each pair has its own frequency exponent computed from the correct dimension index, so vectorization does not alter mathematical correctness
- Memory-bound kernels benefit most from vectorized access at large data volumes where bandwidth pressure is highest — the 1.31x speedup at seq_len=8192 confirms that transaction overhead dominates over compute
