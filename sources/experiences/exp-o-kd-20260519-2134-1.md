## Challenge

MoE topk-token argsort uses CUB DeviceRadixSort::SortPairs to sort all topk_ids by expert ID, then a separate build_perms_from_sorted kernel to build permutation arrays — requiring O(N log N) sort with 4 radix passes over 32K elements, plus auxiliary buffers for sorted_ids, sorted_orig_idx, and orig_idx_in, and an extra init_indices kernel to generate identity indices


## Solution

Replace the CUB radix sort + build_perms two-phase pipeline with a single-pass atomic scatter kernel (compute_arg_sorts) that uses atomicAdd on a per-expert write counter to directly place tokens into the correct slot, eliminating the O(N log N) sort, three auxiliary device buffers, and the init_indices kernel entirely

## Key Lessons

- For MoE topk-token routing with a small number of experts, atomic scatter (atomicAdd per-expert counter) is faster than CUB radix sort because it reduces O(N log N) multi-pass sort traffic to a single O(N) pass — the atomic contention is manageable when expert count is small relative to token count
- Eliminating auxiliary sort buffers (sorted_ids, sorted_orig_idx, orig_idx_in) reduces memory allocation and memory bandwidth overhead, which is significant when the sort dominates runtime for small expert counts
- A compute_offsets kernel that writes initial atomic_buffer values from prefix-sum of expert_counts allows compute_arg_sorts to scatter directly into the correct output range without a separate build_perms pass — the atomic counter acts as a natural running index
