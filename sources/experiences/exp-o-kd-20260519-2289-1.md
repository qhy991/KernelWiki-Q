## Challenge

In the wvSplitKrc_ GEMM kernel's A[] global-to-LDS load path, kOffcp was computed without clamping, so when K does not perfectly divide into tile sizes the K-offset spills beyond valid K-dimension range and reads from wrong rows of A[], producing incorrect reduction values


## Solution

Re-enable the min__ clamping guard on kOffcp so that kOffcp = min__(K - A_CHUNK, k_str + kOff), keeping the K-offset within [0, K - A_CHUNK] and causing out-of-range threads to read the last valid chunk as harmless padding data

## Key Lessons

- Clamp K-dimension offsets with min__(K - CHUNK_SIZE, ...) in global-to-LDS load paths so that non-perfectly-tiled K dimensions read the last valid chunk as padding rather than spilling into wrong matrix rows — the outer address guard prevents hardware OOB but not logical OOB
- A commented-out clamping guard is not dead code to be removed — it may be a correctness safeguard for edge-case tile dimensions that the developer temporarily disabled for debugging
