## Challenge

At 2048 tokens the vectorized int2/int approach achieves 1.7x speedup, lower than the 2.1x at 512 tokens. The before kernel's scalar bf16 pattern reads/writes 6 individual 2-byte values per head iteration (4 NOPE + 2 ROPE) without buffering, generating 16*6 = 96 scalar 2-byte transactions per lan...


## Solution

The after kernel's fully coalesced int2/int transactions maintain optimal bandwidth utilization regardless of data volume, so its absolute latency scales linearly with tokens while the before kernel's partially-coalesced scalar pattern scales somewhat better at larger sizes.

## Results

- **speedup**: 1.707x (improved)


## Key Lessons

- At larger token counts, the CUDA memory controller can partially merge adjacent lane accesses into wider sectors because consecutive lanes read from contiguous addresses (lane_id*4 covers indices 0-3, 4-7, 8-11, etc.), achieving some partial coalescing.
- The before kernel's scalar bf16 pattern reads/writes 6 individual 2-byte values per head iteration (4 NOPE + 2 ROPE) without buffering, generating 16*6 = 96 scalar 2-byte transactions per lane per warp-chunk.
