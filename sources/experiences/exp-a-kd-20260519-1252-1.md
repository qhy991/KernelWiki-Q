## Solution

Exploit the fact that E8M0 value 2^(e-127) maps directly to a BF16 with zero mantissa and exponent e: shift odd-positioned bytes right by 1 (Out1 = (q & 0xFF00FF00) >> 1) and even-positioned bytes left by 7 (q<<=7 then mask 0x7F807F80), placing each exponent into the BF16 exponent field with zero mantissa in just 3 integer ops

## Key Lessons

- E8M0 scale factors (exponent-only format, value = 2^(e-127)) can be converted to BF16 by placing the exponent byte directly into the BF16 exponent field — the bias difference (127 for E8M0, 127 for BF16) is identical, so no bias adjustment is needed
- Batched integer shift-and-mask on packed 32-bit values can simultaneously convert two BF16 values (via >>1 to split odd bytes into high-half exponent slots, and <<7 to shift even bytes into position), reducing 8 FP ops to 3 integer ops
- Latency-dominated kernels with small element counts may not show measurable speedup from instruction-count reductions alone — the 1.025x neutral result suggests launch overhead and memory latency mask the ALU savings at this problem size
