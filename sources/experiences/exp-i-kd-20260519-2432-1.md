## Challenge

CUTLASS LDSM kernel skeleton requires implementing a three-stage data pipeline (gmem→smem→rmem→gmem) using tensor abstractions: a strided global-to-shared load, an LDSM-based shared-to-register copy via TiledCopy, and a register-to-global store


## Solution

Implement the three-stage pipeline: (1) strided gmem→smem load with stride equal to tiled_copy thread count, (2) LDSM copy from smem to registers via copy(tiled_copy, src, dst) which maps to hardware LDSM atoms, (3) register-to-global store via copy(rmem, gmem)

## Key Lessons

- Use `size(tiled_copy)` as the loop stride for gmem→smem loading to evenly distribute elements across all threads in the TiledCopy tile — this matches the thread count the LDSM stage expects and avoids shared memory bank conflicts during the subsequent tiled copy
- Use `copy(tiled_copy, tXsX, tXrX)` to invoke hardware-optimized LDSM instructions from smem to registers — the TiledCopy abstraction encodes the LDSM atom, thread layout, and vectorization automatically so no inline assembly is needed
- The gmem→smem→rmem→gmem pipeline with `__syncthreads()` between stages is the canonical CUTLASS pattern for exercising shared memory tensor loads: the barrier ensures all threads have written to smem before any thread issues an LDSM read
