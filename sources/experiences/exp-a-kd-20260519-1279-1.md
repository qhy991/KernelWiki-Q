## Challenge

The before path performs bf16×bf16 GEMM into a bf16 output buffer via cublasGemmEx, then launches a separate cast_bf16_to_fp32 kernel to convert the result to fp32 — incurring an extra kernel launch, an extra 4 KB bf16 write, 4 KB bf16 read, and 16 KB fp32 write per iteration for M=16, N=256


## Solution

Use cublasGemmEx with CUDA_R_32F as the output type (d_C_fp32 buffer, C-type CUDA_R_32F) so the GEMM internally accumulates in fp32 and writes fp32 output directly, eliminating the intermediate bf16 buffer, the separate cast kernel, and its associated memory traffic and launch overhead

## Key Lessons

- cublasGemmEx supports mixed-precision output (bf16 inputs → fp32 output via CUDA_R_32F output type with CUBLAS_COMPUTE_32F) — a single call replaces GEMM + separate cast, eliminating an extra kernel launch and redundant memory round-trip
- For small GEMMs where the compute time is low, the overhead of a second kernel (launch + memory traffic) becomes a significant fraction of total latency — fusing into one library call yields proportional gains
- Removing the intermediate bf16 output buffer also reduces device memory allocation and improves numerical accuracy since the result is never truncated to bf16 before the fp32 conversion
