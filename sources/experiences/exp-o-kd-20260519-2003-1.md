## Solution

Reconfigure tile to BM=32, BN=64, BK=64, TM=1, TN=2 — smaller BM reduces M-dimension padding waste from 87.5% to 50%, doubled BK amortizes each shared memory load over twice as many multiply-accumulate iterations improving compute-to-memory ratio, and reduced shared memory per block (16KB to 12KB) enables higher SM occupancy

## Key Lessons

- Match BM tile dimension to the actual M problem size to eliminate padding waste — when M << BM, most threads compute on zero-padded rows, wasting both shared memory bandwidth and compute cycles
- Larger BK amortizes shared memory load overhead across more inner-loop multiply-accumulate iterations, improving the compute-to-memory ratio especially for thin-M problems where outer-loop iterations are few
- Reconfiguring tile dimensions while keeping total thread count constant (1024) balances reduced padding waste with maintained parallelism — smaller BM/BN compensated by smaller TM/TN preserves occupancy
