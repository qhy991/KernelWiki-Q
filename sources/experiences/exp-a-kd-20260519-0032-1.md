## Challenge

ReLu functor takes value by const reference (T const&) and copies it into a local result variable before comparison, raising the question of whether pass-by-value would eliminate the indirection and improve register usage in the hot loop


## Solution

Replace the two-arg const-reference functor with a single-arg pass-by-value overload where the threshold is hardcoded as T(), relying on NVCC inlining to emit identical PTX since the compiler dereferences const-refs into registers at the call site anyway

## Key Lessons

- For trivially small functors (one comparison + one conditional move), NVCC inlines both const-reference and pass-by-value call sites into identical PTX — the parameter passing convention has zero effect on generated SASS
- In memory-bandwidth-bound kernels processing large arrays (4M floats), any microarchitectural difference in functor calling convention is completely hidden by global memory access latency — tune the memory access pattern instead of the parameter style
- Prefer pass-by-value for small trivially-copyable types in __device__ functors for cleaner API design, but do not expect performance gains — the compiler already optimizes both conventions identically
