## Solution

Double-stage (kStage=2) double-buffering overlaps async gmem→smem copies with HMMA computation: V is prefetched asynchronously during Q@K^T (no wait), and next-K is prefetched during P@V, using CP_ASYNC_WAIT_GROUP(1) to allow one outstanding group while waiting for V

## Key Lessons

- Double-buffering with cp.async pipeline only improves performance when the async gmem loads can be hidden behind sufficient compute; if the tile size (Bc) is simultaneously reduced (Bc=64→32), the reduced arithmetic intensity (8→4 HMMA per tile_K_d) and increased barrier count outweigh the latency-hiding benefit
- Multi-stage pipelining increases synchronization overhead proportionally to the number of tiles (16 tiles for Bc=32 vs 8 for Bc=64); at short sequence lengths (seqlen=512, d=64) the overhead dominates, making single-stage with larger tiles faster
- When applying software pipelining to Flash Attention, preserve the tile dimensions (Bc) across stage variants — changing both the pipeline depth and tile size simultaneously conflates two variables and may hide the true effect of each optimization
