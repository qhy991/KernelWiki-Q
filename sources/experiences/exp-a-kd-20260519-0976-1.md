## Challenge

In a per-token MoE pre-reorder scaling kernel, the scale factor (1.0f / a1_scales_ptr[token_idx]) is recomputed inside each iteration of the expert loop even though it does not depend on the loop variable, redundantly executing a null-pointer check and float division up to topk=8 times per token


## Solution

Hoist the loop-invariant scale computation (null-pointer check and float division) out of the expert loop so it executes once per token instead of once per expert iteration, applying classic loop-invariant code motion

## Key Lessons

- Loop-invariant code motion (hoisting scale/constant computation out of inner loops) is always correct but may produce no measurable speedup when the hoisted operation is a single scalar division dwarfed by thousands of memory operations per iteration — apply it for clarity but do not expect speedup when the inner loop body is memory-bound with high arithmetic intensity imbalance
- Per-token scale factors in MoE pre-reorder kernels are inherently constant across expert iterations (scale depends on token_idx, not k), making them textbook candidates for hoisting even when compilers may already optimize this — explicit hoisting improves readability and guarantees the optimization regardless of compiler version or optimization level
