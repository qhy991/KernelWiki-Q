## Solution

The after code adds OOB guard infrastructure with ThreadMap coordinate decomposition: for each row it computes row_idx, group_idx, cluster_idx via integer modulo/division, reconstructs output_row, then conditionally fetches data — but on a perfectly aligned tile every fetch passes the guard, so all the extra integer arithmetic and branching is pure overhead with no correctness benefit

## Key Lessons

- OOB guard logic (ThreadMap coordinate decomposition + conditional fetch) is correctness infrastructure, not a performance optimization — on aligned tiles where no row is out-of-bounds, every guard check passes, making all the integer modulo/division/multiply and branch overhead pure waste that roughly doubles instruction count
- Per-row integer arithmetic chains (modulo + division + multiply for row_idx, group_idx, cluster_idx decomposition) are expensive inside tight reduction loops — gate them behind a compile-time or runtime flag (e.g., kOobCheck) so aligned-tile fast paths avoid the overhead entirely
- Branch-free accumulation (direct indexed access into fragment) is significantly faster than guarded conditional fetch when the guard always evaluates to true — the conditional branch adds warp divergence potential and prevents the compiler from optimizing the access pattern
