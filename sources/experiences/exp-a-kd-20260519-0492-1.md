## Challenge

FlashAttention MMA kernel uses single-stage (kStage=1) blocking K/V gmem-to-smem copy: K is loaded with CP_ASYNC_WAIT_GROUP(0)+syncthreads blocking the entire block before Q@K^T compute, and V is loaded serially after compute completes, leaving the async copy engine idle during arithmetic


## Solution

Upgrade to kStage=2 double-buffered software pipelining: allocate separate smem slots for K (slot 0) and V (slot 1), issue V async copy to slot 1 before Q@K^T compute so the transfer overlaps with arithmetic, then prefetch the next iteration's K into slot 0 after P@V consumes the current K, using CP_ASYNC_WAIT_GROUP(1) to wait for V while keeping next K in flight

## Key Lessons

- Double-buffering (kStage=2) overlaps async V gmem-to-smem copy with Q@K^T compute using separate smem slots, but the speedup is marginal when compute time is shorter than the transfer latency being hidden — the overlap must be nearly complete to justify the extra smem cost
- Using CP_ASYNC_WAIT_GROUP(N) with N>0 allows waiting for a specific async group while keeping later groups in flight, enabling fine-grained pipeline control where V is waited on but next K continues transferring in the background
- Doubling smem for double-buffering (2x K/V tile slots) can reduce occupancy, which may offset pipeline benefits — measure whether the extra smem cost is justified by the overlap gain for the specific tile dimensions and sequence length
