## Solution

Read input once in pass 1 and write converted float values to the output array, then read from output (cache-hot) in passes 2 and 3 to eliminate redundant global memory reads and type conversions — however, this is neutral because the 128-float row (512 bytes) fits entirely in L2 cache and convert_to_float<float> is a no-op

## Key Lessons

- Cache-aware output buffering (write converted values to output, read back from cache) only helps when the per-block working set exceeds L2 cache residency — for small rows (e.g., 128 floats = 512 bytes) that fit in L2, the extra write traffic in pass 1 negates any read savings
- Eliminating redundant convert_to_float calls only matters when T is a reduced-precision type (half/bfloat16) with actual conversion cost — for float→float the conversion is a no-op so removing it provides zero compute savings
- Multi-pass softmax kernels with small column counts are already cache-friendly on GPUs with large L2 caches (40 MB on A800); the sequential pass pattern keeps data resident in L2 across passes without explicit output buffering
