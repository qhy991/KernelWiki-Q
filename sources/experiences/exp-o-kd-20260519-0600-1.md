## Challenge

Cutlass int8 GEMM with small TileShape<32,64,128> and WarpShape<16,64,64> launches too many threadblocks each computing only 32×64=2048 output elements, underutilizing SM80 tensor cores and suffering from high threadblock scheduling overhead


## Solution

Enlarge TileShape to <128,128,64> and WarpShape to <64,64,64>: each threadblock computes 8× more output elements (16384 vs 2048), each warp does 4× more MMA work in the M dimension, and reducing K from 128 to 64 cuts shared memory per block to improve SM occupancy

## Key Lessons

- Enlarge TileShape M,N dimensions when problem size (M×N) is large enough to saturate SMs with fewer threadblocks — reduces scheduling overhead and gives each block more work to amortize launch cost
- Increase WarpShape M dimension to give each warp more tensor-core MMA iterations, improving instruction-level parallelism within the warp — works when the TileShape can accommodate the larger warp footprint
- Reduce TileShape K dimension to cut shared memory per block when occupancy is limited by shared memory capacity — trades more K-loop iterations for higher concurrent block count per SM
