## Solution

Split the fused kernel into two: the align kernel computes expert counts and cumsum (single-block), then a separate multi-block sort kernel uses global memory atomicAdd on cumsum_buffer to place tokens, enabling multi-SM parallelism for the sort phase

## Key Lessons

- Splitting a fused single-block kernel into two launches adds kernel launch overhead (~5-10us) that can negate parallelism gains when the workload is small (e.g., 1024 tokens producing only 4 blocks) — the added launch cost exceeds the benefit of distributing work across more SMs
- Global memory atomicAdd replaces shared memory atomicAdd when moving the sort to a multi-block kernel, trading lower-latency shared atomics for higher-latency global atomics — only worthwhile when sufficient blocks saturate enough SMs to hide the increased per-operation latency
- Shared memory local_offsets[256] (1024 bytes) in a single-block kernel is lightweight and cache-friendly for small token counts, making the fused approach more efficient than kernel decomposition for workloads under a few thousand tokens
