## Challenge

Both A and B operand async copies use cp.async.ca (cache-all), allocating fetched data in both L1 and L2 caches; for the B operand this unnecessarily pollutes L1 cache lines that are consumed once from shared memory and never re-accessed via L1


## Solution

Introduce cp.async.cg (cache-global, bypass L1) for B operand async copies so that B data is cached only in L2, reserving L1 capacity exclusively for A operand data which may benefit from L1 temporal locality

## Key Lessons

- Using cp.async.cg to bypass L1 for an operand reduces L1 pollution but yields no benefit when the total working set (32KB for 2-stage 64x32+32x64 tiles) already fits comfortably in L1 (128KB on A800) — L1 bypass is effective only when L1 capacity pressure causes actual eviction of useful lines
- cp.async.ca vs cp.async.cg is a hint to the cache hierarchy; choosing the right policy requires analyzing whether the fetched data has temporal locality in L1 — data consumed once from shared memory typically has none
