## Solution

Replace scalar element-wise loop with float4 vectorized per-group dispatch using a lambda that loads/stores 4 floats at once via reinterpret_cast<float4>, iterating over groups externally; compile-time template parameters (STRIDE_I_ZERO, STRIDE_J_ZERO) select between full-vectorized, per-group-vectorized, and scalar fallback paths based on scale tensor layout

## Key Lessons

- Float4 vectorization regresses when per-group work (group_n / VEC_SIZE) is much smaller than blockDim.x — only group_n/VEC_SIZE threads are active per group iteration, starving the SM of warps; prefer flat strided loops that keep all threads busy when group size is small relative to block size
- For fp32 data where scalar loads are already 4-byte (one full memory transaction), float4 vectorization provides minimal bandwidth benefit over scalar coalesced access, making the thread-utilization cost dominate
- Vectorized per-group dispatch becomes viable when group_n >> blockDim.x * VEC_SIZE (e.g., group_n=2048 with VEC_SIZE=4 gives 512 vectors for 256 threads), or when output is narrower than 4 bytes (e.g., fp8 quantization with VEC_SIZE=16) where vectorization significantly boosts store throughput
