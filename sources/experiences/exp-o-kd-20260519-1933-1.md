## Challenge

Grouped GEMM arguments are constructed without KernelHardwareInfo, leaving CUTLASS to use a default or hardcoded SM count assumption for occupancy-based tile scheduling, which misestimates concurrent CTA count and produces suboptimal tile distribution across expert groups in MoE workloads


## Solution

Query the actual device SM count via cutlass::KernelHardwareInfo::query_device_multiprocessor_count and pass it as the fifth argument to GemmKernel::Arguments, giving the CUTLASS runtime accurate occupancy information for proper tile scheduling that matches real hardware capacity

## Key Lessons

- Always provide explicit KernelHardwareInfo with actual device SM count to CUTLASS grouped GEMM arguments — without it, CUTLASS defaults to a hardcoded SM count that may under or overestimate hardware capacity, causing suboptimal tile scheduling especially for MoE workloads where group sizes vary
- Use static const for KernelHardwareInfo to query the device multiprocessor count once and reuse across kernel launches, avoiding redundant device queries while ensuring occupancy-aware scheduling remains accurate for the lifetime of the process
