## Challenge

In a HGEMM TN kernel with a 2D grid, adjacent thread blocks map sequentially to column tiles (bx=0,1,2,...), causing them to access overlapping rows of the B matrix and compete for the same L2 cache sets, reducing effective L2 reuse across the 32x32 block grid on GPUs with large but set-associative L2 caches


## Solution

Enable BLOCK_SWIZZLE to launch a 3D grid where N_SWIZZLE z-slices remap adjacent x-blocks to distant column tiles separated by gridDim.x*BN columns, spreading B-matrix accesses across different L2 cache regions and reducing set contention for a 3% latency improvement

## Key Lessons

- Block swizzle remaps sequential thread blocks to strided column tiles via a 3D grid (bx = blockIdx.z * gridDim.x + blockIdx.x), reducing L2 cache set contention by ensuring adjacent blocks access different regions of the shared B matrix → applies when GEMM grids have many column-tile blocks competing for B-matrix rows in a set-associative L2 cache
- The same kernel code supports both swizzled and non-swizzled launches via a compile-time bool template and the expression ((int)BLOCK_SWIZZLE) * blockIdx.z * gridDim.x + blockIdx.x, which collapses to blockIdx.x when swizzle is disabled — zero-overhead abstraction for grid-level tuning
