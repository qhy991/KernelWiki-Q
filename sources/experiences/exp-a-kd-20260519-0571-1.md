## Challenge

In a double-buffered tiled SGEMM kernel, B-matrix tiles are loaded from global memory into shared memory using synchronous FLOAT4 assignments, blocking the SM's load/store unit and preventing overlap with ongoing computation in the BK compute loop


## Solution

Replace synchronous B-matrix global-to-shared FLOAT4 loads with cp.async.ca.shared.global PTX async copies (via CP_ASYNC_CA macro), committing the async group before compute and waiting (CP_ASYNC_WAIT_GROUP) after compute, so that B-tile DMA transfers overlap with the BK-step FMA computation and A-matrix register spilling

## Key Lessons

- Use cp.async.ca.shared.global to issue asynchronous global-to-shared copies that proceed in the background while the SM executes FMA compute, eliminating the stall that synchronous FLOAT4 assignments cause on the load/store pipeline — applies to any tiled kernel with a double-buffered shared memory scheme where compute and load phases can overlap
- Decouple the async copy lifecycle with CP_ASYNC_COMMIT_GROUP before entering the compute section and CP_ASYNC_WAIT_GROUP(0) after compute completes, so the async DMA engine works in parallel with arithmetic instructions rather than serializing them
- The relative speedup from async copy is larger when per-tile compute time is short relative to load latency; at larger tile counts (more BK iterations) the compute already partially hides load latency, yielding a more modest gain (~7% at M=N=K=1024 with BK=16)
