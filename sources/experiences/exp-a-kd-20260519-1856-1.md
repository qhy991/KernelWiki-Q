## Challenge

The mmq.cuh permutation-loop ds-loading pattern uses modulo scatter indexing (% mmq_x) to distribute tile columns across warps, which runs with full warp occupancy but cannot express MoE-specific per-token column mapping through pre-loaded token offsets


## Solution

Replace the permutation-loop modulo-scatter pattern with direct threadIdx.y-based indexing that pre-loads per-token column offsets, enabling MoE-specific column mapping at the cost of introducing warp divergence (75% of threads inactive under threadIdx.x < 8 guard)

## Key Lessons

- Replacing a permutation-loop modulo-scatter pattern with direct threadIdx.y indexing introduces severe warp divergence when a threadIdx.x guard deactivates most threads — the 75% inactive threads in this case caused a 2.2x slowdown, which is acceptable only when the new pattern enables functionality (per-token MoE column mapping) that the permutation loop cannot express
- Pre-loading token offsets into registers (token_offs[mmq_x/nwarps]) enables per-warp column mapping for MoE expert routing but adds register pressure and requires padding the token_offs_arr to avoid out-of-bounds access when threadIdx.y + i exceeds the block's column range
