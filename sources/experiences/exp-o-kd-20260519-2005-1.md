## Challenge

Default GEMM tile configuration (BM=128, BN=128, BK=32, TM=4, TN=4) allocates 16KB shared memory per block (128×32+32×128 half elements), limiting the number of concurrent blocks per SM and underutilizing GPU parallel resources even when M=128 perfectly tiles into BM=128


## Solution

Reduce tile BM from 128 to 64 (with TM from 4 to 2) to cut shared memory per block from 16KB to 12KB, enabling more concurrent blocks per SM and higher occupancy; the smaller tile splits M=128 into 2 blocks in the y dimension, distributing work across more parallel blocks for better GPU utilization

## Key Lessons

- Reducing tile BM (and corresponding TM) to cut shared memory per block can yield higher occupancy and throughput even when the larger tile perfectly fits the M dimension — occupancy gains from reduced smem can outweigh the loss of per-block compute efficiency
- Smaller tiles increase the grid dimension (e.g., 1→2 blocks in y), distributing work across more CTAs and improving GPU utilization for medium-sized GEMM problems where the larger tile leaves SM resources idle
- Tile size tuning is M-range dependent: a default large tile (BM=128) may be optimal for very large M, but for M in (64,128] a smaller tile (BM=64) achieves better occupancy through reduced shared memory pressure
