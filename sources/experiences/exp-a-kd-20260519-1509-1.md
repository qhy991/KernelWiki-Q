## Challenge

Custom blockReduceSum uses an if-else conditional dispatch on num_tokens to select between blockReduceSum<float,1024> and blockReduceSum<float,256> template specializations, adding code complexity and requiring a hand-rolled warp-shuffle plus shared-memory cross-warp reduction chain


## Solution

Replace the custom blockReduceSum with if-else dispatch with a single cub::BlockReduce<float,1024> call using runtime numValidItems=blockDim.x, delegating warp-shuffle and cross-warp aggregation to the CUB library

## Key Lessons

- Replacing a hand-rolled block reduction with cub::BlockReduce and runtime numValidItems simplifies code without regressing performance — both follow the same 4-warp shuffle → shared-memory cross-warp pattern, so the library call is a zero-cost abstraction
- If-else dispatch on a runtime-but-launch-constant value (num_tokens) does not cause warp divergence because all threads in a block evaluate the same branch — the branch predictor and compiler constant-folding absorb the cost, making the conditional dispatch harmless but still worth removing for code clarity
