## Challenge

A cudaMemcpy DeviceToHost call between two kernel launches creates an implicit synchronization barrier that drains the CUDA stream and blocks the host thread, producing a pipeline gap where no GPU work executes; for small kernels this sync overhead (~2.4ms) dominates total latency


## Solution

Eliminate the cudaMemcpy DeviceToHost by keeping the scalar on device and passing a device pointer to the kernel, which reads the value via dereference (*d_scale_ptr), enabling back-to-back kernel enqueue on the same stream without host-side blocking

## Key Lessons

- cudaMemcpy DeviceToHost creates an implicit stream synchronization that blocks the host and drains the GPU pipeline — for small kernels where compute takes microseconds, this sync overhead dominates total latency; keep scalars on device and pass device pointers instead
- Reading a scalar via kernel-side device pointer dereference (*d_ptr) enables back-to-back kernel enqueue on the same stream without any host involvement, maximizing stream occupancy and eliminating pipeline bubbles for chained small-kernel operations
