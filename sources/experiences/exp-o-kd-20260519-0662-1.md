## Solution

Flatten the 2D nested loop into a 1D global-index stride pattern where each thread computes globalIdx = blockIdx.x*blockDim.x+threadIdx.x and walks all numRows*colsPerRow elements, deriving rowIdx via division and colIdx via modulo; size the grid by totalWorkSize instead of numRows and apply a grid/block balancing loop that doubles grid count while halving block size to maximize SM occupancy

## Key Lessons

- Flatten 2D nested loops (blockIdx.x for outer, threadIdx.x for inner) into a 1D global-index stride when the outer dimension is small — this allows the grid to be sized by total work rather than by the small outer dimension, enabling full SM occupancy
- Grid/block balancing loops that double grid while halving block size (down to a minimum of 64 threads) redistribute work across more thread blocks, increasing occupancy on GPUs with many SMs — applies whenever the initial grid is smaller than multiProcessorCount
- Size the grid by totalWorkSize (rows × cols) rather than by the row count alone when the iteration pattern is dominated by the column dimension — prevents underutilization when the problem has few rows but many columns
