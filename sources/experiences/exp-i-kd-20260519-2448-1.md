## Solution

Advance each TensorRef to the correct batch via add_pointer_offset(threadIdx.y * stride) before constructing iterators, then build IteratorA/B/Params from layout, compute with the Gemv operator (clear accum → gemv() call), and store through IteratorC which assumes PitchLinear n()-contiguous output

## Key Lessons

- Use add_pointer_offset(threadIdx.y * stride) on each TensorRef before iterator construction to implement batched GEMV — the stride parameter controls inter-batch spacing and threadIdx.y indexes batches in parallel without needing a grid-level batch dimension
- IteratorA for GEMV takes thread_id=0 (full-row access) while IteratorB takes threadIdx.x (column-partitioned) — this asymmetry reflects the matrix-vector nature where each thread reads the full A row but partitions the B vector
- CUTLASS Gemv operator signature is gemv(problem_size, accum, iterA, iterB, accum) where the same accum serves as both input (source for residual add) and output — always call accum.clear() before the first invocation to avoid garbage accumulation
