## Challenge

Paged attention K-cache stores key vectors as FP16 (2 bytes/element), consuming double the memory bandwidth necessary during the QK dot-product phase; each K-cache element is loaded as a half2 vector from fp16 memory, incurring 2 bytes of global memory traffic per element across all blocks


## Solution

Compress K-cache storage from FP16 to FP8-E5M2 (1 byte/element), halving memory traffic; load 2x FP8 values packed as uint16_t, then dequantize to half2 via software bit-manipulation (extract sign/exponent/mantissa, reconstruct fp16 bit pattern using shift/mask/or integer operations) before computing the QK dot product

## Key Lessons

- Compress K-cache to FP8-E5M2 (1 byte/element) instead of FP16 to halve memory traffic in attention kernels — the 2x bandwidth reduction outweighs the small integer conversion cost even with software bit-manipulation dequantization
- Software FP8-to-FP16 conversion via shift/mask/or on integer registers compiles to a compact instruction sequence that avoids hardware emulation overhead — preferable when native FP8 conversion intrinsics are unavailable
- Memory-bound kernels benefit from reduced precision even when computation remains at higher precision — the FP8 storage only affects the load path while the dot product stays in FP32, preserving numerical accuracy
