## Challenge

The before path launches two separate kernels sequentially on the same stream: kernel 1 computes token, seq_len, position, and slot_mapping updates; kernel 2 re-reads the already-incremented seq_lens to derive paged_kv_last_page_len and block_table_bound. ## Solution

## Results

- **speedup**: 1.431x (improved)


## Key Lessons

- Eliminating one kernel launch saves the second launch's driver overhead entirely, and the fused kernel also avoids re-loading seq_lens_ptr and block_tables_ptr from global memory in a second pass, yielding a consistent ~1.4x speedup across all batch sizes.
