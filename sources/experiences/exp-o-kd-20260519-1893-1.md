## Solution

Eliminate the GPU-side segmented_max_reduction entirely by pre-computing the max-abs scale on the CPU host using a simple loop over the already-available host input buffer, then upload the static scale and launch only the quant kernel — removing one full kernel launch, one data traversal, and all shared-memory reduction overhead

## Key Lessons

- For small tensor sizes where input data already resides in host memory, pre-computing reduction-derived scales on CPU and uploading a single scalar eliminates an entire GPU kernel launch plus shared-memory reduction overhead — the CPU loop cost is negligible relative to saved kernel launch latency and data traversal
- Kernel launch overhead is significant relative to compute for lightweight kernels on small workloads (4096 elements); consolidating two kernel launches into one can yield near-2x speedup even when the remaining kernel is unchanged
- When the quantization scale can be pre-determined (static or already-known range), always prefer a static scale path over dynamic GPU-side reduction — the dynamic path is only justified when the input range is truly unknown at dispatch time
