## Solution

Replace the weight-type-specific branch condition (is_fe2m1f) with a scale-property-based condition (is_8bit_scale) so that any 8-bit scale format (including MXFP8 kFE8M0fnu) correctly selects the int2 64-bit load path with doubled stride, while 16-bit scales use the int4 128-bit path

## Key Lessons

- Prefer property-based branch conditions (e.g., is_8bit_scale) over type-ID checks (e.g., b_type_id != kFE2M1f.id()) for data-width decisions — property-based logic is automatically correct for new formats that share the same property, avoiding silent correctness bugs
- Semantic generalizations that compile to identical PTX for existing code paths are zero-risk correctness improvements — when refactoring branch conditions, verify the new condition evaluates identically for all previously-supported types to confirm no performance regression
