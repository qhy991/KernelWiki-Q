## Challenge

Grid size for the quantization kernel is computed from raw row count m instead of effectiveRows (round_up(m, 128)), which may under-launch blocks when the kernel's sf_out zeroing loop iterates over padded rows up to sf_m = round_up(m, 128), leaving some padding rows unzeroed when m is not a multiple of 128


## Solution

Replace raw m with effectiveRows = round_up_int(m, 128) in the grid size calculation so that enough blocks are launched to cover all padded sf_out rows that the kernel zeros out in its first loop

## Key Lessons

- When a kernel iterates over padded dimensions (e.g., sf_out rows up to round_up(m, alignment)), the grid size must be derived from the padded count (effectiveRows) rather than the raw dimension to ensure all iterations are covered — this is a correctness concern that only manifests as a performance difference when the padded and raw counts diverge
- Benchmarks with dimensions that are already aligned (e.g., m=256 with alignment=128) cannot detect grid-sizing bugs — always test with non-aligned dimensions (e.g., m=130) to expose the gap between raw and effective row counts
