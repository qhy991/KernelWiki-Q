## Challenge

Split-K reduction uses kCount=8 with TILE_COLS=256, producing fewer thread blocks in the y-dimension (4 blocks for 1024 columns) and a larger per-thread accumulator array, potentially underutilizing GPU parallelism


## Solution

Reduce kCount from 8 to 4 (TILE_COLS from 256 to 128), doubling the number of thread blocks in y-dimension and halving per-thread accumulator register pressure; however at 1024x1024 the kernel is memory-bound streaming float workspace across 8 partitions, and both variants already achieve near-peak bandwidth, yielding neutral performance

## Key Lessons

- For memory-bound split-K reduction kernels that already saturate memory bandwidth, reducing per-thread tile width (kCount) to increase thread block count yields no performance improvement — the bottleneck is memory throughput, not SM occupancy or parallelism
- Grid size tuning via kCount/TILE_COLS adjustments only benefits performance when the kernel is compute-bound or when the original grid is too small to populate all SMs on the GPU
