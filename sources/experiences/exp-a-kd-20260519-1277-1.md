## Solution

Replace the two-pass bf16-GEMM + cast-kernel pipeline with a single cublasGemmEx call that specifies CUDA_R_32F as the output type — cuBLAS accumulates in fp32 internally and writes fp32 output directly, eliminating the intermediate bf16 buffer, the extra kernel launch, and the global-memory round-trip

## Key Lessons

- cuBLAS supports mixed-type output via cublasGemmEx: bf16 inputs with CUDA_R_32F output type lets the library accumulate in fp32 and write fp32 directly, avoiding a separate dtype-conversion kernel — applies whenever F.linear or similar ops need bf16→fp32 output conversion
- For small M (e.g. single-token inference), the GEMM compute is trivial and kernel launch overhead dominates — eliminating even a lightweight cast kernel yields measurable speedup because the fixed driver/grid launch cost is non-trivial relative to the tiny GEMM
