## Challenge

Two separate kernel launches (copy_kv_c_kernel for 256 kv_c elements, copy_k_pe_kernel for 16 k_pe elements) write to adjacent regions of the same KV cache buffer; with only 4 tokens and tiny dimensions, each kernel's driver-side launch overhead (~5-10us) exceeds its actual GPU execution time, wasting over half the total latency on dispatch


## Solution

Fuse the two kernels into a single launch using a templated kernel (concat_and_cache_mla_kernel) with a device-side lambda that copies kv_c (offset 0) and then k_pe (offset kv_lora_rank) into contiguous cache slots, amortizing one driver dispatch overhead across both copy phases

## Key Lessons

- Fuse multiple small kernels that write to adjacent regions of the same buffer into a single launch using a device-side lambda or sequential copy phases — eliminates one full kernel launch overhead (~5-10us per launch) which dominates total latency when per-kernel work is light (few tokens, small dimensions)
- Device-side lambdas let a single thread block iterate over multiple copy phases sequentially with different sizes; even when most threads are idle in the smaller phase (16 of 256 threads active for k_pe), the cost is negligible compared to saving an entire kernel launch
- Kernel launch overhead is the dominant cost at small scale — two launches for 4 tokens with 256+16 elements wastes more time on driver dispatch than on actual GPU computation, making kernel fusion the highest-impact optimization for low-token-count scenarios
