## Challenge

Two-pass softmax pipeline launches a separate convertBf16ToFloat kernel that writes a 2MB intermediate float buffer (8192×64×4 bytes) to global memory, which the subsequent softmaxFloat kernel must re-read, doubling memory traffic on the input data path and incurring an extra kernel launch overhead


## Solution

Fuse the BF16→float conversion into the softmax kernel by inlining __bfloat162float calls at each of the three softmax passes (max-reduction, exp-sum, output-write), eliminating the intermediate buffer and one kernel launch entirely while reading BF16 input once and writing float output once

## Key Lessons

- Fuse type-conversion kernels with compute kernels when the conversion is lightweight and the intermediate buffer creates a global memory round-trip — inline __bfloat162float at each use site eliminates one kernel launch and the full intermediate buffer allocation/read/write
- Kernel fusion that eliminates a global memory intermediate buffer is especially beneficial when the compute kernel has a small working set per thread (64 experts here), because the relative cost of the extra memory traffic dominates over the duplicated conversion work
- Reading input data once and writing output once (vs. write-then-re-read through an intermediate buffer) halves global memory traffic on the input path — applies to any producer-consumer kernel pair where the intermediate data does not need to persist
