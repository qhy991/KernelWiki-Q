## Solution

Vectorize to half2 loads/stores so each thread processes two elements via the HALF2 reinterpret-cast macro, halving the number of global memory transactions and threads per block (256→128)

## Key Lessons

- Half2 vectorization alone does not guarantee speedup — halving threads per block (256→128) can reduce occupancy enough to offset coalescing gains, especially on memory-bound elementwise kernels at moderate input sizes
- Vectorization benefits are workload-size dependent: at 4M half elements (8 MB) the kernel may be too small for the bandwidth difference to dominate over launch overhead and occupancy effects
- Reusing the same scalar compute function (elu_half) on each half2 lane preserves correctness while attempting to reduce memory traffic — the bottleneck is occupancy, not arithmetic throughput
