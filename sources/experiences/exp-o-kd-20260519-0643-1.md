## Challenge

LSE arrays use a transposed [num_heads, num_tokens] layout with head_idx*num_tokens stride, causing threads in a warp to issue strided loads with 2048-byte spacing when reading different heads for the same token, wasting DRAM bandwidth and L2 cache lines


## Solution

Change LSE storage from transposed [num_heads, num_tokens] to row-major [num_tokens, num_heads] layout so that threads processing consecutive head indices for the same token access contiguous 4-byte floats, enabling coalesced memory loads that fit within 2 cache lines instead of 64 strided lines

## Key Lessons

- Prefer row-major [tokens, heads] layout over transposed [heads, tokens] for per-token multi-head arrays when warp threads iterate over heads for the same token — row-major enables coalesced 128-byte loads while transposed forces strided accesses spanning many cache lines
- When thread-to-data mapping uses consecutive head_idx values within the same token_idx, the innermost array dimension should match head_idx (not token_idx) to satisfy the GPU coalescing requirement that consecutive threads read consecutive addresses
