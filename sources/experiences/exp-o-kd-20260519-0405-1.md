## Challenge

Scalar rms_norm_f32_kernel processes one float per thread, issuing separate 32-bit loads/stores for each element, underutilizing memory bandwidth and requiring K threads per row to enter block_reduce_sum with individual scalar values


## Solution

Vectorize to float4 (128-bit) loads/stores via FLOAT4 reinterpret, processing 4 contiguous floats per thread, reducing thread count from K to K/4 and pre-accumulating 4 partial squares into one float before the block reduction

## Key Lessons

- Vectorize scalar float loads/stores to float4 (128-bit) via FLOAT4 reinterpret for memory-bandwidth-bound kernels like rmsnorm — consolidates 4 separate 32-bit transactions into one 128-bit transaction, improving memory throughput
- Pre-accumulate partial squares from vector lanes (reg_x.x²+reg_x.y²+reg_x.z²+reg_x.w²) into a single float before entering block reduction — reduces the number of values in the reduction chain by 4x, shortening barrier and synchronization overhead
- Scale NUM_THREADS proportionally when vectorizing (256 → 256/4) to maintain full-block occupancy while processing the same data with fewer threads, reducing scheduling overhead
