## Challenge

Fixed launch configuration uses 1024 threads and experts_per_warp=32 (full warp) regardless of actual expert count, wasting shared memory (4096 bytes) and SM occupancy when num_experts=8 only needs 8 expert slots


## Solution

Adapt thread count to 256 and experts_per_warp to 8 for small expert counts, reducing shared memory from 4096 to 32 bytes and eliminating idle warps — however the 4x increase in per-thread token iteration count (8→32 tokens) in the counting loop dominates, causing a net regression for workloads where parallelism outweighs resource savings

## Key Lessons

- Reducing thread count for small expert counts saves shared memory but multiplies per-thread iteration count in the counting loop — when token-counting parallelism dominates (8192 tokens), fewer threads means more serial atomicAdd work and worse latency (0.764x regression)
- Adaptive MoE kernel configurations targeting GPU resource utilization (freeing SMs for concurrent kernels) can regress single-invocation latency for small workloads where parallelism is the bottleneck, not resource contention
