## Solution

Replace the hardcoded float accumulator with an opmath_t<double>=double accumulator and matching identity helpers, so inputs are summed at full fp64 precision without truncation; the opmath_t pattern generalizes this so the same kernel template uses float accumulation for fp16/bf16 inputs and double for fp64

## Key Lessons

- Accumulator type must match or exceed input precision — a float accumulator silently truncates 29 mantissa bits from each double element, producing errors in the 1e-7 range even for modest sum lengths
- The opmath_t<T> pattern (float for fp16/bf16 inputs, double for fp64 inputs) lets a single kernel template select the correct accumulation precision at compile time without runtime overhead
- For memory-bound reduction kernels on small problem sizes, fp64 compute overhead is negligible because latency is dominated by memory access; the primary benefit of correct accumulation is precision, not speed
