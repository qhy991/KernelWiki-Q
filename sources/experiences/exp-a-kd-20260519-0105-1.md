## Solution

Replace the single cooperative reduction kernel with a dispatch to a separate reduction path: lightweight store kernels eliminate the spin-wait barrier for subsequent peers, while a dedicated finalize kernel handles output — freeing SK compute blocks to continue working on other tiles

## Key Lessons

- Separating reduction finalization into a dedicated kernel eliminates spin-wait barriers in the compute path and allows SK blocks to overlap reduction with further tile computation — but only yields a net benefit at scale where launch overhead is amortized across many concurrent tiles
- Microbenchmarks with few tiles can show regression for architectural changes that trade kernel launch overhead for compute overlap, since the overhead dominates at small scale
- Dispatch-based reduction strategies (cooperative vs separate) should be selected dynamically based on tile count and GPU occupancy to avoid paying separation costs when cooperative reduction is sufficient
