## Solution

Replace the scalar stride loop with an alignment-aware vectorized template (vectorize_with_alignment<VEC_SIZE=4>) that computes the pointer misalignment at runtime, processes a small scalar prefix to reach alignment, then issues 128-bit wide vec_n_t<float,4> loads and stores for the main body, finishing with a scalar tail — reducing total memory transactions by 4x and improving L2 cache line utilization.

## Key Lessons

- Use alignment-aware vectorized loads/stores (VEC_SIZE=4 for 128-bit width) instead of scalar element-wise loops for memory-bound row-wise operations on large hidden sizes — the runtime alignment calculation cost is negligible when each thread processes many vectors (e.g., 16384/4/256=16 vectors per thread)
- Structure vectorized kernels with a scalar prefix + vectorized body + scalar tail pattern to handle arbitrary pointer alignment safely — cudaMalloc guarantees at least 256-byte alignment, but row offsets (token_idx * hidden_size * sizeof(float)) may break narrower alignment boundaries
- Express elementwise operations as dual lambda pairs (scalar_op + vec_op) passed to a generic vectorize_with_alignment template — this decouples the memory access pattern from the computation and enables reuse across different elementwise kernels
