## Challenge

Per-token absolute-max finding kernel uses scalar loads (one __half per LDG transaction) in a stride loop, issuing hidden_dim / blockDim separate memory transactions that severely underutilize available memory bandwidth


## Solution

Replace scalar loads with 128-bit vectorized loads via uint4 reinterpret_cast, fetching 8 fp16 elements (vec_size=8, 16 bytes) per LDG.128 transaction to reduce total memory transactions by 8x and saturate memory bandwidth

## Key Lessons

- Use uint4 reinterpret_cast for 128-bit vectorized loads to fetch 8 fp16/bf16 elements per LDG.128 transaction instead of one per scalar LDG — reduces memory transaction count by 8x and delivers consistent ~1.7x speedup across both fp16 and bf16 types since they share the same 2-byte element size
- Vectorized loads via uint4 reinterpret_cast apply uniformly to any 2-byte element type (fp16, bf16, int16) because the vectorization factor depends on element size not data interpretation — same vec_size=8 and same bandwidth improvement
