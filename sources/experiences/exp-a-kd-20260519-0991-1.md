## Solution

Decompose the monolithic kernel into two phases: keep the counting/cumsum in a single-block kernel, then extract the sorting scatter into a dedicated multi-block kernel (<<<128, 256>>>) that uses global-memory atomicAdd on cumsum_buffer, achieving near 1:1 thread-to-token mapping and distributing atomics across all SMs

## Key Lessons

- Decompose monolithic single-block kernels into phase-separated multi-block kernels when a scatter/sort phase can independently scale — the counting/cumsum phase may need single-block serialization, but the scatter phase benefits from full-GPU parallelism
- Prefer global-memory atomicAdd across many SMs over shared-memory atomicAdd within one block for scatter workloads with high contention — distributing atomics to 128 blocks reduces hotspot contention compared to 1 block of 1024 threads
- Use grid-stride loops (for i = tid; i < numel; i += stride) in multi-block kernels to achieve natural load balancing regardless of input size to thread-count ratio
