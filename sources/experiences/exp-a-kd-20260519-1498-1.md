## Challenge

Custom vllm block-wide max reduction uses a two-phase warp-shuffle plus shared-memory approach with function-pointer dispatch (ReduceFnType<T> = T(*)(T,T)), introducing indirect call overhead and manual shared-memory barrier management for cross-warp coordination at blockDim.x=1024


## Solution

Replace the hand-rolled two-phase block reduction (warp shuffle + shared memory with function-pointer dispatch) with CUB BlockReduce<float,1024> using cub::Max{} functor dispatch, consolidating warp reduction, shared-memory staging, and cross-warp reduction into a single well-tested library primitive

## Key Lessons

- CUB BlockReduce is not universally faster than hand-rolled reductions for specific single operations like max — at full block sizes (1024 threads), CUB's general-purpose reduction infrastructure introduced ~3.75% overhead versus a tightly optimized single-operation custom implementation → applies when evaluating whether replacing a custom reduction with CUB will actually improve performance
- Function-pointer dispatch (T(*)(T,T)) may be fully inlined by nvcc at -O3, so the expected functor-over-function-pointer advantage of CUB may not materialize at runtime — the compiler can eliminate the indirect call overhead in both paths → applies when comparing function-pointer vs functor dispatch patterns in CUDA device code
