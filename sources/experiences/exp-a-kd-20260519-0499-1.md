## Challenge

The FlashAttention-2 MMA kernel with kStage=1 uses synchronous Q/K loading, issuing CP_ASYNC_WAIT_GROUP(0) immediately after each async copy, which stalls the MMA compute pipeline while waiting for global-to-shared-memory transfers to complete on every tile_K_d iteration


## Solution

Switch from kStage=1 (synchronous) to kStage=2 (two-stage pipelined) loading using double shared-memory buffers (smem_sel / smem_sel_next) and async prefetch of the next Q/K tile while computing the current tile via HMMA16816F32, overlapping CP_ASYNC_CG transfers with MMA compute — however for seq_len=128 with only Tc=2 outer-loop iterations, the pipeline warmup overhead and extra barriers cause net regression of 0.882x

## Key Lessons

- Multi-stage async pipeline (kStage=2) introduces overhead through double-buffered shared memory, extra CP_ASYNC_COMMIT_GROUP calls, and deferred CP_ASYNC_WAIT_GROUP + __syncthreads() barriers — this overhead only pays off when there are enough outer-loop iterations (Tc) to amortize the pipeline warmup cost
- For short-sequence FlashAttention kernels where Tc is small (e.g., seq_len=128 / Bc=64 = 2 tiles), synchronous single-stage loading (kStage=1) outperforms pipelining because the warmup phase consumes a significant fraction of total iterations, leaving minimal opportunity for compute-memory overlap
- The CP_ASYNC pipeline requires (kStage-1) pre-fetch tiles before the main compute loop starts; when the total iteration count is close to kStage, the pipeline never reaches steady-state and the additional synchronization barriers cause net slowdown
