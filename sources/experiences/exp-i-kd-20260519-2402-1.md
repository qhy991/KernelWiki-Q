## Challenge

Implementing a high-performance GEMM kernel using CUTLASS CUTE library requires orchestrating shared memory tiling, thread partitioning for both data loading and compute, async memory copy for latency hiding, and a mainloop that pipelines global-to-shared copies with register-level MMA operations


## Solution

Use CUTE tensor abstractions to build the GEMM: construct global tensors with make_tensor/make_gmem_ptr, extract CTA tiles via local_tile with coordinate stepping, partition data loading via local_partition with thread layouts tA/tB, partition compute via tC projection, and run a mainloop that copies tiles to shared memory with cp_async_fence/cp_async_wait followed by gemm() on register accumulators, finalizing with axpby epilogue

## Key Lessons

- Use local_tile with Step templates (Step<_1,X,_1> for A, Step<X,_1,_1> for B, Step<_1,_1,X> for C) to extract per-CTA subtensors from global tensors — the Step pattern encodes which dimensions are tiled vs iterated vs skipped
- Use separate thread layouts for loading (tA/tB via raked local_partition) and compute (tC via projection-based local_partition with Step) — this decouples memory access patterns from MMA compute patterns, enabling independent optimization of each
- The CUTE mainloop pattern is copy→cp_async_fence→cp_async_wait→syncthreads→gemm→syncthreads: fence/wait pairs mark async copy boundaries while syncthreads ensures shared memory consistency between the load and compute phases
