## Challenge

CUB BlockReduce with shared-memory TempStorage and __syncthreads() barriers for row-wise max and sum reductions adds unnecessary latency when each row is small enough for a single thread to own entirely


## Solution

Reorganize data ownership so each thread owns VPT=16 elements (entire row), enabling thread-local max/sum reduction via unrolled loops and __shfl_xor_sync for the general THREADS_PER_ROW>1 case, eliminating all shared-memory TempStorage, __shared__ broadcast variables, and __syncthreads() barriers

## Key Lessons

- When VPT >= num_cols, each thread can compute row-wise reductions (max, sum) entirely in registers via unrolled loops, eliminating shared-memory TempStorage, __shared__ broadcast variables, and __syncthreads() barriers — applies to any per-row operation where the row fits in one thread's register budget
- Use __shfl_xor_sync with a THREADS_PER_ROW-scoped mask as a generalization that handles both the single-thread-owns-row case (loop becomes a no-op) and multi-thread-per-row case without branching, making the kernel adaptive to different row widths
