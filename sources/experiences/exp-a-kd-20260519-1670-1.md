## Solution

Partition 256 threads into 32 groups of 8 threads each (group_size=128) to compute per-group absmax scales using shared memory staging (s_max_vals[1024]) and warp-level reduction via warpReduceMaxSpecialized instead of a single CUB BlockReduce — but the interleaved thread-to-group mapping breaks coalesced global memory access and synchronization overhead dominates for small groups

## Key Lessons

- Groupwise thread partitioning (threads_per_group = blockDim.x / num_groups) breaks the natural stride-by-blockDim.x memory coalescing — threads within a warp access non-contiguous memory regions belonging to different groups, and the resulting uncoalesced loads can negate any benefit from finer-grained scaling
- For small per-group thread counts (8 threads/group with group_size=128), the overhead of writing 256 values to shared memory, two __syncthreads() barriers, and a warp reduction operating on only 8 active elements exceeds the cost of a single CUB BlockReduce over the full vector
- To add per-group scaling without losing coalescing, keep the stride-by-blockDim.x scan pattern and accumulate per-group partial results in registers or a shared memory array indexed by group, then reduce within each group — avoids interleaved thread-to-group mapping entirely
