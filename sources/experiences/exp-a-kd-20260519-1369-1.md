## Solution

Replace scalar bf16 copy loops with 128-bit vectorized cache-streaming loads/stores via inline PTX (ld.global.cs.v4.u32 / st.global.cs.v4.u32) for the NOPE section and 32-bit cache-streaming (ld.global.cs.b32 / st.global.cs.b32) for the ROPE section, reducing per-warp instructions from 18 to 3 and signaling non-temporal streaming to bypass L2 allocation

## Key Lessons

- GPU coalescing hardware merges 32 adjacent scalar 2-byte lanes into the same 64-byte memory sector as 128-bit vectorized accesses (16 bytes × 32 lanes), so vectorization reduces instruction count but yields no throughput gain when the kernel is already bandwidth-saturated with enough warps to fill the memory pipeline
- Cache-streaming hints (.cs PTX modifier) reduce L2 cache line allocation for streaming data, but have negligible effect when the kernel runs alone on a large L2 cache (40 MB on A800) without cache contention from other consumers
- A 6× reduction in load/store instruction count per warp (18 → 3) is invisible when memory bandwidth is the bottleneck — instruction count optimization only helps when the kernel is compute-bound or latency-bound
