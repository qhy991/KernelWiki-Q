## Solution

Doubling TILE_K from 64 to 128 halves K iterations and sync barriers but increases cooperative shared-memory loading linearly — without tensor cores the arithmetic-intensity multiplier is absent, making the larger tile a net regression

## Key Lessons

- Larger K-tile sizes improve arithmetic intensity only when the compute path (tensor core HMMA) processes more MACs per shared-memory load — in a simplified scalar dot-product kernel the cooperative loading overhead scales linearly with TILE_K while the compute benefit is negligible
- Doubling the shared-memory tile (BM×TILE_K + TILE_K×BN) can reduce occupancy or increase bank conflicts, further degrading performance when the tile-size tuning is borrowed from a tensor-core CUTLASS config without the corresponding compute backend
- Tile-size parameters are not universally portable: CUTLASS GemmShape<128,128,128> is tuned for tensor-core throughput on sm75; applying the same TILE_K=128 to a naive cooperative-loading kernel without tensor cores yields a regression
