## Solution

Replaced scalar loop with vectorize_with_alignment<16> to batch 16 float loads into a single 64-byte vector load and 16 int8 stores into a single 16-byte store, but simultaneously reduced block size from 1024 to 256 threads, cutting warps per block from 32 to 8 and causing a severe occupancy regression that outweighs the memory coalescing benefit

## Key Lessons

- Vectorized memory access alone does not guarantee speedup — when vectorization increases register pressure (vec_n_t<float,16> = 64 bytes local storage per thread) and the block size is proportionally reduced (1024→256 threads), the resulting occupancy loss (32 warps→8 warps per block) can dominate and cause net regression
- When widening vector access from scalar to VEC_SIZE=16, maintain total parallelism by keeping block size constant or increasing grid dimensions — do not shrink block size proportionally unless occupancy analysis confirms sufficient SM utilization
- For memory-bound quantization kernels on sm_80 (A800), occupancy is the primary lever — aggressive vectorization (16-wide) with few threads leaves SM resources idle; prefer moderate vector width (4–8) with 512–1024 threads to balance coalescing and parallelism
