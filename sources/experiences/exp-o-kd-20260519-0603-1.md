## Challenge

Three separate kernel launches to resolve future token IDs create 256KB of unnecessary intermediate global memory writes and reads (neg_indices and gathered buffers), plus three kernel launch overheads that prevent any inter-kernel optimization such as register carry-over or shared memory reuse


## Solution

Fuse the three separate kernels (compute_neg_indices → gather_from_map → conditional_select) into a single-pass kernel that keeps the loaded value in registers throughout the condition-check, negate, clamp, and map-lookup sequence, eliminating intermediate buffers and reducing global memory traffic from 6 accesses per element to 2

## Key Lessons

- Fuse multi-stage elementwise pipelines into a single kernel when each stage processes the same index range — register-resident values eliminate intermediate global memory round-trips and buffer allocation overhead
- Kernel fusion removes per-launch overhead (grid setup, synchronization) and enables the compiler to keep intermediate values in registers rather than spilling to global memory — applies whenever consecutive kernels operate on the same element indices with trivial data dependencies
- Eliminating intermediate buffers (neg_indices, gathered) reduces memory pressure and frees GPU memory for other concurrent work — particularly impactful at scale where buffer sizes grow linearly with element count
