## Solution

Replacing the custom blockReduceMax with CUB's BlockReduce<float,1024> caused a 5.42% regression because the template dimension (1024) exceeds the actual blockDim.x (512), causing CUB's internal reduction phases to execute logic designed for 32 warps when only 16 are active — the overhead is proportional to the under-population ratio

## Key Lessons

- CUB BlockReduce template dimension should match the maximum expected blockDim.x — over-allocating (e.g., BlockReduce<float,1024> with only 512 threads) causes internal reduction phases dimensioned for 32 warps to execute with only 16 active warps, introducing overhead proportional to the under-population ratio
- Custom warp-shuffle block reductions can outperform CUB's generic BlockReduce when blockDim.x varies at launch time, because the custom implementation uses fixed shared memory sized to the actual warp count rather than a compile-time template parameter that may exceed the runtime thread count
