## Solution

Replace scalar bf16 access with flashinfer::vec_t vectorized load/store (8 bf16 elements per 16-byte transaction) and accumulate in float for higher arithmetic throughput; however the vectorization reduces thread count from 128 to 16, dropping below the minimum warps needed to hide memory latency for this small workload (128×128=16KB)

## Key Lessons

- Vectorized memory access via flashinfer::vec_t reduces memory transactions by N× for N-element vectors, but proportionally reduces thread count — when thread count drops below ~32 (1 warp), the SM has insufficient resident warps to hide memory latency, negating the bandwidth benefit — applies particularly to small row_stride dimensions where the original thread count is already modest
- Float accumulation (instead of bf16 __hadd/__hmul) in reduction loops improves numerical precision since bf16 has only 8-bit mantissa causing significant rounding error in multi-term sums — prefer float for any accumulation of 3+ terms, converting back to bf16 only at the final store
