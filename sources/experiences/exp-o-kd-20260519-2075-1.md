## Challenge

Batch-split grid launches with 1024 threads per CTA to gather paged KV-cache entries, but only 576 of 1024 threads do useful work per entry (< 56% utilization), while split boundary logic, nested block/token loops, and partial-block conditionals add divergent branching and address computation overhead


## Solution

Replace batch-split grid (1024 threads/CTA) with per-token grid (64 threads/CTA) where each CTA handles exactly one token via a token_to_seq lookup, yielding 9 meaningful iterations per thread (576/64) instead of <1, eliminating split/partial-block boundary logic and enabling higher occupancy through smaller CTAs on a memory-bound workload

## Key Lessons

- Size CTAs so each thread has multiple meaningful iterations (576/64=9 vs 576/1024<1) — when per-element work is a simple memory copy, over-provisioning threads wastes registers and reduces occupancy without any throughput benefit
- Prefer flat per-token grid mappings over batch-split grids for memory-bound gather kernels — eliminating split boundary logic, partial-block conditionals, and nested loops removes divergent branching and reduces address computation overhead
- Smaller CTA sizes (64 vs 1024) increase the number of concurrent CTAs per SM, which is the primary lever for hiding memory latency on bandwidth-bound workloads where compute is trivial
