## Challenge

Scalar absmax loop in dynamic scaled int8 quantization reads elements one-by-one with fabsf per iteration; vectorization via vectorize_read_with_alignment<16> is attempted to reduce memory instructions, but at hidden_size=256 with 256 threads the scalar path already achieves fully coalesced stride-1 access


## Solution

Replace the scalar absmax for-loop with vectorize_read_with_alignment<16> template that loads 16-float (128-byte) vectors and applies the absmax lambda per element, adding template indirection and DefaultReadVecOp unrolling without changing the effective per-thread computation

## Key Lessons

- Vectorized loads provide no benefit when the scalar path already achieves full memory coalescing with stride-1 access across all threads — the effective per-thread work is identical and template/lambda indirection can add measurable overhead
- At small hidden sizes (~256 elements), kernel launch overhead and shared-memory block reduction dominate the latency profile, making memory-level optimizations like wider loads unmeasurable and potentially counterproductive
- vectorize_read_with_alignment<VEC_SIZE> via DefaultReadVecOp unrolls the vector into per-element scalar lambda calls internally, so it does not reduce the number of fabsf/fmaxf operations — true vectorized absmax would require SIMD-style intrinsics operating on the vector register directly
