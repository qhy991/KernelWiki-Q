## Challenge

The naive kernel concatenates k_nope and k_rope using a grid-stride loop where each thread processes one bf16 element, requiring expensive per-element integer division and modulo for index decomposition, a conditional branch to select between the two source buffers, and generating uncoalesced scalar memory traffic


## Solution

Replace per-thread scalar processing with warp-cooperative vectorized access: each warp handles one (token, head_chunk) pair using lane_id for dimension-level parallelism with int2 loads (8 bf16 per lane for k_nope) and int loads (2 bf16 per lane for k_rope), register-buffering k_rope to reuse across all 16 heads in a chunk, achieving fully coalesced 128/256-byte transactions and eliminating per-element division, modulo, and conditional branching

## Key Lessons

- Map work to warps instead of individual threads — use flat_warp_id for token/chunk indexing and lane_id for dimension-level parallelism with vectorized int2 loads (8 bf16 per lane), producing fully coalesced 256-byte transactions; applies when head dimensions are multiples of warp_size × element_size
- Buffer shared read-only input in registers across loop iterations — k_rope is identical for all heads of the same token, so loading once and reusing across 16 head writes eliminates 15 redundant global memory reads; applies when the same input feeds multiple output locations
- Replace per-element integer division/modulo index decomposition with warp-level indexing — deriving token_id and head_chunk_id from flat_warp_id via cheaper division by small constants eliminates the expensive idx/(NUM_LOCAL_HEADS*K_HEAD_DIM) computation per element; applies when the problem has a natural hierarchy amenable to warp mapping
