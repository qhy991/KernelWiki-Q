## Solution

Replace 4 scalar __half2float(__low2half/__high2half) calls with 2 vector __half22float2 calls per iteration; on sm_80 this produces identical PTX cvt.rn.f32.f16 instructions with no instruction-level speedup, but provides a unified type-parameterized API across half/bfloat16/fp8 specialization paths

## Key Lessons

- __half22float2(half2) and __half2float(__low2half/__high2half) expand to identical cvt.rn.f32.f16 PTX on sm_80 — there is no hardware paired half2-to-float2 conversion unit, so vector intrinsics do not reduce instruction count on Ampere
- Prefer __half22float2 over scalar decomposition for code maintainability: it provides a unified type-parameterized API that works identically across half, bfloat16, and fp8 specialization paths, enabling a single conversion method name regardless of scalar type
- A measured speedup below ~1% is within benchmark timing noise and should be treated as neutral — do not attribute instruction-level performance gains to API-level substitutions that produce identical PTX
