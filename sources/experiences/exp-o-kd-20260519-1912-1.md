## Solution

Reduce TILE_M from 16 to 8 to match VALID_ROWS=8, halving per-block shared memory from 25152 to 12576 bytes and cutting the cooperative load loop from 2048 to 1024 iterations, which reduces shared memory traffic and eliminates wasted row slots without any structural kernel changes

## Key Lessons

- Match tile dimensions (TILE_M) to actual active row counts (VALID_ROWS) rather than rounding up to power-of-2 defaults — oversized tiles waste shared memory and increase cooperative load/store loop iterations even when guarded by valid-row checks
- Parameterizing shared memory layout by tile size (SH_A_FLOATS, SH_RED_FLOATS as compile-time macros derived from TILE_M) allows the compiler to propagate the smaller tile size through all allocation and loop bounds, achieving the reduction with zero structural code changes
- For MoE workloads with small expert batches (e.g. M=8), a conditional tile size (8 or 16 rows based on actual batch) avoids over-allocation and can yield 5-10% speedup per kernel launch from reduced shared memory traffic alone
