## Challenge

For a 4096-element (16KB) int32_t array fill, the scalar_before kernel writes one int32_t per thread per iteration with 256 threads in 16 blocks, while the vectorized_after kernel uses AlignedArray<int32_t, 4> (Vec) to write 4 elements per thread via reinterpret_cast<Vec*> on the output pointer with


## Solution

Vectorize memory access using float4 (128-bit) loads/stores so each thread processes 4 elements per single memory transaction, reducing total memory transactions by 4x and dramatically improving bandwidth utilization for the memory-bound ELU operation

## Results

- **speedup**: 0.998x (neutral)


## Key Lessons

- Use float4 (128-bit) vectorized loads/stores to process 4 elements per thread instead of scalar 32-bit accesses — reduces memory transaction count by 4x and is the primary optimization for memory-bound elementwise kernels where bandwidth is the bottleneck
- Scale thread count down by the vectorization factor (blockSize / 4) when each thread processes K elements, so total grid coverage remains correct without launching excess threads
- Separate computation into register-level vector components (reg_y.x/y/z/w) then write back as a single float4 — keeps the compute logic identical while gaining coalesced 128-bit memory throughput
