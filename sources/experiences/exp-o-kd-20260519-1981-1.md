## Challenge

getMIndices kernel reads expert_first_token_offset repeatedly from global memory without caching: each block reads up to 9 int64_t offsets multiple times across the token-count computation and aligned-offset accumulation loop, incurring redundant global memory latency


## Solution

Load the small expert offset array (9 int64_t = 72 bytes) into shared memory via __ldg to leverage the read-only texture cache during the load and eliminate L1 cache conflicts across blocks for subsequent accesses in the accumulation loop

## Key Lessons

- Cache small repeatedly-accessed lookup tables (few dozen bytes) in shared memory loaded via __ldg to bypass L1 pollution and avoid inter-block cache conflicts — the read-only texture cache path used by __ldg can service reads without evicting other useful L1 lines
- For compact data structures (<=4KB) shared across all threads in a block, a cooperative smem load with __syncthreads() is preferable to repeated global reads even when the data fits in L1 cache, because it eliminates dependence on L1 hit rate under multi-block contention
