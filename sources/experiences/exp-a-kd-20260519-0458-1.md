## Challenge

Scalar FP32 dot product kernel loads one float per thread from global memory, resulting in low memory throughput for a memory-bound reduction workload where each thread issues a single 4-byte load


## Solution

Replace scalar float loads with float4 vectorized loads (4 floats per thread) to improve memory throughput, while keeping the same grid size but reducing threads per block from 256 to 64 so total data coverage is preserved

## Key Lessons

- Vectorizing loads from float to float4 while proportionally reducing thread count (256→64) can cancel out the memory throughput gain because fewer warps reduce occupancy and the scheduler's ability to overlap memory latency — the net effect is neutral when the kernel is already memory-bound and occupancy-limited
- When applying vectorized loads, keep thread count constant (or reduce less aggressively) so the same grid covers more total data per launch, rather than covering the same data with fewer threads
