## Challenge

The RMS-norm dynamic per-token quantization kernel passes min_scaling_factor as a runtime float parameter (computed on host as std::numeric_limits<float>::epsilon()), adding an unnecessary kernel argument and preventing the compiler from folding the known constant into the instruction stream


## Solution

Replace the runtime min_scaling_factor kernel parameter with a compile-time constexpr type trait (MinScalingFactor<T>) specialized per quantization type, eliminating the kernel argument and enabling the compiler to constant-fold the value directly into the scale computation

## Key Lessons

- Compile-time constexpr type traits can replace runtime kernel parameters for constants determined entirely by the quantization type → enables compiler constant folding and reduces kernel parameter passing overhead → applies when the constant is fully known at compile time, though the performance benefit may be negligible when per-thread compute dominates parameter overhead in small kernels
- Template specialization per quantization type (e.g., MinScalingFactor<int8_t> vs generic MinScalingFactor<T>) provides type-safe per-type constant dispatch without runtime branching → compiles to the exact constant value inline → applies to quantization kernels where scaling floors differ by target integer type
