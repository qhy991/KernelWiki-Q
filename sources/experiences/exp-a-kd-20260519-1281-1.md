## Challenge

Two-pass bf16×bf16 GEMM pipeline writes bf16 output to global memory, then a separate cast kernel reads it back, converts to fp32, and writes again — doubling memory traffic (6 KB intermediate write + 12 KB cast read/write) for the output tensor


## Solution

Use cublasGemmEx with CUDA_R_32F output type instead of CUDA_R_16BF, so the GEMM accumulates in fp32 internally and writes fp32 results directly — eliminating the intermediate bf16 global-memory buffer and the separate cast kernel entirely

## Key Lessons

- Set cublasGemmEx output type to CUDA_R_32F when the consumer needs fp32 — cuBLAS natively supports mixed-precision output so a separate type-conversion kernel is never necessary, saving one full global-memory round-trip on the output tensor
- When a library kernel's output type differs from what downstream code requires, check if the library supports the target type directly before adding a post-processing pass — the internal accumulation precision (CUBLAS_COMPUTE_32F) and the output storage type (CUDA_R_32F) are independent knobs in cuBLAS
