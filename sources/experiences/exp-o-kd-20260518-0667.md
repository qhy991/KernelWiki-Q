## Challenge

Optimization focus: FP32 output accumulator storage (kOStorageAccFloat32=1) preserves precision during online softmax rescaling vs FP16 storage (kOStorageAccFloat32=0) which can lose precision in the rescale step.
Problem before: With kOStorageAccFloat32=0, the output accumulator R_D stores half val


## Solution

6 (__float2half_rn). This repeated FP16->FP32->FP16 round-trip loses precision as FP16 has only ~3-4 decimal digits, and errors accumulate across all Tc tiling steps.
Applied change: When kOStorageAccFloat32=1 (enabled for kHeadDim<256), R_D uses 4 uint32_t per element instead of 2, storing FP32 val

## Key Lessons

- Applied change: When kOStorageAccFloat32=1 (enabled for kHeadDim<256), R_D uses 4 uint32_t per element instead of 2, storing FP32 values directly.
