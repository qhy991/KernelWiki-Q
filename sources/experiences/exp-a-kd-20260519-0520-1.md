## Solution

The attempted optimization widened per-thread vectorization from f16x2 to f16x8 (loading 4 half2 registers and processing 8 elements per thread), but this degraded performance because the thread count dropped 4x (N/8 vs N/2), yielding insufficient warps to hide memory latency, and the 4 separate boundary checks added conditional branch overhead

## Key Lessons

- Increasing per-thread vectorization width reduces thread count proportionally; when total elements are small enough that thread count drops below the occupancy sweet spot (e.g., N/8 = 131K threads for N=1M), the loss of latency-hiding warps outweighs the per-thread memory coalescing benefit
- Each widened vectorization group requires a separate boundary check when the total element count is not guaranteed aligned; 4 conditional stores in f16x8 vs 1 in f16x2 adds branch divergence overhead that compounds the occupancy loss
