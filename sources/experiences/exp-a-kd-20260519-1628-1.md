## Challenge

K-cache fp8-to-half dequantization interleaves memory loads with arithmetic conversion in the same loop iteration, mixing dependent load and convert instructions and preventing the GPU scheduler from efficiently pipelining memory accesses


## Solution

Split the single load+convert loop into two phases: Phase 1 loads all raw fp8 bytes into a compact uint8_t Klocalb8[16][8] buffer (128 bytes, 1 byte/element), then Phase 2 batch-converts all 16 chunks to Half8 Klocal[16] (256 bytes) in a separate unrolled loop, separating pure memory loads from arithmetic conversion

## Key Lessons

- Separating memory loads from arithmetic conversion into distinct phases allows the GPU scheduler to issue pure load instructions without dependent arithmetic, but the benefit may be offset when both the compact raw buffer and the converted buffer coexist in registers simultaneously, increasing total register pressure
- Compact intermediate buffers (1 byte/element for fp8 vs 2 bytes/element for half) reduce register footprint during the load phase, but the overall register budget must account for both buffers alive between phases — at HEAD_SIZE=128 (KHELOOP=16), Klocalb8 needs 128 bytes and Klocal needs 256 bytes for a combined 384 bytes vs the inline approach's 256 bytes
