## Challenge

Query vectors are loaded per-thread into private registers (float q_vecs[64]), causing 128 threads to redundantly read 128 unique query values from global memory; the optimization attempts to eliminate this redundancy via cooperative shared-memory loading but introduces overhead that outweighs the benefit


## Solution

Replace per-thread register query storage with cooperative shared-memory loading (__shared__ float q_vecs[THREAD_GROUP_SIZE][NUM_VECS_PER_THREAD]) where all threads collaboratively load query vectors once and reuse from shared memory across key-token iterations, adding a __syncthreads() barrier before the inner loop

## Key Lessons

- Cooperative shared-memory loading for read-only query data is counterproductive when the inner loop is short (≤8 iterations) — the fixed __syncthreads() barrier and shared-memory access latency are not amortized across enough dot-product iterations to outweigh register-based access
- L2 cache can effectively mask redundant global reads of small working sets, making shared-memory deduplication unnecessary — when 128 threads re-read 128 unique float values, the L2 cache serves subsequent reads at near-register speed
- Register-file access (1-cycle) is fundamentally faster than shared-memory access (5+ cycles with bank conflicts possible); prefer registers for per-thread private data that fits within the register budget, especially when the data is only 64 floats (256 bytes) per thread
