## Challenge

CUTLASS int-output GEMM epilogue always loads the source tensor and computes beta*source + alpha*accum sequentially, even when beta=0 (OnlyAlphaScaling), wasting global memory bandwidth on an unnecessary load and performing a multiply-by-zero followed by a fused multiply-add


## Solution

Specialize the epilogue for OnlyAlphaScaling by removing the source_in parameter entirely, skipping source tensor load and conversion, and replacing the two-step beta*source then alpha*accum+intermediate with a single mul_accum(alpha, accum) call that directly computes D = alpha * accumulator

## Key Lessons

- Eliminate dead data paths at compile-time or via template specialization when ScaleType guarantees beta=0 — removing the source load and beta*source multiply saves memory bandwidth and ALU cycles even when the speedup is masked by kernel launch overhead at small problem sizes
- Kernel launch overhead (~0.1ms) can dominate microbenchmarks of epilogue-only kernels on small datasets (128KB), hiding genuine memory bandwidth savings from source-load elimination — benchmark at realistic problem sizes to reveal the benefit
- CUTLASS epilogue partial specialization by ScaleType (OnlyAlphaScaling vs generic) avoids unnecessary source tensor I/O — applies to any GEMM epilogue where the scaling mode is known at compile or configuration time
