## Challenge

sortAndScanExpert takes a non-const int* parameter for expert IDs, forcing every caller to clone topk_ids via cudaMemcpy DeviceToDevice before each sort invocation; for small batches where the CUB radix sort on 2048 key-value pairs is lightweight, the memcpy API dispatch and data copy overhead dominates total iteration time


## Solution

Change sortAndScanExpert and CubKeyValueSorter::run to accept const int* for the keys input parameter, matching CUB DeviceRadixSort::SortPairs which already takes const keys_in; this allows the original topk_ids buffer to be passed directly, eliminating the device-to-device memcpy clone entirely

## Key Lessons

- Adding const to function parameters that only read data eliminates unnecessary device-to-device memcpy clones — CUB's DeviceRadixSort::SortPairs already takes const KeyT* for input keys, so wrapper functions should propagate this const-correctness rather than forcing callers to allocate and copy mutable buffers
- For small problem sizes where compute kernels are lightweight, fixed API overhead (cudaMemcpy dispatch, synchronization) dominates total latency — removing even a small 8 KB D2D copy yields significant relative speedup when the sort itself is sub-microsecond
