## Challenge

All int8 quantized GEMMs on SM89 (Ada Lovelace) used a single fixed CUTLASS 2.x configuration (TileShape <128,128,64>, WarpShape <64,64,64>, 5 stages) regardless of problem shape, causing small-M problems to waste threads on oversized tiles while large-N problems underutilized shared memory with too few stages


## Solution

Replace the single fixed CUTLASS 2.x GEMM instantiation with cutlass_gemm_sm89_int8_dispatch, which selects from 7 M-range configurations (M16/M32/M64/M128/M256/default) each branching on N sub-ranges to choose tuned TileShape, WarpShape, and stage count that match the actual problem dimensions for higher occupancy and shared-memory utilization

## Key Lessons

- Use shape-aware dispatch to select CUTLASS tile/warp/stage configurations based on actual M and N dimensions rather than a single fixed config — small-M problems (e.g., LLM decoding batch size 1-16) benefit from narrower tiles like <16,64,128> while large-M problems use wider tiles like <128,128,64> or <256,128,64> for higher throughput
- Provide a shared-memory fallback GEMM config (e.g., <32,64,128> with reduced stages) for cases where the primary tuned config exceeds available SMEM per SM — applies whenever dispatch selects configs that may push shared-memory limits on certain problem shapes
