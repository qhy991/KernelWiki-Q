## Solution

Reduce kSmem from 32*1024*sizeof(uint32_t) (128KB) to 8*1024*sizeof(uint32_t) (32KB), since each radix round needs at most ~TopK candidate entries (2048) in the threshold bin, making 4K entries per round (8K total) sufficient while allowing 3–5 concurrent blocks per SM

## Key Lessons

- Reduce shared memory allocation to the minimum required by the algorithm's actual working set — in radix topk, each round only needs ~TopK candidates, not the full buffer size, so clipping SMEM_INPUT_SIZE from 16K to 4K entries per round is safe and dramatically improves occupancy
- When a kernel uses ping-pong double buffering (s_input_idx[][SMEM_INPUT_SIZE]), size each buffer for the realistic per-round candidate count rather than the maximum possible index count — the pos < SMEM_INPUT_SIZE guard already handles overflow safely
