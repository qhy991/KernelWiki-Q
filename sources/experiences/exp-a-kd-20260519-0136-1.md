## Solution

Use SmemLayoutA (Crosswise) layout for TensorRefA instead of Operator::LayoutA (RowMajor), matching the actual shared-memory layout of GEMM0's accumulator output so that TensorRef.at() computes correct permuted offsets via CrosswiseLayout::packed() stride initialization

## Key Lessons

- When a CUTLASS TensorRef layout type doesn't match the actual shared-memory layout, .at() computes wrong offsets — the layout parameter must exactly match how the data was written, not the logical view you want
- Volta tensor core crosswise layouts (RowMajorVoltaTensorOpMultiplicandCrosswise) trade higher addressing arithmetic for bank-conflict-free warp-level matrix loads — microbenchmarks using simple sequential reads will show regression, but real fMHA kernels benefit from eliminated bank conflicts during ldmatrix operations
- Use SmemLayoutA (the shared-memory layout from the upstream GEMM epilogue) rather than Operator::LayoutA (the math-domain layout) when constructing TensorRef for inter-GEMM shared-memory communication in fused kernels
