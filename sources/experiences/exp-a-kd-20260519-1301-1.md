## Challenge

MoE topk ID remapping kernel reads expert_map (only 8 ints = 32 bytes) from global memory on every thread access, prompting an attempt to cache it in shared memory to reduce global memory latency


## Solution

Cache the expert_map into shared memory via cooperative loading with __syncthreads() barrier, then read from smem instead of global memory — however, with only 8 ints (32 bytes) the entire map already fits in one L1 cache line, making shared memory caching ineffective

## Key Lessons

- Shared memory caching of a tiny lookup table (≤32 bytes, single cache line) provides no benefit over global memory reads because the data is already L1-cache-resident after the first thread accesses it — the added __syncthreads() barrier and cooperative load loop only introduce overhead
- Before applying shared memory caching, verify the lookup table exceeds one L1 cache line (128 bytes on most GPUs); otherwise the GPU's L1 cache already provides equivalent latency to shared memory
