## Challenge

SSM scan forward kernel stores recurrent state as __half, requiring __half2float conversion at each state read and __float2half conversion at each state write per dstate iteration, adding 32 conversion instructions per block that execute alongside 8192 FP32 arithmetic operations in the inner time-step loop


## Solution

Change ssm_states storage from __half to float, eliminating all half-to-float and float-to-half conversion instructions at state read/write boundaries; the FP32 state doubles memory footprint (64 vs 32 bytes per block) but state access is infrequent (once per dstate iteration, not per timestep), making bandwidth overhead negligible

## Key Lessons

- Eliminating half-to-float conversion in recurrent state storage is neutral when the conversion count (32 per block) is negligible relative to inner-loop arithmetic (8192 FP32 ops) and the conversions are hardware-accelerated — the optimization only matters when state access frequency approaches inner-loop frequency
- Decoupling state storage precision from input precision doubles state memory footprint with no performance gain when state is accessed infrequently (once per dstate iteration vs once per timestep) — the bandwidth cost of larger state may outweigh conversion savings in memory-bound regimes
- Quantify instruction elimination as a fraction of total instruction count before claiming benefit — removing 32 conversions from 8192+ arithmetic instructions (<0.4%) is within measurement noise for CUDA event timing at sub-millisecond kernels
