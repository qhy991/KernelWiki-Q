## Challenge

Per-block reduction uses atomicAdd to a single global memory address, causing severe atomic contention as all 512 threads in a block contend on the same location; additionally requires a separate global buffer allocation, cudaMemset reset before each launch, and a post-sync global read to retrieve the result


## Solution

Replace the global atomicAdd reduction with CUB BlockReduce, which performs a cooperative shared-memory tree reduction across all threads in the block, eliminating atomic contention entirely and removing the need for a global atomic buffer and its per-launch cudaMemset reset

## Key Lessons

- atomicAdd to a single global address from all threads in a block creates severe serialization; CUB BlockReduce uses a shared-memory reduction tree that avoids atomics entirely and is deterministically O(log N) in thread steps — use it whenever per-block reduction of a per-thread scalar is needed
- The fixed-cost overhead of a global atomic buffer (cudaMemset reset + extra global memory allocation + post-reduction global read) is proportionally dominant at smaller workloads; eliminating the buffer via shared-memory cooperative reduction yields larger speedups when the compute loop is short relative to these overheads
- CUB BlockReduce::TempStorage is allocated in __shared__ memory with size known at compile time from the block dimension — it adds zero global memory traffic and the shared-memory reduction is bandwidth-optimal for the GPU's on-chip interconnect
