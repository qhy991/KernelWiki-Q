## Challenge

Kernel uses cooperative shared-memory load of expert offsets with __syncthreads barrier, then only lane 0 performs a serial accumulation loop over up to 8 experts, leaving 31 warp lanes idle, followed by __shfl_sync broadcast — causing thread divergence, barrier stalls, and wasted warp occupancy


## Solution

Move the expert-aligned offset prefix-sum accumulation to host-side pre-computation into a new aligned_expert_first_token_offset array; each GPU thread then independently computes its result with two __ldg reads from the pre-computed arrays, eliminating shared memory, __syncthreads, serial lane-0 loop, and __shfl_sync broadcast entirely

## Key Lessons

- Prefer host-side pre-computation of prefix-sum-style aligned offsets over per-block GPU accumulation loops when the data (e.g., expert offsets) is small and static per kernel launch — eliminates shared memory, __syncthreads barriers, and warp divergence at the cost of one extra GPU array
- Replace serial lane-0 accumulation + __shfl_sync broadcast with per-thread __ldg reads from pre-computed read-only arrays — __ldg hits the read-only cache and avoids both warp serialization and cross-lane communication overhead
- When a kernel's shared memory is only used for cooperative load of a small read-only table, replacing it with direct __ldg per-thread reads removes the shared memory footprint, the __syncthreads barrier, and the cooperative load loop in one change
