## Challenge

The same vectorization mechanism as the smaller case: the before kernel uses scalar float I/O with per-element modular index decomposition (idx%c_out, cascade integer divisions for w/h/n), processing one 4-byte element per thread iteration. The after kernel casts to float2 and halves the channel dim


## Solution

Vectorize scalar float I/O to float2 by casting pointers and halving channel dimensions (c_in/2, c_out/2), so each thread processes 2 adjacent channels via one 8-byte load/store, halving total elements from 262k to 131k and reducing global memory transactions by 2x.

## Results

- **speedup**: 1.136x (improved)


## Key Lessons

- Vectorize scalar float I/O to float2/float4 when channel dimensions are even — halves memory transactions and loop iterations with minimal code change (pointer cast + divide channels by vector width), applies when adjacent elements share identical processing logic and memory is the bottleneck
- Index arithmetic cost bounds vectorization speedup — when per-element integer modulo/division dominates, doubling memory width only speeds up the memory fraction of total runtime, yielding sub-linear gains (~14% instead of theoretical 2x)
- Casting device pointers to float2* and dividing channel counts by 2 is a low-risk vectorization path that preserves the original index decomposition logic unchanged — suitable for quick wins on memory-bound elementwise kernels
