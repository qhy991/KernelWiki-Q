## Challenge

NaN-propagating max in a reduction kernel is implemented with a multi-instruction branch sequence (two isnan checks plus a ternary comparison), costing 4+ instructions per max operation even though the GPU provides a single hardware FMNMX.NAN instruction


## Solution

Replace the multi-instruction isnan+ternary sequence with a single PTX max.NaN.f32 inline asm instruction on SM80+, which maps directly to the FMNMX.NAN SASS instruction and preserves NaN semantics in hardware; fall back to the original branch sequence on older architectures

## Key Lessons

- PTX max.NaN.f32 maps to a single FMNMX.NAN SASS instruction, replacing a 4+ instruction isnan+branch sequence — but the speedup is negligible in memory-bandwidth-bound kernels where instruction-level savings are hidden by memory stalls
- Single-instruction PTX replacements for comparison operations yield measurable gains only in compute-bound scenarios where the operation is on the critical path and not hidden by memory latency
- Use __CUDA_ARCH__ guards with inline PTX asm to target SM80+ hardware while preserving correct fallback on older architectures
