## Challenge

GEMM epilogue keeps the int32_t accumulator as int32_t before converting to float for scale+bias computation; in CUTLASS CollectiveEpilogue this causes the builder to select an int32-specialized epilogue code path that produces incorrect results for sparse GEMM on SM90


## Solution

Convert the int32_t accumulator to float immediately upon read so the epilogue computes scale+bias entirely in float; in CUTLASS this means changing the 2nd CollectiveEpilogue template parameter from ElementAcc (int32_t) to float, which selects the correct float-precision epilogue builder path for SM90 sparse GEMM

## Key Lessons

- In CUTLASS CollectiveEpilogue, the 2nd template parameter controls the epilogue compute type and dispatches to different builder specializations — using ElementAcc (int32_t for int8 inputs) instead of float selects the wrong code path for SM90 sparse GEMM, even though at the raw CUDA level the I2F + FMUL/FADD instruction sequences are identical
- Always convert int32 GEMM accumulators to float before epilogue arithmetic (scale, bias, activation) to preserve fractional precision and ensure correct CUTLASS template dispatch, regardless of the input element type
