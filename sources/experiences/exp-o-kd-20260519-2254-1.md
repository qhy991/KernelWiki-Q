## Challenge

The baseline gather kernel processes elements one-at-a-time in a single loop, performing an immediate __ldg read followed by a store on each iteration, creating a serialized read-then-write dependency chain where write port occupancy can delay subsequent reads and prevents the GPU memory subsystem from issuing multiple outstanding read requests concurrently


## Solution

Apply loop unrolling with factor 4 to process four elements per iteration, separating the read phase (batching all __ldg loads into register-allocated local arrays keys_to_store/values_to_store) from the write phase (issuing all stores from those arrays), which allows the GPU to issue multiple outstanding read requests concurrently, hides read latency, and avoids read-after-write resource conflicts on the memory pipeline

## Key Lessons

- Separate read and write phases within an unrolled iteration — batching all __ldg loads into register-allocated local arrays before issuing any stores eliminates read-after-write pipeline conflicts and allows the GPU memory subsystem to issue multiple outstanding read requests concurrently, yielding significant latency hiding for memory-bound gather kernels
- Loop unrolling with #pragma unroll on the inner loop gives the compiler full scheduling freedom to reorder independent loads for maximum throughput — applies whenever loop iterations have no cross-iteration dependencies and the unroll factor fits comfortably in registers
- Register-allocated local arrays (scalar_t arr[N] with small N) act as zero-overhead buffers that decouple load latency from store completion — effective when N is small enough to avoid register spilling (typically N ≤ 8 for half-precision)
