## Challenge

Standard min() with uint32_t arguments in LDS load loop boundaries can cause LLVM on AMD ROCm to silently promote the comparison to double-precision (v_min_f64), adding unnecessary FP64 overhead even though the operands are unsigned integers


## Solution

Wrap min() in an explicit __device__ inline min__(uint32_t, uint32_t) function that forces the compiler to stay in the unsigned integer domain, preventing LLVM from silently upcasting to double-precision on ROCm while remaining semantically identical on NVIDIA

## Key Lessons

- Wrap standard library functions (min, max) in explicit-type __device__ helpers when targeting cross-platform CUDA/ROCm code — on ROCm, LLVM's overload resolution can silently promote uint32_t comparisons to double-precision, while NVIDIA's compiler already generates correct integer instructions
- A micro-optimization that is neutral on one platform (NVIDIA IMIN stays integer) may be critical on another (ROCm v_min_f64 promotion) — always validate on target hardware rather than assuming cross-platform parity
