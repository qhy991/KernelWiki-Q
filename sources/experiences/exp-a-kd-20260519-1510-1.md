## Challenge

Custom blockReduceMax uses a function-pointer dispatch pattern (ReduceFnType<T>) for warp-level shuffle reduce followed by shared-memory cross-warp aggregation, which is verbose but compact — only a shared[maxActiveLanes=32] array is allocated, keeping shared-memory usage minimal


## Solution

Replace custom blockReduceMax with cub::BlockReduce<float,1024>.Reduce using cub::Max functor, which provides a library-quality block reduction but at this block size allocates more shared memory (TempStorage includes internal buffers for multi-stage reduce) than the custom implementation's compact shared[32] array, resulting in a 7% regression

## Key Lessons

- CUB BlockReduce's TempStorage can be larger than a hand-rolled warp+shared-memory reduction at large block sizes (1024 threads) — measure shared-memory footprint before assuming CUB is always more efficient, especially for simple max/sum reductions
- A compact custom block reduce using warp shuffles + a small shared[32] array can outperform CUB when the reduction operation is trivial (max) and nvcc inlines the function pointer at -O3, eliminating dispatch overhead
- Library abstractions add value for complex reductions (scan, argmax, segmented) but for single-operator block-wide max, the custom approach's minimal shared-memory allocation is a genuine advantage at occupancy-sensitive block sizes
