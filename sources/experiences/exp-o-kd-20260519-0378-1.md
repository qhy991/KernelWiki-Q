## Solution

Switch to double-stage async copy pipeline (kStage=2): preload first Q/K/V tiles into smem_stage_0, then each iteration issues CP_ASYNC_CG to load the next tile into the alternate shared memory buffer (smem_sel_next) while MMA proceeds on the current stage (smem_sel), with deferred CP_ASYNC_WAIT_GROUP(kStage-2) after MMA to overlap global memory transfers with tensor core computation

## Key Lessons

- Double-stage async copy pipeline (kStage=2) overlaps global memory loads with tensor core MMA computation by loading the next tile into an alternate shared memory buffer while computing on the current one — applies whenever the kernel iterates over multiple tiles and has enough iterations to amortize the preload overhead
- Deferred CP_ASYNC_WAIT_GROUP(kStage-2) after MMA compute is less restrictive than immediate CP_ASYNC_WAIT_GROUP(0) before compute, allowing the async copy engine to continue transferring while tensor cores are active
- Pipeline staging requires sufficient iteration count to amortize the preload cost — with 8 K/V tiles (seq_len=512, Bc=64), the double-stage pipeline has enough work to hide gmem latency effectively
