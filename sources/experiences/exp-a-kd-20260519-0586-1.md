## Challenge

Scalar sigmoid kernel loads and stores one float per thread, issuing separate LDG/STG instructions for each element and leaving vectorized memory bandwidth underutilized


## Solution

Vectorize loads/stores using float4 so each thread processes 4 elements, reducing the number of memory instructions by 4x and enabling wider 128-bit memory transactions

## Key Lessons

- float4 vectorization of loads/stores provides negligible benefit when the working set fits entirely in L2 cache and the scalar kernel already issues coalesced contiguous accesses — the memory controller merges them into wide transactions regardless
- expf-heavy kernels are compute-bound regardless of memory access width; reducing load/store instruction count does not improve throughput when expf latency dominates the critical path
- Vectorized memory access optimizations are most effective when data exceeds L2 cache capacity and the kernel is memory-bound; at small working-set sizes the benefit is absorbed by cache residency
