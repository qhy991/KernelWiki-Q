## Challenge

An explicit local variable (vin_t tmp = v_in[i]) is introduced to materialize a vec4<half> pack before passing it to a __forceinline__ read operator, but NVCC -O3 already fuses the global load directly into the inlined operator when the global-memory reference is passed without copy


## Solution

Pass the global-memory reference v_in[i] directly to the __forceinline__ operator instead of materializing a local copy; NVCC -O3 with __forceinline__ already inlines the callable and emits a single LDG.8 or LDG.16, so the explicit local variable only adds unnecessary register traffic and prevents load-use fusion

## Key Lessons

- Do not materialize local copies of small vector packs (≤16 bytes) before passing them to __forceinline__ operators — NVCC -O3 already fuses the global load into the inlined expansion, and an explicit local copy prevents load-use fusion and adds register traffic overhead
- A 14.6% regression from a single local variable copy demonstrates that compiler load-use fusion is fragile: introducing a named temporary for a read-only operand can break the fusion even at -O3, because the compiler must honor the copy semantics
