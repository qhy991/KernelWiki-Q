## Challenge

Three separate CUDA kernels (update_tokens, update_lens_pos, update_slot_mapping) are launched sequentially for only 64 queries, each launching a single block of 256 threads that does a handful of scalar memory operations, making kernel launch overhead the dominant cost rather than computation


## Solution

Fuse all three kernels into a single advance_step_kernel where each thread computes token update, seq_len increment, position update, and slot mapping in one pass, keeping seq_len in a register to avoid the inter-kernel global memory dependency that forced the slot_mapping kernel to read input_positions from a previous kernel's output

## Key Lessons

- Fuse multiple trivial kernels (each doing only a few scalar memory ops per thread) into a single kernel when the per-kernel computation is too small to amortize launch overhead — applies especially to serving metadata updates with small batch sizes where kernel launch cost dominates total latency
- Keeping intermediate results (seq_len, next_input_pos) in registers across fused operations eliminates inter-kernel global memory round-trips that would otherwise be needed when dependent kernels read output of prior kernels
- For small workloads (e.g., 64 queries), launching with sm_count blocks instead of ceil(N/256)=1 block improves SM utilization by distributing the trivial work across all available multiprocessors
