## Solution

Widen per-thread work from 2 to 8 elements by using 4 half2 register pairs, but this shrinks block size from 128 to 32 threads (1 warp), drastically reducing occupancy and memory-level parallelism for the memory-bound kernel — the scheduler cannot issue enough concurrent memory requests to saturate bandwidth

## Key Lessons

- Increasing per-thread vector width without adjusting total threads-per-block can backfire for memory-bound kernels: halving threads to widen work reduces warps per block, starving the memory scheduler of concurrent requests
- For memory-bound elementwise kernels at modest problem sizes (1M elements), occupancy and memory-level parallelism dominate over instruction-level parallelism — prefer more threads with moderate vector width (half2) over fewer threads with aggressive vectorization (4×half2)
- When widening per-thread work from N to M elements, keep total threads-per-block at or above 128 (≥4 warps) to maintain sufficient occupancy for latency hiding — reduce grid stride or increase blockDim instead of shrinking block size
