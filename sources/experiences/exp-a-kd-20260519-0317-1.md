## Challenge

The before kernel already maps each of the 256 outputs to its own block with 256 threads iterating over 512 keys, providing ample device parallelism. Attempting to further split the short inner key dimension across multiple blocks introduces coordination overhead (atomicAdd per partial result plus cudaMemset every launch) that overwhelms any parallelism gain.


## Solution

Split the 512-key reduction into 2 partial blocks per output using a 2D grid (dim3(kNumSplits, kNumOutputs)), accumulate partial sums via atomicAdd, and zero the output buffer with cudaMemset before each launch — but the atomic contention and memset overhead dominate when the key dimension is short and output-level parallelism already saturates the device.

## Key Lessons

- Splitting a reduction's inner dimension into separate blocks only helps when per-block work is large enough to amortize the atomicAdd and memset coordination cost — with only 512 keys and 256 already-independent output blocks, the split doubles launch overhead for negligible compute gain
- cudaMemset before every kernel launch is a hidden cost that grows with output count; prefer in-kernel zeroing or block-local accumulation with a single final write when the reduction can stay within one block
- Evaluate existing device occupancy before adding parallelism — if gridDim already provides enough blocks to fill the SM (256 blocks × 256 threads on most GPUs), further splitting yields diminishing returns and introduces contention
