## Challenge

The baseline gather_cached_kv_kernel processes each element one-at-a-time in a stride loop: compute source/target indices, read from cache via __ldg, then immediately write to the output. This serialized read-then-write pattern creates a tight dependency chain that limits instruction-level parallelism and can cause memory pipeline stalls on read-after-write hazards, especially with scattered paged KV cache access patterns.


## Solution

Unroll the loop by a factor of 4 and separate read/write into two phases: first batch 4 __ldg reads into local scalar_t arrays in a #pragma unroll inner loop, then write all 4 values to output in a separate #pragma unroll inner loop. This phase separation decouples memory pipelines, giving the compiler freedom to schedule multiple outstanding reads to hide latency.

## Key Lessons

- Separating read and write phases in gather/copy kernels (batch __ldg reads into local arrays before issuing writes) decouples memory pipelines and enables the compiler to schedule multiple outstanding reads for latency hiding — most effective when warp-level parallelism is limited (small token counts)
- Loop unrolling combined with read/write phase separation increases instruction-level parallelism for scattered access patterns like paged KV cache gathers, where the 5D interleaved key layout creates non-coherent memory requests that benefit from having multiple reads in flight simultaneously
