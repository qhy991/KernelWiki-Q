## Challenge

Scalar absmax kernel loads one float per thread per iteration, processing elements individually with fabs+max pairs, which at small data sizes (4096 floats, 16KB) does not benefit from explicit vectorization since the compiler already coalesces scalar loads into wider memory transactions when warp threads access consecutive addresses


## Solution

Replace scalar float loads with 4-wide vec4_t struct loads via reinterpret_cast, unroll the loop 4x, and add a remainder loop for tail elements — however this adds instruction overhead (struct field extraction, reinterpret_cast indirection) that outweighs any bandwidth benefit when data fits entirely in L1/L2 cache and the compiler already coalesces scalar accesses

## Key Lessons

- Explicit vectorized loads (vec4_t via reinterpret_cast) can regress performance on small data sizes (≤16KB) where data fits in L1/L2 cache and the compiler already coalesces scalar warp-wide loads into wide memory transactions — vectorization overhead (struct field extraction, reinterpret_cast, remainder loop) outweighs bandwidth gains
- On sm_80+ architectures (A800/H100), NVCC coalesces consecutive scalar float loads from warp threads into 128-byte memory transactions automatically, making manual vec4_t struct packing redundant for throughput-bound reduction kernels
- Loop unroll pragmas and remainder-element loops add instruction count and register pressure without benefit when the working set is cache-resident — always profile at the target data size before applying vectorization transformations
