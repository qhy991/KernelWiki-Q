## Challenge

Scalar FP16 ReLU kernel processes one half value per thread with separate 16-bit load and store per element, underutilizing available memory transaction width when consecutive half elements could be packed into 32-bit operations


## Solution

Vectorize to half2 (32-bit) load/store so each thread processes two elements, halving the number of memory transactions and naturally coalescing two half values per 32-bit word

## Key Lessons

- Vectorizing half to half2 loads/stores halves memory transaction count and naturally coalesces two half values per 32-bit word — benefits are most pronounced when data exceeds L2 cache capacity, as smaller datasets already saturate bandwidth via scalar coalesced access
- At ~2MB working sets that fit entirely in L2 cache, both scalar and half2 kernels are memory-bandwidth-bound with similar throughput, making the vectorization advantage negligible at small problem sizes
