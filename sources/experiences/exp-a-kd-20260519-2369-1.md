## Solution

Apply XOR-based index remapping (transform_a(i) = a_gl_rd_delta_o * row + (i % a_gl_rd_delta_o) ^ row) to shared memory indices for both writes and reads, precomputing all transformed indices into arrays at kernel startup to eliminate bank conflicts without runtime recomputation

## Key Lessons

- XOR-based shared memory index remapping eliminates bank conflicts for int4/ldmatrix tile access patterns, but the precomputation overhead (register allocation for index arrays + ALU for XOR computation) must be amortized over many tile reads — in microbenchmarks with few accesses per kernel, the setup cost dominates and causes regression
- Bank conflict elimination techniques like XOR remapping show their full benefit only in production kernels (e.g., full Marlin GEMM with hundreds of inner-loop tile iterations) where the one-time index precomputation is reused across many shared memory transactions, making the per-access cost negligible
- Prefer measuring shared memory optimizations in realistic workload contexts — isolated write-barrier-read microbenchmarks understate the benefit of conflict-free layouts because the barrier already hides short conflict stalls that would accumulate in tiled inner loops
