## Solution

Templatize compute_problem_sizes with a SWAP_AB<bool> parameter; when token count M is below a threshold, swap M↔N in the problem_sizes layout (e.g. problem_sizes1 becomes [2*n, occurrences, k] instead of [occurrences, 2*n, k]) so the downstream CUTLASS SM90 GEMM sees a tall-skinny matrix that maps better to TMA tile scheduling

## Key Lessons

- Swapping M↔N dimensions in GEMM problem sizes is a zero-cost metadata transformation on the setup kernel whose real payoff is downstream: when M (token count) is small, making the larger hidden-dim the M dimension lets CUTLASS SM90 TMA tiles cover more work per launch → applies whenever batch-parallel GEMM has a small M dimension that starves the tensor core pipeline
- Use compile-time template parameters (template<bool SWAP_AB>) rather than runtime branching for GEMM layout variants — the compiler eliminates the unused branch entirely and the setup kernel pays no divergence cost
- For lightweight metadata kernels (problem size setup, index computation), kernel-level benchmarking is misleading; the performance impact is in the downstream compute kernels that consume the metadata → validate swap_ab-style optimizations end-to-end, not on the setup kernel alone
