## Challenge

For a skinny GEMM with M=32, N=32 the 2D WMMA grid produces only 4 blocks (2x2 tiles of 16x16), leaving most of the 108 SMs idle while each block sequentially traverses the entire K=2880 dimension in a single loop


## Solution

Split the K dimension into NUM_K_SHARDS=2 shards using a 3D grid (grid.z=2), doubling the block count from 4 to 8; each block computes a partial 16x16 tile over K/2 elements, writes to shared memory via store_matrix_sync, then threads merge partial sums into global C[] using atomicAdd

## Key Lessons

- Split the K dimension into independent shards along grid.z to multiply block count for skinny GEMM shapes where the output tile grid produces too few blocks to saturate SMs — each shard computes a partial result and merges via atomicAdd, trading lightweight reduction overhead for substantially higher SM utilization
- Stage WMMA accumulator results to shared memory before atomic-merging to global memory, since store_matrix_sync can write to shared memory but WMMA fragments cannot be directly scattered to global atomics
