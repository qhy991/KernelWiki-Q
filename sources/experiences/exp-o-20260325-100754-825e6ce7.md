## Challenge

Top-k sampling kernel for vocab_size=256, k=6. Baseline used torch headers (violated pure CUDA constraint). Initial curand-based implementation had 4.7ms latency dominated by curand_init overhead, giving performance_ratio=0.0.


## Solution

Replace curand_init with inline fast LCG (Linear Congruential Generator) RNG seeded with batch_idx XOR clock64(). Use one warp per batch item with warp-shuffle-based argmax for top-k selection.

## Key Lessons

- curand_init is extremely expensive (~4.7ms) and dominates latency for small kernels - always avoid it for latency-sensitive sampling kernels
- Fast LCG RNG (inline, no state init) reduces latency from 4.7ms to 0.008ms - a 587x latency improvement
- For top-k sampling with small vocab (256), one warp per batch item with warp-shuffle argmax is efficient
- The baseline_comparison uses batch_1 gbps by default; if batch_1 data is tiny (1KB), gbps rounds to 0.0 - need low latency to get non-zero gbps
- Always use /home/qinhaiyan/anaconda3/envs/robust_kbench/bin/python for running tests (not system python3 which lacks torch)
- For top-k sampling correctness: the sampled index must be one of the top-k indices - use cumulative probability sampling over top-k
