## Solution

Replace thread-per-expert with cooperative binary search across all threads: every thread iterates over blocks using a stride loop and performs O(log2(num_experts))=6 binary search iterations on the prefix array to determine the owning expert, distributing work evenly but incurring 6 global memory reads per block in isolated testing

## Key Lessons

- Algorithmic parallelism does not guarantee speedup: distributing binary search across all threads introduces O(log N) global memory reads per element that outweigh the benefit of higher thread utilization when the baseline only needs 2 reads per active thread
- Low thread occupancy is acceptable when total memory traffic is minimal — 64 threads each doing 2 reads + sequential writes can beat 1024 threads each doing 6 binary-search reads from global memory
- Memory hierarchy context determines whether cooperative search is profitable: when the prefix array resides in shared memory (e.g., after a Blelloch scan in a fused kernel), binary search avoids global memory round-trips and may become worthwhile, but as a standalone kernel the global memory reads per iteration dominate cost
