## Solution

Add a num_splits_key argument and compute an adaptive split count from the number of key tiles (ceil_div(num_keys, kBlockSizeJ)), with a runtime heuristic that disables splitting when existing grid parallelism is already sufficient and caps the split count when gradient accumulation for gK/gV would create excessive temporary workspace traffic.

## Key Lessons

- Splitting the key-sequence dimension across gridDim.x in backward attention lets independent CTAs process different key tiles of the same batch-head concurrently, dramatically improving occupancy for long-sequence cases where the original grid was nearly empty
- A runtime heuristic should guard split-key dispatch: disable when existing parallelism (grid.x * grid.y * grid.z) already exceeds a threshold and key length is short, and cap the split count when gK/gV accumulation staging would blow up temporary workspace — over-splitting short problems introduces overhead that can regress latency
- Expose a caller-provided num_splits_key override to allow downstream users to force-split or force-disable based on profiling, while the default heuristic handles the common case automatically
