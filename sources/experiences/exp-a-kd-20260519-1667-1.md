## Challenge

The original compute_dynamic_per_token_scales uses a monolithic CUB BlockReduce over all block threads to produce a single per-token absmax scale, sacrificing quantization granularity by using one scale for the entire hidden_size dimension instead of finer per-group scales


## Solution

Replace the single CUB BlockReduce with groupwise shared-memory partitioning: divide block threads into num_groups sub-groups (hidden_size/group_size), each group computes its own absmax into shared memory, then a custom warp-level reduction (warpReduceMaxSpecialized) produces independent per-group scales, enabling finer quantization granularity without CUB dependency

## Key Lessons

- Replacing CUB BlockReduce with groupwise shared-memory + warp-level reduction enables per-group scale computation for finer quantization granularity → applies when quantization accuracy benefits from sub-token scale differentiation at the cost of added synchronization barriers
- At small kernel durations (~6us), kernel launch overhead dominates making algorithmic reduction strategy changes indistinguishable within measurement noise → latency gains from reduction restructuring are only measurable for kernels with substantial compute time beyond launch overhead
- Groupwise thread partitioning using integer division/modulo (threadIdx.x / threads_per_group for group ID, threadIdx.x % threads_per_group for lane ID) maps block-level parallelism to independent data sub-ranges cleanly → applies when a single block must compute multiple independent reductions over contiguous data segments
