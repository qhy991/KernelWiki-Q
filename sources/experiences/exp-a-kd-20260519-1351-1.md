## Challenge

Default GEMM tile configuration (BM=128) applied to M=64 wastes 50% of the M-dimension tile as padding since only 64 of 128 rows carry useful data, while the large shared memory footprint (16KB per block) limits SM occupancy for small-batch workloads


## Solution

Switch to an M-appropriate tile (BM=32, BN=64, BK=64) where M=64 maps to exactly 2 fully-utilized blocks in the y dimension, eliminating all M-dimension padding, reducing shared memory to 12KB for higher occupancy, and doubling BK to improve compute-to-memory ratio per K-tile iteration

## Key Lessons

- Select tile BM that evenly divides the problem M dimension — when M < BM, up to half the threads compute masked-out padding rows, wasting both compute and shared memory bandwidth
- Smaller tiles with larger BK improve the compute-to-memory ratio per K-tile iteration because each shared memory load is amortized over more multiply-accumulate operations along the K dimension
- Adaptive tile selection based on M (large tiles for large M, compact tiles for small M) can yield 3-4x speedup by simultaneously eliminating padding waste, increasing occupancy, and boosting arithmetic intensity
