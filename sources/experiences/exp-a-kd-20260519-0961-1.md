## Challenge

The topk kernel requires the caller to pre-offset the input pointer (d_input + row_start_offset) to handle ragged/non-contiguous data rows, which breaks when multiple rows share a single buffer with different starting offsets and forces host-side pointer arithmetic per row


## Solution

Add a per-row row_start parameter to fast_topk_cuda_tl and a row_starts array in TopKParams, computing input[idx + row_start] inside the kernel so each row can have a variable offset into a shared buffer without host-side pointer pre-adjustment

## Key Lessons

- Adding a runtime integer offset (row_start) to array indexing is essentially free when the kernel's dominant cost is in shared-memory atomics and histogram computation — applies to any radix-based or histogram-heavy kernel where address arithmetic is dwarfed by memory and synchronization costs
- Passing a per-row row_starts array rather than requiring pre-offsetted pointers is the correct API pattern for ragged batch processing, enabling variable start positions across rows without extra host-side pointer arithmetic
- Null-checking the row_starts array (params.row_starts == nullptr ? 0 : ...) provides backward compatibility with existing callers that do not need ragged offsets, avoiding a breaking API change
