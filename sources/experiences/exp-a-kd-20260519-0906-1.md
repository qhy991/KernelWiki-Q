## Challenge

The benchmark loop allocates two small device buffers (tokens_cnts and cumsum) via cudaMalloc/cudaFree inside every iteration, forcing four synchronous device-wide stream synchronization stalls per iteration that dominate the 18.24ms per-iteration cost even though the kernel itself executes in microseconds


## Solution

Pre-allocate the tokens_cnts and cumsum buffers once before the benchmark loop and reuse them across all iterations, relying on the kernel's internal zeroing of per-thread rows to safely reset state between invocations, then synchronize once after all iterations complete

## Key Lessons

- Pre-allocate device buffers once and reuse them across kernel invocations instead of calling cudaMalloc/cudaFree per iteration — each cudaMalloc/cudaFree forces a synchronous device-wide stall (~4-5ms each) that completely dominates cost for fast kernels
- A kernel that internally zeroes its working buffers (e.g., clearing per-thread count rows at the start of each invocation) can safely reuse pre-allocated memory without external reinitialization, making buffer pre-allocation correctness-preserving
- When benchmarking CUDA kernels, always move buffer allocation outside the timed loop and use a single cudaStreamSynchronize after all iterations to measure pure kernel launch and execution overhead
