## Solution

Replace the integer zero-point path with a float16 zero-point path where ZP values are already stored as fp16 and loaded directly via int4 vectorized loads, eliminating the dequant_4bit unpack+convert step entirely from the inner loop

## Key Lessons

- Store zero-points as fp16 instead of packed int4 to eliminate per-iteration dequant arithmetic (shift+mask+__float2half) — applies whenever zero-points are applied repeatedly in an inner loop and memory overhead is acceptable
- Precompute quantized values into their compute format (fp16) at load time rather than dequantizing on every use — trades memory for reduced instruction count in hot loops
- Using int4 vectorized loads to fill fp16 register arrays (frag_zpf) allows bulk loading 8 half values in one instruction, further reducing memory traffic compared to scalar int loads of packed values
