## Challenge

Shared memory allocation for the A-matrix tile always assumes 2 bytes per element (FP16/BF16), even when activation type is INT8 requiring only 1 byte, causing over-allocation that prevents INT8 thread configs from fitting within shared memory limits on constrained GPUs like Turing (SM75, 64KB)


## Solution

Detect 8-bit activation at runtime via a_type.size_bits() == 8 and conditionally select element size in the shared memory calculation: stages * (tb_m * tb_k) * (is_a_8bit ? 1 : 2), halving A-matrix shared memory for INT8 and freeing space for larger tile configurations

## Key Lessons

- Size shared memory allocations by the actual element byte-width of the data type at runtime rather than hardcoding a fixed size — INT8 uses 1 byte vs FP16/BF16's 2 bytes, so halving the per-element footprint directly halves (or quarters with fewer stages) shared memory usage
- On shared-memory-constrained architectures (Turing SM75 with 64KB), correct element sizing is the difference between a valid and invalid thread config — the same tile that fails at 2 bytes/element may fit comfortably at 1 byte/element, yielding large speedups (2.38x here)
