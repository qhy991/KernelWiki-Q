## Challenge

Two independent GEMM kernels are launched on the default CUDA stream (stream 0), which serializes their execution — the second kernel cannot begin until the first completes, leaving SMs idle during the gap between kernel dispatches


## Solution

Assign each independent GEMM to a dedicated CUDA stream (stream1, stream2) so the GPU scheduler can overlap their execution across the SM pool, reducing total wall-clock time from the sum of both serial launches to approximately the max of either launch; also add cudaSetDevice(0) to ensure correct device context on multi-GPU systems

## Key Lessons

- Launch independent kernels on separate CUDA streams to enable concurrent SM occupancy — applies whenever kernels have no data dependencies and each individually uses fewer than all available SMs
- Explicitly set the device context with cudaSetDevice (or at::cuda::CUDAGuard in PyTorch) before stream operations on multi-GPU systems to avoid silent errors from a stale device context
- Use cudaStreamSynchronize per stream instead of cudaDeviceSynchronize when benchmarking overlapping kernels — the former allows concurrent execution to complete naturally while the latter serializes the entire device
