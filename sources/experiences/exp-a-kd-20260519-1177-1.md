## Challenge

SiLU-and-mul kernel uses 128-bit int4 loads via __ldg to read input, issuing one 16-byte LDG per vector chunk and underutilizing available memory bandwidth on hardware that supports wider transactions


## Solution

Replace 128-bit int4 loads/stores with 256-bit ld256/st256 using inline PTX (ld.global.nc.v8.u32 / st.global.v8.u32), halving the number of memory transactions; on cc<100 the helpers decompose to two uint4 loads as fallback

## Key Lessons

- 256-bit ld.global.nc.v8.u32 / st.global.v8.u32 PTX instructions only execute as single 32-byte transactions on cc>=10 (Blackwell+); on sm_80 they decompose into two 128-bit loads, yielding no bandwidth gain — verify target architecture before expecting wider-transaction benefits
- Provide a cc-conditional fallback in ld256/st256 helpers (two uint4 __ldg loads) so the code compiles and runs correctly on older architectures while getting the full 256-bit path on supported hardware
