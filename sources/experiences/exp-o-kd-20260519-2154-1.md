## Challenge

The advance_step_kernel computes four per-query outputs (input_tokens, seq_lens, input_positions, slot_mapping) but FlashInfer additionally needs paged_kv_last_page_len and block_table_bound per query; computing these in a separate kernel incurs an extra kernel launch overhead (~5-10 microseconds) and redundant global memory reads of seq_lens and block_tables data already loaded by the first kernel


## Solution

Fuse paged_kv_last_page_len (block_offset + 1) and block_table_bound (div_ceil(next_seq_len, block_size)) computations directly into the advance_step kernel body, reusing the seq_len, next_seq_len, block_index, and block_offset values already held in registers from the base computation and requiring only two additional global memory stores with zero redundant reads

## Key Lessons

- Fuse per-query metadata computations (paged_kv_last_page_len, block_table_bound) into a kernel that already reads the same source data (seq_lens, block_tables) — eliminates a separate kernel launch and avoids redundant global memory reads by deriving additional outputs from values already in registers
- Kernel fusion is especially beneficial when the additional computation reuses intermediate register values (seq_len, block_index, block_offset) from the base logic, requiring only extra stores rather than extra loads
- Convert early-return control flow (if (cond) return) to positive-guard flow (if (cond) { body }) when extending a kernel with fused outputs — positive guards keep the entire computation body within one scope, simplifying addition of new output stores at the end of the block
