## Solution

Replace the hardcoded FP8_E4M3_MAX macro with a type-dependent template trait fp8_e4m3_adjusted_max_v<fp8_type> that resolves to the correct maximum per FP8 format at compile time (448.0f for e4m3fn on CUDA, 224.0f for e4m3fnuz on ROCm), achieving portable correctness as a zero-cost abstraction

## Key Lessons

- Replace hardcoded format-specific constants with constexpr template traits so the compiler selects the correct value per target platform — identical generated code on current platform, correct behavior on others
- Zero-cost abstraction principle applies to compile-time template constants: NVCC resolves fp8_e4m3_adjusted_max_v<fp8_e4m3fn_t> to 448.0f inline, producing the same SASS as a #define macro
- Cross-vendor FP8 quantization must account for format differences (e4m3fn max=448 vs e4m3fnuz max=224) — a single hardcoded constant causes silent numerical errors on the alternate platform
