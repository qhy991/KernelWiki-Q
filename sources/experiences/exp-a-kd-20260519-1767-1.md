## Challenge

GEMM scheduler uses a conditional heuristic (k > 3*n) to dispatch between StreamK split-K (4-way K-dimension split with a separate accumulation kernel and partial-result buffer) and single-GEMM paths, introducing unnecessary branching complexity, extra memory allocation, and an additional kernel launch for problem shapes where the heuristic selects the simple path anyway


## Solution

Remove the conditional StreamK/Persistent heuristic entirely and always use a single cuBLAS Sgemm call followed by the scale kernel on a dedicated CUDA stream, eliminating the split-K accumulation kernel, conditional dispatch logic, partial-result buffer allocation, and replacing cudaDeviceSynchronize with stream-level synchronization

## Key Lessons

- For balanced GEMM shapes (m ≈ n ≈ k) where k < 3*n, a conditional StreamK split-K heuristic provides no benefit over a single cuBLAS call — the heuristic overhead (extra memory allocation, accumulation kernel launch, conditional branching) is wasted when the simple path is selected anyway
- Using a dedicated CUDA stream with cublasSetStream + cudaStreamSynchronize instead of cudaDeviceSynchronize enables finer-grained synchronization that only waits on the relevant stream's work, reducing synchronization overhead when other stream work is concurrent
- Removing the split_k_accum kernel and partial-result buffer eliminates one kernel launch and nsplits*m*n*sizeof(float) bytes of GPU memory per iteration, reducing both launch overhead and memory footprint for the common single-GEMM path
