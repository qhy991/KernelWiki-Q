## Solution

Split the monolithic single-block kernel into two kernels: a lightweight align/count kernel (single-block, no sorting) and a multi-block sort kernel (64 blocks x 256 threads) using grid-stride loops with global-memory atomicAdd on cumsum_buffer, distributing sorting across multiple SMs to reduce per-SM atomic contention and increase aggregate throughput

## Key Lessons

- Splitting a monolithic single-CTA kernel into align + sort phases and running the sort phase across multiple CTAs with grid-stride loops reduces shared-memory atomic contention and leverages more SMs — applies whenever sorting/scattering dominates kernel runtime and the count phase is relatively cheap
- Grid-stride loops (for i = tid; i < numel; i += stride) distribute work evenly across arbitrary block counts, making the sort kernel scalable without manual workload partitioning
- Reusing the cumsum buffer as atomic counters for the sort phase avoids an extra memory allocation and initialization pass — the prefix-sum output doubles as the scatter base address array
